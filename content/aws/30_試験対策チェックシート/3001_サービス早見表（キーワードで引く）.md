# AWSサービス早見表｜キーワードで引く

> [!info] このシートの目的
> SAAは**暗記**ではなく「**この要件（キーワード）が出たら、このサービス**」という“反射”の試験。このシートは**キーワードから逆引き**して答えのサービスを当てるための実戦用早見表。
> - 考え方・記憶術は → [[AWS学習のコツ_用語整理術]]
> - 演習の進め方は → [[SAA問題演習の進め方]]
> - **問題文そのものが読めない**とき → [[3002_問題文の読み方（4ステップ分解）]]（文の分解手順）／[[3003_問題文頻出用語辞典]]（カタカナIT用語の翻訳表）
> - 各サービスの詳細は分野フォルダ（01〜18）の学習ノートへ。
>
> **使い方**：問題で詰まったら §1で要件の言葉から逆引き → §2で誤答の型を確認 → §3で紛らわしいペアを即断 → §4の自問で詰める。

## §1. 最頻出「キーワード → サービス」反射表（まずここ）

問題文の左の言葉を見たら、右が浮かぶ状態を目指す。**ここが埋まれば選択肢の半分は即消せる。**

**読み方**：左の言葉が問題文に出たら、右が浮かぶように。分野ごとにまとめてあるが、本番は分野を意識せず「言葉→サービス」で引く。

## §2. 「ひっかけ」頻出パターン（誤答の型）

選択肢を切るときの“あるある”。これを知ると罠を踏まない。

- **WAFに「DDoS」と書いてある** → WAFはL7（アプリ層）。大規模DDoSの自動緩和は **Shield**。
- **「Web ACL」をShieldに紐づける** → Web ACLは **WAF** の部品。Shieldではない。
- **IAMポリシー（IDベース）で“他アカウント”に権限を与える** → 不可。他アカウントに使わせるのは **リソースベースポリシー**。
- **Lambdaで30分の処理** → Lambdaは**最大15分**。超えるなら ECS/Fargate や Batch。
- **マルチAZ“インスタンス”で読み取りスケール** → 待機系は読めない。読み取り分散は**リードレプリカ**／マルチAZ**クラスター**。
- **Memcachedでセッション保持** → Memcachedは揮発・複製なし。セッションは **Redis**。
- **egress-only インターネットゲートウェイ** → **IPv6専用**の外向き。IPv4は NAT Gateway。
- **単一AZ構成** → ほぼ**単一障害点＝不正解**サイン。高可用はマルチAZ。
- **自前でKubernetes/サーバーを構築** → 「運用負荷を抑えたい」要件ではほぼ**不正解**（マネージドを選ぶ）。
- **「リアルタイム検知」にLogs Insights/Athena** → これらは**事後の検索・分析**。常時監視は CloudWatch アラーム等。

## §3. 紛らわしいペア “即断” 集

迷いやすい2択を、決め手の一言で。

- **WAF vs Shield** … アプリ層の不正リクエスト＝WAF／物量DDoS＝Shield。
- **CloudFront Functions vs Lambda@Edge** … 軽い(ヘッダ/URL)＝Functions／重い(画像/外部アクセス)＝Lambda@Edge。
- **SQS vs SNS** … 取りに行く(プル/キュー)＝SQS／送りつける(プッシュ/通知)＝SNS。
- **CloudTrail vs Config** … 操作の証跡＝CloudTrail／構成の記録・準拠＝Config。
- **Secrets Manager vs Parameter Store** … 自動ローテーション要＝Secrets Manager／単なる設定値＝Parameter Store(無料)。
- **ECS vs EKS** … AWS独自＝ECS／Kubernetes＝EKS。
- **EC2起動タイプ vs Fargate** … ノードを自分で管理＝EC2／管理不要＝Fargate。
- **Athena vs Redshift** … S3に都度SQL＝Athena／本格DWH常設＝Redshift。
- **EMR vs Glue** … クラスタ細かく制御＝EMR／サーバーレスETL＝Glue。
- **KMS vs CloudHSM vs Secrets Manager** … 鍵をマネージド＝KMS／鍵を専有自己管理＝CloudHSM／秘密情報そのもの＝Secrets Manager。
- **NAT Gateway vs VPCエンドポイント** … ネットへ出る＝NAT／AWSサービスへ閉域＝エンドポイント。
- **マルチAZ vs リードレプリカ** … 可用性(待機系)＝マルチAZ／読み取りスケール＝リードレプリカ。
- **Organizations vs Control Tower** … アカウント管理&SCPの土台＝Organizations／ガードレール自動整備＝Control Tower。

## §4. 仕上げの自問（試験で詰まったら）

1. この要件の**主目的はどのカテゴリ**？（compute/storage/db/network/security/監視…）
2. **「マネージド/運用最小」**と書いてある？ → 自前構築・EC2手組みは切る。
3. **「高可用性」**？ → 単一AZ・単一インスタンスは切る。
4. **「コスト最適」**？ → 常時起動・過剰スペックは切る。
5. **キーワードと§1反射表**は一致してる？ → 一致する選択肢が本命。
