# 問題文頻出用語辞典｜サービス名以外の「読めない言葉」を潰す

> [!info] このノートの目的
> SAAの問題文が読めない原因の半分は、AWSサービス名ではなく**一般IT用語のカタカナ語**（ワークロード・プロビジョニング・疎結合…）。
> この辞典は「問題文に出た言葉 → 頭の中でどう読み替えるか」の翻訳表。**訳語だけでなく、その言葉が出たときの試験上の意味（反射）**までセットで覚える。
> - AWSサービスの略語（DX・SG・MGN等）は → [[0410_AWS略語・用語集]]
> - 文全体の読み方は → [[3002_問題文の読み方（4ステップ分解）]]

---

## §1. 基本のカタカナ語（これが読めないと全文が霞む）

| 用語 | 読み替え | 問題文での意味・反射 |
|---|---|---|
| **ワークロード** | 動かしているシステム・処理 | 「分析ワークロード」＝分析システム。深い意味はない、「仕事のかたまり」程度 |
| **プロビジョニング** | （リソースを）**用意する・確保する** | 「プロビジョニングされたEC2」＝用意済みのEC2。「プロビジョンド〜」＝事前確保型（↔オンデマンド） |
| **デプロイ** | （アプリを本番に）**配置する・展開する** | 「デプロイする」＝動く状態にして置く |
| **インスタンス** | （仮想サーバーの）**1台** | EC2インスタンス＝仮想サーバー1台。RDSインスタンス＝DBサーバー1台 |
| **リソース** | AWS上の**部品全般** | EC2もS3バケットもIAMロールも全部「リソース」 |
| **アーキテクチャ** | システムの**構成・設計図** | 「アーキテクチャを設計する」＝どの部品をどう組むか決める |
| **ソリューション** | **解決策（＝答えの構成）** | 「ソリューションアーキテクト」＝構成を設計する人（＝受験者のロール） |
| **コンポーネント** | システムの**構成部品** | 「アプリケーションのコンポーネント間」＝部品どうしの間 |
| **レガシー** | **古い**既存システム | 「レガシーアプリケーション」＝改修しにくい古いアプリ → 互換性・改修最小の文脈 |
| **モノリシック** | **一枚岩**（全機能が1つのアプリ） | ↔ マイクロサービス（機能ごとに分割）。「モノリスを分割」＝疎結合化の文脈 |
| **スケーラビリティ** | **増減に対応できる能力** | 「将来の成長に対応」＝スケーラブルな構成（Auto Scaling / サーバーレス / DynamoDB） |
| **スケールアップ／アウト** | 1台を**強化**／台数を**増やす** | アウト（水平）が基本の正解。アップ（垂直）は上限あり・再起動要 |
| **エンドポイント** | **接続の入口**（URL・接続先） | 「APIエンドポイント」＝APIの入口URL。VPCエンドポイントとは別物なので文脈で判断 |

## §2. 場所・環境系

| 用語 | 読み替え | 問題文での意味・反射 |
|---|---|---|
| **オンプレミス（オンプレ）** | **自社運用の**データセンター・サーバー室 | 出たら「ハイブリッド接続」の文脈 → DX / VPN / Storage Gateway / DataSync |
| **ハイブリッド** | オンプレ＋クラウドの**併用** | オンプレとAWSをつなぐサービスが答えに絡む合図 |
| **データセンター** | サーバーを収容する施設 | 「自社データセンター」＝オンプレのこと |
| **マルチテナント／シングルテナント** | 他社と**相乗り**／**専有** | 「専有が必要（コンプラ・ライセンス）」→ Dedicated Hosts / CloudHSM |
| **リージョン／AZ** | 地域（例：東京）／その中の**独立したデータセンター群** | AZ障害に耐える＝**マルチAZ**。リージョン障害に耐える＝**マルチリージョン（DR）** |
| **エッジロケーション** | 世界中にある**配信拠点** | CloudFront / Global Accelerator / Route 53 が使う「ユーザーの近く」 |
| **クラウドネイティブ** | クラウド前提で設計された | マネージド・サーバーレスに寄せる合図 |

## §3. 移行系（「〜に移行したい」問題の頻出語）

| 用語 | 読み替え | 問題文での意味・反射 |
|---|---|---|
| **リフト＆シフト（リホスト）** | **そのまま**引っ越し（改修なし） | アプリを変えずにEC2へ → MGN。「最小限の変更で」とセットで出る |
| **リプラットフォーム** | 少しだけ変えて引っ越し | 例：DB on EC2 → RDS（エンジンは同じ） |
| **リファクタリング／リアーキテクト** | **作り直し**て引っ越し | 例：モノリス → マイクロサービス＋サーバーレス。工数最大・効果最大 |
| **移行（マイグレーション）** | 引っ越し全般 | データ移行=DMS/DataSync/Snow、サーバー移行=MGN、と対象で使い分け |
| **レプリケーション** | **複製**（コピーを別の場所に維持） | 同期レプリケーション＝常に同一（マルチAZ）／非同期＝少し遅れる（リードレプリカ・クロスリージョン） |
| **同期／非同期** | **待つ**／**待たない** | 同期＝書き込み完了を待つ（RPOゼロ・遅い）。非同期＝待たない（速い・少し失われうる） |
| **カットオーバー** | 新システムへの**切替の瞬間** | 「ダウンタイム最小でカットオーバー」＝継続レプリケーション（DMS等）の合図 |

## §4. 性能系（数字と一緒に出る言葉）

| 用語 | 読み替え | 問題文での意味・反射 |
|---|---|---|
| **レイテンシ** | **応答までの待ち時間** | 「低レイテンシ／ミリ秒単位」→ ElastiCache / DynamoDB / CloudFront / NLB |
| **スループット** | 単位時間あたりの**処理量・転送量** | 「高スループット」→ st1(EBS) / FSx for Lustre / Kinesis 等、量をさばく文脈 |
| **IOPS** | 1秒あたりの**読み書き回数** | 「高IOPS・トランザクション」→ io2 / gp3(EBS)。DBの文脈で出る |
| **帯域幅（バンド幅）** | 回線の**太さ** | 「帯域が限られる」→ 物理輸送（Snow Family）の合図になることも |
| **バースト** | **一時的な**性能の跳ね上がり | 「バースト可能」＝普段は低性能・一時的に高性能（T系インスタンス / gp2） |
| **スパイク** | アクセスの**急増** | Auto Scaling / SQSバッファ / サーバーレスの合図 |
| **ボトルネック** | 全体を遅くしている**詰まり所** | 「DBがボトルネック」→ キャッシュ（ElastiCache）かリードレプリカ |
| **リアルタイム／ニアリアルタイム** | 即時／**ほぼ**即時（数秒） | ストリーミング系（Kinesis）の合図。「事後にまとめて」ならバッチ（S3+Athena） |

## §5. 可用性・障害系（DR問題の必須語彙）

| 用語 | 読み替え | 問題文での意味・反射 |
|---|---|---|
| **高可用性（HA）** | **止まりにくい**こと | マルチAZが基本の答え。単一AZ・単一インスタンスの選択肢を消す |
| **耐障害性／フォールトトレランス** | 障害が起きても**動き続ける**こと | HAより強い要求（無停止）。冗長化を厚めに |
| **単一障害点（SPOF）** | そこが壊れると**全部止まる**箇所 | 「SPOFをなくす」＝1台構成・1AZ構成を消す |
| **冗長化** | **予備を持つ**こと | マルチAZ・複数NAT GW・レプリカ |
| **フェイルオーバー** | 障害時に**予備へ自動で切り替わる**こと | RDSマルチAZ / Route 53フェイルオーバー / ENI付け替え |
| **ダウンタイム** | システムが**止まっている時間** | 「ダウンタイムを最小限に」＝無停止移行・Blue/Green・継続レプリケーション |
| **RTO／RPO** | 復旧までの**時間**／失ってよい**データの量（時間分）** | 数字が出たら[[1801_DR戦略4パターン（RTO_RPO）]]の4戦略と照合 |
| **耐久性 vs 可用性** | データが**消えない**こと vs **いつでも使える**こと | S3の「イレブンナイン」は**耐久性**。混同させる選択肢に注意 |
| **グレースフル（に終了）** | **後始末をしてから**穏やかに | 「グレースフルシャットダウン」＝処理中のものを終えてから停止 |

## §6. 運用系（「運用負荷」問題の語彙）

| 用語 | 読み替え | 問題文での意味・反射 |
|---|---|---|
| **運用上のオーバーヘッド** | 人手の**管理の手間**（パッチ・監視・スケール調整…） | 「最小限に」→ マネージド／サーバーレス一択（[[AWS学習のコツ_用語整理術]]） |
| **マネージド（サービス）** | 面倒をAWSが見てくれる | Amazon/AWSの名前つき＝マネージド。「EC2に構築」＝自前運用 |
| **サーバーレス** | サーバーの存在を**意識しなくてよい** | Lambda / Fargate / DynamoDB / S3 / Athena / Aurora Serverless |
| **パッチ適用** | OSやソフトの**更新プログラムを当てる**こと | 「パッチ管理をなくしたい」→ サーバーレス。「自動化したい」→ SSM Patch Manager |
| **オンデマンド** | **使うときだけ・使った分だけ** | ↔ プロビジョンド（事前確保）。従量課金の文脈 |
| **プロビジョンドキャパシティ** | **事前に確保した**処理能力 | DynamoDB等。読みが予測可能→プロビジョンド、不規則→オンデマンド |
| **ベストプラクティス** | AWS推奨の**定石** | 「ベストプラクティスに従い」＝奇抜な構成を消す |
| **ガードレール** | **逸脱を防ぐ**ための統制ルール | Control Tower / SCP の文脈 |
| **ガバナンス** | 組織としての**統制・管理** | Organizations / Control Tower / Config の文脈 |

## §7. アプリ設計系（アプリ統合問題の語彙）

| 用語 | 読み替え | 問題文での意味・反射 |
|---|---|---|
| **疎結合（デカップリング）** | 部品どうしを**直接つながない** | 出たらほぼ **SQS**（間にキューを挟む）。SNS / EventBridge も候補 |
| **密結合** | 部品が直接依存し合う（壊れやすい） | 「密結合を解消」＝疎結合化 → SQS |
| **ステートレス** | 状態（セッション等）を**自分の中に持たない** | スケールアウトしやすい。セッションは外出し → ElastiCache / DynamoDB |
| **ステートフル** | 状態を**自分の中に持つ** | スケールしにくい。スティッキーセッションの文脈 |
| **セッション** | ログイン中ユーザーの**状態情報** | 「セッション管理」→ ElastiCache（Redis）が定番 |
| **キュー** | 処理待ちの**行列** | SQS。「一時的にためる」「バッファ」も同義 |
| **バッファ（リング）** | 急増分を**一時的にためて**あとで処理 | 「書き込みが集中」→ SQSを挟んでDBを守る |
| **ファンアウト** | 1つのイベントを**複数の宛先へ同時配布** | SNS → 複数SQS の定番構成 |
| **ポーリング** | 定期的に**取りに行く**（プル） | SQSはポーリング。↔ プッシュ（SNSが送りつける） |
| **イベント駆動** | 何かが**起きたら**動く | Lambda + S3イベント / EventBridge。常駐サーバー不要の合図 |
| **バッチ処理** | ためて**まとめて**処理 | ↔ リアルタイム。夜間バッチ→スポット/Batch/スケジュール |
| **べき等（冪等）** | **何度実行しても結果が同じ** | SQSの「少なくとも1回配信」→ 重複してもよい設計、の文脈 |
| **スロットリング** | リクエストの**流量制限** | API Gatewayの機能。「制限をかけたい/かかって困る」の両方で出る |
| **インジェスト（取り込み）** | データを**システムに入れる**こと | 「ストリーミングデータの取り込み」→ Kinesis / Firehose |
| **ETL** | 抽出→**変換**→書き込み（データの加工パイプライン） | サーバーレスETL＝Glue |
| **データレイク** | 生データを**そのまま全部**貯める湖 | ほぼ S3。「S3に貯めてAthenaで分析」が定番 |
| **データウェアハウス（DWH）** | 分析用に**整理された**大規模DB | Redshift。データレイク（S3）との対比で出る |

## §8. セキュリティ系

| 用語 | 読み替え | 問題文での意味・反射 |
|---|---|---|
| **保管時の暗号化（at rest）** | **置いてあるデータ**の暗号化 | S3 SSE / EBS暗号化 / RDS暗号化 → 鍵はKMS |
| **転送時の暗号化（in transit）** | **通信中のデータ**の暗号化 | TLS/HTTPS。ACM証明書の文脈 |
| **最小権限の原則** | **必要な権限だけ**与える | 広い権限（`*`）の選択肢を消す。IAMポリシーを絞る方向が正解 |
| **認証／認可** | **本人確認**／**何をしてよいかの許可** | 認証=Cognito・IAM、認可=IAMポリシー |
| **フェデレーション** | 外部の認証基盤と**連携**してログイン | 「既存のAD/Googleアカウントで」→ IAM Identity Center / Cognito |
| **多要素認証（MFA）** | パスワード＋**もう1つ**の確認 | ルートユーザー保護・特権操作の文脈 |
| **監査証跡** | 「**誰が何をしたか**」の記録 | CloudTrail。「証跡・監査」と出たら反射でよい |
| **コンプライアンス** | 法規制・業界基準への**準拠** | 暗号化＋監査＋Config（準拠チェック）のセット |
| **PII（個人識別情報）** | 個人を特定できるデータ | S3内のPII検出＝Macie |
| **閉域（接続）** | **インターネットを通らない**経路 | DX / VPCエンドポイント / PrivateLink |
| **インバウンド／アウトバウンド** | **入ってくる**通信／**出ていく**通信 | SG/NACLのルール方向。egress＝外向き、ingress＝内向き |
| **WORM** | 一度書いたら**変更・削除不可** | S3 Object Lock / Backup Vault Lock。「改ざん防止・ランサムウェア対策」 |
| **ペネトレーションテスト** | 侵入テスト | 文脈語。Inspector（脆弱性スキャン）と混同注意 |

## §9. 問題文の定型フレーズ→即翻訳

読み飛ばしがちな「決まり文句」の正体。

| 定型フレーズ | 翻訳（試験上の意味） |
|---|---|
| 「ソリューションアーキテクトは〜を設計する必要があります」 | ＝あなたへの依頼。この後に本命の要件が来る |
| 「既存のアプリケーションへの変更を最小限に抑えたい」 | **改修最小** → 互換のあるマネージド（例：ActiveMQ→Amazon MQ、MySQL→RDS/Aurora MySQL互換） |
| 「開発チームはビジネスロジックに集中したい」 | ＝インフラ管理をしたくない → サーバーレス |
| 「予測できないトラフィック」 | → オンデマンド系（DynamoDBオンデマンド / Aurora Serverless / Lambda） |
| 「アクセスパターンが不明／変化する」（ストレージ文脈） | → S3 Intelligent-Tiering |
| 「〜への直接アクセスを許可すべきではありません」 | 禁止要件（最強）。破る選択肢を即消し → [[3002_問題文の読み方（4ステップ分解）]] |
| 「将来的な拡張／ビジネスの成長に対応」 | スケーラビリティ → 上限のある構成（単一EC2等）を消す |
| 「専門知識を持つ担当者がいません」 | 運用負荷最小 → フルマネージド |
| 「規制により、データは〜に保存する必要があります」 | データレジデンシー（保存場所の制約）→ リージョン指定・Outposts |
| 「数回クリックするだけで／自動的に」 | マネージド機能で完結する選択肢が正解の合図 |
