# 問題文の読み方｜4ステップ分解

> [!info] このノートの目的
> 「解説を読めば分かるのに、**問題文の段階で6割しか理解できない**」を解消するための読解手順書。
> SAAの長文シナリオは、実は**毎回同じ4つの部品**でできている。部品ごとに読む場所と読み方を固定すれば、初見の問題でも迷子にならない。
> - 知らない**一般IT用語**（ワークロード・プロビジョニング等）が原因なら → [[3003_問題文頻出用語辞典]]
> - 知らない**AWSサービス名**が原因なら → [[3001_サービス早見表（キーワードで引く）]] / [[0410_AWS略語・用語集]]
> - このノートは「**単語は分かるのに文全体で何を聞かれているか掴めない**」を担当する。

---

## 大前提：問題文は「上から順に」読まない

SAAの問題文は小説ではないので、頭から丁寧に読むと**飾りの情報**（企業紹介・業種・経緯）に体力を吸われる。
読む順番を次のように固定する。

```
① 設問の締め（最後の1文）   ← 何を選ばされるのか
② 優先軸（「最も〜」の言葉）  ← 同点のときの決着ルール
③ 要件文（〜する必要がある）  ← 満たすべきチェック項目
④ シナリオ（残り全部）      ← ③の補足として流し読み
```

> [!tip] たとえ話：宝探しの地図
> 問題文は宝の地図。最初に見るべきは「**宝の絵（＝何を探すのか）**」で、それは地図の**一番下（設問の締め）**に描いてある。何を探すか知らずに地図の隅々（企業の背景説明）を眺めても、時間だけが溶ける。

## SAA問題文の解剖図（毎回この4部品）

[[401_【問題】SAA模擬試験レベル]] 第1問を例に、部品に分解してみる。

> ある企業は、複数のアベイラビリティーゾーンにまたがるVPC内で、3層構成のウェブアプリケーションを実行しています。**←④シナリオ（状況説明）**
> アプリケーション層のAmazon EC2インスタンスはプライベートサブネットで稼働しており、ソフトウェアの更新プログラムをインターネットからダウンロードする必要があります。**←③要件1**
> ただし、セキュリティ要件により、インターネットからこれらのEC2インスタンスへ直接アクセスできてはなりません。**←③要件2（禁止形＝特に強い）**
> ソリューションアーキテクトは、アベイラビリティーゾーン障害にも耐えられる構成にする必要があります。**←③要件3（＝②優先軸：可用性）**
> これらの要件を満たすために実行すべきアクションの組み合わせはどれですか。**（2つ選択）** **←①設問の締め**

この問題の「読めた」状態とは、次の1行に要約できた状態を指す。

```
プライベートのEC2が (a)外に出られて (b)外から入られず (c)AZ障害に耐える
──そのための操作を2つ選ぶ
```

問題文が長くても、要約は**必ず1〜2行に収まる**。収まらないのは要件と飾りを区別できていないサイン。

## ステップ① 設問の締めを最初に読む

最後の1文（「〜はどれですか。」）で、**解答のタイプ**が決まる。ここを最初に読むと、シナリオを読む目的がハッキリする。

| 締めの文言 | 選ばされるもの | 読み方のモード |
|---|---|---|
| 「使用すべき**サービス**はどれですか」 | サービス1つ | キーワード→サービスの反射（[[3001_サービス早見表（キーワードで引く）]]） |
| 「満たす**構成**はどれですか」 | 組み合わせ構成 | 選択肢の**どの部分が要件とズレるか**を探す |
| 「実行すべき**アクションの組み合わせ**（2つ選択）」 | 補完し合う操作2つ | 2つで1セット（後述） |
| 「**最もコスト効率よく**満たすのはどれですか」 | 全部正解っぽい中の最安 | 「動く選択肢」が複数ある前提で読む |
| 「**運用上のオーバーヘッドが最小**のものはどれですか」 | 全部正解っぽい中の手間なし | マネージド/サーバーレスに寄せる |
| 「ソリューションアーキテクトは何を**推奨すべき**ですか」 | ベストプラクティス | AWSの「模範解答」的な型を選ぶ |

> [!warning] 「全部できそうな選択肢」が並ぶ理由
> SAAの上位問題（4xx/5xx）は「**どれでも一応動く**」選択肢を並べて、優先軸（コスト・運用・可用性）で1つに絞らせる。
> 「Bでもできるのでは？」と感じたら、それは読み違いではなく**仕様**。決着は知識ではなく②の優先軸でつける。

## ステップ② 優先軸を探す（決着ルール）

問題文中の「**最も〜**」「**〜を最小限に**」が優先軸。これは同点決勝のルールブックで、**主に4種類しかない**。

| 優先軸 | 問題文での言い回し（言い換え含む） | 寄せる方向 | 即切りできる選択肢 |
|---|---|---|---|
| 💰 **コスト** | 最も費用対効果が高く／最もコスト効率よく／コストを最小限に | S3・スポット・サーバーレス・Gateway型エンドポイント・ライフサイクル | 常時起動・過剰スペック・専用線（要件外なら） |
| 🔧 **運用負荷** | 運用上のオーバーヘッドを最小限に／管理の手間を減らし／運用作業なしで | マネージド・サーバーレス（Lambda / Fargate / Aurora Serverless / S3） | EC2に自前構築・手動運用・スクリプトcron |
| 🏗️ **可用性** | 高可用性／単一障害点をなくす／AZ障害に耐える／ダウンタイムを最小限に | マルチAZ・Auto Scaling・複数NAT GW・マルチリージョン（DR文脈） | 単一AZ・単一インスタンス・1つのNAT GW |
| ⚡ **性能** | 低レイテンシ／ミリ秒単位／高スループット／リアルタイム | ElastiCache・DynamoDB（+DAX）・CloudFront・NLB・Global Accelerator | バッチ処理・ポーリング・Glacier |

> [!tip] 優先軸が2つ書いてある場合
> 「高可用性を保ちつつ、**最もコスト効率よく**」のような場合、**先に書いてある方が満たすべき条件（足切り）**、「最も〜」が付いている方が**決着ルール**。
> → まず高可用性を満たさない選択肢を消し、残った中から最安を選ぶ。

## ステップ③ 要件文に線を引く（チェック項目化）

シナリオの中で、次の形の文だけが「採点対象」。それ以外は背景説明。

| 文の形 | 意味 | 強さ |
|---|---|---|
| 「〜する**必要があります**」 | 必須要件 | ★★★ 満たさない選択肢は即消し |
| 「〜**できてはなりません**」「〜を許可しない」 | 禁止要件 | ★★★ 破る選択肢は即消し（最強の消去ツール） |
| 「〜したいと**考えています**」 | 要望（実質必須） | ★★☆ |
| 「現在〜しています」「〜で稼働しており」 | 現状（変更の起点） | ★☆☆ 「既存を活かす」選択肢のヒント |

**要件の数 ＝ 選択肢をチェックする項目の数**。要件が3つあれば、各選択肢を3項目で○×採点する。
正解は全部○、誤答は**どれか1つだけ×**になるように作られている（→どの要件で×かを言えれば完全に理解できた状態）。

> [!tip] 数字はすべて拾う
> 「**15分**を超える処理」「**RTOは2分**」「**900人**の同時接続」——問題文中の数字は、ほぼ確実にどれかの選択肢を切るために置かれている。
> 例：処理時間15分超 → Lambda（上限15分）が消える。RPO数秒 → 日次バックアップが消える。

## ステップ④ シナリオは「翻訳しながら」流し読み

残った部分（企業紹介・システム構成の説明）は、③の要件を理解するための**文脈**として流し読みする。
ただし、**業種・状況ワードが暗黙の要件**になっていることがあるので、次の対応だけ頭に入れる。

| シナリオのワード | 暗黙に意味する要件 |
|---|---|
| 医療・金融・政府機関・規制 | コンプライアンス → 暗号化・監査証跡（KMS / CloudTrail） |
| グローバル展開・世界中のユーザー | 低レイテンシ配信 → CloudFront / Global Accelerator / Global Tables |
| キャンペーン・セール・急増 | スパイク対応 → Auto Scaling / SQSでバッファ / サーバーレス |
| オンプレミスから移行 | ハイブリッド接続 → DX / VPN / Storage Gateway / DataSync |
| 分析チーム・データサイエンティスト | 分析基盤 → S3 + Athena / Redshift / QuickSight |
| 開発者が少ない・小規模チーム | 運用負荷最小 → マネージド一択 |

## 「2つ選択」問題の定石

複数選択（5択から2つ）は、独立した正解が2つあるのではなく、**2つで1つの構成が完成する**パターンが大半。

- 例（401第1問）：「NAT Gatewayを**作る**」＋「ルートテーブルで**向ける**」→ 作っただけでは通信できない。**部品＋配線**で1セット。
- 例：「S3にデータを置く」＋「ライフサイクルで移行する」→ **置き場＋運用ルール**で1セット。

> [!tip] 選び方のコツ
> 1つ目の正解候補を選んだら、「**これだけで要件は完結するか？足りない部品は何か？**」と自問する。その足りない部品が2つ目の正解。
> 逆に、2つがまったく無関係（例：NAT Gatewayの話とIAMの話）なら、どちらかが誤り。

## 実演：この手順で1問解いてみる

[[401_【問題】SAA模擬試験レベル]] 第1問での思考の流れ：

1. **①締め**：「アクションの組み合わせ（2つ選択）」→ 部品＋配線のセットを探すモード
2. **②優先軸**：「AZ障害にも耐えられる」→ 可用性。**単一AZ構成の選択肢は即消し** → C（単一AZのNAT GW）・E（単一AZのNATインスタンス）消滅
3. **③要件**：(a)外へダウンロード必要 (b)外から直接アクセス禁止 → 「外向きだけ通す装置」＝NAT Gateway。D は egress-only IGW＝**IPv6専用**なのでIPv4要件で×
4. **残り**：A（各AZにNAT GW＝部品）＋ B（ルートテーブル＝配線）→ 2つで1セット完成 ✅

**シナリオの「3層構成のウェブアプリケーション」は一度も使っていない**ことに注目。長文の相当部分は、実は解答に不要な飾り。

## 練習メニュー：「解かない」練習

理解度6割の段階では、正解を当てる練習より**分解する練習**の方が効く。

> [!success] 1日3問・1問5分の分解ドリル
> 1. 問題を**解かずに**、①締め ②優先軸 ③要件（箇条書き）を書き出す
> 2. 問題全体を**自分の言葉で1〜2行に要約**する（上の要約例の形）
> 3. そのあと初めて選択肢を見て、要件との○×表で消去する
> 4. 答え合わせでは「正解したか」ではなく「**分解が解説と一致していたか**」を見る
>
> 分解が合っていて間違えたなら知識不足（→関連ノートへ）。分解がズレていたなら読解の問題（→このノートに戻る）。**原因を切り分けられる**のがこのドリルの最大の効能。
