# Amazon Redshift

> [!info] AWSのフルマネージド **データウェアハウス（DWH）**。ペタバイト級の構造化データを高速にSQL集計・分析するためのサービス。
> 用語が不安なら先に [[1102_データ分析の超基礎・用語集]] を。担当は分析5段階のうち **④分析**（の本格派）。

## 一言でいうと

Redshift は **「分析専用の巨大データベース」** です。
ふだんの業務で使う [[0301_RDS]] / Aurora は、注文・登録など**1件ずつの読み書き（OLTP）**が得意。一方 Redshift は、**何億行をまとめて集計する（OLAP）**のが得意です。役割がまったく違います。

| | RDS / Aurora（OLTP） | Redshift（OLAP / DWH） |
|---|---|---|
| 用途 | 業務システムの登録・更新 | 蓄積データの集計・分析・BI |
| 処理単位 | 1件ずつ | 何億行をまとめて |
| データの持ち方 | 行指向 | **列指向（カラムナ）** |
| 例 | ECサイトの注文処理 | 「過去3年の地域別売上推移」 |

> [!example] たとえ話
> RDS が **「日々の伝票を1枚ずつさばくレジ」** なら、Redshift は **「全店舗の伝票を倉庫にためて、夜間に一気に分析する経営分析室」**。
> 目的が「さばく」か「分析する」かで道具を分けます。

## S3（データレイク）との違い — 「保存できる」は同じでも中身が別物

[[0201_S3]] も Redshift も「データを貯める場所」だが、**できることの差**で区別する。ラベル（データレイク／DWH）の暗記より、この機能差で覚える方が問題で効く。

| | S3（データレイク） | Redshift（データウェアハウス） |
|---|---|---|
| データの持ち方 | **ファイルをそのまま置くだけ**（フォルダにCSVを放り込むイメージ） | **表（列と行）に整理して格納** |
| 分析 | **S3自身は計算できない**（Athena等の道具を別途足す） | **SQLで即座に分析できる**（「種類ごとの平均は?」がその場で聞ける） |
| 一言で | 何でも安く置ける**物置** | 今すぐ分析できる状態で貯まる**倉庫** |

> [!tip] SAAひっかけ
> 問題文が「**データウェアハウスへ蓄積**」「**分析用に整理して貯める**」と言ったら → **Redshift**。「S3に保存して終わり」の選択肢は、**保存はできても"分析できる倉庫"要件を満たさない**ので切れる（実例：[[508_【問題】外部サイト模擬試験8#第8問（Analytics ｜ IoTストリームの順序保持とRedshiftへの配信）]]）。逆に「データレイク」「まず何でも溜める」ならS3。

## なぜ速いのか（SAAで問われる仕組み）

- **列指向（カラムナ）ストレージ**：「売上列だけ合計」のような分析では、必要な列だけ読むので無駄が少ない。
- **MPP（超並列処理）**：複数ノードにデータを分散し、全ノードで同時に計算する。
- **クラスタ構成**：1つの **リーダーノード**（指揮役）＋複数の **コンピュートノード**（実働）で動く。

## Redshift Spectrum（重要）

**Redshift Spectrum = S3 上のデータを、Redshift に取り込まずに直接クエリできる拡張機能。**

通常、分析するにはデータを Redshift に**ロード（コピー）**します。Spectrum を使うと、**ロードせずに S3 のファイルをそのまま読んで**、しかも **Redshift内のテーブルと JOIN** できます。

> [!example] たとえ話
> Redshift という「分析室」は、ふつう室内の棚（クラスタ）の資料しか見られません。
> Spectrum は **「室内の資料と、外部倉庫（[[0201_S3]]）にある資料を、運び込まずに司書がその場で突き合わせてくれる」** 機能です。

### Athena との違い（試験頻出の引っかけ）

|  | [[1109_Amazon Athena]] | Redshift Spectrum |
|---|---|---|
| 前提 | **不要**（サーバーレス、単体で動く） | **Redshiftクラスタが必要** |
| 得意 | S3を手軽に単発SQL分析 | **DWHの中のデータと S3 のデータを JOIN** |
| 課金 | 読んだデータ量 | Redshift料金 ＋ 読んだデータ量 |
| 選ぶ場面 | サーバーを立てずにS3をたまに分析 | すでにRedshiftがあり、S3の過去ログ等も結合したい |

## 知っておきたい機能（SAAレベル）

| 機能 | 何のため |
|---|---|
| **RA3ノード** | コンピュートとストレージを分離。ストレージはS3ベースで自動拡張（Managed Storage） |
| **Redshift Serverless** | クラスタ管理不要で使った分だけ課金。たまに分析する用途に最適 |
| **Concurrency Scaling** | 同時アクセスが増えたとき、一時的に処理能力を自動追加 |
| **スナップショット** | S3へ自動バックアップ。**別リージョンへコピー**して災害対策にできる |
| **Data Sharing** | クラスタをコピーせず、別クラスタとデータを共有 |
| **Enhanced VPC Routing** | S3との通信をVPC内に閉じ込める（セキュリティ要件向け） |

## Redshift ML

**Redshift ML** は、Redshift内のSQLからAmazon SageMakerを裏側で利用し、機械学習モデルの作成・トレーニング・推論を行う機能。データをDWHから外へ大きく動かさず、SQLで予測列を作るような使い方ができる。

> [!example] たとえ話
> Redshiftという経営分析室の中に「統計に強い分析担当（SageMaker）」を呼び、SQLで「来月の売上予測列を作って」と頼めるイメージ。

| 要件 | 答え |
|---|---|
| DWH上のデータをSQLで分析 | Redshift |
| SQLで機械学習モデルを作成/推論 | **Redshift ML** |
| クラスタ管理を避けたい | **Redshift Serverless** |

> [!tip] SAAひっかけ
> 「DWH」「MPP」「列指向」= Redshift。
> 「SQLで機械学習」= Redshift ML。
> 「可能な限りサーバーレス」= Redshift Serverless。
> 3つが同時に出たら **Redshift Serverless + Redshift ML** が強い候補。

> [!warning] 可用性の注意点（SAAの落とし穴）
> Redshift は伝統的に **単一AZ** で動くサービス（マルチAZ対応は後発機能）。
> 「DWHの高可用性・災害対策」を問われたら、まず **スナップショットの別リージョンコピー → 復元** が王道の答え。RDSのような「マルチAZ自動フェイルオーバー」を即答しないこと。

## こんなときに使う（試験頻出シナリオ）

| やりたいこと | 答え |
|---|---|
| ペタバイト級の構造化データをSQLで集計・BI分析したい | **Redshift** |
| 既存のDWH（Redshift）と、S3にある過去ログを**結合**して分析したい | **Redshift Spectrum** |
| サーバーを立てず、S3のデータを**単発で**手軽にSQL分析したい | **Athena**（Redshiftではない） |
| 注文処理など1件ずつの読み書き（トランザクション） | **RDS / Aurora**（Redshiftではない） |
| DWHの災害対策（別リージョンで復旧したい） | **スナップショットのクロスリージョンコピー** |
| たまにしか分析しないのでクラスタを常時持ちたくない | **Redshift Serverless** |
| DWH上のデータからSQLでMLモデルを作りたい | **Redshift ML** |
