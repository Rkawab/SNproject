# Amazon EMR (Elastic MapReduce)

> [!info] 1台では処理しきれない超大量データを、たくさんのコンピューターで手分けして処理する「大厨房」。
> 用語が不安なら先に [[1102_データ分析の超基礎・用語集]] を。担当は分析5段階のうち**③加工**（大規模）。

## 超かんたんに言うと

データが多すぎて、1台のコンピューターでは何時間かけても終わらない――そんな**ペタバイト級（とてつもない量）のデータ**を、**何十台ものコンピューターで手分けして一気に処理する**ための仕組みです。

この「手分けして処理する仕組み」の代表が [[1102_データ分析の超基礎・用語集|Hadoop や Spark]] と呼ばれる技術。EMR は、それらをAWS上で**まとめて借りて動かせる**サービスです。

> [!example] たとえ話
> - [[1107_AWS Glue]] は「お手軽な町のキッチンスタジオ」（少人数の下ごしらえ向け）
> - EMR は「**ホテルの大厨房を、料理人ごとまるごとレンタル**」（大量調理向け）
> 規模が桁違いに大きいときに EMR を使います。

## どんな技術が動かせる？

EMR の上では、大量データを扱う有名な道具たちが動きます（名前だけ知っておけばOK）。

| 道具 | ざっくり何者 |
|---|---|
| **Hadoop / HDFS** | 大量データを分散して保存・処理する元祖の仕組み |
| **Apache Spark** | Hadoopより高速な、今の主流の処理エンジン |
| **Hive** | 大量データに [[1102_データ分析の超基礎・用語集|SQL]] っぽく問い合わせる道具 |
| **Presto / Trino** | 高速にSQLで分散検索する道具 |
| **HBase / Flink** | 特殊用途（NoSQL DB / ストリーミング処理） |

## Glue との一番の違い

| | EMR | [[1107_AWS Glue]] |
|---|---|---|
| イメージ | 大厨房を借りる | お手軽カッター |
| サーバー管理 | **自分で台数・性能を決める**（手間あり） | 不要（おまかせ） |
| 向いている規模 | 超大量・複雑 | 小〜中規模・シンプル |
| コスト最適化 | 激安の[[0102_EC2購入プラン|スポット]]を活用できる | 使った分だけ |

> [!tip] コスト節約のキモ
> EMR は中身が [[0102_EC2購入プラン|EC2（コンピューター）]] の集まりです。一部のコンピューターに「**スポット**」という激安プランを使うと大幅に安くできます。最近は管理不要の **EMR Serverless** 版もあります。

## クラスタ（コンピューター群）の構成

EMR は複数のコンピューターがチームを組んで動きます。役割が3種類あります。

| 役割 | 仕事 |
|---|---|
| マスターノード | チームのリーダー。全体の指揮・割り振り |
| コアノード | データを保存しながら処理する主力 |
| タスクノード | 処理だけ手伝う応援要員（激安スポットを使う候補） |

## こんなときに使う（試験頻出シナリオ）

| やりたいこと | 答え |
|---|---|
| ペタバイト級の超大量データを分散処理したい | **EMR** |
| すでに Hadoop / Spark を使っていて、AWSへ移したい | **EMR** |
| 大規模処理をできるだけ安くやりたい | **EMR**（スポット活用） |
| 小〜中規模で、サーバー管理なしで手軽に | [[1107_AWS Glue]] |
