# データ分析の超基礎・用語集

> [!info] このフォルダ（11_分析）のノートを読む前に、まずここを読むと一気に分かりやすくなります。
> AWSの分析サービスは「データを料理する厨房の道具たち」です。道具を覚える前に、**そもそもデータ分析とは何をやっているのか**を押さえましょう。

---

## そもそも「データ分析」とは何をしているのか

会社やアプリには、毎日たくさんの「記録」がたまります。

- お店なら「いつ・誰が・何を・いくらで買ったか」
- Webサイトなら「誰が・どのページを・何秒見たか」
- 機械なら「センサーが測った温度・振動の値」

こういう生のデータは、ためただけでは**ただの数字の山**です。
そこから「先月いちばん売れた商品は？」「故障の前兆はどんな数値？」といった**役に立つ答え**を取り出す作業が「データ分析」です。

この作業は、だいたい次の5段階に分かれます。**料理に例える**と腹落ちします。

```
① 収集 → ② 蓄積 → ③ 加工 → ④ 分析 → ⑤ 可視化
（食材を   （冷蔵庫に  （下ごしらえ  （調理）   （お皿に
  仕入れる）  しまう）    する）              盛り付ける）
```

| 段階 | 何をする？ | 料理でいうと | 担当するAWSサービス例 |
|---|---|---|---|
| ① 収集 | データを集めてくる | 食材を仕入れる | [[1103_Amazon Kinesis]] / [[1104_Amazon MSK]] |
| ② 蓄積 | 集めたデータをためる | 冷蔵庫にしまう | [[0201_S3]]（保存場所）/ [[1106_AWS Lake Formation]]（管理） |
| ③ 加工 | 使いやすい形に整える | 野菜を切る・下味をつける | [[1107_AWS Glue]] / [[1108_Amazon EMR]] |
| ④ 分析 | 答えを取り出す | 調理する | [[1109_Amazon Athena]] / [[1110_Amazon OpenSearch Service]] |
| ⑤ 可視化 | 人が見て分かる形にする | お皿に盛り付ける | [[1111_Amazon QuickSight]] |

各サービスのノートは「この5段階のどこを担当する道具か」を意識して読むと、頭に入りやすいです。

## 最初に知っておきたい用語（これが分かれば9割解決）

### SQL（エスキューエル）
データを「表（Excelの表のようなもの）」と見立てて、**「条件に合う行を取り出して」と命令する言葉**です。
例：「売上の表から、3月のぶんだけ合計して」を1行の命令文で書けます。
分析の世界では超基本の道具で、多くのサービスがSQLで操作できます。

### ETL（イーティーエル）
**E**xtract（抽出）→ **T**ransform（変換）→ **L**oad（書き出し）の頭文字。
ようは「**バラバラな形のデータを取ってきて、きれいに整え、使いやすい場所に置き直す**」という一連の下ごしらえ作業のこと。
> [!example] たとえ
> 八百屋から泥つき野菜を仕入れ（Extract）、洗って皮をむいて切り（Transform）、調理台に並べる（Load）。この一連が ETL です。代表的な担当が [[1107_AWS Glue]]。

### データレイク（Data Lake）
**形式バラバラのデータを、加工せずそのまま全部ためておく巨大な置き場**のこと。「レイク（湖）」という名のとおり、何でも流し込んでためる大きな器です。
AWSでは中身の実体は [[0201_S3]] に置きます。「とりあえず全部ためて、後で必要なものだけ取り出して使う」という考え方。
- 似た言葉「データウェアハウス（DWH）」は、**最初からきれいに整理してためる倉庫**。レイクが「ごちゃ混ぜの湖」、DWHが「整理された倉庫」と覚えるとよいです。

### バッチ処理 と ストリーミング（リアルタイム）処理
データの「処理のタイミング」の違いです。

| | バッチ処理 | ストリーミング処理 |
|---|---|---|
| やり方 | データをためてから**まとめて**処理 | 届いたそばから**1件ずつ即座に**処理 |
| たとえ | 1日分の洗濯物を夜にまとめて洗う | 汚れたら都度すぐ手洗い |
| 用途 | 月次レポート、夜間集計 | 不正検知、リアルタイム監視 |
| 担当例 | [[1107_AWS Glue]] / [[1108_Amazon EMR]] | [[1103_Amazon Kinesis]] / [[1104_Amazon MSK]] |

### サーバーレス（Serverless）
「サーバーが無い」という意味ではなく、**自分でサーバー（コンピューター）を用意・管理しなくてよい**という意味です。
普通はデータを処理するのに「何台のコンピューターを、どのくらいの性能で借りるか」を自分で決めて管理する必要があります。サーバーレスなら**AWSが裏で勝手に用意してくれて、使った分だけ課金**。電源やメンテの心配が要りません。
> [!tip] 試験のヒント
> 「運用の手間をかけたくない」「管理不要で」と来たら、サーバーレス系（[[1109_Amazon Athena]] や [[1107_AWS Glue]]）が答えになりやすいです。

### BI（ビーアイ / ビジネスインテリジェンス）
分析結果を**グラフや表のダッシュボードにまとめて、経営判断に使う**こと。担当は [[1111_Amazon QuickSight]]。
「数字の答え」を「人が一目で分かる絵」にする最後の仕上げ役です。

### 全文検索（ぜんぶんけんさく）
大量の文章やログの中から、**キーワードを含むものを一瞬で探し出す**技術。Googleで検索する、あの体験のこと。担当は [[1110_Amazon OpenSearch Service]]。

## やや専門的な用語（個別ノートで出てきたら戻ってくる用）

### Hadoop（ハドゥープ）/ Spark（スパーク）
**1台では処理しきれない超大量のデータを、たくさんのコンピューターで手分けして処理する仕組み**。
- **Hadoop**：その元祖となる仕組みの総称。
- **Spark**：Hadoopより新しくて高速な処理エンジン。今の主流。
AWSでこれらを動かす担当が [[1108_Amazon EMR]]。「ペタバイト級」「分散処理」と来たらこのあたり。

### Apache Kafka（カフカ）
**大量のデータを次々と受け流すための、世界標準のオープンソース仕組み**（ストリーミングの土管）。
AWSが代わりに運用してくれるのが [[1104_Amazon MSK]]。AWS独自の似た役割が [[1103_Amazon Kinesis]]。

### Parquet（パーケイ）/ ORC
データファイルの**保存形式**の一種。CSV（普通の表形式）に対して、Parquetは「**列ごとにまとめて圧縮して保存**」する形式。
分析では「3月の売上だけ」のように一部の列だけ読むことが多く、列ごとにまとまっていると**読む量が減って速く・安くなる**。[[1109_Amazon Athena]] のコスト節約で重要。

### メタデータ / データカタログ
**「データそのもの」ではなく「データの説明書」**のこと。
「このファイルは売上表で、1列目が日付、2列目が金額…」という情報＝メタデータ。それを一覧管理する図書館の目録が「データカタログ」。担当は [[1107_AWS Glue]] Data Catalog。
