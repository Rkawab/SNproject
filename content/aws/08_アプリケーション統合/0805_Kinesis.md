# Kinesis（リアルタイムストリーミング）

> [!info] 概要
> 大量の時系列データを**リアルタイム**で収集・処理・分析するサービス群。ログ・クリックストリーム・IoTセンサー・金融取引データ・動画など「**流れ続けるデータ**」を扱う。

## たとえ話

**大河の流れ**。
- データが川のように途切れず流れてくる
- 川岸（コンシューマー）に何人も立って、それぞれが自分の目的でデータを観察・処理できる
- 川を流れる水を**一定期間は遡って**取れる（保持期間）
- 川幅（シャード数）を広げれば、流せる水量も増える

|------|------|------|

## ① Kinesis Data Streams（KDS）

### コアの仕組み

```
[Producer]──PutRecord──>[Stream]──GetRecords──>[Consumer A]
[Producer]              (シャード)              [Consumer B] ← 同じデータを別Consumerが読める
[Producer]                                      [Consumer C]
```

- **シャード**: スループットの単位。Streamは複数のシャードで構成される
- **データレコード**: 最大1MB。**Partition Key** でシャードに振り分けられる
- **シーケンス番号**: シャード内で順序を示すID
- **複数の消費者**が同じデータを独立に読める（これがSQSと最大の違い）

### シャードの計算（試験頻出！）

| シャード単位の性能 | 書き込み | 読み出し |
|------------------|---------|---------|
| 1シャード | **1 MB/秒** または **1000レコード/秒** | **2 MB/秒** または **5回/秒** |

> [!example] 計算例
> 「毎秒5MBのデータを書き込み、3つのConsumerが同時に読む」
> - 書き込み: 5MB/秒 → **5シャード必要**
> - 読み出し: 各Consumerが5MB/秒で読む → 5シャードでは足りない（2MB×5=10MB/秒で3 Consumer分の15MBに足りない）
> - 解決策: **Enhanced Fan-Out** を使う（各Consumerに専用2MB/秒を提供）

### キャパシティモード

| モード | 説明 |
|--------|------|
| **Provisioned** | シャード数を手動指定。安定したトラフィックなら安い |
| **On-Demand** | 自動スケール。予測不能なトラフィック向け（4MB/秒・4000レコード/秒まで自動） |

### Enhanced Fan-Out

通常: 全Consumerでシャードの読み出し帯域（2MB/秒）を共有
拡張: **各Consumerに専用の2MB/秒**を割り当て＆Push型で低レイテンシ（〜70ms）

### データ保持期間

- デフォルト: **24時間**
- 最大: **365日**（長期保持は追加料金）
- 期間内であれば**過去のデータを何度でも再生**できる

### Producer / Consumer のライブラリ

| ライブラリ | 用途 |
|-----------|------|
| **KPL**（Kinesis Producer Library） | 高効率な書き込み・自動バッチ・再試行 |
| **KCL**（Kinesis Client Library） | DynamoDB を使ったシャード割り当て・チェックポイント管理 |

## ② Amazon Data Firehose（旧 Kinesis Data Firehose）

### 特徴

- **完全マネージド**で配信先に流し込む
- **コード不要**（KDSは自分でConsumer書く必要があるがFirehoseは設定だけ）
- 内部でバッファリング → まとめて配信
- Lambda で途中変換も可能

### 配信先

| 配信先 | 用途 |
|--------|------|
| **S3** | データレイク・長期保存（最頻出） |
| **Redshift** | 分析DWH。実体はS3経由でCOPY |
| **OpenSearch** | ログ検索・可視化 |
| **Splunk** | 既存Splunk基盤連携 |
| **HTTP Endpoint** | サードパーティSaaS（Datadog等） |

### バッファリング

| 単位 | 設定範囲 |
|------|---------|
| **サイズ** | 1〜128 MB |
| **時間** | 60〜900秒 |

どちらか先に達した方で配信される。**ニアリアルタイム**（数十秒〜数分の遅延あり）。

### KDS vs Firehose

| 観点 | Kinesis Data Streams | Amazon Data Firehose |
|------|---------------------|---------------------|
| 配信先 | 自分で書く（カスタム処理可） | S3 / Redshift / OpenSearch 等の決まったサービス |
| レイテンシ | 〜200ms（拡張ファンアウトで70ms） | 60秒以上 |
| データ保持 | 1〜365日 | **保持しない**（即配信） |
| 再生 | 可能 | 不可 |
| シャード管理 | 必要 | **不要**（自動スケール） |
| 用途 | リアルタイム分析・複数Consumer | 単純なデータ蓄積 |

## ③ Managed Service for Apache Flink（旧 Kinesis Data Analytics）

### 特徴

ストリームに対して **SQL** や **Apache Flink** でリアルタイム集計・分析する。

```
[Kinesis Data Streams]
       ↓
[Managed Service for Apache Flink]
   - 直近5分の平均値を計算
   - 異常値を検知
   - データを変換・結合
       ↓
[Kinesis Data Streams / Firehose / Lambda]
```

### よく使うウィンドウ計算

| ウィンドウ | 意味 |
|----------|------|
| **タンブリング** | 固定の時間枠（例: 0:00-0:05, 0:05-0:10...） |
| **スライディング** | ずれた重なりあう枠（例: 直近5分を毎分計算） |
| **セッション** | アクティブな期間を1セッションとする |

## ④ Kinesis Video Streams（KVS）

> [!info] 概要
> カメラ・IoTデバイスからの**映像ストリームをリアルタイムでAWSに取り込み**、保存・再生・機械学習処理を行うサービス。

### たとえ話

**映像専用の高速道路**。
監視カメラの映像をリアルタイムでクラウドに送る場合、ファイルアップロードでは遅延が大きい。KVS は映像専用の高速道路として、途切れることなくAWSに映像を流し込み、そのまま処理・保存・分析できる。

### 主なユースケース

| ユースケース | 詳細 |
|---|---|
| **監視カメラ映像の保存・分析** | 店舗・施設の監視カメラをクラウドで一元管理 |
| **IoTデバイスからの映像収集** | スマートドアベル・ドライブレコーダー・ドローン等 |
| **Rekognitionと連携した顔認識** | 顔検出・物体検知・感情分析をリアルタイムで実行 |
| **機械学習パイプライン** | SageMaker と連携してカスタムモデルで映像分析 |

### 技術的な特徴

```
[カメラ / IoTデバイス]
       ↓  KVS SDK（Producer）
[Kinesis Video Streams]
       ↓
  ├─ [再生] HLS / DASH / WebRTC
  ├─ [Rekognition Video] 顔認識・物体検知
  ├─ [SageMaker] カスタム機械学習モデル
  └─ [S3] 長期アーカイブ
```

| 機能 | 内容 |
|------|------|
| **再生形式** | HLS / DASH（VOD再生）・WebRTC（リアルタイム双方向） |
| **データ保持** | 数時間〜年単位で設定可能 |
| **プロトコル** | RTSP / WebRTC / HLS |
| **SDK** | Producer SDK（デバイス側）/ Parser SDK（分析側） |

### Elastic Transcoder / MediaConvert との違い

| | Kinesis Video Streams | Elastic Transcoder / MediaConvert |
|---|---|---|
| 主な目的 | **リアルタイム映像の取り込み・処理** | **動画ファイルのフォーマット変換** |
| データの流れ | デバイス → AWS（ストリーミング） | S3 → 変換 → S3（バッチ） |
| リアルタイム性 | あり | なし（VOD向け） |
| 試験キーワード | 「監視カメラ」「IoT映像」「Rekognitionと連携」 | 「MP4に変換」「複数デバイス向け配信」 |

## Kinesis vs SQS（試験超頻出）

| 観点 | Kinesis Data Streams | [[0802_SQS]] |
|------|---------------------|---------|
| データモデル | **ストリーム**（連続データ） | キュー（個別メッセージ） |
| 順序保証 | シャード内で**保証** | StandardはNo / FIFOはYes |
| 複数Consumer | **同じデータを複数Consumerが読める** | 1メッセージは1Consumer |
| データ再生 | **可能**（保持期間内） | 削除したら消える |
| メッセージサイズ | 最大1MB | 最大256KB |
| スケール | シャード数 | ほぼ無制限 |
| 削除 | 不要（時間経過で消える） | 明示的に削除必要 |
| 用途 | リアルタイム分析・ログ集約 | タスク処理・非同期ジョブ |

> [!tip] 見分け方の決め手
> - **「同じデータを複数の処理が独立に消費する」** → Kinesis
> - **「1つのタスクを1人のワーカーが処理して完了したら消す」** → SQS
> - **「過去のデータを遡って再処理したい」** → Kinesis（SQSは不可）

---

## Kinesis vs MSK（Managed Kafka）

| 観点 | Kinesis Data Streams | Amazon MSK |
|------|---------------------|-----------|
| ベース | AWS独自 | Apache Kafka |
| 移行 | AWSネイティブのみ | Kafkaエコシステムをそのまま使える |
| 運用負荷 | 低 | やや高い |
| 用途 | AWS中心のリアルタイム | Kafkaに慣れた組織 |

## SAA頻出パターン

| 問題文のキーワード | 答え |
|---------------------|------|
| 「**リアルタイム**」「**ストリーミング**」「**複数の消費者**」 | KDS |
| 「**過去のデータを再生**」「データを**遡って再処理**」 | KDS |
| 「**S3 / Redshift / OpenSearch にデータを流す**」「**コード不要**」 | **Data Firehose** |
| 「ストリームへの**SQLクエリ**・**異常検知**・移動平均」 | Managed Service for Apache Flink |
| 「**IoTセンサー**データ取り込み」 | KDS or IoT Core |
| 「**動画ストリーム**取り込み」 | Kinesis Video Streams |
| 「**シャード数を計算**」 | スループット要件から計算（1MB/秒・1000レコード/秒）|
| 「**専用帯域**」「**低レイテンシ**で多Consumer」 | Enhanced Fan-Out |
