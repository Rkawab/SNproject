# SES（Simple Email Service）

> [!info] 概要
> **メール送受信に特化**したフルマネージドサービス。マーケティング・トランザクション・運用通知メールを、自前でメールサーバーを立てずに大量・高信頼に送れる。受信メールを S3 / Lambda / SNS にルーティングすることも可能。

## たとえ話

**法人向け大量発送に対応した郵便局**。
- 1日数百万通でも安定して捌ける配送能力
- 「届いた／開封された／クリックされた」の追跡レポート発行
- 「宛先不明」「クレーム」の戻りもちゃんと処理してくれる
- 「うちの郵便局から出したものですよ」の認証印（DKIM/SPF）を付けてくれる

---

## なぜ自前のメールサーバーじゃダメか？

自分でPostfix等のメールサーバーを建てると以下の苦労がある:

| 苦労 | SESなら |
|------|--------|
| IPアドレスの**送信レピュテーション**を一から育てる必要 | AWSの良質なIPプールから送信 |
| DKIM / SPF / DMARC の自前設定 | 設定支援＆自動署名 |
| バウンス（届かない）・苦情（迷惑メール報告）の処理 | 自動でフィードバック・対応 |
| ブラックリスト入りリスク | AWSが監視 |
| スケールの設計 | 自動スケール |

---

## サンドボックス vs 本番

> [!warning] 初回アカウントはサンドボックスモード
> 新規アカウントは**サンドボックス**に置かれ、検証済みアドレス間でしか送信できない。AWSサポートに**本番アクセス申請**を出して解放してもらう。

| モード | 制限 |
|--------|------|
| **サンドボックス** | 1日200通まで・1秒1通まで・検証済み宛先のみ |
| **本番アクセス** | 申請後、上限解除（最初は1日5万通程度から徐々に拡大） |

送信上限（**送信クォータ**）はアカウントのレピュテーションに応じて自動で引き上げられる。

## 送信時の認証 ― ドメイン認証3点セット

ちゃんと届くメールを送るには以下3つが必須レベル:

| 認証 | 役割 |
|------|------|
| **SPF**（Sender Policy Framework） | 「このIPからの送信はうちのドメインの正規メールです」をDNSで宣言 |
| **DKIM**（DomainKeys Identified Mail） | メールに電子署名を付けて改ざんを防ぐ |
| **DMARC** | SPF / DKIM が失敗したときの**振る舞いポリシー**を宣言（拒否・隔離・通知） |

> [!tip] SES がやってくれること
> - DKIM 署名は SES が自動で付与（ドメイン設定時に発行されるCNAMEレコードを登録するだけ）
> - 「Easy DKIM」ボタンひとつでセットアップ完了

## 主要機能

### 1. メール送信

- **SMTPインターフェース**（既存のメールライブラリでそのまま使える）
- **SES API**（AWS SDK経由、推奨）

### 2. 設定セット（Configuration Set）

送信ごとに**メトリクス・イベントを記録**するためのプロファイル。

```
配信成功 / バウンス / 苦情 / 開封 / クリック / 拒否
  ↓ Event Publishing
[CloudWatch / SNS / Kinesis Firehose / Pinpoint]
```

これで「マーケティングメールの開封率」「特定キャンペーンのバウンス率」を自動計測できる。

### 3. 受信ルール（Receive Rules）

メールを「受信」する側にもなれる。

```
[user@your-domain.com にメール到着]
        ↓
   SES 受信ルール
   ├→ S3 にメール全文を保存
   ├→ Lambda 関数を起動
   ├→ SNS で通知
   └→ バウンス応答 / 別アドレスに転送
```

→ 「お問い合わせメールをLambdaで自動処理してDynamoDBに保存」のような構成が組める。

### 4. レピュテーション・ダッシュボード

自分のアカウントの**送信レピュテーション**（バウンス率・苦情率）をリアルタイム表示。閾値（バウンス5%、苦情0.1%）を超えるとAWSから警告。

### 5. カスタム MAIL FROM ドメイン

エンベロープ送信元（Return-Path）を独自ドメインのサブドメインにできる。SPF認証で有利。

## バウンスと苦情の処理

| 種類 | 意味 | 処理 |
|------|------|------|
| **ハードバウンス** | 永続的に届かない（存在しないアドレス等） | **すぐにリストから削除** |
| **ソフトバウンス** | 一時的（メールボックス満杯等） | 一定期間後に再送可 |
| **苦情（Complaint）** | 受信者が「迷惑メール」と報告 | **絶対に再送しない**（レピュテーション低下） |

これらの通知を SNS で受け取って自動的に送信先リストから除外する仕組みを作るのが定番。

## SNS との違い

| 観点 | SES | [[0803_SNS]] のメール配信 |
|------|-----|------------------|
| 目的 | **メール専用**で高機能 | 通知の1配信方法としてのメール |
| 大量送信 | ◎ | △ |
| HTMLメール・添付ファイル | ◎ | × |
| バウンス処理 | ◎ | × |
| 受信機能 | ◎ | × |
| カスタムドメイン | ◎ | × |

> 「ユーザー向けのちゃんとしたメール」→ **SES**
> 「運用者へのアラート通知」→ **SNS**

## 構成例

### ユーザー登録時の確認メール

```
[Webアプリ] → SES.SendEmail → ユーザーのGmail
                ↓ 設定セットでイベント記録
            [CloudWatch メトリクス]
```

### お問い合わせメール自動処理

```
[user@contact.example.com 受信]
       ↓ SES 受信ルール
       ├→ S3 にメール本文保存
       └→ Lambda 起動
            ├→ DynamoDB に問い合わせ記録
            └→ Slack 通知
```

## SAA頻出パターン

| 問題文のキーワード | 答え |
|---------------------|------|
| 「**大量のEメール**送信」 | SES |
| 「**バウンス・苦情**を自動処理」 | SES + SNS + Lambda |
| 「メール**受信**を自動化」 | SES 受信ルール → S3 / Lambda |
| 「**DKIM/SPF** によるなりすまし防止」 | SES |
| 「送信レピュテーション・配信スコア」 | SES |
| 「**簡易な障害通知をメールで**」 | [[0803_SNS]]（SESではない） |
