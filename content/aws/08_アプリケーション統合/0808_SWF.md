# SWF（Simple Workflow Service）

> [!warning] 旧世代サービス
> 現在は **AWS Step Functions** が後継として推奨。新規システムでは Step Functions を選ぶのが基本で、AWSの公式ドキュメントも「新規はStep Functions」と明記している。SAAでも基本は Step Functions が答え。

## 概要

複数のステップ・複数のサーバー・場合によっては**人間の手作業**を含む**長時間ワークフロー**を管理するサービス。各タスクの状態（実行中・完了・失敗）を追跡し、進行を制御する。

## たとえ話

**工場の製造ライン管理システム**。
「部品調達 → 組み立て → 検品 → 梱包 → 出荷」という工程で、どこまで終わったか・どの工程で誰が担当しているか・止まっているかを追跡し、次の工程に指示を出す。組み立てが人間の作業でも、検品が自動化機械でも、一元管理する。

---

## SWF の構成要素

| 要素 | 役割 |
|------|------|
| **ドメイン** | ワークフローの分離単位（namespace） |
| **ワークフロー** | 全体のオーケストレーション定義 |
| **Workflow Starter** | ワークフローを開始するアプリケーション |
| **Decider** | 「次に何をするか」を判断するロジック（自分で実装） |
| **Activity Worker** | 実際にタスクを実行する側（EC2 / オンプレ / 人間） |
| **タスクリスト** | Worker への作業指示が積まれるキュー |

### 動きのイメージ

```
[Workflow Starter] → SWFにワークフロー開始要求
        ↓
   [SWF]
        ↓ 履歴イベントを記録
   [Decider]（自分で実装）が履歴を見て次のタスクを決定
        ↓
   [Activity Task] をタスクリストに積む
        ↓
   [Activity Worker] が取り出して実行（人間でもOK）
        ↓ 完了報告
   [SWF] が履歴に記録 → Decider が次のステップを決定 → ...
```

## SWF vs Step Functions

| 観点 | SWF | Step Functions |
|------|-----|----------------|
| **定義方法** | コードで Decider を実装する | JSON（**Amazon States Language**）で宣言的に定義 |
| **可視化** | なし | コンソールで**フロー図表示** |
| **運用負荷** | Decider 用のサーバーが必要 | **完全サーバーレス** |
| **実行時間** | 最大 **1年** | Standard: 1年 / Express: 5分 |
| **コスト** | やや高い | 安い |
| **AWSとの統合** | 限定的 | Lambda / ECS / SNS / SQS / DynamoDB 等を直接呼べる |
| **人間の手作業** | 〇（最大1年） | 〇（Wait For Task Token パターン） |
| **学習コスト** | 高い | 比較的低い |

> [!tip] 現状の選び方
> - 新規 → **Step Functions** 一択
> - 既存SWFの維持 → SWFのまま運用
> - 既存SWFの新規開発分 → Step Functions に切り替えを検討

## いつ SWF が答えになる？

SAAでも基本は Step Functions が正解だが、以下のような問題文では SWF も選択肢になる:

- 「人間の介入を含む**長期間（数日〜数ヶ月）** のワークフロー」
- 「外部システム連携が多く、複雑な状態管理が必要」
- 「既存のSWFベース資産を活用したい」

ただし大抵の場合、Step Functions の **Wait For Task Token** パターンで同じことが実現できる。

---

## Step Functions（後継）の超概要

> 詳細は別ノートで扱うが、SWFの代替として知っておくべきこと。

- **Standard Workflow**: 最大1年・厳密1回実行・監査向け
- **Express Workflow**: 5分以内・高スループット・ストリーミング用
- 各タスクは Lambda / ECS / Batch / SNS / SQS / DynamoDB / API Gateway 等を直接呼べる
- 失敗時の **Retry / Catch** をJSONで定義
- **Parallel / Map / Choice** 等の状態遷移を組み合わせ

## SAA頻出パターン

| 問題文のキーワード | 答え |
|---------------------|------|
| 「**複数Lambda**のオーケストレーション」 | **Step Functions** |
| 「処理フローを**可視化**したい」 | **Step Functions** |
| 「複雑なリトライ・エラーハンドリング」 | **Step Functions** |
| 「**人間の承認**を含むワークフロー」 | Step Functions（Wait For Task Token）or SWF |
| 「既存のSWF構成を活用」 | SWF |
