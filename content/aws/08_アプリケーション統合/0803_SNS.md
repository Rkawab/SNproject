# SNS（Simple Notification Service）

> [!info] 概要
> **Pub/Sub（発行/購読）** モデルのフルマネージド通知サービス。1つのメッセージを**複数の購読者（Subscriber）に同時に**配信する。配信先は SQS / Lambda / HTTP(S) / Email / SMS / モバイルプッシュ通知 / Kinesis Data Firehose と多彩。

## たとえ話

**マンションの一斉放送**。
- 管理人（Publisher）が放送室（トピック）からアナウンスを1回流す
- 各部屋に設置されたスピーカー（Subscriber）から同時に音声が流れる
- スピーカーごとに「メール式・SMS式・電話式」など出力方法は違う
- 放送をやめれば全スピーカーに届かなくなる

---

## コアの仕組み

```
[Publisher（発行者）]
       ↓ Publish
   [SNS Topic]
       ↓ 自動的に全Subscriberに配信
  ┌────┬─────┬──────┬──────┬────────┐
  ↓    ↓     ↓      ↓      ↓        ↓
 SQS  Lambda Email  SMS  HTTP/S Mobile Push
```

- **Topic**: 配信チャネル（任意の名前）
- **Subscription**: Topic に対する購読登録
- **Endpoint**: 実際の配信先（Lambda ARN、メールアドレス等）
- **Protocol**: 配信方法（lambda / sqs / email / sms / http(s) / application）

---

## トピックの種類

| 種別 | 特徴 | ユースケース |
|------|------|------------|
| **Standard Topic** | 高スループット・順序保証なし・少量重複あり | 一般的な通知・ファンアウト |
| **FIFO Topic** | 順序保証・重複排除・**FIFO SQS のみ購読可能** | 金融イベント・在庫変動 |

> [!warning] FIFO Topic の制約
> FIFO Topic は**FIFO SQS にしか配信できない**。Lambda や Email には配信できない点に注意。

## 配信先プロトコル一覧

| Protocol | 配信先 | 備考 |
|----------|--------|------|
| **SQS** | SQSキュー | 最頻出の組み合わせ |
| **Lambda** | Lambda関数 | サーバーレス処理 |
| **HTTP / HTTPS** | 任意のWebhook URL | 外部サービス連携 |
| **Email / Email-JSON** | メールアドレス | 運用通知用 |
| **SMS** | 電話番号 | 障害アラート等 |
| **Application** | モバイルプッシュ（APNs / FCM / ADM 等） | アプリ通知 |
| **Kinesis Data Firehose** | Firehose ストリーム | データ蓄積・分析へ流す |

## Fan-out パターン（SAA超頻出）

1つのイベントを複数のサービスで並列処理する定番構成。

```
   [注文確定イベント]
         ↓ Publish
    [SNS Topic]
   ├→ [SQS A] → 在庫処理ワーカー
   ├→ [SQS B] → 配送手配ワーカー
   ├→ [SQS C] → 分析データ蓄積
   └→ [Lambda] → メール送信
```

### なぜ SQS を挟む？

| 構成 | 耐障害性 |
|------|---------|
| SNS → Lambda 直結 | Lambda 障害時にメッセージ消失 |
| SNS → SQS → Lambda | **SQSが緩衝**。再処理可能・DLQも使える |

「**メッセージの耐久性**」と問題文にあれば SNS+SQS の Fan-out が正解になりやすい。

## メッセージフィルタリング（Subscription Filter Policy）

Subscriberごとに「特定の属性のメッセージだけ受け取る」絞り込みが可能。

### 例: 価格帯で振り分け

```json
// Subscription A のフィルタ
{ "price": [{ "numeric": [">=", 1000] }] }

// Subscription B のフィルタ
{ "price": [{ "numeric": ["<", 1000] }] }
```

これにより、Publisher は1つのTopicに送るだけで、価格帯ごとに違うキューに振り分けられる。**トピック数を増やさずに済む**のがメリット。

## SQS との違い

| 観点 | SNS | [[0802_SQS]] |
|------|-----|---------|
| モデル | **Push型**（即時配信） | **Pull型**（消費者が取りに行く） |
| 配信先数 | 1メッセージ → **N人**に同時配信 | 1メッセージ → 1Consumer |
| メッセージ保持 | **保持しない**（届けたら終わり） | 最大14日保持 |
| 再処理 | 不可（届かなければ消失） | 可（削除しない限り再表示） |
| 用途 | 通知・イベント扇形配信 | タスクキューイング |

## モバイルプッシュ通知

各OS/プラットフォーム向けのプッシュ通知をSNSが一手に引き受ける。

| プラットフォーム | サービス |
|----------------|---------|
| iOS | **APNs**（Apple Push Notification service） |
| Android | **FCM**（Firebase Cloud Messaging） |
| Kindle | **ADM**（Amazon Device Messaging） |
| Windows | **WNS**, **MPNS** |
| Baidu (China) | **Baidu Cloud Push** |

→ Publisher は SNS にメッセージを送るだけで、各端末プラットフォームに自動配信。

---

## クロスアカウント / クロスリージョン

- **クロスアカウント**: Topic のリソースポリシーで他アカウントの Publish / Subscribe を許可
- **クロスリージョン**: SNS Topic は同一リージョン内。別リージョンへは HTTPS 経由 or **SNS の Cross-Region Subscription** を使う

## SAA頻出パターン

| 問題文のキーワード | 答え |
|---------------------|------|
| 「1つのメッセージを**複数のサービスに同時配信**」 | SNS |
| 「**複数のキュー**にメッセージを分散」 | SNS + 複数SQS（Fan-out） |
| 「**耐久性**も欲しい複数配信」 | SNS + SQS Fan-out |
| 「**SMS / モバイルプッシュ**」 | SNS |
| 「[[1001_CloudWatch]] アラームの通知先」 | SNS |
| 「**条件ごとに別の処理に振り分け**」 | SNS Subscription Filter |
| 「順序保証された一斉配信」 | SNS **FIFO Topic** + SQS FIFO |
