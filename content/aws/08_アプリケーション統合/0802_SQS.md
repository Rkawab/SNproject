# SQS（Simple Queue Service）

> [!info] 概要
> AWS最古参のサービスのひとつ（2006年提供開始）。フルマネージドな**メッセージキューイング**サービスで、送信側（Producer）と受信側（Consumer）の間に「行列」を挟むことで両者を **疎結合（loosely coupled）** にする。
> サーバー管理不要・自動スケール・高可用性（複数AZ自動複製）。

## なぜ必要か？ ― 疎結合の威力

### 結合が密なシステムの問題

```
[Webサーバー] → 直接呼び出し → [画像処理サーバー]
```

- 画像処理サーバーが落ちたら、Webサーバーまでエラーになる
- 画像処理に時間がかかるとWebレスポンスも遅くなる
- アクセスが急増したら処理サーバーがパンクする

### キューを挟むと

```
[Webサーバー] → メッセージ投入 → [SQS] ← メッセージ取得 ← [画像処理ワーカー × N台]
```

- 処理ワーカーが落ちてもメッセージはキューに残る（最大14日）
- Webは「投入したら即レスポンス」で速い
- アクセス急増時もキューが緩衝材になり、ワーカーは自分のペースで処理
- ワーカー台数を Auto Scaling でキュー長に応じて増減できる

---

## たとえ話

**コンビニのレジ行列**。
- 客（メッセージ）が一気に来店してもレジ前の行列（キュー）が吸収する
- 店員（コンシューマー）は前から順に処理
- 店員が増えれば（スケールアウト）行列の捌けるスピードも上がる
- 店員が休憩中（障害）でも、客は行列に並んで待っていられる

## キューの種類

| 観点 | Standard | FIFO |
|------|----------|------|
| **スループット** | ほぼ無制限 | 300 TPS（バッチで3000 TPS） |
| **順序保証** | ベストエフォート（保証なし） | **厳密に保証** |
| **配信回数** | At-Least-Once（重複あり） | **Exactly-Once**（重複排除） |
| **キュー名** | 任意 | 末尾に `.fifo` 必須 |
| **料金** | 安い | やや高い |
| **ユースケース** | ログ収集、画像処理、注文受付（順序不問） | 金融取引、在庫、注文順序保証 |

> [!tip] FIFO で重複排除される仕組み
> 5分間のウィンドウ内で**同じ MessageDeduplicationId** のメッセージは1回しか配信されない。明示的にIDを付けるか、本文ベースのハッシュで自動生成。

## メッセージのライフサイクル（最重要）

```
[Producer] --SendMessage--> [SQS キュー]
                                ↓ Consumer が ReceiveMessage
                          [可視性タイムアウト中]
                          （他のConsumerには見えない）
                                ↓ 処理完了
                          [DeleteMessage で削除] ← これを忘れると再配信される
```

### 各段階のポイント

1. **SendMessage**: Producer がメッセージを投入。投入時に **遅延（Delay）** を最大15分まで指定可能
2. **ReceiveMessage**: Consumer が取得。同時に**可視性タイムアウト**が開始
3. **可視性タイムアウト中**: 他のConsumerからは「見えない」状態（二重処理防止）
4. **DeleteMessage**: 処理完了したら **明示的に削除** する必要がある
5. **タイムアウト超過**: 削除されないとメッセージは再びキューに表示される → 再配信

> [!warning] 削除を忘れると地獄
> 処理が遅くて可視性タイムアウトを超えると、別のワーカーが**同じメッセージを再処理**する。冪等性（同じ処理を何度やっても結果が同じ）を確保するか、長時間処理には `ChangeMessageVisibility` で延長する。

## ポーリング方式の違い

| 方式 | 動作 | コスト | 用途 |
|------|------|--------|------|
| **ショートポーリング** | メッセージがなくても即レスポンス | API呼び出しが増えて**高コスト** | リアルタイム性最優先 |
| **ロングポーリング** | メッセージが来るまで最大20秒待機 | **低コスト**・低レイテンシ | 通常はこちらを使う |

> [!tip] 試験でよく出る
> 「コスト最適化のために設定すべきは？」→ **ロングポーリング** が答え。`ReceiveMessageWaitTimeSeconds = 20` を設定。

## デッドレターキュー（DLQ）

何度処理しても失敗するメッセージを別キューに**退避**する仕組み。

```
通常キュー（maxReceiveCount = 5）
   ↓ 5回失敗
DLQ（デッドレターキュー）
   ↓ 開発者が後で原因調査
```

- 「毒メッセージ」がワーカーを延々と苦しめるのを防ぐ
- DLQ への移動回数（**maxReceiveCount**）を設定
- DLQ もただのSQSキュー（Standardなら StandardのDLQ、FIFOなら FIFOのDLQ）

## その他の重要機能

### 遅延キュー（Delay Queue）

メッセージ投入後、指定時間（最大15分）は Consumer から見えない。
**ユースケース**: 注文後5分は変更可能にしたい → 5分遅延してワーカーに渡す

### バッチ操作

`SendMessageBatch` / `DeleteMessageBatch` で**最大10件まとめて**処理。API呼び出し数を1/10にできてコスト削減。

### メッセージサイズ

- 最大 **256KB**
- それ以上は **Extended Client Library** で S3 と組み合わせて大きなペイロードを扱う

## SQS の限界

| 項目 | 制限 |
|------|------|
| メッセージ最大サイズ | 256 KB |
| 保持期間 | 1分 〜 **14日**（デフォルト4日） |
| 可視性タイムアウト | 0秒 〜 12時間（デフォルト30秒） |
| 遅延 | 0秒 〜 15分 |
| FIFOスループット | 300 TPS（バッチで3000、ハイスループットモードで更に上） |

| 「処理を**疎結合**にしたい」 | SQS |
