# ネットワーク防御（壁・門で防ぐ）

> [!info] このノートの範囲
> Firewall Manager / Network Firewall / Shield / WAF。
> 攻撃を「壁・門」で物理的に防ぐ層。

> [!tip] 最重要の対比
> **Shield＝量で潰す攻撃（DDoS, L3/4）／WAF＝中身が不正なリクエスト（L7）**。

---

## AWS WAF（Web Application Firewall）

**Webアプリ層（L7）の攻撃を防ぐフィルター。**

> [!example] たとえ
> 受付で「怪しい申込書（SQLインジェクション・XSS等の不正リクエスト）」をはじく審査員。

- **CloudFront / ALB / API Gateway** に紐づけ、IP・地域・レート制限・SQLi/XSSルールで防御。

> [!tip] SAAポイント
> WAFは **ALB / CloudFront / API Gateway** に付ける（EC2に直接ではない）、が頻出。

---

## AWS Shield

**DDoS攻撃（大量アクセスでパンクさせる攻撃）専門の盾。**

> [!example] たとえ
> 大群が一斉に押し寄せても建物が潰れないようにする防御壁。

| 種類 | 内容 |
|---|---|
| **Standard** | 全員に自動・無料で適用（L3/L4） |
| **AWS Shield Advanced** | 有料。手厚い保護＋24/7対応＋**課金保護**（DDoSで増えた料金を補償） |

> [!tip] SAAポイント
> 「DDoS対策」＝ Shield。手厚い保証・専門サポートが欲しい→ Shield Advanced。
> WAFのWeb ACLやGeo matchはL7のリクエスト制御。Shield AdvancedはDDoS緩和が主役で、国別制限を作る機能ではない。

---

## AWS Network Firewall

**VPCの入口に置く、本格的な「関所」（ステートフルファイアウォール）。**

> [!example] たとえ
> 建物の門に立つ、通行人を1人ずつ細かく検査する警備門。

- VPCトラフィックをL3〜L7で検査・フィルタリング（ドメイン単位の制御など）。SG/NACLより高度。

---

## AWS Firewall Manager

**複数アカウントのファイアウォール設定を一括管理する「本部」。**

> [!example] たとえ
> 全店舗の防犯ルールを本社が一括で配布・統一する仕組み。

- WAF・Shield・Network Firewall・SGのルールを **Organizations全体に一元適用**。

> [!tip] SAAポイント
> 「組織全体でセキュリティルールを統一管理」→ Firewall Manager。

---

## 実践：国ごとのアクセス制限（Geo matching）

**シナリオ：ALB配下のEC2でWebコンテンツを公開。自国内のクライアントのみアクセスを許可したい。**

### 正解パターン（2つある）

#### 1️⃣ AWS WAFをALBに割り当ててGeo matchルール設定

```
Web ACLでGeo matchルール
→ 「○○国は許可 / それ以外は拒否」を設定
→ ALBに紐づけ
```

- **メリット**：ALBの前段で国別フィルタリング可能
- **対応可能な付与先**：ALB / CloudFront / API Gateway / AppSync
- **実装**：Web ACL → Geo match rule で国の許可/拒否リスト

> [!tip] 重要
> **「Web ACL」は WAF の機能**。Shield に Web ACL は作れない（これが出題の罠）。

#### 2️⃣ CloudFront経由にして地理的制限を設定

```
CloudFront配下のALB/EC2
↓
CloudFront標準機能：Geographic Restrictions
→ 「許可国リスト」「拒否国リスト」を設定
```

- **メリット**：CloudFront自体が国ごとにコンテンツへのアクセスを制御。そもそもCDNを使っているなら標準機能で対応可
- **効果**：CloudFront→ALB→EC2 という流路の前段で弾く

### ❌ よくある誤り

| 選択肢 | なぜ誤り？ |
|---|---|
| **Security Group で許可国設定** | SGはIP/CIDR単位でしか制御できず、「国」という単位の指定は不可 |
| **AWS Shield をALBに割り当てて Web ACL で国設定** | ShieldはDDoS専門（L3/L4量的攻撃）。「Web ACL」は WAF の機能で、ShieldにはWeb ACLがない。「Web ACL」という単語に引っ張られるひっかけ |
| **API Gateway経由にして地理的制限** | API Gatewayには「地理的制限」機能そのものが無い（CloudFront側の機能）。そもそも不自然な構成 |

### 🎓 SAA的な着眼点

- **「国・地域でアクセス制限」 → WAF（Geo match）or CloudFront（Geographic Restrictions）**。この2つがセット
- **「Web ACL」という語が出たら主語は WAF**。Shield/SG に "Web ACL" をくっつけるのは誤り
- **Shield = DDoS（量・L3/L4） / WAF = L7のルール審査（SQLi/XSS/Geo/レート制限）**。役割を明確に分ける

## WAF のレートベースルール（重要：DDoS防御の第一手段）

**攻撃者が同じIPを使い続ける、または異なるIPで大量リクエストを送ってくる場合の防御ルール。**

### 何をするルールか

- **送信元IPアドレスごとに、直近5分間のリクエスト数を自動カウント**
- あらかじめ設定した**しきい値**（例：「5分間に2000回まで」）を**超えたIPを自動ブロック**
- 回数が落ち着けば、そのIPを再び通す

> [!key] 最大のポイント
> **攻撃元IPが事前に分かっていなくても大丈夫**。WAF が勝手に「お前さっきから叩きすぎ」と数えて締め出す。

### 実装パターン

```
Web ACL に "Rate-based rule" を追加
→ "Action" = "Block" を設定
→ "Limit" = 2000 requests / 5 min
→ ALB / CloudFront / API Gateway に紐づけ
```

結果：同じIPから短時間に大量のリクエストが来たら自動ブロック。

### なぜ他の方法はダメか（重要な対比）

| 方法 | 判定 | 理由 |
|---|---|---|
| **Network ACL に攻撃IPを手動追加してブロック** | ❌ | 攻撃者がIPを変えてくるイタチごっこ。手作業で追いつけない。「事前にIPが分かっている」前提が崩れている |
| **Amazon Inspector で検出して制限** | ❌ | **Inspector は脆弱性スキャナ**（EC2/コンテナに古いソフトや設定ミスが無いか点検）。**リアルタイムにリクエストをブロックする機能は持たない**。役割が違う |
| **WAF の地理的一致（Geo match）ルール** | ❌ | 「特定の国・地域からのアクセスを制限」ルール。**IPの回数で見るのではなく"どの国から来たか"で見る**。問題文に「特定の国から攻撃」と書いてなければ、正規ユーザーも巻き添えになる。要件と噛み合わない |
| **Security Group で攻撃IPをブロック** | ❌ | Network ACL と同じ理由。IPを変えられるとすぐ別IPで来る。IP単位の手動登録では追いつけない |

### SAA での着眼点

> [!tip] 
> - **「IPを変えて大量リクエスト」「DDoS的な負荷」 → WAF のレートベースルール** が定番解
> - 「攻撃元IPが変わる」と問題文に書いてあったら、**手動でIP指定するブロック（Network ACL / SG）は全部 ❌**
> - **Inspector = 脆弱性の点検役**。「アクセスを弾く」文脈で出たら基本ひっかけ。防御して弾くのは WAF / Shield
> - **Shield = DDoS（量・L3/L4） / WAF = L7のルール審査**。役割が違う
