# アイデンティティ系（誰かを証明する）

> [!info] このノートの範囲
> IAM Identity Center / Cognito / Directory Service / RAM。
> 「誰を、どのリソースに、どう入れるか」を担当するサービス群。土台の認可は [[0601_IAM基礎]] を参照。

---

## AWS IAM Identity Center（旧 AWS SSO）

**複数のAWSアカウント＆業務アプリに、1回のログインで入れる「統合受付」。**
シングルサインオン

> [!example] たとえ
> 大企業ビルで、入口で社員証をかざせば全フロア（経理部・開発部…）に入れる仕組み。フロアごとに別の鍵を持たなくていい。

- IAMユーザーを各アカウントに作る代わりに、**1か所でユーザー管理**し、各アカウントへ権限（パーミッションセット）を割り当てる。
- 外部IdP（Microsoft Entra ID / Okta 等）と連携してSAML SSOも可能。

> [!tip] SAAポイント
> 「**複数アカウント（Organizations）を一元的にアクセス管理したい**」→ IAM Identity Center が定番解答。

---

## Amazon Cognito

**自分が作る「アプリの利用者（一般ユーザー）」向けのログイン基盤。**

> [!example] たとえ
> 自社が運営するスマホアプリの「会員登録・ログイン画面」を丸ごと用意してくれるサービス。

- **ユーザープール**＝ サインアップ／サインインの認証（誰であるか）。Google・Apple・Facebookログインも統合。
- **IDプール（フェデレーテッドアイデンティティ）**＝ 認証済みユーザーに一時的なAWS権限を付与（S3へ直接アップロード等）。

> [!tip] SAAポイント
> **IAM＝社内の人/サービス、Cognito＝アプリのエンドユーザー（B2C）**。「モバイル/Webアプリの何百万人ユーザーを認証」→ Cognito。

---

## AWS Directory Service

**Microsoft Active Directory（AD）をAWS上で使うためのマネージドサービス。**

> [!example] たとえ
> 会社が昔から使う社員名簿システム（AD）を、クラウドにそのまま引っ越し or 連携させる。

| 種類 | 用途 |
|---|---|
| **AWS Managed Microsoft AD** | 本格的なフルADがクラウドに必要なとき |
| **AD Connector** | オンプレADへの「橋渡し（プロキシ）」。認証はオンプレ側で実行 |
| **Simple AD** | 小規模・低コストの簡易AD |

> [!tip] SAAポイント
> 「オンプレADの認証情報をそのまま使いたい」→ AD Connector。「クラウドに独立したADが欲しい」→ Managed Microsoft AD。

---

## AWS Resource Access Manager（AWS RAM）

**自分のAWSリソースを、他アカウントと「共有」する仕組み。**

> [!example] たとえ
> 会議室（VPCサブネット等）を、隣の部署にも使わせてあげる「共用設定」。各部署がそれぞれ建てなくて済む。

- 共有できる代表例：**VPCサブネット、Transit Gateway、Route 53 Resolverルール、ライセンス**など。

> [!tip] SAAポイント
> 「複数アカウントでVPCを共有してコスト削減・一元管理」→ RAM（VPC Sharing）。
