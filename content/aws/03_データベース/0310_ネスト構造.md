# ネスト構造（入れ子） — プログラミング・データ構造の基礎

**ネスト（入れ子）** は、プログラミングやデータ全般で出てくる基礎概念です。DynamoDBの学習で必須となるため、ここで丁寧に説明します。

---

## ネスト構造 = 「箱の中に箱が入っている」状態

まず**ネストしていない（フラットな）データ**から見ます。1階建てで、項目と値がただ並んでいるだけ：

```json
{
  "名前": "田中",
  "年齢": 30,
  "電話": "090-1234-5678"
}
```

`名前→田中`、`年齢→30` のように、**値がそれ以上分かれない**。これがフラット（平ら）な状態です。

---

## パターン① 値の中に、さらに項目が入る（入れ子）

ここで「住所」を持たせたいとします。住所は「都道府県・市・番地」に分けられますよね。**値の中にさらに小さな項目のかたまりを入れる**のがネストです：

```json
{
  "名前": "田中",
  "年齢": 30,
  "住所": {              ← 「住所」の値が、ただの文字ではなく"かたまり"
    "都道府県": "東京都",
    "市": "渋谷区",
    "番地": "1-2-3"
  }
}
```

`住所` を開けると、**中にまた `都道府県`・`市`・`番地` という項目が入っている**。これが「入れ子（ネスト）」＝**箱を開けたら中にまた箱がある**状態です。

---

## パターン② 値がリスト（複数）になっていて、その中身もかたまり

もう一つのネストの形。1人が複数の電話番号を持つとします。**値が「リスト（並び）」になり、しかもその一つ一つがかたまり**：

```json
{
  "名前": "田中",
  "電話一覧": [                          ← リスト（複数入る）
    { "種類": "自宅", "番号": "03-1111" },   ← 中身もかたまり
    { "種類": "携帯", "番号": "090-2222" }
  ]
}
```

`電話一覧` の中に、電話が2件並んでいて、それぞれが `種類`・`番号` を持つ。**リスト × かたまり**の二重の入れ子です。

---

## DynamoDBのチケット例で読む

これでネスト構造が読めるようになります：

```json
{
  "ticket_id": "1",
  "顧客プロフィール": { "名前": "田中", "会員ランク": "ゴールド" },   ← パターン①（かたまり）
  "メッセージ一覧": [                                              ← パターン②（リスト×かたまり）
    { "誰": "顧客",   "本文": "問い合わせ内容…" },
    { "誰": "サポート", "本文": "返信です…" }
  ]
}
```

- `顧客プロフィール` = 中に `名前`・`会員ランク` が入ったかたまり（①）
- `メッセージ一覧` = メッセージが複数並んだリストで、各メッセージも `誰`・`本文` を持つ（②）

**1件のチケットという「大きな箱」の中に、プロフィールという箱、メッセージ一覧という箱…が入っている**。これが「ネストしたドキュメント構造」の正体です。

---

## なぜこれがDynamoDBの話で重要か

> [!tip]
> - **RDB（表のDB）は「箱の中の箱」を1つのマスに入れられない**。だから住所やメッセージを別テーブルに割って、JOINで組み立て直す必要がある。
> - **DynamoDBはこの入れ子構造をそのまま1件として保存でき、そのまま丸ごと取り出せる**。だから「ネストしたドキュメント構造で保存したい」という要件にピッタリだった。

---

## 補足：ネストの応用範囲

この「ネスト＝入れ子」は JSON に限らず、**フォルダの中にフォルダがある（階層構造）** のも同じ発想です。

**あるものの中に、同じ種類のものがさらに入っている構造**を全部「ネスト」と呼びます。

- 📁 フォルダ（中にさらにフォルダ）
- 📑 XMLやJSON（要素の中に要素）
- 🗂️ HTML（タグの中にタグ）
- 💾 プログラムコード（ブロックの中にブロック）

すべて同じ「ネスト」という概念です。
