# AWS RDS Proxy

## 🗂 概要

> **RDS Proxy = アプリとRDSの間に置く「受付窓口」（コネクションプーラー）**
> コネクションを使い回すことで、RDSへの接続数を減らして安定させる。

---

## 🏦 銀行で例えると

```
❌ Proxy なし
お客さん1000人 → 全員が直接、行員に話しかける → 行員パンク💥

✅ Proxy あり
お客さん1000人 → 受付が整理して → 少人数の行員に振り分ける → 安定✅
```

| たとえ | 実際 |
|---|---|
| お客さん | Lambda関数 |
| 行員 | RDSのコネクション |
| 受付 | RDS Proxy |

---

## ❓ なぜLambdaが問題になるのか

通常のサーバー（EC2など）は**起動しっぱなし**なので、DBコネクションを張りっぱなしにできる。

しかしLambdaは…

```
呼ばれる → 起動 → 処理 → 終了  ← これを何千回も繰り返す
```

同時に1000個呼ばれると → 1000本のコネクションがRDSに刺さる → **RDSがコネクション上限オーバーで死ぬ** 💥

RDS Proxyが間に入ることで、コネクションを**プール（使い回し）**して数を激減させる。

---

## 🛠 主な機能

| 機能 | 内容 |
|---|---|
| **コネクションプーリング** | コネクションを使い回してRDSへの接続数を削減 |
| **フェイルオーバー短縮** | Multi-AZ障害時のフェイルオーバーを最大**66%短縮** |
| **IAM認証の強制** | DBパスワードを使わずIAMで接続を一元管理 |
| **Secrets Manager連携** | DB認証情報をSecrets Managerで安全に管理 |

---

## 📐 構成イメージ

```
❌ Proxy なし
Lambda × 1000 ──── 1000コネクション ────▶ RDS 💥

✅ Proxy あり
Lambda × 1000 ──▶ RDS Proxy ──── 少数コネクション ────▶ RDS ✅
                 （プール・再利用）
```

---

## 🧠 試験頻出ポイント

| キーワード | 答え |
|---|---|
| LambdaからRDSに大量接続 → コネクション枯渇 | **RDS Proxy**を間に挟む |
| フェイルオーバーをより速くしたい | **RDS Proxy**（最大66%短縮） |
| DBパスワードを使わずに安全に接続したい | **RDS Proxy + IAM認証** |

> 💡 **「Lambda + RDS」の組み合わせが出たら、ほぼRDS Proxyが答え**

## ⚠️ よくある勘違い（試験の罠）

**「RDS Proxyは読み取りを別ノードに逃がす」と思い込む学生が多い** ← これは**間違い**。

| 機能 | RDS Proxy | Multi-AZ DBクラスター | リードレプリカ |
|---|---|---|---|
| コネクションプーリング | ✅ **本来の役割** | - | - |
| 読み取り分離 | ✗ **できない** | ✅ リーダーEPで自動 | ✅ 手動で構築 |
| 高可用性（自動フェイルオーバー） | △（フェイルオーバー短縮のみ） | ✅ 完全自動 | ✗ 手動対応 |

→ **Proxyは「接続を整理する案内係」に過ぎず、読み取りを別フロア（リーダー）に振り分ける機能はない。**
