# Amazon DynamoDB

**フルマネージドのNoSQL（キー・バリュー型）データベース**。
数百万リクエスト/秒にも対応できる、スケーラビリティと低レイテンシ（1桁ミリ秒）に特化したDB。サーバー管理・容量設計が不要。

---

## そもそも何者か（初心者向け：RDBとの違いで掴む）

DynamoDBを理解する鍵は、**昔ながらのRDB（MySQL / PostgreSQL）とデータの持ち方が根本的に違う**こと。3つの言葉に分けて押さえる。（「ネスト（入れ子）」自体が曖昧な場合は先に [[0310_ネスト構造]] を読む。IT基礎側の詳説は [[JSONとネスト構造]]）

### ① NoSQL＝「表に割らない」DB

**RDB** は、1件のデータを**複数のテーブルに分けて**持ち、取り出すときに**JOIN（結合）で突き合わせる**。例：1件のサポートチケットが3テーブルに散らばる。

```
ticketsテーブル      : ticket_id=1, 顧客名="田中", 状態="対応中"
messagesテーブル     : ticket_id=1, "問い合わせ内容…" / ticket_id=1, "返信です…"
attachmentsテーブル  : ticket_id=1, "スクショ.png"
→ 1件を取るのに3テーブルをJOINで結合
```

**DynamoDB（NoSQL）** は、関連するものを**1個のアイテム（かたまり）にまるごと**入れる。JOINが要らない。

### ② 「JSONのようなネスト（入れ子）構造をそのまま保存」

DynamoDBの1アイテムは、こういう**入れ子のJSON**をそのまま格納できる。

```json
{
  "ticket_id": "1",
  "顧客プロフィール": { "名前": "田中", "会員ランク": "ゴールド" },   // ← 中にさらに項目（入れ子）
  "メッセージ一覧": [                                              // ← リストの中に複数（入れ子）
    { "誰": "顧客",   "本文": "問い合わせ内容…" },
    { "誰": "サポート", "本文": "返信です…" }
  ],
  "添付ファイル": [ { "ファイル名": "スクショ.png", "サイズ": 20480 } ]
}
```

RDBは「表の1マスに表を入れる」ができないのでテーブルを割ってJOINが要る。DynamoDBは**この入れ子を1アイテムのまま保存でき、取り出すときもかたまりごと一発**。「スレッド形式のメッセージ一覧・添付メタデータ・顧客プロフィールをネストしたドキュメント構造で保存」といった要件はこの形がぴったりハマる。

### ③ なぜ「1桁ミリ秒の低レイテンシ」か

- **レイテンシ** = リクエストを投げてから応答が返るまでの待ち時間。**1桁ミリ秒** = 1〜9ミリ秒でほぼ一瞬。
- **速い理由**：上のように**1アイテム取れば済む（JOINで複数テーブルを突き合わせない）**設計だから。さらに裏で自動的に複数サーバーへデータを分散して持つので、**アクセスが増えても1件あたりの速さが落ちにくい**（＝「大規模でも低レイテンシ」）。
- 割り切り：**取り方をシンプルにする代わりに、大量アクセスでも安定して速い**。逆に「複雑な集計・分析クエリ」「表をまたいだ柔軟な検索」が主役ならRDBやAthenaの方が向く（→ [[0301_RDS]]）。

> [!tip] SAAでの合図
> 「**ネスト（入れ子）のドキュメントをそのまま保存**」「**トラフィック増でも低レイテンシ**」「**スキーマ柔軟／ミリ秒応答／無制限スケール**」→ **DynamoDB**。DynamoDBはS3に直接スキャンされない（Macie等に回すなら一旦S3へエクスポート）点もセットで押さえる。

## 基本構造

### RDBとの対応

| RDB | DynamoDB |
|-----|----------|
| テーブル | テーブル |
| 行（Row） | アイテム（Item） |
| 列（Column） | 属性（Attribute）※アイテムごとに自由 |
| 主キー | パーティションキー（＋ソートキー） |

### キーの種類

| キー | 必須 | 役割 | 例 |
|---|---|---|---|
| **パーティションキー** | 必須 | データを**どのパーティション（物理的な置き場）に分散するか**を決めるハッシュキー | `userId`, `orderId` |
| **ソートキー** | 任意 | 同じパーティション内での**並び順**。範囲検索が可能になる | `timestamp`, `productId` |

> [!tip] 例え：図書館の分類
> パーティションキー＝「棚の番号」、ソートキー＝「棚の中での並び順」。棚番号さえ分かれば目的の本を高速で取り出せる。逆に言うと、**キー以外の属性での検索は苦手**（→ 後述のGSIで補う）。

## キャパシティモード

| | オンデマンド | プロビジョンド |
|---|---|---|
| 設定 | 不要（自動スケール） | **RCU / WCU** を事前設定 |
| 課金 | リクエスト数に応じた従量課金 | 確保した分を常時課金（安い） |
| 向き | **予測不能・スパイクのあるトラフィック** | **予測可能な安定トラフィック** |
| 備考 | — | Auto Scaling と組み合わせ可能 |

### RCU / WCU の定義

| 単位 | 内容 |
|------|------|
| 1 RCU | 強整合性読み取り 4KB×1回/秒（結果整合性なら4KB×2回/秒） |
| 1 WCU | 書き込み 1KB×1回/秒 |

---

## 整合性モデル

| | 結果整合性（デフォルト） | 強整合性 |
|---|---|---|
| 動作 | 書き込み直後は古いデータを返す可能性あり（数秒で収束） | 必ず最新データを返す |
| コスト | **0.5 RCU（半額）** | 1 RCU |
| 指定 | — | `ConsistentRead: true` |

## インデックス（LSI / GSI）

| | LSI（Local Secondary Index） | GSI（Global Secondary Index） |
|---|---|---|
| パーティションキー | テーブルと**同じ** | **別のキー**を指定できる |
| ソートキー | 別のキーを指定 | 別のキーを指定 |
| 作成タイミング | **テーブル作成時のみ** | **後から追加可能** |
| 整合性 | 強整合性も可能 | 結果整合性のみ |

> 「キー以外の属性で検索したい」→ **GSI** を張る、が基本パターン。

> [!note] 用語
> **グローバルセカンダリインデックス（GSI）** は、テーブル本体とは別のパーティションキー/ソートキーで問い合わせるための副索引。選択肢では「Global Secondary Index」より日本語の正式名で出ることもある。

### スパースインデックス（SAAで出やすい）

GSIは、インデックスキー属性を持つアイテムだけを含められる。これを利用して「未発送注文だけ `statusIndexKey` を持たせる」ようにすると、未発送注文だけを小さなインデックスとして高速・低コストに検索できる。

> [!tip] ひっかけ
> 「条件に合う一部のアイテムだけを効率的に検索」→ **GSI（スパースインデックス）**。
> DAXは読み取りキャッシュであり、Scanの対象を減らすためのインデックスではない。

## 主要機能

| 機能 | 内容 | 典型用途 |
|---|---|---|
| **DynamoDB Streams** | テーブル変更（INSERT/MODIFY/REMOVE）をイベントストリームとして記録（保持**24時間**）。Lambdaと連携してリアルタイム処理 | 変更をトリガーに後続処理 |
| **TTL** | アイテムに有効期限（**Epoch秒**）を設定し自動削除（無料・即時ではない） | セッション・一時データ |
| **DynamoDB Accelerator（DAX）** | DynamoDB専用**インメモリキャッシュ**。レイテンシを**ミリ秒→マイクロ秒**へ | 読み取り集中ワークロード |
| **トランザクション** | `TransactWriteItems` / `TransactGetItems` で**ACID**操作（最大**100アイテム**） | 残高の減算＋加算をセットで成功/失敗させる等 |
| **グローバルテーブル** | **複数リージョンへ自動レプリケーション**。マルチリージョン読み書き | DR・グローバル低レイテンシ |

```json
// TTLの例：expireAt が過ぎたら自動削除
{
  "userId": "user123",
  "sessionToken": "abc...",
  "expireAt": 1735689600
}
```

## SAA試験ポイント

| 問題文のキーワード | 答え |
|---|---|
| 「ミリ秒レイテンシ」「無制限にスケールするNoSQL」「サーバー管理不要」 | **DynamoDB** |
| 予測不能なトラフィック | **オンデマンドモード** |
| 読み取りコストを半分にしたい（多少古くてもよい） | **結果整合性読み取り（0.5 RCU）** |
| 後から別の属性で検索できるようにしたい | **GSI** |
| テーブルの変更をトリガーに処理したい | **Streams + Lambda** |
| 「マイクロ秒」の読み取りレイテンシが必要 | **DAX** |
| 検索条件を変えたい/一部アイテムだけ効率的に取りたい | **GSI** |
| セッションデータを期限切れで自動削除 | **TTL** |
| 複数リージョンで読み書き・DR | **グローバルテーブル** |
