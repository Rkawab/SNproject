# Amazon ElastiCache

**よく使うデータをメモリ上に置いておくことで、DBへのアクセスを減らして高速化するマネージドのインメモリキャッシュサービス**。
RDSの手前に置いて読み取り負荷を肩代わりさせるのが典型構成。

関連：[[0301_RDS]] / [[0201_S3]]

> [!tip] 例え：コンビニのレジ横
> 倉庫から商品を取り出す＝RDS（時間がかかる）、売れ筋商品をレジ横に並べておく＝ElastiCache（すぐ渡せる）。
> 技術的には「ディスク（RDS）に対するメモリ（RAM）」の関係で、速いが高価・容量は小さい。

## 動作の流れ（キャッシュアサイド）

```
ユーザー → アプリ → [ElastiCache] → キャッシュHITなら即返却
                         ↓ キャッシュMISSなら
                        RDS → 結果をキャッシュに保存して返却
```

## 2つのエンジン

| | **Redis** | **Memcached** |
|---|---|---|
| データ構造 | リスト・セット・ソート済みセットなど豊富 | シンプルなKey-Value |
| 永続化 | **あり**（スナップショット・AOF） | なし |
| レプリケーション | **あり（Multi-AZ対応・自動フェイルオーバー）** | なし |
| クラスタリング | あり | あり（シンプルなスケールアウト） |
| Pub/Sub | あり | なし |
| 用途 | セッション管理・リーダーボード・Pub/Sub | 純粋なキャッシュ（マルチスレッドで単純高速） |

> [!info] 迷ったらRedis。Memcachedが正解になるのは「シンプルなキャッシュを水平スケールしたいだけ」「永続化・冗長化が不要」と明言されたときくらい。

## 典型的なユースケース

- DBクエリ結果のキャッシュ（頻繁に変わらないマスターデータなど）
- セッション情報の保存（ログイン状態の管理）→ ステートレスなWeb層の実現
- リーダーボード（ゲームのランキング：Redisのソート済みセット）
- レート制限（API呼び出し回数の管理）

## SAA試験ポイント

| シナリオ | 答え |
|---|---|
| RDSへの読み取りアクセスが多くて遅い | **ElastiCacheを前に置く**（またはリードレプリカ。「同じクエリの繰り返し」ならキャッシュが有利） |
| セッション管理が必要 | **Redis** |
| 永続化・レプリケーション・複雑なデータ型・Pub/Subが必要 | **Redis** |
| シンプルなキャッシュを単純にスケールアウトしたい | **Memcached** |
| DynamoDB専用のキャッシュ | ElastiCacheではなく **DAX** → [[0303_DynamoDB]] |
| Redis互換で**データを失えない**（プライマリDBにする） | ElastiCacheではなく **MemoryDB** → [[0309_Keyspaces・Timestream・MemoryDB]] |
