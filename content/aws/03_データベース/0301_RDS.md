# AWS RDS（Relational Database Service）

## 🗂 概要

> **RDS = AWSが提供するマネージド型リレーショナルデータベースサービス** OSのパッチ適用・バックアップ・レプリケーションなどの運用をAWSが肩代わりしてくれる。

自前でEC2にDBを立てる場合と比較すると、運用負荷が大幅に下がる。

---

## 🆚 EC2 on DB vs RDS

|項目|EC2に自分でDB|RDS（マネージド）|
|---|---|---|
|OSの管理|自分|AWS|
|DBエンジンのパッチ|自分|AWS|
|バックアップ|自分|AWS（自動）|
|フェイルオーバー|自分で設計|Multi-AZで自動|
|スケーリング|手動|比較的容易|
|SSHアクセス|○|✗（できない）|

> ⚠️ **SSHでOSにログインできない**のがRDSの大きな特徴。完全にAWS管理下。

---

## 🛠 対応DBエンジン（6種類）

| エンジン                 | 特徴                          |
| -------------------- | --------------------------- |
| MySQL                | 最も広く使われるOSS DB              |
| PostgreSQL           | 高機能OSSで型が豊富                 |
| MariaDB              | MySQLのフォーク                  |
| Oracle               | 商用DB、ライセンス注意                |
| Microsoft SQL Server | Windows環境に多い                |
| **Amazon Aurora**    | AWSが独自開発。MySQLとPostgreSQL互換 |

## 🌟 Amazon Aurora（重要）

RDSの中でも特別扱いされる特別なエンジン。

|項目|内容|
|---|---|
|互換性|MySQL / PostgreSQL と互換|
|パフォーマンス|MySQLの**5倍**、PostgreSQLの**3倍**|
|ストレージ|3 AZ × 2コピー = **6コピー**自動で保持|
|フェイルオーバー|**30秒以内**に自動完了|
|自動スケール|10GB から最大**128TB**まで自動拡張|
|リードレプリカ|最大**15台**|

> 💡 **試験ポイント：** Auroraは「高可用性・高パフォーマンス」が求められる問題で頻出。
> 詳細: [[0307_Aurora詳細]]

## 🔁 Multi-AZ（マルチAZ）（高可用性）

> [!info] 表記について
> 試験・問題文では「**マルチAZ**」とカタカナで出ることが多い（このノートの「Multi-AZ」と同じ意味）。「マルチAZ DBインスタンス」＝下記の従来型、「マルチAZ DBクラスター」＝下記の新型を指す。

### Multi-AZ DB インスタンス（マルチAZ DBインスタンス・従来型）

```
プライマリDB（AZ-a）  ──── 同期レプリケーション ────▶  スタンバイDB（AZ-b）
       ↑
   通常アクセス（スタンバイは待機専用）
```

- スタンバイはあくまで**待機専用**（読み取りには使えない）
- プライマリ障害時に**自動フェイルオーバー**（DNSが切り替わる）
- 目的は**高可用性のみ（自動フェイルオーバー）**
- **読み取り分散はできない** → 監査レポートなど負荷の高い読み取りを本番に影響させたい場合には不適切

> [!note] そもそも「クラスター」「DBクラスター」とは
> **クラスター（cluster）＝「房・かたまり」**。複数のサーバー（インスタンス）を束ねて、外からは“1つのまとまり”として扱う仕組み。AWSでは ECSクラスター・Redshiftクラスター等にも同じ意味で出てくる。
> - **DBインスタンス**（単数）＝ サーバー**1台**そのもの。
> - **DBクラスター**（束）＝ 役割の違う**複数台**（ライター＋リーダー等）を1セットに束ねた構成。
> - だから「マルチAZ DB**インスタンス**」＝本番1台＋待機1台のシンプル構成、「マルチAZ DB**クラスター**」＝ライター＋リーダー複数台を束ねた構成、という対比になる（Auroraも中身はこのクラスター方式）。

### 🌟 Multi-AZ DB クラスター（マルチAZ DBクラスター・新型・推奨）

```
ライター（AZ-a）
     ↓（同期レプリケーション）
┌─────────┬──────────┐
│ リーダー  │ リーダー  │（AZ-b, AZ-c）
│ EP指定   │ EP指定   │（読み取りのみ・スケールアウト可能）
└─────────┴──────────┘
```

**構成**
- **ライター EP（Writer Endpoint）** × 1：書き込み用（AZ-a）
- **リーダー EP（Reader Endpoint）** × 2+：読み取り用（別AZ）
- **自動フェイルオーバー**：ライター障害時は別AZのリーダーが即座に昇格

**利点**
- ✅ **読み取りスケール**：監査・レポート・分析を別リーダーEPで分離 → 本番への影響ゼロ
- ✅ **高可用性**：ライター障害時も自動フェイルオーバー（数秒で復旧）
- ✅ **マネージド**：スタンバイ管理を自動化（リードレプリカより運用が楽）
- ⚠️ リーダーへの反映は数秒の遅延あり（許容範囲なら OK）

**選択基準**

| 要件                    | Multi-AZ インスタンス | Multi-AZ クラスター |
|-------------------------|-------|----------|
| 高可用性のみ             | ✅    | ✅      |
| 読み取り分離＋高可用性     | ✗     | ✅ **推奨** |
| 読み取り数秒の遅延許容     | ✗     | ✅      |
| 読み取り負荷高い（監査/分析） | ✗     | ✅ **推奨** |
| 運用負荷を最小化          | △     | ✅      |

> ⚠️ **試験の罠**：「マルチAZ」という単語だけで判断すると、スタンバイ読み取り不可に気づかないまま B を選んでしまう。

## 📖 リードレプリカ（読み取りスケールアウト）

```
書き込み ──▶ プライマリDB
                │
                │ 非同期レプリケーション
                ▼
読み取り ──▶ リードレプリカ（最大15台 ※Oracle/SQL Serverは5台 / Auroraも15台）
```

- **非同期**レプリケーション（わずかな遅延あり）
- 別AZ・別リージョンにも作成可能
- レプリカを**昇格させて**新しいプライマリDBにすることも可能
- 目的は**読み取り負荷の分散**

---

## 💾 バックアップ

### 自動バックアップ

- **1〜35日**の保持期間を設定
- 毎日バックアップウィンドウの時間帯に自動取得
- ポイントインタイムリカバリ（5分前まで戻せる）

### 手動スナップショット

- 手動で任意タイミングに取得
- 保持期間の制限なし（削除するまで残る）

## 🔐 セキュリティ

|機能|内容|
|---|---|
|保存時暗号化|AWS KMSを使ったAES-256|
|転送時暗号化|SSL/TLSを使用|
|IAM認証|パスワード不要でIAMで接続可|
|VPCサポート|VPC内に配置して外部からのアクセスを遮断|
|セキュリティグループ|インバウンドルールでアクセス制御|

> ⚠️ 暗号化は**作成時にのみ**設定可能。後からONにするにはスナップショット→復元が必要。
