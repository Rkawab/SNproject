# AWS RDS（Relational Database Service）

## 🗂 概要

> **RDS = AWSが提供するマネージド型リレーショナルデータベースサービス** OSのパッチ適用・バックアップ・レプリケーションなどの運用をAWSが肩代わりしてくれる。

自前でEC2にDBを立てる場合と比較すると、運用負荷が大幅に下がる。

---

## 🆚 EC2 on DB vs RDS

|項目|EC2に自分でDB|RDS（マネージド）|
|---|---|---|
|OSの管理|自分|AWS|
|DBエンジンのパッチ|自分|AWS|
|バックアップ|自分|AWS（自動）|
|フェイルオーバー|自分で設計|Multi-AZで自動|
|スケーリング|手動|比較的容易|
|SSHアクセス|○|✗（できない）|

> ⚠️ **SSHでOSにログインできない**のがRDSの大きな特徴。完全にAWS管理下。

---

## 🛠 対応DBエンジン（6種類）

| エンジン                 | 特徴                          |
| -------------------- | --------------------------- |
| MySQL                | 最も広く使われるOSS DB              |
| PostgreSQL           | 高機能OSSで型が豊富                 |
| MariaDB              | MySQLのフォーク                  |
| Oracle               | 商用DB、ライセンス注意                |
| Microsoft SQL Server | Windows環境に多い                |
| **Amazon Aurora**    | AWSが独自開発。MySQLとPostgreSQL互換 |

---

## 🌟 Amazon Aurora（重要）

RDSの中でも特別扱いされる特別なエンジン。

|項目|内容|
|---|---|
|互換性|MySQL / PostgreSQL と互換|
|パフォーマンス|MySQLの**5倍**、PostgreSQLの**3倍**|
|ストレージ|3 AZ × 2コピー = **6コピー**自動で保持|
|フェイルオーバー|**30秒以内**に自動完了|
|自動スケール|10GB から最大**128TB**まで自動拡張|
|リードレプリカ|最大**15台**|

> 💡 **試験ポイント：** Auroraは「高可用性・高パフォーマンス」が求められる問題で頻出。

---

## 🔁 Multi-AZ（高可用性）

```
プライマリDB（AZ-a）  ──── 同期レプリケーション ────▶  スタンバイDB（AZ-b）
       ↑
   通常アクセス
```

- スタンバイはあくまで**待機専用**（読み取りには使えない）
- プライマリ障害時に**自動フェイルオーバー**（DNSが切り替わる）
- 目的は**DR（ディザスタリカバリ）**

> ⚠️ スタンバイDBへの読み取り分散はできない → それは **リードレプリカ** の役割

---

## 📖 リードレプリカ（読み取りスケールアウト）

```
書き込み ──▶ プライマリDB
                │
                │ 非同期レプリケーション
                ▼
読み取り ──▶ リードレプリカ（最大15台 ※Oracle/SQL Serverは5台 / Auroraも15台）
```

- **非同期**レプリケーション（わずかな遅延あり）
- 別AZ・別リージョンにも作成可能
- レプリカを**昇格させて**新しいプライマリDBにすることも可能
- 目的は**読み取り負荷の分散**

---

## 💾 バックアップ

### 自動バックアップ

- **1〜35日**の保持期間を設定
- 毎日バックアップウィンドウの時間帯に自動取得
- ポイントインタイムリカバリ（5分前まで戻せる）

### 手動スナップショット

- 手動で任意タイミングに取得
- 保持期間の制限なし（削除するまで残る）

---

## 🔐 セキュリティ

|機能|内容|
|---|---|
|保存時暗号化|AWS KMSを使ったAES-256|
|転送時暗号化|SSL/TLSを使用|
|IAM認証|パスワード不要でIAMで接続可|
|VPCサポート|VPC内に配置して外部からのアクセスを遮断|
|セキュリティグループ|インバウンドルールでアクセス制御|

> ⚠️ 暗号化は**作成時にのみ**設定可能。後からONにするにはスナップショット→復元が必要。

---

## 📐 ストレージ種別

|種別|用途|
|---|---|
|gp2 / gp3（汎用SSD）|一般的なワークロード|
|io1 / io2（プロビジョンドIOPS）|高I/O要件のDBに|
|マグネティック|レガシー（非推奨）|

---

## 📊 モニタリング

- **CloudWatch** でCPU・ストレージ・接続数などを監視
- **Enhanced Monitoring** でOS内部のメトリクス（プロセスごとのCPUなど）も取得可能
- **Performance Insights** でSQLレベルのボトルネック分析が可能

---

## 🧠 試験頻出まとめ

| テーマ      | 覚えること                      |
| -------- | -------------------------- |
| 高可用性     | Multi-AZ（同期・自動フェイルオーバー）    |
| 読み取りスケール | リードレプリカ（非同期・最大15台 ※Oracle/SQL Serverは5台）          |
| 最強構成     | Aurora（6コピー・15レプリカ・自動スケール） |
| SSHアクセス  | できない（マネージドだから）             |
| 暗号化      | 作成時のみ設定可                   |
| バックアップ保持 | 自動は最大35日、スナップショットは無制限      |

---

## 🔗 関連サービス

- `Amazon Aurora` → RDSのハイパフォーマンス版
- `ElastiCache` → インメモリキャッシュ（DBの手前に置いて高速化）
- `DynamoDB` → NoSQL、RDSとは別物
- `AWS Backup` → RDSスナップショットを集中管理
