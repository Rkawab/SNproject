# S3 イベント通知

## 一言で言うと

> **「S3に何かが起きたら、自動で次の処理を動かす」トリガーの仕組み**

S3単体では「保存するだけ」だが、イベント通知を使うと「保存 → 自動処理」がつながる。

---

## アナロジー：宅配ボックスの通知機能

| | 説明 |
|---|---|
| 📦 **通知なし** | 荷物が届く → ボックスに入る → **自分で確認しに行く**（手間） |
| 🔔 **通知あり** | 荷物が届く → **自動で「届いたよ！」と通知** → すぐ対応できる |

S3も同じ。ファイルが届いたら自動で次のアクションを起こせる。

---

## 検知できるイベント

| カテゴリ     | 代表的なイベント                            |
| -------- | ----------------------------------- |
| オブジェクト作成 | PUT / POST / COPY / マルチパートアップロード完了  |
| オブジェクト削除 | DELETE（バージョンあり・なし両方）                |
| オブジェクト復元 | Glacierからの復元開始・完了                   |
| レプリケーション | 失敗・タイムアウト・レプリケーション済み[[0206_S3のレプリケーション]] |
| ライフサイクル  | オブジェクトの期限切れ・移行                      |

---

## 通知の送信先（3種類）

```
S3バケット
    ├─→ Lambda  ：届いたら即、コードを動かす
    ├─→ SQS     ：届いたら一旦キューに積んで、順番に処理
    └─→ SNS     ：届いたらメール・Slack等に知らせる
```

| 送信先 | ひと言 | 主なユースケース |
|---|---|---|
| **Lambda** | 即時にコードを実行 | 画像リサイズ、ETL処理、自動変換 |
| **SQS** | キューで非同期処理 | 順番制御が必要なジョブ、処理の平滑化 |
| **SNS** | 複数宛先へ一斉通知 | 管理者アラート、複数サービスへのファンアウト |

---

## 設定のポイント

- 設定場所：**バケット単位**（S3コンソール → プロパティ → イベント通知）
- **プレフィックス・サフィックスでフィルタリング**可能
  - 例：`images/` 配下の `.jpg` ファイルだけ通知する
- 通知先に **リソースポリシーの付与が必要**（S3からのアクセスを許可）
- より複雑なルーティングには **EventBridge連携** も可能

---

## 典型的なアーキテクチャ例

```
ユーザー → S3にCSVをアップロード
                ↓（オブジェクト作成イベント発火）
           Lambda が自動起動
                ↓
           DynamoDBに集計結果を書き込み
```

## 📝 SAA試験の出題パターン

| 問題のキーワード | 正解の構成 |
|---|---|
| 「アップロードされたら自動処理」 | S3 + **Lambda** |
| 「複数サービスに同時通知」 | S3 + **SNS**（ファンアウト） |
| 「処理の順番を制御・キューイング」 | S3 + **SQS** |
| 「複雑なルーティング条件」 | S3 + **EventBridge** |
