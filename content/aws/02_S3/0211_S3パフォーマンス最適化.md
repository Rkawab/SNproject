[[0201_S3]]
# S3 パフォーマンス最適化

## 概念整理

| 用語 | 意味 | 例え |
|------|------|------|
| **レイテンシ** | リクエストが返るまでの時間 | 信号が青になるまでの待ち時間 |
| **スループット** | 単位時間あたりの処理量 | 1時間に通れる車の台数 |

---

## 1. レイテンシの削減：S3 Transfer Acceleration

### 仕組み

```
通常：  クライアント ──[インターネット]──────────→ S3バケット
TA後：  クライアント → [近くのエッジ] → [AWSバックボーン] → S3バケット
```

CloudFrontのエッジロケーション経由でAWSバックボーンネットワークを使う。

### ポイント

| 項目 | 内容 |
|------|------|
| 対象 | 遠距離からのアップロード（海外クライアント → S3等） |
| 効果 | 長距離転送で最大60〜500%高速化 |
| 料金 | 追加コストあり（通常転送より高い） |
| 有効化 | バケットプロパティで有効化 → エンドポイントが変わる |

> 📝 **試験ポイント**：グローバルなユーザーがS3に大容量ファイルをアップロード → **Transfer Acceleration**

---

## 2. キャッシュの使用：CloudFront

### 仕組み

```
ユーザー → CloudFront（キャッシュあり）→ S3（オリジン）
                ↑
          2回目以降はここで返す（S3にリクエストが届かない）
```

### CloudFront vs ElastiCache

| | CloudFront | ElastiCache |
|---|---|---|
| 対象 | 静的コンテンツ（画像・JS等） | 動的データ（DBクエリ結果等） |
| キャッシュ場所 | エッジロケーション | VPC内のインメモリ |
| S3との組み合わせ | ◎ よく使う | △ 直接的でない |

| 項目 | 内容 |
|------|------|
| TTL | キャッシュの有効期限（デフォルト24時間） |
| 無効化 | CloudFront Invalidationでキャッシュを強制削除 |
| メリット | S3へのリクエスト数・コスト削減、レイテンシ激減 |

> 📝 **試験ポイント**：S3の静的Webサイトを世界中に高速配信 → **CloudFront**

---

## 3. スループットの最適化

### S3のスループット上限（プレフィックスあたり）

| 操作 | 上限 |
|------|------|
| PUT / COPY / POST / DELETE | **3,500 req/秒** |
| GET / HEAD | **5,500 req/秒** |

この上限にぶつかると、リクエストがエラーや遅延になる。

---

### 3-1. プレフィックス分散（上限を突破する）

**銀行の窓口に例えると：**

```
❌ 窓口1つに全員並ぶ → 5,500人/時間の壁

✅ 窓口を4つに分ける → 5,500 × 4 = 22,000人/時間
```

**S3のキー設計：**

```
❌ 全部同じフォルダ
s3://bucket/photos/001.jpg
s3://bucket/photos/002.jpg  → 5,500 req/秒が上限

✅ 日付や種類でフォルダを分ける
s3://bucket/2024-01/001.jpg → 5,500 req/秒
s3://bucket/2024-02/002.jpg → 5,500 req/秒
s3://bucket/2024-03/003.jpg → 5,500 req/秒
```

> ⚠️ **昔の話**：以前はキー名にランダムプレフィックスを付ける（`abc123/images/...`）テクニックが必要だったが、**2018年以降は不要**。日付やカテゴリで普通に分けるだけでOK。

> 📝 **試験ポイント**：GETリクエストが多すぎてスローダウン → **プレフィックスを分散**

---

### 3-2. マルチパートアップロード（並列アップロード）

**引越しの荷物運びに例えると：**

```
❌ 1人で1箱ずつ運ぶ → 10箱 × 5分/箱 = 50分
✅ 10人で同時に運ぶ → 10箱 ÷ 10人 = 5分
```

**S3での動き：**

```
[アプリ]
  ├─ Part1（0〜1GB）  → S3 ─┐
  ├─ Part2（1〜2GB）  → S3 ─┤ 同時並行
  ├─ Part3（2〜3GB）  → S3 ─┤
  └─ Part4（3〜4GB）  → S3 ─┘
                              → S3が自動で結合して1ファイルに
```

**ポイントまとめ：**

| 項目 | 内容 |
|------|------|
| 推奨開始サイズ | **100MB以上** |
| 必須サイズ | **5GB以上**（これ以上は分割しないとアップロード不可） |
| 最大ファイルサイズ | 5TB |
| 最小パートサイズ | 5MB（最後のパートだけ例外） |
| パート数上限 | 最大10,000パート |

> 📝 **試験ポイント**：大ファイルのアップロードを速くしたい → **マルチパートアップロード（100MB以上推奨・5GB以上必須）**

---

### 3-3. Byte-Range Fetch（並列ダウンロード）

マルチパートアップロードの**ダウンロード版**。NetflixやYouTubeの動画ダウンロードと同じ発想。

```
[10GBファイル]
  ├─ 0〜2GB    → アプリへ ─┐
  ├─ 2〜4GB    → アプリへ ─┤ 同時並行
  ├─ 4〜6GB    → アプリへ ─┤
  └─ 6〜10GB   → アプリへ ─┘
```

**HTTPのRange指定（実際のリクエスト）：**

```http
GET /myfile.zip HTTP/1.1
Range: bytes=0-1073741823   ← 最初の1GBだけ取得
```

**応用：特定部分だけ取得してコスト節約**

```
s3://bucket/logs/huge-log.csv（50GB）
  → Range: bytes=0-1000 だけ取得して内容確認
  → 全部DLせずに済む！コスト節約にもなる
```

> 📝 **試験ポイント**：大ファイルの一部だけ取得したい → **Byte-Range Fetch**

## 4. S3 Select（オブジェクト内の必要部分だけ読む）

**S3 Select** は、CSV/JSON/Parquet などのS3オブジェクトに対してSQL式を実行し、必要な行・列だけを取り出す機能。アプリ側で巨大ファイルを丸ごとダウンロードしてから絞り込むより、転送量と処理量を減らせる。

> [!example] たとえ
> 分厚い台帳を全部コピーしてから探すのではなく、倉庫番に「2026年の東京支店の行だけ抜き出して」と頼んで、その部分だけ受け取る。

| 向いていること | 向いていないこと |
|---|---|
| 1つの大きなオブジェクトから必要な行/列だけ抽出 | 複数ファイルを横断する分析 |
| アプリのダウンロード量削減 | データレイク全体へのSQL分析 |
| 一部抽出によるレイテンシ/コスト削減 | 削除防止・保持・ライフサイクル管理 |

> [!tip] SAAひっかけ
> **S3 Select は「読み取り量を減らす」機能**。S3上の多数ファイルをSQL分析するなら [[1109_Amazon Athena]]、削除・上書きを防ぐなら [[0205_S3のバージョン管理|S3 Object Lock]]。

---

## 5. S3 Storage Lens（可視化・分析）

**S3 Storage Lens** は、複数バケット/アカウントにまたがるS3利用状況を可視化する分析機能。容量増加、ストレージクラス分布、古い未完了マルチパートアップロード、暗号化状況などをダッシュボードで確認できる。

| できること | できないこと |
|---|---|
| S3の利用傾向・コスト最適化ポイントを可視化 | オブジェクトを自動で安いクラスへ移動 |
| 組織/アカウント/リージョン単位で集計 | WORM保持や削除防止 |
| 推奨事項を見て改善計画を立てる | アプリのSQLクエリ実行 |

> [!tip] SAAひっかけ
> 「アクセス頻度が不明なので自動でコスト最適化」なら [[0203_S3ストレージクラス|S3 Intelligent-Tiering]]。
> 「利用状況を見える化して分析」なら **S3 Storage Lens**。

---

## 全体像

```
【アップロード側】
クライアント
  ├─ 小ファイル（<100MB）  → 普通のPUT
  └─ 大ファイル（>100MB）  → マルチパートアップロード（並列）
                                  ↓
                              S3（プレフィックス分散でスループット確保）

【ダウンロード側】
S3
  ├─ 小ファイル            → 普通のGET
  └─ 大ファイル            → Byte-Range Fetch（並列）
                                  ↓
                              クライアント
```

---

## 試験頻出ポイント まとめ

| 問われるシナリオ | 答え |
|----------------|------|
| 海外からのアップロードが遅い | **Transfer Acceleration** |
| 静的コンテンツを世界中に高速配信 | **CloudFront** |
| 大ファイル（5GB以上）のアップロード | **マルチパートアップロード**（必須） |
| アップロードを速くしたい（100MB以上） | **マルチパートアップロード**（推奨） |
| GETリクエストが多すぎてスローダウン | **プレフィックスを分散** |
| 大ファイルの一部だけ取得したい | **Byte-Range Fetch** |
| オブジェクト内の必要な行/列だけ取得したい | **S3 Select** |
| S3利用状況・コスト最適化ポイントを可視化したい | **S3 Storage Lens** |

---

> 💡 **核心のひとこと**：S3のパフォーマンス最適化は「距離を縮める（Transfer Acceleration / CloudFront）」と「仕事を分担して同時にこなす（マルチパート / Byte-Range Fetch / プレフィックス分散）」の2つに集約される。
