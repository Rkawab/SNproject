# S3 バケットポリシー

> **一言で言うと：** 「このバケットに、誰が、何をしていいか・ダメか」をJSONで書いたルール

---

## 🗂️ そもそも何のためにあるの？

S3バケットはデフォルトで**全部非公開**。  
アクセスを許可・拒否するには「ポリシー」が必要。

ポリシーには2種類ある：

| | バケットポリシー | IAMポリシー |
|---|---|---|
| どこに設定する？ | **S3バケット側** | IAMユーザー/ロール側 |
| 誰に適用できる？ | 誰でも（他アカウント・匿名も可） | AWSアカウント内のユーザーのみ |
| 何ができる？ | 「このバケットへのアクセス」を制御 | 「このユーザーの権限」を制御 |

💡 **アナロジー：**  
- バケットポリシー ＝ **建物の入口に貼った入館ルール**（誰でも読める）  
- IAMポリシー ＝ **社員証に書かれた権限**（その人専用）

---

## 📝 ポリシーの書き方（基本構造）

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "名前（任意）",
      "Effect": "Allow または Deny",
      "Principal": "誰に？",
      "Action": "何を？",
      "Resource": "どのバケット/オブジェクトに？"
    }
  ]
}
```

### 各項目の意味

| 項目 | 意味 | 例 |
|---|---|---|
| `Effect` | 許可 or 拒否 | `"Allow"` / `"Deny"` |
| `Principal` | 誰に適用するか | `"*"`（全員）、特定のAWSアカウント |
| `Action` | 何の操作を | `"s3:GetObject"`（読む）、`"s3:PutObject"`（書く） |
| `Resource` | どのバケット/オブジェクト | `"arn:aws:s3:::my-bucket/*"` |
| `Condition` | 条件（任意） | 特定のIPからのみ、など |

---

## 🔑 よく使うActionまとめ

| Action | 意味 |
|---|---|
| `s3:GetObject` | オブジェクトを読む（ダウンロード） |
| `s3:PutObject` | オブジェクトを書く（アップロード） |
| `s3:DeleteObject` | オブジェクトを削除する |
| `s3:ListBucket` | バケット内のファイル一覧を見る |
| `s3:*` | S3の全操作 |

---

## 💡 よくある使用パターン

### ① 静的ウェブサイト公開（全員に読み取り許可）

```json
{
  "Effect": "Allow",
  "Principal": "*",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-website-bucket/*"
}
```

→ `Principal: "*"` ＝ インターネット上の誰でも読める

---

### ② 別のAWSアカウントに許可（クロスアカウント）

```json
{
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:root"
  },
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-bucket/*"
}
```

→ IAMポリシーだけでは他アカウントへの許可は不可。バケットポリシーが必要。

---

### ③ 特定IPからのみアクセス許可

```json
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": "arn:aws:s3:::my-bucket/*",
  "Condition": {
    "NotIpAddress": {
      "aws:SourceIp": "203.0.113.0/24"
    }
  }
}
```

→ 指定IP以外は全拒否

---

### ④ CloudFrontからのみアクセス許可（OAC）

```json
{
  "Effect": "Allow",
  "Principal": {
    "Service": "cloudfront.amazonaws.com"
  },
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-bucket/*",
  "Condition": {
    "StringEquals": {
      "AWS:SourceArn": "arn:aws:cloudfront::アカウントID:distribution/ディストリビューションID"
    }
  }
}
```

→ CloudFront経由でのみアクセス可能にするセキュアな構成

---

## ⚠️ 優先順位のルール（SAA頻出！）

```
① パブリックアクセスブロック（最優先）
    ↓ ブロックしていなければ
② Denyポリシーがあれば → 拒否（他の設定を上書き）
    ↓ Denyがなければ
③ Allowポリシーがあれば → 許可
    ↓ どちらもなければ
④ デフォルト → 拒否
```

💡 **覚え方：** `Deny` は `Allow` より常に強い。ただし「パブリックアクセスブロック」は最強。

---

## 🔗 クロスアカウントアクセスの注意点

他アカウントのユーザーがS3にアクセスする場合、**両方のポリシーが必要**：

```
バケットポリシー（バケット側）：他アカウントに許可
    +
IAMポリシー（ユーザー側）：そのS3への操作を許可
```

→ どちらか一方だけでは動かない！

---

## 📌 SAA試験チェックリスト

- [ ] `Principal: "*"` は匿名アクセス（パブリック公開）を意味する
- [ ] `Deny` は `Allow` より優先される
- [ ] パブリックアクセスブロックは全ポリシーより優先される
- [ ] クロスアカウントアクセスはバケットポリシー＋IAMポリシーの両方が必要
- [ ] CloudFront経由のみにする場合はOAC（Origin Access Control）を使う
- [ ] `s3:ListBucket` はバケット自体に、`s3:GetObject` はオブジェクト（`/*`）に設定

---

## 🧠 混同しやすいポイント

| よくある間違い | 正しい理解 |
|---|---|
| IAMポリシーだけで他アカウントに許可できる | ❌ バケットポリシーも必要 |
| `s3:ListBucket` を `/*` に設定する | ❌ バケット本体（`arn:aws:s3:::my-bucket`）に設定 |
| `Deny` と `Allow` が両方あると `Allow` が勝つ | ❌ `Deny` が勝つ |
| パブリックアクセスブロックをオフにすれば公開できる | △ バケットポリシーも必要 |
