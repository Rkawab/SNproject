# S3 暗号化

## 概要

S3の暗号化には2種類ある。

- **保存時の暗号化（SSE: Server-Side Encryption）**：S3に保存されるデータを暗号化
- **転送時の暗号化**：S3へのデータ送受信を暗号化（HTTPS/TLS）

---

## 保存時の暗号化（SSE）

「**誰が鍵を管理するか**」によって3種類に分かれる。

|種類|鍵の管理者|追加料金|特徴|
|---|---|---|---|
|SSE-S3|AWS（S3）|なし|最もシンプル。デフォルト有効|
|SSE-KMS|AWS KMS|あり（KMS料金）|CloudTrailで監査可能。コンプライアンス向け|
|SSE-C|顧客（自前）|なし|AWSは鍵を保存しない。CLI/SDKのみ対応|

### SSE-S3

- AWSが鍵を完全自動管理
- AES-256で暗号化
- **新規バケットにデフォルト適用**
- 追加料金なし・設定が最もシンプル

### SSE-KMS

- AWS KMS（Key Management Service）で鍵を管理
- 誰がいつ鍵を使ったか **CloudTrail で監査**できる
- 独自の鍵（CMK: Customer Managed Key）を作成・持ち込みも可能
- ⚠️ 大量リクエスト時に **KMS の API コールがボトルネック**になることがある（スロットリング）

### SSE-C

- 鍵は自前で用意し、リクエスト毎に送付
- AWSは鍵を保存しない（使用後に破棄）
- **HTTPS 必須**
- **コンソール不可 → CLI / SDK のみ使用可能**

---

## 転送時の暗号化

- **HTTPS（TLS）** を使うことで実現
- バケットポリシーで `aws:SecureTransport: false` を **Deny** することで HTTP を強制拒否できる

```json
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": "arn:aws:s3:::my-bucket/*",
  "Condition": {
    "Bool": { "aws:SecureTransport": "false" }
  }
}
```

---

## 試験頻出ポイント

- **デフォルト暗号化**：現在は全バケットで SSE-S3 が自動適用される
- **監査が必要 → SSE-KMS**（CloudTrail とセット）
- **自前鍵を AWS に渡したくない → SSE-C**
- **SSE-C はコンソール不可**（CLI / SDK のみ）
- **コンプライアンス要件が厳しい → SSE-KMS or SSE-C**
- **KMS スロットリング**：高頻度アクセス時は SSE-S3 の方が有利な場合あり

---

## アナロジー

| SSE種類   | 貸し金庫で例えると          |
| ------- | ------------------ |
| SSE-S3  | 金庫屋が鍵も全部管理してくれる    |
| SSE-KMS | 金庫屋が鍵を預かるが、使用記録が残る |
| SSE-C   | 自分の鍵だけ持参。金庫屋は複製しない |
