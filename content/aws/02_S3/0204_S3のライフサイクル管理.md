# S3 ライフサイクル管理

オブジェクトを**時間とともに自動的に別のストレージクラスへ移行したり削除したりする仕組み**。

![[Pasted image 20260402201244.png]]

## 2種類のアクション

| アクション | 内容 |
|---|---|
| **Transition（移行）** | 指定日数後に別のストレージクラスへ自動移行 |
| **Expiration（有効期限）** | 指定日数後にオブジェクトを自動削除（バージョニングON時は「削除マーカー付け」） |

## よく使われる設定例

- ログファイル → 30日後にStandard-IA → 90日後にGlacier → 365日後に削除
- バックアップ → 即日Glacier Deep Archive、7年後に削除
- 現行バージョン以外の古いバージョン → 30日後に削除

ストレージクラスの詳細 → [[0203_S3ストレージクラス]]

## SAA試験ポイント

| ポイント | 内容 |
|---|---|
| 移行の最低期間 | Standard-IA へ移行するには**最低30日間Standardに置く**必要がある |
| 小さいオブジェクト | **128KB未満のオブジェクトはIAへ移行されない** |
| バージョニングとの関係 | バージョニング有効時にExpireすると、消えずに**削除マーカーが付く**（完全削除には「非現行バージョンの削除」設定が別途必要）→ [[0205_S3のバージョン管理]] |
| Intelligent-Tiering | アクセスパターンが予測できないときの**自動**振り分け機能で、ライフサイクル（日数ベースの固定ルール）とは別の仕組み |
| 未完了マルチパート | **未完了パーツも課金対象** → ライフサイクルで自動削除がベストプラクティス |

> [!important] 頻出の流れ
> 「Standard → IA → Glacier → 削除」というコスト最適化の流れは試験頻出。**日数の下限（IA系30日・Glacier系90日/180日の最低保存期間）**とセットで覚える。

## マルチパートアップロードとライフサイクルの関係

### マルチパートアップロード（Multipart Upload）とは

大きいファイルを「小分け」にして**並行**アップロードし、S3で自動的に1つに組み立てる仕組み。

**たとえ話：大型家具の配送**
```
❌ タンス（大きいファイル）を丸ごと1便で送る
   → 途中でトラックが故障したら、最初からやり直し

✅ 分解して複数の箱（パート）にする
   1. 分解して複数の箱にする
   2. 何台ものトラックで同時に運ぶ（並行＝速い）
   3. 1箱だけ落ちても、その箱だけ再送すればいい
   4. 到着先（S3）で組み立てて元のタンスに戻す
```

### 仕組み（3ステップ）

| ステップ | 動作 | 内容 |
|---------|------|------|
| **1. Initiate（開始）** | 宣言 | 「これから分割アップロードします」とS3に宣言、IDをもらう |
| **2. Upload Parts（パート送信）** | 並行送信 | ファイルを複数パートに分けて送る（失敗パートだけ再送OK） |
| **3. Complete（完了）** | 確定 | 「全部送った」と通知 → S3がパートを連結して1つのオブジェクトにする |

### ポイント

| 項目 | 数値 |
|-----|------|
| 推奨開始サイズ | **100MB以上** |
| 必須サイズ | **5GB以上**（これ以上は分割しないとアップロード不可） |
| 最大ファイルサイズ | 5TB |
| 最小パートサイズ | 5MB（最後のパートだけ例外） |
| パート数上限 | 最大10,000パート |

---

### ⚠️ 「不完全なマルチパートアップロード」がコストの罠

**ステップ2まで送ってステップ3（完了）をしないまま放置すると、送ったパートがS3に残ったまま課金され続けます。**

> [!danger] なぜ気づきにくいのか？
> - パーツは「マルチパートアップロード中」という状態になる
> - **通常のオブジェクト一覧には表示されない**
> - だから削除し忘れたパーツが、知らずに課金され続けている

### 解決策：ライフサイクルで自動掃除

```xml
<LifecycleConfiguration>
  <Rule>
    <Id>AbortIncompleteMultipartUpload</Id>
    <AbortIncompleteMultipartUpload>
      <DaysAfterInitiation>7</DaysAfterInitiation>
    </AbortIncompleteMultipartUpload>
    <Status>Enabled</Status>
  </Rule>
</LifecycleConfiguration>
```

**意味**：不完全なマルチパートアップロード（開始後7日経過）を自動削除して、課金を防ぐ。

### SAA試験ポイント

> [!info] 「マルチパートアップロード使用」という文が問題に出たら
> **「不完全なマルチパートアップロードを削除するライフサイクル」** はほぼセットで正解になる。
>
> 典型的な問題：「S3でマルチパートアップロードを使っているが、コストが想定より高い」
> → 答え：「不完全なマルチパートアップロードを削除するライフサイクルルールを設定する」
