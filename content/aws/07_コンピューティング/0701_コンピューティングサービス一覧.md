# AWSコンピューティングサービス一覧

## サービス選択の全体像

```
どこでコードを動かすか？
│
├── 自分でサーバー管理したい      → EC2（仮想マシン）
├── コンテナで動かしたい          → ECS / EKS / Fargate
├── コードだけ書いてあとはお任せ  → Lambda（サーバーレス）
├── バッチ処理（大量データ処理）  → AWS Batch
├── デプロイだけ楽にしたい        → Elastic Beanstalk
├── オンプレに近い感覚で使いたい  → AWS Outposts
└── 小規模・シンプルに使いたい    → Lightsail
```

---

## EC2（Elastic Compute Cloud）

仮想サーバーをレンタルするサービス。「AWSの基本」。
OSの選択・ネットワーク設定・スケーリングまで自分で制御できる反面、管理コストもかかる。

> 関連: [[0101_EC2やVPCとは]]

---

## Lambda（サーバーレス）

サーバーなしでコードだけ動かせるサービス。イベント駆動で動作する。
実行時間だけ課金され、アイドル時はゼロコスト。ただし最大15分の実行制限あり。

> たとえ話: 「レストランを借りる（EC2）」ではなく「料理人だけ呼ぶ（Lambda）」
> 詳細: [[0708_Lambda]]

---

## ECS / EKS / Fargate（コンテナ）

| サービス | 特徴 |
|----------|------|
| **ECS** | AWSが独自で提供するコンテナ管理 |
| **EKS** | Kubernetes（業界標準）でコンテナ管理 |
| **Fargate** | ECS/EKSのサーバー管理をAWSに任せるモード |

## AWS Batch

> 詳細: [[0703_AWS_Batch]]

### 概要

大量のバッチジョブを自動で並列処理するサービス。

> たとえ話: 「1万枚の画像をリサイズしたい」とき、1枚ずつ処理するより、100台のサーバーで100枚ずつ同時に処理したほうが速い。その「振り分け・管理」を自動でやってくれる。

### 構成要素

| 要素 | 役割 |
|------|------|
| **ジョブ定義** | 何をするか（Dockerイメージ・CPU・メモリ等） |
| **ジョブキュー** | ジョブを溜めておく待ち行列 |
| **コンピューティング環境** | 実際に動かすEC2またはFargate |

### SAA試験ポイント

- Lambdaの15分制限を超える**長時間バッチ処理**に使う
- **EC2スポットインスタンスと組み合わせてコスト削減**が定番パターン
- コンテナベース（Dockerイメージを使う）

### 向いているケース

```
✅ 機械学習モデルの学習
✅ 遺伝子解析・科学計算
✅ 動画変換・画像一括処理
✅ 月次の大規模データ集計
```

## AWS Elastic Beanstalk

### 概要

コードをアップロードするだけでWebアプリを動かしてくれるPaaSサービス。
EC2・ELB・Auto Scaling等のインフラ設定をBeanstalkが全自動で行う。

> たとえ話: 「引っ越し業者に段ボールを渡したら、家具の組み立てから電気の接続まで全部やってくれた」

### 裏側で自動作成されるもの

```
Elastic Beanstalk
│
├── EC2インスタンス（アプリサーバー）
├── Elastic Load Balancer
├── Auto Scaling Group
├── RDS（オプション）
└── CloudWatch（モニタリング）
```

### 対応言語・環境

Java / .NET / PHP / Node.js / Python / Ruby / Go / Docker

### SAA試験ポイント

- **PaaS（Platform as a Service）** の典型例
- Beanstalk自体への課金はなし（使ったEC2等は課金される）
- 裏のリソース（EC2等）は自分のアカウントに作られるので細かい設定も可能
- **Blue/Greenデプロイ**が得意

### 向いているケース

```
✅ 開発者がインフラを学ばずにWebアプリを公開したい
✅ スタートアップや小規模チームの素早いデプロイ
✅ 既存のWebアプリをAWSに移行したい
```

## AWS Outposts

> 詳細: [[0705_AWS_Outposts]]

### 概要

AWSのインフラをそのまま自社のデータセンターに設置するサービス。
オンプレミス環境でAWSのサービスを利用できる。

> たとえ話: 「AWSという巨大なスーパーのレジを、自分の家に持ってきてもらう」

### なぜ必要なのか

| 課題 | 具体例 |
|------|--------|
| **レイテンシ問題** | クラウドまでの通信が遅すぎる（工場の制御システム等） |
| **データ主権** | 法律でデータを国外に出せない |
| **オフライン環境** | インターネットが不安定・つながらない場所 |

### 利用可能なサービス

EC2・EBS・RDS・ECS・EKS などをオンプレで動かせる。
AWS Consoleから一元管理（クラウドと同じ操作感）。

### 種類

| 種類 | 概要 |
|------|------|
| **Outposts Rack** | ラックごとレンタル。大規模向け |
| **Outposts Servers** | サーバー単位でレンタル。スペースが限られる場合 |

### SAA試験ポイント

- **ハイブリッドクラウドの実現手段**として出題される
- 「低レイテンシが必要」「データをオンプレに置きたい」「規制で国外送信不可」→ **Outpostsが答え**
- ハードウェアはAWS所有・AWSが設置・管理する

### 向いているケース

```
✅ 製造業の工場（IoTデータをその場で処理）
✅ 病院（患者データを外に出せない）
✅ 金融機関（国内データ保持の義務）
✅ 通信の基地局（低レイテンシ必須）
```

## AWS Serverless Application Repository (SAR)

> 詳細: [[0704_Serverless_Application_Repository]]

### 概要

サーバーレスアプリの「App Store」。
Lambda・API Gateway・DynamoDBなどを組み合わせた構成セットをワンクリックでデプロイできるマネージドリポジトリ。

> たとえ話: スマートフォンの App Store と同じ。他の人が作ったアプリ（サーバーレス構成）を簡単にインストール（デプロイ）できる。

### SAA試験ポイント

- 内部は **SAM（CloudFormationを拡張したサーバーレス用IaC）テンプレート**
- **公開・非公開・組織内共有（Organizations）** の3段階の公開範囲設定が可能
- 単一Lambdaではなく**複数サービスの組み合わせをまとめてデプロイ**できる点が特徴
- **再利用・標準化**のユースケースで登場する

## その他のコンピューティングサービス

| サービス | 特徴 | 向いているケース |
|----------|------|----------|
| **Lightsail** | EC2を超シンプルにした版。固定料金 | 個人ブログ・小規模サイト |
| **App Runner** | コードを渡すだけでコンテナアプリ動作 | Webアプリの素早いデプロイ |
| **Wavelength** | 5G基地局内にAWS環境を設置。[[0706_AWS_Wavelength]] | モバイル向け超低遅延 |
| **Local Zones** | 特定都市にAWSのリソースを拡張 | その都市近くで低レイテンシ |

## SAA試験での選び方まとめ

>[!info] 状況別サービス選択
> | 状況 | 答え |
> |------|------|
> | 長時間のバッチ処理・並列処理 | **AWS Batch** |
> | コードだけ渡してアプリ公開したい | **Elastic Beanstalk** |
> | オンプレ環境にAWSが欲しい | **AWS Outposts** |
> | イベント駆動の短時間処理 | **Lambda** |
> | サーバーレスアプリを共有・再利用したい | **SAR** |
> | コンテナをマネージドで使いたい | **ECS/EKS + Fargate** |
> | シンプルな仮想サーバーが欲しい | **EC2** |
> | 5G・超低遅延・エッジ処理 | **Wavelength** |
