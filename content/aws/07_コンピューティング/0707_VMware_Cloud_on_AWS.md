# VMware Cloud on AWS

## 前提知識：VMwareとは

### サーバー仮想化が生まれた背景

昔の企業ITは、アプリごとに専用の物理サーバーを1台用意していた。

```
メールサーバー用マシン   → 1台
Webサーバー用マシン     → 1台
DBサーバー用マシン      → 1台
社内システム用マシン     → 1台  ...
```

各サーバーのCPUは20〜30%しか使っていないのに、電気代・設置スペース・管理コストだけかかる状態だった。

### VMwareの解決策：1台に複数の仮想マシンを詰め込む

> [!info] たとえ話：マンション
> - 物理サーバー1台 ＝ 土地
> - VMwareのソフト ＝ マンションの建設業者
> - 仮想マシン（VM） ＝ 各部屋（それぞれ独立した生活空間）

VMwareのソフトウェアを使うと、1台の物理サーバーの中に **複数の仮想的なコンピューター（仮想マシン＝VM）** を作れる。各VMは完全に独立しており、別々のOSやアプリが動く。

```
【物理サーバー 1台】
┌──────────────────────────────────┐
│          VMwareのソフト           │
│  ┌────────┐ ┌────────┐ ┌──────┐ │
│  │メール  │ │ Web   │ │  DB  │ │  ← 各部屋が「仮想マシン」
│  │サーバー│ │サーバー│ │      │ │
│  └────────┘ └────────┘ └──────┘ │
└──────────────────────────────────┘
```

### VMwareとは

| 項目 | 内容 |
|------|------|
| VMware | 1998年創業のアメリカのソフトウェア会社（現在はBroadcomが買収） |
| 主力製品 | vSphere（仮想化ソフトのシリーズ名） |
| 立ち位置 | 企業向けサーバー仮想化の分野で長年シェアNo.1 |
| 使っている場所 | 大企業・官公庁・銀行などのオンプレミスのデータセンター |

### なぜAWS移行が難しいか

大企業のデータセンターでは、VMware上で何百〜何千台もの仮想マシンが稼働しており、10年以上かけて積み上げた資産になっている。これをAWSネイティブに作り直すには莫大なコスト・時間・リスクがかかる。

そこでVMwareとAWSが組んで作ったのが **VMware Cloud on AWS**。

---

## たとえ話：「自社工場をそのままAWSの敷地内に移転する」

会社が長年使ってきた自社工場（オンプレミスのVMware環境）を、そのまま丸ごとAWSの広大な土地（AWS上の専用ハードウェア）に持ち込んで動かすイメージ。

工場の機械（VMwareのソフトウェア）はそのまま。場所だけAWSのインフラに移る。
従来のやり方を変えずに、AWSのサービス（RDS・S3・Lambda等）も隣で使えるようになる。

---

## 基本概要

| 項目 | 内容 |
|------|------|
| 正式名称 | VMware Cloud on AWS（VMC on AWS） |
| 提供形態 | VMwareとAWSの共同開発サービス |
| 基盤 | AWSのベアメタルEC2（`i3.metal` 等）を専用利用 |
| 管理 | VMwareが管理（ユーザーはVMware Consoleから操作） |
| 目的 | オンプレのVMware環境をそのままAWSへ移行 |

---

## なぜ使うのか（ユースケース）

### 1. リフト＆シフト移行（ほぼそのまま移行）
VMware環境を **アプリの改修なし** でクラウドに移せる。

> 工場の機械を解体してゼロから組み直すのではなく、「工場ごとトラックで運ぶ」イメージ。

### 2. データセンター廃止・縮小
自社DCのハードウェアリース切れのタイミングで、更新せずにクラウドに乗り換える。

### 3. 災害対策（DR）
オンプレ環境のバックアップ先・フェイルオーバー先としてVMC on AWSを使う。

### 4. AWSネイティブサービスとの連携
VMware上のVMから **RDS・S3・Lambda** 等を低レイテンシで呼び出せる（同一リージョン内）。

## 構成イメージ

```
オンプレミス
┌────────────────────┐
│  VMware vSphere    │ ←── 既存のサーバー仮想化環境
│  （vCenter管理）   │
└─────────┬──────────┘
          │ HCX（ライブマイグレーション）で接続
          ▼
AWS クラウド
┌──────────────────────────────────────────┐
│   VMware Cloud on AWS（専用ベアメタル）  │
│   ┌──────────┐  ┌──────────┐            │
│   │  VM (A)  │  │  VM (B)  │   VMware   │
│   └──────────┘  └──────────┘   SDDCスタック│
│                                          │
│   ← 同じVPCでAWSサービスと連携可能 →    │
│   ┌──────┐ ┌────┐ ┌──────────┐          │
│   │  S3  │ │RDS │ │ Lambda  │          │
└──────────────────────────────────────────┘
```

## キーワード

| 用語 | 意味 |
|------|------|
| **SDDC** | ソフトウェア定義データセンター。VMwareの仮想化基盤一式（vSphere・NSX・vSAN） |
| **HCX** | VMwareのライブマイグレーションツール。稼働中のVMを止めずにAWSへ移行できる |
| **ベアメタルEC2** | OSなしの物理サーバー。VMwareがOS・ハイパーバイザーを直接管理するために必要 |
| **vCenter** | VMwareのVM管理コンソール。VMC on AWSでもこれを使って操作する |

---

## SAA試験ポイント

> [!info] 試験での出題パターン
> 「既存のVMware環境を変更せずにAWSへ移行したい」→ **VMware Cloud on AWS** が正解。

- VMware環境の **リフト＆シフト移行** の手段として登場する
- AWSが直接管理するのではなく、**VMwareが管理** する点が独自
- ベアメタルEC2を使うため、通常のEC2より **高コスト・高性能**
- オンプレとのネットワーク接続は **AWS Direct Connect** or **VPN** を使う
- **HCX** を使うことで無停止マイグレーション（ライブマイグレーション）が可能
