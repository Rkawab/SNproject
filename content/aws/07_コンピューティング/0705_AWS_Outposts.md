# AWS Outposts

> 関連: [[0701_コンピューティングサービス一覧]]

---

## 概要

AWSのインフラ（ハードウェア・サービス）をそのまま**自社データセンターや施設に設置**するサービス。
クラウドの運用・管理体験はそのままに、物理的なハードウェアだけオンプレミスに置ける。

> **たとえ話**: AWSというクラウドの「本店」から、ラックごとお店を「のれん分け」してもらい、自社のデータセンターに置く感覚。営業権・店員・オペレーションはすべてAWS流儀のまま、でも物理的には自社の建物の中にある。

---

## なぜ必要なのか（3大ユースケース）

| 課題 | 詳細 | 具体例 |
|------|------|--------|
| **超低レイテンシ** | AWSリージョンまでの往復が遅すぎる | 工場の機械制御（1ms以下が必要）、自動運転、医療機器 |
| **データ主権・コンプライアンス** | 法律・規制でデータを国外・社外に出せない | 金融業（銀行法）、医療（HIPAA）、政府機関 |
| **オフライン・不安定環境** | インターネット接続が保証できない | 海底油田、艦船、山間部の施設 |

## 仕組み（アーキテクチャ）

```
自社データセンター                    AWSクラウド（例: ap-northeast-1）
┌──────────────────────┐            ┌────────────────────────┐
│  AWS Outposts        │            │                        │
│  ┌──────────────┐   │            │  ・コントロールプレーン │
│  │ EC2インスタ  │   │            │  ・APIエンドポイント   │
│  │ EBS ボリューム│  │ ←Service→  │  ・管理コンソール      │
│  │ S3バケット   │   │    Link    │  ・CloudWatch          │
│  └──────────────┘   │            │                        │
│         ↑           │            └────────────────────────┘
│  ローカルゲートウェイ│
│         ↓           │
│  オンプレネットワーク│
└──────────────────────┘
```

### 重要な2つの接続経路

| 接続 | 役割 | 経由 |
|------|------|------|
| **Service Link（サービスリンク）** | OutpostsとAWSリージョン間の制御プレーン（管理・監視・API） | Direct Connect または VPN |
| **ローカルゲートウェイ（LGW）** | OutpostsとオンプレLAN間のデータ通信 | 自社ネットワーク |

> [!info] インターネット切断時の挙動
> Service Link（リージョン接続）が切れると、コントロールプレーン機能は停止するが、**既に起動中のインスタンスはローカルで稼働を継続できる**。

## VPCとの関係

Outpostsは**既存VPCのサブネットを自社施設に"延伸"**するイメージ。

```
VPC（10.0.0.0/16）
├── ap-northeast-1a のサブネット（10.0.1.0/24） ← AWSクラウド上
├── ap-northeast-1c のサブネット（10.0.2.0/24） ← AWSクラウド上
└── Outposts サブネット（10.0.3.0/24）          ← 自社データセンター内
```

AWSコンソールから見ると、Outpostsのサブネットも同じVPCの一部として見える。
通常のAWSリソースと同じようにセキュリティグループ・ルートテーブルで制御できる。

## 2種類の形態

| | **Outposts Rack** | **Outposts Servers** |
|---|---|---|
| 形状 | 標準42Uラック | 1U または 2U サーバー |
| 規模 | 大規模（本格的なデータセンター向け） | 小規模（スペース制約あり） |
| 電力 | 約10kW | 1.5〜2kW |
| 重量 | 約2000kg | 数十kg |
| 設置場所 | 本格的なデータセンター | 工場フロア、支店、医療現場 |
| ネットワーク推奨 | 25Gbps以上 | 1Gbps以上 |
| 利用可能サービス | フルセット | EC2・ECS・EBS（一部） |

> **たとえ話（Rack）**: 銀行の本格的な金庫室ごと家に運んでくる
> **たとえ話（Servers）**: キャッシュレジスター1台だけ持ってくる

## 利用可能なAWSサービス（Outposts Rack）

| カテゴリ | サービス |
|---------|---------|
| コンピューティング | EC2（M・C・R・G・I・D系インスタンス） |
| ストレージ | EBS（gp2）、**S3 on Outposts** |
| データベース | RDS（MySQL・PostgreSQL）、ElastiCache |
| コンテナ | ECS、EKS |
| ネットワーク | ALB（Application Load Balancer） |

### S3 on Outposts の特徴

- 通常のS3とは**別物**。APIは同じだがデータはOutposts内に留まる
- ARNに `outposts` が含まれる（例: `arn:aws:s3-outposts:ap-northeast-1:123456789012:outpost/op-01234567890abcdef/bucket/my-bucket`）
- エンドポイント例: `s3-outposts.ap-northeast-1.amazonaws.com`
- **ストレージは物理ラック内のEBSから確保される**ため容量に上限あり

## 他のエッジ/ハイブリッドサービスとの比較

> [!info] 比較表
>
> | サービス | 設置場所 | 目的 | ハードウェア所有者 | オフライン対応 |
> |---------|---------|------|-----------------|-------------|
> | **Outposts** | 自社データセンター・施設 | ハイブリッド運用・規制対応 | AWS（顧客施設に設置） | 一部可 |
> | **Local Zones** | AWSが選定した都市の施設 | 特定都市への低レイテンシ提供 | AWS（AWS管理） | 不可 |
> | **Wavelength** | 通信キャリアの5G基地局内 | 5G/モバイル向け超低レイテンシ | AWS（AWS管理） | 不可 |
> | **Snow Family** | 顧客が持ち運ぶ（一時的） | オフライン処理・大容量データ転送 | AWSからレンタル | 可（Snowball Edge） |

### 判断の鍵

```
「自社ビル・データセンター内に設置したい」 → Outposts
「大阪・福岡など国内の特定都市で低レイテンシ」 → Local Zones
「5Gスマホアプリの超低遅延」 → Wavelength
「ネットが全くない砂漠・海の上でデータ処理」 → Snow Family（Snowball Edge）
```

## 価格モデル

- **ハードウェアレンタル料** + **使用したAWSサービスの料金**の2段構え
- ラックは **1年または3年のキャパシティ予約**で契約（EC2 Reserved Instanceに近い概念）
- AWSエンジニアが設置・メンテナンスを担当するコストも込み
- **Outposts → AWSリージョンへのデータ転送**は課金対象

## SAA試験ポイント

> [!info] 出題パターン別まとめ
>
> | パターン | キーワード | 答え |
> |---------|-----------|------|
> | ハイブリッドクラウド | 「オンプレとAWSを一元管理したい」 | Outposts |
> | 規制・データ主権 | 「データを国内・自社施設から出せない」「金融・医療の規制」 | Outposts |
> | 超低レイテンシ + ローカル処理 | 「クラウドまでの遅延が許容できない」「オフラインでも動作必須」 | Outposts |
> | Local Zones との選択 | 設置場所が「自社施設」か「AWS管理施設」かで判断 | 自社→Outposts |

### 細かい試験知識

- ハードウェアの所有者は **AWS**（顧客が購入するわけではない）
- AWSコンソールから通常リージョンと **同じ操作感で管理可能**
- ネットワーク接続は Direct Connect **推奨**（VPNも可）
- インターネット切断時も、**起動済みインスタンスはローカルで稼働継続**
- Outpostsのサブネットは**既存VPCの延伸**として機能する
