# Lambda・ECS・RDS・Beanstalk 比較：マイクロサービスとの適性

## 学習背景
AWS Solutions Architect Associate (SAA) 試験対策における、マイクロサービスアーキテクチャに最適なAWSサービス選択の学習。

---

## ユースケース
- マイクロサービスアーキテクチャの実装
- ピーク時 400 req/s の自動スケール対応
- 運用の手間を最小化
- Python 対応

---

## 正解：**AWS Lambda（サーバーレス）**

### Lambda が最適な理由
| 項目 | Lambda の対応 |
|---|---|
| **サーバー管理** | 完全に不要 ✅ |
| **自動スケーリング** | リクエスト数に応じて自動スケール（関数単位） ✅ |
| **運用コスト** | 最小限。デプロイと監視のみ ✅ |
| **Python 対応** | ネイティブサポート ✅ |
| **マイクロサービス** | イベント駆動で疎結合を実現 ✅ |

### Lambda の特徴
- **イベント駆動**: API Gateway、SQS、SNS 等からトリガー
- **スケーリング粒度**: 関数単位で細かく制御可能
- **課金モデル**: 実行時間と実行回数のみ（サーバーを立てていない時間は無料）
- **デプロイ**: コードのアップロードのみで即座に反映

---

## 不正解：他の3つのサービス

### ① Amazon RDS（❌ 完全に間違い）

**問題点**
- **役割の違い**: RDS はデータベースサービス。スケーリングやデプロイの要件に答えない
- **スケーリング・デプロイ**: マイクロサービスを実装するコンピューティング層ではない
- **説明の誤り**: 「全データを一元管理」という説明は、**マイクロサービス思想（分散・疎結合）と真逆**
  - マイクロサービス = 各サービスが独立したデータストアを持つ（データの分散化）
  - RDS の説明 = 全データ一元化（モノリシック思想）

---

### ② Amazon ECS（⚠️ 間違いではないが「最適」ではない）

**できること**
- ✅ コンテナオーケストレーション
- ✅ 自動スケーリング（タスク単位）
- ✅ マイクロサービス実装可能

**Lambda より劣る理由**
| 項目 | ECS | Lambda |
|---|---|---|
| **サーバー管理** | EC2 インスタンス管理が必要 | 不要 |
| **デプロイ手順** | コンテナイメージのビルド・プッシュ・タスク定義更新 | コードアップロードのみ |
| **運用コスト** | 高（クラスター・タスク定義の管理） | 低 |
| **スケーリング粒度** | タスク単位（粗い） | 関数単位（細か） |

**ECS が向いている場合**
- コンテナ化が既に完了している
- 複雑な分散処理が必要
- ステートフルなサービス

---

### ③ AWS Elastic Beanstalk（❌ 最適ではない）

**できること**
- ✅ Web アプリのデプロイ自動化
- ✅ 自動スケーリング
- ✅ Python サポート

**問題点**
| 項目 | Beanstalk | Lambda |
|---|---|---|
| **サーバー管理** | 裏側は EC2 なので管理が発生 | 不要 |
| **スケーリング粒度** | インスタンス単位（粗い） | 関数単位（細か） |
| **アーキテクチャ向き** | モノリシック Web アプリ向け | マイクロサービス・イベント駆動向け |
| **API ベース** | ウェブサーバー（常時起動） | イベント駆動（呼び出し時のみ実行） |

**Beanstalk が向いている場合**
- 既存の Django/Flask アプリをすぐに EC2 で動かしたい
- インスタンス管理を最小化したいが、完全サーバーレスはいらない
- 単一の Web アプリケーション

## SAA 試験の判断フロー

```
要件確認
  │
  ├─ マイクロサービス ＆ イベント駆動 ＆ 自動スケール ＆ サーバー管理なし
  │  → ★ Lambda ★
  │
  ├─ コンテナが既にある ＆ 複雑な分散処理が必要
  │  → ECS / EKS
  │
  ├─ Web アプリをすぐデプロイしたい（EC2 管理は最小限）
  │  → Beanstalk
  │
  └─ データベースが主役
     → RDS / DynamoDB
```

---

## 重要な用語説明

### マイクロサービスアーキテクチャ
- 大きなアプリを小さな独立サービスに分割
- 各サービスが独立したデータストアを持つ（データの疎結合）
- サービス間通信: API / メッセージキュー（SQS・SNS）
- **各サービスの起動・停止が独立** → Lambda に最適

### イベント駆動アーキテクチャ
- サービスが「イベント」に反応して動作
- 常にサーバーを起動しておく必要がない
- API Gateway → Lambda、SQS → Lambda など
- **サーバーレスコンピューティングの基本思想**

### スケーリング粒度
- **関数単位**（Lambda）: 400 req/s を 400 個の関数で処理 → 細かく対応
- **インスタンス単位**（EC2・Beanstalk）: インスタンスを 2 個に増やす → 粗い
