# MySQLとPostgreSQLの違い

## 一言で言うと

| | MySQL | PostgreSQL |
|---|---|---|
| イメージ | フォルダにExcelファイルが並んでる | 番号管理された図書館の書庫 |
| 設計思想 | 速さ・シンプルさ重視 | 標準準拠・高機能重視 |

---

## 直感的なイメージ（専門用語なし）

### MySQL：フォルダにExcelファイルが並んでる

テーブルごとにファイルが一つ存在する。フォルダを開けばどのファイルが何のデータか一目でわかる。保存場所も設定ファイルを書き換えるだけで自由に変えられる。

### PostgreSQL：図書館の書庫

本（データ）は全部書庫の中にまとまっているが、背表紙には番号しか書いていない。司書（PostgreSQL自身）が管理台帳を持っていて、番号と中身の対応を把握している。利用者は司書に聞かないと目的のデータにたどり着けない。

---

## 内部構造の違い（専門用語あり）

### ファイル構造

MySQLはテーブルごとに `.ibd` ファイルが独立して存在する（InnoDBエンジンの場合）。

```
# MySQLのdatadir
/var/lib/mysql/mydb/
  users.ibd       ← テーブル名がそのままファイル名
  orders.ibd
  products.ibd
```

PostgreSQLはクラスタ（データとメタ情報の管理単位）の中に全テーブルが格納され、ファイル名は内部OIDの番号になる。

```
# PostgreSQLのデータディレクトリ
/var/lib/postgresql/data/base/16384/
  1259            ← pg_class（システムカタログ）
  16385           ← 何のテーブルかは内部管理
  16386
```

### プロセスモデル

| | MySQL | PostgreSQL |
|---|---|---|
| 接続方式 | スレッド型（同一プロセス内） | プロセス分離型（接続ごとにfork） |
| 安定性 | スレッド間でメモリ共有 → クラッシュリスクあり | プロセス分離 → 安定性が高い |
| 接続コスト | 軽量 | 重い（コネクションプーラーのpgBouncerが必要になることも） |

### MVCC（同時実行制御）の実装

どちらも「読み書きが互いをブロックしない」MVCCの仕組みを持つが、実装方法が異なる。

**MySQL（InnoDB）**
- 最新データは本体に、古いバージョンはUndo Logに保存
- 本体は常に最新版のみ → VACUUMは不要
- ただしUndo Logが肥大化することがある

**PostgreSQL**
- 古いバージョンをテーブル内にそのまま保持（タプル多版管理）
- 定期的な `VACUUM` で古いバージョンを掃除する必要がある
- その代わり、バージョン管理が厳密

### ストレージエンジン

**MySQL**：ストレージエンジンを差し替えられる設計（プラガブル）
- InnoDBがデフォルト（トランザクション対応）
- MyISAM（高速読み取り・トランザクションなし）なども選択可能

**PostgreSQL**：エンジンは一枚岩
- 選択肢はないが、一貫して高機能・高信頼性

### トランザクションの厳密さ

PostgreSQLはSQL標準に忠実。DDL（`CREATE TABLE`など）もトランザクションに含められる。

```sql
-- PostgreSQLではDDLもロールバックできる
BEGIN;
CREATE TABLE test (id INT);
ROLLBACK; -- テーブルが作られなかったことになる

-- MySQLではDDLが暗黙的にコミットされるのでロールバック不可
-- ROLLBACKしてもテーブルは残る
```

## SQLの主な差異

基本的なCRUD・JOINは共通だが、細部に違いがある。

| 項目 | MySQL | PostgreSQL |
|---|---|---|
| 文字列結合 | `CONCAT(a, b)` | `a \|\| b`（SQL標準） |
| 自動連番 | `AUTO_INCREMENT` | `SERIAL` / `GENERATED AS IDENTITY` |
| 真偽値型 | 実態は`TINYINT(1)` | ネイティブな`BOOLEAN`型 |
| DDLのロールバック | 不可 | 可 |
| ウィンドウ関数 | MySQL 8.0以降 | 古くから充実 |
| JSON型 | あり | `JSON` / `JSONB`（インデックス対応） |

---

## まとめ

MySQLは「速くて使いやすい実用主義」、PostgreSQLは「SQL標準・理論的な正しさ重視」という設計思想が、アーキテクチャの隅々まで反映されている。

DjangoはPostgreSQLとの組み合わせを公式推奨しており、Supabaseも内部はPostgreSQL。
