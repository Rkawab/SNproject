# AWS IPAM（IP Address Manager）

## 概要

複数のVPCやAWSアカウントにまたがるIPアドレスを**一元管理・自動割り当て**するサービス。
AWS Organizationsと連携してマルチアカウント環境のIP管理を効率化する。

> [!info] たとえ話
> **IPAM = マンション管理組合**
> 各部屋（VPC）の番号（IPアドレス）を住人が勝手に決めると重複が起きる。
> IPAMが一括管理することで「301号室が2つある！」という混乱を防ぐ。

---

## 基本構造

```
IPAM
├── スコープ（Scope）
│   ├── プライベートスコープ（VPCなどのプライベートIP）
│   └── パブリックスコープ（グローバルIPアドレス）
│
├── プール（Pool）← CIDRブロックをツリー状に管理
│   ├── 親プール: 10.0.0.0/8（会社全体）
│   ├── 子プール: 10.1.0.0/16（東京リージョン）
│   └── 孫プール: 10.1.1.0/24（本番環境）
│
└── 割り当て（Allocation）← VPCへの実際のCIDR割り当て
```

| 構成要素 | 役割 |
|----------|------|
| スコープ | プライベート/パブリックの管理範囲を区分け |
| プール | CIDR範囲を階層的に分割・委譲する単位 |
| 割り当て | VPCや他リソースへの実際のIP割り当て |

---

## 主な機能

| 機能 | 内容 |
|------|------|
| IP割り当ての自動化 | VPC作成時に自動でCIDRを割り当て |
| 重複検知・防止 | IPアドレスの衝突を事前に防ぐ |
| 使用状況の可視化 | どのIPがどのリソースに使われているか一目で確認 |
| マルチアカウント対応 | AWS Organizationsと連携して組織全体を一元管理 |
| 履歴追跡 | 過去のIP割り当て履歴を確認可能 |

---

## 活用シナリオ

```
【問題】 複数アカウントで同じCIDRを使ってしまった
  開発アカウント: 10.0.0.0/16
  本番アカウント: 10.0.0.0/16  ← 重複！VPCピアリング不可

【解決】 IPAMで事前に割り当てを管理
  開発アカウント: 10.1.0.0/16（IPAMが自動割り当て）
  本番アカウント: 10.2.0.0/16（重複なし → ピアリングOK）
```

## SAA試験ポイント

> [!tip] 試験でよく出るシナリオ
> - **複数アカウント・VPCでIPが重複しないようにしたい** → IPAM
> - **VPCピアリングできない原因** → CIDRが重複 → IPAM導入で予防
> - **AWS Organizationsと組み合わせた中央IP管理** → IPAM のマルチアカウント機能

> [!warning] 混同注意
> - **IPAM ≠ Route 53 Resolver**：名前解決はRoute 53、IP管理はIPAM
> - **IPAM ≠ VPC CIDR手動設定**：IPAMは自動化・一元管理が目的
