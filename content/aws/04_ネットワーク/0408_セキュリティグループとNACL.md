# セキュリティグループ(SG)と ネットワークACL(NACL)

> [!info] このノートの範囲
> VPC内の通信を制御する2大ファイアウォール、**セキュリティグループ(SG)**と**ネットワークACL(NACL)**を、SAA観点で対比して整理する。
> 「どっちを使う?」を即答できるようにするのが目標。

> [!tip] 最重要の一文
> **SG＝インスタンス単位／ステートフル／許可のみ。NACL＝サブネット単位／ステートレス／許可も拒否もできる。**

---

## 全体像のたとえ

VPCを**会社のビル**に例えると:

- **NACL = フロアの正門に立つ警備員**(サブネット＝フロアの入口)
  - フロアに出入りする全員を**入る時も出る時も**チェックする。
  - 「このIPは入れるな」という**ブラックリスト(拒否ルール)も書ける**。
  - ただし融通が利かない:入れた人が出ていく時も**別途チェック**が必要(＝**ステートレス**)。
- **SG = 各部屋のドアの鍵**(EC2＝社員＝部屋単位)
  - 「**許可した相手しか入れない**」ホワイトリスト方式(拒否ルールは書けない)。
  - 一度入れた相手の応答(戻りの通信)は**自動で通す**(＝**ステートフル**)。

---

## 1. セキュリティグループ(Security Group / SG)

**EC2など各リソースのENI(仮想NIC)に紐づく、インスタンス単位の仮想ファイアウォール。**

### 特徴

- **適用単位:インスタンス(ENI)単位**。1つのSGを複数インスタンスに付けたり、1インスタンスに複数SGを付けたりできる。
- **ステートフル**:インバウンドを許可すれば、その**戻りの通信(アウトバウンド)は自動的に許可**される。戻りポートを別途開ける必要がない。
- **許可(Allow)ルールのみ**。**拒否(Deny)ルールは書けない**(＝書いていないものは暗黙拒否＝ホワイトリスト)。
- **全ルールを評価**(順序の概念なし)。どれか1つでも許可に一致すれば通す(論理和)。
- **送信元/宛先にSG IDを指定できる**:「このSGが付いたインスタンスからの通信を許可」と書ける(IP固定不要)。ALB→EC2、踏み台→DBなどの構成で多用。
- **デフォルト**:インバウンド=全拒否 / アウトバウンド=全許可。

> [!example] たとえ
> 部屋のドアの鍵。鍵を渡した(許可した)相手だけ入れる。入れた人が部屋から出る(応答する)のはノーチェックで通す(ステートフル)。「この人は入れるな」という貼り紙(Deny)は貼れない。

---

## 2. ネットワークACL(Network ACL / NACL)

**サブネットの境界に置く、サブネット単位のファイアウォール。**

### 特徴

- **適用単位:サブネット単位**。そのサブネット内の全インスタンスにまとめて効く。
- **ステートレス**:インバウンドを許可しても、**戻りの通信(アウトバウンド)を別途明示的に許可**しないと通信が成立しない。
- **許可(Allow)も拒否(Deny)も書ける**。特定IPを**ピンポイントで遮断(Deny)**できるのはNACLだけ。
- **ルール番号の小さい順に評価し、最初に一致したルールを適用**(以降は見ない)。番号設計が重要。
- **送信元はIPアドレス(CIDR)のみ**指定可能(SG IDは指定できない)。
- **デフォルトNACL**=全許可 / **カスタムで新規作成したNACL**=全拒否(明示許可するまで全遮断)。

> [!warning] ステートレスの落とし穴(SAA頻出)
> NACLでインバウンド80番を許可しても、応答は**エフェメラルポート(1024–65535)**で返るため、**アウトバウンドでエフェメラルポート範囲を開けないと通信が成立しない**。SGはステートフルなのでこの考慮は不要。「片方向だけ許可して通信できない」シナリオはNACLのステートレス忘れを疑う。

> [!example] たとえ
> フロア正門の警備員。出入りする全員を毎回チェックし、「このIPは入館禁止」という名簿(Deny)も持てる。ただし入館を許しても退館は別ルール(ステートレス)。

---

## 3. 核心の対比表

| 観点 | セキュリティグループ (SG) | ネットワークACL (NACL) |
|---|---|---|
| 適用単位 | **インスタンス(ENI)単位** | **サブネット単位** |
| ステート | **ステートフル**(戻りは自動許可) | **ステートレス**(戻りも明示許可) |
| ルール種別 | **許可(Allow)のみ** | **許可・拒否(Deny)両方** |
| 評価方法 | 全ルールを評価(順序なし／論理和) | **番号順**、最初に一致したルールを適用 |
| 送信元の指定 | IP / CIDR / **SG ID** | **IP / CIDR のみ** |
| デフォルト | IN=全拒否 / OUT=全許可 | デフォルト=全許可 / カスタム=全拒否 |
| 役割イメージ | 部屋のドアの鍵 | フロア正門の警備員 |

---

## 4. 通信の評価順序

どちらが先に効くか(イメージ):

```
受信(インバウンド): サブネット入口 NACL  →  インスタンス SG  →  EC2
送信(アウトバウンド): EC2  →  インスタンス SG  →  サブネット出口 NACL
```

- 外から入る時は**NACLが先**(サブネット境界)、次にSG(インスタンス境界)。
- どちらか一方でも拒否すれば通信は通らない(両方の許可が必要)。

---

## 5. SAA出題パターン早見

> [!tip] 問題文のキーワード → 答え
> - 「**特定IPをサブネット全体で拒否(Deny)したい**」 → **NACL**(SGはDeny不可・インスタンス単位)
> - 「**サブネット単位で一括ブロック**」 → **NACL**
> - 「**ステートレスに制御**」 → **NACL**
> - 「**特定のインスタンス/ロード分散構成で許可制御**」「**SGを送信元に指定**」 → **SG**
> - 「**戻り通信のポートを意識せず許可だけ書きたい**」 → **SG**(ステートフル)
> - 「**片方向だけ許可したのに通信できない**」 → **NACLのステートレス**(エフェメラルポート未開放)を疑う

> [!example] 典型問題と解き方
> **問**:特定のIPアドレス範囲からの通信を、サブネット全体に対してステートレスに「拒否(Deny)」したい。適切な仕組みは?
> **答**:**NACL**。
> - 「サブネット全体」→ 単位がサブネットなのでNACL(SGはインスタンス単位)。
> - 「Deny(拒否)」→ SGは許可ルールしか書けない。明示的拒否はNACLだけ。
> - 「ステートレス」→ NACLの特徴そのもの。
> - ルートテーブルは経路制御であり「拒否」の仕組みではない。IAMはAWS API認可で、ネットワーク層の通信制御ではない。

## まとめ:一行整理

| | 一言で |
|---|---|
| **SG** | インスタンスの鍵。許可だけ・ステートフル・SG ID指定可 |
| **NACL** | サブネットの門番。許可も拒否も・ステートレス・番号順 |
| **使い分け** | 通常はSGで制御。**特定IPの遮断やサブネット一括ブロックが必要な時だけNACL** |
