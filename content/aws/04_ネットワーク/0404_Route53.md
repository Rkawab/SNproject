# AWS Route 53

## 概要

Route 53 は AWS が提供するフルマネージドの **DNS（Domain Name System）サービス**。

### 名前の由来
- **Route** → ルーティング（経路制御）
- **53** → DNS が使う標準ポート番号（53番ポート）

---

## DNS とは（電話帳のたとえ）

電話帳のように「名前 → 番号」の対応表。DNS の場合は「ドメイン名 → IP アドレス」。

```
電話帳:  山田太郎 → 090-1234-5678
DNS:     example.com → 203.0.113.10
```

この対応表の1行1行を **「DNS レコード」** と呼ぶ。

---

## DNS レコードの種類

### A レコード（基本）

ドメインを **IP アドレスに直接** 変換する最もシンプルなレコード。

```
example.com  →  203.0.113.10（固定IP）
```

### CNAME レコード（別名・エイリアス）

「この名前は、別の名前と同じです」という転送ルール。

```
shop.example.com  →  example.com  →  203.0.113.10
     ↑ 別名（CNAME）     ↑ 本名（A レコード）
```

**たとえ：転送電話**
```
03-0000-0000（代表）にかけると
→「03-1111-2222（直通）にかけ直してください」
→ もう1ステップ経由して繋がる
```

**制限：ルートドメイン（example.com）には使えない。**  
`www.example.com` のようなサブドメインにしか設定できない。

### ALIAS レコード（AWS 独自）

AWS Route 53 が独自に追加した特別なレコード。  
CNAME の上位互換で、ルートドメインにも使えて、かつ直接解決できる。

**なぜ必要か？** ELB の IP アドレスは変わるため A レコードを直接書けない。

```
ELB の IP アドレス
  月曜: 203.0.113.10
  火曜: 198.51.100.5   ← 変わる！
```

**たとえ：直通案内**
```
example.com にアクセス
→ Route 53 が裏で ELB の現在の IP を自動で調べて直接つなぐ
（ユーザーは1回のアクセスで完結）
```

### CNAME vs ALIAS の比較（試験頻出）

| 比較項目 | CNAME | ALIAS |
|----------|-------|-------|
| 使える場所 | サブドメインのみ | ルートドメインも OK |
| DNS 解決の回数 | 2回（別名 → 本名 → IP） | 1回（直接 IP） |
| AWS リソースへの紐づけ | △ できるが非推奨 | ◎ 推奨 |
| クエリ料金 | 課金あり | **無料**（AWS リソース宛のみ） |

## 主な機能

| 機能 | 説明 |
|------|------|
| **ドメイン登録** | example.com などのドメイン名を購入・管理 |
| **DNS ルーティング** | ドメインをIPアドレスや AWS リソースに紐づける |
| **ヘルスチェック** | エンドポイントの死活監視。障害時に自動で切り替え |

---

## ルーティングポリシー（試験頻出）

| ポリシー | 動作 | 使いどころ |
|----------|------|-----------|
| **シンプル** | 単一リソースに固定で向ける | 最もシンプルな構成 |
| **加重（Weighted）** | 複数リソースに割合で振り分け（70% / 30%） | カナリアリリース・段階的移行 |
| **フェイルオーバー** | プライマリ障害時にセカンダリへ自動切り替え | 高可用性構成 |
| **レイテンシー** | 応答が最速のリージョンへ誘導 | グローバル展開でのパフォーマンス最適化 |
| **地理的（Geolocation）** | ユーザーの地域（国・大陸）で向き先を変える | 地域ごとにコンテンツを出し分け |
| **IPベース** | ユーザーの IP アドレス範囲でルーティング | オンプレ→ AWS 移行期など |
| **複数値回答** | 複数の IP をランダムに返す | 簡易ロードバランシング |

---

## AWS サービスとの組み合わせ

```
ユーザー
  │
  ▼
Route 53（DNS解決 + ルーティング）
  │
  ├─▶ ELB（ロードバランサー）← ALIAS レコードで紐づける
  ├─▶ CloudFront（CDN）      ← ALIAS レコードで紐づける
  └─▶ S3（静的サイト）       ← ALIAS レコードで紐づける
```

---

## SAA 試験 頻出パターン

| 問題パターン | 答え |
|-------------|------|
| ルートドメインを ELB に紐づけたい | **ALIAS レコード**（CNAME は不可） |
| 障害時に自動切り替えしたい | **ヘルスチェック + フェイルオーバールーティング** |
| 新バージョンに 10% だけ流したい | **加重ルーティング（Weighted）** |
| 日本ユーザーは東京、米国ユーザーはバージニアへ | **地理的ルーティング（Geolocation）** または **レイテンシーベース** |

## ヘルスチェック

### 概要

サーバーが「生きているか・死んでいるか」を定期的に確認する仕組み。

**たとえ：コンビニのレジ巡回**
```
レジA（メイン）・レジB（サブ）がある
店長が5分おきに「レジA、動いてる？」と確認
  → 正常 → そのままレジAに案内
  → 故障 → 自動でレジBへ案内を切り替え
```

### Route 53 での動き

```
Route 53
  │
  ├─ 30秒ごとに HTTP リクエストを送る
  │        ↓
  │    EC2 サーバー
  │        ↓
  │    200 OK → 「正常」
  │    タイムアウト / エラー → 「異常」（3回連続で異常判定）
  │
  └─ 異常と判定したら DNS の向き先を切り替える
```

### ヘルスチェックの種類

| 種類 | 何を監視するか |
|------|--------------|
| **エンドポイント監視** | IP / ドメインに HTTP・HTTPS・TCP で疎通確認 |
| **Calculated（計算式）** | 複数のヘルスチェックを AND / OR で組み合わせて判定 |
| **CloudWatch アラーム監視** | CPU 使用率などのメトリクスが閾値を超えたら異常と判定 |

### フェイルオーバーとの組み合わせ（SAA最頻出）

ヘルスチェック単体は「生死を確認するだけ」。  
**フェイルオーバールーティング**と組み合わせて「自動切り替え」が実現する。

```
通常時:
ユーザー → Route 53 → プライマリ（EC2 東京） ← ヘルスチェック: 正常

障害時:
プライマリのヘルスチェックが「異常」を検知
          ↓
ユーザー → Route 53 → セカンダリ（S3 静的ページなど）← 自動切り替え
```

### ヘルスチェック SAA 頻出ポイント

| 問題パターン | 答え |
|-------------|------|
| 自動フェイルオーバーの実現方法は？ | **ヘルスチェック + フェイルオーバールーティング** |
| ヘルスチェックはどこで設定するか | **Route 53 側**（サーバー側ではない） |
| デフォルトのチェック間隔は？ | **30秒**（高速オプションで10秒も可） |
| 何回失敗で異常と判定されるか | **3回連続失敗** |
