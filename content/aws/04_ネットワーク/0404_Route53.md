# AWS Route 53

## 概要

Route 53 は AWS が提供するフルマネージドの **DNS（Domain Name System）サービス**。

### 名前の由来
- **Route** → ルーティング（経路制御）
- **53** → DNS が使う標準ポート番号（53番ポート）

---

## DNS とは（電話帳のたとえ）

電話帳のように「名前 → 番号」の対応表。DNS の場合は「ドメイン名 → IP アドレス」。

```
電話帳:  山田太郎 → 090-1234-5678
DNS:     example.com → 203.0.113.10
```

この対応表の1行1行を **「DNS レコード」** と呼ぶ。

---

## DNS レコードの種類

### A レコード（基本）

ドメインを **IP アドレスに直接** 変換する最もシンプルなレコード。

```
example.com  →  203.0.113.10（固定IP）
```

### CNAME レコード（別名・エイリアス）

「この名前は、別の名前と同じです」という転送ルール。

```
shop.example.com  →  example.com  →  203.0.113.10
     ↑ 別名（CNAME）     ↑ 本名（A レコード）
```

**たとえ：転送電話**
```
03-0000-0000（代表）にかけると
→「03-1111-2222（直通）にかけ直してください」
→ もう1ステップ経由して繋がる
```

**制限：ルートドメイン（example.com）には使えない。**  
`www.example.com` のようなサブドメインにしか設定できない。

### ALIAS レコード（AWS 独自）

AWS Route 53 が独自に追加した特別なレコード。  
CNAME の上位互換で、ルートドメインにも使えて、かつ直接解決できる。

**なぜ必要か？** ELB の IP アドレスは変わるため A レコードを直接書けない。

```
ELB の IP アドレス
  月曜: 203.0.113.10
  火曜: 198.51.100.5   ← 変わる！
```

**たとえ：直通案内**
```
example.com にアクセス
→ Route 53 が裏で ELB の現在の IP を自動で調べて直接つなぐ
（ユーザーは1回のアクセスで完結）
```

### CNAME vs ALIAS の比較（試験頻出）

| 比較項目 | CNAME | ALIAS |
|----------|-------|-------|
| 使える場所 | サブドメインのみ | ルートドメインも OK |
| DNS 解決の回数 | 2回（別名 → 本名 → IP） | 1回（直接 IP） |
| AWS リソースへの紐づけ | △ できるが非推奨 | ◎ 推奨 |
| クエリ料金 | 課金あり | **無料**（AWS リソース宛のみ） |

## 主な機能

| 機能 | 説明 |
|------|------|
| **ドメイン登録** | example.com などのドメイン名を購入・管理 |
| **DNS ルーティング** | ドメインをIPアドレスや AWS リソースに紐づける |
| **ヘルスチェック** | エンドポイントの死活監視。障害時に自動で切り替え |

---

## ルーティングポリシー（試験頻出）

| ポリシー | ひとことで | 使いどころ |
|----------|------|-----------|
| **シンプル** | 1レコードを単純に返す | 単一リソース |
| **加重（Weighted）** | 割合で振り分け（例:90:10） | カナリアリリース・A/Bテスト |
| **フェイルオーバー** | **本番ダウンで予備に自動切替** | **DR・アクティブ/パッシブ** |
| **レイテンシー** | **一番速いリージョン**へ | 多リージョンで低遅延 |
| **位置情報（Geolocation）** | **ユーザーの国/地域**で振り分け | 地域別コンテンツ・規制対応 |
| **地理的近接（Geoproximity）** | 地理的な近さ（＋バイアス調整） | リージョン間の負荷寄せ |
| **複数値回答（Multivalue）** | 複数IPを返す＋ヘルスチェック | 簡易的な負荷分散 |
| **IPベース** | ユーザーの IP アドレス範囲でルーティング | オンプレ→ AWS 移行期など |

---

## フェイルオーバールーティングポリシー（SAAキモ）

### 概要

**アクティブ/パッシブ（本番/予備）の自動切替**を行うポリシー。

- **プライマリ（本番）レコード**と**セカンダリ（予備）レコード**の2つを用意する
- Route 53が**ヘルスチェックでプライマリの生死を常時監視**
- **プライマリが健全 → プライマリを返す**／**プライマリがダウン → 自動でセカンダリを返す**

→ つまり「**普段は本番に流し、本番が落ちたら予備に切り替える**」をDNSレベルでやる仕組み。**ヘルスチェックが必須**。

### たとえ話：代表電話の自動転送

```
普通時:
  ユーザー（外部）
      ↓
  交換手（Route 53）
      ↓
  本店（プライマリ・EC2 東京）← 常時確認「本店、生きてる？」

本店ダウン時:
  プライマリのヘルスチェック = 異常
      ↓
  ユーザー（外部）
      ↓
  交換手（Route 53）[自動切替]
      ↓
  支店（セカンダリ・S3静的ページなど）
```

### 運用構成例

```
通常時:
ユーザー → Route 53 → EC2 東京（本番）✓ ヘルスチェック: 正常

EC2東京がダウン:
プライマリのヘルスチェックが「異常」を3回連続で検知
      ↓
ユーザー → Route 53 → S3 静的フォールバック or EC2 大阪（予備）✓ 自動切り替え
```

### 他の主要ポリシーとの違い（SAAの覚え方）

- **「本番が落ちたら予備へ自動で」→ フェイルオーバー**（＋ヘルスチェック必須）
- **「一番速いリージョンへ」→ レイテンシー**
- **「ユーザーの国で出し分け」→ 位置情報（Geolocation）**
- **「割合で振り分け／徐々に移行」→ 加重（Weighted）**

---

## AWS サービスとの組み合わせ

```
ユーザー
  │
  ▼
Route 53（DNS解決 + ルーティング）
  │
  ├─▶ ELB（ロードバランサー）← ALIAS レコードで紐づける
  ├─▶ CloudFront（CDN）      ← ALIAS レコードで紐づける
  └─▶ S3（静的サイト）       ← ALIAS レコードで紐づける
```

## SAA 試験 頻出パターン

| 問題パターン | 答え |
|-------------|------|
| ルートドメインを ELB に紐づけたい | **ALIAS レコード**（CNAME は不可） |
| 障害時に自動切り替えしたい | **ヘルスチェック + フェイルオーバールーティング** |
| 新バージョンに 10% だけ流したい | **加重ルーティング（Weighted）** |
| 日本ユーザーは東京、米国ユーザーはバージニアへ | **地理的ルーティング（Geolocation）** または **レイテンシーベース** |

## ヘルスチェック

### 概要

サーバーが「生きているか・死んでいるか」を定期的に確認する仕組み。

**たとえ：コンビニのレジ巡回**
```
レジA（メイン）・レジB（サブ）がある
店長が5分おきに「レジA、動いてる？」と確認
  → 正常 → そのままレジAに案内
  → 故障 → 自動でレジBへ案内を切り替え
```

### Route 53 での動き

```
Route 53
  │
  ├─ 30秒ごとに HTTP リクエストを送る
  │        ↓
  │    EC2 サーバー
  │        ↓
  │    200 OK → 「正常」
  │    タイムアウト / エラー → 「異常」（3回連続で異常判定）
  │
  └─ 異常と判定したら DNS の向き先を切り替える
```

### ヘルスチェックの種類

| 種類 | 何を監視するか |
|------|--------------|
| **エンドポイント監視** | IP / ドメインに HTTP・HTTPS・TCP で疎通確認 |
| **Calculated（計算式）** | 複数のヘルスチェックを AND / OR で組み合わせて判定 |
| **CloudWatch アラーム監視** | CPU 使用率などのメトリクスが閾値を超えたら異常と判定 |

### フェイルオーバーとの組み合わせ（SAA最頻出）

ヘルスチェック単体は「生死を確認するだけ」。  
**フェイルオーバールーティング**と組み合わせて「自動切り替え」が実現する。

```
通常時:
ユーザー → Route 53 → プライマリ（EC2 東京） ← ヘルスチェック: 正常

障害時:
プライマリのヘルスチェックが「異常」を検知
          ↓
ユーザー → Route 53 → セカンダリ（S3 静的ページなど）← 自動切り替え
```

### ヘルスチェック SAA 頻出ポイント

| 問題パターン | 答え |
|-------------|------|
| 自動フェイルオーバーの実現方法は？ | **ヘルスチェック + フェイルオーバールーティング** |
| ヘルスチェックはどこで設定するか | **Route 53 側**（サーバー側ではない） |
| デフォルトのチェック間隔は？ | **30秒**（高速オプションで10秒も可） |
| 何回失敗で異常と判定されるか | **3回連続失敗** |

## Route 53 Resolver（ハイブリッドDNS）

VPCには標準でDNSサーバーが備わっている（VPCのCIDR**+2**のアドレス。Amazon Provided DNS）。**オンプレミスとAWSをつないだハイブリッド構成での名前解決**を担うのが「**Resolverエンドポイント**」。

### 2種類のエンドポイント（問い合わせの「向き」で覚える）

| エンドポイント | 問い合わせの向き | 用途 |
|---|---|---|
| **インバウンドエンドポイント** | オンプレ **→** AWS | オンプレのサーバーが**VPC内のプライベートDNS名**（EC2等）を解決したい |
| **アウトバウンドエンドポイント** | AWS **→** オンプレ | VPC内のリソースが**オンプレの社内ドメイン**を解決したい（**転送ルール**で社内DNSへフォワード） |

> [!tip] たとえ話：内線電話帳の相互接続
> AWSとオンプレはそれぞれ自分の「内線電話帳（DNS）」を持っている。Direct ConnectやVPNで**回線**はつながっても、**相手の電話帳は勝手に引けない**。
> - オンプレ側からAWSの電話帳を引く窓口 ＝ **イン**バウンドエンドポイント（AWSに問い合わせが入ってくる）
> - AWS側からオンプレの電話帳を引きに行く窓口 ＝ **アウト**バウンドエンドポイント（AWSから問い合わせが出ていく）

### SAA頻出パターン

| 問題文 | 答え |
|---|---|
| オンプレのサーバーからVPC内のプライベートDNS名を解決したい | Resolver **インバウンド**エンドポイント |
| EC2からオンプレの社内ドメイン名を解決したい | Resolver **アウトバウンド**エンドポイント＋転送ルール |
| 双方向で名前解決したい | 両方のエンドポイントを作成 |

- （参考）**Route 53 Resolver DNS Firewall**：VPCから悪性ドメインへのDNS問い合わせをブロックする機能。「**DNSレベルでの防御**」と出たらこれ。
