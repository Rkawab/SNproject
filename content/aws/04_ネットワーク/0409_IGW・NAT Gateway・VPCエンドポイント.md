# Internet Gateway / NAT Gateway / VPCエンドポイント(VPCの出入口)

> [!info] このノートの範囲
> VPC内のリソースが「**外(インターネット)とどう出入りするか**」「**AWSサービスへインターネットを通らず行く方法**」を司る3つの仕組み、**Internet Gateway(IGW)／NAT Gateway／VPCエンドポイント**をSAA観点で対比して整理する。
> SAAの不足#1(VPCコア)に挙がっていた頻出テーマ。サブネットの基礎は [[0101_EC2やVPCとは]] を前提とする。

> [!tip] 最重要の一文
> **IGW＝VPCの正面玄関(双方向)。NAT Gateway＝中の人だけ外出できる片方向の出口(外からは呼べない)。VPCエンドポイント＝AWSサービスへインターネットを通らず行く専用通路。**

---

## 全体像のたとえ

VPCを**会社のビル**に例えると(サブネット＝フロア、EC2＝社員):

- **Internet Gateway(IGW) = ビルの正面玄関**(公道に直結)
  - 玄関にルートが通っているフロア＝**パブリックサブネット**。外から来客も入れるし、中の人も出られる(**双方向**)。
- **NAT Gateway = 受付経由の「代理外出窓口」**
  - 奥のフロア(プライベートサブネット)の社員が、**外出(アウトバウンド)はできる**が、外部から名指しで**呼び出されることはない**(**片方向**)。外出した先の返事(応答)だけは受け取れる。
- **VPCエンドポイント = 取引先(S3/DynamoDB等のAWSサービス)への専用通路**
  - 公道(インターネット)を一切通らず、ビル内から直接つながる**社内連絡通路**。

```
インターネット
   │
 [IGW] ← VPCに1つだけアタッチ
   │
パブリックサブネット ── [NAT Gateway](EIP付き)
   │                         ▲ 0.0.0.0/0 をここへ向ける
プライベートサブネット ───────┘
   │
 [VPCエンドポイント] ──→ S3 / DynamoDB / 他AWSサービス(インターネット非経由)
```

---

## 1. Internet Gateway(IGW)

**VPCとインターネットを双方向につなぐ玄関。VPCに1つだけアタッチする、水平スケール済みの冗長なマネージドコンポーネント。**

### 特徴

- **VPCに1つ**アタッチ(0または1)。可用性・帯域はAWS側が自動で確保(自分で冗長化不要)。
- サブネットを**パブリックにする条件**＝そのサブネットのルートテーブルに **`0.0.0.0/0 → igw-xxxx`** がある(IPv6は `::/0`)。
- IGWは**パブリックIP(またはEIP)を持つリソースだけ**を外と通信させる(NAT変換はしない。あくまで経路)。
- 料金は**無料**(IGW自体には課金されない。転送量は別)。

> [!example] たとえ
> ビルの正面玄関。玄関へ続く廊下(ルート)があるフロアだけが外と行き来できる。玄関自体は勝手に混雑を捌いてくれる(冗長)。

---

## 2. NAT Gateway

**プライベートサブネットのリソースが、外部IPを持たないままインターネットへ"出ていく(アウトバウンド)"ためのマネージドな出口。外から中へは入れない。**

### 特徴

- **片方向**:中→外(例:OSパッチ取得、外部API呼び出し)はOK。**外→中の新規接続は不可**(応答の戻りだけ通す)。
- **パブリックサブネットに置き、Elastic IPを1つ持つ**。プライベートサブネット側のルートに **`0.0.0.0/0 → nat-xxxx`** を向ける。
- **AZ単位のサービス**。**各AZにNAT Gatewayを1つずつ**置くのが高可用構成(1つのAZのNATが死んでも他AZは生存)。
- フルマネージドで**自動スケール(最大100Gbps)・高可用**。
- 料金は**有料**(時間課金＋**処理データ量(GB)課金**)。SAAでは「NAT経由の転送料が高い」がコスト最適化の論点になりがち。

> [!warning] NAT Gateway vs NAT インスタンス(SAA頻出比較)
> | 観点 | **NAT Gateway** | **NATインスタンス**(旧式) |
> |---|---|---|
> | 実体 | マネージドサービス | 自前のEC2(amzn-ami-vpc-nat) |
> | 可用性 | AZ内で自動冗長・自動スケール | 自分でスケール/監視/フェイルオーバー |
> | 管理 | 不要 | OS・SGパッチ等を自分で運用 |
> | 帯域 | 最大100Gbps | インスタンスタイプ依存 |
> | 踏み台/ポート転送 | 不可 | 可能(汎用EC2なので) |
> 基本は **NAT Gateway 一択**。「運用を任せたい/高可用」→ Gateway、「自前で細かく制御」という特殊要件のみインスタンス。

> [!example] たとえ
> 受付を通した代理外出窓口。社員(プライベートのEC2)は外に買い物に行けて、その品物(応答)を持ち帰れる。でも外部の人が窓口越しに社員を名指しで呼び出すことはできない(片方向)。

## 3. VPCエンドポイント

**VPC内から、AWSのサービス(S3・DynamoDB等)や自前サービスへ、インターネット(IGW/NAT)を一切通さずプライベートに接続する仕組み。**

2種類ある。ここが最頻出。

### 3-1. Gateway型エンドポイント

- **対象は S3 と DynamoDB の2つだけ**(超重要)。
- **ルートテーブルにエントリを追加**する方式(宛先＝そのサービスのプレフィックスリスト)。
- **無料**。
- 「プライベートサブネットのEC2からS3へ、NAT経由の転送料も払わず・インターネットにも出さずアクセスしたい」→ **これ**。

### 3-2. Interface型エンドポイント(AWS PrivateLink)

- **対象は多数のAWSサービス**(SQS, SNS, KMS, CloudWatch, ECR, Systems Manager…)＋**自前/サードパーティのサービス**。
- **サブネット内にENI(プライベートIP)を作る**方式。DNS名がそのプライベートIPに解決される。
- **有料**(ENIの時間課金＋データ処理量)。
- S3はInterface型でも使える(オンプレ→DX/VPN経由でS3にプライベートアクセスしたい時など)が、VPC内だけならGateway型(無料)で十分。

> [!note] PrivateLink との関係
> **Interface型エンドポイント＝PrivateLink の実体**。PrivateLinkは「サービスをVPC内のENIとして露出し、プライベートIPで呼ぶ」技術の総称。詳しい接続系(DX/VPN/TGW)との位置づけは [[0405_ネットワークとコンテンツ配信]] を参照。

### Gateway型 vs Interface型

| 観点 | **Gateway型** | **Interface型(PrivateLink)** |
|---|---|---|
| 対象 | **S3 / DynamoDB のみ** | 多数のAWSサービス＋自前サービス |
| 実体 | ルートテーブルのエントリ | サブネット内の**ENI(プライベートIP)** |
| 料金 | **無料** | 時間課金＋データ処理料 |
| オンプレから利用 | 不可 | **可能**(DX/VPN経由) |
| DNS | 変わらず | エンドポイント用のプライベートDNS |

> [!example] たとえ
> Gateway型＝「S3行き/DynamoDB行きの社内直通通路を案内板(ルートテーブル)に書き足す」。Interface型＝「相手サービスの内線電話(ENI)を自分のフロアに引いて、社内番号で直接かける」。

## 4. 「どれを使う?」決定フロー(SAA頻出)

> [!tip] 問題文のキーワード → 答え
> - 「**外から来客も受ける/中の人も出る(Webサーバ・ALB)**」 → **IGW＋パブリックサブネット**
> - 「**プライベートのEC2をパッチ取得など外へ出したいが、外からは入らせたくない**」 → **NAT Gateway**
> - 「**プライベートサブネットからS3/DynamoDBへ、NAT料金もインターネットも避けたい**」 → **Gateway型VPCエンドポイント**(無料)
> - 「**プライベートにSQS/KMS/ECR等やオンプレからS3へ接続したい**」 → **Interface型エンドポイント(PrivateLink)**
> - 「**NAT Gatewayが高可用になっていない**」 → **各AZに1台ずつ**配置していないことを疑う
> - 「**NAT経由のS3転送料が高い**」 → **Gateway型エンドポイント**に切替でコスト削減

> [!example] 典型問題と解き方
> **問**:プライベートサブネットのEC2が、毎晩S3へ大量のバックアップを送っている。現状はNAT Gateway経由で、データ処理料が高額。インターネットには一切出したくない。最もコスト効率の良い構成は?
> **答**:**S3用のGateway型VPCエンドポイント**を作成し、ルートテーブルでS3宛てをそこへ向ける。
> - Gateway型は**無料**＆**インターネット非経由**なので、NATのデータ処理料が丸ごと消える。
> - Interface型でもS3に届くが**有料**なので、VPC内だけならGateway型が最適。
> - NAT Gatewayの増設や別AZ化はコスト削減にならない。

## まとめ:一行整理

| 仕組み | 一言で | 料金 |
|---|---|---|
| **Internet Gateway** | VPCの正面玄関(双方向)。VPCに1つ、ルートに `0.0.0.0/0→igw` | 無料 |
| **NAT Gateway** | プライベートの片方向の出口(中→外のみ)。AZごとに1台、EIP付き | 有料(時間＋処理量) |
| **Gateway型エンドポイント** | S3/DynamoDB専用の社内直通路。ルートに足すだけ | **無料** |
| **Interface型(PrivateLink)** | 多数サービス＋自前サービスをENIで露出 | 有料 |

## 5. SAAでの「ひっかけ」パターン 🎣

VPCエンドポイント周辺は試験での引っかけが多い。よくある誤選択：

### ❌ 誤選択例

**問題文：** 「プライベートサブネットのEC2が、毎晩S3へ大量のバックアップを送っている。NAT Gateway経由で処理料が高額。コスト最適化するには？」

| 選択肢 | 判定 | 理由 |
|---|---|---|
| **A. S3用Gateway型VPCエンドポイント** | ✅ **正解** | 無料＋インターネット非経由。NAT処理料が丸ごと消える。 |
| **B. Internet GatewayをEC2に直接アタッチ** | ❌ ×「IGWは VPC にアタッチするもの、EC2にはアタッチしない」＋インターネット経由で要件(VPC内経路)に反する |
| **C. AWS Direct Connect(DX)** | ❌ ×「オンプレ ↔ AWS用」。AWS内部のEC2→S3の話には関係ない。月額単価も非常に高い。 |
| **D. Interface型VPCエンドポイント** | ❌ **超ひっかけ** 「技術的には動く」が、「時間課金＋データ処理料がかかる」。「コスト最適化」の目的に反する。問題文に「**必ず**Interface型を使う」等の特殊要件がない限り、無料のGateway型が答え。 |

> [!warning] 「必ず」「絶対」のワードに注意
> 選択肢で「Interface型VPCエンドポイントを**必ず**作成する」「**絶対に**Interface型を使う必要がある」という断定があれば、その場面は「S3/DynamoDB以外(SQS/KMS等)も接続する要件がある」か「オンプレから接続する」という追加要件を暗に示しています。単にS3だけなら Gateway型一択。

### 試験での判断ロジック

```
「NAT経由でコスト高い」「S3へのアクセス」「インターネット出したくない」
           ↓
    → S3用Gatewayエンドポイント(無料＋内部経路)
           ↓
        正解はこれ
```

## 6. ネットワーク系学習の心がけ 🧠

「IGW/NAT/エンドポイントどれ？」という問題を解く時の最強の着眼点：

> [!tip] **最重要：「その通信は VPC『の中』を通るのか、『外(インターネット)』を通るのか」を毎回問い直す**

- **インターネット(外)経由** → IGW or NAT
- **VPC内完結**(AWS内部リソース向け) → VPCエンドポイント
  - **S3/DynamoDB** → Gateway型(無料)
  - **その他サービス** → Interface型(有料)

このフレームで考えると、複雑に見えるネットワーク問題も一気に整理できる。
