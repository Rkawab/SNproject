[[0501_高可用性とスケーラビリティ]]

## 1. ELB（Elastic Load Balancer）とは

> **「複数のサーバーにトラフィックを自動で振り分ける仕組み」**

### アナロジー：交通整理の警察官
大きな交差点で、警察官が各車線（サーバー）の混雑を見ながらトラフィックを誘導する。  
1台に集中しないよう均等に分散 → **過負荷・単一障害点を防ぐ**

### ELBが提供する機能（共通）

| 機能 | 内容 |
|------|------|
| **負荷分散** | 複数ターゲットにトラフィックを均等配分 |
| **ヘルスチェック** | 異常なインスタンスを自動で切り離す |
| **高可用性** | 複数AZに自動展開 |
| **スケーリング** | ELB自身の処理能力が自動スケール（EC2の台数は増減しない） |
| **SSL/TLS終端** | HTTPS通信の復号をLBで行い、バックエンドの負荷を軽減 |

> [!warning] ALBはインスタンスを直さない
> ALBは、ヘルスチェックで異常になったターゲットへの新規リクエスト送信を停止するだけ。EC2の終了・起動・修復・置換は行わない。
> 障害EC2を置き換えて必要台数を維持するのは **Auto Scaling Group（ASG）**。問題文にASGがなければ、置換まで行われるとは判断しない。

**試験用の一行**：ALBは交通整理、ASGは人員補充。正常なターゲットが残っていれば、ALBは通常そちらへ振り分ける。

---

## 2. ELBの3種類 + GWLB の全体像

```
ELB（総称）
├── ALB（Application Load Balancer）  ← L7（アプリ層）
├── NLB（Network Load Balancer）      ← L4（トランスポート層）
└── GWLB（Gateway Load Balancer）     ← L3（ネットワーク層）

※ CLB（Classic Load Balancer）は旧世代。試験にはほぼ出ない
```

### OSI参照モデルとの対応

```
L7 アプリケーション層  ← ALB  （HTTP/HTTPS・WebSocket）
L6 プレゼンテーション層
L5 セッション層
L4 トランスポート層    ← NLB  （TCP・UDP・TLS）
L3 ネットワーク層      ← GWLB （IPパケット）
L2 データリンク層
L1 物理層
```

## 3. ALB（Application Load Balancer）

> **「HTTPの中身を見て、振り分け先を変えられるLB」**

### アナロジー：カスタマーサポートの受付
電話の内容（問い合わせ種別）を聞いてから、適切な部署（サーバー）につなぎ変える。  
→ ALBはHTTPのパスやヘッダーを見て**振り分け先を柔軟に変える**

### 主な特徴

| 項目 | 内容 |
|------|------|
| **対応プロトコル** | HTTP / HTTPS / WebSocket / HTTP/2 |
| **動作レイヤー** | L7（アプリケーション層） |
| **ルーティング** | パス・ホスト・ヘッダー・クエリパラメータで振り分け先を変更可 |
| **ターゲット** | EC2・ECS・Lambda・IPアドレス |
| **認証** | Cognitoと統合してユーザー認証をLBで処理可能 |
| **WAF統合** | AWS WAFと連携してWebアプリ保護 |

### ルーティング例

```
https://example.com/api/*    → APIサーバー群（EC2）
https://example.com/images/* → 画像サーバー群（EC2）
https://example.com/        → フロントエンド（ECS）

api.example.com  → APIクラスター
www.example.com  → Webクラスター
```

### 向いているユース

- マイクロサービス・コンテナ環境（ECS・EKS）
- REST API のルーティング
- Webアプリケーション全般

---

## 4. NLB（Network Load Balancer）

> **「超高速・超低遅延で、TCPをそのまま流すLB」**

### アナロジー：高速道路の料金所（ETC専用）
中身を確認せず（HTTPの内容は見ない）、ひたすら高速にパケットを通過させる。  
→ NLBは**極限のパフォーマンスと低遅延が必要なとき**に使う

### 主な特徴

| 項目 | 内容 |
|------|------|
| **対応プロトコル** | TCP / UDP / TLS |
| **動作レイヤー** | L4（トランスポート層） |
| **パフォーマンス** | 毎秒数百万リクエスト処理可能、超低遅延 |
| **静的IP** | AZごとに固定IPを付与可能（Elastic IP割当可） |
| **ターゲット** | EC2・IPアドレス・ALB |
| **送信元IP保持** | クライアントのIPがそのままバックエンドに届く |

### 向いているユース

- ゲームサーバー（UDP通信）
- IoT・リアルタイム通信
- 固定IPが必要なシステム（ファイアウォールのホワイトリスト設定）
- ALBの前段に置いてDDoS対策を強化

---

## 5. GWLB（Gateway Load Balancer）

> **「ファイアウォール・IDS/IPSなどのセキュリティアプライアンスを透過的に挟み込むLB」**

### アナロジー：税関検査
荷物（パケット）が目的地に届く前に必ず税関（セキュリティ機器）を通過させる。  
荷物の中身（アプリデータ）は変えずに検査だけ行う。

### 主な特徴

| 項目 | 内容 |
|------|------|
| **対応プロトコル** | GENEVE（ポート6081） |
| **動作レイヤー** | L3（ネットワーク層） |
| **用途** | サードパーティのセキュリティアプライアンス（Palo Alto等）を冗長化・スケール |
| **透過性** | パケットを変更せずそのまま転送（透過プロキシ） |
| **ターゲット** | セキュリティアプライアンスのEC2インスタンス |

### 構成イメージ

```
インターネット
     ↓
  GWLB（トラフィックを受け取る）
     ↓
  セキュリティアプライアンス群（Palo Alto / Fortinet等）
  ※ 検査・フィルタリング
     ↓
  GWLB（検査済みトラフィックを転送）
     ↓
  アプリケーションサーバー
```

### 向いているユース

- エンタープライズのネットワークセキュリティ基盤
- IDS/IPS・ファイアウォールの冗長化
- セキュリティ製品を透過的にスケールアウトしたい場合

## 6. ALB vs NLB vs GWLB 比較表

| 項目 | ALB | NLB | GWLB |
|------|-----|-----|------|
| **レイヤー** | L7 | L4 | L3 |
| **プロトコル** | HTTP/HTTPS/WS | TCP/UDP/TLS | GENEVE |
| **ルーティング** | パス・ホスト・ヘッダー | IPアドレス・ポート | IPパケット |
| **パフォーマンス** | 高い | 超高い（最速） | 高い |
| **静的IP** | ❌（DNS名のみ） | ✅（Elastic IP可） | ❌ |
| **送信元IP保持** | X-Forwarded-For ヘッダー | ✅ネイティブ | ✅ |
| **Lambda対応** | ✅ | ❌ | ❌ |
| **WAF統合** | ✅ | ❌ | ❌ |
| **主な用途** | Webアプリ・API・マイクロサービス | ゲーム・IoT・固定IP | セキュリティアプライアンス |

## 7. どれを選ぶか？ 判断フロー

```
HTTP/HTTPS を扱う？
├── YES → パスやホストで振り分けたい？
│         ├── YES → ALB
│         └── NO  → NLB（TLS終端だけでよければNLBも可）
└── NO
    ├── TCP/UDP が必要？ → NLB
    ├── 固定IPが必要？  → NLB
    └── セキュリティアプライアンスを挟む？ → GWLB
```

## 8. SAA 試験の重要ポイント

| ポイント | 内容 |
|----------|------|
| **ALB + Lambda** | ALBはLambdaをターゲットにできる（NLBはできない） |
| **固定IP** | 固定IPが必要 → NLB（ALBはDNS名のみ） |
| **マイクロサービス** | パスベースルーティング → ALB |
| **超低遅延・UDP** | ゲーム・リアルタイム → NLB |
| **セキュリティ製品の冗長化** | GWLB（GENEVEプロトコル） |
| **WebSocket** | ALBがサポート |
| **HTTP/2** | ALBがサポート（バックエンドとの間はHTTP/1.1に変換） |
| **CLB（旧世代）** | 新規採用は非推奨。試験でも「ALBかNLBを使え」が正解になる |

---

## 9. まとめ図

```
用途別まとめ

ALB  → 「賢い」LB。HTTPの中身を見て振り分け。Webアプリの主役
NLB  → 「速い」LB。中身は見ない。高速・固定IPが武器
GWLB → 「監視役」LB。セキュリティ機器をパケットに透過的に挟む
```
