# スティッキーセッション（Sticky Sessions）

## 概要

ロードバランサーが同じクライアントからのリクエストを、**常に同じバックエンドサーバーに送り続ける**仕組み。

## 身近な例え：銀行の「担当スタッフ制」

```
【普通のロードバランサー】
→ 毎回空いてる窓口に案内（バラバラ）

1回目: 田中さん（窓口A）→「口座開設の途中です」
2回目: 鈴木さん（窓口B）→「え、初めてのお客様ですよね？」← 困る！

【スティッキーセッション】
→「このお客様は田中さん担当」と固定

1回目: 田中さん（窓口A）→「口座開設の途中です」
2回目: 田中さん（窓口A）→「続きから対応します！」← OK
```

## 仕組み（Cookieのパス証的な役割）

初回アクセス時にALBが「この人はAサーバー担当」というCookieをクライアントに発行。
次回からそのCookieを見て同じサーバーへルーティングする。

```
1回目: クライアント → ALB → サーバーA
              ← 「AWSALB=xxxxx（Aの証明書）」

2回目: クライアント → ALB（Cookieを確認）→ サーバーA（固定！）
```

## AWSでの設定（ALB）

| 設定項目 | 内容 |
|---|---|
| 有効化場所 | ターゲットグループの「属性」タブ |
| Cookieの種類 | ロードバランサー生成 / アプリケーション生成 |
| 持続時間 | 1秒〜7日（設定可能） |

## メリット・デメリット

| メリット | デメリット |
|---|---|
| セッション管理が簡単 | 特定サーバーに負荷集中のリスク |
| 既存アプリの改修不要 | サーバー障害時にセッション消失 |

## ベストプラクティス

スティッキーセッションは**応急処置**。本来は「全スタッフが見られる共有の引き継ぎノート」を用意する設計が理想。

```
引き継ぎノート = ElastiCache（Redis）や DynamoDB
→ どのサーバーに振られても状態を参照できる（ステートレス設計）
```

## SAA試験ポイント

| 問われ方 | 答え |
|---|---|
| 「ログイン状態が途中で切れる」の応急処置 | **スティッキーセッション有効化**（ターゲットグループの属性） |
| スケールイン・サーバー障害でもセッションを失わない設計 | セッションを **ElastiCache（Redis）/ DynamoDB** に外部保存（ステートレス化） |
| 負荷が特定サーバーに偏る原因 | スティッキーセッションの副作用 |
