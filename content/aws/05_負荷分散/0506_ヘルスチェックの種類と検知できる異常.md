# ヘルスチェックの種類と検知できる異常

Auto Scalingのヘルスチェックは**2種類**あり、**何を見るか・どう反応するか**が大きく異なる。可用性要件によって使い分けが必須。

---

## ヘルスチェックの2つのタイプ

| ヘルスチェック種別 | 何を見る？ | 見ている層 | HTTP エラーを検知？ | 自動置換される？ |
|---|---|---|---|---|
| **EC2 ヘルスチェック**（既定） | インスタンスのステータス（running/stopped など） | **インフラ層**（HW/OS） | ❌ できない | EC2 が完全に停止した場合のみ |
| **ELB ヘルスチェック** | 指定 URL への HTTP 応答 | **アプリ層** | ✅ できる | HTTP エラー/タイムアウト時に自動置換 |

---

## EC2 ヘルスチェック（デフォルト）

### 特徴
- **インスタンスのシステムステータスだけを確認**
  - 電源が入っているか
  - OS が起動しているか
  - ディスクは応答しているか

### 検知できる異常
- ❌ CPU スパイク
- ❌ メモリ逼迫
- ❌ **アプリケーションエラー（HTTP 500 など）**
- ❌ データベース接続切断
- ✅ ハードウェア障害（インスタンス停止など）
- ✅ ネットワークが完全に断絶

### 実際の問題
```
EC2 インスタンス: running（起動している）
Web アプリケーション: クラッシュ → HTTP 500 を返す
期待: 異常なので置き換え
実際: EC2 ステータスが「running」なので「健全」と判定
結果: **壊れたインスタンスがロードバランサーから要求を受け続ける** 💥
```

---

## ELB ヘルスチェック（推奨）

### 特徴
- **ALB/NLB/ELB を経由して、実際にアプリケーションへリクエストを送信**
- 指定パス（例：`/health`）への HTTP レスポンスコードで判定
- ASG の側でこのヘルスチェック結果を参照する必要がある

### 検知できる異常
- ✅ HTTP エラー（HTTP 5xx）
- ✅ タイムアウト（応答がない）
- ✅ アプリケーション起動失敗
- ✅ データベース接続切断
- ✅ ディスク容量不足（エラーページ返却）
- ❌ **検知後、自動置換は ELB ヘルスチェック設定があってもまだ不完全**

### 重要：ALB ヘルスチェック ≠ 自動置換

**ALB のヘルスチェック単独では、異常インスタンスの自動置換は起きない**：

```
ALB のヘルスチェック:
  ├─ HTTP 500 を検知
  ├─ 「このターゲットは不健全」と判定
  └─ 今後の新規リクエストは此処に振らない

但し:
  × インスタンスの強制終了 ❌
  × 新インスタンスの起動 ❌
  × ASG 側では「何が起きたか」知らない ❌
```

**自動置換を実現するには**、ASG 側で「**ELB ヘルスチェック**」設定を有効にして、ALB の判定結果を参照させる必要がある。

---

## 正しい構成：「ALB + ASG の ELB ヘルスチェック連動」

```
[HTTP リクエスト]
    ↓
[ALB]
  ├─ ターゲットグループに紐付け
  └─ ヘルスチェックパス: /health に定期的にリクエスト送信
    
[ASG] ← ALB のヘルスチェック結果を参照（重要！）
  ├─ ヘルスチェックタイプ: **ELB** に設定
  └─ ALB が「不健全」と判定したインスタンスを自動終了 → 新インスタンス起動
  
[EC2 インスタンス群]
  ├─ インスタンス A: /health → HTTP 200 OK ✅ 正常
  ├─ インスタンス B: /health → HTTP 500 ERROR ❌ 不健全
  │   → ASG が終了 → 新インスタンス C を起動
  └─ インスタンス C: 新規起動
```

## たとえ話：レジ係の「健康チェック」

### EC2 ヘルスチェック
- **毎日、レジ係が「立ってるか・息してるか」だけ確認**
- 接客品質が下がってても（HTTP エラー）気づかない
- 結果：壊れたレジ係が顧客に当たり続ける

### ELB（ヘルスチェック単独）
- **店長が「実際に注文を処理できるか」テスト**
- テストで「ダメだ」とわかる
- でも「この係を外す」までしかしない
- 結果：レジ係は次々来る新規顧客に当たり続ける 😰

### ASG + ELB ヘルスチェック連動（推奨）
- **店長がテストで異常を検知**
- **その場で「この係は終了」と指示**
- **新しい係が流れ込んでくる**（Launch Template から起動）
- 結果：顧客は常に正常な係に当たる ✅

## SAA 試験での頻出パターン

### Q: 「HTTPエラーが発生しても、可用性を保ちたい」

**必須要件**：
1. ✅ ALB（ELB）を配置
2. ✅ ALB にヘルスチェックパス（例 `/health`）を設定
3. ✅ **Auto Scaling のヘルスチェックタイプを「ELB」に設定**（ここが重要）

**よくある誤り**：
- ❌ 「ALB にヘルスチェック設定したからいいだろう」
- ❌ 「EC2 ステータスだけ見れば十分」
- ❌ 「ロードバランサーがなんとかしてくれる」

---

## 設定例（概念図）

### ASG 側の設定
```
Health Check Type: ELB ← ★重要（既定は EC2）
Health Check Grace Period: 300秒（起動直後の待機時間）
```

### ALB 側の設定
```
Target Group
  ├─ Health Check Protocol: HTTP
  ├─ Health Check Path: /health
  ├─ Matcher (正常と判定するコード): 200
  ├─ Interval: 30秒
  └─ Timeout: 5秒
```

## SAA 着眼点

| 試験での出題 | 覚えるべきこと |
|---|---|
| 「HTTP エラーが起きてもサービス継続」 | **ASG のヘルスチェック = ELB** に設定 |
| 「アプリが落ちた時は自動置換」 | **ALB（ELB）のヘルスチェック結果を ASG が参照** する必要がある |
| 「インスタンスが生きてても HTTPエラー」 | **EC2 ヘルスチェックは HTTP を見ないので、この場合は使い分け必須** |
| 「ロードバランサーだけで十分？」 | ❌ ALB だけでは振り分けを止めるだけ。**自動置換には ASG との連動が必須** |

---

**要点**：健全性の定義は「**何の層を見るか**」で決まる。可用性要件が高ければ、アプリ層（HTTP）を見る ELB ヘルスチェックを ASG に設定するしかない。
