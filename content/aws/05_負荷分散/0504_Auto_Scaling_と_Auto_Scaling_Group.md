# Auto Scaling と Auto Scaling Group

## Auto Scaling とは

需要に合わせてリソースを自動で増減させる仕組み。
アクセスが増えたらサーバーを増やし、減ったら減らすことで、コスト効率と可用性を両立する。

---

## Auto Scaling Group (ASG) とは

EC2インスタンスをまとめて管理する**グループ（論理的な単位）**。
ASGが「このグループには常にN台のEC2を維持せよ」という役割を担う。

```
インターネット
    ↓
[ロードバランサー (ALB/ELB)]
    ↓
[Auto Scaling Group]
  ├── EC2 インスタンス A
  ├── EC2 インスタンス B  ← トラフィック増加 → 自動追加
  └── EC2 インスタンス C  ← トラフィック減少 → 自動削除
```

---

## ASGの3つの台数設定

| 設定 | 意味 | 例 |
|------|------|-----|
| **Minimum** | 最小台数（絶対下限） | 2台 |
| **Desired** | 通常時の目標台数 | 4台 |
| **Maximum** | 最大台数（絶対上限） | 10台 |

---

## スケーリングの種類

### スケールアウト / スケールイン（水平スケーリング）
- **スケールアウト** → インスタンスを**増やす**（負荷増大時）
- **スケールイン** → インスタンスを**減らす**（負荷低下時）

### スケーリングポリシーの種類

| ポリシー              | 説明                                               |
| ----------------- | ------------------------------------------------ |
| **シンプルスケーリング**    | CloudWatchアラームが発火したら+N台 or -N台                   |
| **ステップスケーリング**    | 負荷の程度に応じて段階的に増減                                  |
| **ターゲット追跡スケーリング** | CPU使用率70%など目標値を維持するよう自動調整（最もシンプル）エアコンの温度設定みたいなもの |
| **スケジュールスケーリング**  | 曜日・時間帯で事前に台数を指定（例: 毎朝9時に増やす）                     |

---

## Launch Template（起動テンプレート）

ASGがインスタンスを起動する際の**設計図**。以下を定義する：

- AMI（どのOSイメージを使うか）
- インスタンスタイプ（t3.micro など）
- セキュリティグループ
- キーペア
- ユーザーデータ（起動時に実行するスクリプト）

---

## ヘルスチェックとの連携

ASGはインスタンスのヘルスチェックを行い、**不健全なインスタンスを自動で置き換える**。

```
EC2インスタンスが異常検知
    ↓
ASGが自動的にそのインスタンスを終了
    ↓
新しいインスタンスをLaunch Templateに基づいて起動
    ↓
Desired台数を維持
```

### ヘルスチェックの種類
- **EC2ヘルスチェック** → インスタンスの状態（running/stopped など）
- **ELBヘルスチェック** → ロードバランサー経由でアプリの応答を確認（推奨）

---

## ALBとの組み合わせ（よくある構成）

```
ALB（Application Load Balancer）
    ↓  ターゲットグループに紐付け
Auto Scaling Group
    ↓  インスタンス自動増減
EC2インスタンス群
```

ASGとALBを組み合わせることで、増減したインスタンスへの**トラフィック振り分けも自動化**される。

関連: [[0502_Elastic Load Balancing]] [[0503_ALB・NLB・GWLBの違い]] [[0501_高可用性とスケーラビリティ]]

## コスト面のメリット

| 状況 | 固定台数運用 | ASG運用 |
|------|-------------|---------|
| 深夜（低負荷） | 10台分のコスト | 2台分のコスト |
| 昼間（高負荷） | 10台（余裕あり） | 10台（自動増加）|

## SAA試験ポイント

| 問われ方 | 答え |
|---|---|
| 「CPU使用率を50%に保ちたい」最も簡単な方法 | **ターゲット追跡スケーリング** |
| 「毎朝9時のアクセス急増に備えたい」 | **スケジュールスケーリング** |
| 異常なインスタンスを自動復旧したい | ASGの**ヘルスチェック**（ELBヘルスチェック推奨）による自動置き換え |
| マルチAZで最低2台を常に維持したい | ASGを**複数AZに配置**し Minimum=2 |
| ASGがEC2を起動する元になるものは？ | **起動テンプレート（Launch Template）**＋その中の**AMI** → [[0104_スナップショット・AMIとは]] |
| スケールイン時にセッションが消える | セッションを **ElastiCache / DynamoDB** に外出し → [[0505_スティッキーセッション]] |
