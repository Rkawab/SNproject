## 1. 一般用語の整理

### 可用性（Availability）

> **「システムが使える状態にある時間の割合」**

$$可用性 = \frac{稼働時間}{稼働時間 + 停止時間} \times 100\%$$

| 可用性 | 年間ダウンタイム | 呼び方 |
|--------|----------------|--------|
| 99% | 約87.6時間 | 2ナイン |
| 99.9% | 約8.7時間 | 3ナイン |
| 99.99% | 約52分 | 4ナイン |
| 99.999% | 約5分 | 5ナイン |

**高可用性（High Availability / HA）**とは、システムが障害・メンテナンスの影響を最小化し、継続して稼働し続ける設計思想。  
キーワード：**冗長化・フェイルオーバー・単一障害点（SPOF）の排除**

---

### スケーラビリティ（Scalability）

> **「需要の変化に応じてリソースを増減できる能力」**

| 種類 | 別名 | 内容 | アナロジー |
|------|------|------|-----------|
| **スケールアップ** | 垂直スケーリング | サーバー1台を強化（CPU・メモリ増強） | 軽トラ→大型トラックに乗り換え |
| **スケールアウト** | 水平スケーリング | サーバーの台数を増やす | 軽トラの台数を増やす |
| **スケールイン** | — | 台数を減らす（コスト削減） | 閑散期に台数を減らす |
| **スケールダウン** | — | 1台のスペックを下げる | 大型トラック→軽トラに戻す |

**エラスティシティ（Elasticity）**：スケールアウト・スケールインを自動で行う能力（クラウドの強み）

---

### 関連用語

| 用語 | 意味 |
|------|------|
| **フォールトトレランス** | 一部が故障しても処理を継続できる性質（可用性より強い概念） |
| **フェイルオーバー** | 障害発生時にスタンバイ機へ自動切り替えすること |
| **SPOF（単一障害点）** | そこが壊れると全体が止まるボトルネック。HAはSPOFを排除する設計 |
| **RTO** | 目標復旧時間（障害発生から復旧までの許容時間） |
| **RPO** | 目標復旧時点（どこまでのデータ損失を許容するか） |

## 2. AWSのインフラ構造

```
世界
└── リージョン（例：ap-northeast-1 東京）
    ├── AZ-a（アベイラビリティゾーン）
    ├── AZ-b
    └── AZ-c
        └── 各AZは独立した電源・ネットワーク・物理施設
```

- **リージョン**：地理的に離れた拠点（東京、大阪、バージニア等）
- **AZ**：1リージョン内の独立施設。AZ間は低遅延の専用線で接続
- **複数AZ展開** = 高可用性の基本

---

## 3. AWSにおける高可用性

> **「1か所が壊れても全体は止まらない」設計**

### アナロジー
新幹線が止まっても飛行機・在来線で移動できる。  
→ AWSも複数AZにリソースを分散し、「どこか1つが壊れても動き続ける」状態を作る。

### 主要サービス・手段

| 手段 | 仕組み |
|------|--------|
| **Multi-AZ構成** | RDS・ELBなどを複数AZに配置し、障害時に自動フェイルオーバー |
| **ELB（ロードバランサー）** | 正常なインスタンスだけにトラフィックを振り分ける |
| **Route 53ヘルスチェック** | 障害リージョンから正常リージョンへ自動DNS切り替え |
| **S3・DynamoDB** | AWS側が自動でMulti-AZ保存（ユーザーが意識不要） |
| **Auto Scaling（最小台数設定）** | インスタンスが落ちても最小台数を自動維持 |

## 4. AWSにおけるスケーラビリティ

> **「需要に合わせてリソースを増減できる」設計**

### Auto Scaling の流れ

```
需要急増
  ↓
CloudWatch が CPU使用率80%超を検知
  ↓
Auto Scaling が EC2 を自動追加
  ↓
ELB が新インスタンスにトラフィック分散
  ↓
需要落ち着く → 自動でインスタンス削減（コスト節約）
```

### スケーリングポリシーの種類

| ポリシー | 内容 |
|----------|------|
| **ターゲット追跡スケーリング** | 指標（例：CPU 50%）を目標に自動調整。最もシンプル |
| **ステップスケーリング** | 指標の変化幅に応じてステップで台数を変える |
| **スケジュールスケーリング** | 時間帯・曜日に合わせて事前にスケール |
| **予測スケーリング** | 過去のパターンをMLで学習し、事前にスケール |

## 5. 高可用性 × スケーラビリティの合わせ技（典型構成）

```
インターネット
      ↓
   Route 53
      ↓
  CloudFront（CDN・エッジキャッシュ）
      ↓
  ALB（Application Load Balancer）
   ↙              ↘
AZ-a                AZ-b
EC2 × n             EC2 × n   ← Auto Scaling で増減
   ↓                   ↓
  RDS Primary  ⇄  RDS Standby（Multi-AZ 自動フェイルオーバー）
```

| 目的 | 担当サービス |
|------|------------|
| 負荷分散 | ALB / NLB |
| 自動スケール | Auto Scaling Group |
| DB 冗長化 | RDS Multi-AZ |
| DNS 切り替え | Route 53 |
| グローバル配信 | CloudFront |
| キャッシュ（読み取り負荷軽減） | ElastiCache |

## 6. SAA 試験の重要ポイント

### 高可用性 vs スケーラビリティ の対応

| 目的 | キーワード | 代表サービス |
|------|-----------|-------------|
| 高可用性 | 冗長化・フェイルオーバー | RDS Multi-AZ, ELB, Route 53 |
| スケーラビリティ | 負荷分散・台数増減 | Auto Scaling, RDS Read Replica |

### よく出る引っかかりポイント

- **Single-AZ = 高可用性ではない**
- **RDS Read Replica** → スケーラビリティ（読み取り負荷分散）
- **RDS Multi-AZ** → 高可用性（冗長化・フェイルオーバー）。読み取りには使えない
- **スケールアップには上限がある** → だからスケールアウトが重要
- **ElastiCache** → DBへの読み取り負荷を減らす（スケーラビリティ補助）
- **ELB単体では高可用性にならない** → Multi-AZのEC2と組み合わせて初めて有効
