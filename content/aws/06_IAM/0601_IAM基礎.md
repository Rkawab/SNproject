# AWS IAM 基礎

## IAMとは

「誰が・何に・何をできるか」を管理するサービス。  
**Identity and Access Management** の略。

---

## たとえ話

会社の**社員証（誰か）＋権限バッジ（何ができるか）**の仕組みがIAM。  
社員証だけでは何もできず、必要な権限バッジをつけることで初めて操作可能になる。

---

## 主要な構成要素

### IAMユーザー（User）
- AWSアカウント内の「人」の単位
- アクセスキーまたはパスワードで認証
- **ルートユーザーは日常使いしない**（全権限を持つ社長IDのイメージ）

### IAMグループ（Group）
- 複数ユーザーをまとめる「部署」
- グループにポリシーをアタッチ → メンバー全員に権限が適用される
- グループのネスト（グループ内にグループ）は**不可**

### IAMロール（Role）
- 「人」ではなく **AWSサービスや外部システム** に権限を与える
- たとえ：EC2がS3を読みたいとき → EC2にロールをアタッチ
- **アクセスキーのハードコードは不要**（セキュリティのベストプラクティス）
- 「ロールを引き受ける（Assume Role）」が基本の動作

### IAMポリシー（Policy）
- JSON形式で「何を許可・拒否するか」を定義
- ユーザー・グループ・ロールにアタッチして使う

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

| ポリシー種類 | 説明 |
|---|---|
| AWS管理ポリシー | AWSが用意した汎用ポリシー（例: AmazonS3ReadOnlyAccess） |
| カスタマー管理ポリシー | 自作・再利用可能 |
| インラインポリシー | ユーザー/ロールに直接埋め込み・再利用不可 |

---

## 重要な概念

### 最小権限の原則（Least Privilege）
必要最低限の権限だけ付与する。「とりあえず全部許可」はNG。

### 権限の評価順序
1. **明示的なDeny → 即拒否（最優先）**
2. 明示的なAllow → 許可
3. どちらもなし → 暗黙的にDeny（デフォルト拒否）

### MFA（多要素認証）
- パスワード＋ワンタイムコードの2段階認証
- ルートユーザー・管理者ユーザーには必須レベル

### STS（Security Token Service）
- 一時的な認証情報（アクセスキー＋有効期限付き）を発行
- `AssumeRole` で別アカウントや外部IDPのユーザーに一時権限を与える

---

## SAA試験でよく出るポイント

| ポイント | 内容 |
|---|---|
| EC2からS3へのアクセス | ロールを使う（アクセスキー埋め込みはNG） |
| クロスアカウントアクセス | ロール＋Assume Roleで実現 |
| フェデレーション | 社内AD/IdPとIAMを連携（SAML/OIDC） |
| Permission Boundary | ユーザーに付与できる権限の上限を設定（委任時の安全弁） |
| SCPs（Service Control Policies） | AWS Organizationsで複数アカウントを一括制御 |
| ポリシーのJSON構造 | Effect / Action / Resource / Condition を読み取れるようにする |
