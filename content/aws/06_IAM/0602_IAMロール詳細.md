# AWS IAMロール 詳細

## IAMロールとは（本質）

IAMロールは **「一時的に身にまとう権限の衣装」**。
ユーザーやグループが「所有する」永続的な権限なのに対し、ロールは **「一時的に引き受ける（Assume）」** 権限。

### IAMユーザーとの決定的な違い

| 項目 | IAMユーザー | IAMロール |
|---|---|---|
| 帰属 | 特定の「人」 | 誰でも引き受け可能（信頼関係次第） |
| 認証情報 | アクセスキー・パスワード（永続） | 一時的なトークン（STS発行） |
| 有効期限 | なし（手動失効） | あり（15分〜12時間） |
| 主な用途 | 人間がCLI/コンソールで使う | サービス・他アカウント・フェデレーション |

---

## たとえ話：「派遣社員の入館証」

### IAMユーザー＝正社員の社員証
- 自分専用、ずっと使える
- 紛失したら即危険（永続的）

### IAMロール＝来客用の一時パス
- 受付（STS）で「私はAさんの代わりに会議室に入ります」と申請
- 受付が「2時間有効のパス」を発行
- 時間が来たら自動失効
- **誰が引き受けるか**は事前に登録された「信頼ポリシー」で決まる

```
[EC2インスタンス]──「S3を読みたい」──→[STS]
                                          │
                                          ↓
                                   一時的な認証情報を発行
                                   （アクセスキー＋秘密鍵＋トークン）
                                          │
                                          ↓
                                   [S3にアクセス可能]
```

---

## ロールの2つのポリシー（超重要）

ロールには **2種類のポリシー** がアタッチされる。SAA試験で頻出。

### (1) 信頼ポリシー（Trust Policy）= **誰が引き受けられるか**
「このロールを **Assume できる人/サービス** は誰か」を定義。

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "ec2.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}
```
→ EC2サービスがこのロールを引き受けられる、という意味。

### (2) アクセス許可ポリシー（Permissions Policy）= **何ができるか**
「このロールを引き受けた人は **何を操作できるか**」を定義。
通常のIAMポリシー（Effect/Action/Resource）と同じ構造。

> [!info] ドアの例え
> 信頼ポリシー＝鍵穴に合う鍵を持つ人を決める
> 許可ポリシー＝部屋の中で何ができるかを決める

## 典型的なユースケース（SAA頻出パターン）

### ケース①：EC2からS3アクセス（**最頻出**）
```
EC2インスタンス
  └ IAMロールをアタッチ（インスタンスプロファイル経由）
     └ アプリは自動的にメタデータから一時認証情報を取得
        → S3 API呼び出し成功
```
**ポイント**: アクセスキーをコードに書かない（漏洩防止）

### ケース②：クロスアカウントアクセス
```
[アカウントA: 開発環境] ──→ [アカウントB: 本番環境のS3]

アカウントBで:
  - ロール作成（信頼ポリシーでアカウントAを許可）
  - 許可ポリシーでS3:GetObjectを許可

アカウントAのユーザーが:
  - sts:AssumeRole でアカウントBのロールを引き受ける
  - 一時認証情報でアカウントBのS3へアクセス
```

### ケース③：フェデレーション（外部ID連携）
| 種類 | 用途 |
|---|---|
| SAML 2.0 | 社内ActiveDirectory・社員アカウントと連携 |
| OIDC | Google/Facebookログイン・GitHub Actions |
| Cognito | スマホアプリのユーザー認証 |

社員はAWSアカウントを持たずに、社内IDでAWSにログイン → ロールを引き受ける

### ケース④：Lambda・ECS等のサービスロール
- Lambda関数の実行ロール → CloudWatch Logsへの書き込み権限など
- ECSタスクロール → タスクごとに権限を分離

---

## インスタンスプロファイル（よく勘違いされる）

EC2にロールをアタッチする際、**実は「インスタンスプロファイル」というラッパー** を介している。

```
EC2 ── インスタンスプロファイル ── IAMロール ── ポリシー
```

- マネジメントコンソールで操作すると自動で作成される
- CloudFormationやCLIだと **明示的に作成が必要**（試験で問われる）

---

## STS（Security Token Service）との関係

ロールを「引き受ける」の正体は **STSのAPI呼び出し**。

| API | 用途 |
|---|---|
| `AssumeRole` | IAMユーザーが別ロールを引き受け（クロスアカウントなど） |
| `AssumeRoleWithSAML` | SAMLフェデレーション |
| `AssumeRoleWithWebIdentity` | OIDC（Google・Cognitoなど） |
| `GetSessionToken` | MFA認証付きの一時トークン |

**返却されるもの**: `AccessKeyId`・`SecretAccessKey`・`SessionToken`・`Expiration`

## SAA試験で狙われる罠

> [!warning] こんな問題が出る

### 罠①：「アクセスキーを設定ファイルに書いた」
→ **間違い**。EC2ならIAMロールを使う。

### 罠②：「ユーザーをグループに入れて権限を与えたい、でもサービスにも同じ権限が必要」
→ サービスにはロール、ユーザーにはグループ。**サービスにグループは使えない**。

### 罠③：「クロスアカウントでバケットにアクセスしたい」
→ **2つのアプローチ**がある
- **IAMロール方式**: アカウントBにロール作成 → アカウントAがAssumeRole（推奨）
- **バケットポリシー方式**: S3側でアカウントAを許可（簡易だが管理が分散）

### 罠④：「Permission Boundary」と「SCP」の違い
| 機能 | 適用範囲 | 設定者 |
|---|---|---|
| Permission Boundary | 個別IAMユーザー/ロール | アカウント管理者 |
| SCP（サービスコントロールポリシー） | アカウント全体 | Organizations管理者 |

両方とも **「上限」を設定する**（許可ではなく制限）のがポイント。

## ベストプラクティスまとめ

> [!tip] 覚えておくべき原則
> 1. **アクセスキー埋め込みは絶対NG** → ロール使用
> 2. **最小権限の原則** → 必要なActionだけ許可
> 3. **MFA併用** → 特権ロールはMFA必須条件付き
> 4. **ロールの有効期限は短く** → 漏洩時の被害を最小化
> 5. **CloudTrailで監査** → 誰がいつAssumeRoleしたか追跡
