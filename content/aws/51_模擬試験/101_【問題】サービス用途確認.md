# 【問題】サービス用途確認（4択・全20問）

## この問題集の目的

主要AWSサービスの「基本用途」を、シナリオ（場面）で結びつけて覚える。サービス名を見たときに「何のためのサービスか」「どの場面で使うか」を即答できるようにする、最初の一歩。

## 対象レベル

初心者向け（SAA入門の前段）。用語の丸暗記ではなく「この要件ならこのサービス」を選ぶ練習。

## 問題数

20問（1サービス＝1問。S3 / EC2 / RDS / IAM / VPC / CloudWatch / CloudTrail / Lambda / ALB / Auto Scaling / Route 53 / CloudFront / EBS / EFS / DynamoDB / SQS / SNS / EventBridge / KMS / Secrets Manager）

## 使い方

まず選択肢を見ずに「自分なら何を使うか」を考える。回答後は、正解の理由だけでなく**不正解の選択肢がなぜ違うか**まで読む。迷ったら 📎 のノートに戻る。

## 第1問（ストレージ）
画像・動画・バックアップなどの大量ファイルを、容量を気にせず高い耐久性で安価に保存したい。最初に検討すべきサービスはどれか。

- A. Amazon S3
- B. Amazon EBS
- C. Amazon EFS
- D. Amazon RDS

> [!success]- 答え：**A**（S3）
> オブジェクトストレージで容量は実質無制限、耐久性99.999999999%（イレブンナイン）。静的ファイル・バックアップ・ログの「倉庫」として真っ先に候補になる。
> **他の選択肢が違う理由**
> - B：EBSはEC2に1台ずつ接続するブロックストレージ（ディスク）。単独の保管庫用途ではない。
> - C：EFSは複数サーバーで共有するファイルシステム（NAS）。大量ファイルの安価な保管が主目的ではない。
> - D：RDSはリレーショナルDB。ファイル置き場ではない。
> **覚えるべきキーワード**：オブジェクトストレージ / 容量無制限 / 高耐久（11ナイン）
> **関連サービス**：CloudFront（前段CDN） / S3 Glacier（アーカイブ）
> **SAAでの出題イメージ**：「大量の静的ファイル/バックアップを安く高耐久に」と来たらS3。
> 📎 参照：[[0201_S3]] / [[0203_S3ストレージクラス]]

## 第2問（コンピューティング）
OS・ミドルウェア・ソフト構成を自分で自由に設定できる仮想サーバーを、必要なときに起動したい。最も適切なサービスはどれか。

- A. AWS Lambda
- B. Amazon EC2
- C. Amazon S3
- D. Amazon RDS

> [!success]- 答え：**B**（EC2）
> EC2はAWS上の仮想サーバー。OSへログインしてミドルウェアやアプリを自由に構成できる「フルコントロール」が特徴。
> **他の選択肢が違う理由**
> - A：Lambdaはサーバー管理不要のサーバーレス。OSを自由に触る用途には向かない。
> - C：S3はストレージで計算処理はできない。
> - D：RDSはマネージドDBでOSへログインできない。
> **覚えるべきキーワード**：仮想サーバー / フルコントロール / インスタンス
> **関連サービス**：EBS（ディスク） / Auto Scaling（台数自動増減） / AMI（テンプレート）
> **SAAでの出題イメージ**：「OSレベルの細かい設定が必要」「既存アプリをそのまま動かす」ならEC2。
> 📎 参照：[[0101_EC2やVPCとは]] / [[0102_EC2購入プラン]]

## 第3問（データベース）
MySQLやPostgreSQLを使いたいが、パッチ適用・バックアップ・冗長化などの運用作業はできるだけ任せたい。最も適切なサービスはどれか。

- A. EC2にMySQLを自前インストール
- B. Amazon RDS
- C. Amazon DynamoDB
- D. Amazon S3

> [!success]- 答え：**B**（RDS）
> RDSはリレーショナルDBのマネージドサービス。パッチ・バックアップ・Multi-AZ冗長化・リードレプリカなどをAWSが面倒を見てくれる。
> **他の選択肢が違う理由**
> - A：EC2自前構築は可能だが、パッチ・バックアップ・冗長化を全部自分で運用する必要があり要件に反する。
> - C：DynamoDBはNoSQL（キー値型）。SQLのリレーショナルDBではない。
> - D：S3はストレージでDBではない。
> **覚えるべきキーワード**：マネージドRDB / 運用負荷の削減 / Multi-AZ・リードレプリカ
> **関連サービス**：Aurora（高性能版） / RDS Proxy（接続プール）
> **SAAでの出題イメージ**：「RDB＋運用を楽にしたい」ならRDS（運用まで自前ならEC2）。
> 📎 参照：[[0301_RDS]]

## 第4問（認証・認可）
「誰が」「どのAWSリソースに」「何をできるか」を細かく制御し、ユーザーやグループ、権限を管理したい。適切なサービスはどれか。

- A. AWS IAM
- B. Amazon Cognito
- C. AWS KMS
- D. AWS Shield

> [!success]- 答え：**A**（IAM）
> IAMはAWSアカウント内の認証・認可の中核。ユーザー/グループ/ロールにポリシーを付与し、AWSリソースへのアクセス権限を制御する。
> **他の選択肢が違う理由**
> - B：Cognitoは「アプリのエンドユーザー（会員）」の認証用。AWSリソースの管理権限制御ではない。
> - C：KMSは暗号鍵の管理サービス。
> - D：ShieldはDDoS防御。
> **覚えるべきキーワード**：ユーザー/グループ/ロール / ポリシー / 最小権限
> **関連サービス**：IAMロール（サービスへの権限委譲） / Organizations（複数アカウント統制）
> **SAAでの出題イメージ**：「AWSリソースへの権限を誰に与えるか」ならIAM。アプリ会員の認証はCognito。
> 📎 参照：[[0601_IAM基礎]] / [[0602_IAMロール詳細]]

## 第5問（ネットワーク）
AWS上に、自社専用に論理的に分離した仮想ネットワーク空間を作り、その中にサブネットを切ってサーバーを配置したい。適切なサービスはどれか。

- A. Amazon Route 53
- B. Amazon VPC
- C. AWS Direct Connect
- D. Amazon CloudFront

> [!success]- 答え：**B**（VPC）
> VPCはAWS上に作る「自分専用のネットワーク」。サブネット・ルートテーブル・ゲートウェイ・セキュリティグループなどで通信を設計する。
> **他の選択肢が違う理由**
> - A：Route 53はDNS（名前解決）。ネットワーク空間そのものは作らない。
> - C：Direct ConnectはオンプレとAWSを専用線で結ぶ接続サービス。
> - D：CloudFrontはCDN（コンテンツ配信）。
> **覚えるべきキーワード**：仮想ネットワーク / サブネット / ルートテーブル
> **関連サービス**：サブネット / インターネットゲートウェイ / NAT Gateway
> **SAAでの出題イメージ**：「ネットワークを分離・設計する」土台は常にVPC。
> 📎 参照：[[0101_EC2やVPCとは]] / [[0406_VPC応用機能]]

## 第6問（監視）
EC2のCPU使用率などのメトリクスを収集し、しきい値を超えたらアラームで通知したい。適切なサービスはどれか。

- A. AWS CloudTrail
- B. Amazon CloudWatch
- C. AWS Config
- D. Amazon Inspector

> [!success]- 答え：**B**（CloudWatch）
> CloudWatchはメトリクス収集・アラーム・ダッシュボード・ログ集約を担う「性能の見張り役」。アラームからAuto ScalingやSNS通知にもつなげられる。
> **他の選択肢が違う理由**
> - A：CloudTrailは「誰がどのAPI操作をしたか」の監査ログ。性能メトリクスではない。
> - C：Configはリソース構成の変更履歴・準拠チェック。
> - D：InspectorはEC2等の脆弱性診断。
> **覚えるべきキーワード**：メトリクス / アラーム / ダッシュボード / Logs
> **関連サービス**：SNS（通知） / Auto Scaling（連動） / CloudWatch Logs
> **SAAでの出題イメージ**：「性能を監視してしきい値で動かす」ならCloudWatch。
> 📎 参照：[[1001_CloudWatch]]

## 第7問（監査）
「いつ・誰が・どのAPIを呼び出して何を操作したか」をアカウント全体で記録し、後から追跡できるようにしたい。適切なサービスはどれか。

- A. Amazon CloudWatch
- B. AWS CloudTrail
- C. VPCフローログ
- D. AWS Config

> [!success]- 答え：**B**（CloudTrail）
> CloudTrailはAWS API呼び出しの「操作の足跡（証跡）」を記録する。セキュリティ調査・コンプライアンス・原因追跡の定番。
> **他の選択肢が違う理由**
> - A：CloudWatchは性能メトリクスの監視。操作者の記録ではない。
> - C：VPCフローログはネットワーク通信（IP/ポート）の記録で、API操作者の記録ではない。
> - D：Configはリソース構成の状態・変更履歴（誰がAPIを叩いたかが主目的ではない）。
> **覚えるべきキーワード**：API操作履歴 / 証跡（Trail） / 監査
> **関連サービス**：CloudWatch（性能） / Config（構成） / GuardDuty（脅威検知）
> **SAAでの出題イメージ**：「誰が操作したかを追跡・監査」ならCloudTrail。
> 📎 参照：[[1002_CloudTrail]]

## 第8問（サーバーレス）
短時間で終わる処理を、サーバーを用意・管理せずイベントで自動実行し、実行時間分だけ課金されるようにしたい。適切なサービスはどれか。

- A. Amazon EC2
- B. AWS Lambda
- C. AWS Batch
- D. Amazon ECS on EC2

> [!success]- 答え：**B**（Lambda）
> Lambdaはコードをアップするだけで動くサーバーレス。イベント駆動・ミリ秒課金・インフラ管理不要が3大特徴。最大実行時間は15分。
> **他の選択肢が違う理由**
> - A：EC2は常時起動のサーバーで、サーバー管理と起動時間課金が発生する。
> - C：Batchは長時間バッチジョブ向け。短いイベント処理には過剰。
> - D：ECS on EC2はホスト管理が残り、サーバーレスではない。
> **覚えるべきキーワード**：サーバーレス / イベント駆動 / 実行時間課金 / 15分上限
> **関連サービス**：S3イベント通知 / API Gateway / Step Functions
> **SAAでの出題イメージ**：「サーバー管理なし・短時間・イベント駆動」ならLambda。
> 📎 参照：[[0702_Lambda・ECS・RDS・Beanstalk比較]]

## 第9問（負荷分散）
複数のEC2に対してHTTP/HTTPSリクエストを振り分け、URLパスごとに転送先を変えたい。適切なサービスはどれか。

- A. Application Load Balancer（ALB）
- B. Network Load Balancer（NLB）
- C. Amazon Route 53
- D. Amazon CloudFront

> [!success]- 答え：**A**（ALB）
> ALBはレイヤー7（HTTP/HTTPS）のロードバランサー。パスベース/ホストベースのルーティングやコンテナ・Lambdaへの振り分けが得意。
> **他の選択肢が違う理由**
> - B：NLBはレイヤー4（TCP/UDP）。超低遅延・固定IP向けで、パスベース振り分けはできない。
> - C：Route 53はDNSレベルの分散で、HTTPパスを見た振り分けはしない。
> - D：CloudFrontはCDN。配信の最適化が主目的。
> **覚えるべきキーワード**：L7 / パスベースルーティング / HTTP/HTTPS
> **関連サービス**：Auto Scaling / ターゲットグループ / NLB（L4）
> **SAAでの出題イメージ**：「Webアプリ（HTTP）を複数台へ・パスで振り分け」ならALB。
> 📎 参照：[[0502_Elastic Load Balancing]] / [[0503_ALB・NLB・GWLBの違い]]

## 第10問（スケーリング）
昼はアクセスが多く夜は少ないWebアプリで、負荷に応じてEC2の台数を自動で増減し、コストと可用性を両立したい。適切なサービスはどれか。

- A. AWS Auto Scaling（EC2 Auto Scaling）
- B. Elastic Load Balancing
- C. Amazon CloudFront
- D. AWS Lambda

> [!success]- 答え：**A**（Auto Scaling）
> EC2 Auto Scalingは需要に応じてインスタンス台数を自動で増減（スケールアウト/イン）する。閑散時はコスト削減、ピーク時は不足を防ぐ。
> **他の選択肢が違う理由**
> - B：ELBは「トラフィックの振り分け」担当で、台数を増減はしない（ペアで使う）。
> - C：CloudFrontはキャッシュ配信でEC2台数制御ではない。
> - D：Lambdaは別のサーバーレス実行基盤で、EC2の台数調整ではない。
> **覚えるべきキーワード**：スケールアウト/イン / 台数自動増減 / 希望/最小/最大台数
> **関連サービス**：ALB（併用） / CloudWatch（トリガー） / 起動テンプレート
> **SAAでの出題イメージ**：「負荷に応じてEC2台数を自動調整」ならAuto Scaling。振り分けはELB。
> 📎 参照：[[0504_Auto_Scaling_と_Auto_Scaling_Group]] / [[0501_高可用性とスケーラビリティ]]

## 第11問（DNS）
独自ドメインの名前解決を行い、加重・レイテンシ・フェイルオーバーなどのルーティングを設定したい。適切なサービスはどれか。

- A. Amazon CloudFront
- B. Amazon Route 53
- C. AWS Global Accelerator
- D. Amazon VPC

> [!success]- 答え：**B**（Route 53）
> Route 53はマネージドDNS。ドメイン名→IPの解決に加え、加重・レイテンシ・位置情報・フェイルオーバーなど多彩なルーティングポリシーとヘルスチェックを持つ。
> **他の選択肢が違う理由**
> - A：CloudFrontはCDN（配信）であり名前解決サービスではない。
> - C：Global AcceleratorはAWSグローバル網で通信を高速化するが、DNSルーティングポリシーの設定が主目的ではない。
> - D：VPCはネットワーク空間で名前解決サービスではない。
> **覚えるべきキーワード**：DNS / ルーティングポリシー / ヘルスチェック
> **関連サービス**：加重ルーティング / フェイルオーバー / ALIASレコード
> **SAAでの出題イメージ**：「ドメインの名前解決とトラフィック制御」ならRoute 53。
> 📎 参照：[[0404_Route53]]

## 第12問（コンテンツ配信）
世界中のユーザーへ静的・動的コンテンツを低遅延で配信し、オリジン（S3やALB）の負荷も下げたい。適切なサービスはどれか。

- A. Amazon CloudFront
- B. Amazon S3 Transfer Acceleration
- C. Elastic Load Balancing
- D. Amazon Route 53

> [!success]- 答え：**A**（CloudFront）
> CloudFrontは世界中のエッジロケーションでコンテンツをキャッシュ配信するCDN。ユーザーに近い場所から返すことで低遅延化し、オリジン負荷も軽減する。
> **他の選択肢が違う理由**
> - B：S3 Transfer AccelerationはS3への「アップロード」高速化機能で、一般的なコンテンツ配信CDNではない。
> - C：ELBはリージョン内の負荷分散で、グローバルなキャッシュ配信ではない。
> - D：Route 53は名前解決。配信のキャッシュはしない。
> **覚えるべきキーワード**：CDN / エッジロケーション / キャッシュ / 低遅延配信
> **関連サービス**：S3（オリジン） / WAF（エッジで保護） / Global Accelerator（非キャッシュ高速化）
> **SAAでの出題イメージ**：「静的/動的コンテンツをグローバルに低遅延配信」ならCloudFront。
> 📎 参照：[[0405_ネットワークとコンテンツ配信]]

## 第13問（ブロックストレージ）
EC2にアタッチして、OSやデータベースのデータを置く「ディスク」として使う、永続的なブロックストレージが欲しい。適切なサービスはどれか。

- A. Amazon S3
- B. Amazon EBS
- C. インスタンスストア
- D. Amazon EFS

> [!success]- 答え：**B**（EBS）
> EBSはEC2に接続して使う永続ブロックストレージ（仮想ディスク）。インスタンスを停止してもデータは残り、スナップショットでバックアップできる。
> **他の選択肢が違う理由**
> - A：S3はオブジェクトストレージで、OSのディスク（ブロック）としては使わない。
> - C：インスタンスストアは物理ホスト内蔵の一時ディスクで、停止すると消える（揮発性）。
> - D：EFSは共有ファイルシステムで、用途が「1台のディスク」とは異なる。
> **覚えるべきキーワード**：ブロックストレージ / 永続 / スナップショット
> **関連サービス**：EC2 / スナップショット（S3保管） / gp3・io2（ボリュームタイプ）
> **SAAでの出題イメージ**：「EC2の永続ディスク」ならEBS。揮発でよければインスタンスストア。
> 📎 参照：[[0103_EC2のストレージ]]

## 第14問（共有ファイルストレージ）
複数のLinux EC2から、同じファイルシステムを同時にマウントして共有したい。適切なサービスはどれか。

- A. Amazon EBS
- B. Amazon EFS
- C. Amazon S3
- D. インスタンスストア

> [!success]- 答え：**B**（EFS）
> EFSはNFSベースのフルマネージド共有ファイルシステム。複数のEC2から同時マウントでき、容量は自動で伸縮する。
> **他の選択肢が違う理由**
> - A：EBSは原則1インスタンスに接続するディスク（Multi-Attachは特殊・限定的）。
> - C：S3はオブジェクトストレージで、POSIX互換のファイルシステムマウントが本来の用途ではない。
> - D：インスタンスストアはホスト内蔵の一時ディスクで共有できない。
> **覚えるべきキーワード**：共有ファイルシステム / NFS / 複数同時マウント / 自動伸縮
> **関連サービス**：EC2 / EBS（占有ディスク） / S3（オブジェクト）
> **SAAでの出題イメージ**：「複数サーバーで同じファイルを共有」ならEFS。
> 📎 参照：[[0202_S3とEBS・EFSの違い]]

## 第15問（NoSQL）
キー値アクセスで1桁ミリ秒の応答が求められ、トラフィックが急増しても自動でスケールする、サーバー管理不要のデータベースが欲しい。適切なサービスはどれか。

- A. Amazon RDS
- B. Amazon DynamoDB
- C. Amazon Redshift
- D. Amazon Aurora

> [!success]- 答え：**B**（DynamoDB）
> DynamoDBはフルマネージドのキー値型NoSQL。1桁ミリ秒の低遅延、オンデマンドキャパシティで急なスパイクにも自動対応する。
> **他の選択肢が違う理由**
> - A：RDSはリレーショナルDBで、サーバー（インスタンス）管理が残りスパイク追従も弱い。
> - C：Redshiftは分析（OLAP）向けのデータウェアハウス。
> - D：AuroraもRDBで高性能だが、「キー値・サーバーレスで急増対応」の定番解はDynamoDB。
> **覚えるべきキーワード**：NoSQL / キー値 / 1桁ミリ秒 / オンデマンド
> **関連サービス**：DAX（マイクロ秒キャッシュ） / グローバルテーブル / Lambda
> **SAAでの出題イメージ**：「キー値・超低遅延・サーバーレス・スパイク」ならDynamoDB。
> 📎 参照：[[0303_DynamoDB]]

## 第16問（メッセージキュー）
注文処理システムで、急な大量リクエストをいったん受け止めて、後段の処理を自分のペースで進めたい（疎結合・負荷の平準化）。適切なサービスはどれか。

- A. Amazon SQS
- B. Amazon SNS
- C. AWS Lambda
- D. Amazon API Gateway

> [!success]- 答え：**A**（SQS）
> SQSはメッセージをいったん「キュー（行列）」にためて、後段が取り出して処理する。急な負荷をバッファし、コンポーネントを疎結合にする。
> **他の選択肢が違う理由**
> - B：SNSは1対多の即時通知（Pub/Sub）で、ためて平準化する用途ではない。
> - C：Lambdaは処理の実行側で、バッファ（行列）そのものではない。
> - D：API Gatewayはリクエストの入口で、キューではない。
> **覚えるべきキーワード**：キュー / 疎結合 / バッファリング / 負荷平準化
> **関連サービス**：SNS（ファンアウト） / Lambda（コンシューマ） / FIFOキュー
> **SAAでの出題イメージ**：「急な負荷を受け止めて非同期処理・疎結合」ならSQS。
> 📎 参照：[[0802_SQS]]

## 第17問（通知・Pub/Sub）
1件のイベントを、メール・SMS・複数のLambda・複数のSQSへ同時にファンアウト（一斉配信）したい。適切なサービスはどれか。

- A. Amazon SQS
- B. Amazon SNS
- C. Amazon SES
- D. AWS Step Functions

> [!success]- 答え：**B**（SNS）
> SNSはPub/Sub型の通知サービス。1つのトピックに発行したメッセージを、購読している複数の宛先（メール/SMS/Lambda/SQS等）へ同時配信する。
> **他の選択肢が違う理由**
> - A：SQSは1対1でためて処理するキューで、一斉配信（ファンアウト）が主目的ではない。
> - C：SESはメール送信特化のサービス（マーケメール/通知メール）。多宛先ファンアウトの汎用Pub/Subではない。
> - D：Step Functionsはワークフローのオーケストレーション。
> **覚えるべきキーワード**：Pub/Sub / トピック / ファンアウト / 即時配信
> **関連サービス**：SQS（SNS→複数SQSの定番構成） / Lambda / SES（メール特化）
> **SAAでの出題イメージ**：「1イベントを複数宛先へ即時に一斉配信」ならSNS。
> 📎 参照：[[0803_SNS]]

## 第18問（イベントルーティング）
AWSサービスやSaaSから発生する様々なイベントを、ルールに基づいて適切な宛先へ振り分けたり、cronのようにスケジュール実行したりしたい。適切なサービスはどれか。

- A. Amazon SQS
- B. Amazon EventBridge
- C. AWS CloudTrail
- D. Amazon Kinesis Data Streams

> [!success]- 答え：**B**（EventBridge）
> EventBridgeはイベントバス。多数のAWSサービス/SaaSのイベントをルールでフィルタ・ルーティングし、Lambda等へ届ける。スケジュール（cron）実行にも使える。
> **他の選択肢が違う理由**
> - A：SQSは単純なメッセージキューで、ルールベースのイベントルーティングやスケジュール機能はない。
> - C：CloudTrailは操作の監査ログ記録。イベントの配信基盤ではない。
> - D：Kinesisは大量ストリーミングデータのリアルタイム処理向け。イベントのルール振り分けが主目的ではない。
> **覚えるべきキーワード**：イベントバス / ルール / スケジュール（cron） / SaaS連携
> **関連サービス**：Lambda / Step Functions / SNS・SQS
> **SAAでの出題イメージ**：「多様なイベントをルールで振り分け/定期実行」ならEventBridge。
> 📎 参照：[[0804_EventBridge]]

## 第19問（暗号鍵管理）
保存データを暗号化するための鍵を一元管理し、ローテーションやアクセス制御、CloudTrailによる使用監査を行いたい。適切なサービスはどれか。

- A. AWS KMS
- B. AWS Secrets Manager
- C. AWS Certificate Manager（ACM）
- D. AWS WAF

> [!success]- 答え：**A**（KMS）
> KMSは暗号鍵（CMK/KMSキー）の作成・管理・ローテーション・アクセス制御を担う。S3・EBS・RDS等の暗号化（SSE-KMS等）と統合され、鍵の使用はCloudTrailで監査できる。
> **他の選択肢が違う理由**
> - B：Secrets ManagerはDBパスワードやAPIキーなどの「シークレット（秘密情報）」管理。暗号鍵そのものの管理基盤はKMS。
> - C：ACMはTLS/SSL証明書の発行・管理。
> - D：WAFはWebアプリ攻撃のフィルタ。
> **覚えるべきキーワード**：暗号鍵管理 / ローテーション / SSE-KMS / 使用監査
> **関連サービス**：Secrets Manager（鍵でシークレット暗号化） / S3/EBS/RDS暗号化
> **SAAでの出題イメージ**：「暗号鍵の集中管理・ローテーション・監査」ならKMS。
> 📎 参照：[[1605_データ保護・暗号化]]

## 第20問（シークレット管理）
データベースのパスワードやサードパーティAPIキーを安全に保管し、定期的に自動ローテーションして、アプリから安全に取得したい。適切なサービスはどれか。

- A. AWS KMS
- B. AWS Secrets Manager
- C. 環境変数に平文で保存
- D. Amazon S3に平文で保存

> [!success]- 答え：**B**（Secrets Manager）
> Secrets ManagerはDB認証情報やAPIキーなどの秘密情報を暗号化保管し、RDS等と連携した自動ローテーション、アプリからのAPI取得を提供する。
> **他の選択肢が違う理由**
> - A：KMSは暗号鍵の管理基盤で、シークレット値そのものの保管・ローテーション機能を提供するのはSecrets Manager（内部でKMSの鍵を利用）。
> - C：環境変数に平文は漏洩リスクが高く非推奨。
> - D：S3に平文保存は暗号化・ローテーション・アクセス制御の観点で不適切。
> **覚えるべきキーワード**：シークレット管理 / 自動ローテーション / 暗号化保管
> **関連サービス**：KMS（暗号化） / RDS（ローテーション連携） / Systems Manager Parameter Store（簡易な代替）
> **SAAでの出題イメージ**：「DBパスワード等を安全に保管・自動ローテーション」ならSecrets Manager。
> 📎 参照：[[1605_データ保護・暗号化]] / [[1405_Systems Manager]]
