## 第1問（Containers ｜ マルチテナントEKSでのワークロード完全分離）
ある医療データ分析企業は、Amazon EKSの単一クラスター上で、複数の顧客向けアプリケーションをホストする必要があります。コンプライアンス要件により、各顧客のワークロードはCPU、メモリ、ストレージ、ENIを他の顧客と共有せず、独自の環境で実行されなければなりません。また、インフラ管理コストを抑える必要があります。

この要件を最も効率的に満たすAWSソリューションはどれですか。

- A. EKSとDedicated Hosts上で動作するEC2ワーカーノードを使用して、各顧客のワークロードを分離する。
- B. EKSとFargateを使用し、Fargateが提供する専用リソース環境で各ポッドを分離した環境で実行する。
- C. EC2インスタンス上にユーザーが一から構築したKubernetesクラスターを使用し、顧客ごとにノードを分けてワークロードを分離する。
- D. EKSとEC2ワーカーノードを使用し、Horizontal Pod AutoscalerとCluster Autoscalerで各顧客のポッドを分離する。

> [!success]- 答え：**B**
> EKS on Fargateは、ポッドごとに専用のマイクロVM（独立したCPU・メモリ・カーネル・ENI）を割り当てて実行する。つまり顧客のポッド同士がハードウェア/カーネルレベルで分離され、しかもワーカーノードのプロビジョニング・パッチ・スケールをAWSが受け持つため、インフラ管理コストも最小化できる。「完全分離」と「低い運用負荷」を同時に満たす唯一の選択肢。
> - A：Dedicated Hostsは物理サーバーを専有する仕組みで、顧客単位に物理ホストを割り当てれば分離はできるが、ホストの確保・容量管理・コストが重く「インフラ管理コストを抑える」に反する。粒度も過剰。
> - B：正解。Fargateはポッド単位に専用リソース環境（マイクロVM）を割り当て、リソース非共有の分離とサーバーレスな低運用を両立する。
> - C：EC2上に自前でKubernetesを構築すると、コントロールプレーンからノードまで全て自己管理になり、運用負荷・コストが最大。「インフラ管理コストを抑える」と正反対。
> - D：HPA/Cluster Autoscalerはスケーリングの仕組みであって、ワークロードの分離（リソース非共有）を保証するものではない。共有EC2ノード上に複数顧客のポッドが同居しうるため要件を満たさない。
> **関連ノート**：[[0906_EKS]] / [[0901_AWS_コンテナサービス概要]] / [[0904_ECS]]
> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第1問（Containers ｜ マルチテナントEKSでのワークロード完全分離）]]

## 第2問（Containers ｜ GPUコンテナワークロード（FargateはGPU非対応））
ある動画配信企業が、ディープラーニングを用いたリアルタイムな映像解析アプリケーションを開発しています。このアプリケーションは複数のコンテナ化されたマイクロサービスで構成されており、その一部のコンポーネントでは高負荷の解析を実行するために GPU リソースを利用する必要があります。

これらの要件を満たすために最も適切な構成はどれですか。

- A. コンテナイメージを ECR に格納し、GPU リソースを必要とするコンポーネントを ECR で直接実行する。
- B. GPU リソースを必要とするジョブを AWS Batch のジョブキューに送信し、GPU 対応の EC2 インスタンスで実行する。
- C. ECS を用いてコンテナをオーケストレーションし、GPU リソースを必要とするコンポーネントを Fargate 起動タイプで実行する。
- D. ECS を用いてコンテナをオーケストレーションし、GPU リソースを必要とするコンポーネントを GPU 対応の EC2 インスタンスで実行する。

> [!success]- 答え：**D**
> AWS Fargate は GPU をサポートしていないため、コンテナでGPUを使うには ECS（または EKS）の EC2 起動タイプで GPU 対応インスタンス（g4・p系など）を割り当てる。リアルタイムなマイクロサービス構成にはECSのオーケストレーションが適し、GPU要件はEC2起動タイプで満たす。
> - A：ECRはコンテナイメージのレジストリ（保管庫）で、コンテナを実行する場所ではない。「ECRで直接実行」が誤り。
> - B：AWS Batchはバッチ処理（大量ジョブの非同期実行）向け。要件はリアルタイム映像解析のマイクロサービスで性質が異なる。
> - C：ECSは適切だが、FargateはGPU非対応。GPUを必要とするコンポーネントを実行できない（最大の罠）。
> - D：正解。ECSでオーケストレーションし、GPU対応EC2インスタンスでGPUコンポーネントを実行する。
> **関連ノート**：[[0904_ECS]] / [[0902_オーケストレーション_フルマネージド]] / [[0703_AWS_Batch]]
> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第2問（Containers ｜ GPUコンテナワークロード（FargateはGPU非対応））]]

## 第3問（Application Integration ｜ Apache ActiveMQの移行とAmazon MQ・高可用構成）
ある会社は、オンプレミスの多層 Web アプリケーションシステムを AWS へ移行する計画を立てています。Web アプリケーションはデータ受信用の Web サーバーとデータ処理用のアプリケーションサーバーに分かれており、Apache Active MQ を利用してシステム間を疎結合化しています。保守担当者は、できる限りシステムの改修をせずに AWS へ移行したいと考えています。

AWS へ移行するにあたり、高可用性かつ運用にかかる負荷が少なくなる方法は次のうちどれですか。

- A. EC2 インスタンスに Web サーバーとアプリケーションサーバーをそれぞれ移行する。EC2 インスタンスを複数の AZ に配置する。システム間の連携にはアクティブ/スタンバイ構成の Amazon MQ を利用する。
- B. EC2 インスタンスに Web サーバーとアプリケーションサーバーをそれぞれ移行する。複数の AZ にまたがる Auto Scaling グループに EC2 インスタンスを所属させる。システム間の連携には Amazon SQS を利用する。
- C. EC2 インスタンスに Web サーバーとアプリケーションサーバーをそれぞれ移行する。EC2 インスタンスを複数の AZ に配置する。システム間の連携には Active MQ 用の EC2 インスタンスを利用し、複数の AZ にまたがる Auto Scaling グループに所属させる。
- D. EC2 インスタンスに Web サーバーとアプリケーションサーバーをそれぞれ移行する。複数の AZ にまたがる Auto Scaling グループに EC2 インスタンスを所属させる。システム間の連携にはアクティブ/スタンバイ構成の Amazon MQ を利用する。

> [!success]- 答え：**D**
> 既存がApache ActiveMQで「改修最小」なので、同プロトコルのマネージドブローカー Amazon MQ（アクティブ/スタンバイ）で載せ替えるのが定番。コンピュートは複数AZにまたがるAuto Scalingグループで高可用性と低運用を両立する。
> - A：Amazon MQ採用は良いが、EC2が単なる複数AZ配置でAuto Scalingがなく、障害時の自動復旧・スケールが弱い。運用も手作業寄りでDに劣る。
> - B：連携をSQSに変更するとActiveMQからのコード改修が必要で、「できる限り改修せず」に反する。
> - C：「Active MQ用のEC2インスタンスを利用」とは、素のEC2上に自分でActiveMQを構築・運用する（セルフマネージド）という意味で、パッチ・冗長化・監視を自己管理することになり運用負荷が重い。さらにActiveMQは状態を持つブローカーのためAuto Scalingで増減する構成自体も不適。マネージドのAmazon MQがあるのに不適。
> - D：正解。複数AZのAuto Scaling（高可用＋低運用）＋Amazon MQアクティブ/スタンバイ（マネージド冗長＋ActiveMQ互換で改修最小）。
> **関連ノート**：[[0806_Amazon_MQ]] / [[0802_SQS]] / [[0504_Auto_Scaling_と_Auto_Scaling_Group]]
> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第3問（Application Integration ｜ Apache ActiveMQの移行とAmazon MQ・高可用構成）]]

## 第4問（Security / IAM ｜ 特定IPからの攻撃をブロック（WAF））
ある会社は AWS を利用してウェブアプリケーションを運用しています。このアプリケーションは、ALB の背後にある EC2 インスタンス上で動作し、CloudFront を通じてコンテンツを配信しています。また、WAF を導入してクロスサイトスクリプティングなどの攻撃からアプリケーションを保護しています。ところが最近、特定の IP アドレスから悪意のあるトラフィックが増加しており、現行のセキュリティ設定では十分な対応ができていません。

特定の IP アドレスからの攻撃に対処するための最も適切な方法はどれですか。

- A. CloudFront のディストリビューションで、特定の IP アドレスをブロックするルールを追加する。
- B. WAF の設定を変更し、特定の IP アドレスをブロックする条件を追加する。
- C. Shield を利用して、悪意のあるトラフィックを軽減するための DDoS プロテクションを有効にする。
- D. ネットワーク ACL を設定して、特定の IP アドレスからのアクセスをサブネットレベルでブロックする。

> [!success]- 答え：**B**
> 既にWAFが導入されているので、WAFにIPセット（IP match）ルールで特定IPを拒否する条件を追加するのが最小変更で最適。WAFはCloudFrontのエッジで本当のクライアントIPを見て判定できる。
> - A：CloudFront単体に「IPブロックルールを追加」という機能はなく、CloudFrontでのIP制御は結局WAFで行う。実質Bの遠回りな表現で適切でない。
> - B：正解。既存WAFにIPセットで特定IPを拒否する条件を追加。最小変更で的確に対処できる。
> - C：ShieldはDDoS（量による攻撃）の緩和サービスで、特定IPからのL7攻撃をピンポイントで遮断する道具ではない。用途違い。
> - D：CloudFront配下ではサブネットに届くのはCloudFrontのエッジIPで、攻撃者の実IPはNACLに見えない。サブネットレベルでの特定IPブロックは成立しない。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[0408_セキュリティグループとNACL]] / [[0405_ネットワークとコンテンツ配信]]
> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第4問（Security / IAM ｜ 特定IPからの攻撃をブロック（WAF））]]

## 第5問（Database ｜ ライター/リーダーEPで本番と監査を分離（マルチAZ DBクラスター））
ある金融サービス企業は、Amazon RDS for PostgreSQLを使用してミッションクリティカルな決済処理アプリケーションを運用しています。このアプリケーションは高可用性が求められ、障害発生時には自動的にフェイルオーバーが実行されなければなりません。また、企業の監査チームは、取引データを最大5時間以内の遅延で日次監査レポートに反映させる必要があり、監査処理は本番トランザクションのパフォーマンスに影響を与えてはなりません。

これらの要件を最も効率的かつ運用負荷を抑えて満たす構成はどれですか。

- A. アプリケーションとRDS for PostgreSQLの間にRDS Proxyを導入し、トランザクション処理と監査処理の接続を振り分けて処理する。
- B. RDS for PostgreSQLのマルチAZ DBインスタンスを採用し、トランザクション処理はプライマリDBインスタンスに、監査処理はスタンバイDBインスタンスに向けて実行する。
- C. RDS for PostgreSQLのマルチAZ DBクラスターを導入し、トランザクション処理はライターエンドポイントに、監査処理はリーダーエンドポイントに向けて実行する。
- D. RDS for PostgreSQLに複数のリードレプリカを作成し、トランザクション処理はプライマリDBインスタンスに、監査処理はリードレプリカに向けて実行する。

> [!success]- 答え：**C**
> マルチAZ DBクラスターは「ライター1台＋リーダー2台」を別AZに自動配置し、①ライター障害時に自動フェイルオーバー（高可用性）、②リーダーエンドポイントで読み取り（監査）を本番書き込みと分離、を1つの構成でまとめて満たす。フェイルオーバーもレプリケーションもマネージドで運用負荷が低い。
> - A：RDS Proxyは接続プーリング（コネクションの使い回し）で、読み取りを別ノードへ分離する仕組みではない。監査処理は結局同じDBに当たり本番性能に影響する。役割違い。
> - B：マルチAZ DB"インスタンス"のスタンバイは待機専用で、通常は読み取りにも使えない（フェイルオーバー先）。スタンバイに監査処理を向ける構成自体が成立しない。最大の罠。
> - D：リードレプリカでも読み取り分離はできるが、レプリカは自動フェイルオーバー対象ではなく高可用性は別途設計が必要。クラスター（読み取り分離＋自動フェイルオーバーを一体提供）より運用負荷が高く「最も効率的」ではない。
> **関連ノート**：[[0301_RDS]] / [[0303_DynamoDB]] / [[0501_高可用性とスケーラビリティ]]
> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第5問（Database ｜ ライター/リーダーEPで本番と監査を分離（マルチAZ DBクラスター））]]

## 第6問（Storage ｜ S3アクセスポイントによる大規模なプレフィックス別アクセス制御）
ある企業は、複数の部門が生成するアプリケーションログや監査ログを、セキュリティチームが管理する1つの S3 バケットに集約しています。各部門には専用のプレフィックス（例：finance/、hr/、engineering/ など）が割り当てられ、そこにログを保存します。今後さらに多くの部門やチームからログを収集する予定で、プレフィックスの数は大幅に増える見込みです。同社は、S3 バケットポリシーが複雑化・肥大化することを避けつつ、各部門が自分のプレフィックスのみにアクセスできるよう制御する仕組みを必要としています。

この要件を最も適切に満たす設計はどれですか。

- A. S3 Storage Lens を使用して、各部門ごとのプレフィックス単位でのアクセス権限を分離し、S3 バケットへのアクセスを制御する。
- B. 署名付き URL を使用して一時的なアクセス権限を付与し、対象オブジェクトへのアクセスを行う。
- C. バケットポリシーでデフォルト拒否を設定し、アカウントごとに個別の許可文を追加して対象プレフィックスへのアクセスを許可する。
- D. 各部門ごとに専用の S3 アクセスポイントを作成し、各アクセスポイントポリシーで対応するプレフィックスのみにアクセスを許可する。

> [!success]- 答え：**D**
> S3アクセスポイントは、1つの共有バケットに多数のチームがアクセスする状況で、巨大な単一バケットポリシーを避けて権限を分散管理するための機能。部門ごとに専用アクセスポイント＋小さなポリシーを付ければ、プレフィックスが増えてもアクセスポイントを足すだけで済み、バケットポリシーは肥大化しない。
> - A：S3 Storage Lensは利用状況の可視化・分析ツールで、アクセス制御の機能ではない。
> - B：署名付きURLはオブジェクト単位の一時アクセスを渡す仕組みで、部門ごとの恒常的な大規模アクセス統制には向かない。
> - C：バケットポリシーにアカウントごとの許可文を追記する設計は、まさにポリシーが肥大化・複雑化するため要件と正反対。
> - D：正解。部門ごとに専用アクセスポイント＋アクセスポイントポリシーで、プレフィックスが増えてもバケットポリシーを膨らませず統制できる。
> **関連ノート**：[[0208_S3バケットポリシー]] / [[0201_S3]] / [[1601_セキュリティサービス一覧]]
> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第6問（Storage ｜ S3アクセスポイントによる大規模なプレフィックス別アクセス制御）]]

## 第7問（Migration / Transfer ｜ RTO2分のDR戦略（ウォームスタンバイ））
ある企業は、商品検索機能を提供する3層のWebアプリケーションをAWS上で運用しています。アプリケーション層は、ALBの背後にある複数のEC2インスタンスで構成されています。ユーザーが参照する商品情報などのドキュメントはS3バケットに保存されています。この企業は、プライマリリージョンで障害が発生した場合でも、2分以内にサービスを再開できることを求めています。運用チームは、できるだけ自動化された方法で、短時間でセカンダリリージョンへ切り替えられる構成を採用したいと考えています。

この要件を最も適切に満たす選択肢はどれですか。

- A. Amazon S3バケットのデータを別リージョンへレプリケートする。プライマリリージョンのEC2インスタンスについてAmazon Machine Image（AMI）を定期的に作成し、セカンダリリージョンへコピーしておく。障害発生時には、そのAMIからセカンダリリージョンでEC2インスタンスを起動し、ALBのターゲットグループに登録する。Amazon Route 53を使用して、セカンダリリージョンへトラフィックを切り替える。
- B. Amazon S3バケットのデータを別リージョンへレプリケートする。セカンダリリージョンにALBと最小限のEC2インスタンスを作成しておくが、EC2インスタンスは必要になるまで停止しておく。Amazon Route 53のフェイルオーバールーティングを使用して、プライマリリージョンで障害が発生した場合にセカンダリリージョンへトラフィックを切り替える。
- C. Amazon S3バケットのデータを別リージョンへレプリケートする。セカンダリリージョンにもALBと最小限のEC2インスタンスを常時稼働させる。Amazon Route 53のフェイルオーバールーティングを使用して、プライマリリージョンで障害が発生した場合にセカンダリリージョンへトラフィックを切り替える。
- D. AWS Backupを使用して、EC2インスタンスとS3バケットのデータを定期的にバックアップする。障害発生時には、バックアップからセカンダリリージョンにEC2インスタンスを復元し、ALBを新規作成してアプリケーションを再構成する。復旧後にAmazon Route 53のレコードを更新して、セカンダリリージョンへトラフィックを切り替える。

> [!success]- 答え：**C**
> 「2分以内に再開（RTO=2分）」「自動化」「短時間で切替」は、セカンダリリージョンに縮小版の本番を**常時稼働**させておく**ウォームスタンバイ**を指す。S3はクロスリージョンレプリケーションでデータ同期、Route 53フェイルオーバールーティングでヘルスチェック連動の自動切替を行う。
> - A：AMIから障害時にEC2を起動する＝起動・登録に時間がかかる（パイロットライト寄り）。2分以内のRTOには間に合わない。
> - B：セカンダリのEC2を「停止しておく」＝障害時に起動を待つ必要があり、起動時間ぶんRTOが延びる。常時稼働のCに対して2分要件で劣る。
> - C：正解。最小限のEC2を常時稼働（ウォームスタンバイ）＋Route 53フェイルオーバーで、起動待ちなく即トラフィックを受けられる。スケールは切替後に拡張。
> - D：バックアップからの復元＋ALB新規作成＝復旧に長時間かかる（バックアップ&リストア）。最もRTOが長く、2分要件と正反対。最大の罠。

> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第7問（Migration / Transfer ｜ RTO2分のDR戦略（ウォームスタンバイ））]]

## 第8問（Networking / CDN / LB ｜ Route 53フェイルオーバーとヘルスチェック）
ALB 配下に複数の EC2 インスタンスを配置したシステムがあります。可用性の向上のため、すでにシステムを展開しているリージョンとは異なるリージョンに、バックアップ環境の作成を計画しています。バックアップ環境へのフェイルオーバーには、Amazon Route 53 のフェイルオーバールーティングポリシーを使用します。

フェイルオーバーの実現のために、次のうちどの設定が必要ですか。

- A. Amazon Route 53 のヘルスチェックを有効にする。
- B. ALB のヘルスチェックを有効にする。
- C. バックアップ環境の ALB を指す A レコードを作成する。
- D. Amazon CloudWatch の監視アラートを有効にする。

> [!success]- 答え：**A**
> Route 53のフェイルオーバールーティングは、Route 53のヘルスチェックでプライマリ（本番）の死活を監視し、不健全になったらセカンダリ（バックアップ）へDNSを切り替える。よってRoute 53ヘルスチェックの有効化が必須。
> - A：正解。Route 53ヘルスチェックがプライマリの生死を監視し、フェイルオーバーの起点になる。
> - B：ALBのヘルスチェックはリージョン内で配下のEC2を振り分けるためのもので、DNSレベルのリージョン間フェイルオーバー判断には使われない。
> - C：レコード自体は必要だが、フェイルオーバーにはプライマリ/セカンダリのフェイルオーバーレコード＋ヘルスチェックが要点。単なるAレコード作成では切り替わらない（ALB宛ては本来Aliasレコード）。
> - D：CloudWatchアラームはヘルスチェックの一形態として使える場合もあるが、Route 53フェイルオーバーで直接必要な設定はRoute 53ヘルスチェック。
> **関連ノート**：[[0404_Route53]] / [[0501_高可用性とスケーラビリティ]] / [[0502_Elastic Load Balancing]]
> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第8問（Networking / CDN / LB ｜ Route 53フェイルオーバーとヘルスチェック）]]

## 第9問（Networking / CDN / LB ｜ NATインスタンスからNATゲートウェイへの移行と高可用化）
ある企業は、数年前に構築したシステムで2つのNATインスタンスを利用し、プライベートサブネット内のバックエンドサーバーがインターネットへアクセスできるようにしています。これらのバックエンドサーバーはAuto Scalingグループで構成されており、サービス利用者の増加に伴ってソフトウェアパッケージのダウンロードや外部APIへの接続、外部ログ管理サービスへのデータ送信などのアウトバウンドトラフィックが増えています。その結果、NATインスタンスの帯域が限界に近づき、処理遅延や障害が発生するリスクが高まっています。

今後のさらなるトラフィック増加に備え、高可用性と耐障害性を確保しつつ、運用負荷を最小限に抑える構成にするために、最も適切な対応はどれですか。

- A. 既存のNATインスタンスを廃止し、同一のAZに2つのNATゲートウェイを配置する。
- B. 既存のNATインスタンスを廃止し、別のVPCに配置した1つのNATゲートウェイをTransit Gateway経由で共有利用する。
- C. 既存のNATインスタンスを廃止し、異なるAZにそれぞれNATゲートウェイを1つずつ配置する。
- D. NATインスタンスの数を増やし、異なるAZに配置した上で、Auto Scalingを利用してインスタンスの起動と障害復旧を自動化する。

> [!success]- 答え：**C**
> NATインスタンスは自前運用のEC2で帯域も手動管理。マネージドなNATゲートウェイ（最大100Gbpsまで自動スケール）に置き換えるのが定石。さらに各AZにNATゲートウェイを1つずつ置くと、AZ障害時も他AZが生き残り、AZ内通信で完結して高可用性・耐障害性・低運用を満たす。
> - A：同一AZに2つ置いてもそのAZが落ちれば両方ダウン＝単一AZ障害に弱い。NATゲートウェイは1台で自動スケールするので2台に分割する意味も薄い。
> - B：別VPCの1つのNATゲートウェイをTransit Gateway共有＝そのNATゲートウェイ／AZが単一障害点。構成も複雑で運用負荷・コストが増える。
> - C：正解。各AZにNATゲートウェイを1つずつ配置し、各プライベートサブネットのルートを同一AZのNATゲートウェイへ向ける。マネージドで自動スケール、AZ障害の影響も分離できる。
> - D：NATインスタンスを増やす＝自前運用の継続でパッチ・帯域・冗長化を自分で抱える。マネージドなNATゲートウェイがあるのに運用負荷最小に反する。
> **関連ノート**：[[0406_VPC応用機能]] / [[0501_高可用性とスケーラビリティ]] / [[0405_ネットワークとコンテンツ配信]]
> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第9問（Networking / CDN / LB ｜ NATインスタンスからNATゲートウェイへの移行と高可用化）]]

## 第10問（Storage ｜ POSIX共有ストレージとEFSライフサイクル）
ある企業は、分析用のレポート生成アプリケーションを複数の AZ に配置した Amazon EC2 Linux インスタンスで実行しています。アプリケーションは、大きなレポートファイルを生成・共有するため、複数の EC2 インスタンス間で同時アクセスできる共有ストレージが必要です。このストレージは POSIX 準拠である必要があり、生成されたレポートは最初の1か月間は頻繁にアクセスされますが、その後はほとんど参照されません。高可用性を維持しつつ、コストを最適化する最も適切なソリューションはどれですか。

- A. S3 Standard ストレージクラスを使用し、ライフサイクルルールを設定して、生成後1カ月が経過したデータを S3 Glacier Flexible Retrieval に自動移行する。
- B. EFS の Standard ストレージクラスを使用し、ライフサイクルポリシーを設定して、アクセス頻度の低いデータを EFS Standard-IA に移行する。
- C. EBS ボリュームを EC2 インスタンス間で共有し、スナップショットを利用する。
- D. S3 Standard ストレージクラスを使用し、ライフサイクルルールを設定して、生成後1カ月が経過したデータを S3 Standard-IA に自動移行する。

> [!success]- 答え：**B**
> 「複数のEC2から同時アクセスできる共有ストレージ」＋「POSIX準拠」を満たすのはEFS（共有ファイルシステム）だけ。EFSは複数AZのEC2から同時マウントでき高可用性も備える。コスト最適化はEFSのライフサイクル管理でアクセス頻度の低いファイルをEFS Standard-IAへ自動移行して実現する。
> - A：S3はオブジェクトストレージでPOSIX非準拠、複数EC2から同時マウントする共有ファイルシステムではない。要件を満たさない。
> - B：正解。EFS（POSIX・複数AZ同時アクセス・高可用）＋ライフサイクルでStandard-IAへ移行しコスト最適化。
> - C：EBSは基本1インスタンスにアタッチするブロックストレージで、複数AZ・多数同時アクセスの共有ファイルシステムにならない。スナップショットも共有アクセス手段ではない。
> - D：AもDもS3でPOSIX非準拠・共有ファイルシステムではないため、ストレージ種別が要件に合わない。

> **詳しい解説**：[[504_【解説】外部サイト模擬試験4#第10問（Storage ｜ POSIX共有ストレージとEFSライフサイクル）]]
