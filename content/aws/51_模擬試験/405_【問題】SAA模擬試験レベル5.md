# 【問題】SAA模擬試験レベル5（公式サンプル寄せ改訂版・全10問）

## この問題集の目的

旧405とは別問題。SAA-C03公式サンプル問題に寄せて、Security / IAMを中心に、複数の要件を同時に満たす構成を選ぶ練習を行う。

> [!info] 出題形式
> - シナリオ段落と設問文を分け、です・ます調で記述。
> - 選択肢はサービス名だけでなく、具体的な操作・構成で記述。
> - 複数選択問題は5択（A〜E）、単一選択問題は4択（A〜D）。

10問

## 第1問（Security / IAM ｜ S3の保存時暗号化と鍵の監査）
ある金融サービス企業は、顧客の機密ドキュメントをAmazon S3バケットに保存しています。コンプライアンス要件により、すべての新規オブジェクトを保存時に暗号化し、暗号化キーのローテーションと利用履歴を監査できる必要があります。企業は、鍵の管理権限を自社側に置きながら、運用負荷をできるだけ抑えたいと考えています。

これらの要件を満たす最も適切なソリューションはどれですか。

- A. AWS KMSのカスタマーマネージドキーを作成し、S3バケットのデフォルト暗号化をSSE-KMSに設定する。KMSキーの自動ローテーションを有効にし、AWS CloudTrailでキーの使用状況を監査する。
- B. S3マネージドキーによるサーバー側暗号化（SSE-S3）をバケットのデフォルト暗号化に設定し、S3サーバーアクセスログでキーの利用状況を確認する。
- C. AWS CloudHSMクラスターを作成し、アプリケーションでクライアント側暗号化を実装してからS3へアップロードする。
- D. S3バケットポリシーで暗号化されていないPutObjectを拒否し、暗号化キーにはAWS管理のKMSキーを使用する。

> [!success]- 答え：**A**
> SSE-KMSとカスタマーマネージドキーを使うと、企業側で鍵の権限とローテーションを管理しながら、CloudTrailで鍵利用を監査できる。
> - A：正解。鍵管理、ローテーション、監査、運用負荷のバランスが最もよい。
> - B：保存時暗号化はできるが、鍵の管理権限やKMSキー利用の監査要件を満たしにくい。
> - C：強い管理は可能だが、HSM運用とアプリ実装が重く、運用負荷最小化に反する。
> - D：暗号化の強制には役立つが、AWS管理キーでは企業側の鍵管理要件が弱い。
> **関連ノート**：[[1607_AWS_KMS]] / [[1605_データ保護・暗号化]] / [[0207_S3の暗号化]]

## 第2問（Security / IAM ｜ パブリック証明書の自動更新）
ある企業は、Application Load Balancerの背後でHTTPSのWebアプリケーションを公開しています。セキュリティチームは、信頼されたパブリック証明書を使用し、証明書更新の手作業と期限切れリスクを減らしたいと考えています。ロードバランサー終端のTLS構成は維持する必要があります。

最も運用負荷が小さいソリューションはどれですか。

- A. サードパーティーCAで証明書を購入し、IAMにアップロードしてALBに関連付ける。期限が近づいたら手動で証明書を差し替える。
- B. AWS Certificate Manager（ACM）でパブリック証明書をリクエストし、ドメイン検証後にALBのHTTPSリスナーへ関連付ける。
- C. AWS KMSで非対称キーを作成し、その公開鍵をブラウザに配布してTLS証明書として使用する。
- D. 証明書と秘密鍵をAWS Secrets Managerに保存し、Lambda関数で定期的にALBへコピーする。

> [!success]- 答え：**B**
> ACMで発行したパブリック証明書をALBに関連付けると、サポート対象サービス上で証明書更新をマネージドに扱える。
> - A：機能的には可能だが、手動更新が残り、期限切れリスクを下げる要件に弱い。
> - B：正解。ALBとACMの組み合わせは、公開TLS証明書の発行・更新運用を大きく減らせる。
> - C：KMSキーはTLS証明書そのものとしてブラウザに信頼されるものではない。
> - D：Secrets Managerは秘密情報の保管には使えるが、ALB証明書更新の標準的な低運用解ではない。
> **関連ノート**：[[1605_データ保護・暗号化]] / [[0406_ELB]]

## 第3問（Security / IAM ｜ Web攻撃と過剰リクエストの軽減）
あるECサイトは、Amazon CloudFrontの背後にあるApplication Load BalancerでWebアプリケーションを配信しています。最近、SQLインジェクションやクロスサイトスクリプティングを狙ったHTTPリクエストと、短時間に大量のリクエストを送るクライアントが増えています。アプリケーションコードを変更せず、HTTPレイヤーで攻撃を軽減したいと考えています。

最も適切なソリューションはどれですか。

- A. ALBのセキュリティグループで、既知の攻撃元IPアドレスからのインバウンド通信を拒否する。
- B. VPCネットワークACLで、Webサーバーのサブネットへ向かう一部のエフェメラルポートを拒否する。
- C. AWS WAFのWeb ACLをCloudFrontディストリビューションに関連付け、AWSマネージドルールとレートベースルールを設定する。
- D. AWS Shield Advancedだけを有効化し、アプリケーション層の攻撃パターン検査は設定しない。

> [!success]- 答え：**C**
> AWS WAFはHTTPリクエストの内容を評価し、マネージドルールやレートベースルールでWeb攻撃と過剰リクエストを軽減できる。
> - A：IP単位の制御は一部に有効だが、SQLインジェクションやXSSの内容検査にはならない。
> - B：ネットワークレイヤーの粗い制御で、HTTPリクエスト内容に基づく防御はできない。
> - C：正解。CloudFront前段で広く適用でき、HTTPレイヤーの攻撃とレート制御を扱える。
> - D：DDoS対策には役立つが、WAFルールなしではアプリケーション層の攻撃パターン検査が不足する。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[0405_ネットワークとコンテンツ配信]]

## 第4問（Management / Monitoring ｜ マルチアカウントの予防的ガードレール）
ある企業は、AWS Organizationsで本番、開発、検証用の複数アカウントを管理しています。セキュリティ部門は、承認されていないリージョンでのリソース作成と、各アカウントでのCloudTrail無効化を組織全体で防止したいと考えています。個別アカウントでの手動設定に依存しない、予防的な制御が必要です。

要件を満たす最も適切なソリューションはどれですか。

- A. すべてのアカウントにAWS Configルールを作成し、非準拠リソースを検出したら管理者へ通知する。
- B. IAM Identity Centerの許可セットで、管理者ユーザーにCloudTrail変更権限を付与しないようにする。
- C. Amazon GuardDutyを組織全体で有効化し、未承認リージョンの使用を検出したら調査する。
- D. AWS Organizationsのサービスコントロールポリシー（SCP）で、未承認リージョンの操作とCloudTrail停止操作を拒否する。

> [!success]- 答え：**D**
> SCPはアカウント内のIAM権限より上位のガードレールとして、組織単位で特定操作を拒否できる。
> - A：検出と通知には有効だが、作成や無効化を事前に防ぐ予防的制御ではない。
> - B：人のアクセス権限管理には有効だが、全プリンシパルに対する組織横断の拒否としてはSCPが適切。
> - C：脅威検出サービスであり、リージョン使用やCloudTrail停止を直接拒否しない。
> - D：正解。マルチアカウントに中央から予防的ガードレールを適用できる。
> **関連ノート**：[[1403_Organizations]] / [[1002_CloudTrail]]

## 第5問（Security / IAM ｜ S3内の機密データ検出）
ある企業は、複数のAmazon S3バケットに保存されたログと顧客アップロードファイルを管理しています。監査チームは、個人情報や機密データが含まれるオブジェクトを自動的に検出し、検出結果をセキュリティ運用チームへ通知したいと考えています。対象はS3内のデータそのものです。

最も適切なサービスはどれですか。

- A. Amazon Inspectorを使用して、S3バケット内のオブジェクトに含まれる脆弱性をスキャンする。
- B. Amazon Macieを有効化し、S3バケット内の機密データ検出ジョブと検出結果通知を設定する。
- C. Amazon GuardDutyを有効化し、S3オブジェクト内の個人情報を分類する。
- D. IAM Access Analyzerを使用して、S3オブジェクト本文に含まれる機密情報を抽出する。

> [!success]- 答え：**B**
> MacieはS3内の機密データを検出・分類するマネージドサービスで、PIIなどの発見に使う。
> - A：Inspectorは主にワークロードやコンテナイメージなどの脆弱性管理であり、S3本文のPII分類ではない。
> - B：正解。S3に保存された機密データの検出という要件に合う。
> - C：GuardDutyは脅威検出であり、S3オブジェクト本文の分類サービスではない。
> - D：外部アクセス可能なリソースの分析には有効だが、オブジェクト本文の機密情報抽出は行わない。
> **関連ノート**：[[1601_セキュリティサービス一覧]]

## 第6問（Security / IAM ｜ AWSアカウント内の脅威検出）
ある企業は、複数のVPCでEC2インスタンスを実行しています。セキュリティチームは、既知の悪性IPとの通信、異常なAPI操作、DNSクエリなどを継続的に分析し、侵害の可能性を早期に検出したいと考えています。エージェントの導入や独自の分析基盤の構築は避けたい方針です。

最も適切なソリューションはどれですか。

- A. AWS Configを有効化し、リソース設定の変更履歴を記録する。
- B. Amazon Inspectorを使用し、EC2インスタンスにあるソフトウェア脆弱性を継続的に検出する。
- C. Amazon GuardDutyを有効化し、CloudTrailイベント、VPC Flow Logs、DNSログなどを分析して脅威を検出する。
- D. AWS Trusted Advisorを使用し、サービス制限とコスト最適化の推奨事項を確認する。

> [!success]- 答え：**C**
> GuardDutyはAWSログをもとに、侵害の兆候や不審な通信、異常なAPI操作をマネージドに検出する。
> - A：設定変更の記録には有効だが、脅威検出の主役ではない。
> - B：脆弱性検出には有効だが、悪性IP通信や異常API操作の総合的な脅威検出とは目的が異なる。
> - C：正解。エージェントレスで複数ログソースを分析し、脅威検出に使える。
> - D：ベストプラクティス確認には有効だが、侵害兆候の継続分析サービスではない。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[1002_CloudTrail]]

## 第7問（Storage ｜ S3アクセスの強制条件）
ある医療関連企業は、Amazon S3バケットに診療関連ファイルを保存しています。セキュリティ要件により、すべてのリクエストはTLS経由でなければならず、新規アップロードされるすべてのオブジェクトは指定したAWS KMSカスタマーマネージドキーで暗号化されている必要があります。要件に違反するリクエストはS3バケット側で拒否したいと考えています。

この要件を満たすためにバケットポリシーで設定すべき拒否条件はどれですか。（2つ選択）

- A. `aws:SecureTransport` が `false` のリクエストを拒否する。
- B. `s3:x-amz-acl` が `public-read` ではないPutObjectリクエストを拒否する。
- C. リクエスト元IPアドレスが社内ネットワークではない場合、すべてのGetObjectを拒否する。
- D. PutObjectで `s3:x-amz-server-side-encryption` が `aws:kms` ではない、または指定KMSキーIDではないリクエストを拒否する。
- E. バケットでS3 Object Lockのコンプライアンスモードを有効化して、削除を防止する。

> [!success]- 答え：**A, D**
> TLS強制には`aws:SecureTransport`、SSE-KMSと特定KMSキーの強制にはPutObject時の暗号化ヘッダー条件を使う。
> - A：正解。HTTPなどTLSでない通信をバケットポリシーで拒否できる。
> - B：ACL条件であり、TLSやKMS暗号化の強制とは関係しない。
> - C：IP制限は追加要件があれば有効だが、今回のTLSとKMSキー強制の要件ではない。
> - D：正解。アップロード時に指定方式・指定キーで暗号化されていないオブジェクトを拒否できる。
> - E：削除保護や改ざん防止には有効だが、通信経路や保存時暗号化の強制ではない。
> **関連ノート**：[[0207_S3の暗号化]] / [[0208_S3セキュリティ・アクセス制御]]

## 第8問（Security / IAM ｜ 従業員のマルチアカウントアクセス）
ある企業は、社内IdPで従業員アカウントを管理しています。AWS環境では開発、検証、本番の複数アカウントがあり、チームや職務に応じて権限を割り当てたいと考えています。長期的なIAMユーザーを各アカウントに作成せず、入退社や異動に合わせて中央管理したい方針です。

最も適切なソリューションはどれですか。

- A. 各AWSアカウントにIAMユーザーを作成し、社内IdPのユーザー名と同じ名前で管理する。
- B. 各AWSアカウントに同じIAMグループを作成し、アクセスキーをSecrets Managerに保存して従業員へ配布する。
- C. IAM Identity CenterをAWS Organizationsと連携し、社内IdPをIDソースとして設定して、許可セットを各アカウントへ割り当てる。
- D. AWS OrganizationsのSCPだけを使用し、従業員ごとのサインインと権限割り当てを管理する。

> [!success]- 答え：**C**
> IAM Identity Centerは、外部IdPと連携した従業員のフェデレーションアクセスと、複数AWSアカウントへの許可セット割り当てに適している。
> - A：アカウントごとのIAMユーザー管理が増え、中央管理と長期ユーザー回避の要件に反する。
> - B：長期アクセスキー配布はセキュリティ上不適切で、従業員アクセス管理にも向かない。
> - C：正解。社内IdPを起点に、マルチアカウントの権限を中央管理できる。
> - D：SCPはガードレールであり、個々の従業員認証や権限割り当ての仕組みではない。
> **関連ノート**：[[0601_IAM基礎]] / [[1403_Organizations]]

## 第9問（Security / IAM ｜ RDS認証情報の安全なローテーション）
ある企業は、Amazon EC2上のアプリケーションからAmazon RDS for MySQLへ接続しています。現在、データベースのユーザー名とパスワードはアプリケーション設定ファイルに保存されています。セキュリティチームは、認証情報を設定ファイルから分離し、定期的な自動ローテーションを行い、アプリケーションが実行時に安全に取得できるようにしたいと考えています。

これらの要件を満たすために実行すべきアクションはどれですか。（2つ選択）

- A. Systems Manager Parameter StoreのSecureStringに認証情報を保存し、ローテーションは運用担当者が月次で手動実行する。
- B. AWS Secrets ManagerにRDS認証情報を保存し、RDS向けの自動ローテーションを有効化する。
- C. 認証情報をEC2インスタンスのユーザーデータに暗号化して保存し、起動時に環境変数へ展開する。
- D. EC2インスタンスプロファイルのIAMロールに、対象シークレットを取得する最小限の権限を付与する。
- E. アプリケーション用のIAMユーザーを作成し、アクセスキーを設定ファイルに保存してSecrets Managerを呼び出す。

> [!success]- 答え：**B, D**
> Secrets ManagerはRDS認証情報の保管と自動ローテーションに対応しており、EC2のIAMロールで実行時取得を安全に行える。
> - A：SecureStringは暗号化保管には有効だが、今回の「自動ローテーション」要件ではSecrets Managerが適切。
> - B：正解。RDS認証情報の自動ローテーションをマネージドに実装できる。
> - C：設定ファイルから移しても、インスタンス内に固定値として残りやすく、ローテーションにも弱い。
> - D：正解。アプリケーションが長期アクセスキーなしで必要なシークレットだけ取得できる。
> - E：長期アクセスキーを設定ファイルに置く設計になり、要件に反する。
> **関連ノート**：[[1605_データ保護・暗号化]] / [[0602_IAMロール詳細]]

## 第10問（Networking / CDN / LB ｜ CloudFront経由の限定配信）
ある教育サービス企業は、受講者向けの動画ファイルをAmazon S3に保存し、Amazon CloudFrontで配信しています。動画ファイルは契約中の受講者だけが閲覧できる必要があり、S3バケットへ直接アクセスされてはなりません。企業は、S3をパブリックにせず、CloudFront経由の認可済みリクエストだけを許可したいと考えています。

最も適切なソリューションはどれですか。

- A. CloudFrontで署名付きURLまたは署名付きCookieを使用し、Origin Access Control（OAC）を設定する。S3バケットポリシーではCloudFrontディストリビューションからのアクセスだけを許可する。
- B. S3バケットを静的Webサイトホスティングとして有効化し、バケットポリシーで全オブジェクトにパブリック読み取りを許可する。
- C. S3オブジェクトにランダムなファイル名を付け、CloudFrontのデフォルトキャッシュ動作で公開配信する。
- D. CloudFrontのオリジンをS3からALBへ変更し、ALBのセキュリティグループでインターネットからのHTTPアクセスを許可する。

> [!success]- 答え：**A**
> 署名付きURLまたはCookieで視聴者を制御し、OACとバケットポリシーでCloudFront経由だけを許可すると、S3の直接公開を避けられる。
> - A：正解。認可済みユーザーへの限定配信と、S3への直接アクセス防止を両立できる。
> - B：S3が公開され、契約中ユーザー限定の要件に反する。
> - C：推測困難な名前はアクセス制御ではなく、漏えい後の保護にもならない。
> - D：ALBへ変えても、S3上の動画をCloudFront経由で限定配信する要件の解ではない。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0208_S3セキュリティ・アクセス制御]]
