# 【問題】サービス用途確認13（ストレージ・データベース・統合）

## この問題集の目的

ストレージ、データベース、アプリケーション統合で使う代表的なサービスを、要件のキーワードから選べるようにする。

## 対象レベル

SAA基礎。各サービスの得意分野を1問1論点で確認する。

## 問題数

10問

---

## 第1問（Storage ｜ 高性能コンピューティング向けファイルシステム）
S3に保存した大量のデータを、機械学習やHPCのLinuxワークロードから高スループット・低レイテンシーのPOSIXファイルシステムとして処理したいと考えています。
最も適したサービスはどれですか。

- A. Amazon FSx for Lustre
- B. Amazon EFS
- C. Amazon FSx for Windows File Server
- D. Amazon S3 Glacier Deep Archive

> [!success]- 答え：**A**
> FSx for Lustreは、HPCや機械学習などの高速処理向けに設計されたマネージド並列ファイルシステムです。
> - A：正解。S3との連携にも対応し、大規模データを高性能に処理できる。
> - B：汎用的な共有NFSであり、HPC向け並列ファイルシステムを明示した要件では最適ではない。
> - C：WindowsのSMBワークロード向け。
> - D：長期アーカイブ用のオブジェクトストレージクラス。
> **着眼点（SAA）**：「HPC」「Lustre」「S3データを高速処理」ならFSx for Lustre。
> **関連ノート**：[[0213_FSxファミリー4種の使い分け]]
> **詳しい解説**：[[113_【解説】サービス用途確認13#第1問（Storage ｜ 高性能コンピューティング向けファイルシステム）]]

## 第2問（Storage ｜ NetApp互換ストレージ）
オンプレミスで利用しているNetApp ONTAP環境をAWSへ移行します。NFS、SMB、iSCSI、スナップショット、重複排除など、使い慣れたONTAP機能との互換性を維持したいと考えています。
最も適したサービスはどれですか。

- A. Amazon FSx for NetApp ONTAP
- B. Amazon EFS
- C. Amazon FSx for OpenZFS
- D. AWS Storage Gateway Volume Gateway

> [!success]- 答え：**A**
> FSx for NetApp ONTAPは、ONTAPのデータ管理機能と複数プロトコルをマネージドで提供します。
> - A：正解。既存NetAppワークロードの移行要件に直接対応する。
> - B：Linux向けの共有NFSが中心で、ONTAP互換機能を提供しない。
> - C：ZFSベースのLinuxファイルサーバー向け。
> - D：オンプレミスからクラウドストレージへ接続するハイブリッドサービスで、ONTAP互換ファイルシステムではない。
> **着眼点（SAA）**：「NetApp」「ONTAP」「NFSとSMBの両方」ならFSx for NetApp ONTAP。
> **関連ノート**：[[0213_FSxファミリー4種の使い分け]]
> **詳しい解説**：[[113_【解説】サービス用途確認13#第2問（Storage ｜ NetApp互換ストレージ）]]

## 第3問（Storage ｜ S3オブジェクトの一括処理）
S3バケット内にある数億個の既存オブジェクトに対して、タグ付け、コピー、復元開始などの処理を一括で実行し、進捗と完了レポートを確認したいと考えています。
最も適した機能はどれですか。

- A. S3 Batch Operations
- B. S3 Transfer Acceleration
- C. S3 Select
- D. S3イベント通知

> [!success]- 答え：**A**
> S3 Batch Operationsは、オブジェクト一覧を基に大規模な一括処理ジョブを実行・追跡します。
> - A：正解。数百万から数十億の既存オブジェクトに同じ操作を適用できる。
> - B：遠距離からS3への転送を高速化する機能。
> - C：単一オブジェクト内から必要なデータだけを抽出する機能。
> - D：新しいイベントを契機に処理を起動する仕組みで、既存オブジェクト群の管理ジョブとは異なる。
> **着眼点（SAA）**：「大量の既存オブジェクトへ一括操作」ならS3 Batch Operations。
> **関連ノート**：[[0201_S3]] / [[0204_S3のライフサイクル管理]]
> **詳しい解説**：[[113_【解説】サービス用途確認13#第3問（Storage ｜ S3オブジェクトの一括処理）]]

## 第4問（Storage ｜ S3利用状況の可視化）
企業は多数のアカウントとリージョンにS3バケットを所有しています。ストレージ使用量、オブジェクト数、暗号化状況、コスト効率などを組織横断のダッシュボードで可視化したいと考えています。
最も適した機能はどれですか。

- A. S3 Storage Lens
- B. S3 Inventory
- C. AWS Cost Explorer
- D. Amazon CloudWatch Logs Insights

> [!success]- 答え：**A**
> S3 Storage Lensは、S3ストレージの利用状況と保護・コスト最適化に関するメトリクスを組織横断で可視化します。
> - A：正解。アカウント、リージョン、バケットなどの粒度で集約ダッシュボードを提供する。
> - B：オブジェクトとメタデータの一覧レポートを定期出力する機能。
> - C：AWS全体のコスト分析には使えるが、S3固有の詳細な利用・保護メトリクスが主目的ではない。
> - D：ログデータへの対話的なクエリ機能。
> **着眼点（SAA）**：「S3全体をダッシュボードで可視化・最適化」ならStorage Lens。
> **関連ノート**：[[0201_S3]] / [[1701_コスト管理サービス一覧]]
> **詳しい解説**：[[113_【解説】サービス用途確認13#第4問（Storage ｜ S3利用状況の可視化）]]

## 第5問（Database ｜ データベース接続のプール）
多数のLambda関数がAmazon RDSへ短時間に接続するため、接続数が急増してデータベースが不安定になります。アプリケーションの変更を抑えながら接続をプールし、フェイルオーバー時の復旧も速めたいと考えています。
最も適したサービスはどれですか。

- A. Amazon RDS Proxy
- B. Amazon ElastiCache
- C. Amazon API Gateway
- D. AWS Database Migration Service

> [!success]- 答え：**A**
> RDS Proxyは、アプリケーションとRDS/Auroraの間でデータベース接続をプールして再利用します。
> - A：正解。サーバーレスの接続急増を吸収し、DBフェイルオーバーの影響も軽減する。
> - B：データ自体をメモリにキャッシュして読み取り負荷を減らすサービス。
> - C：APIの公開・認証・スロットリングなどを管理する。
> - D：データベース移行・継続レプリケーション用。
> **着眼点（SAA）**：「LambdaからRDS」「接続枯渇」「コネクションプール」ならRDS Proxy。
> **関連ノート**：[[0302_RDSProxy]] / [[0301_RDS]]
> **詳しい解説**：[[113_【解説】サービス用途確認13#第5問（Database ｜ データベース接続のプール）]]

> - D：時系列データベース。

## 第8問（Application Integration ｜ SaaSデータ連携）
Salesforceに保存された顧客データを、カスタム連携コードをできるだけ書かずにS3へ定期転送して分析に利用したいと考えています。
最も適したサービスはどれですか。

- A. Amazon AppFlow
- B. AWS DataSync
- C. Amazon EventBridge
- D. AWS Transfer Family

> [!success]- 答え：**A**
> Amazon AppFlowは、SalesforceなどのSaaSとAWSサービスの間でデータフローを構成するマネージド統合サービスです。
> - A：正解。スケジュールやイベントを契機にSaaSデータを安全に転送できる。
> - B：NFS/SMBやオブジェクトストレージ間の大容量ファイル転送向け。
> - C：イベントのルーティングが主目的で、大量データ本体の抽出・転送サービスではない。
> - D：SFTP、FTPS、FTPなど既存プロトコルでファイルを授受する。
> **着眼点（SAA）**：「SalesforceなどSaaSのデータをAWSへノーコード連携」ならAppFlow。
> **関連ノート**：[[0812_AppFlow]]

## 第9問（Containers ｜ サービスディスカバリ）
ECS上のマイクロサービスが動的に増減します。各サービスを論理名で登録し、他のサービスがDNS名やAPIを使って正常なエンドポイントを発見できるようにしたいと考えています。
最も適したサービスはどれですか。

- A. AWS Cloud Map
- B. Amazon Route 53 Resolver
- C. AWS App Mesh
- D. Elastic Load Balancing

> [!success]- 答え：**A**
> AWS Cloud Mapは、クラウドリソースを名前で登録・検出するサービスディスカバリ機能を提供します。
> - A：正解。動的なマイクロサービスの場所を名前空間とサービス名で発見できる。
> - B：VPCとオンプレミス間などのDNS問い合わせ転送を扱う。
> - C：サービス間通信の可視化やトラフィック制御を行うサービスメッシュ。
> - D：トラフィックを複数ターゲットへ分散する。
> **着眼点（SAA）**：「動的なサービスを名前で登録・発見」ならCloud Map。

モバイルアプリ向けにGraphQL APIを提供し、複数のデータソースへのアクセス、リアルタイム更新、オフライン同期をマネージドで実装したいと考えています。
最も適したサービスはどれですか。

- A. AWS AppSync
- B. Amazon API Gateway REST API
- C. Amazon MQ
- D. AWS Step Functions

> [!success]- 答え：**A**
> AWS AppSyncは、GraphQL APIとリアルタイムデータ同期をマネージドで提供します。
> - A：正解。GraphQLスキーマを使い、DynamoDBやLambdaなどのデータソースを統合できる。
> - B：REST/HTTP/WebSocket APIの管理に適するが、GraphQL専用機能はAppSyncが担う。
> - C：メッセージブローカー互換サービス。
> - D：複数処理を順序・分岐・再試行付きでオーケストレーションする。
> **着眼点（SAA）**：「GraphQL」「リアルタイム」「オフライン同期」ならAppSync。
