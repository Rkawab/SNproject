# 【問題】類似サービス比較9（公式サンプル風・全10問）

## この問題集の目的

役割が似ていて混同しやすいサービス/オプションの「使い分け」を要件文で判断する。201〜208で扱っていない比較（Lightsail vs EC2・S3 Intelligent-Tiering vs 手動ライフサイクル・DynamoDBオンデマンド vs プロビジョンド・Route 53ルーティングポリシー・KMS vs CloudHSM・Organizations SCP vs IAMポリシー・SQS vs SNS・CloudFront Functions vs Lambda@Edge・EMR vs Glue・MGN vs DMSなど）を中心に、13分野を横断する。

## 対象レベル

SAA。サービス名は知っている前提で、要件に最も合う1つを選ぶ。

## 問題数

10問

## 第1問（Compute ｜ Lightsail vs EC2）
小規模なブログを、月額固定で予測しやすい料金で、最小限の設定でさっと立ち上げたい。VPCやIAMの細かな設計はせず、あらかじめパッケージ化された構成で始めたい。最も適切なサービスはどれか。

- A. Amazon Lightsail
- B. Amazon EC2（オンデマンドインスタンス）
- C. AWS Elastic Beanstalk
- D. Amazon ECS on EC2

> [!success]- 答え：**A**
> Lightsailは仮想サーバー・ストレージ・ネットワークを月額固定でパッケージ化した入門向けサービスで、最小設定ですぐに使い始められる。
> - A：正解。固定料金・簡単セットアップという要件に最も合う。
> - B：自由度は高いが、料金が従量で見積もりにくく、ネットワーク等の設計も自前で必要。
> - C：アプリのデプロイ自動化が主目的で、月額固定の簡易サーバーという要件には過剰。
> - D：コンテナオーケストレーションが前提で、小規模ブログを最小設定で始める用途としては複雑。
> **関連ノート**：[[0701_コンピューティングサービス一覧]] / [[0101_EC2やVPCとは]]

## 第2問（Storage ｜ S3 Intelligent-Tiering vs 手動ライフサイクル）
S3に置くオブジェクトのアクセス頻度が予測できず、時期によって大きく変動する。運用の手間をかけずにコストを最適化したい。最も適切なアプローチはどれか。

- A. S3 Intelligent-Tiering を使う
- B. S3 Standard に固定する
- C. 一定日数で Standard-IA → Glacier に移すライフサイクルルールを手動設計する
- D. すべて S3 Glacier Deep Archive に保存する

> [!success]- 答え：**A**
> Intelligent-Tieringはアクセスパターンを監視して自動で最適な階層へ移動し、アクセス頻度が読めないデータの運用レスなコスト最適化に向く。
> - A：正解。予測困難なアクセスでも自動で階層化し、手動設計なしでコストを抑えられる。
> - B：常に高めの料金で、アクセスが減った分の最適化ができない。
> - C：アクセスパターンが予測できる場合は有効だが、変動が読めないケースでは誤った階層移動で逆効果になりうる。
> - D：取り出しに時間がかかるため、随時アクセスされる可能性があるデータには不適切。
> **関連ノート**：[[0203_S3ストレージクラス]] / [[0204_S3のライフサイクル管理]]

## 第3問（Database ｜ DynamoDB オンデマンド vs プロビジョンド）
新規アプリのトラフィックがまったく予測できず、急なスパイクも想定される。キャパシティの事前見積もりや調整をせず、使った分だけ支払いたい。最も適切なDynamoDBのキャパシティモードはどれか。

- A. オンデマンドキャパシティモード
- B. プロビジョンドキャパシティモード（固定）
- C. プロビジョンドキャパシティモード＋手動スケーリング
- D. リザーブドキャパシティ

> [!success]- 答え：**A**
> オンデマンドモードは事前のキャパシティ設定なしにトラフィックへ自動対応し、実際のリクエスト量に応じて課金される。予測不能なワークロードに向く。
> - A：正解。見積もり不要・スパイク自動対応・従量課金という要件に合致する。
> - B：一定のキャパシティを事前確保する方式で、予測困難なスパイクでスロットリングや過剰確保が起きやすい。
> - C：手動調整が必要で、予測できないトラフィックには追従しきれない。
> - D：安定して高い消費量がある場合の割引購入で、予測不能・見積もり不要という要件には合わない。
> **関連ノート**：[[0303_DynamoDB]]

## 第4問（Networking / CDN / LB ｜ Route 53 ルーティングポリシー）
同一アプリを複数リージョンに展開し、利用者を最も応答が速いリージョンへ自動的に振り向けたい。最も適切なRoute 53のルーティングポリシーはどれか。

- A. レイテンシーベースルーティング
- B. 加重ルーティング
- C. フェイルオーバールーティング
- D. 位置情報（Geolocation）ルーティング

> [!success]- 答え：**A**
> レイテンシーベースルーティングは、利用者から各リージョンへの測定レイテンシーが最も低いエンドポイントへ誘導する。
> - A：正解。最も応答が速いリージョンへ自動的に振り向ける要件に合致する。
> - B：指定した比率でトラフィックを分配する方式で、レイテンシー最小化が目的ではない（カナリアリリース等向け）。
> - C：プライマリ障害時にセカンダリへ切り替える可用性向けで、最速リージョン選択ではない。
> - D：利用者の地理的な所在地でルーティングする方式で、必ずしも最速のレイテンシーになるとは限らない。
> **関連ノート**：[[0404_Route53]] / [[0405_ネットワークとコンテンツ配信]]

## 第5問（Security / IAM ｜ KMS vs CloudHSM）
暗号鍵を、自社だけが専有する単一テナントのハードウェアで管理し、FIPS 140-2 レベル3準拠と鍵の完全な管理権限が求められるコンプライアンス要件を満たしたい。最も適切なサービスはどれか。

- A. AWS CloudHSM
- B. AWS KMS（AWSマネージドキー）
- C. AWS KMS（カスタマー管理キー）
- D. AWS Secrets Manager

> [!success]- 答え：**A**
> CloudHSMは専有（単一テナント）のハードウェアセキュリティモジュールを提供し、FIPS 140-2 レベル3準拠と鍵の完全な自己管理を可能にする。
> - A：正解。専有HSM・FIPSレベル3・完全な鍵管理権限という厳格な要件に合致する。
> - B：AWSが管理するマルチテナントの鍵で、専有ハードウェアや完全な自己管理という要件は満たさない。
> - C：顧客が作成・管理する鍵だが基盤はマルチテナントのKMSで、専有HSMが必須の要件には合わない。
> - D：認証情報・シークレットの保管とローテーションが主で、暗号鍵を専有HSMで管理するサービスではない。
> **関連ノート**：[[1607_AWS_KMS]] / [[1605_データ保護・暗号化]]

## 第6問（Management / Monitoring ｜ Organizations SCP vs IAMポリシー）
組織内のすべてのアカウントに対し、「特定リージョン以外は利用禁止」という上限（ガードレール）を、各アカウントのIAM設定に依存せず強制したい。最も適切な方法はどれか。

- A. AWS Organizations のサービスコントロールポリシー（SCP）を適用する
- B. 各アカウントの全IAMユーザーに拒否ポリシーをアタッチする
- C. 各アカウントのIAMロールを個別に修正する
- D. IAM パスワードポリシーを統一する

> [!success]- 答え：**A**
> SCPは組織（OU/アカウント）に適用する権限の上限（ガードレール）で、配下アカウントのIAMで何を許可しても、その範囲を超える操作を一律に禁止できる。
> - A：正解。アカウント横断で許可の上限を一括強制でき、個別IAM設定に依存しない。
> - B：アカウント・ユーザーごとの個別運用となり、漏れや上書きが起きやすく一括強制にならない。
> - C：ロールごとの個別修正で運用負荷が高く、新規作成分への抜け漏れも防げない。
> - D：パスワード強度の設定で、リージョン制限などの操作ガードレールとは無関係。
> **関連ノート**：[[1403_Organizations]] / [[0601_IAM基礎]]

## 第7問（Application Integration ｜ SQS vs SNS）
1つのイベントを、複数の独立したサブスクライバー（Lambda・メール通知・別システム）へ同時に一斉配信（プッシュ）したい。最も適切なサービスはどれか。

- A. Amazon SNS
- B. Amazon SQS
- C. Amazon SQS FIFOキュー
- D. AWS Step Functions

> [!success]- 答え：**A**
> SNSはPub/Subモデルで、1つのメッセージを複数のサブスクライバーへ同時にプッシュ配信（ファンアウト）できる。
> - A：正解。1対多の一斉プッシュ配信という要件に合致する。
> - B：1メッセージは原則1コンシューマーがポーリングで取り出すキューで、複数宛先への同時プッシュには向かない。
> - C：順序保証・重複排除のキューだが、これも複数宛先への一斉プッシュ配信の仕組みではない。
> - D：複数ステップのワークフロー制御が主で、イベントの一斉ファンアウト配信ハブではない。
> **関連ノート**：[[0803_SNS]] / [[0802_SQS]]

## 第8問（Serverless ｜ CloudFront Functions vs Lambda@Edge）
CloudFrontの閲覧リクエストに対し、HTTPヘッダーの書き換えやURLリライトといった非常に軽量・高頻度な処理を、最小のレイテンシーとコストでエッジ実行したい。ネットワークアクセスやファイルシステムは不要。最も適切なサービスはどれか。

- A. CloudFront Functions
- B. Lambda@Edge
- C. リージョンの AWS Lambda
- D. Amazon API Gateway

> [!success]- 答え：**A**
> CloudFront Functionsは超軽量・短時間のJavaScript処理をエッジロケーションで実行し、ヘッダー操作やURLリライトを極小のレイテンシー・コストで行える。
> - A：正解。軽量・高頻度・低レイテンシのエッジ処理という要件に最適。
> - B：オリジンリクエスト処理やネットワークアクセスなど高機能だが、その分レイテンシー・コストが高く、軽量処理にはオーバースペック。
> - C：リージョンで実行されるためエッジでの極小レイテンシ要件を満たさない。
> - D：APIの受け口・管理が主目的で、CloudFrontの閲覧リクエストをエッジで書き換える用途ではない。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0708_Lambda]]

## 第9問（Analytics ｜ EMR vs Glue）
既存のApache SparkとHadoopのジョブを大規模に実行したい。クラスターのインスタンスタイプや台数を細かく制御してチューニングし、長時間のビッグデータ処理を行いたい。最も適切なサービスはどれか。

- A. Amazon EMR
- B. AWS Glue
- C. Amazon Athena
- D. Amazon Redshift

> [!success]- 答え：**A**
> EMRはマネージドなHadoop/Sparkクラスターを提供し、インスタンス構成やチューニングを細かく制御して大規模なビッグデータ処理を実行できる。
> - A：正解。Spark/Hadoopのクラスターを制御・チューニングする大規模処理に最適。
> - B：サーバーレスETLでクラスター管理が不要な反面、インスタンスレベルの細かな制御には向かない。
> - C：S3上データへのアドホックなSQLクエリ用途で、Spark/Hadoopジョブの実行基盤ではない。
> - D：データウェアハウスで、既存のSpark/Hadoopジョブをそのまま動かす基盤ではない。
> **関連ノート**：[[1108_Amazon EMR]] / [[1107_AWS Glue]]

## 第10問（Migration / Transfer ｜ MGN vs DMS）
オンプレミスで稼働する多数のサーバー（OS・アプリ・設定を含む）を、できるだけ手を加えずそのままAWSのEC2へ「リフト＆シフト」で移行したい。最も適切なサービスはどれか。

- A. AWS Application Migration Service（MGN）
- B. AWS Database Migration Service（DMS）
- C. AWS DataSync
- D. AWS Snowball Edge

> [!success]- 答え：**A**
> MGN（Application Migration Service）はサーバーをブロックレベルで複製し、OS・アプリ・設定ごとEC2へリホスト（リフト＆シフト）移行する。
> - A：正解。サーバー全体をそのままEC2へ移行するリホストに最適。
> - B：データベースのスキーマ・データ移行に特化しており、サーバーOSごとの移行ではない。
> - C：ファイル/オブジェクトのデータ同期向けで、稼働サーバー全体の移行ではない。
> - D：大容量データの物理オフライン搬送向けで、サーバーのリホスト移行サービスではない。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[0301_RDS]]
