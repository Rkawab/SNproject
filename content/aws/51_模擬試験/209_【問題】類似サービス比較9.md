# 【問題】類似サービス比較9（移行・バックアップ・DR編・全10問）

## この問題集の目的

第2版（2026-07-14全面新作）。移行系（MGN/DMS/DataSync/Snow/Transfer Family/Storage Gateway）とバックアップ・DR系の使い分けを要件文で判断する。誤答はすべて実在の構成で、**別の要件なら正解になるもの**だけを置く。

## 対象レベル

SAA。「何を運ぶか（サーバー/DB/ファイル）」「オンラインかオフラインか」「守るのは何か」で切り分ける。

## 問題数

10問

## 第1問（Migration / Transfer ｜ データベースの移行）
オンプレのOracleデータベースを、**エンジンの異なる**Aurora PostgreSQLへ移行したい。移行中も業務は継続し、切替時のダウンタイムを最小化する必要がある。最も適切な組み合わせはどれか。

- A. AWS Application Migration Service（MGN）
- B. AWS DMS＋AWS Schema Conversion Tool（SCT）
- C. データベースのダンプを取得しS3経由でリストアする
- D. AWS DataSync

> [!success]- 答え：**B**
> 異種エンジン移行は「スキーマ変換＝SCT」＋「データ移行と継続レプリケーション＝DMS」のセット。稼働したまま同期し、短時間で切り替えられる。
> - A：**サーバーごと**（OS込み）そのまま移すリホストなら正解になる。エンジンを変える移行はサーバーコピーでは実現できない。
> - B：正解。「異種」「最小ダウンタイム」の2語で確定。
> - C：ダウンタイムが許容できる小規模・同種エンジンなら実在解。異種変換もダウンタイム最小化もカバーできない。
> - D：**ファイル**の転送サービス。データベースの論理移行はできない。
> **関連ノート**：[[1501_移行と転送サービス一覧]]

## 第2問（Migration / Transfer ｜ サーバー群の移行）
データセンター閉鎖に伴い、数百台の物理・仮想サーバーを**OS・ミドルウェア込みで変更なしに**EC2へ移行する。移行中も本番を動かし続け、カットオーバーは数分に抑えたい。最も適切なサービスはどれか。

- A. AWS Application Migration Service（MGN）
- B. VM Import/Export
- C. AWS DMS
- D. AWS Snowball Edgeでイメージを物理搬送

> [!success]- 答え：**A**
> MGNはブロックレベルの**継続レプリケーション**で稼働中サーバーをAWSへ複製し続け、テスト起動を挟んで数分のカットオーバーを実現する。リホスト（リフト&シフト）の第一候補。
> - A：正解。
> - B：一番きわどい誤答。仮想マシンイメージを**一度だけ**取り込むなら実在解。継続レプリケーションがないため、エクスポートからカットオーバーまでの差分が失われ、大規模移行の運びにも向かない。
> - C：**データベースの中身**を移すサービス。OSやミドルウェアは運ばない。
> - D：ネットワークが細い拠点からの**大容量データ**搬送なら正解になる。稼働中サーバーの数分カットオーバーは物理搬送では成立しない。
> **関連ノート**：[[1501_移行と転送サービス一覧]]

## 第3問（Migration / Transfer ｜ オンラインの大容量転送）
オンプレNAS上の500TBのファイルをS3へ移行する。**10Gbpsの専用線（Direct Connect）が既にあり**、転送の整合性検証・帯域制御・差分の再同期を自動化したい。最も適切なサービスはどれか。

- A. AWS Snowball Edge
- B. AWS DataSync
- C. AWS Transfer Family
- D. S3 マルチパートアップロードのスクリプト運用

> [!success]- 答え：**B**
> DataSyncはNFS/SMB/S3/EFS間のオンライン転送に特化し、並列化・チェックサム検証・帯域制御・差分同期が組み込み。「太い回線がある × 検証と差分同期」ならオンラインで運ぶ。
> - A：一番きわどい誤答。**回線が細い・期日が迫っている**なら正解になる。10Gbpsなら500TBは理論上1週間弱で送れるため、物理搬送の往復日数と手間をかける理由がない。
> - B：正解。
> - C：**取引先がSFTPで送ってくる**受け口を作るサービス。自社主導の大量移行の道具ではない。
> - D：実在する手だが、再試行・検証・差分管理を自作することになる。DataSyncがまさにその作り込みの製品化。
> **関連ノート**：[[1502_DataSync]] / [[1501_移行と転送サービス一覧]]

## 第4問（Storage ｜ 移行後のハイブリッド継続）
ファイルサーバーをS3へ移行した後も、オンプレの業務アプリは**NFSマウントのまま**同じデータを使い続ける必要がある。頻繁に使うファイルは**ローカルにキャッシュ**して低レイテンシで読みたい。最も適切なサービスはどれか。

- A. AWS DataSyncの定期同期タスク
- B. Amazon EFSをDirect Connect経由でマウント
- C. Amazon S3 File Gateway
- D. AWS Storage Gateway（Volume Gateway）

> [!success]- 答え：**C**
> S3 File GatewayはオンプレにNFS/SMBの入口を提供し、実体をS3に置きつつホットデータをローカルキャッシュする。「オンプレからNFSのまま × 実体はS3 × キャッシュ」の3点でこれ。
> - A：一方向の**定期コピー**なら正解になる。アプリが常時読み書きする「同じデータの窓口」にはならず、同期間隔分の不整合も生じる。
> - B：実在する構成で、クラウド側と**同一のファイルシステム**を共有したいなら候補になる。ただしローカルキャッシュがなく、全アクセスが回線越えになる。
> - C：正解。
> - D：オンプレへ**iSCSIブロックボリューム**を見せる型。ファイル共有（NFS）の要件とは型が違う。
> **関連ノート**：[[1503_Storage Gateway]]

## 第5問（Migration / Transfer ｜ オフライン転送の判断）
遠隔地の観測拠点に1.2PBの動画データがある。拠点の回線は**100Mbps**しかなく、**60日以内**にS3へ取り込みたい。最も適切な方法はどれか。

- A. AWS Snowball Edgeを複数台利用する
- B. AWS DataSyncで転送する
- C. S3 Transfer Accelerationで転送する
- D. Direct Connectを新規契約して転送する

> [!success]- 答え：**A**
> 100Mbpsでは全力でも約1.2PB≒3年かかる計算で、オンライン転送は物理的に間に合わない。「回線×期日で計算して無理ならSnow」が鉄則。
> - A：正解。数十TB〜PB級×細い回線×期日ありの定番解。
> - B：**回線が十分太ければ**第一候補になる（検証・差分同期も揃う）。本問は物理法則で脱落。
> - C：遠距離の**インターネット経路を改善する**機能で、100Mbpsという蛇口自体は太くならない。
> - D：契約から開通まで数週間〜数か月かかるうえ、観測拠点まで専用線を引く費用対効果が合わない。恒常的なハイブリッド接続の答え。
> **関連ノート**：[[1501_移行と転送サービス一覧]]

## 第6問（Migration / Transfer ｜ 取引先とのファイル授受）
数十社の取引先が、**既存のSFTPクライアントと手順を変えずに**自社へファイルを送ってくる。受け取ったファイルはS3に直接保存し、サーバー管理はなくしたい。最も適切なサービスはどれか。

- A. EC2でSFTPサーバーを自前運用しS3へ転送
- B. S3署名付きURLを取引先へ都度発行
- C. AWS Transfer Family
- D. AWS DataSync

> [!success]- 答え：**C**
> Transfer FamilyはSFTP/FTPS/AS2のマネージドエンドポイントを提供し、バックエンドをS3/EFSにできる。「相手の手順を変えられない × SFTP × サーバーレス」が決め手。
> - A：固定IPや特殊な認証方式など細かい要件があれば実在解だが、パッチ・冗長化・監視を自前で抱える。マネージドで足りるなら選ばない。
> - B：取引先が**S3のHTTP APIを使える**なら軽量な正解になる。SFTPクライアント運用を変えられないという前提に反する。
> - C：正解。
> - D：**自社が管理する両端**（NASとS3等）の間の転送サービス。不特定の取引先の受け口ではない。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[1502_DataSync]]

## 第7問（Storage ｜ バックアップの一元化）
EC2・EBS・RDS・DynamoDB・EFSが混在する環境で、バックアップの取得スケジュール・保持期間・クロスリージョンコピーを**ポリシーで一元管理**し、準拠状況をレポートしたい。最も適切なサービスはどれか。

- A. Amazon Data Lifecycle Manager（DLM）
- B. AWS Backup
- C. 各サービスの自動バックアップ機能を個別に設定
- D. EventBridge＋Lambdaでスナップショット取得を自動化

> [!success]- 答え：**B**
> AWS Backupはサービス横断のバックアップポリシー（プラン）を組織規模で適用し、監査レポートまで一元化する。「複数サービス横断 × ポリシー一元管理」が決め手。
> - A：一番きわどい誤答。対象が**EBSスナップショットとAMIだけ**なら正解になる軽量ツール。RDSやDynamoDBはカバーしない。
> - B：正解。
> - C：それぞれは実在の機能だが、統制・レポート・保持ポリシーがバラバラになり、監査要件に応えられない。
> - D：AWS Backup登場以前の自前構成。特殊な取得タイミング制御が要るなら今も使うが、標準要件では車輪の再発明。
> **関連ノート**：[[1802_AWS_Backup]]

## 第8問（Storage ｜ バックアップの削除防止）
ランサムウェア対策として、AWS Backupで取得したバックアップを、保持期間中は**管理者や侵害されたアカウントでも削除できない**ようにしたい。最も適切な方法はどれか。

- A. AWS Backup Vault Lock（コンプライアンスモード）
- B. S3 Object Lock
- C. バックアップ専用アカウントへコピーしSCPで削除を拒否
- D. バックアップをKMSで暗号化する

> [!success]- 答え：**A**
> Vault Lockはバックアップボールトに対するWORM設定で、コンプライアンスモードならrootでも削除・設定変更ができない。「AWS Backupの保管庫を不可侵にする」専用機能。
> - A：正解。
> - B：守る対象が**S3バケット内の自前オブジェクト**なら正解になる同思想の機能。AWS Backupのボールトには適用できない。
> - C：実在する多層防御で、**併用は推奨**される。ただしSCPは組織管理者が変更でき、「誰にも削除できない保証」の一次装置にはならない。
> - D：暗号化は**盗み見**への対策。削除・破壊への対策にはならない。
> **関連ノート**：[[1802_AWS_Backup]] / [[0205_S3のバージョン管理]]

## 第9問（Migration / Transfer ｜ DR戦略の選択）
基幹システムのDRを設計する。要件はRTO 30分・RPO数分。**平常時のDRサイトのコストは最小限**にしたいが、データベースだけは常時レプリケーションしておきたい。最も適切なDR戦略はどれか。

- A. バックアップ＆リストア
- B. パイロットライト
- C. ウォームスタンバイ
- D. マルチサイト（アクティブ/アクティブ）

> [!success]- 答え：**B**
> パイロットライトは「種火」＝データ層だけ常時同期し、アプリ層は障害時に起動する。RTO数十分・RPO数分・低コストの交点。「DBだけ常時、他は止めておく」の記述そのもの。
> - A：RTOが**数時間〜1日**許されるなら最安の正解になる。30分では復元が間に合わない。
> - C：縮小版とはいえ**アプリ層まで常時稼働**させる戦略。RTOを数分に縮めたいなら正解になるが、平常時コスト最小の要件に反する。
> - D：RTO/RPOほぼゼロが必須のミッションクリティカルなら正解になる最高コスト構成。
> - B：正解。
> **関連ノート**：[[1801_DR戦略4パターン（RTO_RPO）]]

## 第10問（Migration / Transfer ｜ 低RPOのサーバーDR）
オンプレの基幹サーバー群について、**RPO秒単位・RTO数分**のDRをAWS上に構築したい。平常時のAWS側は最小限の安価なリソースで待機させたい。最も適切なサービスはどれか。

- A. AWS Backupで日次バックアップをクロスリージョンコピー
- B. AWS Elastic Disaster Recovery（DRS）
- C. AWS Application Migration Service（MGN）
- D. EBSスナップショットを1時間ごとに取得

> [!success]- 答え：**B**
> DRSはブロックレベルの**継続レプリケーション**を低スペックのステージング領域で受け、被災時に本番サイズで数分起動する。「RPO秒 × RTO分 × 平常時は安く」のDR専用サービス。
> - A：RPOが**24時間**でよい一般システムなら正解になる王道。秒単位のRPOには桁が足りない。
> - B：正解。
> - C：一番きわどい誤答。技術基盤はDRSと同系だが、こちらは**片道の移行（カットオーバーして終わり）**用。継続的なDR運用（定期訓練・フェイルバック）の機能はDRSが持つ。
> - D：RPOは最良でも1時間。スナップショット間隔を縮める発想には転送・コストの限界がある。
> **関連ノート**：[[1802_AWS_Backup]] / [[1801_DR戦略4パターン（RTO_RPO）]]
