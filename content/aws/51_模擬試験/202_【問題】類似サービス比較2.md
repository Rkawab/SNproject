セキュリティ、ネットワーク、分析、移行、ガバナンスの紛らわしい選択肢を見分ける。

## 第1問（Networking / CDN / LB ｜ グローバル配信）
企業は世界中のユーザーへ静的コンテンツを配信するWebサイトを運営している。ユーザーに近いエッジロケーションでコンテンツをキャッシュし、オリジンへの負荷を削減したい。最も適切なサービスはどれか。

- A. Amazon CloudFront
- B. AWS Global Accelerator
- C. Amazon S3 Transfer Acceleration
- D. Amazon Route 53のレイテンシールーティング

> [!success]- 答え：**A**
> CloudFrontはCDNとしてエッジでコンテンツを**キャッシュ**する。
> - A：正解。「エッジでキャッシュ」「オリジン負荷削減」はCloudFrontの定義そのもの。
> - B：きわどい誤答。同じくエッジを使うが**経路の最適化**であってキャッシュはしない。「キャッシュ」の一語で切る。
> - C：エッジ経由なのは同じだが**S3へのアップロードを速くする**機能。配信（ダウンロード）ではなく向きが逆。
> - D：近いリージョンへDNSで振り分けるだけで、エッジのキャッシュもオリジン負荷削減もない。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]]

## 第2問（Networking / CDN / LB ｜ 固定IPのグローバル入口）
企業は複数リージョンのアプリケーションに対し、固定のグローバルIPアドレスを提供し、AWSグローバルネットワークで最適なリージョンへルーティングしたい。最も適切なサービスはどれか。

- A. AWS Global Accelerator
- B. Amazon CloudFront
- C. Amazon Route 53のレイテンシールーティング
- D. 各リージョンのNetwork Load BalancerにElastic IPを割り当てる

> [!success]- 答え：**A**
> Global Acceleratorは固定のAnycast IPを2つ提供し、AWSグローバルネットワークで最適リージョンへ転送する。
> - A：正解。「固定のグローバルIP」＋「AWSネットワークで最適ルーティング」の両方を1つで満たす。
> - B：CDN（キャッシュ配信）であり、固定IPも提供しない。
> - C：近いリージョンへの振り分けはできるが、返るIPはリージョンごとに異なり「固定のグローバルIP」にならない。
> - D：きわどい誤答。NLBのEIPは固定だが**リージョンごとに別のIP**で、グローバルに単一のIPを提示できず、リージョン間の自動ルーティングもない。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0403_Elastic IP]]

## 第3問（Security / IAM ｜ 鍵管理）
企業はS3とRDSの保存時暗号化に使用する暗号鍵のアクセス制御と利用履歴を管理したい。最も適切なサービスはどれか。

- A. AWS Key Management Service
- B. AWS Secrets Manager
- C. AWS Certificate Manager
- D. AWS CloudHSM

> [!success]- 答え：**A**
> KMSは暗号鍵の作成、管理、アクセス制御、監査連携（CloudTrail）を提供する。
> - A：正解。S3/RDSのSSE-KMS連携・キーポリシーによる制御・利用履歴の監査をまとめて満たす標準解。
> - B：管理するのは鍵ではなく**パスワードやAPIキー等の秘密の値**。「暗号鍵」と「認証情報」の混同を突く。
> - C：管理するのは**TLS証明書**。暗号つながりのひっかけ。
> - D：きわどい誤答。専有HSMで鍵を管理できるが、S3/RDSの標準的な保存時暗号化には過剰で運用負荷も高い。「FIPS 140-2 Level 3必須」のような明示がない限りKMSを選ぶ。
> **関連ノート**：[[1607_AWS_KMS]] / [[1605_データ保護・暗号化]]

## 第4問（Security / IAM ｜ 脅威検知）
セキュリティチームは、VPC Flow Logs、DNSログ、CloudTrailイベントなどを分析し、不審なAPI操作や通信を自動検知したい。最も適切なサービスはどれか。

- A. Amazon GuardDuty
- B. Amazon Inspector
- C. Amazon Macie
- D. AWS Security Hub

> [!success]- 答え：**A**
> GuardDutyは各種ログ（VPC Flow Logs・DNS・CloudTrail）を機械学習で分析する「見張り役」。
> - A：正解。問題文の「ログを分析して不審な挙動を自動検知」はGuardDutyの定義そのもの。
> - B：Inspectorが見るのは**ソフトウェアの脆弱性（CVE）**。「攻撃されている兆候」ではなく「攻撃されうる穴」を探す人。
> - C：Macieが見るのは**S3内の機密データ**。ログ分析はしない。
> - D：きわどい誤答。Security Hubは各サービスの検出結果を**集約・整理**する係で、自らログを分析して検知はしない。「検知する人」と「まとめる人」の混同を突く。
> **関連ノート**：[[1603_脅威検知・監査]]

## 第5問（Security / IAM ｜ 脆弱性管理）
企業はEC2インスタンスとコンテナイメージのソフトウェア脆弱性を継続的に検出したい。最も適切なサービスはどれか。

- A. Amazon Inspector
- B. Amazon GuardDuty
- C. AWS Systems Manager Patch Manager
- D. AWS Security Hub

> [!success]- 答え：**A**
> InspectorはEC2・ECRコンテナイメージ・Lambdaの**脆弱性（CVE）を継続スキャン**する。
> - A：正解。「ソフトウェア脆弱性を継続的に検出」はInspectorの担当そのもの。
> - B：GuardDutyは**不審な挙動（脅威）の検知**。「穴を探す（Inspector）」と「侵入の兆候を見張る（GuardDuty）」で役割が違う。
> - C：きわどい誤答。Patch Managerは**パッチを適用する**係。脆弱性を「見つける」のはInspectorで、見つけた後に「直す」のがPatch Manager。
> - D：検出結果を集約するダッシュボードで、スキャン自体は行わない。
> **関連ノート**：[[1603_脅威検知・監査]] / [[1405_Systems Manager]]

## 第6問（Analytics ｜ ETL）
分析基盤では、S3上の生データを変換し、パーティション化して別のS3プレフィックスへ保存する必要がある。サーバーレスでETLジョブとデータカタログを管理したい。最も適切なサービスはどれか。

- A. AWS Glue
- B. Amazon Athena
- C. Amazon EMR
- D. AWS Data Pipeline

> [!success]- 答え：**A**
> GlueはサーバーレスETLとデータカタログを提供する。
> - A：正解。「サーバーレスでETL＋カタログ」の2条件を1つで満たす。
> - B：AthenaはS3データへの**SQLクエリ（読む側）**。変換して書き出すETLの主役ではない。
> - C：きわどい誤答。EMRでも同じ変換はできるが**クラスター管理が必要**。「サーバーレス」の一語で切る（細かい制御が要件ならEMRが逆に正解になる）。
> - D：レガシーなデータ移動オーケストレーション（現在は非推奨方向）。新規設計でGlueより優先する理由がない。
> **関連ノート**：[[1107_AWS Glue]] / [[1108_Amazon EMR]] / [[1112_AWS Data Pipeline]]

## 第7問（Analytics ｜ BI可視化）
経営層は、RedshiftとS3にある分析データをダッシュボードで可視化したい。マネージドなBIサービスを利用したい。最も適切なサービスはどれか。

- A. Amazon QuickSight
- B. Amazon Managed Grafana
- C. Amazon OpenSearch Service
- D. AWS Lake Formation

> [!success]- 答え：**A**
> QuickSightはAWSのマネージドBI（ビジネスインテリジェンス）サービス。
> - A：正解。Redshift/S3(Athena)をデータソースに、経営層向けのダッシュボード・グラフを作る用途そのもの。
> - B：きわどい誤答。同じ「可視化」でもGrafanaは**運用メトリクス**（CloudWatch/Prometheus）向け。ビジネスデータのBIならQuickSight。
> - C：可視化機能（Dashboards）はあるが**ログ検索・分析**が主戦場。経営層向けBIではない。
> - D：データレイクの**権限管理・構築支援**であり、可視化ツールではない。
> **関連ノート**：[[1111_Amazon QuickSight]] / [[1110_Amazon OpenSearch Service]]

## 第8問（Migration / Transfer ｜ 移行方式）
オンプレミスからAWSへ数百TBのアーカイブデータを初回移行する。ネットワーク帯域が限られており、オンライン転送では期限に間に合わない。最も適切なサービスはどれか。

- A. AWS Snowball Edge
- B. AWS DataSync
- C. Amazon S3 Transfer Acceleration
- D. AWS Transfer Family

> [!success]- 答え：**A**
> Snowball Edgeは物理デバイスにデータを積んで運ぶ「オフライン輸送」。回線がボトルネックなら物理で運ぶ。
> - A：正解。「数百TB＋帯域が細い＋期限あり」の3点が揃ったら物理輸送一択。
> - B：きわどい誤答。DataSyncは高速なオンライン転送だが、**細い回線の物理的な限界は超えられない**。「帯域が限られ、間に合わない」で切る。
> - C：アップロードの高速化はできるが、これも**回線の帯域以上は出ない**。エッジ経由で遅延を減らすだけ。
> - D：Transfer FamilyはSFTP等の**プロトコルを維持した受け口**の話で、大容量移行の速度問題は解決しない。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[1502_DataSync]]

## 第9問（Management / Monitoring ｜ 構成管理）
企業はAWSリソースの設定変更履歴を記録し、S3バケットが公開状態になっていないかなどのルールを継続的に評価したい。最も適切なサービスはどれか。

- A. AWS Config
- B. AWS CloudTrail
- C. Amazon CloudWatch
- D. AWS Trusted Advisor

> [!success]- 答え：**A**
> Configはリソース構成を記録し、ルールで準拠状態を評価する。
> - A：正解。設定変更履歴とコンプライアンス評価に使う。
> - B：API操作証跡の記録が中心。
> - C：メトリクス、ログ、アラームの監視が中心。
> - D：ベストプラクティスチェックを提供するが、継続的なリソース設定記録の主サービスではない。
> **関連ノート**：[[1003_AWS_Config]]

## 第10問（Security / IAM ｜ 公開API防御）
インターネット公開APIに対して、HTTPリクエストのパターンを検査し、SQLインジェクションや既知の悪性IPからのアクセスをブロックしたい。ALBまたはCloudFrontに関連付けて利用できるサービスはどれか。

- A. AWS WAF
- B. AWS Network Firewall
- C. AWS Shield Advanced
- D. Amazon GuardDuty

> [!success]- 答え：**A**
> WAFはWeb ACLをALB/CloudFront/API Gatewayに関連付けて、HTTPリクエストの中身を検査・ブロックする。
> - A：正解。SQLi/XSS等のパターン検査（マネージドルールあり）・IPセット・レートベースルールが揃い、ALB/CloudFrontに直接アタッチできる。
> - B：きわどい誤答。Network Firewallは**VPCのサブネット境界**に置くネットワーク層の防御で、ALB/CloudFrontに関連付けるものではない。
> - C：Shield Advancedは**DDoS対策**。SQLiのようなリクエスト内容の検査はWAFの領分（実はShield AdvancedにWAFが付属する関係も覚えておくと強い）。
> - D：GuardDutyは**事後の脅威検知**（ログ分析）で、リクエストをインラインでブロックできない。
> **関連ノート**：[[1604_ネットワーク防御]]
