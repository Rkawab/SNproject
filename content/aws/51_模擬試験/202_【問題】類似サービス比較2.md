# 【問題】類似サービス比較 ②（4択・全20問）

## この問題集の目的

[[201_【問題】類似サービス比較]]の続編。1問目では扱わなかった「混同しやすい組み合わせ」（ストレージクラス・分析・ネットワーク接続・セキュリティ）の違いを、要件から選び分けられるようにする。

## 対象レベル

基礎〜SAA標準。各問で「なぜ似た他サービスではないか」「外したサービスはどんな場面なら正解か」を必ず確認する。

## 問題数

20問（S3ストレージクラス・Glacier取り出し・Athena/Redshift・EMR/Athena・Kinesis Streams/Firehose・Kinesis/SQS・ElastiCache/DAX・Redis/Memcached・CloudFront/Global Accelerator・Direct Connect/VPN・VPCピアリング/Transit Gateway・ゲートウェイ型/インターフェース型エンドポイント・NAT GW/IGW・Cognito/IAM・STS/アクセスキー・SCP/IAMポリシー・Config/CloudTrail・GuardDuty/Inspector・DataSync/Snowball・EBS gp3/io2）

## 使い方

正解の根拠だけでなく、**外した似たサービスがどんな場面なら正解になるか**をセットで覚える。比較表を自分で書けるようになるのがゴール。

## 第1問（S3ストレージクラス）
アクセス頻度は低いが、必要時にはすぐ取り出したいデータを、複数AZの高耐久を保ちつつ保管コストを下げたい。最適なS3クラスはどれか。

- A. S3 Standard
- B. S3 Standard-IA
- C. S3 One Zone-IA
- D. S3 Glacier Deep Archive

> [!success]- 答え：**B**（Standard-IA）
> 「低頻度アクセスだが即時取り出し・複数AZ耐久」はStandard-IA。保管は安く、取り出し時に取り出し料金がかかる。
> **他の選択肢が違う理由**
> - A：Standardは頻繁アクセス向けで保管コストが高い。
> - C：One Zone-IAは単一AZで安いが、AZ障害でデータを失うため「高耐久維持」の要件に反する。
> - D：Deep Archiveは取り出しに数時間かかり「すぐ取り出したい」に反する。
> **覚えるべきキーワード**：Standard-IA＝低頻度・即時・複数AZ / One Zone-IA＝単一AZで安い
> **関連サービス**：Intelligent-Tiering / Glacier / ライフサイクル
> **SAAでの出題イメージ**：「低頻度だが即時・高耐久」=Standard-IA、「再取得容易で安く」=One Zone-IA。
> 📎 参照：[[0203_S3ストレージクラス]]

## 第2問（Glacierの取り出し時間）
規制対応で長期保管するデータを、年に数回・取り出しに「数時間待てる」前提で、保管コストを最小にしたい。最適なクラスはどれか。

- A. S3 Glacier Instant Retrieval
- B. S3 Glacier Flexible Retrieval
- C. S3 Glacier Deep Archive
- D. S3 Standard-IA

> [!success]- 答え：**C**（Glacier Deep Archive）
> 「滅多に取り出さない・数時間待てる・最安」はDeep Archive。S3で最も保管コストが低く、規制アーカイブの定番。
> **他の選択肢が違う理由**
> - A：Instant Retrievalはミリ秒取り出しだが保管単価は高め（即時が要る低頻度向け）。
> - B：Flexible Retrievalは分〜時間の取り出しで、Deep Archiveより保管単価が高い。
> - D：Standard-IAはアーカイブほど安くない。
> **覚えるべきキーワード**：Deep Archive＝最安・数時間 / Instant＝即時取り出し / Flexible＝分〜時間
> **関連サービス**：ライフサイクル / Object Lock（改ざん防止）
> **SAAでの出題イメージ**：「7〜10年保管・滅多に出さない・最安」=Glacier Deep Archive。
> 📎 参照：[[0203_S3ストレージクラス]]

## 第3問（Athena / Redshift）
S3に置いた数百GBのログを、たまに・その都度SQLで調べたい。常時稼働のクラスタは持ちたくない。最適なのはどれか。

- A. Amazon Athena
- B. Amazon Redshift
- C. Amazon RDS
- D. Amazon DynamoDB

> [!success]- 答え：**A**（Athena）
> 「たまに・アドホック・クラスタ管理なし・スキャン量課金」はAthena。S3を直接SQLで分析でき、待機コストがない。
> **他の選択肢が違う理由**
> - B：Redshiftは常時稼働クラスタが前提の本格DWH。継続的に大量分析する場合に有利だが、たまの利用には過剰・割高。
> - C：RDSはOLTP向けでS3直接クエリ不可。
> - D：DynamoDBはキー値で集計分析向きではない。
> **覚えるべきキーワード**：Athena＝サーバーレス・アドホック / Redshift＝常時DWH
> **関連サービス**：Glue / S3 / QuickSight
> **SAAでの出題イメージ**：「たまにS3をSQL」=Athena、「継続的に大量分析」=Redshift。
> 📎 参照：[[1109_Amazon Athena]] / [[1113_Amazon Redshift]]

## 第4問（EMR / Athena）
ペタバイト級データに対し、Spark/Hadoopによる大規模な分散ETL・機械学習前処理をカスタムコードで実行したい。最適なのはどれか。

- A. Amazon Athena
- B. Amazon EMR
- C. Amazon Redshift
- D. AWS Glue DataBrew

> [!success]- 答え：**B**（EMR）
> 大規模分散処理（Spark/Hadoop/Hive等）をフレームワーク込みで柔軟に動かすならEMR。クラスタを制御しカスタム処理を組める。
> **他の選択肢が違う理由**
> - A：Athennaはサーバーレスのアドホック「SQLクエリ」向けで、Sparkジョブのフル制御や大規模分散ETLの主役ではない。
> - C：RedshiftはDWH（SQL分析）でフレームワーク任意の分散処理基盤ではない。
> - D：DataBrewはノーコードのデータ準備で、ペタバイト級カスタム分散処理向けではない。
> **覚えるべきキーワード**：EMR＝Spark/Hadoopクラスタ / Athena＝サーバーレスSQL
> **関連サービス**：Glue / S3 / Athena
> **SAAでの出題イメージ**：「大規模分散・フレームワーク制御」=EMR、「手軽なSQL」=Athena。
> 📎 参照：[[1108_Amazon EMR]] / [[1109_Amazon Athena]]

## 第5問（Kinesis Data Streams / Firehose）
ストリーミングデータを、コードをほとんど書かずにS3へバッファして配信・蓄積したい（ニアリアルタイムで可）。最適なのはどれか。

- A. Kinesis Data Streams
- B. Kinesis Data Firehose
- C. Amazon SQS
- D. Amazon MSK

> [!success]- 答え：**B**（Firehose）
> FirehoseはストリームをS3/Redshift/OpenSearch等へ「自動でバッファ配信」するフルマネージドのデリバリー。コード最小・ニアリアルタイム蓄積に向く。
> **他の選択肢が違う理由**
> - A：Data Streamsは順序保持・複数コンシューマの「リアルタイム処理」基盤で、配信先への投入は自前実装が必要。
> - C：SQSはキューで、ストレージへの自動配信機能ではない。
> - D：MSKはマネージドKafkaで、配信先への自動デリバリーが主目的ではない。
> **覚えるべきキーワード**：Firehose＝自動配信・コード最小 / Streams＝リアルタイム処理・順序
> **関連サービス**：S3 / Redshift / OpenSearch / Lambda（変換）
> **SAAでの出題イメージ**：「ストリームをS3へ楽に蓄積」=Firehose、「順序保持で処理」=Data Streams。
> 📎 参照：[[0805_Kinesis]] / [[1103_Amazon Kinesis]]

## 第6問（Kinesis / SQS）
同じデータを複数の独立した処理系が、それぞれ自分の進捗で（巻き戻し再処理も可能に）並行して読みたい。最適なのはどれか。

- A. Amazon SQS 標準キュー
- B. Amazon Kinesis Data Streams
- C. Amazon SQS FIFOキュー
- D. Amazon SNS

> [!success]- 答え：**B**（Kinesis Data Streams）
> Kinesisはデータを一定期間保持し、複数コンシューマが各自のシーケンス位置で並行に読め、巻き戻し再処理もできる。
> **他の選択肢が違う理由**
> - A/C：SQSはメッセージを取得・削除すると消え、基本1コンシューマが取り切るモデル。複数系の並行読み・巻き戻しには不向き。
> - D：SNSはPub/Sub通知で、ストリームの保持・再処理はしない。
> **覚えるべきキーワード**：Kinesis＝保持・複数コンシューマ・再処理 / SQS＝取得で消える
> **関連サービス**：Firehose / Lambda / SNS+SQS（ファンアウト）
> **SAAでの出題イメージ**：「同じデータを複数系が並行・巻き戻し」=Kinesis、「1回処理の疎結合」=SQS。
> 📎 参照：[[0805_Kinesis]] / [[0802_SQS]]

## 第7問（ElastiCache / DAX）
DynamoDBの読み取りを、アプリ改修を最小にしてマイクロ秒級まで高速化したい。最適なのはどれか。

- A. Amazon ElastiCache for Redis
- B. DynamoDB Accelerator（DAX）
- C. Amazon RDS Read Replica
- D. Amazon CloudFront

> [!success]- 答え：**B**（DAX）
> DAXはDynamoDB専用のインメモリキャッシュで、DynamoDB互換APIのまま前段に挟むだけ。マイクロ秒級読み取りをアプリ改修最小で得られる。
> **他の選択肢が違う理由**
> - A：ElastiCacheは汎用キャッシュで、キャッシュ整合・取得ロジックを自前実装する必要があり改修が大きい。
> - C：リードレプリカはRDSの用語でDynamoDBにはない概念。
> - D：CloudFrontはコンテンツ配信CDNでDBキャッシュではない。
> **覚えるべきキーワード**：DAX＝Dynamo専用・互換API・改修最小 / ElastiCache＝汎用・自前管理
> **関連サービス**：DynamoDB / ElastiCache / グローバルテーブル
> **SAAでの出題イメージ**：「DynamoDBをマイクロ秒・改修最小」=DAX。
> 📎 参照：[[0303_DynamoDB]] / [[0304_ElastiCache]]

## 第8問（ElastiCache Redis / Memcached）
セッションやリーダーボードを、レプリケーション・自動フェイルオーバー・永続化・Pub/Subなど高機能に扱える分散キャッシュが欲しい。最適なエンジンはどれか。

- A. ElastiCache for Memcached
- B. ElastiCache for Redis
- C. DynamoDB
- D. Amazon MQ

> [!success]- 答え：**B**（Redis）
> Redisはレプリケーション・Multi-AZ自動フェイルオーバー・スナップショット永続化・データ構造（ソート済セット等）・Pub/Subを備える高機能キャッシュ。リーダーボードやセッション共有に向く。
> **他の選択肢が違う理由**
> - A：Memcachedはシンプル・マルチスレッドで水平スケールに強いが、レプリケーション/永続化/高度なデータ型を持たない。
> - C：DynamoDBはNoSQL DBでインメモリキャッシュエンジンではない。
> - D：Amazon MQはメッセージブローカーでキャッシュではない。
> **覚えるべきキーワード**：Redis＝高機能・永続化・フェイルオーバー / Memcached＝単純・水平スケール
> **関連サービス**：DAX / DynamoDB / セッション外部化
> **SAAでの出題イメージ**：「高機能・冗長・永続が要る」=Redis、「単純な揮発キャッシュ」=Memcached。
> 📎 参照：[[0304_ElastiCache]]

## 第9問（CloudFront / Global Accelerator）
キャッシュできない動的・非HTTPのトラフィック（ゲーム/VoIP/APIなど）を、AWSのグローバル網経由で高速・安定化し、固定の入口IP（Anycast）も欲しい。最適なのはどれか。

- A. Amazon CloudFront
- B. AWS Global Accelerator
- C. Amazon Route 53
- D. Elastic Load Balancing

> [!success]- 答え：**B**（Global Accelerator）
> Global Acceleratorはエッジから最寄りのAWSリージョンへAWSバックボーン経由で誘導し、動的・非HTTPでも低遅延・安定化する。固定のAnycast IP（2つの静的IP）を提供。
> **他の選択肢が違う理由**
> - A：CloudFrontはコンテンツをエッジでキャッシュするCDNで、キャッシュできない動的/非HTTPの高速化が主目的ではない。
> - C：Route 53はDNS誘導で、経路そのものの高速化やAnycast固定IPは提供しない。
> - D：ELBはリージョン内の負荷分散でグローバル高速化ではない。
> **覚えるべきキーワード**：Global Accelerator＝非キャッシュ・固定IP・AWS網 / CloudFront＝キャッシュCDN
> **関連サービス**：CloudFront / Route 53 / NLB
> **SAAでの出題イメージ**：「動的/非HTTPを高速化・固定IP」=Global Accelerator、「静的配信」=CloudFront。
> 📎 参照：[[0405_ネットワークとコンテンツ配信]]

## 第10問（Direct Connect / VPN）
オンプレとAWS間で、安定した広帯域・低遅延・一貫した品質が必須の本番接続を確立したい。最適なのはどれか。

- A. AWS Site-to-Site VPN
- B. AWS Direct Connect
- C. AWS Client VPN
- D. VPCピアリング

> [!success]- 答え：**B**（Direct Connect）
> 安定・広帯域・低遅延・一貫品質はDirect Connect（専用線）。インターネットを経由しないため帯域と遅延が安定する。
> **他の選択肢が違う理由**
> - A：Site-to-Site VPNはインターネット経由で構築が速く安価だが品質はネット依存（DXのバックアップ向き）。
> - C：Client VPNは個々の端末からの接続用で、拠点間の専用接続ではない。
> - D：VPCピアリングはVPC間接続でオンプレ接続ではない。
> **覚えるべきキーワード**：DX＝専用線・安定品質 / VPN＝ネット経由・即時・安価
> **関連サービス**：Direct Connect Gateway / Transit Gateway / VPN（冗長化）
> **SAAでの出題イメージ**：「安定品質の専用接続」=DX、「即時/安価/バックアップ」=VPN。
> 📎 参照：[[0405_ネットワークとコンテンツ配信]] / [[0406_VPC応用機能]]

## 第11問（VPCピアリング / Transit Gateway）
多数（数十）のVPCとオンプレを相互接続する。1対1接続が増えすぎて管理が破綻しないよう、ハブ＆スポークで一元管理したい。最適なのはどれか。

- A. VPCピアリングを全VPC間にフルメッシュで張る
- B. AWS Transit Gateway
- C. インターネットゲートウェイ
- D. NAT Gateway

> [!success]- 答え：**B**（Transit Gateway）
> Transit Gateway（TGW）は多数のVPC・オンプレVPN/DXを中央ハブで接続するルーター。ピアリングのフルメッシュ（N×N）の複雑さを解消し、一元管理できる。
> **他の選択肢が違う理由**
> - A：ピアリングは1対1・非推移的で、数が増えると接続数が爆発し管理が破綻する（少数の単純接続向き）。
> - C：IGWはインターネット出入口でVPC相互接続ではない。
> - D：NAT Gatewayは外向き通信用でVPC間接続ではない。
> **覚えるべきキーワード**：TGW＝ハブ＆スポーク・多数集約 / ピアリング＝1対1・非推移
> **関連サービス**：VPCピアリング / Direct Connect Gateway / VPN
> **SAAでの出題イメージ**：「多数VPC/オンプレを一元接続」=Transit Gateway。
> 📎 参照：[[0406_VPC応用機能]]

## 第12問（ゲートウェイ型 / インターフェース型エンドポイント）
プライベートサブネットのEC2から、CloudWatchやSQSなどのAWSサービスへ、プライベートIP経由（PrivateLink）で接続したい。最適なエンドポイント種別はどれか。

- A. ゲートウェイ型VPCエンドポイント
- B. インターフェース型VPCエンドポイント（PrivateLink）
- C. NAT Gateway
- D. インターネットゲートウェイ

> [!success]- 答え：**B**（インターフェース型エンドポイント）
> インターフェース型はENI（プライベートIP）を作り、PrivateLinkで多くのAWSサービス（CloudWatch/SQS/KMS等）へプライベート接続する。
> **他の選択肢が違う理由**
> - A：ゲートウェイ型はS3とDynamoDBのみ対応（ルートテーブル経由・追加料金なし）。CloudWatch/SQS等には使えない。
> - C：NAT Gatewayはインターネット経由の外向き通信で、プライベート接続ではない。
> - D：IGWはインターネット出入口でプライベート接続ではない。
> **覚えるべきキーワード**：インターフェース型＝ENI/PrivateLink/多サービス / ゲートウェイ型＝S3・DynamoDB限定・無料
> **関連サービス**：PrivateLink / S3・DynamoDB（ゲートウェイ型） / Route 53
> **SAAでの出題イメージ**：「S3/DynamoDBへ無料でプライベート」=ゲートウェイ型、「他サービスへ」=インターフェース型。
> 📎 参照：[[0406_VPC応用機能]]

## 第13問（NAT Gateway / Internet Gateway）
プライベートサブネットのEC2に、OS更新のための「外向きインターネット通信のみ」を許可したい（外部からの着信は不可）。最適なのはどれか。

- A. インターネットゲートウェイ＋パブリックIP
- B. NAT Gateway
- C. ゲートウェイ型VPCエンドポイント
- D. Egress-Onlyインターネットゲートウェイ（IPv4で）

> [!success]- 答え：**B**（NAT Gateway）
> NAT Gatewayはプライベートサブネットのリソースに「外向きのみ」のインターネット接続を提供する一方通行。外部から開始する通信は通さない。
> **他の選択肢が違う理由**
> - A：IGW＋パブリックIPは双方向（外部からの着信も可能）で「外向きのみ」に反する。
> - C：ゲートウェイ型エンドポイントはS3/DynamoDB限定で一般的なインターネット更新には使えない。
> - D：Egress-Only IGWはIPv6専用の外向き専用ゲートウェイ。IPv4の外向きはNAT Gatewayが正解。
> **覚えるべきキーワード**：NAT GW＝IPv4外向きのみ / IGW＝双方向 / Egress-Only IGW＝IPv6外向き
> **関連サービス**：ルートテーブル / インターネットゲートウェイ / VPCエンドポイント
> **SAAでの出題イメージ**：「プライベートからIPv4で外向きだけ」=NAT Gateway。
> 📎 参照：[[0406_VPC応用機能]]

## 第14問（Cognito / IAM）
スマホアプリのエンドユーザー（会員）のサインアップ・ログインを管理したい。AWSアカウント内の管理者・サービスの権限管理ではない。最適なのはどれか。

- A. Amazon Cognito
- B. AWS IAM
- C. AWS IAM Identity Center
- D. AWS Directory Service

> [!success]- 答え：**A**（Cognito）
> アプリの「会員」認証はCognito（ユーザープール）。サインアップ/サインイン・ソーシャル/SAML連携・MFAを担う。
> **他の選択肢が違う理由**
> - B：IAMはAWSリソースを操作する利用者・サービスの認可で、アプリ会員向けではない。
> - C：Identity Centerは組織のAWSアカウント/アプリへのSSO（従業員向け）でアプリ会員向けではない。
> - D：Directory Serviceは企業ADの提供・連携で、コンシューマ会員認証の主役ではない。
> **覚えるべきキーワード**：Cognito＝アプリ会員 / IAM＝AWSリソース操作者
> **関連サービス**：API Gateway / IAM / Identity Center
> **SAAでの出題イメージ**：「アプリ会員の認証」=Cognito、「AWSの権限」=IAM。
> 📎 参照：[[0810_Cognito]] / [[0601_IAM基礎]]

## 第15問（STS一時認証 / 長期アクセスキー）
EC2やLambdaからAWSサービスへアクセスする際、認証情報を埋め込まず、自動でローテーションされる一時認証情報を使いたい。最適なのはどれか。

- A. IAMユーザーの長期アクセスキーを環境変数に置く
- B. IAMロール（STSの一時認証情報）を使う
- C. ルートユーザーのアクセスキーを使う
- D. アクセスキーをコードにハードコードする

> [!success]- 答え：**B**（IAMロール／STS）
> サービスへの権限付与はIAMロールが鉄則。STSが短期間有効な一時認証情報を自動発行・自動ローテーションし、キーの埋め込み・漏洩リスクをなくす。
> **他の選択肢が違う理由**
> - A：長期アクセスキーの埋め込みは漏洩・ローテーション漏れのリスクが高いアンチパターン。
> - C：ルートユーザーの使用・キー発行は厳禁。
> - D：ハードコードは最悪（コード共有/流出で即漏洩）。
> **覚えるべきキーワード**：IAMロール / STS / 一時認証情報 / 自動ローテーション
> **関連サービス**：インスタンスプロファイル / AssumeRole / IAMポリシー
> **SAAでの出題イメージ**：「サービスに権限・キーを埋め込まない」=IAMロール（STS）。
> 📎 参照：[[0602_IAMロール詳細]] / [[0601_IAM基礎]]

## 第16問（SCP / IAMポリシー）
組織内の全アカウントに対し、「許可の上限（ガードレール）」として特定操作を一律に禁止したい。個々のIAM設定に依存せず強制したい。最適なのはどれか。

- A. 各アカウントのIAMポリシー
- B. AWS Organizations の SCP（サービスコントロールポリシー）
- C. IAMアクセスキーのローテーション
- D. セキュリティグループ

> [!success]- 答え：**B**（SCP）
> SCPは組織配下アカウントの「権限の上限」を定める。IAMでいくら許可してもSCPで禁止された操作はできず、全アカウントへ予防的・強制的に効く。
> **他の選択肢が違う理由**
> - A：IAMポリシーはアカウント内の許可付与で、組織全体への一律強制・上限設定にはならない。
> - C：キーのローテーションは資格情報管理で、操作の一律禁止ではない。
> - D：セキュリティグループはネットワーク制御でAPI操作の禁止ではない。
> **覚えるべきキーワード**：SCP＝権限の上限・ガードレール / IAM＝許可の付与
> **関連サービス**：Organizations / Control Tower / IAM
> **SAAでの出題イメージ**：「全アカウントに一律の利用制限」=SCP。
> 📎 参照：[[1403_Organizations]] / [[0601_IAM基礎]]

## 第17問（Config / CloudTrail）
「あるセキュリティグループの設定がいつ・どう変更され、現在ルール違反していないか」を継続的に評価したい。最適なのはどれか。

- A. AWS CloudTrail
- B. AWS Config
- C. Amazon CloudWatch メトリクス
- D. VPCフローログ

> [!success]- 答え：**B**（Config）
> Configはリソース「構成（設定）」の変更履歴と現在状態を記録し、ルールで準拠を継続評価する。設定の時系列と違反判定はConfigの担当。
> **他の選択肢が違う理由**
> - A：CloudTrailは「誰がどのAPIを呼んだか」の操作証跡。構成の状態・準拠評価そのものは行わない（補完関係）。
> - C：CloudWatchは性能メトリクスで構成評価ではない。
> - D：VPCフローログは通信記録で設定変更履歴ではない。
> **覚えるべきキーワード**：Config＝構成の状態・履歴・準拠評価 / CloudTrail＝操作の証跡
> **関連サービス**：CloudTrail / Security Hub / Systems Manager（是正）
> **SAAでの出題イメージ**：「構成の変更履歴・ルール準拠」=Config、「誰が操作したか」=CloudTrail。
> 📎 参照：[[1003_AWS_Config]] / [[1002_CloudTrail]]

## 第18問（GuardDuty / Inspector）
EC2やコンテナイメージに既知の脆弱性（CVE）や危険な設定がないかをスキャンして評価したい。最適なのはどれか。

- A. Amazon GuardDuty
- B. Amazon Inspector
- C. AWS Config
- D. AWS CloudTrail

> [!success]- 答え：**B**（Inspector）
> Inspectorはワークロードの脆弱性管理。EC2・コンテナイメージ（ECR）・Lambdaを継続スキャンし、CVE等の既知脆弱性やネットワーク到達性のリスクを評価する。
> **他の選択肢が違う理由**
> - A：GuardDutyはログ分析による「挙動ベースの脅威検知」で、脆弱性スキャンとは目的が異なる。
> - C：Configは構成準拠チェックで脆弱性（CVE）診断ではない。
> - D：CloudTrailは操作証跡で脆弱性診断ではない。
> **覚えるべきキーワード**：Inspector＝脆弱性スキャン（CVE） / GuardDuty＝脅威検知（挙動）
> **関連サービス**：Security Hub / ECR / Systems Manager
> **SAAでの出題イメージ**：「脆弱性をスキャン」=Inspector、「不審な挙動を検知」=GuardDuty。
> 📎 参照：[[1603_脅威検知・監査]]

## 第19問（DataSync / Snowball）
オンプレの数十TBを、回線がそれなりにある環境からAWSへ「オンラインで」高速・自動的に転送し、以後も定期同期したい。最適なのはどれか。

- A. AWS DataSync
- B. AWS Snowball
- C. AWS Storage Gateway
- D. S3 へ手動アップロード

> [!success]- 答え：**A**（DataSync）
> DataSyncはオンプレ⇄AWS間のオンラインデータ転送を高速・自動化し、増分の定期同期もできる。回線があるなら移行・継続同期に適する。
> **他の選択肢が違う理由**
> - B：Snowballは物理デバイスでのオフライン転送。回線が細い・ペタ級・オフラインが要る場合向け。
> - C：Storage Gatewayはオンプレからクラウドストレージを「日常利用」するハイブリッド用途で、一括移行ツールではない。
> - D：手動アップロードは大量・継続同期に不向き。
> **覚えるべきキーワード**：DataSync＝オンライン高速転送・定期同期 / Snowball＝オフライン物理転送
> **関連サービス**：Storage Gateway / Snowball / S3 / EFS
> **SAAでの出題イメージ**：「回線ありでオンライン移行・同期」=DataSync、「回線細い・大容量」=Snowball。
> 📎 参照：[[1501_移行と転送サービス一覧]] / [[1502_DataSync vs Firehose vs Storage Gateway]]

## 第20問（EBS gp3 / io2）
ミッションクリティカルなDB用に、非常に高いIOPSと高い耐久性（99.999%）が一貫して求められるEC2用ブロックストレージが欲しい。最適なボリュームタイプはどれか。

- A. gp3（汎用SSD）
- B. io2 / io2 Block Express（プロビジョンドIOPS SSD）
- C. st1（スループット最適化HDD）
- D. sc1（コールドHDD）

> [!success]- 答え：**B**（io2 / io2 Block Express）
> io2系はプロビジョンドIOPS SSDで、非常に高いIOPSを明示指定でき、耐久性も高い（99.999%）。ミッションクリティカルDB向け。
> **他の選択肢が違う理由**
> - A：gp3はコスパの良い汎用SSDで多くの用途に十分だが、最高水準のIOPS・耐久が要る重DBではio2が適切。
> - C：st1はスループット重視のHDDで、高IOPSランダムアクセスには不向き（ログ/ビッグデータ向け）。
> - D：sc1は最安のコールドHDDでアクセス頻度の低いデータ向け。
> **覚えるべきキーワード**：io2＝高IOPS・高耐久（重DB） / gp3＝汎用コスパ / HDD系＝スループット/コールド
> **関連サービス**：EBSスナップショット / EC2 / インスタンスストア（揮発高速）
> **SAAでの出題イメージ**：「重要DBで最高IOPS・高耐久」=io2、「汎用でコスパ」=gp3。
> 📎 参照：[[0103_EC2のストレージ]]
