# 【問題】外部サイト模擬試験 5（複数分野・全10問）

## この問題集の目的

外部サイトの模擬問題を、答えの暗記ではなく本番形式（長文シナリオ）で解く練習として収録する。`504` が10問で満杯になったため、その続きを本ファイルに蓄積する。分野は固定せず、出題された問題を順次追加していく。

> [!info] 出題形式
> - シナリオ段落と設問文を分け、です・ます調で記述。
> - 選択肢はサービス名ではなく、具体的な操作・構成で記述。
> - 原則は単一選択（4択 A〜D）。複数選択の問題は5択（A〜E）とし、設問文に「2つ選択」等を明記。

> [!note] 運用ルール
> 外部サイト模擬試験は分野横断で1ファイルに蓄積する。**1ファイルが10問を超えたら `506_…` 以降に新規ファイルを作成**して分割する（解説側 `61_模擬問題解説/` も同じ番号で対応させる）。

## 第1問（Management / Monitoring ｜ マルチリージョンの確実なフェイルオーバー（Application Recovery Controller））
ある動画配信企業は、AWS上で会員向け配信プラットフォームを運用しており、2つのAWSリージョンにまたがるアクティブ/アクティブ構成で稼働しています。同社は、いずれかのリージョンで大規模障害が発生した場合でも、利用者のリクエストを正常なリージョンへ切り替えられるようにしたいと考えています。また、フェイルオーバー時には、切り替え先リージョンが確実にリクエストを処理できる状態である必要があります。運用管理の負荷をできるだけ抑えながら、これらの要件を満たす最も適切な選択肢はどれですか。

- A. Amazon Route 53のレイテンシーベースルーティングを設定し、各リージョンのエンドポイントへユーザーを最も低レイテンシーなリージョンにルーティングする。さらに、ヘルスチェックを関連付けて、障害が発生したリージョンへのルーティングを停止するように設定する。
- B. Amazon Route 53のヘルスチェックを利用して各リージョンのエンドポイントの正常性を監視し、障害発生時には利用可能なリージョンへトラフィックを切り替える。さらに、Amazon EventBridge Schedulerでプライマリリージョンのリソースに対する定期的なテストを実行し、フェイルオーバー先リージョンがリクエストを処理できる状態であることを確認する。
- C. AWS Fault Injection Serviceを使用してリージョン障害をシミュレートし、障害が検出された場合はAmazon Route 53のDNSレコードを更新して正常なリージョンへトラフィックを向けるように設定する。
- D. Amazon Application Recovery Controllerのマルチリージョンリカバリ機能を使用する。

> [!success]- 答え：**D**
> Amazon Application Recovery Controller（ARC）は、マルチリージョン/マルチAZのフェイルオーバーを「確実に・マネージドに」実行するためのサービス。**Readiness Check（準備状況チェック）**で切替先リージョンの容量・構成・クォータが「確実にリクエストを処理できる状態」かを継続的に検証し、**ルーティングコントロール**で安全に手動/自動フェイルオーバーを行える。「確実に処理できる状態の保証」＋「低運用」を同時に満たすのはDだけ。
> - A：レイテンシーベースルーティング＋ヘルスチェックは正常時の最適ルーティングと障害時の切り離しはできるが、「切替先が確実に処理できる状態か」を保証する仕組みがない。
> - B：ヘルスチェックは「エンドポイントが応答するか」しか見ず、容量やクォータまでは検証しない。EventBridge Schedulerで定期テストを自作するのは運用負荷が高く、ARCのReadiness Checkを自前で作り直しているだけ。
> - C：Fault Injection Serviceは障害を意図的に注入してレジリエンスを「テスト」するためのサービスで、本番フェイルオーバーを制御する仕組みではない。DNS更新の自作も運用負荷が高い。
> - D：正解。ARCのReadiness Checkで切替先の処理可能性を保証し、ルーティングコントロールでフェイルオーバーをマネージドに実行。低運用で要件を満たす。
> **関連ノート**：[[1406_Application_Recovery_Controller]] / [[1407_Fault_Injection_Service]] / [[0404_Route53]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第1問（Management / Monitoring ｜ マルチリージョンの確実なフェイルオーバー（Application Recovery Controller））]]

## 第2問（Security / IAM ｜ WAFの保護対象（NLBは非対応）とL7攻撃対策）
あるゲーム開発会社が、AWS 上でリアルタイムのマルチプレイヤーオンラインゲームを運営しています。ゲームアプリケーションは、Network Load Balancer（NLB）配下にある Auto Scaling グループ内の複数の Amazon EC2 インスタンス上で稼働しており、低レイテンシーの TCP/UDP 通信を使用しています。ユーザーは、Amazon API Gateway によって公開された API 経由で EC2 インスタンスにアクセスします。会社は、脆弱性を突く攻撃（クロスサイトスクリプティングや SQL インジェクションなど）からゲームアプリケーションを保護したいと考えています。

次のうち、もっともセキュリティ効果が高い方法はどれですか。

- A. AWS WAF を使用して、Amazon API Gateway を保護する。
- B. AWS Shield Advanced を使用して、NLB を保護する。
- C. AWS Shield Advanced を使用して、Amazon API Gateway を保護する。
- D. AWS WAF を使用して、NLB を保護する。

> [!success]- 答え：**A**
> XSS/SQLインジェクションはL7（Webアプリ層）の攻撃なので、対策はWAF（ShieldはDDoS対策で対象外）。さらにWAFはL7リソースにしか関連付けできず、ユーザーの入口はAPI Gatewayなので、WAFをAPI Gatewayに適用するのが最適。
> - A：正解。L7攻撃(XSS/SQLi)はWAFで防ぎ、WAFを関連付け可能かつ実際の入口であるAPI Gatewayに適用する。
> - B：ShieldはDDoS（量による攻撃）緩和でXSS/SQLiには対応しない。対象もNLB。
> - C：ShieldはDDoS対策でXSS/SQLiには効かない（対象がAPI GatewayでもNG）。
> - D：WAFはL7でHTTPの中身を見て判定するため、L4（TCP/UDP）のNLBには関連付けできない。設定自体が成立しない。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[0503_ALB・NLB・GWLBの違い]] / [[0809_API_Gateway]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第2問（Security / IAM ｜ WAFの保護対象（NLBは非対応）とL7攻撃対策）]]

## 第3問（Networking / CDN / LB ｜ Global AcceleratorとNLB（UDP・静的IP・エッジ低遅延））
金融商品を取り扱う企業は、現在の株価をリアルタイムで配信するアプリケーションを構築しています。このアプリケーションは OS カーネルがカスタマイズされた Linux 上で動作し、UDP 通信を使用しています。企業は世界中のユーザーにサービスを提供する予定であり、可能な限り低遅延で通信するためにユーザーから最も近い AWS エッジロケーションにリクエストをルーティングさせたいと考えています。さらに、ユーザーからアプリケーションへは固定 IP アドレスを使用してアクセスできる必要があります。

次のうち、要件に合うソリューションはどれですか。

- A. フロントエンドに Global Accelerator を配置し、リクエストを NLB へ転送する。NLB 配下の Auto Scaling グループに所属した複数の EC2 インスタンスでリクエストを処理する。
- B. フロントエンドに Global Accelerator を配置し、リクエストを ALB へ転送する。ALB 配下の Auto Scaling グループに所属した複数の EC2 インスタンスでリクエストを処理する。
- C. フロントエンドに CloudFront を配置し、リクエストを ALB へ転送する。ALB 配下の Auto Scaling グループに所属した複数の EC2 インスタンスでリクエストを処理する。
- D. フロントエンドに CloudFront を配置し、リクエストを NLB へ転送する。NLB 配下の Auto Scaling グループに所属した Lambda 関数でリクエストを処理する。

> [!success]- 答え：**A**
> UDP通信なのでロードバランサーはNLB（L4・TCP/UDP対応。ALBはHTTP専用でUDP不可）。世界中で最寄りエッジから低遅延＋固定IPが要るのでフロントはAWS Global Accelerator（TCP/UDP対応・静的Anycast IPを2つ提供）。CloudFrontはHTTPコンテンツのキャッシュ配信用で、UDPも固定IPも要件に合わない。
> - A：正解。Global Accelerator（エッジ＋静的IP）→NLB（UDP対応）→EC2。UDP・固定IP・低遅延をすべて満たす。
> - B：Global Acceleratorは正しいが、ALBはHTTP専用でUDP非対応。UDP要件で落ちる。
> - C：CloudFront（HTTP用でUDP不可・固定IPなし）＋ALB（UDP不可）で二重に不適。
> - D：CloudFrontはUDP非対応。バックエンドがLambdaでリアルタイムUDP配信にも合わない。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0503_ALB・NLB・GWLBの違い]] / [[0502_Elastic Load Balancing]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第3問（Networking / CDN / LB ｜ Global AcceleratorとNLB（UDP・静的IP・エッジ低遅延））]]

## 第4問（Storage ｜ HPC向け高性能ファイル（FSx for Lustre×S3統合））
ある気候研究機関が気候変動に関連する大量の気象データを集め、S3 に保存しています。この研究機関は、気候変動の影響をシミュレートするために高性能コンピューティング（HPC）環境を導入しました。HPC 環境から S3 上のデータへ低レイテンシーかつ高スループットでアクセスする必要がありますが、この要件を最も効果的に満たす選択肢はどれですか。

- A. FSx for Lustre ファイルシステムを設定し、S3 上のデータを手動でインポートする。HPC 環境からファイルシステムにアクセスする。
- B. S3 Transfer Acceleration を活用し、S3 データへのアクセス速度を向上させ、HPC 環境からのアクセスを高速化する。
- C. FSx for Lustre ファイルシステムを設定し、既存の S3 バケットと統合して、HPC 環境から直接ファイルシステムにアクセスする。
- D. CloudFront を利用して、S3 に保存されたデータに対するアクセスをキャッシュし、HPC 環境からのアクセスを高速化する。

> [!success]- 答え：**C**
> HPCの低レイテンシ・高スループットなファイルアクセスは、HPC/ML向けに設計された高性能並列ファイルシステム Amazon FSx for Lustre が最適。FSx for LustreはS3とネイティブ統合でき、既存バケットにリンクすればS3オブジェクトをファイルとして直接・高速に扱える（手動コピー不要）。
> - A：FSx for Lustreは正しいが、S3データを手動でインポートする運用が余計。ネイティブ統合のCに劣る。
> - B：S3 Transfer Accelerationは遠距離のS3アップロード/ダウンロードを高速化する機能で、HPCの低遅延ファイルアクセスとは別物。
> - C：正解。FSx for LustreをS3にネイティブ統合し、HPCから直接高速アクセス。低遅延・高スループット・手間なし。
> - D：CloudFrontはHTTPコンテンツのキャッシュ配信で、HPCのファイルシステムアクセス高速化には使えない。
> **関連ノート**：[[0202_S3とEBS・EFSの違い]] / [[0103_EC2のストレージ]] / [[0201_S3]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第4問（Storage ｜ HPC向け高性能ファイル（FSx for Lustre×S3統合））]]

## 第5問（Analytics ｜ ネスト文書のDynamoDB保存とMacieによるPII定期スキャン）
ある企業は、自社のカスタマーサポートセンター向けに問い合わせチケットを管理するシステムを構築しています。各チケットはスレッド形式のメッセージ一覧、添付ファイルのメタデータ、顧客プロフィールなどをネストしたドキュメント構造で保存し、トラフィック増大時でも低レイテンシで応答できることが求められています。また、顧客がメッセージ内に誤って住所や電話番号などの個人識別情報（PII）を入力してしまうリスクがあるため、毎月全データをスキャンし、PIIが検出された場合にはコンプライアンス部門へメールで通知したいと考えています。

これらの要件を満たすために、ソリューションアーキテクトが取るべき方法の組みあわせは次のうちどれですか。（2つ選択）

- A. Macieを有効化して、月次の分類ジョブ完了イベントをEventBridgeで捕捉してLambdaに送信する。
- B. S3にJSONファイルでチケットを保存し、Athenaでクエリを実行する。
- C. Macieを有効化して、月次スケジュールの分類ジョブを設定してスキャンを実行する。ジョブ完了イベントをEventBridgeで捕捉してSNSトピックに転送する。
- D. DynamoDBにチケットをネスト構造で保存し、エクスポート機能で毎月S3に出力する。
- E. DynamoDB Streamsを有効化して書き込みイベントをLambdaに送り、各レコードでComprehendを実行してPIIを検出し、SNSで通知する。

> [!success]- 答え：**C ＋ D**
> 要件は2つ。①「ネスト構造・低レイテンシ・大規模」＝DynamoDB。②「毎月全データをスキャンしてPII検出→メール通知」＝Macie（PII検出はS3が対象）。DynamoDBにネスト保存しつつ毎月S3へエクスポート（D）し、そのS3をMacieの月次分類ジョブでスキャン、完了イベントをEventBridge→SNSでメール通知（C）する組み合わせが最適。
> - A：分類ジョブの「スケジュール設定（月次スキャンの実行）」がなく、通知先もLambda（メール送信は追加実装が必要）。Cの劣化版で不十分。
> - B：S3＋JSON＋Athenaは低レイテンシなチケット応答（ネスト文書の高速読み書き）に向かず、要件①を満たさない。
> - C：正解。Macieの月次分類ジョブでS3をスキャンし、EventBridge→SNSでコンプラ部門へメール通知できる。
> - D：正解。DynamoDBはネスト（JSON）文書を低レイテンシ・大規模に扱え、S3エクスポートでMacieのスキャン対象を用意できる。
> - E：DynamoDB Streams＋Comprehendは「書き込みごとのリアルタイム検出」で、「毎月全データを一括スキャン」という要件と方式が違う。自前実装で運用負荷も高い。
> **関連ノート**：[[0303_DynamoDB]] / [[1603_脅威検知・監査]] / [[1601_セキュリティサービス一覧]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第5問（Analytics ｜ ネスト文書のDynamoDB保存とMacieによるPII定期スキャン）]]

## 第6問（Migration / Transfer ｜ Application Migration Service（MGN）によるVM移行（3つ選択））
あるオンライン小売企業は、オンプレミスのデータセンター内で複数の仮想マシン（VM）を使用して販売管理システムを運用しています。最近、データセンターの賃料高騰や設備維持費用の増加が課題となっており、コスト最適化のために AWS への移行を検討しています。同社はまず、ビジネスへの影響が比較的少ないバッチ処理系のワークロードを対象として AWS への移行を開始することにしました。移行時のコードやシステム構成への変更を最小限に留め、可能な限り短期間で移行作業を完了させたいと考えています。また、移行作業に伴うダウンタイムも最小限に抑える必要があります。これらの要件を満たすために適切な移行手順の組み合わせはどれですか。（3つ選択）

- A. VM の初期レプリケーション完了後、テストインスタンスを起動して、移行後のワークロードの動作確認を行う。
- B. DataSync で VM のディスクイメージを S3 にコピーし、コピー後に AMI を作成して EC2 を起動する。
- C. Systems Manager Patch Manager を有効化し、移行対象の VM に関するデータを収集する。
- D. Application Migration Service を利用するために、移行対象の VM に AWS Replication Agent をインストールする。
- E. Elastic Disaster Recovery を利用するために、移行対象の VM に AWS Replication Agent をインストールする。
- F. カットオーバー時にオンプレミスの VM への書き込みを停止し、カットオーバーインスタンスを起動して最終データ同期を実施する。

> [!success]- 答え：**A ＋ D ＋ F**
> 「変更最小・短期間・低ダウンタイムでVMをリフト＆シフト」はAWS Application Migration Service（MGN）の定番。標準手順は①Replication Agentを導入して継続的ブロックレベル複製を開始（D）、②初期レプリケーション後にテストインスタンスで検証（A）、③カットオーバーで書き込み停止→最終同期→カットオーバーインスタンス起動（F）。継続複製＋一瞬の切替でダウンタイムを最小化する。
> - A：正解。MGNのテスト工程。本番切替前にテストインスタンスで動作確認する。
> - B：DataSyncはファイル/オブジェクトのデータ転送サービスで、VM全体の移行手段ではない。手作業が多く継続複製もなくダウンタイムも大きい。
> - C：Systems Manager Patch Managerはパッチ適用の道具で、移行の発見・収集用ではない（発見はApplication Discovery Service）。無関係。
> - D：正解。MGNを使う起点。VMにAWS Replication Agentを入れて継続的レプリケーションを開始する。
> - E：Elastic Disaster Recovery（DRS）は災害復旧サービス。エージェントは似ているが、今回は「移行」なので使うのはMGN。用途違いの罠。
> - F：正解。MGNのカットオーバー工程。書き込み停止→最終同期→カットオーバーインスタンス起動で本番切替する。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[1502_DataSync]] / [[0202_S3とEBS・EFSの違い]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第6問（Migration / Transfer ｜ Application Migration Service（MGN）によるVM移行（3つ選択））]]

## 第7問（Application Integration ｜ API Gatewayのキャッシュで重複参照を削減）
ある金融サービス企業では、大量のトランザクションデータを処理するETLジョブを運用しています。このETLジョブは、顧客プロファイルや取引条件などの参照データを取得するために、オンプレミスでホストされている参照データサービスにアクセスしています。市場が活発な時間帯には、同じ参照データに対する多数のリクエストが集中し、データサービスが過負荷になる問題が発生しています。企業は、ETLジョブの効率を保ちつつ、参照データサービスへの負荷を軽減し、管理コストを最小限に抑える解決策を探しています。

次のソリューションの中から、最も適切なのはどれですか。

- A. API Gateway REST APIを作成し、使用量プランでスロットルとクォータを設定する。VPCリンクを使用してオンプレミスサービスに接続し、参照データを取得する。
- B. API Gateway WebSocket APIを作成し、ETLジョブがオンプレミス参照データサービスと持続的接続を維持することで、接続確立のオーバーヘッドを削減し参照データサービスへの負荷を軽減する。
- C. API Gateway HTTP APIを作成し、Lambda関数でオンプレミス参照データを取得する。突発的な大量リクエストに対する自動スケールアウト機能を活用して負荷を分散する。
- D. API Gateway REST APIを作成し、顧客IDや商品コードをキャッシュキーとして設定して、API Gatewayのキャッシュ機能を有効にする。

> [!success]- 答え：**D**
> 問題の本質は「同じ参照データへの重複リクエストがバックエンドに殺到している」こと。API GatewayのキャッシュはREST APIで使え、顧客IDや商品コードをキャッシュキーにすると、同じキーの2回目以降はキャッシュから即応答し、オンプレの参照データサービスまでリクエストが到達しない。重複取得そのものを減らすので負荷が根本的に下がり、マネージド機能なので管理コストも小さい。
> - A：使用量プランのスロットル/クォータは「リクエスト数を上限で絞る」機能で、絞られたETL側の効率が落ちるだけ。同じデータの重複取得は減らず、バックエンド負荷の根本解決にならない。
> - B：WebSocketは双方向のリアルタイム通信用。参照データの取得はリクエスト/レスポンス型で、持続的接続にしても重複リクエスト自体は減らない。用途違い。
> - C：HTTP API＋Lambdaは自動スケールするが、スケールすればするほどオンプレのデータサービスへのアクセスが増え、過負荷をむしろ悪化させる。
> - D：正解。キャッシュで重複リクエストをAPI Gatewayが肩代わりし、バックエンド到達数を削減。効率維持・負荷軽減・低管理コストを満たす。
> **関連ノート**：[[0809_API_Gateway]] / [[0304_ElastiCache]] / [[0405_ネットワークとコンテンツ配信]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第7問（Application Integration ｜ API Gatewayのキャッシュで重複参照を削減）]]

## 第8問（Networking / CDN / LB ｜ オンプレからS3への閉域アクセス（Direct Connect＋インターフェイスエンドポイント））
ある企業は S3 へのデータ保存を行っており、このデータに対して、自社の AWS リージョン内のシステムとオンプレミス環境の双方から安全にアクセスする必要があります。また、企業ポリシー上、すべての通信はインターネットを経由せず、閉域ネットワーク経由で行う必要があります。この要件を満たす最も適切なソリューションはどれですか。

- A. Site-to-Site VPN の IPsec 暗号化通信を利用し、仮想プライベートゲートウェイ経由で S3 にアクセスする。
- B. Transit Gateway と Direct Connect ゲートウェイを設定し、各 VPC でゲートウェイ型エンドポイント経由で S3 にアクセスする。
- C. Direct Connect 接続を確立し、S3 に対してゲートウェイ型エンドポイントを作成してオンプレミスからのアクセスを許可する。
- D. Direct Connect 接続を確立し、S3 に対してインターフェイス型エンドポイントを構成し、プライベート IP によるアクセスを実現する。

> [!success]- 答え：**D**
> 閉域接続はDirect Connect（Site-to-Site VPNはインターネット経由なので不可）。オンプレからS3をプライベートに叩くには、VPC内にプライベートIP（ENI）を持つインターフェイス型エンドポイント（PrivateLink）が必要。ゲートウェイ型エンドポイントはVPC内からのみ有効でオンプレからは到達できない。
> - A：Site-to-Site VPNはインターネット経由（暗号化しても通り道はインターネット）で、「インターネット非経由」ポリシーに反する。
> - B：閉域接続は良いが、S3をゲートウェイ型エンドポイントで受けているため、オンプレからはそのエンドポイントに到達できずアクセスが成立しない。
> - C：ゲートウェイ型エンドポイントはVPC内専用で、オンプレミスからのアクセスを許可できない。
> - D：正解。Direct Connect（閉域）＋インターフェイス型エンドポイント（プライベートIP）で、VPCからもオンプレからもインターネットを通らずS3へアクセスできる。
> **関連ノート**：[[0409_IGW・NAT Gateway・VPCエンドポイント]] / [[0405_ネットワークとコンテンツ配信]] / [[0406_VPC応用機能]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第8問（Networking / CDN / LB ｜ オンプレからS3への閉域アクセス（Direct Connect＋インターフェイスエンドポイント））]]

## 第9問（Security / IAM ｜ マルチアカウント統制とAD連携シングルサインオン）
ある多国籍企業は、グローバルなITガバナンスを強化するため、各地域の事業部ごとに個別のAWSアカウントを使用したマルチアカウント構成に移行することを計画しています。ユーザー認証は、本社で管理しているオンプレミスのディレクトリサービス（Microsoft Active Directory）を用いて、全AWSアカウントを一元的に管理したいと考えています。

この要件を満たすために有効なアクションの組み合わせはどれですか。（2つ選択）

- A. AWS IAM Identity Centerを設定し、AWS Directory Serviceを介して本社のActive Directoryと連携する。
- B. 各AWSアカウントごとに個別のIAMユーザーを作成し、本社のActive Directoryから一時的な認証情報を配布する。
- C. AWS Organizationsですべての機能を有効にした組織を作成し、各事業部用のAWSアカウントを追加する。
- D. 本社のActive DirectoryにAmazon Cognitoを接続し、Cognitoを用いてすべてのAWSアカウントのユーザー認証を行う仕組みを構成する。
- E. AWS Organizationsで新しい組織を作成し、認証メカニズムにAWS Directory Serviceを直接使用するように組織を設定する。

> [!success]- 答え：**A ＋ C**
> 要件は2つ。①「マルチアカウントを一元管理する土台」＝AWS Organizations（全機能有効化＋事業部ごとにアカウント追加）。②「オンプレADで全アカウントの認証を一元化」＝IAM Identity Center（旧AWS SSO）＋AWS Directory ServiceでオンプレADと連携。AがADシングルサインオン、Cがマルチアカウントの器で、2つ揃って要件を満たす。
> - A：正解。IAM Identity CenterはOrganizations配下の全アカウントへのSSOを提供し、Directory Service経由でオンプレADのユーザーをそのまま利用できる。
> - B：アカウントごとにIAMユーザーを個別作成＝一元管理と真逆で、管理が煩雑。ADとの一元認証にもならない。
> - C：正解。Organizationsの全機能を有効化した組織を作り、事業部ごとにアカウントを追加＝マルチアカウント統制の土台。
> - D：Cognitoはアプリ（Webモバイル）のエンドユーザー認証用で、AWSアカウント/コンソールへの従業員SSOの仕組みではない。用途違い。
> - E：Organizations自体に「認証メカニズムにDirectory Serviceを直接使う」設定は存在しない。AD連携SSOはIAM Identity Centerの役割で、この選択肢は成立しない。
> **関連ノート**：[[1602_アイデンティティ系]] / [[1403_Organizations]] / [[0601_IAM基礎]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第9問（Security / IAM ｜ マルチアカウント統制とAD連携シングルサインオン）]]

## 第10問（Security / IAM ｜ セキュリティグループのSG参照（ALB経由のみ許可））
あるサービスは ALB 配下の EC2 インスタンスで Web コンテンツを公開しています。インターネットから EC2 インスタンスへの直接アクセスを防ぎ、ALB 経由のアクセスのみ許可したいと考えています。

どのようなネットワークアクセス制御の設定をしたらよいですか。

- A. EC2 インスタンスのセキュリティグループで、ALB のプライベート IP アドレスからの通信を許可する。
- B. EC2 インスタンスのセキュリティグループで、インターネットからの通信を拒否する。
- C. ALB のセキュリティグループで、EC2 インスタンスのセキュリティグループへの通信を許可する。
- D. EC2 インスタンスのセキュリティグループで、ALB のセキュリティグループからの通信を許可する。

> [!success]- 答え：**D**
> セキュリティグループは許可（allow）ルールのみで拒否ルールは書けない。EC2側SGのインバウンドで送信元にALBのセキュリティグループを指定して許可すれば、ALB経由の通信だけが通り、インターネットからの直接アクセスは自動的に遮断される（SG参照/チェイン）。ALBのIPが変動しても追従できるベストプラクティス。
> - A：ALBのプライベートIPを許可しても、ALBのIPは動的に変わる/増えるため壊れやすい。IP直指定は非推奨で、SG参照のDが正道。
> - B：セキュリティグループには拒否ルールが書けない（許可のみ）。「インターネットからの通信を拒否する」という設定自体が成立しない。拒否ルールはNACLの機能。
> - C：制御をかける場所が逆。「EC2に誰を入れるか」はEC2側SGのインバウンドで決める話で、ALB側SGをいじってもEC2への直アクセス制限にはならない。
> - D：正解。EC2のSGで送信元＝ALBのSGを許可。ALB経由のみ通り直アクセスは自動遮断、IP変動にも強い。
> **関連ノート**：[[0408_セキュリティグループとNACL]] / [[0502_Elastic Load Balancing]] / [[1601_セキュリティサービス一覧]]
> **詳しい解説**：[[505_【解説】外部サイト模擬試験5#第10問（Security / IAM ｜ セキュリティグループのSG参照（ALB経由のみ許可））]]
