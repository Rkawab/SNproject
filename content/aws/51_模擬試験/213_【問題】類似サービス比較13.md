# 【問題】類似サービス比較13（ストレージ・データベース・分析）

## この問題集の目的

ストレージ、データベース、ログ分析で混同しやすい機能を、データ形式・アクセス方法・処理目的から選び分ける。

## 対象レベル

SAA基礎〜標準。サービス境界と構成上の違いを確認する。

## 問題数

10問

## 第1問（Compute ｜ EBSスナップショット と AMI）
同じOS、ミドルウェア、起動設定を持つEC2インスタンスを複数リージョンで繰り返し起動したいと考えています。
最も適した方法はどれですか。

- A. EC2インスタンスからAMIを作成し、必要なリージョンへコピーする
- B. データ用EBSボリュームのスナップショットだけをコピーする
- C. 稼働中インスタンスのメモリ内容をS3へ保存する
- D. AWS BackupでRDSのバックアッププランを作成する

> [!success]- 答え：**A**
> EBSスナップショットはボリュームのバックアップ、AMIはEC2を起動するための設計図です。
> - A：正解。ルートボリュームのスナップショットに加え、起動に必要な情報をまとめて再利用できる。
> - B：ボリュームデータは復元できるが、それだけではEC2起動テンプレートにならない。
> - C：通常のAMIやスナップショットは実行中メモリを保存する仕組みではない。
> - D：EC2起動イメージの複製という要件に合わない。
> **着眼点（SAA）**：ディスク復元はスナップショット、同じEC2を起動するならAMI。
> **関連ノート**：[[0104_スナップショット・AMIとは]] / [[0202_S3とEBS・EFSの違い]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第1問（Compute ｜ EBSスナップショット と AMI）]]

## 第2問（Storage ｜ EFS と FSx for Lustre）
数百台のLinux計算ノードが同じデータセットへ並列アクセスし、HPC処理で最大限のスループットを得る必要があります。処理元データはS3にあります。
最も適したストレージはどれですか。

- A. Amazon FSx for Lustre
- B. Amazon EFS General Purpose
- C. Amazon S3 Glacier Flexible Retrieval
- D. Amazon FSx for Windows File Server

> [!success]- 答え：**A**
> EFSは汎用共有NFS、FSx for LustreはHPC向けの並列ファイルシステムです。
> - A：正解。S3データと連携し、高い並列スループットを必要とする計算処理に適する。
> - B：Webサーバーの共有コンテンツなど汎用Linuxファイル共有に適する。
> - C：アーカイブ用で、計算ノードからの低遅延ファイルアクセスには使わない。
> - D：WindowsのSMBワークロード向け。
> **着眼点（SAA）**：汎用NFSはEFS、HPC・機械学習・LustreはFSx for Lustre。
> **関連ノート**：[[0213_FSxファミリー4種の使い分け]] / [[0214_EFS詳細]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第2問（Storage ｜ EFS と FSx for Lustre）]]

## 第3問（Storage ｜ FSx for Windows と FSx for ONTAP）
Windowsアプリケーションが、Active Directory統合されたSMB共有とWindows ACLを必要としています。NetApp固有機能やNFS/iSCSIは不要です。
最も適したストレージはどれですか。

- A. Amazon FSx for Windows File Server
- B. Amazon FSx for NetApp ONTAP
- C. Amazon EFS
- D. Amazon FSx for Lustre

> [!success]- 答え：**A**
> FSx for Windows File ServerはWindowsネイティブのSMB共有、FSx for ONTAPはNetApp互換とマルチプロトコルが強みです。
> - A：正解。Microsoft AD、SMB、Windows ACLなどのWindows機能に適する。
> - B：ONTAPのデータ管理機能やNFS/SMB/iSCSIを必要とする場合に選ぶ。
> - C：Linux向けNFS共有が中心。
> - D：HPC向け並列ファイルシステム。
> **着眼点（SAA）**：純粋なWindows共有はFSx for Windows、NetApp移行はFSx for ONTAP。
> **関連ノート**：[[0213_FSxファミリー4種の使い分け]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第3問（Storage ｜ FSx for Windows と FSx for ONTAP）]]

## 第4問（Storage ｜ S3 Inventory と Storage Lens）
特定のS3バケットに保存された全オブジェクトについて、キー、サイズ、ストレージクラス、暗号化状態、レプリケーション状態をCSVまたはParquetの一覧として毎日出力したいと考えています。
最も適した機能はどれですか。

- A. S3 Inventory
- B. S3 Storage Lens
- C. S3 Batch Operations
- D. AWS Config

> [!success]- 答え：**A**
> S3 Inventoryはオブジェクト明細、Storage Lensは集約メトリクスとダッシュボードを提供します。
> - A：正解。オブジェクト単位のメタデータ一覧を定期レポートとして出力する。
> - B：組織やバケット全体の使用傾向・保護状況を集約して可視化する。
> - C：一覧を入力にして既存オブジェクトへ一括操作を実行する。
> - D：対応AWSリソースの構成履歴を記録するが、S3オブジェクト一覧の代替ではない。
> **着眼点（SAA）**：オブジェクトごとの一覧はInventory、全体傾向の可視化はStorage Lens。
> **関連ノート**：[[0201_S3]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第4問（Storage ｜ S3 Inventory と Storage Lens）]]

## 第5問（Database ｜ RDS Proxy と ElastiCache）
Lambda関数からRDSへ同時に多数の接続が発生し、データベース接続数の上限に達しています。SQL結果のキャッシュではなく、接続自体を再利用したいと考えています。
最も適したサービスはどれですか。

- A. Amazon RDS Proxy
- B. Amazon ElastiCache
- C. Amazon DynamoDB Accelerator（DAX）
- D. Amazon API Gatewayのキャッシュ

> [!success]- 答え：**A**
> RDS Proxyは接続をプールし、ElastiCacheはデータをキャッシュします。
> - A：正解。短命なLambda実行からの多数のDB接続を集約・再利用する。
> - B：頻繁に読むデータをメモリへ置いてクエリ負荷を減らす場合に使う。
> - C：DynamoDB専用のインメモリキャッシュ。
> - D：APIレスポンスの重複を手前で返す機能。
> **着眼点（SAA）**：接続数問題はRDS Proxy、読み取りデータの再利用はElastiCache。
> **関連ノート**：[[0302_RDSProxy]] / [[0304_ElastiCache]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第5問（Database ｜ RDS Proxy と ElastiCache）]]

## 第6問（Database ｜ DAX と ElastiCache）
DynamoDBを利用するアプリケーションで、コード変更を最小限に抑えながら、結果整合性でよい読み取りをマイクロ秒単位に高速化したいと考えています。
最も適したサービスはどれですか。

- A. Amazon DynamoDB Accelerator（DAX）
- B. Amazon ElastiCache for Redis
- C. Amazon RDS Proxy
- D. DynamoDB Global Tables

> [!success]- 答え：**A**
> DAXはDynamoDB互換APIを持つ専用キャッシュ、ElastiCacheは汎用インメモリデータストアです。
> - A：正解。DynamoDB向けの読み取りキャッシュとしてアプリ変更を抑えやすい。
> - B：キャッシュ戦略やキー管理をアプリケーション側で設計する汎用選択肢。
> - C：RDS/Auroraのデータベース接続をプールする。
> - D：複数リージョンでDynamoDBテーブルを複製する。
> **着眼点（SAA）**：「DynamoDB専用」「マイクロ秒」「互換API」ならDAX。
> **関連ノート**：[[0303_DynamoDB]] / [[0304_ElastiCache]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第6問（Database ｜ DAX と ElastiCache）]]

## 第7問（Database ｜ Neptune と DocumentDB）
不正取引の調査で、口座、端末、IPアドレス、取引の関係を複数段たどり、共通する関連ノードを高速に発見したいと考えています。
最も適したデータベースはどれですか。

- A. Amazon Neptune
- B. Amazon DocumentDB（MongoDB互換）
- C. Amazon Timestream
- D. Amazon Keyspaces（Apache Cassandra向け）

> [!success]- 答え：**A**
> Neptuneは関係性をたどるグラフDB、DocumentDBはJSONドキュメントを格納・検索するDBです。
> - A：正解。ノードとエッジを使う多段の関係探索に適する。
> - B：柔軟なドキュメント構造には適するが、グラフ走査が中心の要件ではない。
> - C：時間順に蓄積される測定値の分析向け。
> - D：大規模なワイドカラムデータ向け。
> **着眼点（SAA）**：つながりの探索はNeptune、MongoDB互換の文書はDocumentDB。
> **関連ノート**：[[0305_Neptune]] / [[0308_DocumentDB]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第7問（Database ｜ Neptune と DocumentDB）]]

## 第8問（Database ｜ Aurora Global Database と DynamoDB Global Tables）
複雑なSQL、トランザクション、外部キーを使う既存のMySQL互換アプリケーションを、複数リージョンへ展開します。主リージョン障害時のリージョン切り替えと、他リージョンでの低遅延読み取りが必要です。
最も適した構成はどれですか。

- A. Amazon Aurora Global Database
- B. Amazon DynamoDB Global Tables
- C. RDSリードレプリカを同一AZに作成する
- D. Amazon ElastiCache Global Datastoreだけを使用する

> [!success]- 答え：**A**
> Aurora Global DatabaseはリレーショナルSQL、DynamoDB Global Tablesはマルチリージョンのキー・バリュー型NoSQLに使います。
> - A：正解。MySQL/PostgreSQL互換の関係モデルを維持し、リージョン間レプリケーションとDRに対応する。
> - B：複数リージョンでアクティブ・アクティブに読み書きできるNoSQLだが、既存のSQLや外部キーをそのまま使えない。
> - C：同一AZではリージョン障害対策にならない。
> - D：キャッシュであり、永続的なリレーショナルDBの代替ではない。
> **着眼点（SAA）**：関係モデルならAurora Global Database、キー・バリューの各リージョン書き込みならGlobal Tables。
> **関連ノート**：[[0307_Aurora詳細]] / [[0303_DynamoDB]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第8問（Database ｜ Aurora Global Database と DynamoDB Global Tables）]]

## 第9問（Storage ｜ S3 Batch Operations と イベント通知）
バケットに既に存在する5億個のオブジェクトへ、新しいタグを一括付与したいと考えています。今後追加されるオブジェクトだけでなく、既存データ全体が対象です。
最も適した方法はどれですか。

- A. S3 Batch Operationsで一括ジョブを実行する
- B. S3イベント通知からLambdaを呼び出す
- C. S3 Transfer Accelerationを有効化する
- D. S3のマルチパートアップロードを使用する

> [!success]- 答え：**A**
> Batch Operationsは既存オブジェクト群への管理処理、イベント通知は新しく発生するイベントへの反応に使います。
> - A：正解。マニフェストを基に既存オブジェクトへ大規模な反復操作を実行できる。
> - B：新規PUTなどには向くが、過去の全オブジェクトにイベントを遡及発生させない。
> - C：遠距離転送の高速化機能。
> - D：大きな単一オブジェクトのアップロードを分割する機能。
> **着眼点（SAA）**：既存データへの一括処理はBatch Operations、新規イベントへの反応はイベント通知。
> **関連ノート**：[[0201_S3]] / [[0209_S3イベント通知]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第9問（Storage ｜ S3 Batch Operations と イベント通知）]]

## 第10問（Analytics ｜ Logs Insights と OpenSearch Service）
CloudWatch Logsに集約済みのLambdaログについて、障害時に数時間分を一時的に絞り込み、集計クエリを実行したいと考えています。専用検索クラスターや常設ダッシュボードは不要です。
最も適した方法はどれですか。

- A. CloudWatch Logs Insightsで対話的にクエリする
- B. Amazon OpenSearch Serviceへ常時取り込む
- C. Amazon AthenaでRDSへ直接クエリする
- D. AWS X-Rayだけでログ全文を検索する

> [!success]- 答え：**A**
> Logs InsightsはCloudWatch Logsのその場の調査、OpenSearch Serviceは継続的な全文検索・可視化基盤に向きます。
> - A：正解。保存済みログへオンデマンドで検索・集計でき、追加基盤が不要。
> - B：大規模な全文検索や常設ダッシュボードには適するが、今回の要件には運用が過剰。
> - C：Athenaは主にS3上のデータへSQLクエリを実行する。
> - D：分散トレースには使えるが、CloudWatch Logsの全文検索を代替しない。
> **着眼点（SAA）**：CloudWatch Logsをその場で調査するならLogs Insights、検索基盤を常設するならOpenSearch。
> **関連ノート**：[[1110_Amazon OpenSearch Service]] / [[1001_CloudWatch]]
> **詳しい解説**：[[213_【解説】類似サービス比較13#第10問（Analytics ｜ Logs Insights と OpenSearch Service）]]
