# 【問題】外部サイト模擬試験 3（複数分野・全9問）

## この問題集の目的

外部サイトの模擬問題を、答えの暗記ではなく本番形式（長文シナリオ）で解く練習として収録する。`502` が10問で満杯になったため、その続きを本ファイルに蓄積する。分野は固定せず、出題された問題を順次追加していく。

> [!info] 出題形式
> - シナリオ段落と設問文を分け、です・ます調で記述。
> - 選択肢はサービス名ではなく、具体的な操作・構成で記述。
> - 原則は単一選択（4択 A〜D）。複数選択の問題は5択（A〜E）とし、設問文に「2つ選択」等を明記。

> [!note] 運用ルール
> 外部サイト模擬試験は分野横断で1ファイルに蓄積する。**1ファイルが10問を超えたら `504_…` 以降に新規ファイルを作成**して分割する（解説側 `61_模擬問題解説/` も同じ番号で対応させる）。

## 第1問（Containers ｜ Kubernetes運用のマネージド化（EKS Auto Mode））
ある企業が、アプリケーションを AWS 上で大規模にデプロイしようとしています。このアプリケーションはコンテナ化されており、企業はオープンソースソフトウェアである Kubernetes を積極的に活用したいと考えています。インフラ管理の負担を最小限に抑え、開発チームがアプリケーション開発に集中できる環境を構築する必要があります。

これらの要件を満たすソリューションはどれですか。

- A. Amazon EKS クラスターにアプリケーションをデプロイし、EKS マネージドノードグループを使用する。
- B. EC2 インスタンス上でセルフマネージド Kubernetes クラスターを構築し、アプリケーションをデプロイする。
- C. Amazon ECS クラスターを使用してコンテナアプリケーションをデプロイし、AWS Fargate で実行する。
- D. Amazon EKS クラスターにアプリケーションをデプロイし、EKS Auto Mode を使用する。

> [!success]- 答え：**D**
> 「Kubernetesを使いたい」→ EKS（ECSは非Kubernetesなので除外）、「インフラ管理を最小・開発に集中」→ EKS Auto Mode（ノードの調達・スケール・パッチ・OS管理までAWSが自動運用）。両軸を満たすのがD。
> - A：EKSでKubernetesは満たすが、マネージドノードグループはノードの構成・AMI更新・スケーリング設計などが依然こちらに残る。「インフラ管理を最小」ではDに劣る。
> - B：セルフマネージドKubernetesはコントロールプレーンもノードも全部自前で、管理負担が最大。要件と真逆。
> - C：Fargateで運用は楽だが、ECSはKubernetesではない。「オープンソースのKubernetesを活用したい」という要件を満たさない。
> - D：正解。EKS（Kubernetes）＋Auto Modeでノード〜スケール〜パッチまで自動化され、インフラ管理を最小化しつつ開発に集中できる。
> **関連ノート**：[[0906_EKS]] / [[0908_Kubernetes]] / [[0901_AWS_コンテナサービス概要]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第1問（Containers ｜ Kubernetes運用のマネージド化（EKS Auto Mode））]]

## 第2問（Management / Monitoring ｜ コンソールログイン失敗の検知と通知）
ある企業のセキュリティ担当者は、AWS マネジメントコンソールへ不正なログインが行われていないかを継続的に監視したいと考えています。具体的には、サインインに3回以上失敗した場合に、管理用のメーリングリスト宛にメールで通知が届くようにする必要があります。

この要件を満たすために必要な設定はどれですか。

- A. CloudWatch Logs Insights のクエリにログインイベントを指定する。CloudWatch のアラームにメール通知を設定する。
- B. CloudTrail と EventBridge を連携し、ログインイベントをメトリクスとして登録する。EventBridge のアクションにメール通知を設定する。
- C. EventBridge の監視対象にログインイベントを設定する。EventBridge のアクションにメール通知を設定する。
- D. CloudTrail と CloudWatch Logs を連携し、ログインイベントのフィルタを設定する。CloudWatch のアラームにメール通知を設定する。

> [!success]- 答え：**D**
> コンソールのサインイン記録は CloudTrail が取得し、それを CloudWatch Logs に流してメトリクスフィルタで「ログイン失敗」だけを抽出・カウントできる。「3回以上」というしきい値判定と通知は CloudWatch アラーム＋SNS の定番構成。回数をメトリクス化してしきい値で通知できるのはこの組み合わせだけ。
> - A：Logs Insights は溜まったログを人が後から検索・分析するツール（事後調査向け）。クエリ結果へリアルタイムにアラームを直接張る仕組みではなく、常時監視に不向き。
> - B：EventBridge は「イベント発生で何かを起動する」役で、イベントを“メトリクスとして登録・集計”する機能を持たない。「3回以上」という回数のしきい値判定ができない。
> - C：EventBridge は単体でコンソールログインを直接監視対象にできず（ソースは CloudTrail 経由）、回数を数えてしきい値判定することもできない。
> - D：正解。CloudTrail（記録）→CloudWatch Logs メトリクスフィルタ（失敗だけ抽出・カウント）→CloudWatch アラーム→SNS（メール通知）の鉄板パターン。
> **関連ノート**：[[1002_CloudTrail]] / [[1001_CloudWatch]] / [[0804_EventBridge]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第2問（Management / Monitoring ｜ コンソールログイン失敗の検知と通知）]]

## 第3問（Security / IAM ｜ DB認証情報のローテーションとコード最小化）
ある会社では、Amazon Auroraデータベースを利用したWebアプリケーションを運用しています。現在、Auroraデータベースへの認証情報はアプリケーションコードにハードコーディングされていますが、セキュリティ上のリスクを減らし、より安全に資格情報を管理したいと考えています。なお、既存のアプリケーションコードの変更や、追加のコードの記述は、できる限り避けたいと考えています。

この要件を満たす最も適切なアプローチはどれですか。

- A. AWS Systems Manager Parameter Store にAuroraデータベースへ接続するための資格情報をパラメータとして保存する。資格情報をローテーションするためのLambda関数を作成し、定期的に実行する。
- B. Amazon Cognito を使用してWebアプリケーションのユーザー認証を管理し、認証されたユーザーにのみAuroraデータベースへのアクセスを許可するように設定する。Cognitoユーザープールの機能で資格情報のローテーションを行う。
- C. AWS Secrets Manager にAuroraデータベースへ接続するための資格情報を保存する。Secrets Manager の自動ローテーション機能を用いて資格情報のローテーションを行う。
- D. AWS Secrets Manager にAuroraデータベースへ接続するための資格情報を保存する。資格情報をローテーションするためのLambda関数を作成し、AWS EventBridge を用いて定期的に実行する。

> [!success]- 答え：**C**
> 「DB認証情報をコードから分離して保管」「定期的にローテーション」「コードの追加・変更は最小化」を同時に満たすのは Secrets Manager の自動ローテーション。Aurora/RDS とはネイティブ統合があり、ローテーションの仕組みをAWS側が提供するため自前コードが不要。
> - A：Parameter Store には自動ローテーション機能が無く、ローテーション用Lambdaの自作が必須。「追加コードを避けたい」要件に反する。
> - B：Cognito はアプリのエンドユーザー認証（サインアップ／サインイン）の仕組みで、アプリ↔DB間の接続資格情報の保管・ローテーション機能は持たない。役割違いの最大の罠。
> - C：正解。コードからの分離・実行時のAPI取得・マネージドな自動ローテーションを一体で満たし、追加コードも最小。
> - D：Secrets Manager を使う点は良いが、ローテーションを自前Lambda＋EventBridgeで実装しており、組込みの自動ローテーションを使えば不要な手間。「最も適切」ではない。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[1605_データ保護・暗号化]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第3問（Security / IAM ｜ DB認証情報のローテーションとコード最小化）]]

## 第4問（Database ｜ 3層Webアプリの高可用DB・セッション・キャッシュ構成）
あるEコマース会社は、オンプレミスで運用している3層のWebアプリケーションをAWSに移行しようとしています。このアプリケーションはオンラインストアの運営を支援し、ユーザーが商品を検索・購入・レビューできる機能を提供しています。現在のアプリケーション構成は、ユーザーからのリクエストを受け取りレスポンスを返すWebサーバー、ビジネスロジックを処理するアプリケーションサーバー、およびユーザーデータや商品情報を保存するデータベース（PostgreSQL）です。会社は、アプリケーションの高可用性とスケーラビリティを確保し、将来のトラフィックの急増に対応するため、AWSに移行しようとしています。特に、高可用性を担保するために、Webサーバーとアプリケーションサーバーを3つのアベイラビリティーゾーンにまたがってAuto Scalingで稼働させることを計画しています。さらに、ユーザーのセッション情報を効率的に管理し、キャッシュを利用してパフォーマンスを向上させたいと考えています。

これらの要件を満たすソリューションはどれですか。

- A. データベースをAurora for PostgreSQLに移行し、セッション管理にはDynamoDB、キャッシュにはRDS Proxyを使用する。
- B. データベースをマルチAZ配置のRDS for PostgreSQLに移行し、セッション管理とキャッシュにはElastiCache for Memcachedを使用する。
- C. データベースをマルチAZ DBクラスター配置のRDS for PostgreSQLに移行し、セッション管理とキャッシュにはElastiCache for Memcachedを使用する。
- D. データベースをマルチAZ DBクラスター配置のRDS for PostgreSQLに移行し、セッション管理とキャッシュにはElastiCache for Redisを使用する。

> [!success]- 答え：**D**
> 3軸で決まる。①高可用性＋読み取りスケール＋将来の急増 → マルチAZ DBクラスター（リーダー2台が読み取り可）。②セッション管理 → Redis（レプリケーション・永続化・フェイルオーバーがあり、ノード障害でもセッションを失わない）。③キャッシュもRedisに一本化。すべて満たすのはD。
> - A：「キャッシュにRDS Proxy」が決定的な誤り。RDS Proxyはコネクションプーリング（接続の使い回し）でキャッシュではない。最大の罠。
> - B：マルチAZ"インスタンス"は待機系が読めず読み取りスケールに劣る。さらにセッションをMemcached（永続化・レプリケーションなし）に置くのは不適。二重に弱い。
> - C：DBはマルチAZ DBクラスターで良いが、セッションをMemcachedに置く点がNG（ノード障害でセッション消失）。
> - D：正解。マルチAZ DBクラスター（高可用＋読み取りスケール）＋Redis（セッションを安全に保持＆キャッシュ）。
> **関連ノート**：[[0304_ElastiCache]] / [[0301_RDS]] / [[0504_Auto_Scaling_と_Auto_Scaling_Group]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第4問（Database ｜ 3層Webアプリの高可用DB・セッション・キャッシュ構成）]]

## 第5問（Security / IAM ｜ クロスアカウントでのSQSアクセス許可）
ある大手製造業は、複数の部門ごとに異なるAWSアカウントを運用しています。マーケティング部門は、お客様からのフィードバックを処理するためのアプリケーションを開発しており、そのデータをキューに格納するためにAmazon SQSを使用しています。一方、データ分析部門は、このフィードバックデータを定期的に収集し、分析する必要があります。そのため、データ分析部門は、自部門内のAWSリソースから、マーケティング部門が管理しているSQSキューにポーリングしてデータを取得したいと考えています。

マーケティング部門はどのようにSQSキューへのアクセスを提供すべきですか。

- A. SQSキューのアクセスポリシーを設定して、データ分析部門のAWSリソースからSQSキューへのアクセスを許可する。
- B. データ分析部門のアカウント内に新しいIAMユーザーを作成し、クロスアカウントアクセス設定でSQSキューへのアクセスを許可する。
- C. マーケティング部門のIAMポリシーを設定して、データ分析部門のAWSリソースからSQSキューへのアクセスを許可する。
- D. データ分析部門のアカウント内にSNSトピックを作成し、SQSキューのアクセスポリシーにSNSトピックからSQSキューへのアクセスを許可する。

> [!success]- 答え：**A**
> SQSキューはマーケティング部門（リソース所有側）のものなので、「誰に使わせるか」は所有側が**リソースベースポリシー（SQSのアクセスポリシー）**で宣言するのが定番。`Principal` にデータ分析部門のアカウント/ロールを指定すれば、クロスアカウントのポーリングを許可できる。
> - A：正解。リソース所有側がアクセスポリシーで相手アカウントのプリンシパルを許可する、クロスアカウントSQSの王道。
> - B：別アカウント（データ分析側）に新規IAMユーザーを作っても、マーケ側キューへの許可は所有側のポリシーが要る。IAMユーザー新設は管理負担も増え、ベストプラクティス（ロール/リソースポリシー）から外れる。
> - C：マーケ部門のIDベースポリシー（IAMポリシー）は「マーケ部門のプリンシパルが何をできるか」を定めるもので、他アカウント（データ分析部門）のリソースに権限を与えることはできない。IDベースとリソースベースを取り違えた最大の罠。
> - D：SNSを挟むのはプッシュ型のファンアウト構成で、要件は「データ分析側からポーリングして取得」。不要なSNSトピックを足す過剰設計で要件と方向が違う。
> **関連ノート**：[[0801_アプリケーション統合サービス一覧]] / [[1601_セキュリティサービス一覧]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第5問（Security / IAM ｜ クロスアカウントでのSQSアクセス許可）]]

## 第6問（Management / Monitoring ｜ IAMリソースの構成変更検知）
ある開発チームでは、監査チームからセキュリティポリシーを適切に運用するよう通達があったため、IAMユーザーおよびIAMグループ、IAMロールなどの運用を見直しました。以降、IAMリソースの設定が変更された場合に検知したいと考えています。

適切なアクションは次のうちどれですか。

- A. Amazon CloudWatch で IAM のメトリクスを監視する。設定変更がされた時に Lambda 関数で検知し、検知した際は Lambda 関数から SNS FIFO トピックへの通知を行う。
- B. AWS CloudTrail で IAM リソースの運用を監視し、Lambda 関数で設定変更を検知する。検知した際は SQS キューから Lambda 関数へ通知を行う。
- C. AWS Trusted Advisor で IAM の運用を監視する。設定変更があった場合に SNS トピックへ通知されるよう設定する。
- D. IAM リソースを監視する AWS Config ルールを作成する。設定変更があった場合、SNS トピックから通知を行う。

> [!success]- 答え：**D**
> 「リソースの設定（構成）が変更されたら検知」は AWS Config の定義そのもの。Configルールで IAM リソースを監視し、構成変更時に SNS で通知する。CloudTrailは「誰が何をしたか（操作の証跡）」、Configは「設定がどう変わったか（構成の記録・評価）」という役割の違いがポイント。
> - A：IAMの設定変更はメトリクス（数値）ではないため、CloudWatchメトリクス監視では構成変更を捉えられない。
> - B：CloudTrailでも変更イベントは拾えるが、本問の直球は構成変更を評価するConfig。さらに「SQSキューからLambdaへ通知」という配線も不自然（通常はイベント→Lambda→SNS）で最適でない。
> - C：Trusted Advisor はベストプラクティス診断（コスト・セキュリティの推奨）で、リアルタイムの設定変更検知ツールではない。
> - D：正解。AWS Config ルールでIAMリソースを監視し、構成変更を検知してSNSで通知。要件に直結。
> **関連ノート**：[[1003_AWS_Config]] / [[1002_CloudTrail]] / [[0601_IAM基礎]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第6問（Management / Monitoring ｜ IAMリソースの構成変更検知）]]

## 第7問（Serverless ｜ スパイク＋アイドルなワークロードのサーバーレス化）
あるグローバル企業は、ユーザーがテキストファイルをアップロードすると、指定した言語に翻訳されてダウンロードできる Web アプリケーションを公開しています。Web アプリケーションは、ユーザーからアップロードされたテキストファイルを S3 に保存し、EC2 インスタンスで翻訳処理をした後、再び S3 に別ファイルで保存してダウンロードできる仕組みになっています。ある日ソリューションアーキテクトが過去3か月間の利用状況を確認したところ、翻訳処理が遅延するほどユーザーから大量のリクエストが突然発生する一方、長時間利用されていない期間もありました。

ソリューションアーキテクトは、リソースの効率化とコスト削減のためにどのような提案をすべきですか。

- A. スクリプトを修正して翻訳処理を EC2 から AWS Lambda に変更する。Amazon SQS と Lambda を連携させてリクエストをキューに投入する。
- B. Auto Scaling で動的スケーリングを設計し、インスタンスの増加分はスポットインスタンスを契約する。
- C. Amazon CloudFront を配置し、S3 のデータをキャッシュしてユーザーに配信する。
- D. リクエストのない期間はインスタンスを停止し、S3 にテキストファイルがアップロードされたらインスタンスを起動する Amazon EventBridge を作成する。

> [!success]- 答え：**A**
> 負荷が「突発的なスパイク＋長いアイドル」で、コスト削減も求められるワークロードはサーバーレスの独壇場。Lambdaが需要に応じて自動スケールし従量課金（アイドル時はコストほぼゼロ）、SQSがスパイクを緩衝して非同期処理する。
> - A：正解。Lambdaで自動スケール＋従量課金、SQSでスパイクを吸収。アイドル時コストゼロで効率・コスト両取り。
> - B：Auto Scaling＋スポットは固定台数よりマシだが、EC2の土台コストと管理が残り、アイドルでも最低台数ぶん課金が発生する。長いアイドルにはサーバーレスが有利。
> - C：CloudFrontは配信のキャッシュ。翻訳処理（コンピュート）のスパイク・コストには無関係で論点違い。
> - D：停止EC2をEventBridgeで起動する案は、起動の待ち時間と運用の複雑さが残り、依然EC2を管理する。サーバーレス化のAに劣る。
> **関連ノート**：[[0708_Lambda]] / [[0802_SQS]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第7問（Serverless ｜ スパイク＋アイドルなワークロードのサーバーレス化）]]

## 第8問（Application Integration ｜ SQSバックログに基づくAuto Scaling）
ある会社が分析システムを開発しています。分析システムは、データを変換するアプリケーションと、変換したデータを分析するアプリケーションの2つにプロセスが分かれており、それぞれが Auto Scaling グループ内の複数の EC2 インスタンスで実行されています。データ変換処理は1件につき1分以内に完了しますが、分析処理は完了までに数分かかります。システムの各処理でデータの損失が発生しないように、Auto Scaling に適切なスケーリングポリシーを設定する必要があります。

ソリューションアーキテクトの提案として、もっとも適切なものはどれですか。

- A. SQS キューを変換用アプリケーションと分析用アプリケーションにそれぞれ作成する。SQS キューからの通知に基づいたスケーリングポリシーを設定する。アプリケーションは SQS キューへポーリングしてリクエストを処理する。
- B. SNS トピックをデータ変換用アプリケーションと分析用アプリケーションにそれぞれ作成する。SNS トピックからの通知に基づいたスケーリングポリシーを設定する。アプリケーションは SNS トピックからリクエストを受け取って処理する。
- C. SQS キューを変換用アプリケーションと分析用アプリケーションにそれぞれ作成する。SQS キューのバックログに基づいたスケーリングポリシーを設定する。アプリケーションは SQS キューへポーリングしてリクエストを処理する。
- D. 変換用アプリケーションは処理後に5分間待機してから分析用アプリケーションにデータを送信する。それぞれの EC2 インスタンスの CPU 使用率に基づいたスケーリングポリシーを設定する。

> [!success]- 答え：**C**
> キュー処理のワーカーをAuto Scalingする際の正しい指標は「SQSのバックログ（未処理メッセージ数／インスタンスあたりのバックログ）」。溜まったら増やすことで、捌ききれず溢れてデータ損失する前にスケールアウトできる。2段それぞれに独立のSQSキューを置く疎結合も適切。
> - A：SQSはプル型（ポーリングで取得）で“通知”をしない。「SQSキューからの通知に基づくスケーリング」という指標が成立しない。Cの劣化版。
> - B：SNSはプッシュ型のファンアウトで、処理待ちを溜めるバッファではない。バックログで測れず、データ損失防止にも不向き。キュー処理にはSQS。
> - C：正解。各段をSQSで疎結合し、キューのバックログでスケール。溢れる前に増やしてデータ損失を防ぐ。
> - D：CPU使用率は待ち件数（行列の長さ）を反映しないため、溜まっていてもスケールせず取りこぼす恐れ。「5分待機」も不要なハック。
> **関連ノート**：[[0802_SQS]] / [[0504_Auto_Scaling_と_Auto_Scaling_Group]] / [[0803_SNS]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第8問（Application Integration ｜ SQSバックログに基づくAuto Scaling）]]

## 第9問（Security / IAM ｜ EBS暗号化の予防ガードレールと自動修復）
ある企業は、AWS 環境で運用している EC2 インスタンスに接続された EBS ボリュームのデータが暗号化されていることを確実にしたいと考えています。企業は、既存の暗号化されていない EBS ボリュームを自動的に検出し、暗号化するプロセスを求めています。また、暗号化されていない EBS ボリュームが新たに作成されるのも防止したいと考えています。この要件を満たすために最適なアプローチはどれですか。（2つ選択）

- A. セキュリティグループを設定して、暗号化されていない EBS ボリュームの作成を防ぐ。
- B. IAM ポリシーを設定し、暗号化されていない EBS ボリュームの作成を防ぐ。
- C. Macie を用いて、暗号化されていない EBS ボリュームを特定し、Systems Manager Automation で暗号化の修正アクションを実行する。
- D. GuardDuty を導入して、暗号化されていない EBS ボリュームを検出し、Systems Manager Session Manager を使って、コンプライアンス違反を修正し、データを自動的に暗号化する。
- E. Config で暗号化されていない EBS ボリュームを自動的に検出し、Systems Manager Automation で暗号化の修正アクションを実行する。

> [!success]- 答え：**B ＋ E**
> 要件は「新規作成を防ぐ（予防）」と「既存を検出して直す（検出＆修復）」の2本立て。予防はIAMポリシーの条件（ec2:Encryptedがfalseなら作成拒否）でB、検出＆自動修復はAWS Config（encrypted-volumesルールで検出）＋SSM Automation（修復アクションで暗号化）でE。
> - A：セキュリティグループはネットワーク通信のファイアウォールで、EBSの作成可否や暗号化とは無関係。
> - B：正解。IAMポリシーの条件で未暗号化ボリュームの作成APIを拒否し、新規作成を防げる（予防ガードレール）。
> - C：MacieはS3の機密データ（PII等）を検出・分類するサービスで、EBSの暗号化状態は対象外。
> - D：GuardDutyは脅威検知で設定コンプライアンスの検出はしない。Session ManagerはEC2へのシェル接続で自動修復ツールではない。2つとも役割違い。
> - E：正解。Configで未暗号化EBSを検出し、SSM Automationで暗号化の修復を自動実行する王道パターン。
> **関連ノート**：[[1003_AWS_Config]] / [[1405_Systems Manager]] / [[0601_IAM基礎]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第9問（Security / IAM ｜ EBS暗号化の予防ガードレールと自動修復）]]

## 第10問（Serverless ｜ エッジでの動的画像処理（Lambda@Edge））
イラストや写真素材を販売するストックフォト企業では、世界中のユーザーに高解像度の画像を届けるためAmazon CloudFrontを用いて配信しています。Webアプリケーション本体はALBの背後で稼働するEC2インスタンスが担当し、素材データはAmazon S3バケットに保存されています。購入前のプレビュー表示では改ざんや不正利用を防ぐため、ユーザーIDとアクセス日を画像の右下に埋め込んだ透かし（ウォーターマーク）を動的に合成して返す必要があります。プラットフォームには世界各地から毎秒数百件規模のプレビュー要求が届くため、オリジンサーバーに過度の負荷をかけず、運用上のオーバーヘッドを最小限に抑えられるアーキテクチャを選定したいと考えています。

この要件を最も満たす構成はどれですか。

- A. ALB配下のEC2インスタンスに画像処理ライブラリをインストールし、リクエストごとにS3からダウンロードした素材に透かしを合成して返却する。
- B. CloudFront Response Headers Policyを作成し、ユーザーIDとアクセス日をHTTPヘッダーに追加して透かしを合成する。
- C. API Gatewayのエッジ最適化エンドポイントでプレビューリクエストを受け、指定リージョン内の通常のLambda関数に転送する。LambdaがS3から原本を取得し、透かしを合成してAPIレスポンスとして返却する。
- D. CloudFrontでLambda@Edgeを実行し、S3から取得した素材に画像処理ライブラリで透かしを合成して返却する。

> [!success]- 答え：**D**
> 「CloudFront配信」「画像を動的に加工して返す（＝計算が必要）」「世界各地から毎秒数百件」「オリジン負荷を抑え運用オーバーヘッド最小」が揃うと、ユーザーに近いエッジで本格的な処理を走らせられるLambda@Edgeが定番解。
> - A：EC2でも実装できるが、リクエスト毎の合成負荷がオリジン（EC2/S3）に集中し、世界規模のトラフィックにスケールさせる運用負荷も大きい。「オリジン負荷を抑え運用最小」に反する。
> - B：Response Headers PolicyはレスポンスのHTTPヘッダーを足し引きする機能で、画像のピクセルを加工する能力はない。透かし合成は不可能。
> - C：単一リージョンのLambdaに集約すると、世界各地のユーザーから遠く、毎秒数百件が1リージョンに集中してレイテンシ・負荷面で不利。エッジ実行のLambda@Edgeに劣る。
> - D：正解。Lambda@Edgeはエッジロケーションで画像処理ライブラリを使った重めの動的処理を実行でき、S3原本に透かしを合成して返せる。ユーザー近接でオリジン負荷を抑え、サーバーレスで運用も最小。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0708_Lambda]]
> **詳しい解説**：[[503_【解説】外部サイト模擬試験3#第10問（Serverless ｜ エッジでの動的画像処理（Lambda@Edge））]]
