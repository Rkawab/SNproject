> [!info] 出題形式
> - シナリオ段落と設問文を分け、です・ます調で記述。

> - 原則は単一選択（4択 A〜D）。複数選択の問題は5択（A〜E）とし、設問文に「2つ選択」等を明記。

## 第1問（Analytics ｜ S3ログのサーバーレス分析（Glueクローラー＋Athena））
あるメディア企業は、自社Webサイトの各Webサーバーから出力されるアクセスログをS3バケットに保存しています。アクセスログはS3バケットに1分ごとにCSV形式で継続的にアップロードされ、1日の合計データ量は数百MBになります。同社の分析担当者は、サイト別・時間帯別のアクセス傾向を速やかに把握したいと考えています。

開発・運用工数を最小限に抑える最適なソリューションはどれですか。

- A. AWS GlueクローラーでS3バケット内のアクセスログファイルをカタログ化する。データカタログに登録されたテーブル定義を利用して、Amazon AthenaでS3バケット内のアクセスログファイルにSQLクエリを実行する。
- B. Amazon SageMaker AIを利用し、S3バケット内のアクセスログファイルを読み込み、Pythonスクリプトで集計や統計分析を行う。
- C. Amazon Redshift Spectrumを使用して、S3バケット内のアクセスログファイルを外部テーブルとして定義する。定義した外部テーブルに対してRedshiftで分析クエリを実行する。
- D. Amazon EMRクラスターを起動し、SparkジョブでS3バケット内のアクセスログファイルをバッチ処理し、データの集計や傾向分析を行う。

> [!success]- 答え：**A**
> S3上のログを「速やかに・最小工数」で分析したいなら、サーバーレスの組み合わせが最適。Glueクローラーがログのスキーマを自動検出してデータカタログにテーブル定義を作り、Athenaがそのテーブルに対してSQLでクエリする。どちらもサーバーレスでクラスター管理不要、数百MB程度のアドホック分析に最も軽い。
> - A：正解。Glueクローラー（スキーマ自動カタログ化）＋Athena（サーバーレスSQL）でクラスター無し・従量課金・最小運用を満たす。
> - B：SageMakerは機械学習の構築・学習基盤で、単純なログ集計には過剰。Pythonスクリプトの開発・運用工数もかかる。
> - C：Redshift SpectrumはS3を外部テーブルとしてクエリできるが、**前提としてRedshiftクラスターの構築・運用が必要**。数百MBのアドホックなログ分析にはクラスターのコスト・運用が過剰。最大の罠。
> - D：EMR＋Sparkはクラスターの起動・管理が必要で、大規模ビッグデータ向け。数百MBのログ分析には重く工数も大きい。
> **関連ノート**：[[1109_Amazon Athena]] / [[1107_AWS Glue]] / [[1101_AWS分析サービス一覧]]
> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第1問（Analytics ｜ S3ログのサーバーレス分析（Glueクローラー＋Athena））]]

## 第2問（Security / IAM ｜ KMSキー削除の防止と通知（EventBridge＋SSM Automation））
ある会社は AWS 上で社内用アプリケーションを運用しています。アプリケーションのデータは、AWS KMS（Key Management Service）のカスタマーマネージド型のキー（鍵）によって暗号化されています。同社は、アプリケーションの運用者が誤ってキーを削除しようとしたときに、キーの削除を防止するのと同時に、管理者にメールで通知するソリューションを検討しています。

要件を満たしつつ、運用上のオーバーヘッドを最小限に抑えたソリューションは次のうちどれですか。

- A. キーの削除操作を監視する CloudWatch アラームを作成する。アラーム発生時にキーの削除をキャンセル、および管理者へメール通知するアクションを作成する。
- B. AWS Config を利用して、キーの削除が実行されたときにキーを非準拠とするルールを作成する。非準拠となった場合、SNS トピックを通じて管理者へメールを送信するように EventBridge ルールを構成する。
- C. CloudTrail で運用者の操作ログを記録し、キーの削除操作をキャンセルするイベントを設定する。キーの削除を実行しようとした場合、CloudTrail によって操作のキャンセル、および SNS トピックを通じて管理者へメールを送信する。
- D. Systems Manager Automation を利用して、特定のキーの削除をキャンセルするランブックを設定する。キーの削除操作をトリガーとした EventBridge ルールを構成し、ランブックの実行、および SNS トピックを通じて管理者へメールを送信する。

> [!success]- 答え：**D**
> KMSキーは即時削除できず、削除を要求（ScheduleKeyDeletion）すると7〜30日の待機期間に入る。この「削除がスケジュールされた」という**単発の操作イベント**をEventBridgeで捕まえ、SSM Automationのランブックで CancelKeyDeletion を自動実行して取り消し、同時にSNSで管理者へ通知する。イベント駆動でサーバーレスに完結し運用負荷が最小。
> - A：CloudWatchアラームは**メトリクス（数値・しきい値）**を監視する仕組みで、「キー削除操作が起きた」という単発イベントの検知には不向き。さらにアラームのアクションはSNS通知やEC2/Auto Scaling操作に限られ、CancelKeyDeletionのような任意のAPIを実行して削除を取り消すことはできない。
> - B：Configは構成変更を事後に検知して「非準拠」と記録する検出型。削除操作を止める（キャンセルする）機能はなく、通知はできても防止にならない。
> - C：CloudTrailはあくまで操作の記録係。CloudTrail自身が操作をキャンセルすることはできない。「CloudTrailによって操作のキャンセル」が誤り。
> - D：正解。EventBridge（イベント検知）＋SSM Automationランブック（CancelKeyDeletionを実行して取り消し）＋SNS（通知）で、防止・通知・低運用を同時に満たす。
> **関連ノート**：[[1607_AWS_KMS]] / [[1405_Systems Manager]] / [[0804_EventBridge]]
> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第2問（Security / IAM ｜ KMSキー削除の防止と通知（EventBridge＋SSM Automation））]]

## 第3問（Application Integration ｜ SNS FIFO＋SQS FIFOによる順序保証ファンアウト）
ある企業では、イベント駆動型の業務サービスについて検討しています。イベントが投入されると業務サービスは処理を行い、処理結果を解析処理アプリケーションやデータベースへの保存処理アプリケーション、事務処理アプリケーションなど複数のアプリケーションへ通知します。各アプリケーションはそれぞれの処理が完了した後に新しい通知を受け取ります。そのうち、一部のアプリケーションでは処理の完了に即時〜数日かかります。なお、同じ処理を複数回実行してはなりません。

業務サービスの構成として、最も適切なものはどれですか。

- A. 処理結果の通知を Amazon SNS FIFO トピックへ行う。FIFO トピックをサブスクライブする各アプリケーション用の SQS FIFO キューを作成し、各アプリケーションはキューからメッセージを取得する。
- B. 処理結果の通知を各アプリケーション用の Amazon SQS FIFO キューへ行う。各アプリケーションはそれぞれのキューから通知を取得する。
- C. 処理結果の通知を Amazon SNS FIFO トピックへ行う。各アプリケーションはトピックのサブスクライバー（購読者）として登録する。
- D. 処理結果の通知は Amazon SNS トピックへ行う。SNS トピックをサブスクライブする1つの SQS FIFO キューを作成し、各アプリケーションはキューからメッセージを取得する。

> [!success]- 答え：**A**
> 要件は3つ。①複数アプリへ同じ通知を配る（ファンアウト）→ SNS。②同じ処理を複数回実行しない＋順序を守る（重複排除・順序保証）→ FIFO。③各アプリが自分のペースで処理し、一部は数日かかる（受信のバッファリング・独立消費）→ アプリごとの SQS キュー。この3つを同時に満たすのが「SNS FIFOトピック→各アプリ専用のSQS FIFOキュー」のファンアウト構成。
> - A：正解。SNS FIFOで重複排除・順序保証しつつ複数のSQS FIFOキューへファンアウト。各アプリは自分専用キューから処理完了後に次を取得でき、数日かかる処理もキューが吸収する。
> - B：業務サービスが各アプリ用キューへ個別に通知を送る形で、送信側が宛先を全て把握・管理する密結合。アプリ追加のたびに送信側の改修が必要でファンアウトの利点を失う。
> - C：各アプリがSNSを直接サブスクライブすると、通知はプッシュで届くだけでキューに溜められない。数日かかる処理の間に届いた通知を保持できず、「処理完了後に次を受け取る」独立消費も成立しない。
> - D：標準SNSトピック＋単一SQS FIFOキュー。標準SNSは順序保証・重複排除がなく「複数回実行しない」に反する。さらに1つのキューを全アプリで共有するとメッセージを取り合ってしまい、各アプリが全通知を受け取れない。
> **関連ノート**：[[0803_SNS]] / [[0802_SQS]] / [[0801_アプリケーション統合サービス一覧]]
> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第3問（Application Integration ｜ SNS FIFO＋SQS FIFOによる順序保証ファンアウト）]]

## 第4問（Networking / CDN / LB ｜ Lambdaからオンプレへのアクセス（VPC Lambda））
ある会社は、オンプレミスのシステムの一部を AWS へ移行しています。移行するにあたり、AWS Site-to-Site VPN を契約してオンプレミスと AWS プライベートネットワークを接続しました。オンプレミスで稼働していたアプリケーションを AWS Lambda に移行する計画を進めていますが、Lambda 関数からオンプレミス内のプライベートネットワーク上にあるデータベースにアクセスする必要があります。なお、システム移行先の VPC は、デフォルトルートが仮想プライベートゲートウェイ（VGW）に設定されています。

要件を満たすには、どうすればよいですか。

- A. Lambda 関数をインターネットに接続できるようにして、インターネット経由でオンプレミスへアクセスする。
- B. Lambda 関数が VPC へアクセスできるように、VPC Lambda の設定をする。
- C. Lambda 関数から直接仮想プライベートゲートウェイにアクセスするようにコードを書き換える。
- D. Lambda 関数が AWS Site-to-Site VPN 経由でオンプレミスへアクセスできるように、VPC のルートテーブルを更新する。

> [!success]- 答え：**B**
> Lambdaはデフォルトでは AWS 管理のネットワーク上で動き、VPC内リソースやオンプレのプライベートIPには到達できない。Lambda関数を VPC に接続（VPC Lambda 設定＝ENIをVPC内に作成）すれば、VPCの経路が使える。問題文で「VPCのデフォルトルートがVGWに設定済み」＝VPC→オンプレの経路は既にあるので、LambdaをVPCに入れさえすればVPN経由でオンプレDBに到達できる。
> - A：オンプレの「プライベートネットワーク上」のDBへインターネット経由でアクセスするのは要件（プライベート接続）に反し、せっかくのSite-to-Site VPNを使わない構成で不適切。
> - B：正解。LambdaをVPCにアタッチする設定を行えば、既存のVGWデフォルトルート経由でオンプレDBにプライベート接続できる。
> - C：VGWはVPCのコンポーネントで、Lambdaのコードから直接呼び出す対象ではない。コード書き換えで到達できるものではなく無意味。
> - D：VPCのルートテーブルは既にデフォルトルートがVGWを向いており、経路は整っている（更新不要）。足りないのは経路ではなく「LambdaがまだVPCの中に居ない」こと。ルートテーブルを触っても、VPC未接続のLambdaはそのルートを使えない。惜しい罠。
> **関連ノート**：[[0708_Lambda]] / [[0405_ネットワークとコンテンツ配信]] / [[0406_VPC応用機能]]
> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第4問（Networking / CDN / LB ｜ Lambdaからオンプレへのアクセス（VPC Lambda））]]

## 第5問（Networking / CDN / LB ｜ 静的＋動的コンテンツのグローバル低遅延配信（CloudFront複数オリジン））
あるメディア企業は、ニュースコンテンツを世界中の視聴者に向けて配信するWebサイトを運営しています。Webサイトは動的に生成される記事と、画像や動画などの静的コンテンツがあります。動的コンテンツはALB配下の複数のEC2インスタンス上で実行されるアプリケーションで生成され、静的コンテンツはS3バケットに保存されています。同社は視聴者のグローバル化を見据えて、静的コンテンツと動的コンテンツの配信パフォーマンスを向上させて、可能な限り低遅延で配信したいと考えています。なお、WebサイトはRoute 53に登録した独自ドメインを使用しています。

世界中の視聴者に対してニュースコンテンツを低遅延で配信するために、もっとも適切な方法はどれですか。

- A. CloudFrontでALBとS3バケットをオリジンとして設定する。Route 53でCloudFrontにトラフィックをルーティングするように独自ドメインのDNSを設定する。
- B. Global AcceleratorでALBとS3バケットをそれぞれエンドポイントとして設定する。Global Acceleratorに割り当てられたIPアドレスにトラフィックをルーティングするように、Route 53で独自ドメインのDNSを設定する。
- C. ALBとEC2インスタンスを世界各地のリージョンに配置する。S3バケットのクロスリージョンレプリケーションを有効にする。Route 53で地理的近接性ルーティングポリシーを使用して、独自ドメインのDNSを設定する。
- D. CloudFrontでS3バケットをオリジンとして設定する。Global AcceleratorでALBとCloudFrontをエンドポイントとして設定する。Route 53でGlobal Acceleratorにトラフィックをルーティングするように独自ドメインのDNSを設定する。

> [!success]- 答え：**A**
> Webサイト（HTTP）の静的＋動的コンテンツを世界中へ低遅延配信するのはCloudFront（CDN）の役割。CloudFrontは複数オリジンを設定でき、静的はS3オリジン（エッジでキャッシュ）、動的はALBオリジン（エッジのキャッシュ動作＋AWSバックボーンで加速）を1つのディストリビューションで扱える。Route 53のエイリアスでCloudFrontに独自ドメインを向ければ完成。
> - A：正解。CloudFront＋複数オリジン（S3・ALB）で静的キャッシュと動的加速を両立し、Route 53でCloudFrontへルーティングする王道パターン。
> - B：Global AcceleratorはTCP/UDPアプリの高速化・静的IP向けで、コンテンツのキャッシュはしない。HTTPのWebコンテンツ（特に静的）の低遅延配信にはCloudFrontが適切。
> - C：ALB/EC2を世界各地のリージョンに展開＋CRR＋地理的近接性ルーティングは、莫大なインフラコストと運用負荷。CloudFrontで同じ低遅延を実現できるのに過剰。
> - D：CloudFrontの前段にGlobal Acceleratorを置く多段構成は不自然で無意味（CloudFront自体が世界中のエッジを持つ）。構成が複雑化するだけ。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0404_Route53]] / [[0503_ALB・NLB・GWLBの違い]]
> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第5問（Networking / CDN / LB ｜ 静的＋動的コンテンツのグローバル低遅延配信（CloudFront複数オリジン））]]

## 第6問（Security / IAM ｜ 組織全体の利用サービス制限（RootへのSCP））
ある大手製造業の企業では、部署ごとに存在する AWS アカウントを AWS Organizations を用いて一元管理しています。同社では、組織全体でのセキュリティの強化を目指しており、メンバーアカウントのルートユーザーを含む全てのユーザーに対して、Amazon EC2 と Amazon RDS のみの使用を許可し、その他の全ての AWS サービスの使用を禁止したいと考えています。なお、同社では AWS Organizations の「すべての機能」の機能セットを有効にしています。

これらの要件を効率的に満たすために最適な方法はどれですか。

- A. AWS CloudTrail と AWS CloudWatch を連携させ、EC2 と RDS 以外の AWS サービスの利用が検出されたら通知を送信するように構成する。
- B. 組織（Root）に対して、EC2 と RDS 以外の AWS サービスの利用を禁止する SCP（サービスコントロールポリシー）を設定する。
- C. 各部署の OU に対して、EC2 と RDS 以外の AWS サービスの利用を禁止する SCP（サービスコントロールポリシー）を設定する。
- D. 各部署のルートユーザーを含む全ての IAM ユーザーが所属する IAM グループを作成し、EC2 と RDS 以外の AWS サービスに対するアクションを拒否する IAM ポリシーをアタッチする。

> [!success]- 答え：**B**
> メンバーアカウントの「ルートユーザーを含む全ユーザー」を制限できるのは SCP だけ（IAMポリシーはメンバーアカウントのルートユーザーに効かない）。かつ「組織全体」が対象なので、Root に1つ SCP を付ければ全メンバーアカウントを一括でカバーでき最も効率的。「すべての機能」有効が前提でSCPが使える。
> - A：CloudTrail＋CloudWatchは「使われたら検出して通知」する事後の検出型で、利用を禁止（予防）できない。要件は禁止なので不適。
> - B：正解。RootにSCPを1つ設定すれば、全メンバーアカウントのルートユーザーを含む全プリンシパルにEC2/RDS以外を禁止でき、組織全体を一括制御できて効率的。
> - C：SCP自体は正しいが、各OUごとに個別設定する必要があり、Rootに一括で付けるBより手間が多い。新しいOUの追加時に付け忘れるリスクもあり「効率的」でBに劣る。
> - D：IAMポリシーは**メンバーアカウントのルートユーザーには効かない**（ルートは常にフルアクセス）。「ルートユーザーを含む全ユーザー」を制限する要件を満たせない。そもそも複数アカウントのルートを1つのIAMグループにまとめる構成も不可能。
> **関連ノート**：[[1403_Organizations]] / [[0601_IAM基礎]] / [[1601_セキュリティサービス一覧]]
> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第6問（Security / IAM ｜ 組織全体の利用サービス制限（RootへのSCP））]]

## 第7問（Security / IAM ｜ プライベートEC2への安全接続とコマンド監査（SSM Session Manager））
ある製薬会社は、機密性の高い業務データを処理する EC2 インスタンスを、VPC 内のプライベートサブネットで運用しています。これらのインスタンスにはパブリック IP アドレスが割り当てられておらず、インターネットからのインバウンド通信も許可されていません。運用管理者は、障害対応や定期メンテナンスのため、これらのインスタンスに安全にリモート接続する必要があります。加えて、管理者の認証には多要素認証（MFA）を必須としたいです。さらに、監査要件として、管理者が実行したすべてのシェルコマンドを一元的に記録し、後から確認できる必要があります。これらの要件を、運用上のオーバーヘッドを最小限に抑えながら満たす選択肢として、最も適切なものはどれですか。

- A. AWS Client VPN エンドポイントを構成し、管理者が VPC に接続したうえで、プライベートサブネット内の EC2 インスタンスへ SSH で接続できるようにする。管理者認証には MFA を必須とし、EC2 インスタンスで実行したコマンドをローカルのシェル履歴として保持し、必要に応じて監査証跡として確認する。
- B. EC2 Instance Connect エンドポイントを設定し、管理者がプライベートサブネット内の EC2 インスタンスへ SSH で接続できるようにする。IAM ポリシーによって MFA を必須化し、接続に関するアクティビティの監査には AWS CloudTrail で記録する。
- C. パブリックサブネットに踏み台ホストを配置し、管理者には MFA を用いた SSH アクセスを許可する。踏み台ホストに Amazon CloudWatch エージェントを導入し、SSH セッションログを Amazon CloudWatch Logs へ送信するよう設定する。
- D. EC2 インスタンスに SSM Agent をインストールする。AWS Systems Manager Session Manager を使用して EC2 インスタンスへ接続し、セッションログを Amazon CloudWatch Logs に送信する。運用管理者には MFA を必須とする IAM ポリシーを適用する。

> [!success]- 答え：**D**
> Session Managerは、SSM Agent経由のアウトバウンド接続でプライベートEC2に接続するため、インバウンドポート開放・パブリックIP・踏み台が一切不要（運用最小）。セッション中の全シェルコマンド・出力をCloudWatch Logs（またはS3）へ一元記録でき、監査要件を満たす。IAMポリシーの条件でMFAを必須化できる。
> - A：Client VPN＋SSHで接続はできるが、「コマンドをローカルのシェル履歴として保持」では改ざん可能で一元的な監査記録にならない。監査要件を満たさず、VPN管理の運用負荷も増える。
> - B：EC2 Instance ConnectエンドポイントでプライベートEC2へSSH接続自体は可能だが、監査がCloudTrailのみ。CloudTrailは「接続したというAPI操作」は記録しても、セッション内で実行した個々のシェルコマンドの中身は記録しない。「すべてのシェルコマンドを記録」の要件を満たせない（最大の罠）。
> - C：踏み台ホストをパブリックサブネットに置く時点で攻撃面が増え、踏み台のパッチ・可用性管理など運用負荷が高い。CloudWatch AgentでのSSHセッションログ収集も設定が煩雑で確実性に欠ける。
> - D：正解。Session Managerで踏み台レス・ポート開放レス接続、CloudWatch Logsへ全コマンドを一元記録、IAMでMFA必須。監査・セキュリティ・低運用を全て満たす。
> **関連ノート**：[[1405_Systems Manager]] / [[0601_IAM基礎]] / [[1001_CloudWatch]]
> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第7問（Security / IAM ｜ プライベートEC2への安全接続とコマンド監査（SSM Session Manager））]]

## 第8問（Analytics ｜ IoTストリームの順序保持とRedshiftへの配信）
ある企業では、家電製品として販売しているIoTデバイスからリアルタイムに情報を収集し、AWSクラウドへ保存しています。収集したデータは、デバイスの種類ごとに分類し、データを取得した順序を維持してデータウェアハウスへ蓄積し、将来の分析などに利用する予定です。

最も適切なソリューションはどれですか。

- A. Amazon Kinesis Data Streamsでデータを受信し、Amazon Data Firehoseを利用してAmazon Redshiftへデータを送信する。
- B. Amazon Data Firehoseでデータを受信し、Amazon S3バケットへデータを送信する。
- C. Amazon SNS FIFOトピックでデータを受信し、Amazon SQS FIFOキューからAWS Lambda関数で呼び出してAmazon S3バケットへデータを送信する。
- D. Amazon Kinesis Data Streamsでデータを受信し、Amazon Data Firehoseを利用してAmazon ElastiCacheへデータを送信する。

> [!success]- 答え：**A**
> 要件は「リアルタイム収集」「種類ごとのグループ化と順序」「データウェアハウスへの蓄積」。Kinesis Data Streamsではデバイス種別などをパーティションキーにして同じシャードへ振り分け、シャード内のレコードをシーケンス番号順に扱える。Data Firehoseを配信役にして、データウェアハウスであるRedshiftへロードする構成が全要件を満たす。
> - A：正解。Data Streamsでグループ化・順序を扱い、Data FirehoseでRedshiftへマネージドに配信する。FirehoseはS3を中間保存先にしてRedshiftのCOPYを実行する。
> - B：Data FirehoseからS3への配信はリアルタイムデータの保存には使えるが、S3はオブジェクトストレージ／データレイクであり、指定されたデータウェアハウスではない。データの順序を扱うData Streamsもない。
> - C：SNS FIFO＋SQS FIFOはメッセージングの順序保証には使えるが、IoTストリーミング分析基盤としては構成が過剰で、保存先もS3のままなのでデータウェアハウス要件を満たさない。
> - D：ElastiCacheは低遅延キャッシュであり、長期分析用のデータウェアハウスではない。Data Firehoseの配信先として選ぶ構成でもない。
> **関連ノート**：[[1103_Amazon Kinesis]] / [[1113_Amazon Redshift]] / [[1101_AWS分析サービス一覧]]
> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第8問（Analytics ｜ IoTストリームの順序保持とRedshiftへの配信）]]

## 第9問（Cost Optimization ｜ SQSワーカーのベースラインRI＋変動分Spot）
ある会社は、AWS上で本番用Webアプリケーションを運用しています。このWebアプリケーションはAmazon SQSキューからデータを取得し、Auto Scaling配下にある複数のEC2インスタンスで並列に処理しています。アプリケーションに対する需要は不規則に変動し、予測が困難です。会社は費用対効果を最大化しつつ、アプリケーションのダウンタイムが発生することなくユーザーが継続的に利用できるようにしたいと考えています。

最適なソリューションはどれですか。

- A. ベースラインとなる台数のEC2インスタンスにリザーブドインスタンスの割引を適用し、需要増加に対応するEC2インスタンスをスポットインスタンスとして起動する。
- B. 最大の需要に対応できる台数のEC2インスタンスにリザーブドインスタンスの割引を適用する。
- C. ベースラインとなる台数のEC2インスタンスをオンデマンドインスタンスとして起動し、需要増加に対応するEC2インスタンスをスポットインスタンスとして起動する。
- D. ベースラインとなる台数のEC2インスタンスにリザーブドインスタンスの割引を適用し、需要増加に対応するEC2インスタンスをオンデマンドインスタンスとして起動する。

> [!success]- 答え：**A**
> 常時必要なベースライン容量には長期利用割引のReserved Instancesを適用し、予測困難な増加分には最安のSpot Instancesを使う。処理はSQSで疎結合化され、複数EC2で並列実行されているため、Spotワーカーが中断されても未完了メッセージは可視性タイムアウト後に再び処理できる。Auto Scalingも代替容量を起動するため、個々のワーカー中断をアプリケーション全体の停止にせず、コストを最大限削減できる。
> - A：正解。安定部分をRI、SQSが中断を吸収できる変動部分をSpotにする、可用性とコストの定番の組み合わせ。
> - B：最大需要分までRIを購入すると、需要が少ない期間に割引枠を使い切れず、予測困難な変動負荷に対して過剰契約になる。
> - C：増加分のSpotは適切だが、常時稼働するベースラインまでOn-Demandにすると、RIの長期利用割引を逃す。
> - D：可用性は高いが、SQSで再試行可能な変動分までOn-Demandにするため、費用対効果の最大化ではAに劣る。
> **関連ノート**：[[0102_EC2購入プラン]] / [[0802_SQS]] / [[0103_Auto Scaling]]
> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第9問（Cost Optimization ｜ SQSワーカーのベースラインRI＋変動分Spot）]]

## 第10問（Database ｜ リアルタイムチャットの最近メッセージをRedisでキャッシュ）
あるベンチャー企業がリアルタイムのチャットアプリケーションを開発しています。このアプリケーションは、データベースにAmazon Auroraを利用しています。ユーザーが数百人のグループチャットでも、遅延を最小限にしてリアルタイムでメッセージを表示させたいと考えています。また、ユーザーがログアウトしても、再度ログインした際に最近のメッセージを即座に表示する必要があります。

最も要件に合うソリューションはどれですか。

- A. Aurora Serverlessを利用してデータベースのコンピューティング容量を自動的にスケーリングする。
- B. Amazon CloudFrontを利用してメッセージをキャッシュする。
- C. Amazon ElastiCache for Memcachedを利用してメッセージをキャッシュする。
- D. Amazon ElastiCache for Redisを利用してメッセージをキャッシュする。

> [!success]- 答え：**D**
> Auroraを全メッセージの永続的な正本として維持し、頻繁に参照する最近のメッセージをElastiCache for Redisへ置けば、インメモリから低遅延で表示できる。Redisはリストなどの豊富なデータ構造、レプリケーション、スナップショット、Pub/Subを利用でき、チャットやセッションのような状態を持つ用途に適する。
> - A：Aurora Serverlessは負荷に応じてACU（CPU・メモリ等のDBコンピューティング容量）を増減する。DBの処理能力には効くが、最近の同じメッセージを毎回DBから読む経路をキャッシュで短縮する回答ではない。
> - B：CloudFrontはHTTPコンテンツをエッジにキャッシュして世界配信するCDN。ユーザーごとに頻繁に変化するチャットメッセージの共有状態を保持する用途ではない。
> - C：Memcachedも高速なインメモリキャッシュだが、単純なKey-Valueキャッシュ向けで、Redisのようなレプリケーション、永続化、Pub/Sub、豊富なデータ構造を持たない。最近のメッセージを保持するチャット用途ではRedisが適切。
> - D：正解。Auroraへの読み取りを減らし、最近のメッセージを低遅延で返しながら、再ログイン後も共有キャッシュから即座に表示できる。

> **詳しい解説**：[[508_【解説】外部サイト模擬試験8#第10問（Database ｜ リアルタイムチャットの最近メッセージをRedisでキャッシュ）]]
