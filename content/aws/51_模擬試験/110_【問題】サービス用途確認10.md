# 【問題】サービス用途確認10（公式サンプル風）

## この問題集の目的

要件文から主要サービスの役割を選ぶドリル。101〜109で扱っていない切り口（運用支援サービスのチェック項目の区別など）を中心に、13分野を横断する。本番（SAA-C03）相当に届かない短めの用途確認問題はこのファイルに蓄積する。

## 対象レベル

SAA入口。単語暗記ではなく、各サービスが「何を見てくれる人か」を区別する練習にする。

## 問題数

蓄積中（10問を超えたら111へ分割）

## 第1問（Management / Monitoring ｜ Trusted Advisorのチェック項目）
AWS Trusted Advisor は、アカウントの構成をベストプラクティスと照らし合わせ、5つのカテゴリ（コスト最適化・パフォーマンス・セキュリティ・フォールトトレランス・サービス制限）で推奨事項を提示します。
Trusted Advisor のチェック対象として正しいものはどれですか。（3つ選択）

- A. アクセス元が 0.0.0.0/0 で設定されたセキュリティグループが定義されていないか
- B. EC2インスタンスのCPU使用率が90%を超えたまま長期間運用されていないか
- C. ELB配下のインスタンスが異なるAZで稼働しているか
- D. 請求額が前月と比較して10%以上増加していないか
- E. EC2インスタンスが故障していないか

> [!success]- 答え：**A, B, C**
> Trusted Advisorは「アカウントの健康診断をしてくれるお医者さん」。構成を5カテゴリで横断的に点検し、改善を提案する。
> - A：正解。**セキュリティ**カテゴリ。特定ポートが 0.0.0.0/0（全開放）になっているセキュリティグループを検出する。
> - B：正解。**パフォーマンス**カテゴリ「使用率の高いEC2インスタンス」。CPUが長期間90%超ならサイズアップ等を推奨する。
> - C：正解。**フォールトトレランス**カテゴリ。ELB配下が単一AZに偏っていないか（複数AZに分散しているか）を点検する。
> - D：不正解。「前月比◯%増」のような請求額の閾値監視は **AWS Budgets / Cost Anomaly Detection** の役割。Trusted Advisorのコスト最適化は"アイドルなELB・低使用率EC2・未使用EIP・RI提案"など無駄の発見が中心で、月次増加率の通知は行わない。
> - E：不正解。インスタンスの故障検知は **EC2ステータスチェック / CloudWatch** の領域で、Trusted Advisorの担当ではない。
> **着眼点（SAA）**：問われるのは「Trusted Advisorの5本柱」と、"請求の増減監視＝Budgets"・"リソースの死活監視＝CloudWatch/ステータスチェック" との**役割分担**。個別チェック項目の閾値暗記までは不要。
> **関連ノート**：[[1201_Trusted_Advisor]] / [[1701_コスト管理サービス一覧]]

## 第2問（Security / IAM ｜ 機密データ検出（Macie））
S3バケットに保存された大量のドキュメントの中に、住所・電話番号・クレジットカード番号などの個人識別情報（PII）が紛れていないかを、自動で検出・分類したいと考えています。
この用途に最も適したサービスはどれですか。

- A. Amazon Macie
- B. Amazon Inspector
- C. Amazon GuardDuty
- D. AWS Config

> [!success]- 答え：**A**
> Macieは「**S3専門の“個人情報探偵”**」。機械学習でS3内のPII・機密データを検出・分類する。
> - A：正解。S3の機密データ検出・分類はMacie。検出結果はEventBridge→SNSでメール通知できる。
> - B：Inspectorは**EC2/ECR/Lambdaの脆弱性(CVE)スキャン**。データ中身の分類はしない。
> - C：GuardDutyは**不審な挙動・脅威の検知**（ログ分析）。PII分類はしない。
> - D：Configは**構成（設定）の記録・準拠評価**。データ中身は見ない。
> **着眼点（SAA）**：「**S3の中の個人情報/機密データを見つける**」＝Macie。脆弱性＝Inspector、脅威検知＝GuardDuty、構成評価＝Config、と役割で切り分ける。
> **関連ノート**：[[1603_脅威検知・監査]] / [[1601_セキュリティサービス一覧]]

## 第3問（Containers ｜ コンテナのサーバー管理（Fargate））
ECSまたはEKSでコンテナを動かしたいが、ワーカーノード（EC2）のOSパッチ・容量管理・スケーリングを自分で運用したくありません。
この要件に最も適した起動方法はどれですか。

- A. AWS Fargate 起動タイプ
- B. EC2 起動タイプ
- C. Amazon EC2 Dedicated Hosts
- D. 自前で構築したKubernetesクラスター

> [!success]- 答え：**A**
> Fargateは「**サーバーを持たずにコンテナを動かす**」仕組み。ノードのパッチ・容量・スケールをAWSが受け持つ。
> - A：正解。ノード管理不要でコンテナを実行できる（ECS/EKS両対応）。
> - B：EC2起動タイプは**ノードを自分で管理**する。運用したくない要件に反する。
> - C：Dedicated Hostsは物理サーバー専有で管理はさらに重い。
> - D：自前Kubernetesは全自己管理で運用最大。
> **着眼点（SAA）**：「**コンテナ＋ノード管理したくない**」→ Fargate。ただし**FargateはGPU非対応**なので、GPUが要るコンテナは**EC2起動タイプ＋GPUインスタンス**に切り替える（頻出ひっかけ）。
> **関連ノート**：[[0901_AWS_コンテナサービス概要]] / [[0906_EKS]] / [[0904_ECS]]

## 第4問（Security / IAM ｜ WAFの保護対象）
AWS WAF の Web ACL を関連付けて、L7（HTTP）レベルの攻撃（SQLインジェクション・XSS等）から保護したいと考えています。
次のうち、WAF の Web ACL を**関連付けできない**（保護対象にできない）ものはどれですか。

- A. Amazon CloudFront ディストリビューション
- B. Application Load Balancer（ALB）
- C. Amazon API Gateway
- D. Network Load Balancer（NLB）

> [!success]- 答え：**D**
> WAFは**HTTPの中身を読んで判定するL7の道具**。中身を見ないL4の**NLBには関連付けできない**。
> - A/B/C：いずれもL7を扱えるのでWAF関連付け可（他にAppSync・Cognito・App Runnerも）。
> - D：正解（＝関連付け不可）。NLBはL4（TCP/UDP）で中身を見ないためWAF非対応。
> **着眼点（SAA）**：**WAFを付けられる＝CloudFront / ALB / API Gateway / AppSync / Cognito / App Runner**。**NLB・EC2直・RDSには付けられない**。「NLBをWAFで保護」は成立しない頻出ひっかけ。DDoS(量)はShield。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[0503_ALB・NLB・GWLBの違い]]

## 第5問（Security / IAM ｜ マルチアカウントSSO（IAM Identity Center））
AWS Organizations 配下の複数アカウントに対し、社内で管理している Active Directory のユーザーIDを使って、一元的にシングルサインオン（SSO）でログインさせたいと考えています。
この用途に最も適したサービスはどれですか。

- A. AWS IAM Identity Center（旧 AWS SSO）
- B. Amazon Cognito
- C. 各アカウントに個別のIAMユーザーを作成する
- D. IAMアクセスキーを全社員に配布する

> [!success]- 答え：**A**
> IAM Identity Centerは「**複数AWSアカウントへの共通ログイン受付**」。Directory Service経由でオンプレADと連携し、既存IDのままSSOできる。
> - A：正解。Organizations配下の全アカウントSSO＋外部IdP/AD連携が役割。
> - B：Cognitoは**アプリ（Web/モバイル）のエンドユーザー認証**用で、従業員のAWSコンソールSSOではない。
> - C：IAMユーザー個別作成は一元管理と真逆で管理煩雑。
> - D：アクセスキー配布は長期認証情報の乱立で危険。
> **着眼点（SAA）**：「**複数アカウント＋社内AD/外部IdPで一元SSO**」→ IAM Identity Center。**Cognito＝アプリのエンドユーザー認証**との取り違えが頻出。
> **関連ノート**：[[1602_アイデンティティ系]] / [[1403_Organizations]]

## 第6問（Management / Monitoring ｜ 確実なフェイルオーバー（Application Recovery Controller））
マルチリージョン構成で、障害時に別リージョンへ切り替えたい。その際、**切替先リージョンが確実にリクエストを処理できる状態か**を継続的に検証しつつ、確実にフェイルオーバーしたいと考えています。
この用途に最も適したサービスはどれですか。

- A. Amazon Application Recovery Controller（ARC）
- B. Amazon Route 53 のヘルスチェックのみ
- C. AWS Fault Injection Service（FIS）
- D. Amazon CloudWatch アラーム

> [!success]- 答え：**A**
> ARCは「**切替先の準備を点検してから、確実に切り替える本部＋非常用スイッチ**」。Readiness Checkで容量・構成・クォータを検証、ルーティングコントロールで安全に切替。
> - A：正解。切替先の処理可能性を保証しつつマネージドにフェイルオーバー。
> - B：ヘルスチェックは「応答するか」だけで**切替先の容量までは保証しない**。
> - C：FISは**障害をわざと起こしてテスト**する道具で、本番切替の制御ではない。
> - D：CloudWatchは監視・アラームで、切替の保証・実行はしない。
> **着眼点（SAA）**：「**切替先が確実に処理できる状態を保証（Readiness）＋確実なフェイルオーバー**」→ ARC。**FIS＝障害テスト（カオス）**との違いに注意。
> **関連ノート**：[[1406_Application_Recovery_Controller]] / [[1407_Fault_Injection_Service]]

## 第7問（Application Integration ｜ API Gatewayのキャッシュ）
API Gateway（REST API）の背後にあるバックエンドに、**同じ内容のリクエストが繰り返し届いて過負荷**になっています。バックエンドへの到達数そのものを減らして負荷を下げたいと考えています。
この用途に最も適した対応はどれですか。

- A. API Gateway のキャッシュを有効化し、キャッシュキーを設定する
- B. 使用量プランでスロットルとクォータを設定する
- C. WebSocket API に変更する
- D. バックエンドを Lambda にして自動スケールさせる

> [!success]- 答え：**A**
> キャッシュは「**よくある質問の答えを受付が貼っておく**」仕組み。同じキーの2回目以降はバックエンド未到達で即応答＝重複が消える。
> - A：正解。重複リクエストを手前で吸収し、バックエンド到達数を根本的に削減。
> - B：使用量プラン（スロットル/クォータ）は**リクエストを上限で絞る**だけで、重複取得は減らず正当なリクエストまで制限。
> - C：WebSocketは双方向リアルタイム用で、重複リクエスト削減とは無関係。
> - D：自動スケールは**バックエンドへのアクセスをむしろ増やす**。
> **着眼点（SAA）**：「**同じデータへの繰り返しで過負荷**」→ **キャッシュ**（API GatewayはREST APIのみ）。**使用量プラン＝流量を絞る/課金管理**で負荷の根本解決ではない、が頻出ひっかけ。
> **関連ノート**：[[0809_API_Gateway]] / [[0304_ElastiCache]]
