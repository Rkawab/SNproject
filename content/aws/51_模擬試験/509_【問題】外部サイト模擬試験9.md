# 【問題】外部サイト模擬試験 9（複数分野）

## この問題集の目的

外部サイトの模擬問題を、答えの暗記ではなく本番形式（長文シナリオ）で解く練習として収録する。`508` が10問で満杯になったため、その続きを本ファイルに蓄積する。分野は固定せず、出題された問題を順次追加していく。

> [!info] 出題形式
> - シナリオ段落と設問文を分け、です・ます調で記述。
> - 選択肢はサービス名ではなく、具体的な操作・構成で記述。
> - 原則は単一選択（4択 A〜D）。複数選択の問題は5択（A〜E）とし、設問文に「2つ選択」等を明記。

> [!note] 運用ルール
> 外部サイト模擬試験は分野横断で1ファイルに蓄積する。**1ファイルが10問を超えたら `510_…` 以降に新規ファイルを作成**して分割する（解説側 `61_模擬問題解説/` も同じ番号で対応させる）。

## 第1問（Networking / CDN / LB ｜ 特定ドメインへのアウトバウンド制限（Network Firewall））
ある企業では、プライベートサブネット内の EC2 インスタンス上でデータ分析アプリケーションをホストしています。このアプリケーションは、インターネット上に公開されている外部 API の URL に定期的にアクセスして気象情報に関するデータを取得しています。同社では現在、ネットワークセキュリティの強化が求められており、インターネットを介した通信については、この API と EC2 上のアプリケーション間のみ許可したいと考えています。

これらの要件を満たす方法はどれですか。

- A. AWS Shield Advanced を構成し、API との通信以外のすべてのネットワークトラフィックをブロックするルールを作成する。
- B. AWS Firewall Manager を用いて、VPC を出入りするすべてのネットワークトラフィックを検査する。指定した URL とインターネットゲートウェイ間の通信のみ許可するファイアウォールの設定を行う。
- C. AWS Network Firewall を構成し、API との通信のみを許可するドメイン名によるフィルタリングのルールを作成する。プライベートサブネットからのアウトバウンド通信が Network Firewall を経由するようにルートテーブルを修正する。
- D. EC2 インスタンスのセキュリティグループで、API の URL とのインバウンド及びアウトバウンド通信のみ許可するように設定する。

> [!success]- 答え：**C**
> 「特定のドメイン（URL）へのアウトバウンドのみ許可、他は遮断」を実現できるのは、ドメイン名（FQDN）フィルタリングを持つ AWS Network Firewall。プライベートサブネットのアウトバウンドをルートテーブルで Network Firewall 経由に向け、許可ドメイン以外を落とす。外部APIのIPは可変でも、ドメイン名で許可できる。
> - A：Shield AdvancedはDDoS攻撃の防御サービス。アウトバウンドを特定ドメインに限定する用途ではなく、要件と無関係。
> - B：Firewall Managerは複数アカウント/リソースのWAF・SGルール等を一元管理する“管理”サービスで、それ自体がドメイン名でトラフィックを検査・許可する実体ではない。記述も不正確。
> - C：正解。Network Firewallのドメイン名フィルタリングで特定APIのみ許可し、ルートテーブルでアウトバウンドをFirewall経由に通す。
> - D：セキュリティグループのルールは**IPアドレス（CIDR）やセキュリティグループ**でしか指定できず、**URL/ドメイン名を条件にできない**。外部APIのIPは変動もするため、SGでは要件を満たせない（最大の罠）。
> **関連ノート**：[[1604_ネットワーク防御]] / [[0408_セキュリティグループとNACL]] / [[1601_セキュリティサービス一覧]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第1問（Networking / CDN / LB ｜ 特定ドメインへのアウトバウンド制限（Network Firewall））]]

## 第2問（Security / IAM ｜ マルチアカウント構成とオンプレADによる認証一元化）
ある多国籍企業は、グローバルな IT ガバナンスを強化するため、各地域の事業部ごとに個別の AWS アカウントを使用したマルチアカウント構成に移行することを計画しています。ユーザー認証は、本社で管理しているオンプレミスのディレクトリサービス（Microsoft Active Directory）を用いて、全 AWS アカウントを一元的に管理したいと考えています。

この要件を満たすために有効なアクションの組み合わせはどれですか。（2つ選択）

- A. AWS Organizations で新しい組織を作成し、認証メカニズムに AWS Directory Service を直接使用するように組織を設定する。
- B. 本社の Active Directory に Amazon Cognito を接続し、Cognito を用いてすべての AWS アカウントのユーザー認証を行う仕組みを構成する。
- C. 各 AWS アカウントごとに個別の IAM ユーザーを作成し、本社の Active Directory から一時的な認証情報を配布する。
- D. AWS Organizations ですべての機能を有効にした組織を作成し、各事業部用の AWS アカウントを追加する。
- E. AWS IAM Identity Center を設定し、AWS Directory Service を介して本社の Active Directory と連携する。

> [!success]- 答え：**D, E**
> マルチアカウントの一元管理には2層が要る。①**器**：AWS Organizations（すべての機能を有効）で組織を作り、事業部ごとのアカウントを束ねる（D）。②**認証**：IAM Identity Center を本社ADと（AWS Directory Service経由で）連携し、AD の資格情報で全アカウントへSSOする（E）。DとEはセットで初めて「オンプレADで全AWSアカウントを一元管理」を実現する。
> - A：Organizationsは複数アカウントを束ねる仕組みで、それ自体が「認証メカニズム」を持たない。認証をADと繋ぐのはIAM Identity Centerの役割で、「組織に認証を直接設定」は誤り。
> - B：CognitoはWeb/モバイルアプリのエンドユーザー認証向け。従業員がAWSアカウント/コンソールへSSOする主役ではない。
> - C：各アカウントに個別IAMユーザーを作るのは一元管理と真逆で、アカウント数ぶん管理が増えスケールしない。ADと繋ぐならフェデレーション（IAM Identity Center）にすべきで、個別ユーザー乱立はアンチパターン。
> - D：正解。Organizations（すべての機能）で組織を作り事業部アカウントを追加＝マルチアカウント統制の土台。
> - E：正解。IAM Identity CenterをAWS Directory Service経由で本社ADと連携＝オンプレADの資格情報で全アカウントへSSOする認証の一元化。
> **関連ノート**：[[1602_アイデンティティ系]] / [[1403_Organizations]] / [[0601_IAM基礎]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第2問（Security / IAM ｜ マルチアカウント構成とオンプレADによる認証一元化）]]

## 第3問（Security / IAM ｜ IPを変える大量リクエストへのWAFレートベース防御）
ある企業は、ALB（Application Load Balancer）の背後に配置されている Web サービスを運用しています。最近、Web サービスに対する膨大なアクセスの影響で、応答遅延や障害が発生するレベルの負荷が発生しました。調査の結果、IP アドレスを変更して送信される不審なリクエストが、Web サービスに対して大量に発生していることが確認されました。

ソリューションアーキテクトの対応として適切なものはどれですか。

- A. AWS WAF を使用し、レートベースのルールを設定する。
- B. ネットワーク ACL にルールを追加し、特定の IP アドレスからのリクエストをブロックする。
- C. Amazon Inspector を使用し、リクエストの不正な挙動を検出したら自動的にアクセスを制限する。
- D. AWS WAF を使用し、特定の国や地域からのアクセスを制限する地理的一致ルールを設定する。

> [!success]- 答え：**A**
> 「送信元 IP を変えながら大量リクエストで負荷をかける」攻撃には、AWS WAF のレートベースルールが適する。レートベースルールは送信元 IP ごとに直近5分間のリクエスト数を数え、あらかじめ決めたしきい値を超えた IP を自動でブロックする。攻撃元 IP を事前に知らなくても、叩きすぎている IP を都度自動で締め出せる。
> - A：正解。IP ごとの回数でしきい値超過を自動ブロック。IP が変わっても「大量に叩いてきた IP」をその都度ブロックできる。
> - B：ネットワーク ACL はブロックする IP を手作業で登録する静的な方式。攻撃者が IP を変えてくるとイタチごっこになり追いつけない。
> - C：Amazon Inspector は脆弱性スキャナ（EC2 やコンテナ内の古いソフト・設定ミスを点検・報告するツール）で、リクエストをリアルタイムに弾く機能はない。役割違いの最大の罠。
> - D：WAF の地理的一致（Geo match）ルールは国・地域で制御する。問題文に「特定の国から」とは書かれておらず、同一国内で IP を変えられれば素通り。正規ユーザーも巻き添えになる。
> **関連ノート**：[[1604_ネットワーク防御]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第3問（Security / IAM ｜ IPを変える大量リクエストへのWAFレートベース防御）]]

## 第4問（Compute ｜ 単一EC2のハードウェア障害からの迅速復旧（自動復旧））
あるスタートアップ企業が、AWS 上で重要なビジネスアプリケーションを開発しています。予算の制約により、このアプリケーションは単一の EC2 インスタンスで運用される予定ですが、ハードウェア障害によるサービス停止は最小限に抑えたいと考えています。

インスタンスの障害から迅速に回復する、最も効率的な方法はどれですか。

- A. Amazon EventBridge からインスタンスの復旧を行う Lambda 関数を呼び出し、障害が発生したインスタンスを復旧する。
- B. 定期的なバックアップを作成し、障害発生時にインスタンスのスナップショットから復元する。
- C. ELB を使用してインスタンスの健全性を監視し、障害が発生したインスタンスを交換する。
- D. CloudWatch アラームとインスタンス復旧アクションを設定して、インスタンスのヘルスチェックで異常があった場合に自動的に復旧する。

> [!success]- 答え：**D**
> 「単一の EC2 インスタンス」＋「ハードウェア障害からの迅速復旧」＋「最も効率的」が揃うと、EC2 の自動復旧（CloudWatch アラームの復旧アクション）が定番解。システムステータスチェックの失敗（＝ホスト側のハード障害）を検知し、同じインスタンスを別の正常なホスト上で自動的に立て直す。インスタンスID・プライベートIP・Elastic IP・アタッチ済みEBS はそのまま引き継ぐため、データ損失なく最速で戻せる。
> - A：EventBridge＋Lambda で復旧を自作する案。動作はするが、AWSが標準機能で用意している自動復旧をわざわざコードで作り込むぶん運用オーバーヘッドが増える。「最も効率的」でDに劣る。
> - B：定期バックアップ＋スナップショット復元は、復元に時間がかかり（新規作成）前回バックアップ以降のデータも失われる。「迅速に復旧」に反する。
> - C：ELBでの「交換」は裏に予備インスタンスやAuto Scalingグループ（複数台）がある前提。本問は単一インスタンスなので交換先がなく、予算制約の1台運用とも噛み合わない。最大の罠。
> - D：正解。CloudWatchアラームの復旧アクションでハード障害を検知→同一インスタンスを別ホストで自動復旧。単一インスタンスのまま、追加コストなく最速で戻せる。
> **関連ノート**：[[0101_EC2やVPCとは]] / [[1001_CloudWatch]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第4問（Compute ｜ 単一EC2のハードウェア障害からの迅速復旧（自動復旧））]]

## 第5問（Management / Monitoring ｜ ハイブリッド環境のパッチ適用と構成情報収集（SSM））
ある企業は、EC2 インスタンスとオンプレミスの Linux サーバーを併用して業務システムを運用しています。セキュリティ部門からは、すべてのサーバーにインストールされている OS の種類やバージョン、インストール済みパッケージなどの構成情報を自動的に収集し、可視化できるようにすることが求められています。また、OS のセキュリティ維持のために定期的なパッチ適用を自動化したいと考えています。最小限の管理労力でこれを実現し、コンプライアンス基準を満たすことが求められています。

最も適したソリューションはどれですか。

- A. AWS Systems Manager Patch Manager とインベントリ機能を使用して、すべての EC2 インスタンスおよびオンプレミスサーバーを対象にパッチ適用と構成情報の自動収集を行う。
- B. cron で定期実行されるシェルスクリプトを用いて、各サーバーに SSH 接続し、夜間バッチで構成情報の収集およびパッチ適用を実行する。
- C. AWS Security Hub を使用して、サーバーのインベントリ情報とパッチ適用状況を可視化し、未適用パッチがある場合はアップデートを定期実行する。
- D. 既存サーバーを Elastic Beanstalk 環境として登録し、プラットフォーム更新機能を利用してパッチ適用とインベントリ収集を行う。

> [!success]- 答え：**A**
> AWS Systems Manager は SSM Agent を入れれば **EC2 もオンプレサーバーも同じ「マネージドノード」として一元管理**できる。インベントリ機能が OS・パッケージ等の構成情報を自動収集・可視化し、Patch Manager がパッチ適用の自動化とコンプライアンス状況のレポートまで担う。要件（構成収集＋パッチ自動化＋最小の管理労力＋ハイブリッド対応）をすべてマネージドで満たす。
> - A：正解。SSMのインベントリ＝構成情報の自動収集、Patch Manager＝パッチ自動適用＋準拠レポート。ハイブリッドアクティベーションでオンプレも対象にできる。
> - B：cron＋SSHの自作スクリプトは動くが、スクリプト保守・SSH鍵管理・失敗時対応がすべて自前になり「最小限の管理労力」に反する。可視化・コンプライアンスレポートも別途作り込みが必要。
> - C：Security Hub はセキュリティ検出結果（findings）を集約・可視化するサービスで、インベントリ収集もパッチ適用の実行もできない。パッチ準拠情報はPatch Manager側が持つもの（最大の罠）。
> - D：Elastic Beanstalk はアプリを新規にデプロイする環境構築サービス。既存のEC2やオンプレサーバーを「登録」して管理する仕組みはなく、前提から成立しない。
> **関連ノート**：[[1405_Systems Manager]] / [[1401_マネジメント・ガバナンスサービス一覧]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第5問（Management / Monitoring ｜ ハイブリッド環境のパッチ適用と構成情報収集（SSM））]]

## 第6問（Security / IAM ｜ ブルー/グリーン切替中のDDoS検知（Shield Advanced保護グループ））
ある企業は、本番アプリケーションを ALB の背後にある複数の EC2 インスタンス上で稼働させています。ALB はすでに Shield Advanced で保護されており、Route 53 を使用して独自ドメインから ALB にトラフィックをルーティングしています。同社は新バージョンのアプリケーションを導入するにあたり、一定期間、現行環境と新環境を並行稼働させつつ、トラフィックを新 ALB へ切り替えるブルー/グリーンデプロイを実施する計画です。運用チームは、切り替え中もアプリケーション全体として一貫したトラフィックベースラインに基づいて、DDoS 攻撃の検知・保護が行われるようにしたいと考えています。

これらの要件を最も適切に満たすソリューションはどれですか。

- A. 現行の ALB と新 ALB を同一の Shield Advanced の保護グループにまとめ、グループの集計タイプとして Sum（合計）を設定する。
- B. 現行の ALB と新 ALB をそれぞれ別の Shield Advanced の保護グループに登録し、グループの集計タイプとして Sum（合計）を設定する。
- C. 現行の ALB と新 ALB を同一の Shield Advanced の保護グループにまとめ、グループの集計タイプとして Mean（平均）を設定する。
- D. 新たに CloudFront ディストリビューションを作成し、オリジンとして新 ALB を登録する。CloudFront ディストリビューションと新 ALB を同一の Shield Advanced の保護グループに含め、グループの集計タイプとして Max（最大）を設定する。

> [!success]- 答え：**A**
> ブルー/グリーンの切り替え中は、旧 ALB のトラフィックが減り新 ALB が増えるため、1本ずつ見るとどちらも「異常な変動」に見える。しかし2本の合計＝アプリ全体のトラフィックは一定なので、両 ALB を同一の保護グループにまとめ、集計タイプ Sum（合計）でグループ全体のベースラインを維持するのが正解。誤検知を防ぎつつ、合計の異常増（本物の DDoS）は検知できる。
> - A：正解。同一グループ＋Sum で「アプリ全体として一貫したベースライン」を実現。切り替えでトラフィックが移動しても合計は変わらない。
> - B：別々の保護グループでは各グループが個別ベースラインを持つだけで、全体で1本のベースラインにならない。切り替え中は旧（激減）も新（激増）も個別に異常扱いになりうる。
> - C：Mean（平均）は各メンバーがほぼ均等にトラフィックを受ける構成向け。ブルー/グリーンでは配分が 100:0→0:100 と偏っていくため均等前提が崩れる。最大の罠。
> - D：CloudFront の新設は要件にないアーキテクチャ変更。さらに Max（最大）は最多メンバーの値でグループを代表させる方式で、全体の合計ベースラインにはならない。
> **関連ノート**：[[1604_ネットワーク防御]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第6問（Security / IAM ｜ ブルー/グリーン切替中のDDoS検知（Shield Advanced保護グループ））]]

## 第7問（Cost Optimization ｜ 大量データ配布の転送コスト最小化（既存Direct Connect活用））
ある会社は、EC2 インスタンスで稼働する勤怠管理アプリケーションを構築しました。同社は既存の AWS Direct Connect 回線を利用しています。勤怠管理アプリケーションは、毎月1か月分の勤怠実績を集計し、全社員分の勤怠データを作成します。作成された勤怠データは暗号化して社員に配布します。データは1件あたり平均10MBで、配布する勤怠データの月合計は約500GB以上になります。

データ転送コストを削減するために、最も適切なソリューションはどれですか。

- A. 勤怠管理アプリケーションから Direct Connect を経由して、オンプレミスのファイルサーバーへ勤怠データをアップロードする。社員はファイルサーバーからダウンロードする。
- B. 社員はインターネット経由で勤怠管理アプリケーションに HTTPS 接続して勤怠データをダウンロードする。
- C. 勤怠管理アプリケーションから S3 バケットへサーバー側の暗号化を利用してデータをアップロードする。社員はインターネット経由で S3 バケットから勤怠データをダウンロードする。
- D. 勤怠管理アプリケーションから SFTP 接続で NAT ゲートウェイを経由して、オンプレミスのファイルサーバーへ勤怠データをアップロードする。社員はファイルサーバーからダウンロードする。

> [!success]- 答え：**A**
> 要件は純粋に「毎月500GB超の配布の転送コストを最小化」。鍵はデータ転送アウトの単価で、**Direct Connect経由（約0.041 USD/GB）はインターネット経由（約0.114 USD/GB）より安い**。同社は既にDirect Connectを契約済みなので、それを使ってオンプレのファイルサーバーへ配る構成が最も転送コストを抑えられる。
> - A：正解。既存のDirect Connectのデータ転送アウトはインターネットより単価が安く、追加契約も不要。大量配布の転送コストを最小化できる。
> - B：EC2からインターネットへの下り転送（約0.114 USD/GB）は割高。DC経由より高くつく。
> - C：S3からのインターネット配信もアウト単価はインターネット料金（約0.114 USD/GB）で、DC経由（0.041 USD/GB）より高い。SSEで暗号化はできるが「転送コスト最小」ではDCに負ける。
> - D：NATゲートウェイは処理データ量（GB）課金（約0.062 USD/GB）が上乗せされ、500GB超では最もコストがかさむ。既存DCがあるのにNATを挟む必然性もない。
> **関連ノート**：[[0411_データ転送料金]] / [[0405_ネットワークとコンテンツ配信]] / [[0409_IGW・NAT Gateway・VPCエンドポイント]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第7問（Cost Optimization ｜ 大量データ配布の転送コスト最小化（既存Direct Connect活用））]]

## 第8問（Management / Monitoring ｜ 承認済み環境のセルフサービス払い出し（Service Catalog））
ある会社は AWS 上で新規サービス用のシステムを構築しています。システムには複数のアプリケーションがあり、アプリケーションごとに開発グループが存在します。システム管理者は、構築している AWS 環境と同一の環境を、各アプリケーション開発者がセルフサービスで開発・テスト用に構築できるようにしたいと考えています。システム管理者は、共有する AWS 環境のバージョンや使用履歴を一元管理する必要があります。

要件を満たすために、AWS のどの機能を利用すればよいですか。

- A. AWS Service Catalog の製品
- B. AWS CloudFormation のテンプレート
- C. AWS CloudTrail の証跡
- D. Amazon CloudWatch のメトリクス

> [!success]- 答え：**A**
> 決め手は「各開発者がセルフサービスで構築」＋「バージョン・使用履歴を一元管理」。承認済みの構成を「製品」としてカタログ化し、利用者にセルフサービスで払い出しつつ、バージョン・権限・利用履歴を統制できるのが AWS Service Catalog。中身は CloudFormation テンプレートだが、その上にガバナンスの層をかぶせる。
> - A：正解。承認済み環境をカタログ化し、開発者がセルフサービスで払い出し、管理者がバージョン・利用履歴を一元管理できる。要件に完全一致。
> - B：CloudFormation テンプレートは環境を作る設計図そのもの。単体では「開発者向けのセルフサービス窓口」も「バージョン・使用履歴の一元管理」も無く、要件の後半を満たせない。Service Catalog の部品でしかない（最大の罠）。
> - C：CloudTrail の証跡は「誰がどのAPIを叩いたか」の監査ログ。環境の払い出し機能ではなく用途違い。
> - D：CloudWatch のメトリクスはリソースの監視・数値計測。払い出しやバージョン管理とは無関係。
> **関連ノート**：[[1401_マネジメント・ガバナンスサービス一覧]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第8問（Management / Monitoring ｜ 承認済み環境のセルフサービス払い出し（Service Catalog））]]

## 第9問（Serverless ｜ 高頻度・短時間ワークフローのコスト最適化（Step Functions Express））
ある企業は、数十万台規模の IoT デバイスを管理・制御するプラットフォームを AWS 上で運用しています。これらのデバイスは温度・湿度・バッテリー残量などのセンサー計測値やイベント情報を大量に送信しており、ピーク時には1秒間に10万回以上の呼び出しが発生する見込みです。現在、受信したデータをワークフローとして自動化しており、実行時間は5分以内です。しかし、大量のリクエストに伴う実行コストが問題になり始めており、できるだけコストを抑えて同時実行数をさばきたいと考えています。

このように高いイベントレートが想定され、大規模に拡張しつつもコスト効率を重視する場合、どの選択肢が最も適切ですか。

- A. 標準ワークフローで Step Functions を使用する。
- B. 標準ワークフローと Express ワークフローを組み合わせて Step Functions を使用する。
- C. Express ワークフローで Step Functions を使用する。
- D. 予約済み同時実行を設定した複数の Lambda 関数を使用する。

> [!success]- 答え：**C**
> 決め手は「1秒10万回超の高頻度」「実行5分以内」「コスト効率重視」。Step Functions の Express ワークフローは高頻度・短時間（最大5分）のイベント処理向けで、状態遷移ごとではなく実行回数・時間・メモリで課金されるため、大量イベントでも安い。3条件すべてに一致する。
> - A：標準ワークフローは状態遷移1回ごとに課金。1秒10万回のような高頻度では遷移が膨大になりコストが跳ね上がる。長時間・厳密な監査向けで、コスト重視の要件と逆。
> - B：標準の中にExpressを入れ子にする構成は可能だが、標準ぶんの状態遷移課金が乗るためコスト効率最優先ではExpress単体に負ける。わざわざ高い標準を混ぜる過剰構成。
> - C：正解。Expressは高頻度・短時間・大量向けで、状態遷移ごと課金でないため大量イベントでも安い。5分以内という条件もExpressの上限に一致。
> - D：Lambdaの予約済み同時実行は同時実行枠を確保・制限する設定でワークフローの自動化ではない。要件は「ワークフローとして自動化」が前提で、予約はコスト削減手段でもない。
> **関連ノート**：[[0807_Step_Functions]] / [[0708_Lambda]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第9問（Serverless ｜ 高頻度・短時間ワークフローのコスト最適化（Step Functions Express））]]

## 第10問（Security / IAM ｜ 組織横断のSGルール統一と監査自動化（Firewall Manager））
ある企業は、Organizations を用いて数十の AWS アカウントを一元管理しており、大規模な企業システムを複数リージョンで EC2 上に運用しています。長年の運用で各アカウントのセキュリティグループの設定にばらつきが生じ、重複や不要なルールも見られるなどの問題を抱えるようになりました。セキュリティ本部は、組織全体で共通のセキュリティグループルールを統一的に適用し、ルールの監査と未使用・冗長ルールの検出を自動化できるソリューションを求めています。

運用コストを抑えつつ、これらの要件を満たすにはどのアプローチが最も適切ですか。

- A. Control Tower のガードレールを使用して、複数アカウントおよびリージョンにわたる VPC セキュリティグループ構成の準拠性を強制し、共通のポリシーを適用する。
- B. Security Hub を有効化し、すべてのアカウントおよびリージョンでの VPC セキュリティグループの構成を一元的に監視・管理する。
- C. Firewall Manager のセキュリティグループポリシーで共通のセキュリティルールを定義し、すべてのアカウントおよびリージョンに適用する。
- D. Network Firewall のファイアウォールポリシーで共通のトラフィック制御ルールを定義し、すべての VPC とアカウントに適用する。

> [!success]- 答え：**C**
> 決め手は「セキュリティグループのルールそのものを組織全体に統一適用」＋「未使用・冗長ルールの監査を自動化」。Firewall Manager のセキュリティグループポリシーは、Organizations 配下の全アカウント・全リージョンへ共通SGルールを強制適用でき、冗長・未使用ルールの監査も自動化できる。要件に完全一致する。
> - A：Control Tower のガードレールはアカウントの準拠性を保つ大枠の統制（ログ有効化・公開禁止など）。SGの個別ルールを定義して配布し、冗長ルールを検出する粒度の道具ではない。統制の層が違う。
> - B：Security Hub は各サービスの検出結果を集約・可視化する掲示板。「一元的に監視」はできても「ルールを統一的に適用」ができず、要件の核（適用）を満たせない。最大の罠（"一元管理"の響きに釣られる）。
> - C：正解。Firewall Manager のSGポリシーで共通ルールを定義し全アカウント・全リージョンへ適用。監査自動化も含めて要件を満たす。
> - D：Network Firewall はSGとは別レイヤーのファイアウォール本体。トラフィック制御ルールは作れるが、要件は「セキュリティグループのルールを統一」で対象が違う。
> **関連ノート**：[[1604_ネットワーク防御]] / [[0408_セキュリティグループとNACL]] / [[1403_Organizations]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第10問（Security / IAM ｜ 組織横断のSGルール統一と監査自動化（Firewall Manager））]]

## 第10問（Database ｜ 生成AI向けベクトル検索をコスト効率よく構築（RDS PostgreSQL＋pgvector））
ある企業は、生成 AI を活用した検索機能を自社の Web アプリケーションに統合しようとしています。この機能では、ユーザーからの問い合わせ内容と過去の FAQ データをベクトル形式で比較し、最も類似した回答を検索する必要があります。そのため企業はベクトル検索をサポートし、コスト効率の高い方法で構築可能なデータベースソリューションを探しています。

これらの要件を最もコスト効率よく満たすソリューションはどれですか。

- A. ElastiCache を使用し、FAQ データをインメモリで管理してベクトル検索を実装する。
- B. OpenSearch Service を使用し、FAQ データのベクトルインデックスを管理するクラスターを構築する。
- C. RDS for PostgreSQL で pgvector 拡張を有効化し、FAQ ベクトルデータをベクトルテーブルに保存して検索機能を実装する。
- D. S3 バケットに FAQ データのベクトルファイルを保存し、Athena で検索クエリを実行する。

> [!success]- 答え：**C**
> 決め手は「ベクトル検索をサポート」かつ「最もコスト効率よく」。RDS for PostgreSQL は pgvector という拡張機能を有効化するだけでベクトル型の保存と類似度検索ができるようになる。既存のマネージド RDB に拡張を1つ足すだけなので、専用のベクトル検索基盤を新設せずに済み、小〜中規模の FAQ 検索では最も安価。
> - A：ElastiCache（Redis/Valkey）はインメモリのキャッシュで、全 FAQ をメモリに常駐させるとデータ量に比例してメモリ課金が高くつく。永続的なベクトル検索基盤としてはコスト効率が悪い。
> - B：OpenSearch Service はベクトル検索に対応するが、常時稼働の専用クラスターを立てる必要があり、大規模・低レイテンシ用途向けでコストが高い。FAQ 程度の規模には過剰（最大の罠）。
> - C：正解。既存の RDS PostgreSQL に pgvector 拡張を足すだけで、追加基盤なくベクトル類似検索を実装でき、最もコスト効率が良い。
> - D：S3＋Athena はスキャン量課金のクエリサービスで、そもそもベクトルの類似度検索（近傍検索）を行う仕組みがない。用途違いで要件を満たせない。
> **関連ノート**：[[0301_RDS]] / [[1110_Amazon OpenSearch Service]] / [[0304_ElastiCache]]
> **詳しい解説**：[[509_【解説】外部サイト模擬試験9#第10問（Database ｜ 生成AI向けベクトル検索をコスト効率よく構築（RDS PostgreSQL＋pgvector））]]
