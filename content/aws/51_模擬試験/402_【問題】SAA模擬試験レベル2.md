# 【問題】SAA模擬試験レベル2（公式サンプル寄せ改訂版・全10問）

## この問題集の目的

ネットワーク、認証、DB接続、疎結合、データ移行を中心に、公式模擬問題に近い条件の読み分けを練習する。

> [!info] 出題形式
> - シナリオ段落と設問文を分け、です・ます調で記述。
> - 選択肢はサービス名だけでなく、具体的な操作・構成で記述。
> - 複数選択問題は5択（A〜E）、単一選択問題は4択（A〜D）。
> - 旧402とは別問題。解き直し用の大幅改訂版。

## 第1問（Database ｜ LambdaからRDSへの接続急増）
ある企業は、Amazon API GatewayとAWS Lambdaで注文APIを構築しています。注文データはAmazon RDS for PostgreSQLに保存しています。セール時にLambdaの同時実行が急増すると、RDSの接続数が上限に達し、注文APIでエラーが発生します。アプリケーションコードの変更を最小限にし、データベース接続を効率的に再利用したいと考えています。

この要件を満たすソリューションはどれですか。

- A. RDSをマルチAZ配置に変更し、スタンバイDBインスタンスへ読み書きを分散する。
- B. Amazon RDS Proxyを導入し、Lambda関数からRDS Proxyエンドポイントへ接続する。
- C. API Gatewayのキャッシュを有効化し、注文作成リクエストをキャッシュする。
- D. Lambda関数のタイムアウトを最大値に延長し、DB接続が空くまで待機させる。

> [!success]- 答え：**B**
> RDS ProxyはアプリケーションとRDSの間で接続をプールし、Lambdaの急増時にDB接続数の消費を抑えられる。
> - A：Multi-AZは可用性向上であり、通常時の書き込み分散や接続プールにはならない。
> - B：正解。接続の再利用とフェイルオーバー時の接続管理を低変更で導入できる。
> - C：注文作成のような書き込みリクエストをキャッシュすると整合性を損なう。
> - D：同時実行枠の占有が長くなり、接続集中を悪化させる可能性がある。
> **関連ノート**：[[0302_RDSProxy]] / [[0708_Lambda]]

## 第2問（Security / IAM ｜ 外部委託先のクロスアカウント監査）
ある企業は、監査ログを保存するS3バケットを外部監査会社に読み取らせたいと考えています。監査会社は複数の顧客を担当しており、自社のAWSアカウントからアクセスします。企業は長期アクセスキーの配布を避け、監査会社が別顧客の依頼と混同してロールを引き受けるリスクも下げたいと考えています。

この要件を満たす最も適切なソリューションはどれですか。

- A. 企業アカウントにIAMユーザーを作成し、アクセスキーを監査会社へ渡す。
- B. S3バケット内の監査ログをパブリック読み取り可能にし、監査会社にURLを共有する。
- C. 企業アカウントにIAMロールを作成し、監査会社のAWSアカウントを信頼する。外部IDを条件に含め、S3読み取りの最小権限を付与する。
- D. 監査ログを監査会社のS3バケットへ毎日コピーし、コピー後のアクセス管理を監査会社に任せる。

> [!success]- 答え：**C**
> クロスアカウントアクセスではIAMロールとSTSを使う。外部IDを条件に含めると、委託先が複数顧客を扱う場合の混同代理問題を抑えられる。
> - A：長期アクセスキーの配布になり、要件に反する。
> - B：監査ログの公開は最小権限と機密性の要件に反する。
> - C：正解。一時認証情報、最小権限、外部IDによる安全な委託先アクセスを実現できる。
> - D：コピー運用が増え、企業側でアクセス範囲を統制しにくい。
> **関連ノート**：[[0602_IAMロール詳細]] / [[0208_S3バケットポリシー]]

## 第3問（Networking / CDN / LB ｜ オンプレミスからVPC内DNSを解決）
ある企業は、オンプレミスの業務システムからAWS上の内部アプリケーションへ接続しています。内部アプリケーションの名前はRoute 53プライベートホストゾーンで管理され、VPC内では解決できます。オンプレミスのDNSサーバーからも同じ内部ドメイン名を解決できるようにしたいと考えています。オンプレミスとVPCの間には既にVPN接続があります。

これらの要件を満たすアクションの組み合わせはどれですか。（2つ選択）

- A. 対象VPCに関連付けたRoute 53プライベートホストゾーンに、内部アプリケーションのレコードを作成する。
- B. Route 53 ResolverインバウンドエンドポイントをVPCに作成する。
- C. Route 53 Resolverアウトバウンドエンドポイントを作成し、VPCからオンプレミスDNSへすべての名前解決を転送する。
- D. オンプレミスDNSサーバーに条件付きフォワーダーを設定し、内部ドメインの問い合わせをResolverインバウンドエンドポイントへ転送する。
- E. パブリックホストゾーンに内部アプリケーションのプライベートIPを登録する。

> [!success]- 答え：**B, D**
> オンプレミスからVPC内のプライベートDNSを解決するには、VPC側にResolverインバウンドエンドポイントを置き、オンプレミスDNSから条件付き転送する。
> - A：プライベートホストゾーンのレコードは前提として必要だが、オンプレミスから到達する経路を作るアクションではない。
> - B：正解。オンプレミスDNSからの問い合わせをVPC内Resolverへ受けられる。
> - C：VPCからオンプレミス名を解決する方向の機能で、今回の方向とは逆。
> - D：正解。オンプレミス側の問い合わせを適切なエンドポイントへ転送できる。
> - E：内部名を公開DNSに出すのはセキュリティ上不適切で、プライベートホストゾーンの要件とも異なる。
> **関連ノート**：[[0404_Route53]] / [[0406_VPC応用機能]]

## 第4問（Database ｜ DynamoDBの未処理データ検索）
あるECサイトは、注文データをAmazon DynamoDBに保存しています。アプリケーションは注文IDをパーティションキーとして注文を取得します。新たに、出荷担当者が「未出荷」の注文だけを高頻度に一覧表示する機能を追加します。テーブル全体のScanは避け、読み取りコストとレイテンシを抑えたいと考えています。

この要件を満たす最も適切な設計はどれですか。

- A. 未出荷注文だけに設定される属性をキーにしたグローバルセカンダリインデックスを作成し、そのインデックスをQueryする。
- B. 注文テーブルを毎回Scanし、アプリケーション側で未出荷注文だけをフィルタする。
- C. DynamoDB Accelerator（DAX）を導入し、全テーブルScanの結果を永続的に保存する。
- D. すべての注文データを毎分Amazon Redshiftへエクスポートし、未出荷注文をSQLで検索する。

> [!success]- 答え：**A**
> DynamoDBで特定条件の項目を効率的に検索するには、そのアクセスパターンに合うキー設計やGSIを用意する。未出荷項目だけに属性を持たせると疎なインデックスとして扱える。
> - A：正解。Queryで必要な項目だけを効率的に取得できる。
> - B：データ量が増えるほどコストとレイテンシが悪化する。
> - C：DAXは読み取りキャッシュであり、Scanを効率的なQueryに変えるものではない。
> - D：分析用途ならあり得るが、運用画面の高頻度検索には重すぎる。
> **関連ノート**：[[0303_DynamoDB]]

## 第5問（Application Integration ｜ 顧客単位の順序保証）
あるECサイトでは、顧客ごとに注文イベントを発生順に処理する必要があります。処理ワーカーは複数台にスケールしますが、同じ顧客のイベントだけは順序を崩せません。異なる顧客のイベントは並列に処理できます。重複処理もできるだけ抑えたいと考えています。

この要件を満たす最も適切なソリューションはどれですか。

- A. Amazon SNS標準トピックからすべてのワーカーへイベントを配信する。
- B. Amazon SQS標準キューを使用し、ワーカー側で受信順に処理されることを前提にする。
- C. Amazon SQS FIFOキューを使用し、顧客IDをMessageGroupId、イベントIDを重複排除IDとして送信する。
- D. Amazon EventBridgeイベントバスへ送信し、複数のルールで同じ顧客イベントを複数ワーカーへ配信する。

> [!success]- 答え：**C**
> SQS FIFOキューはメッセージグループ内の順序を保持する。顧客IDをグループにすると、同じ顧客は順序維持、異なる顧客は並列処理しやすい。
> - A：ファンアウト向けで、顧客単位の順序制御と単一処理には向かない。
> - B：標準キューは高スループットだが、厳密な順序保証には向かない。
> - C：正解。顧客単位の順序と重複抑制を両立できる。
> - D：複数配信は重複処理を招きやすく、順序保証にもならない。
> **関連ノート**：[[0802_SQS]]

## 第6問（Security / IAM ｜ モバイルアプリのユーザー認証とAWS一時認証）
ある企業は、一般消費者向けのモバイルアプリを提供します。利用者はメールアドレスやソーシャルIDプロバイダーでサインインできる必要があります。サインイン後、アプリはユーザーごとに許可された範囲でAWSリソースへアクセスします。企業は独自の認証基盤や長期アクセスキーの埋め込みを避けたいと考えています。

この要件を満たす最も適切な構成はどれですか。

- A. IAMユーザーを利用者ごとに作成し、アクセスキーをモバイルアプリ内に保存する。
- B. Amazon Cognitoユーザープールでサインインを管理し、IDプールで一時的なAWS認証情報を払い出す。
- C. AWS IAM Identity Centerで一般消費者を管理し、各ユーザーをAWSアカウントへ招待する。
- D. API GatewayのAPIキーをアプリに埋め込み、ユーザー認証の代わりにする。

> [!success]- 答え：**B**
> Cognitoはアプリ利用者の認証や外部IDプロバイダー連携に向く。IDプールを使うと、認証済みユーザーへ一時的なAWS認証情報を発行できる。
> - A：長期アクセスキーの埋め込みは危険で、ユーザーごとの管理にも向かない。
> - B：正解。ユーザー認証と一時認証情報の払い出しをマネージドに実現できる。
> - C：従業員などのワークフォースアクセス管理向けで、一般消費者アプリの認証基盤ではない。
> - D：APIキーは利用量管理などに使うもので、ユーザー認証・認可の代替ではない。
> **関連ノート**：[[0810_Cognito]] / [[0602_IAMロール詳細]]

## 第7問（Migration / Transfer ｜ 初回大容量移行と継続同期）
ある企業は、オンプレミスのファイルサーバーに保存された400TBのデータをAmazon S3へ移行します。既存のインターネット回線だけで初回全量を転送すると、業務期限に間に合いません。初回移行後は、オンプレミス側で追加・変更されるファイルを定期的に差分同期したいと考えています。

これらの要件を満たすアクションの組み合わせはどれですか。（2つ選択）

- A. 初回の大容量データをAWS Snowball EdgeデバイスでAWSへ転送する。
- B. 初回移行後の差分同期にAWS DataSyncを使用する。
- C. 初回移行と差分同期の両方にAmazon SQSを使用し、ファイルをメッセージ本文として送信する。
- D. Amazon Route 53 Resolverでオンプレミスファイルサーバー名をS3バケット名へ切り替える。
- E. S3 Transfer Accelerationだけを使用し、既存回線で400TBをアップロードする。

> [!success]- 答え：**A, B**
> 回線だけでは時間がかかる初回大容量移行にはSnowball Edgeが適する。移行後の差分同期にはDataSyncが適している。
> - A：正解。物理デバイスで大容量データを効率よく移行できる。
> - B：正解。オンプレミスとS3間の差分同期をマネージドに実行できる。
> - C：SQSはメッセージキューであり、ファイル移行サービスではない。
> - D：DNS切り替えではデータ移行や差分同期は行えない。
> - E：高速化できる場合はあるが、回線制約が大きい初回400TB移行では物理搬送が現実的。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[1502_DataSync]]

## 第8問（Networking / CDN / LB ｜ SaaSのプライベート公開）
あるSaaS企業は、自社VPC内のサービスを複数の顧客AWSアカウントから利用できるようにしたいと考えています。サービスはインターネットへ公開せず、顧客VPCと自社VPCのCIDR重複やルートテーブル調整の影響を避けたい方針です。顧客側からはプライベートIPでサービスへ接続できる必要があります。

この要件を満たす最も適切なソリューションはどれですか。

- A. 顧客VPCごとにVPCピアリングを作成し、CIDRが重複する場合は顧客に変更してもらう。
- B. サービスをパブリックALBで公開し、セキュリティグループで顧客の送信元IPだけを許可する。
- C. AWS PrivateLinkを使用し、提供側でエンドポイントサービスを公開し、顧客側でインターフェイスVPCエンドポイントを作成する。
- D. 顧客ごとにSite-to-Site VPNを構成し、双方のVPCルートをすべて交換する。

> [!success]- 答え：**C**
> PrivateLinkはVPC同士を広く接続せず、サービス単位でプライベート接続を提供できる。CIDR重複やルート管理の影響も抑えられる。
> - A：VPCピアリングはCIDR重複に弱く、顧客ごとのルート管理も増える。
> - B：送信元制限はできるが、サービス自体はインターネット公開になる。
> - C：正解。インターネット非公開、CIDR重複の影響軽減、顧客ごとのプライベート接続を満たす。
> - D：VPN運用が重く、VPC間の広いルーティング関係を作るため要件に合いにくい。
> **関連ノート**：[[0406_VPC応用機能]] / [[0409_IGW・NAT Gateway・VPCエンドポイント]]

## 第9問（Storage ｜ Windowsファイル共有のマネージド移行）
ある企業は、オンプレミスのWindowsファイルサーバーで部門共有フォルダを運用しています。Active Directoryと統合したSMBアクセス、Windows ACL、ユーザーごとの権限管理を維持したままAWSへ移行したいと考えています。ファイルサーバーのOS管理はできるだけ避けたい方針です。

この要件を満たす最も適切なサービスはどれですか。

- A. Amazon EFSを作成し、WindowsクライアントからSMBで直接マウントする。
- B. Amazon S3を静的ウェブサイトホスティングとして公開し、共有フォルダの代替にする。
- C. Amazon FSx for Windows File Serverを使用し、Active Directoryと統合してSMB共有を提供する。
- D. Amazon EBSボリュームを作成し、複数のWindowsクライアントから同時に直接マウントする。

> [!success]- 答え：**C**
> FSx for Windows File Serverは、SMB、AD統合、Windows ACLを必要とするWindowsファイル共有に適している。
> - A：EFSはNFSベースで、Windows SMB共有とACL要件にはFSx for Windowsが適する。
> - B：S3はオブジェクトストレージであり、Windows共有フォルダの権限モデルをそのまま提供しない。
> - C：正解。Windowsファイルサーバー移行の要件に合う。
> - D：EBSはEC2向けブロックストレージであり、複数クライアント向けの共有ファイルサービスではない。
> **関連ノート**：[[0213_FSxファミリー4種の使い分け]]

## 第10問（Security / IAM ｜ RDS認証情報の自動ローテーション）
ある企業は、Amazon EC2上のアプリケーションからAmazon RDS for PostgreSQLへ接続しています。現在、DBユーザー名とパスワードはアプリケーション設定ファイルに保存されています。セキュリティチームは、認証情報を設定ファイルから分離し、定期的に自動ローテーションし、アプリケーションには実行時に安全に取得させたいと考えています。

これらの要件を満たすアクションの組み合わせはどれですか。（2つ選択）

- A. AWS Secrets ManagerにRDS認証情報を保存し、自動ローテーションを有効化する。
- B. EC2インスタンスのIAMロールに、対象シークレットを取得する最小限の権限を付与する。
- C. Systems Manager Parameter StoreのSecureStringに認証情報を保存し、自動ローテーションは行わない。
- D. 認証情報をEC2ユーザーデータに記載し、起動時に環境変数へ設定する。
- E. RDSスナップショットを毎日作成し、スナップショット取得時にDBパスワードを変更する。

> [!success]- 答え：**A, B**
> Secrets ManagerはRDS認証情報の安全な保管と自動ローテーションに対応している。EC2にはIAMロールで必要な取得権限だけを付与する。
> - A：正解。認証情報の保管と自動ローテーションをマネージドに実現できる。
> - B：正解。アプリケーションから安全に取得しつつ、最小権限にできる。
> - C：SecureStringは暗号化保管には使えるが、RDS認証情報の自動ローテーション要件ではSecrets Managerが適する。
> - D：固定的な保存になり、漏えいやローテーションに弱い。
> - E：バックアップ操作であり、認証情報の安全な保管・取得・ローテーションとは関係しない。
> **関連ノート**：[[1605_データ保護・暗号化]] / [[0602_IAMロール詳細]]
