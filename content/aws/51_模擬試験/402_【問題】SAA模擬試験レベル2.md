# 【問題】SAA模擬試験レベル ②（4択・全20問）

## この問題集の目的

[[401_【問題】SAA模擬試験レベル]]の続編。実際のSAA（SAA-C03）に近い、複数サービスを組み合わせた設計判断に慣れる。1問目では扱わなかった領域（マルチアカウント統制・ハイブリッド/グローバル接続・サーバーレス連携・セキュリティ運用・コスト最適化）を中心に出題する。

## 対象レベル

SAA本番レベル。ただし初学者の復習に耐えるよう解説は厚めにしている。1問あたり数分かけて要件を読み解く練習に使う。

## 問題数

20問（マルチアカウント基盤・ハイブリッドDNS・多VPC接続・プライベート接続・グローバル高速化・クロスリージョンDB・グローバルKVS・改ざん防止/WORM・HPC共有ストレージ・非同期信頼性・ワークフロー・認証付きAPI・大量DB接続・機密データ検出・脅威自動対応・脆弱性管理・静的サイト保護・コスト/アクセス不明・耐障害コンピュート・ハイブリッド階層ストレージ を横断）

## 使い方

要件文から「可用性／コスト／セキュリティ／性能／運用負荷」のどれが問われているかを最初に判定する。本問は**アプリ採点の都合上すべて単一正解（4択1つ）**で作成しているが、本番には「2つ選べ」も出る点は別途意識する。回答後は全選択肢の当落理由を読む。

## 第1問（マルチアカウント基盤）
急成長企業が、複数の新規AWSアカウントを、ガードレールとログ集約・SSOを備えたベストプラクティス構成（ランディングゾーン）で素早く払い出せるようにしたい。最適な方法はどれか。

- A. AWS Control Tower（Organizations＋SCP＋一元ログ/SSO）でランディングゾーンを構築
- B. アカウントごとに手作業でIAM・ログ・SSOを個別設定
- C. 単一アカウントに全部入れてIAMで分離
- D. 各アカウントでCloudFormationを手動実行するだけ

> [!success]- 答え：**A**（Control Tower）
> Control TowerはOrganizations・SCPによるガードレール・一元的なログ集約・IAM Identity Center（SSO）を含むランディングゾーンを自動構築し、新規アカウントの標準化された払い出し（Account Factory）を提供する。
> **他の選択肢が違う理由**
> - B：手作業の個別設定は漏れ・運用負荷が大きく標準化されない。
> - C：単一アカウント集約は分離境界・ブラスト半径の観点で不適切。
> - D：CloudFormation手動だけでは統制（SCP/ログ/SSO）の枠組みにならない。
> **覚えるべきキーワード**：Control Tower / ランディングゾーン / ガードレール / Account Factory
> **関連サービス**：Organizations / SCP / IAM Identity Center / CloudTrail
> **SAAでの出題イメージ**：「標準化されたマルチアカウント基盤を素早く」=Control Tower。
> 📎 参照：[[1404_Control Tower]] / [[1403_Organizations]]

## 第2問（ハイブリッドDNS）
オンプレとAWSをDirect Connectで接続済み。オンプレからAWS（Route 53プライベートホストゾーン）の名前解決と、AWSからオンプレDNSの名前解決を双方向で行いたい。最適な構成はどれか。

- A. Route 53 Resolver のインバウンド/アウトバウンドエンドポイント
- B. パブリックホストゾーンを使う
- C. 各EC2のhostsファイルに手書きする
- D. NAT Gatewayで名前解決する

> [!success]- 答え：**A**（Route 53 Resolver エンドポイント）
> Route 53 Resolverのインバウンドエンドポイントでオンプレ→AWSの名前解決を受け、アウトバウンドエンドポイント＋転送ルールでAWS→オンプレの名前解決を転送する。ハイブリッドDNSの定番。
> **他の選択肢が違う理由**
> - B：パブリックホストゾーンは内部の双方向ハイブリッド解決の仕組みではない。
> - C：hosts手書きは規模・保守の面で破綻する。
> - D：NAT Gatewayは通信経路でDNS解決の仕組みではない。
> **覚えるべきキーワード**：Route 53 Resolver / インバウンド/アウトバウンドEP / 転送ルール
> **関連サービス**：プライベートホストゾーン / Direct Connect / VPN
> **SAAでの出題イメージ**：「オンプレ⇄AWSの双方向名前解決」=Route 53 Resolverエンドポイント。
> 📎 参照：[[0404_Route53]] / [[0406_VPC応用機能]]

## 第3問（多VPC/オンプレ集約）
複数リージョン・多数のVPCとオンプレ拠点が増え、相互接続が複雑化した。中央でルーティングを集約し、将来の拡張も容易にしたい。最適なのはどれか。

- A. AWS Transit Gateway（必要に応じTGWピアリングでリージョン間接続）
- B. 全VPC間にVPCピアリングをフルメッシュ
- C. 各VPCにインターネットゲートウェイ
- D. 各VPCを個別にDirect Connectで接続

> [!success]- 答え：**A**（Transit Gateway）
> Transit Gatewayは多数のVPC・VPN・DXを中央ハブで接続し、リージョン間はTGWピアリングで拡張できる。ピアリングのフルメッシュの管理爆発を避けつつ集約・拡張に強い。
> **他の選択肢が違う理由**
> - B：フルメッシュのピアリングは接続数が爆発し管理破綻、推移的ルーティングもできない。
> - C：IGWはインターネット出入口でVPC相互接続の集約ではない。
> - D：VPC個別DXは非効率で、集約・一元管理にならない。
> **覚えるべきキーワード**：Transit Gateway / ハブ＆スポーク / TGWピアリング / 集約
> **関連サービス**：VPCピアリング / Direct Connect Gateway / VPN
> **SAAでの出題イメージ**：「多数VPC/拠点/リージョンを集約接続」=Transit Gateway。
> 📎 参照：[[0406_VPC応用機能]]

## 第4問（プライベート接続・自社サービス公開）
自社が運用するアプリを、他のVPC（または顧客）にインターネットを経由させずプライベートに公開・提供したい。最適な仕組みはどれか。

- A. AWS PrivateLink（エンドポイントサービス＋インターフェース型エンドポイント）
- B. VPCピアリングを顧客全員と張る
- C. パブリックALBで公開しIPで制限
- D. NAT Gateway経由で公開

> [!success]- 答え：**A**（PrivateLink）
> PrivateLinkは、NLBの背後のサービスを「エンドポイントサービス」として公開し、利用側はインターフェース型エンドポイント（ENI）で非インターネットに接続する。VPC同士のIP重複も気にせずサービス単位で安全に公開できる。
> **他の選択肢が違う理由**
> - B：多数の顧客とのピアリングは管理が破綻し、CIDR重複にも弱い。
> - C：パブリック公開＋IP制限はインターネット経由で要件（非経由）に反する。
> - D：NAT Gatewayはサービス公開の仕組みではない。
> **覚えるべきキーワード**：PrivateLink / エンドポイントサービス / NLB背後 / サービス単位公開
> **関連サービス**：インターフェース型VPCエンドポイント / NLB / Transit Gateway
> **SAAでの出題イメージ**：「自社サービスをプライベートに公開」=PrivateLink。
> 📎 参照：[[0406_VPC応用機能]]

## 第5問（グローバル高速化・固定IP）
世界中のユーザーが使うTCPベースのアプリ（キャッシュ不可）で、AWS網経由の低遅延化と、ファイアウォール許可用に変わらない固定IP（2つのAnycast IP）が欲しい。最適なのはどれか。

- A. AWS Global Accelerator
- B. Amazon CloudFront
- C. Amazon Route 53 レイテンシルーティングのみ
- D. 単一リージョンにNLBを1台

> [!success]- 答え：**A**（Global Accelerator）
> Global Acceleratorはエッジから最寄りリージョンへAWSバックボーン経由で誘導し、非キャッシュ・TCP/UDPでも低遅延・安定化する。2つの固定Anycast IPを提供し、許可リスト運用にも向く。
> **他の選択肢が違う理由**
> - B：CloudFrontはキャッシュCDNで、キャッシュできない動的TCPアプリの主役ではない。
> - C：Route 53は名前解決の誘導で、経路高速化や固定IPは提供しない。
> - D：単一NLBではグローバル低遅延・リージョン障害回避を満たせない。
> **覚えるべきキーワード**：Global Accelerator / 固定Anycast IP / 非キャッシュ高速化 / フェイルオーバー
> **関連サービス**：NLB / CloudFront / Route 53
> **SAAでの出題イメージ**：「動的TCPをグローバル高速化＋固定IP」=Global Accelerator。
> 📎 参照：[[0405_ネットワークとコンテンツ配信]]

## 第6問（クロスリージョンDB／DR）
グローバルなRDBアプリで、リージョン障害時に短いRTO/RPOで別リージョンへ切り替え、平常時は各地域での低遅延読み取りも実現したい。最適なのはどれか。

- A. Aurora Global Database
- B. 単一リージョンのRDS Multi-AZ
- C. リードレプリカを同一リージョンに追加
- D. EC2上の自前MySQLをcronでバックアップ

> [!success]- 答え：**A**（Aurora Global Database）
> Aurora Global Databaseは1つのプライマリリージョンから複数のセカンダリリージョンへ低遅延レプリケーションし、各地域の低遅延読み取りと、リージョン障害時の高速昇格（短いRTO/RPO）を両立する。
> **他の選択肢が違う理由**
> - B：Multi-AZは同一リージョン内の可用性のみで、リージョン障害には対応しない。
> - C：同一リージョンのリードレプリカは他地域低遅延・リージョンDRにならない。
> - D：自前cronバックアップはRTO/RPO・運用の面で要件を満たせない。
> **覚えるべきキーワード**：Aurora Global Database / クロスリージョン / 低RTO/RPO / 低遅延読み取り
> **関連サービス**：Aurora / DynamoDBグローバルテーブル / Route 53（誘導）
> **SAAでの出題イメージ**：「RDBでリージョンDR＋各地で低遅延読み取り」=Aurora Global Database。
> 📎 参照：[[0301_RDS]]

## 第7問（グローバルKVS）
複数リージョンのユーザーに、低遅延でアクティブ/アクティブな読み書きを提供するキー値データストアが必要。さらに古いセッションは自動失効させたい。最適なのはどれか。

- A. DynamoDB グローバルテーブル＋TTL
- B. 単一リージョンのDynamoDB
- C. RDS Multi-AZ
- D. ElastiCache 単一ノード

> [!success]- 答え：**A**（DynamoDBグローバルテーブル＋TTL）
> グローバルテーブルは複数リージョンにマルチアクティブで複製し、各地域で低遅延な読み書きを提供する。TTL属性で期限切れアイテム（古いセッション等）を自動削除できる。
> **他の選択肢が違う理由**
> - B：単一リージョンでは他地域の低遅延・リージョン冗長を満たせない。
> - C：RDSはリレーショナルでキー値の超低遅延・マルチアクティブ用途とは異なる。
> - D：ElastiCache単一ノードは耐障害・グローバル分散を満たさない。
> **覚えるべきキーワード**：グローバルテーブル / マルチアクティブ / TTL自動失効 / 低遅延
> **関連サービス**：DynamoDB / DAX / Aurora Global Database
> **SAAでの出題イメージ**：「キー値をグローバルにアクティブ/アクティブ＋自動失効」=グローバルテーブル＋TTL。
> 📎 参照：[[0303_DynamoDB]]

## 第8問（改ざん防止／WORM）
金融規制で、取引ログを「保持期間中は管理者でも削除・変更できない（WORM）」状態でS3に保管する必要がある。最適な構成はどれか。

- A. S3 Object Lock コンプライアンスモード（バージョニング＋保持期間）
- B. バケットポリシーで削除をDenyするだけ
- C. IAMで管理者以外の削除を禁止
- D. Glacierに置けば自動でWORMになる

> [!success]- 答え：**A**（Object Lock コンプライアンスモード）
> Object Lockのコンプライアンスモードは、保持期間中はrootを含む誰も上書き・削除できないWORMを強制する（バージョニング前提）。規制対応の改ざん防止に適する。
> **他の選択肢が違う理由**
> - B：バケットポリシーのDenyは権限変更で外せ、WORMの強制力に欠ける。
> - C：IAM禁止も管理者が変更可能で、保持の強制にならない。
> - D：Glacierはストレージクラスで、保管しただけではWORMにならない（Object Lockの設定が必要）。
> **覚えるべきキーワード**：Object Lock / コンプライアンスモード / WORM / 保持期間
> **関連サービス**：バージョニング / ガバナンスモード / リーガルホールド
> **SAAでの出題イメージ**：「保持中はrootでも消せない」=Object Lockコンプライアンスモード。
> 📎 参照：[[0205_S3のバージョン管理]] / [[0203_S3ストレージクラス]]

## 第9問（HPC共有ストレージ）
HPC（数値計算）クラスタの多数ノードから、S3のデータに対して非常に高いスループットと低遅延でアクセスする共有ファイルシステムが必要。最適なのはどれか。

- A. Amazon FSx for Lustre
- B. Amazon EFS 標準
- C. Amazon S3 を直接マウント
- D. 各ノードに独立したEBS

> [!success]- 答え：**A**（FSx for Lustre）
> FSx for LustreはHPC/機械学習向けの高性能並列ファイルシステムで、S3とシームレス連携し、超高スループット・低遅延を提供する。大規模並列計算の共有ストレージに最適。
> **他の選択肢が違う理由**
> - B：EFS標準は汎用NFS共有で、HPC級の極端なスループット要件には専用のLustreが適する。
> - C：S3直接マウントはHPCの並列高スループットI/Oには不向き。
> - D：独立EBSは各ノード占有で共有にならない。
> **覚えるべきキーワード**：FSx for Lustre / HPC / 並列ファイルシステム / 高スループット / S3連携
> **関連サービス**：EFS（汎用Linux共有） / FSx for Windows / S3
> **SAAでの出題イメージ**：「HPCで超高速な共有ファイル」=FSx for Lustre。
> 📎 参照：[[0202_S3とEBS・EFSの違い]]

## 第10問（非同期・信頼性）
注文処理をLambdaで非同期に行う。スパイク時も取りこぼさず、処理に繰り返し失敗したメッセージは隔離して後で調査したい。最適な構成はどれか。

- A. SQSでバッファ＋Lambdaで処理＋失敗メッセージはDLQ（デッドレターキュー）へ
- B. Lambdaを同期呼び出しし、失敗は無視する
- C. メッセージをLambdaのメモリ内キューに保持
- D. 失敗したら自動で破棄する

> [!success]- 答え：**A**（SQS＋Lambda＋DLQ）
> SQSでスパイクをバッファし、Lambdaがポーリング処理する。一定回数失敗したメッセージはDLQへ退避して本流を止めず、後で原因調査・再処理できる。信頼性の高い非同期構成。
> **他の選択肢が違う理由**
> - B：同期＋失敗無視は取りこぼし・データ損失を招く。
> - C：Lambdaメモリ内保持は揮発し、スケール/再実行で失われる。
> - D：自動破棄は調査・再処理ができず不適切。
> **覚えるべきキーワード**：SQSバッファ / DLQ / 失敗隔離 / 再処理 / 非同期
> **関連サービス**：SQS / Lambda / SNS / CloudWatch（DLQ監視）
> **SAAでの出題イメージ**：「非同期で取りこぼさず失敗を隔離」=SQS＋Lambda＋DLQ。
> 📎 参照：[[0802_SQS]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]]

## 第11問（ワークフロー・オーケストレーション）
複数のLambda・承認待ち・外部API呼び出し・条件分岐・自動リトライを含む長時間の業務フローを、状態を可視化しながら確実に実行したい。最適なのはどれか。

- A. AWS Step Functions
- B. 1つの巨大なLambdaに全処理を詰め込む
- C. EventBridgeのスケジュールでcron実行
- D. SQSにためて順次処理するだけ

> [!success]- 答え：**A**（Step Functions）
> Step Functionsはステートマシンで分岐・並列・自動リトライ・待機（承認待ち含む）・エラーハンドリングを宣言的に組め、各ステップの実行状況を可視化できる。複雑な業務フローの調停に最適。
> **他の選択肢が違う理由**
> - B：巨大Lambdaは15分制限・可観測性・保守性の面で破綻しやすい。
> - C：cronは定期起動で、分岐・リトライ・状態管理の調停役ではない。
> - D：SQSは単純なキューで、複数ステップの状態遷移・分岐管理はしない。
> **覚えるべきキーワード**：ステートマシン / 分岐・並列・リトライ / 可視化 / 長時間フロー
> **関連サービス**：Lambda / SQS / SNS / EventBridge
> **SAAでの出題イメージ**：「複雑な多段フローをリトライ・状態管理込みで」=Step Functions。
> 📎 参照：[[0807_Step_Functions]]

## 第12問（認証付きサーバーレスAPI）
スマホアプリのバックエンドAPIを、サーバー管理なしで構築し、認証済みユーザーだけがアクセスでき、リクエスト量で自動スケールするようにしたい。最適な構成はどれか。

- A. API Gateway（Cognitoオーソライザ）＋ Lambda ＋ DynamoDB
- B. EC2 ＋ 自前認証 ＋ RDS
- C. ALB ＋ Auto Scaling ＋ EC2 ＋ 自前セッション
- D. S3静的ホスティングのみ

> [!success]- 答え：**A**（API Gateway＋Cognito＋Lambda＋DynamoDB）
> フルサーバーレスAPIの王道。API GatewayがCognitoオーソライザでトークン検証して認証を担い、Lambdaがロジック、DynamoDBがデータ層。いずれも自動スケールし従量課金。
> **他の選択肢が違う理由**
> - B：EC2＋自前認証＋RDSはサーバー管理・常時課金が残る。
> - C：ALB＋ASG＋EC2もインスタンス管理が残り「サーバーレス」要件に反する。
> - D：S3静的のみでは認証付き動的APIを提供できない。
> **覚えるべきキーワード**：サーバーレス / Cognitoオーソライザ / API Gateway / Lambda / DynamoDB
> **関連サービス**：Cognito / Step Functions / SQS
> **SAAでの出題イメージ**：「管理なし・認証付き・自動スケールAPI」=API GW＋Cognito＋Lambda＋DynamoDB。
> 📎 参照：[[0809_API_Gateway]] / [[0810_Cognito]]

## 第13問（大量DB接続）
LambdaがスパイクでRDS（Aurora MySQL）へ大量に同時接続を張り、コネクション枯渇でエラーになる。アプリ改修を抑えて接続を効率化したい。最適なのはどれか。

- A. Amazon RDS Proxy
- B. RDSのインスタンスを最大化する
- C. Lambdaの同時実行を無制限にする
- D. 接続を毎回張り直して都度切断する

> [!success]- 答え：**A**（RDS Proxy）
> RDS ProxyはRDS/Aurora前段の接続プール。多数のLambda接続をプールして再利用し、コネクション枯渇を防ぎ、フェイルオーバーも高速化する。Lambda＋RDSの定番。
> **他の選択肢が違う理由**
> - B：インスタンス最大化は一時しのぎで接続数の根本問題が残りコストも増える。
> - C：同時実行無制限は接続をさらに増やし悪化する。
> - D：都度接続・切断はオーバーヘッドと枯渇を悪化させる。
> **覚えるべきキーワード**：RDS Proxy / 接続プール / コネクション枯渇防止 / フェイルオーバー高速化
> **関連サービス**：Aurora / RDS / Lambda / Secrets Manager（認証）
> **SAAでの出題イメージ**：「Lambda大量接続でRDSが枯渇」=RDS Proxy。
> 📎 参照：[[0302_RDSProxy]] / [[0301_RDS]]

## 第14問（機密データ検出と保護）
S3データレイクに個人情報（PII）が混在していないか継続的に検査し、検出時に通知・対応したい。最適な組み合わせはどれか。

- A. Amazon Macie（検出）＋ EventBridge ＋ 通知/自動対応
- B. AWS KMSで暗号化すれば検査は不要
- C. GuardDutyでS3内のPIIを分類する
- D. Inspectorで個人情報をスキャンする

> [!success]- 答え：**A**（Macie＋EventBridge）
> MacieはS3を機械学習でスキャンしPII・機密データを検出・分類する。検出結果（findings）をEventBridgeで受けてSNS通知やLambda自動対応につなげるのが定番。
> **他の選択肢が違う理由**
> - B：暗号化は保護策だが「混在しているかの検出」はしない（目的が別）。
> - C：GuardDutyは脅威検知（挙動分析）で、S3内のPII分類はMacieのHttps領域。
> - D：Inspectorは脆弱性スキャンでデータ分類ではない。
> **覚えるべきキーワード**：Macie / PII検出 / findings / EventBridge連携 / 自動対応
> **関連サービス**：EventBridge / SNS / Lambda / Security Hub / KMS
> **SAAでの出題イメージ**：「S3の機密データを検出して通知/対応」=Macie＋EventBridge。
> 📎 参照：[[1603_脅威検知・監査]] / [[1605_データ保護・暗号化]]

## 第15問（脅威検知と自動対応）
複数アカウントのセキュリティアラートを一元集約し、GuardDutyの検出をトリガーに自動で初動対応（隔離・通知）したい。最適な構成はどれか。

- A. GuardDuty（検出）＋ Security Hub（集約）＋ EventBridge ＋ Lambda（自動対応）
- B. 各アカウントのCloudTrailを手動で確認する
- C. WAFのログだけ見る
- D. Configルールで脅威を検知する

> [!success]- 答え：**A**（GuardDuty＋Security Hub＋EventBridge＋Lambda）
> GuardDutyで脅威を検知し、Security Hubで複数アカウント/サービスのアラートを集約・標準化、EventBridgeで検出をトリガーにLambda等で自動初動対応（隔離・通知）する。SOC自動化の定番。
> **他の選択肢が違う理由**
> - B：手動確認は集約・自動対応にならず、スケールしない。
> - C：WAFログのみでは脅威検知の全体像・集約・自動対応にならない。
> - D：Configは構成準拠チェックで脅威の挙動検知ではない。
> **覚えるべきキーワード**：GuardDuty / Security Hub集約 / EventBridge / 自動対応（SOAR的）
> **関連サービス**：Detective / Macie / SNS / Lambda
> **SAAでの出題イメージ**：「脅威検知を集約し自動初動対応」=GuardDuty＋Security Hub＋EventBridge。
> 📎 参照：[[1603_脅威検知・監査]] / [[0804_EventBridge]]

## 第16問（脆弱性管理）
本番のEC2群とECRのコンテナイメージに、既知の脆弱性（CVE）がないかを継続的・自動的にスキャンして可視化したい。最適なのはどれか。

- A. Amazon Inspector
- B. Amazon GuardDuty
- C. AWS Trusted Advisor
- D. AWS CloudTrail

> [!success]- 答え：**A**（Inspector）
> Inspectorはワークロードの脆弱性管理サービス。EC2・ECRコンテナイメージ・Lambdaを自動・継続スキャンし、CVE等の既知脆弱性やネットワーク到達性リスクを評価・可視化する。
> **他の選択肢が違う理由**
> - B：GuardDutyは挙動ベースの脅威検知で、脆弱性（CVE）スキャンとは目的が異なる。
> - C：Trusted Advisorは横断的なベストプラクティス助言で、CVE脆弱性スキャンが主目的ではない。
> - D：CloudTrailは操作証跡で脆弱性診断ではない。
> **覚えるべきキーワード**：Inspector / 脆弱性スキャン / CVE / EC2/ECR/Lambda
> **関連サービス**：ECR / Security Hub / Systems Manager / GuardDuty
> **SAAでの出題イメージ**：「脆弱性（CVE）を継続スキャン」=Inspector、「挙動の脅威検知」=GuardDuty。
> 📎 参照：[[1603_脅威検知・監査]]

## 第17問（静的サイトの保護配信）
S3でホストする静的サイトを、世界中へ低遅延配信しつつ、S3は直接非公開にし、SQLi/XSSやボット等のWebアプリ攻撃も防ぎたい。最適な構成はどれか。

- A. CloudFront＋OAC（S3非公開）＋ AWS WAF
- B. S3を公開しURLで配る
- C. ALB＋EC2にサイトを置く
- D. Route 53だけで配信する

> [!success]- 答え：**A**（CloudFront＋OAC＋WAF）
> CloudFrontでグローバルにキャッシュ配信し、OACでS3を「CloudFrontからのみ」に限定して非公開を保ち、WAFでSQLi/XSS・ボット等のアプリ層攻撃を遮断する。静的サイトの保護配信の定番。
> **他の選択肢が違う理由**
> - B：S3公開＋URL配布は保護・限定にならない。
> - C：ALB＋EC2は静的サイトに過剰で運用負荷・コストが高い。
> - D：Route 53は名前解決のみで配信・保護はしない。
> **覚えるべきキーワード**：CloudFront / OAC / WAF / S3非公開 / グローバル配信
> **関連サービス**：S3 / WAF / Shield / ACM（HTTPS）
> **SAAでの出題イメージ**：「静的サイトを低遅延配信＋S3非公開＋アプリ攻撃防御」=CloudFront＋OAC＋WAF。
> 📎 参照：[[0405_ネットワークとコンテンツ配信]] / [[1604_ネットワーク防御]]

## 第18問（コスト最適化／アクセス不明）
新しいデータ群のアクセスパターンが予測できず、頻繁・低頻度が入り混じる。運用の手間をかけずにストレージコストを自動最適化したい。最適なS3クラスはどれか。

- A. S3 Intelligent-Tiering
- B. S3 Standard 固定
- C. S3 Glacier Deep Archive 固定
- D. ライフサイクルで30日後にStandard-IA固定

> [!success]- 答え：**A**（Intelligent-Tiering）
> Intelligent-Tieringはアクセス頻度に応じてオブジェクトを自動で階層移動し、取り出し料金なしでコストを最適化する。アクセスパターンが読めない/変動する場合に最適。
> **他の選択肢が違う理由**
> - B：Standard固定は低頻度分も高コストのまま。
> - C：Deep Archive固定は頻繁アクセス分の取り出しが遅く・割高になり不適。
> - D：固定ルールは予測前提で、パターン不明・変動には自動最適化のIntelligent-Tieringが勝る。
> **覚えるべきキーワード**：Intelligent-Tiering / 自動階層化 / 取り出し料金なし / パターン不明
> **関連サービス**：ライフサイクル / Standard-IA / Glacier
> **SAAでの出題イメージ**：「アクセス予測不能・手間なくコスト最適化」=Intelligent-Tiering。
> 📎 参照：[[0203_S3ストレージクラス]] / [[0204_S3のライフサイクル管理]]

## 第19問（耐障害＋コスト最適なコンピュート）
ステートレスなWeb/ワーカー層を、可用性を保ちながら最大限コストを下げたい。中断されても自動補充できる設計にしたい。最適なのはどれか。

- A. Auto Scaling Group で オンデマンド＋スポットを混在（複数AZ・複数インスタンスタイプ）
- B. 全てスポットインスタンス1種類・単一AZ
- C. 全てオンデマンドで固定
- D. 単一の大型オンデマンドEC2

> [!success]- 答え：**A**（ASGでオンデマンド＋スポット混在）
> ステートレス層は、ASGの混合インスタンスポリシーでベース容量をオンデマンド、上乗せをスポットにし、複数AZ・複数インスタンスタイプに分散すれば、スポット中断にも強く大幅なコスト削減と可用性を両立できる。
> **他の選択肢が違う理由**
> - B：単一タイプ・単一AZのスポットは中断/在庫枯渇/AZ障害に弱い。
> - C：全オンデマンドは安定だが割高でコスト最適化にならない。
> - D：単一大型EC2は単一障害点でスケール・耐障害に劣る。
> **覚えるべきキーワード**：混合インスタンスポリシー / スポット＋オンデマンド / 複数AZ/タイプ分散 / 中断耐性
> **関連サービス**：Auto Scaling / スポット / Savings Plans（ベース容量に） / ALB
> **SAAでの出題イメージ**：「ステートレスを安く＋耐障害」=ASGでスポット/オンデマンド混在。
> 📎 参照：[[0102_EC2購入プラン]] / [[0504_Auto_Scaling_と_Auto_Scaling_Group]]

## 第20問（ハイブリッド階層ストレージ）
オンプレのバックアップ/ファイルサーバーの容量が逼迫。アプリは変えずに、古いデータはS3/Glacierへ自動的に階層化してオンプレ容量を空けたい。最適なのはどれか。

- A. AWS Storage Gateway（File/Volume/Tape Gateway）
- B. すべてのデータをDataSyncで一度に移行して削除
- C. オンプレにEBSを増設
- D. S3をインターネット経由で直接マウント

> [!success]- 答え：**A**（Storage Gateway）
> Storage GatewayはオンプレからNFS/SMB/iSCSI/VTLでアクセスでき、よく使うデータはローカルキャッシュ、実体や古いデータはS3/Glacierへ階層化する。アプリ改修なしでオンプレ容量を逃がせる。
> **他の選択肢が違う理由**
> - B：DataSyncは転送ツールで、日常的にローカル感覚で使う階層化ゲートウェイではない。
> - C：EBS増設はオンプレ機器に接続できず（EC2用）、根本解決にならない。
> - D：S3直接マウントは日常的なファイルサーバー用途に不向き。
> **覚えるべきキーワード**：Storage Gateway / 階層化 / ローカルキャッシュ / S3/Glacierバックエンド
> **関連サービス**：DataSync（移行） / S3 / Glacier / FSx
> **SAAでの出題イメージ**：「オンプレ容量逼迫を改修なしでS3階層化」=Storage Gateway。
> 📎 参照：[[1502_DataSync vs Firehose vs Storage Gateway]] / [[1501_移行と転送サービス一覧]]
