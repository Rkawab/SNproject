# 【問題】サービス用途確認8（公式サンプル風・全10問）

## この問題集の目的

要件文から主要サービスの役割を選ぶドリル。101〜107で扱っていない切り口（短時間イベント処理のサーバーレス・共有ファイルシステム・インメモリキャッシュ・DNSルーティング・脅威検知・構成変更の記録・業務ワークフロー・サーバーレスRDB・BIダッシュボード・画像認識など）を中心に、13分野を横断する。

## 対象レベル

SAA入口。単語暗記ではなく、要件からサービスを選ぶ練習にする。

## 問題数

10問

## 第1問（Compute ｜ 不定期・短時間のイベント処理）
画像がアップロードされた時だけ実行される短時間の処理を動かしたい。サーバーのプロビジョニングや常時起動はしたくなく、実行した分だけ課金されるのが理想。最も適切なサービスはどれか。

- A. AWS Lambda
- B. Amazon EC2 Auto Scaling
- C. AWS Batch
- D. Amazon EC2 リザーブドインスタンス

> [!success]- 答え：**A**
> Lambdaはイベント駆動で関数を実行し、サーバー管理不要・実行時間課金で、不定期・短時間の処理に最適。
> - A：正解。常時起動せず、イベント発生時だけ実行され従量課金になる。
> - B：サーバーを起動し続ける前提のスケーリングで、短時間イベント処理にはオーバーヘッドが大きい。
> - C：大量のバッチジョブのキューイング実行向けで、軽量な単発イベント処理には重い。
> - D：1年以上の継続利用を割引前提に確保する購入方法で、不定期処理には合わない。
> **関連ノート**：[[0702_Lambda・ECS・RDS・Beanstalk比較]] / [[0708_Lambda]]

## 第2問（Storage ｜ 複数サーバーからの共有ファイルシステム）
複数のEC2インスタンスから同時にマウントして読み書きできる、自動拡張するフルマネージドなNFS共有ファイルシステムが欲しい。最も適切なサービスはどれか。

- A. Amazon EFS
- B. Amazon EBS
- C. Amazon S3
- D. インスタンスストア

> [!success]- 答え：**A**
> EFSは複数AZの複数EC2から同時マウントできる、自動拡張のマネージドNFSファイルシステム。
> - A：正解。共有ファイルシステムとして複数インスタンスから同時利用できる。
> - B：原則1インスタンスにアタッチするブロックストレージで、共有NFS用途ではない。
> - C：オブジェクトストレージで、POSIXなファイルシステムとしてマウントする用途とは異なる。
> - D：インスタンスに物理的に付随する一時ストレージで、共有も永続もできない。
> **関連ノート**：[[0202_S3とEBS・EFSの違い]]

## 第3問（Database ｜ マイクロ秒応答のインメモリキャッシュ）
頻繁に読まれる同一データへのアクセスを高速化したい。データベースの前段に、ミリ秒未満で応答するマネージドなインメモリキャッシュを置きたい。最も適切なサービスはどれか。

- A. Amazon ElastiCache
- B. Amazon RDS
- C. Amazon Athena
- D. Amazon Redshift

> [!success]- 答え：**A**
> ElastiCache（Redis/Memcached）はインメモリでマイクロ〜ミリ秒未満の応答を提供し、DB前段のキャッシュに最適。
> - A：正解。高頻度アクセスをメモリ上でキャッシュし、DB負荷とレイテンシを下げられる。
> - B：永続的なリレーショナルDBで、インメモリキャッシュ層そのものではない。
> - C：S3上のデータへのアドホックなSQLクエリ用途で、低レイテンシのキャッシュではない。
> - D：分析向けデータウェアハウスで、キャッシュ用途ではない。
> **関連ノート**：[[0304_ElastiCache]] / [[0301_RDS]]

## 第4問（Networking / CDN / LB ｜ ドメイン登録とDNSルーティング）
独自ドメインの登録と、DNSによる名前解決、ヘルスチェックや加重・レイテンシーに基づくルーティングをマネージドに行いたい。最も適切なサービスはどれか。

- A. Amazon Route 53
- B. Amazon CloudFront
- C. AWS Global Accelerator
- D. Elastic Load Balancing

> [!success]- 答え：**A**
> Route 53はドメイン登録・権威DNS・各種ルーティングポリシー・ヘルスチェックを提供するマネージドDNSサービス。
> - A：正解。ドメイン登録からDNSルーティングまでを担う。
> - B：エッジからのコンテンツ配信（CDN）が主目的で、DNSの権威サーバーそのものではない。
> - C：固定IPとAWSバックボーン経由の高速化が主で、ドメイン登録やDNSルーティング機能ではない。
> - D：VPC内のターゲットへの負荷分散で、DNSの名前解決サービスではない。
> **関連ノート**：[[0404_Route53]] / [[0405_ネットワークとコンテンツ配信]]

## 第5問（Security / IAM ｜ アカウントの脅威検知）
AWSアカウント内の不審なAPI呼び出しや、悪意あるIPとの通信、想定外の挙動を、機械学習で継続的に検知して通知したい。エージェント導入なしで始めたい。最も適切なサービスはどれか。

- A. Amazon GuardDuty
- B. AWS WAF
- C. AWS Shield
- D. Amazon Macie

> [!success]- 答え：**A**
> GuardDutyはCloudTrail・VPCフローログ・DNSログ等を分析し、脅威を継続検知するマネージドな脅威検知サービス。
> - A：正解。ログ分析で不審な挙動を機械学習により検知でき、エージェント不要で始められる。
> - B：L7のHTTPリクエストをルールで遮断するWebアプリ防御で、アカウント全体の脅威検知ではない。
> - C：DDoS攻撃の緩和が主目的で、アカウント内の不審な挙動の検知ではない。
> - D：S3上の機密データ（個人情報等）の発見・分類が主目的で、アカウントの脅威検知エンジンではない。
> **関連ノート**：[[1603_脅威検知・監査]] / [[1601_セキュリティサービス一覧]]

## 第6問（Management / Monitoring ｜ 構成変更の記録と準拠評価）
AWSリソースの構成変更を時系列で記録し、「セキュリティグループが全開放になっていないか」などの社内ルールへの準拠を自動評価したい。最も適切なサービスはどれか。

- A. AWS Config
- B. AWS CloudTrail
- C. Amazon CloudWatch
- D. AWS Trusted Advisor

> [!success]- 答え：**A**
> AWS Configはリソース構成の変更履歴を記録し、Configルールで望ましい状態への準拠を継続評価する。
> - A：正解。構成のスナップショットと準拠評価（コンプライアンス）に特化している。
> - B：APIコールの監査ログ（誰が何をしたか）が主で、リソース構成の準拠評価そのものではない。
> - C：メトリクス・ログ・アラームの監視が主で、構成変更履歴と準拠評価の専用機能ではない。
> - D：ベストプラクティスの推奨チェックを提供するが、任意の構成ルールの継続評価・変更履歴管理ではない。
> **関連ノート**：[[1003_AWS_Config]] / [[1002_CloudTrail]]

## 第7問（Application Integration ｜ 複数ステップの業務ワークフロー）
複数のLambda関数やサービスを、条件分岐・リトライ・並列実行を含む順序立てたワークフローとして実行し、各ステップの進行状況を可視化したい。最も適切なサービスはどれか。

- A. AWS Step Functions
- B. Amazon SQS
- C. Amazon SNS
- D. Amazon EventBridge

> [!success]- 答え：**A**
> Step Functionsはステートマシンとして複数ステップのワークフローを定義し、分岐・リトライ・並列・可視化を提供する。
> - A：正解。順序立てた業務フローのオーケストレーションと進行可視化に最適。
> - B：メッセージのキューイング（バッファ）が主で、ステップ間の制御フローを定義する仕組みではない。
> - C：1対多のプッシュ通知（Pub/Sub）で、ワークフローの制御フローではない。
> - D：イベントのルーティングが主で、複数ステップの状態遷移をステートマシンとして管理する用途ではない。
> **関連ノート**：[[0807_Step_Functions]] / [[0801_アプリケーション統合サービス一覧]]

## 第8問（Serverless ｜ 需要に応じて自動スケールするリレーショナルDB）
不定期にしか使われない開発・検証用のMySQL互換DBが欲しい。アクセスがある時だけ容量を自動で増減し、アイドル時はスケールを抑えてコストを下げたい。サーバー管理はしたくない。最も適切なサービスはどれか。

- A. Amazon Aurora Serverless v2
- B. Amazon RDS for MySQL（プロビジョンド）
- C. Amazon DynamoDB
- D. Amazon EC2 上に構築したMySQL

> [!success]- 答え：**A**
> Aurora Serverless v2は負荷に応じて容量を自動で増減するサーバーレスなリレーショナルDBで、断続的なワークロードに向く。
> - A：正解。需要に応じた自動スケールで、断続利用のコストを抑えられる。
> - B：固定のインスタンス容量を確保する方式で、アイドル時も課金され自動スケールしない。
> - C：キー・バリューのNoSQLで、MySQL互換のリレーショナル用途ではない。
> - D：自前管理が必要で、サーバーレスの自動スケール・無管理という要件に反する。
> **関連ノート**：[[0307_Aurora詳細]] / [[0301_RDS]]

## 第9問（Analytics ｜ ビジネスインテリジェンスのダッシュボード）
売上データを取り込み、対話的なグラフやダッシュボードを作成して、部門のメンバーがブラウザから可視化・共有できるようにしたい。マネージドなBIサービスを使いたい。最も適切なサービスはどれか。

- A. Amazon QuickSight
- B. Amazon OpenSearch Service
- C. AWS Glue
- D. Amazon Athena

> [!success]- 答え：**A**
> QuickSightはマネージドなBIサービスで、対話的なダッシュボードの作成・共有に最適。
> - A：正解。ビジネスデータの可視化ダッシュボードを手軽に作成・共有できる。
> - B：ログの全文検索・分析エンジンが主で、汎用のBIダッシュボード作成ツールではない。
> - C：サーバーレスETLとデータカタログが主で、可視化ダッシュボードのサービスではない。
> - D：S3上のデータへのSQLクエリエンジンで、それ自体がBIダッシュボードを描画するわけではない。
> **関連ノート**：[[1111_Amazon QuickSight]] / [[1101_AWS分析サービス一覧]]

## 第10問（Machine Learning ｜ 画像・動画の物体や顔の検出）
アップロードされた画像や動画から、物体・シーン・顔・不適切コンテンツを検出してタグ付けしたい。機械学習モデルの構築はせず、APIを呼ぶだけで使いたい。最も適切なサービスはどれか。

- A. Amazon Rekognition
- B. Amazon Comprehend
- C. Amazon Textract
- D. Amazon Polly

> [!success]- 答え：**A**
> Rekognitionは画像・動画の物体検出・顔分析・コンテンツモデレーションをAPI経由で提供するマネージドサービス。
> - A：正解。学習不要で、画像・動画の解析をAPIで実行できる。
> - B：自然言語テキストの感情・キーフレーズ抽出向けで、画像解析ではない。
> - C：書類画像からの文字・帳票データ抽出（OCR）向けで、物体・顔の検出ではない。
> - D：テキストを音声に変換する用途で、画像解析ではない。
> **関連ノート**：[[1301_AWS機械学習サービス一覧]]
