# 【問題】類似サービス比較14（実行基盤・統合・エッジ・移行・コスト）

## この問題集の目的

実行時間、通信モデル、配置場所、移行フェーズ、割引対象の違いから、似たサービスを選び分ける。

## 対象レベル

SAA基礎〜標準。選択肢を切るための決定的な制約を確認する。

## 問題数

10問

---

## 第1問（Compute ｜ Lambda と Fargate）
既存のDockerコンテナで動く動画変換処理があります。1回の処理は45分かかり、OSライブラリとCPU・メモリの割り当てを細かく制御する必要があります。サーバーは管理したくありません。
最も適した実行基盤はどれですか。

- A. Amazon ECS on AWS Fargate
- B. AWS Lambda
- C. Amazon EC2の常時稼働インスタンス1台
- D. Amazon API Gateway

> [!success]- 答え：**A**
> Lambdaは短時間のイベント処理、Fargateはサーバー管理なしで長時間のコンテナを実行する用途に向きます。
> - A：正解。コンテナ、長時間処理、リソース指定という要件に合う。
> - B：実行時間上限があり、45分の単一実行には適さない。
> - C：実行は可能だが、OSや容量の管理が必要になる。
> - D：APIの入口を管理するサービスで、動画変換コンテナの実行基盤ではない。
> **着眼点（SAA）**：短いイベント処理はLambda、長時間・任意コンテナはFargate。
> **関連ノート**：[[0702_Lambda・ECS・RDS・Beanstalk比較]] / [[0708_Lambda]]
> **詳しい解説**：[[214_【解説】類似サービス比較14#第1問（Compute ｜ Lambda と Fargate）]]

## 第2問（Compute ｜ AWS Batch と Step Functions）
数万件の独立した科学計算ジョブをキューに入れ、優先度に応じてEC2 Spotを含む計算環境へスケジュールし、必要な計算容量を自動で増減させたいと考えています。
最も適したサービスはどれですか。

- A. AWS Batch
- B. AWS Step Functions
- C. Amazon EventBridge Scheduler
- D. AWS Lambda

> [!success]- 答え：**A**
> AWS Batchはバッチジョブのキューと計算環境を管理し、Step Functionsは複数サービスの処理順序を管理します。
> - A：正解。大量ジョブのスケジューリングとコンピューティング容量のプロビジョニングを行う。
> - B：ジョブ間の分岐や再試行のオーケストレーションには使えるが、計算環境そのものは提供しない。
> - C：時刻に基づく起動には使えるが、大量バッチの計算資源を管理しない。
> - D：個々の短時間処理には使えるが、長時間・大規模バッチ計算の専用基盤ではない。
> **着眼点（SAA）**：計算ジョブと計算環境はBatch、処理の手順・状態遷移はStep Functions。
> **関連ノート**：[[0703_AWS_Batch]] / [[0807_Step_Functions]]
> **詳しい解説**：[[214_【解説】類似サービス比較14#第2問（Compute ｜ AWS Batch と Step Functions）]]

## 第3問（Application Integration ｜ EventBridge Pipes と Step Functions）
SQSキューのメッセージをフィルタリングし、入力形式を変換してStep Functionsへ渡したいと考えています。独自のポーリング用Lambdaコードは作成せず、単一の送信元と単一の送信先を直接接続する必要があります。
最も適したサービスはどれですか。

- A. Amazon EventBridge Pipes
- B. AWS Step Functionsだけを使用する
- C. Amazon SNSトピックを作成する
- D. Amazon Kinesis Data Streamsへ移行する

> [!success]- 答え：**A**
> EventBridge Pipesは送信元と送信先をポイントツーポイントで接続し、フィルタ・変換・エンリッチメントを提供します。
> - A：正解。SQSのポーリングを含む統合をマネージドで構成できる。
> - B：複数ステップのワークフローは管理できるが、SQSからのポーリング接続を単独で置き換える選択肢ではない。
> - C：プッシュ型ファンアウト用で、既存SQSのフィルタ付きポーリング接続とは異なる。
> - D：順序付きストリームが必要という要件はなく、移行が過剰。
> **着眼点（SAA）**：1送信元→1送信先のフィルタ付き接続はPipes、複数処理の状態管理はStep Functions。
> **関連ノート**：[[0804_EventBridge]] / [[0807_Step_Functions]]
> **詳しい解説**：[[214_【解説】類似サービス比較14#第3問（Application Integration ｜ EventBridge Pipes と Step Functions）]]

## 第4問（Application Integration ｜ SNS と EventBridge）
自社アプリケーションが発行する業務イベントを、イベント内容の属性に応じて異なるAWSアカウントのターゲットへルーティングし、必要に応じてイベントをアーカイブして再送したいと考えています。
最も適したサービスはどれですか。

- A. Amazon EventBridge
- B. Amazon SNS
- C. Amazon SQS FIFOキュー1つ
- D. Amazon MQ

> [!success]- 答え：**A**
> SNSはプッシュ型の通知・ファンアウト、EventBridgeはイベントパターンによるルーティングとイベントバス機能が中心です。
> - A：正解。コンテンツベースのルーティング、クロスアカウント、アーカイブとリプレイに適する。
> - B：購読先への高速なプッシュ配信には向くが、イベントバスとしての高度なルーティングが決め手ならEventBridge。
> - C：単一キューは競合コンシューマーのバッファで、複数ターゲットへのルーティングではない。
> - D：既存のActiveMQ/RabbitMQ互換アプリを移行する場合に使う。
> **着眼点（SAA）**：単純な通知・ファンアウトはSNS、ルールでイベントを選別・再送するならEventBridge。
> **関連ノート**：[[0803_SNS]] / [[0804_EventBridge]]
> **詳しい解説**：[[214_【解説】類似サービス比較14#第4問（Application Integration ｜ SNS と EventBridge）]]

## 第5問（Application Integration ｜ Kinesis Data Streams と SQS）
センサーイベントを順序付きのストリームとして数日間保持し、複数の独立した分析アプリケーションがそれぞれ同じイベントを繰り返し読み取れるようにしたいと考えています。
最も適したサービスはどれですか。

- A. Amazon Kinesis Data Streams
- B. Amazon SQS Standardキュー
- C. Amazon SNSから単一SQSキューへ配信する
- D. AWS Step Functions

> [!success]- 答え：**A**
> Kinesis Data Streamsは再読可能なイベントログ、SQSは処理後に削除する作業キューとして使うのが基本です。
> - A：正解。保持期間内のレコードを複数コンシューマーが独立した位置から読める。
> - B：メッセージをコンシューマー間で分担する用途に向き、同じキューの各メッセージを全分析アプリが読む設計ではない。
> - C：単一キューでは全コンシューマーへの複製にならない。
> - D：ワークフローの状態遷移を管理するサービス。
> **着眼点（SAA）**：再読可能な時系列ストリームはKinesis、仕事の受け渡しとバッファはSQS。
> **関連ノート**：[[0805_Kinesis]] / [[0802_SQS]]
> **詳しい解説**：[[214_【解説】類似サービス比較14#第5問（Application Integration ｜ Kinesis Data Streams と SQS）]]

## 第6問（Compute ｜ Outposts と Local Zones）
工場設備との通信遅延とデータ所在地の要件により、AWS計算基盤を自社工場の建物内に設置する必要があります。AWSがハードウェアを管理し、リージョンと一貫したAPIを使いたいと考えています。
最も適した選択肢はどれですか。

- A. AWS Outposts
- B. AWS Local Zones
- C. AWS Wavelength
- D. AWS Direct Connect

> [!success]- 答え：**A**
> Outpostsは顧客施設内、Local ZonesはAWSが選定した都市圏内に置かれます。
> - A：正解。自社工場内へAWS管理のインフラを設置できる。
> - B：利用者の近くにAWSサービスを配置するが、任意の顧客施設内へ設置するものではない。
> - C：5G通信事業者のネットワークエッジ向け。
> - D：専用接続を提供するが、オンプレミスにAWS計算基盤を設置しない。
> **着眼点（SAA）**：顧客拠点内はOutposts、AWSの都市拠点はLocal Zones。
> **関連ノート**：[[0705_AWS_Outposts]] / [[0701_コンピューティングサービス一覧]]
> **詳しい解説**：[[214_【解説】類似サービス比較14#第6問（Compute ｜ Outposts と Local Zones）]]

## 第7問（Compute ｜ Local Zones と Wavelength）
5G接続のARアプリケーションで、モバイル端末から通信事業者ネットワーク内の計算処理までの遅延を最小にする必要があります。

> - C：顧客施設内のハイブリッド基盤。
> - D：軽量なHTTPエッジ処理用で、5G向けEC2ワークロードの配置先ではない。
> **着眼点（SAA）**：5Gキャリア網はWavelength、都市圏の低遅延はLocal Zones。

## 第8問（Migration / Transfer ｜ Application Discovery Service と Migration Hub）
移行プロジェクトの計画段階で、オンプレミスサーバーのCPU使用率、構成、実行中プロセス、依存関係を収集したいと考えています。
最も適したサービスはどれですか。

- A. AWS Application Discovery Service
- B. AWS Migration Hub
- C. AWS Application Migration Service
- D. AWS Database Migration Service

> [!success]- 答え：**A**
> Application Discovery Serviceは情報を収集し、Migration Hubは各移行ツールの進捗を一元追跡します。
> - A：正解。移行前の棚卸し・利用状況・依存関係分析に使う。
> - B：収集済み情報や移行状況をアプリケーション単位で管理する画面を提供する。
> - C：サーバーをAWSへ継続複製してリホストする。
> - D：データベースを移行する。
> **着眼点（SAA）**：現状を発見するのはDiscovery Service、移行の進捗を見るのはMigration Hub。
> **関連ノート**：[[1501_移行と転送サービス一覧]]
> **詳しい解説**：[[214_【解説】類似サービス比較14#第8問（Migration / Transfer ｜ Application Discovery Service と Migration Hub）]]

クリックストリームに対して、過去10分間の状態を保持したウィンドウ集計と異常パターン検出をリアルタイムで行いたいと考えています。

## 第10問（Cost Optimization ｜ Savings Plans と Reserved Instances）
企業は、リージョンやインスタンスファミリーが変わる可能性のあるEC2利用に加えて、FargateとLambdaも継続利用します。一定額のコンピューティング使用量を1年または3年コミットして、広い範囲へ柔軟に割引を適用したいと考えています。
最も適した購入オプションはどれですか。

- A. Compute Savings Plans
- B. EC2 Standard Reserved Instances
- C. EC2 Dedicated Hosts
- D. EC2 Spot Instancesだけを使用する

> [!success]- 答え：**A**
> Compute Savings Plansは広いコンピューティング使用量への金額コミット、EC2 Reserved Instancesは特定のEC2属性により強く結び付く割引です。
> - A：正解。EC2のファミリー・リージョン変更に柔軟で、FargateとLambdaにも適用できる。
> - B：高い割引を得られる場合があるが、対象はEC2で柔軟性が低い。
> - C：物理ホスト専有やライセンス要件に使う購入形態で、広範な割引コミットではない。
> - D：中断可能な耐障害ワークロードには安価だが、継続利用全体を安定してカバーするコミット割引ではない。
> **着眼点（SAA）**：EC2・Fargate・Lambdaを横断する柔軟性ならCompute Savings Plans。
> **関連ノート**：[[1701_コスト管理サービス一覧]]
> **詳しい解説**：[[214_【解説】類似サービス比較14#第10問（Cost Optimization ｜ Savings Plans と Reserved Instances）]]
