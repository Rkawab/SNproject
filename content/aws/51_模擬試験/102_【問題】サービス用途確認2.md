# 【問題】サービス用途確認 ②（4択・全20問）

## この問題集の目的

[[101_【問題】サービス用途確認]]の続編。1問目では扱わなかった「2歩目のサービス」（キャッシュ・分析・コンテナ・運用・セキュリティ系）の基本用途を、シナリオで結びつけて覚える。サービス名を見たら「何のためか」を即答できるようにする。

## 対象レベル

初心者〜SAA入門。用語の丸暗記ではなく「この要件ならこのサービス」を選ぶ練習。

## 問題数

20問（1サービス＝1問。ElastiCache / Aurora / Redshift / Athena / Glue / Kinesis / API Gateway / Step Functions / Cognito / Fargate / ECR / CloudFormation / Systems Manager / Organizations / WAF / Shield / GuardDuty / Config / Direct Connect / AWS Backup）

## 使い方

まず選択肢を見ずに「自分なら何を使うか」を考える。回答後は、正解の理由だけでなく**不正解の選択肢がなぜ違うか**まで読む。迷ったら 📎 のノートに戻る。

## 第1問（インメモリキャッシュ）
DBへの同じ参照クエリが大量に繰り返され負荷が高い。結果をメモリ上にキャッシュし、マイクロ秒〜ミリ秒で返してDB負荷を下げたい。適切なサービスはどれか。

- A. Amazon ElastiCache
- B. Amazon S3
- C. Amazon EBS
- D. Amazon Athena

> [!success]- 答え：**A**（ElastiCache）
> ElastiCacheはRedis/Memcached互換のインメモリキャッシュ。頻出クエリ結果やセッションをメモリに保持し、超低遅延で返してバックエンドDBの負荷を軽減する。
> **他の選択肢が違う理由**
> - B：S3はオブジェクトストレージで、低遅延キャッシュ用途ではない。
> - C：EBSはEC2のディスクで、共有キャッシュ層ではない。
> - D：AthenaはS3へのSQL分析で、リアルタイムキャッシュではない。
> **覚えるべきキーワード**：インメモリ / Redis・Memcached / 読み取り高速化 / セッション外部化
> **関連サービス**：DynamoDB DAX（Dynamo専用キャッシュ） / RDS（オリジンDB）
> **SAAでの出題イメージ**：「頻出参照をキャッシュしてDB負荷を下げる」ならElastiCache。
> 📎 参照：[[0304_ElastiCache]]

## 第2問（高性能リレーショナルDB）
MySQL/PostgreSQL互換のまま、標準RDSより高いスループットと高速フェイルオーバー・ストレージ自動拡張を、できるだけマネージドに使いたい。適切なサービスはどれか。

- A. Amazon RDS for MySQL（シングルAZ）
- B. Amazon Aurora
- C. Amazon DynamoDB
- D. Amazon Redshift

> [!success]- 答え：**B**（Aurora）
> AuroraはMySQL/PostgreSQL互換のクラウド最適化RDB。共有ストレージ・高速フェイルオーバー・最大15のリードレプリカ・自動拡張を備え、標準RDSより高性能・高可用。
> **他の選択肢が違う理由**
> - A：シングルAZのRDSはAuroraの高可用機構を持たない。
> - C：DynamoDBはNoSQL（キー値）で互換性が異なる。
> - D：Redshiftは分析（OLAP）向けDWHで、トランザクション用途ではない。
> **覚えるべきキーワード**：MySQL/PostgreSQL互換 / 高速フェイルオーバー / 共有ストレージ
> **関連サービス**：Aurora Global Database / Aurora Serverless / RDS Proxy
> **SAAでの出題イメージ**：「RDBで高性能・高可用・互換維持」ならAurora。
> 📎 参照：[[0301_RDS]]

## 第3問（データウェアハウス）
数億行の販売データを対象に、列指向で高速に集計・分析するBI基盤（DWH）を構築したい。適切なサービスはどれか。

- A. Amazon RDS
- B. Amazon Redshift
- C. Amazon DynamoDB
- D. Amazon ElastiCache

> [!success]- 答え：**B**（Redshift）
> Redshiftは列指向のマネージドデータウェアハウス。大量データの集計・分析（OLAP）に最適で、BIツールとの連携も定番。
> **他の選択肢が違う理由**
> - A：RDSは日々の取引（OLTP）向けで、大規模集計には不向き。
> - C：DynamoDBはキー値NoSQLで集計分析向きではない。
> - D：ElastiCacheはキャッシュで分析DWHではない。
> **覚えるべきキーワード**：OLAP / 列指向 / DWH / BI
> **関連サービス**：Athena（S3を直接SQL） / QuickSight（可視化） / Glue（ETL）
> **SAAでの出題イメージ**：「大量データを集計・分析するDWH」ならRedshift。
> 📎 参照：[[1113_Amazon Redshift]]

## 第4問（S3への直接SQL）
S3に蓄積したログを、サーバーを立てずに標準SQLでアドホックに分析し、スキャンしたデータ量だけ課金されるようにしたい。適切なサービスはどれか。

- A. Amazon Redshift
- B. Amazon Athena
- C. Amazon EMR
- D. Amazon RDS

> [!success]- 答え：**B**（Athena）
> AthenaはS3上のデータに対し、サーバーレスで標準SQLを実行できる。スキャン量課金で、クラスタ管理不要。ログのアドホック分析の定番。
> **他の選択肢が違う理由**
> - A：Redshiftは事前にロード・クラスタ運用が要る本格DWH。アドホック・サーバーレスの手軽さではAthena。
> - C：EMRはHadoop/Sparkの大規模分散処理クラスタで、運用負荷が高い。
> - D：RDSはS3を直接クエリできない。
> **覚えるべきキーワード**：サーバーレスSQL / S3直接クエリ / スキャン量課金
> **関連サービス**：Glue（カタログ） / S3 / QuickSight
> **SAAでの出題イメージ**：「S3をそのままSQLで・管理なし」ならAthena。
> 📎 参照：[[1109_Amazon Athena]]

## 第5問（ETL・データカタログ）
複数ソースのデータを抽出・変換・ロード（ETL）し、スキーマをカタログ化してAthenaやRedshiftから参照できるようにしたい。サーバーレスで運用負荷を抑えたい。適切なサービスはどれか。

- A. AWS Glue
- B. AWS Lambda
- C. Amazon Kinesis Data Streams
- D. AWS DataSync

> [!success]- 答え：**A**（Glue）
> GlueはサーバーレスのETL基盤＋データカタログ。クローラでスキーマを自動カタログ化し、Athena/Redshift Spectrum/EMRから共通メタデータとして参照できる。
> **他の選択肢が違う理由**
> - B：Lambdaは汎用の関数実行で、ETL・カタログ機能を専門に持たない。
> - C：Kinesisはストリーミング取り込みで、ETLカタログが主目的ではない。
> - D：DataSyncはストレージ間のデータ転送サービスでETL変換ではない。
> **覚えるべきキーワード**：ETL / データカタログ / クローラ / サーバーレス
> **関連サービス**：Athena / Redshift / Lake Formation / S3（データレイク）
> **SAAでの出題イメージ**：「ETL＋カタログをサーバーレスで」ならGlue。
> 📎 参照：[[1107_AWS Glue]]

## 第6問（ストリーミング取り込み）
IoTやクリックストリームから秒間数万件のデータが届く。順序を保ちつつ複数のコンシューマでリアルタイムに処理したい。適切なサービスはどれか。

- A. Amazon SQS（標準キュー）
- B. Amazon Kinesis Data Streams
- C. Amazon SNS
- D. AWS Step Functions

> [!success]- 答え：**B**（Kinesis Data Streams）
> Kinesis Data Streamsは大量データをシャードで順序保持しながら取り込み、複数コンシューマが同じストリームを並行に読める。リアルタイムストリーム処理の基盤。
> **他の選択肢が違う理由**
> - A：標準SQLは順序保証がなく、取得後にメッセージが消えるため複数コンシューマの並行再処理に不向き。
> - C：SNSはPub/Sub通知で、ストリームの順序保持・再読込はしない。
> - D：Step Functionsはワークフロー制御で、ストリーム取り込み基盤ではない。
> **覚えるべきキーワード**：ストリーミング / シャード / 順序保持 / 複数コンシューマ
> **関連サービス**：Kinesis Data Firehose（S3等へ配信） / Lambda / MSK
> **SAAでの出題イメージ**：「大量ストリームを順序保持・リアルタイム」ならKinesis Data Streams。
> 📎 参照：[[0805_Kinesis]] / [[1103_Amazon Kinesis]]

## 第7問（API管理）
モバイル/Webクライアント向けのREST APIを公開し、認証・流量制限（スロットリング）・ステージ管理・Lambda連携をマネージドに行いたい。適切なサービスはどれか。

- A. Application Load Balancer
- B. Amazon API Gateway
- C. Amazon Route 53
- D. Amazon CloudFront

> [!success]- 答え：**B**（API Gateway）
> API GatewayはマネージドなAPIフロント。認証（Cognito/IAM/Lambdaオーソライザ）、スロットリング、ステージ/バージョン管理、Lambdaやバックエンド統合を提供する。
> **他の選択肢が違う理由**
> - A：ALBもHTTP/Lambdaを裁けるが、APIキー・スロットリング・SDK生成などAPI管理機能はAPI Gatewayが本職。
> - C：Route 53はDNSでAPI管理ではない。
> - D：CloudFrontはCDN（配信）。
> **覚えるべきキーワード**：REST/HTTP API / スロットリング / オーソライザ / ステージ
> **関連サービス**：Lambda / Cognito / WAF
> **SAAでの出題イメージ**：「マネージドAPI・認証・流量制限」ならAPI Gateway。
> 📎 参照：[[0809_API_Gateway]]

## 第8問（ワークフロー制御）
複数のLambdaやサービス呼び出しを、分岐・並列・リトライ・待機を含む順序立てたワークフローとして、状態を可視化しながら実行したい。適切なサービスはどれか。

- A. Amazon SQS
- B. AWS Step Functions
- C. Amazon EventBridge
- D. AWS Lambda

> [!success]- 答え：**B**（Step Functions）
> Step Functionsはサーバーレスのワークフローオーケストレーション。各ステップ（状態）を定義し、分岐・並列・自動リトライ・エラーハンドリングを宣言的に組め、実行状況を可視化できる。
> **他の選択肢が違う理由**
> - A：SQSは単純なメッセージキューで、ステップの順序制御やリトライ管理はしない。
> - C：EventBridgeはイベント振り分けで、複数ステップの状態遷移管理が主目的ではない。
> - D：Lambdaは単一の処理実行で、複数処理の調停（オーケストレーション）役ではない。
> **覚えるべきキーワード**：ステートマシン / 分岐・並列・リトライ / オーケストレーション
> **関連サービス**：Lambda / SQS / SNS / EventBridge
> **SAAでの出題イメージ**：「複数処理を順序・リトライ込みで調停」ならStep Functions。
> 📎 参照：[[0807_Step_Functions]]

## 第9問（アプリのユーザー認証）
モバイル/Webアプリの「エンドユーザー（会員）」のサインアップ・サインイン・ソーシャルログインを管理し、認証後にAWSリソースへの一時アクセスも付与したい。適切なサービスはどれか。

- A. AWS IAM
- B. Amazon Cognito
- C. AWS Directory Service
- D. AWS Secrets Manager

> [!success]- 答え：**B**（Cognito）
> Cognitoはアプリの会員認証基盤。ユーザープールでサインアップ/サインインやソーシャル/SAML連携を担い、IDプールで認証後の一時的なAWSアクセス権限を付与できる。
> **他の選択肢が違う理由**
> - A：IAMは「AWSアカウント内の利用者・サービス」の認可で、アプリ会員の認証用ではない。
> - C：Directory Serviceは企業のActive Directory連携向け。
> - D：Secrets Managerは秘密情報の管理でユーザー認証ではない。
> **覚えるべきキーワード**：ユーザープール / IDプール / ソーシャルログイン / JWT
> **関連サービス**：API Gateway（オーソライザ） / IAM / Lambda
> **SAAでの出題イメージ**：「アプリ会員の認証」はCognito、「AWSリソースの権限」はIAM。
> 📎 参照：[[0810_Cognito]]

## 第10問（サーバーレスコンテナ）
Dockerコンテナを動かしたいが、コンテナを載せるEC2（ホスト）のパッチ・スケール・管理から解放されたい。適切な実行環境はどれか。

- A. AWS Fargate（ECS/EKS）
- B. Amazon EC2にDockerを手動構築
- C. AWS Lambda
- D. Amazon S3

> [!success]- 答え：**A**（Fargate）
> FargateはECS/EKSのためのサーバーレスコンテナ実行環境。ホスト（EC2）のプロビジョニング・パッチ・スケール管理が不要で、コンテナ単位のリソース指定だけで動かせる。
> **他の選択肢が違う理由**
> - B：EC2手動構築はホストのパッチ・スケール管理が残る。
> - C：Lambdaは関数（最大15分）向けで、常時稼働の既存コンテナをそのまま動かす用途とは異なる。
> - D：S3はストレージで実行基盤ではない。
> **覚えるべきキーワード**：サーバーレスコンテナ / ホスト管理不要 / ECS/EKS連携
> **関連サービス**：ECS / EKS / ECR / EC2起動タイプ（ホスト管理あり）
> **SAAでの出題イメージ**：「コンテナのままホスト管理をなくす」ならFargate。
> 📎 参照：[[0902_オーケストレーション_フルマネージド]] / [[0901_AWS_コンテナサービス概要]]

## 第11問（コンテナレジストリ）
ビルドしたDockerイメージを、AWS内でプライベートに保管・バージョン管理し、ECS/EKSからプルしたい。適切なサービスはどれか。

- A. Amazon ECR
- B. Amazon S3
- C. AWS CodeCommit
- D. Amazon EFS

> [!success]- 答え：**A**（ECR）

> **他の選択肢が違う理由**
> - B：S3は汎用ストレージで、コンテナイメージ専用のレジストリ機能（タグ・プル認証・スキャン）を持たない。
> - C：CodeCommitはGitソースコードのリポジトリでイメージ保管ではない。
> - D：EFSは共有ファイルシステムでレジストリではない。
> **覚えるべきキーワード**：コンテナイメージ / プライベートレジストリ / イメージスキャン
> **関連サービス**：ECS / EKS / Fargate / Docker
> **SAAでの出題イメージ**：「Dockerイメージの保管・配布」ならECR。
> 📎 参照：[[0903_Amazon_ECR]]

## 第12問（IaC：構成のコード化）
VPC・EC2・RDSなどの構成をテンプレート（コード）で宣言的に定義し、同じ環境を何度でも再現・更新・削除できるようにしたい。適切なサービスはどれか。

- A. AWS CloudFormation
- B. AWS Systems Manager
- C. AWS Config
- D. AWS CloudTrail

> [!success]- 答え：**A**（CloudFormation）
> CloudFormationはインフラをテンプレート（JSON/YAML）で定義し、スタックとして一括作成・更新・削除できるIaCサービス。環境の再現性と変更管理を実現する。
> **他の選択肢が違う理由**
> - B：Systems Managerは運用管理（パッチ・パラメータ・Run Command等）で、インフラのプロビジョニング宣言が主目的ではない。
> - C：Configは構成の変更履歴・準拠チェックで、構成を作るIaCではない。
> - D：CloudTrailはAPI操作の監査ログ。
> **覚えるべきキーワード**：IaC / テンプレート / スタック / 再現性
> **関連サービス**：CDK / Terraform（サードパーティ） / Service Catalog
> **SAAでの出題イメージ**：「インフラをコードで再現・一括管理」ならCloudFormation。
> 📎 参照：[[1402_CloudFormation]]

## 第13問（運用管理・パラメータ保管）
多数のEC2へのパッチ適用やコマンド一括実行、設定値（パラメータ）の集中管理を、エージェント経由でまとめて行いたい。適切なサービスはどれか。

- A. AWS Systems Manager
- B. AWS CloudFormation
- C. Amazon CloudWatch
- D. AWS Organizations

> [!success]- 答え：**A**（Systems Manager）
> Systems Manager（SSM）は運用の統合ツール。Patch Manager（パッチ）、Run Command（一括実行）、Session Manager（踏み台なし接続）、Parameter Store（設定値・シークレットの集中管理）などを提供する。
> **他の選択肢が違う理由**
> - B：CloudFormationはインフラ構築のIaCで、日々の運用操作が主目的ではない。
> - C：CloudWatchは監視・メトリクスで、パッチ・コマンド実行は行わない。
> - D：Organizationsは複数アカウント統制で、個々のEC2運用操作はしない。
> **覚えるべきキーワード**：Patch Manager / Run Command / Session Manager / Parameter Store
> **関連サービス**：Parameter Store / Secrets Manager / CloudWatch
> **SAAでの出題イメージ**：「EC2群のパッチ・一括操作・設定集中管理」ならSystems Manager。
> 📎 参照：[[1405_Systems Manager]]

## 第14問（複数アカウント統制）
事業部ごとに分けた複数のAWSアカウントを一元管理し、一括請求（Consolidated Billing）と、SCPによる利用範囲のガードレールを設けたい。適切なサービスはどれか。

- A. AWS Organizations
- B. AWS IAM
- C. AWS Control Tower
- D. AWS Config

> [!success]- 答え：**A**（Organizations）
> Organizationsは複数AWSアカウントをOU（組織単位）で階層管理し、一括請求とSCP（サービスコントロールポリシー）による権限の上限（ガードレール）を提供する統制の土台。
> **他の選択肢が違う理由**
> - B：IAMは単一アカウント内の認可で、アカウント横断の統制は範囲外。
> - C：Control TowerはOrganizationsの上にベストプラクティスなランディングゾーンを自動構築する上位サービス（土台はOrganizations）。
> - D：Configは構成準拠チェックで、アカウント統制の土台ではない。
> **覚えるべきキーワード**：OU / SCP（ガードレール） / 一括請求 / マルチアカウント
> **関連サービス**：Control Tower / SCP / IAM Identity Center
> **SAAでの出題イメージ**：「複数アカウントを統制・一括請求・SCP」ならOrganizations。
> 📎 参照：[[1403_Organizations]]

## 第15問（Webアプリ層の保護）
公開WebアプリへのSQLインジェクションやクロスサイトスクリプティング（XSS）、特定IP/国からの不正アクセスを、リクエスト内容のルールで遮断したい。適切なサービスはどれか。

- A. AWS WAF
- B. AWS Shield
- C. セキュリティグループ
- D. ネットワークACL

> [!success]- 答え：**A**（WAF）
> WAFはL7（HTTP/HTTPS）のWebアプリケーションファイアウォール。SQLi/XSS・レートベース・IP/地理・マネージドルールでリクエスト内容を検査し、許可/ブロックする。CloudFront/ALB/API Gatewayに適用できる。
> **他の選択肢が違う理由**
> - B：ShieldはDDoS（L3/L4中心）防御で、SQLi/XSSなどアプリ層攻撃の中身は判定しない。
> - C/D：セキュリティグループ/NACLはIP・ポートの制御で、HTTPの中身（攻撃パターン）は見られない。
> **覚えるべきキーワード**：L7 / SQLi・XSS対策 / マネージドルール / レート制限
> **関連サービス**：Shield（DDoS） / CloudFront / ALB / API Gateway
> **SAAでの出題イメージ**：「Webアプリ攻撃をリクエスト内容で遮断」ならWAF。
> 📎 参照：[[1604_ネットワーク防御]]

## 第16問（DDoS防御）
公開サービスへの大規模なDDoS攻撃（SYNフラッド・UDPリフレクション等）から、ネットワーク/トランスポート層で自動的に保護したい。適切なサービスはどれか。

- A. AWS WAF
- B. AWS Shield
- C. Amazon GuardDuty
- D. AWS Config

> [!success]- 答え：**B**（Shield）
> ShieldはDDoS保護サービス。Standardは全顧客に自動適用（L3/L4の一般的なDDoSを軽減）、Advancedは高度なL7保護・24/7対応・DDoSコスト保護を追加する。
> **他の選択肢が違う理由**
> - A：WAFはアプリ層（L7）のリクエスト内容フィルタで、ボリューム型DDoSの主役ではない（併用する）。
> - C：GuardDutyは脅威検知（分析・通知）で、攻撃の自動軽減そのものではない。
> - D：Configは構成管理で防御機能ではない。
> **覚えるべきキーワード**：DDoS軽減 / Standard（自動） / Advanced（高度・コスト保護）
> **関連サービス**：WAF（L7併用） / CloudFront / Route 53 / Global Accelerator
> **SAAでの出題イメージ**：「DDoS対策＝Shield、Webアプリ攻撃＝WAF」をセットで。
> 📎 参照：[[1604_ネットワーク防御]]

## 第17問（脅威検知）
アカウント内の不審なAPI操作・不正アクセスの兆候・暗号通貨マイニング等の脅威を、CloudTrail/VPCフローログ/DNSログを機械学習で分析して継続的に検知したい。適切なサービスはどれか。

- A. Amazon GuardDuty
- B. AWS CloudTrail
- C. AWS WAF
- D. Amazon Inspector

> [!success]- 答え：**A**（GuardDuty）
> GuardDutyはマネージドな脅威検知サービス。CloudTrail・VPCフローログ・DNSログ等を機械学習と脅威インテリジェンスで継続分析し、不審な挙動を検出して通知する（エージェント不要）。
> **他の選択肢が違う理由**
> - B：CloudTrailは操作の記録（証跡）そのもので、脅威の検知・分析はしない（GuardDutyの入力源）。
> - C：WAFはWebアプリ層のリクエスト遮断で、アカウント全体の脅威分析ではない。
> - D：InspectorはEC2/コンテナ/Lambdaの脆弱性診断で、挙動ベースの脅威検知ではない。
> **覚えるべきキーワード**：脅威検知 / 機械学習 / ログ分析 / エージェント不要
> **関連サービス**：Security Hub（集約） / Detective（深掘り） / EventBridge（自動対応）
> **SAAでの出題イメージ**：「ログを分析して脅威を継続検知」ならGuardDuty。
> 📎 参照：[[1603_脅威検知・監査]]

## 第18問（構成変更の追跡・準拠チェック）
AWSリソースの構成（設定）の変更履歴を記録し、「全EBSが暗号化されているか」などのルールに継続的に準拠しているかを自動評価したい。適切なサービスはどれか。

- A. AWS CloudTrail
- B. AWS Config
- C. Amazon CloudWatch
- D. AWS Trusted Advisor

> [!success]- 答え：**B**（Config）
> Configはリソース構成のスナップショットと変更履歴を記録し、Configルールで「望ましい状態」への準拠を継続評価する。逸脱の検知・是正（Remediation）にも使える。
> **他の選択肢が違う理由**
> - A：CloudTrailは「誰がどのAPIを呼んだか」の操作証跡で、構成の状態・準拠評価はしない。
> - C：CloudWatchは性能メトリクス監視で、構成準拠チェックではない。
> - D：Trusted Advisorは横断的なベストプラクティス助言で、カスタムルールの継続評価・変更履歴の主役ではない。
> **覚えるべきキーワード**：構成変更履歴 / Configルール / 準拠評価 / 是正
> **関連サービス**：CloudTrail（操作） / Security Hub / Systems Manager（是正）
> **SAAでの出題イメージ**：「構成の変更履歴・ルール準拠チェック」ならConfig。
> 📎 参照：[[1003_AWS_Config]]

## 第19問（専用線接続）
オンプレデータセンターとAWS間を、インターネットを経由しない安定した広帯域・低遅延の専用接続で結びたい。適切なサービスはどれか。

- A. AWS Site-to-Site VPN
- B. AWS Direct Connect
- C. VPCピアリング
- D. NAT Gateway

> [!success]- 答え：**B**（Direct Connect）
> Direct Connect（DX）はオンプレとAWSを結ぶ専用線接続。インターネットを経由せず、安定した広帯域・低遅延・一貫した品質を提供する。
> **他の選択肢が違う理由**
> - A：VPNはインターネット経由の暗号化トンネルで、品質はネット依存（DXのバックアップに向く）。
> - C：VPCピアリングはVPC間接続で、オンプレ接続の手段ではない。
> - D：NAT Gatewayはプライベートサブネットの外向き通信用でオンプレ専用線ではない。
> **覚えるべきキーワード**：専用線 / 非インターネット / 広帯域・低遅延・安定
> **関連サービス**：Site-to-Site VPN（バックアップ） / Transit Gateway / Direct Connect Gateway
> **SAAでの出題イメージ**：「安定した専用接続」ならDirect Connect（冗長化にVPN併用）。
> 📎 参照：[[0405_ネットワークとコンテンツ配信]] / [[0406_VPC応用機能]]

## 第20問（バックアップの一元管理）
EBS・RDS・DynamoDB・EFS・Storage Gatewayなど複数サービスのバックアップを、ポリシーで一元的にスケジュール・保持・クロスリージョンコピーしたい。適切なサービスはどれか。

- A. AWS Backup
- B. Amazon S3
- C. RDS自動バックアップのみ
- D. EBSスナップショットを手動取得

> [!success]- 答え：**A**（AWS Backup）
> AWS Backupは複数のAWSサービスのバックアップをバックアッププランで一元管理する。スケジュール・保持期間・ライフサイクル・クロスリージョン/クロスアカウントコピーを統一ポリシーで適用できる。
> **他の選択肢が違う理由**
> - B：S3は保存先になり得るが、複数サービス横断のバックアップ管理機能そのものではない。
> - C：RDS自動バックアップはRDS単体向けで、複数サービス横断の一元管理にはならない。
> - D：手動スナップショットは運用負荷・取り忘れリスクがあり一元管理にならない。
> **覚えるべきキーワード**：一元管理 / バックアッププラン / クロスリージョン/アカウント / 保持ポリシー
> **関連サービス**：EBSスナップショット / RDSスナップショット / Organizations（横断適用）
> **SAAでの出題イメージ**：「複数サービスのバックアップを統一ポリシーで」ならAWS Backup。
> 📎 参照：[[0301_RDS]] / [[1501_移行と転送サービス一覧]]
