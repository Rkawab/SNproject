# 【問題】サービス用途確認2（公式サンプル風・全10問）

## この問題集の目的

アプリケーション統合、コンテナ、分析、移行、ガバナンス系サービスをシナリオで確認する。

10問

## 第1問（Application Integration ｜ メッセージキュー）
注文受付APIは短時間に大量のリクエストを受ける。注文処理コンポーネントは処理に時間がかかるため、API応答を遅くしないように非同期で処理したい。メッセージを一時的に保持し、処理側が自分のペースで取得できるサービスはどれか。

- A. Amazon SQS
- B. Amazon SNS
- C. Amazon EventBridge
- D. AWS Step Functions

> [!success]- 答え：**A**
> SQSはメッセージキューで、コンポーネント間を疎結合にし、負荷の急増を吸収できる。
> - A：正解。メッセージを保持し、ワーカーがポーリングして処理できる。
> - B：Pub/Sub通知であり、キューとしてためる用途とは異なる。
> - C：イベントルーティングに向くが、処理待ちキューの基本解ではない。
> - D：ワークフロー制御サービスであり、単純なキューイング要件とは異なる。
> **関連ノート**：[[0802_SQS]]

## 第2問（Application Integration ｜ ファンアウト）
ECサイトで注文が確定したら、請求処理、在庫更新、メール通知の3つのシステムへ同じイベントを届けたい。各システムは独立して処理する。最も適切なサービスはどれか。

- A. Amazon SNS
- B. Amazon SQS
- C. Amazon EventBridge
- D. Amazon Kinesis Data Streams

> [!success]- 答え：**A**
> SNSはトピックに発行されたメッセージを複数のサブスクライバーへ配信できる。
> - A：正解。複数の処理先へ同じイベントを配るファンアウトに適している。
> - B：単一キューに処理をためる用途が中心で、同一メッセージを独立した複数システムへ同時配信するファンアウトではない（SQSはSNSの配信先として組み合わせる）。
> - C：イベントのルーティング/フィルタリングや多数のAWS・SaaS連携には向くが、シンプルなPub/Subファンアウトの基本解はSNS。
> - D：ストリーミングデータの取り込み・複数コンシューマー処理向けで、イベント通知のファンアウト用途とは異なる。
> **関連ノート**：[[0803_SNS]]

## 第3問（Application Integration ｜ ワークフロー）
画像審査システムは、画像解析、手動承認、メタデータ更新、通知の複数ステップを順番に実行する。失敗時の再試行や分岐をマネージドに扱いたい。最も適切なサービスはどれか。

- A. AWS Step Functions
- B. Amazon SQS
- C. Amazon EventBridge
- D. AWS Glue

> [!success]- 答え：**A**
> Step Functionsは複数の処理を状態遷移として組み合わせ、分岐、再試行、エラー処理を管理できる。
> - A：正解。複数ステップのワークフロー制御に適している。
> - B：メッセージの保持と非同期処理には有効だが、分岐・再試行を含む状態管理の主サービスではない。
> - C：イベントのルーティングには向くが、再試行・分岐・手動承認を含む多段ワークフローの状態を一元管理するサービスではない。
> - D：ETLジョブやデータカタログに使うサービスで、汎用的なステップ制御の主サービスではない。
> **関連ノート**：[[0807_Step_Functions]]

## 第4問（Serverless ｜ サーバーレスAPI）
企業は、HTTPS APIを公開し、バックエンドのLambda関数へリクエストを渡したい。認証、スロットリング、ステージ管理もマネージドに行いたい。最も適切なサービスはどれか。

- A. Amazon API Gateway
- B. Application Load Balancer
- C. Amazon Route 53
- D. AWS App Runner

> [!success]- 答え：**A**
> API GatewayはAPIの公開、認証連携、スロットリング、Lambda統合に適している。
> - A：正解。LambdaベースのAPI公開でよく使う。
> - B：HTTPロードバランサーだが、API管理機能はAPI Gatewayほど豊富ではない。
> - C：DNSサービス。
> - D：コンテナ化Webアプリの実行サービスであり、API管理の主サービスではない。
> **関連ノート**：[[0809_API_Gateway]] / [[0708_Lambda]]

## 第5問（Containers ｜ コンテナイメージ）
開発チームは、Dockerイメージをプライベートに保存し、ECSタスクから取得して実行したい。最も適切なサービスはどれか。

- A. Amazon ECR
- B. Amazon ECS
- C. Amazon EKS
- D. AWS CodeArtifact

> [!success]- 答え：**A**
> ECRはコンテナイメージのマネージドレジストリ。
> - A：正解。Dockerイメージを保存し、ECSやEKSから取得できる。
> - B：コンテナを実行・管理するオーケストレーションサービス。
> - C：Kubernetesベースのコンテナ実行基盤。
> - D：ソフトウェアパッケージリポジトリサービス。
> **関連ノート**：[[0903_Amazon_ECR]]

## 第6問（Containers ｜ マネージドコンテナ実行）
企業はコンテナ化したWebアプリケーションをAWSで実行したい。Kubernetesの管理は不要で、AWSネイティブなオーケストレーションを使いたい。最も適切なサービスはどれか。

- A. Amazon ECS
- B. Amazon EKS
- C. Amazon ECR
- D. AWS Fargate

> [!success]- 答え：**A**
> ECSはAWSのマネージドコンテナオーケストレーションサービス。
> - A：正解。Kubernetesを使わずにコンテナを実行・スケールできる。
> - B：Kubernetesベースのオーケストレーションで、「Kubernetesの管理は不要」という要件に反する。
> - C：コンテナイメージの保存先（レジストリ）であり、オーケストレーターではない。
> - D：コンテナを動かすサーバーレスの起動タイプで、それ自体はオーケストレーション制御を担うサービスではない。
> **関連ノート**：[[0904_ECS]]

## 第7問（Analytics ｜ S3上のデータ分析）
企業はS3に保存されたCSVログを、サーバーやデータベースを用意せずSQLで随時分析したい。最も適切なサービスはどれか。

- A. Amazon Athena
- B. Amazon Redshift Spectrum
- C. Amazon EMR
- D. Amazon RDS for MySQL

> [!success]- 答え：**A**
> AthenaはS3上のデータをサーバーレスにSQLクエリできる。
> - A：正解。S3データに対するアドホックSQL分析に適している。
> - B：S3上のデータを直接クエリできるが、Redshiftクラスターの作成・管理が前提で「サーバーを用意しない」要件から外れる。
> - C：Spark/PrestoでS3データを分析できるが、クラスターのプロビジョニング・管理が必要。
> - D：事前にデータをDBへロードして使うリレーショナルDBで、S3上のCSVへ直接アドホックSQLを実行する用途ではない。
> **関連ノート**：[[1109_Amazon Athena]]

## 第8問（Analytics ｜ データウェアハウス）
分析チームは、複数システムの大量データを集約し、BIツールから高速に集計したい。長期的な分析基盤として列指向のデータウェアハウスを使いたい。最も適切なサービスはどれか。

- A. Amazon Redshift
- B. Amazon Athena
- C. Amazon RDS for PostgreSQL
- D. Amazon DynamoDB

> [!success]- 答え：**A**
> Redshiftは分析向けのマネージドデータウェアハウス。
> - A：正解。大量データの集計分析やBI用途に適している。
> - B：S3上データへのサーバーレスなアドホックSQLには向くが、繰り返し高速集計する永続的な列指向データウェアハウスとは用途が異なる。
> - C：行指向のリレーショナルDBでOLTP向け。大量データを列指向で高速集計する分析基盤ではない。
> - D：低レイテンシなNoSQLデータベースで、BIの集計分析向けデータウェアハウスではない。
> **関連ノート**：[[1113_Amazon Redshift]]

## 第9問（Migration / Transfer ｜ ファイル同期）
オンプレミスのファイルサーバーにある数十TBのデータを、継続的にAmazon S3へ同期したい。ネットワーク経由で転送し、差分同期とスケジュール実行を利用したい。最も適切なサービスはどれか。

- A. AWS DataSync
- B. AWS Snowball Edge
- C. AWS Storage Gateway
- D. Amazon Kinesis Data Streams

> [!success]- 答え：**A**
> DataSyncはオンプレミスストレージとAWSストレージ間のオンライン転送・同期を自動化する。
> - A：正解。差分同期やスケジュール転送に適している。
> - B：物理デバイスを使うオフライン移行に向く。
> - C：オンプレミスからAWSストレージを利用するハイブリッド構成に向く。
> - D：ストリーミングデータ処理サービス。
> **関連ノート**：[[1503_Storage Gateway]] / [[1502_DataSync]]

## 第10問（Management / Monitoring ｜ 複数アカウント管理）
企業は複数のAWSアカウントを一元管理し、組織単位で権限の上限や請求を管理したい。使用すべきサービスはどれか。（2つ選択）

- A. AWS Organizations
- B. Service Control Policies
- C. AWS Config
- D. AWS IAM Identity Center

> [!success]- 答え：**A, B**
> Organizationsでアカウントを組織管理し、SCPで権限の上限を設定できる。
> - A：正解。複数アカウントの組織管理と一括請求に使う。
> - B：正解。アカウントやOUに対する権限の最大範囲を制御できる。
> - C：設定の記録・準拠評価が主目的で、アカウント組織管理や請求統合の主サービスではない。
> - D：複数アカウントへのSSOアクセスを提供するが、組織単位の権限上限（SCP）や一括請求の土台を担うのはOrganizations側。
> **関連ノート**：[[1403_Organizations]]
