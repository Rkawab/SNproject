# 【問題】類似サービス比較（公式サンプル風・全10問）

## この問題集の目的

SAAで頻出の「似たサービスの使い分け」を、要件ベースで判断する。

## 問題数

10問

---

## 第1問（Storage ｜ 共有ファイルストレージ）
複数のLinux EC2インスタンスで構成されるWebアプリケーションが、同じアップロードファイルを同時に読み書きする必要がある。インスタンスの増減に合わせて共有ストレージを使い続けたい。最も適切なサービスはどれか。

- A. Amazon EFS
- B. Amazon EBS（マルチアタッチ）
- C. Amazon S3
- D. Amazon FSx for Windows File Server

> [!success]- 答え：**A**
> EFSは複数のLinuxインスタンスから同時マウントできるマネージドNFSファイルシステム。
> - A：正解。複数EC2（Linux）で共有するファイルシステム要件に合う。インスタンスの増減にも自動で追従する。
> - B：きわどい誤答。マルチアタッチは実在するが**同一AZ内・io1/io2限定**で、クラスター対応ファイルシステムも必要。「複数EC2で共有」の一般解ではない。
> - C：オブジェクトストレージはAPI経由のアクセスで、ファイルシステムとしてマウントして同時読み書きする用途には合わない。
> - D：共有ファイルシステムだが**SMB＝Windows向け**。Linux EC2の要件で切る。
> **関連ノート**：[[0202_S3とEBS・EFSの違い]] / [[0214_EFS詳細]]

## 第2問（Storage ｜ ブロックストレージ）
データベースをEC2上で実行しており、低レイテンシな永続ブロックストレージをインスタンスに接続したい。最も適切なサービスはどれか。

- A. Amazon EBS
- B. Amazon EFS
- C. Amazon S3
- D. インスタンスストア

> [!success]- 答え：**A**
> EBSはEC2にアタッチして使う永続ブロックストレージ。
> - A：正解。低レイテンシ・永続・ブロックの3条件を満たす。EC2上のDBのデータ領域の定番。
> - B：ファイルシステム（NFS）であり、ブロックデバイスとしてDBに使う用途ではない。レイテンシもEBSより大きい。
> - C：オブジェクトストレージであり、ブロックデバイスではない（マウントして使えない）。
> - D：きわどい誤答。ブロックで**最も低レイテンシ**だが、インスタンス停止でデータが消える**揮発性**。「永続」の一語で切る。
> **関連ノート**：[[0103_EC2のストレージ]] / [[0109_EBSボリュームタイプ]]

## 第3問（Database ｜ NoSQLとRDB）
アプリケーションは、ユーザーIDをキーにしてプロファイル情報をミリ秒単位で読み書きする。結合や複雑なSQLクエリは不要で、アクセス量は大きく変動する。運用負荷を最小化したい。最も適切なデータベースはどれか。

- A. Amazon DynamoDB
- B. Amazon RDS for PostgreSQL
- C. Amazon Redshift
- D. Amazon Neptune

> [!success]- 答え：**A**
> DynamoDBはキー・バリュー/ドキュメント型のマネージドNoSQLで、大規模な低レイテンシアクセスに向く。
> - A：正解。キーアクセス中心でスケール要件がある場合に適している。
> - B：SQLやリレーションが必要な場合に向く。
> - C：分析用データウェアハウス。
> - D：グラフデータベース。
> **関連ノート**：[[0303_DynamoDB]]

## 第4問（Database ｜ 読み取りスケール）
企業はRDS for MySQLを使用している。書き込みはプライマリDBで処理し、読み取りクエリの増加に対応したい。アプリケーションは読み取り専用クエリを別エンドポイントへ振り分けられる。最も適切な方法はどれか。

- A. RDSリードレプリカを追加する
- B. RDS Multi-AZを有効化する
- C. Amazon ElastiCache をDBの前段に導入する
- D. DBインスタンスを大きいインスタンスタイプへスケールアップする

> [!success]- 答え：**A**
> リードレプリカは読み取りトラフィックを分散するために使う。
> - A：正解。読み取り専用エンドポイントへクエリを振り分け、読み取り性能をスケールできる。
> - B：高可用性とフェイルオーバー目的で、読み取り分散の主解ではない。
> - C：頻繁な読み取りのキャッシュには有効だが、DB自体の読み取りを別エンドポイントへスケールする仕組みではない（アプリ改修も別途必要）。
> - D：単一インスタンスの性能を上げるだけで上限があり、読み取りを別エンドポイントへ分散する方法ではない。
> **関連ノート**：[[0301_RDS]]

## 第5問（Database ｜ 高可用性DB）
企業はRDSデータベースの可用性を高めたい。単一AZ障害時に自動フェイルオーバーできる構成を求めている。読み取り性能の向上は主目的ではない。最も適切な方法はどれか。

- A. RDS Multi-AZ配置を有効化する
- B. RDSリードレプリカを別AZに作成し、障害時に昇格させる
- C. Amazon RDS Proxyを導入して接続をプールする
- D. DBスナップショットを毎時取得する

> [!success]- 答え：**A**
> RDS Multi-AZはスタンバイDBを別AZに配置し、障害時に**自動**フェイルオーバーする。
> - A：正解。同じエンドポイントのまま自動で切り替わる。可用性要件の第一候補。
> - B：きわどい誤答。レプリカの昇格は**手動**（または自前の自動化）で、データも非同期＝直前の書き込みを失いうる。「自動フェイルオーバー」の一語で切る。
> - C：RDS Proxyは接続枯渇対策＋フェイルオーバー時間の短縮に寄与するが、フェイルオーバーの仕組みそのものはMulti-AZが提供する。
> - D：復元には使えるが、自動フェイルオーバーではない（復元には時間もかかる）。
> **関連ノート**：[[0301_RDS]] / [[0302_RDSProxy]]

## 第6問（Networking / CDN / LB ｜ ロードバランサー選択）
ゲームサーバーはTCPで低レイテンシな通信を処理する。HTTPヘッダーやパスに基づくルーティングは不要で、非常に高いスループットが必要である。最も適切なロードバランサーはどれか。

- A. Network Load Balancer
- B. Application Load Balancer
- C. Gateway Load Balancer
- D. Classic Load Balancer

> [!success]- 答え：**A**
> NLBはレイヤー4で高性能、低レイテンシなTCP/UDP/TLS分散に向く。
> - A：正解。TCPベースの高性能分散に適している。
> - B：HTTP/HTTPSのレイヤー7処理に向く。
> - C：仮想アプライアンスを経由させる用途。
> - D：旧世代で、新規設計ではALB/NLBを優先する。
> **関連ノート**：[[0503_ALB・NLB・GWLBの違い]]

## 第7問（Networking / CDN / LB ｜ プライベートなAWSサービス接続）
プライベートサブネットのEC2インスタンスからS3へアクセスする。NAT Gatewayを通るデータ転送コストを避け、VPC内の経路で接続したい。最も適切な構成はどれか。

- A. Gateway型VPCエンドポイントを作成する
- B. Interface型VPCエンドポイント（PrivateLink）を作成する
- C. NAT Gateway を経由してS3へアクセスする
- D. Internet Gateway 経由でパブリックにS3へアクセスする

> [!success]- 答え：**A**
> S3とDynamoDBにはGateway型VPCエンドポイントを使用でき、NAT Gatewayを経由せずにアクセスできる。
> - A：正解。S3向けのコスト効率がよいプライベート経路で、データ処理料金もかからない。
> - B：S3にもPrivateLinkで接続できるが、追加のENIと時間課金が発生し、S3の典型かつ無料の経路はGateway型。
> - C：S3へ到達はできるが、まさに避けたいNAT Gatewayのデータ処理料金が発生する。
> - D：インターネット経由となり、VPC内のプライベート経路でアクセスする要件に反する。
> **関連ノート**：[[0409_IGW・NAT Gateway・VPCエンドポイント]]

## 第8問（Database ｜ キャッシュ）
WebアプリケーションはRDSへの読み取りクエリが多く、同じデータを繰り返し取得している。アプリケーション変更は可能で、低レイテンシなインメモリキャッシュを導入したい。最も適切なサービスはどれか。

- A. Amazon ElastiCache
- B. Amazon DynamoDB Accelerator（DAX）
- C. Amazon RDS リードレプリカ
- D. Amazon MemoryDB for Redis

> [!success]- 答え：**A**
> ElastiCacheはRedisまたはMemcached互換のインメモリキャッシュを提供する。
> - A：正解。DB読み取り負荷の軽減と低レイテンシ化に使う。
> - B：インメモリキャッシュだがDynamoDB専用のアクセラレーターで、RDSの前段キャッシュには使えない。
> - C：読み取り負荷の分散はできるが、インメモリのマイクロ秒級キャッシュとはレイテンシ特性が異なる。
> - D：Redis互換のインメモリDBだが、キャッシュ層ではなく永続データストアの位置づけで、要件のキャッシュ導入には過剰。
> **関連ノート**：[[0304_ElastiCache]]

## 第9問（Application Integration ｜ 通知とキューの組み合わせ）
注文イベントを複数の下流システムへ配信し、各システムは処理失敗時に再試行できるように独自のキューを持ちたい。最も適切な組み合わせはどれか。（2つ選択）

- A. Amazon SNSトピック
- B. 各下流システム用のAmazon SQSキュー
- C. Amazon Kinesis Data Streams
- D. Amazon EventBridge

> [!success]- 答え：**A, B**
> SNSから複数のSQSキューへ配信するファンアウト構成は、疎結合と再試行に適している。
> - A：正解。イベントを複数購読者へ配信できる。
> - B：正解。各システムが自分のペースで処理でき、失敗時の再試行にも使える。
> - C：ストリーミングデータの取り込み向けで、下流ごとに独立した再試行キューを持つ構成の部品ではない。
> - D：イベントのルーティングには使えるが、各システムが自分のペースで再試行できる専用キューを提供する役割はSQSが担う。
> **関連ノート**：[[0802_SQS]] / [[0803_SNS]]

## 第10問（Analytics ｜ リアルタイムストリーミング）
IoTデバイスから継続的に送信されるセンサーデータを、複数のコンシューマーでほぼリアルタイムに処理したい。データストリームとして取り込み、順序を保ちながら処理する必要がある。最も適切なサービスはどれか。

- A. Amazon Kinesis Data Streams
- B. Amazon Kinesis Data Firehose
- C. Amazon SQS 標準キュー
- D. Amazon SQS FIFOキュー

> [!success]- 答え：**A**
> Kinesis Data Streamsはストリーミングデータの取り込みとリアルタイム処理に使う。
> - A：正解。継続的なデータストリームを、複数コンシューマーが順序を保ちつつ並行処理・再処理できる。
> - B：ストリームを保存先へ自動ロードするマネージドサービスで、複数コンシューマーが順序を保って独自に読み出す用途には向かない。
> - C：順序保証がなく、複数コンシューマーが同じストリームを並行処理する用途には適さない。
> - D：順序は保てるが1対1のメッセージ消費が基本で、複数コンシューマーが同じデータを並行・再処理するストリーム処理とは異なる。
> **関連ノート**：[[1103_Amazon Kinesis]]
