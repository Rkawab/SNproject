# 【問題】類似サービス比較（ストレージ編・全10問）

## この問題集の目的

第2版（2026-07-14全面新作）。ストレージサービスの「似た者同士」を要件文で切り分ける。誤答はすべて実在の構成で、**別の要件なら正解になるもの**だけを置く。解説では「どの要件ならその選択肢が勝つか」まで書くので、間違えた選択肢からも学べる。

## 対象レベル

SAA。サービス名は知っている前提で、要件の決め手ワードから最も適切な1つを選ぶ。

## 問題数

10問

## 第1問（Storage ｜ Windowsファイル共有の移行）
オンプレミスのWindowsファイルサーバーをAWSへ移行したい。数百名の利用者がSMBプロトコルでアクセスし、Active Directory によるアクセス権管理とNTFSのアクセス制御リストを維持する必要がある。最も適切なサービスはどれか。

- A. Amazon EFS
- B. Amazon FSx for Windows File Server
- C. Amazon FSx for Lustre
- D. Amazon S3 File Gateway

> [!success]- 答え：**B**
> FSx for Windows File ServerはSMB・Active Directory統合・NTFS ACLをネイティブサポートするフルマネージドのWindowsファイルサーバー。「SMB」「AD」「NTFS」が決め手。
> - A：LinuxからNFSで共有するなら正解になる。ただしNFS専用で、SMB・NTFS ACLは扱えない。
> - B：正解。Windowsファイルサーバーの移行先はこれ一択。
> - C：HPCや機械学習で高スループットの並列処理をするなら正解になる。用途がファイルサーバーではない。
> - D：オンプレにアプリを残したままS3をSMB/NFSで見せるハイブリッド構成なら正解になる。本問はサーバーごとAWSへ移行するのでゲートウェイを挟む理由がない。
> **関連ノート**：[[0213_FSxファミリー4種の使い分け]] / [[0202_S3とEBS・EFSの違い]]

## 第2問（Storage ｜ 共有ブロックストレージ）
同一アベイラビリティゾーン内の複数のEC2インスタンス（Nitroベース）から、クラスター対応の独自ファイルシステムで単一のブロックデバイスへ同時に読み書きするフェイルオーバークラスタを構築したい。最も適切な構成はどれか。

- A. io2ボリュームのEBSマルチアタッチ
- B. Amazon EFS
- C. 各インスタンスにgp3ボリュームを個別アタッチし、アプリでレプリケーション
- D. Amazon FSx for Lustre

> [!success]- 答え：**A**
> EBSマルチアタッチは、同一AZ内の複数のNitroインスタンスへ1つのio1/io2ボリュームを同時アタッチできる。**ブロックレベル**の共有が必要なクラスタ用途はこれ。
> - A：正解。「ブロックデバイスを共有」「同一AZ」「クラスタ対応ファイルシステム」が決め手。
> - B：複数インスタンス・複数AZから**ファイルレベル**（NFS）で共有するなら正解になる。ブロックデバイスとしては見せられない。
> - C：レプリケーション遅延と整合性管理をアプリが背負う自前構成。共有ボリュームが不要な独立ノードなら普通の構成だが、単一デバイスの共有にはならない。
> - D：HPC向けの高性能**ファイル**システムであり、ブロックデバイスではない。
> **関連ノート**：[[0103_EC2のストレージ]] / [[0109_EBSボリュームタイプ]]

## 第3問（Storage ｜ アーカイブの取り出し速度）
医療画像を長期保存する。アクセスは四半期に数回程度と稀だが、必要になったときは**ミリ秒**で取り出せなければならない。ストレージコストは可能な限り抑えたい。最も適切なS3ストレージクラスはどれか。

- A. S3標準-IA
- B. S3 Glacier Flexible Retrieval
- C. S3 Glacier Instant Retrieval
- D. S3 Glacier Deep Archive

> [!success]- 答え：**C**
> Glacier Instant Retrievalは「ほぼアクセスしないがミリ秒取り出しが必須」のための最安クラス。標準-IAより約4割安く、取り出しは即時。
> - A：月1回程度アクセスするなら正解になる。ミリ秒取り出しは満たすが、四半期数回の頻度に対して保存単価が高い。
> - B：取り出しが数分〜数時間待てるなら正解になり、保存単価はさらに下がる。「ミリ秒」の要件で脱落。
> - C：正解。「稀なアクセス × 即時取り出し × 最安」の交点。
> - D：取り出しに12時間程度かかってよい規制アーカイブなら正解になる最安クラス。即時性で脱落。
> **関連ノート**：[[0203_S3ストレージクラス]]

## 第4問（Storage ｜ 規制対応の最安アーカイブ）
法規制により契約書類を7年間保存する。取り出しは監査時のみで、依頼から**12時間以内**に応じられればよい。ストレージ単価を最小化したい。最も適切なS3ストレージクラスはどれか。

- A. S3 Glacier Flexible Retrieval
- B. S3 Intelligent-Tiering
- C. S3標準-IA
- D. S3 Glacier Deep Archive

> [!success]- 答え：**D**
> Deep ArchiveはS3最安のクラスで、標準取り出しは12時間以内。「年単位保存 × 取り出しは半日待てる」なら迷わずこれ。
> - A：取り出しを数分〜数時間で行いたいなら正解になる。Deep Archiveより保存単価が高い。
> - B：アクセスパターンが読めず自動でコスト最適化したいなら正解になる。パターンが「ほぼゼロ」と確定している本問では、最初からDeep Archiveへ置く方が安い。
> - C：月次程度でアクセスし即時取り出しが要るなら正解になる。監査時のみの用途では割高。
> - D：正解。「7年保存」「12時間以内でよい」が決め手。
> **関連ノート**：[[0203_S3ストレージクラス]] / [[0204_S3のライフサイクル管理]]

## 第5問（Storage ｜ EBSボリュームタイプ）
ログ集計基盤のEC2で、大きなファイルをシーケンシャルに大量読み書きする。IOPSよりも**スループット（MB/秒）**が性能を決めるワークロードで、コストも抑えたい。アクセスは日中頻繁にある。最も適切なEBSボリュームタイプはどれか。

- A. gp3（汎用SSD）
- B. st1（スループット最適化HDD）
- C. io2（プロビジョンドIOPS SSD）
- D. sc1（コールドHDD）

> [!success]- 答え：**B**
> st1はビッグデータ・ログ処理などのシーケンシャル大容量アクセス向けに、HDDながら高スループットを安価に提供する。「スループット重視 × 頻繁アクセス × 低コスト」の決め手。
> - A：小さなランダムI/Oが混ざる汎用用途なら正解になる。GB単価はst1より高く、大容量のシーケンシャル用途では割高。
> - B：正解。
> - C：トランザクションDBのような高IOPS・低レイテンシ要件なら正解になる。本問はIOPSを求めていない。
> - D：同じHDDでも**アクセス頻度が低い**アーカイブ寄りのデータなら正解になる最安タイプ。スループット上限が低く、日中頻繁に読むワークロードでは足りない。
> **関連ノート**：[[0109_EBSボリュームタイプ]]

## 第6問（Storage ｜ 一時データの超高速ストレージ）
分散処理クラスターの中間シャッフルデータ置き場として、数百万IOPS級の極めて高速なストレージが必要である。データは再計算で復元でき、インスタンス停止時に消えても問題ない。最もコスト効率よく性能要件を満たすのはどれか。

- A. インスタンスストア
- B. io2 Block Express
- C. gp3を複数本ストライピング
- D. Amazon EFS（Max I/Oモード）

> [!success]- 答え：**A**
> インスタンスストアはホストに物理接続されたNVMeで、EBSを超えるIOPS・最低レイテンシを追加料金なしで提供する。「消えてよい一時データ」が決め手。
> - A：正解。揮発性という弱点が、本問では欠点にならない。
> - B：**永続化が必要**な高性能DBなら正解になる。性能は高いが有償で、消えてよいデータに使うのは過剰投資。
> - C：実在するテクニックだが、ストライピングしてもネットワーク経由のEBSはローカルNVMeのレイテンシに届かず、コストも本数分かかる。
> - D：多数のインスタンスから共有したい分析データなら正解になる。共有ファイルシステムであり、単一ノードの超高速スクラッチ領域には向かない。
> **関連ノート**：[[0103_EC2のストレージ]]

## 第7問（Storage ｜ 削除・改ざんの禁止）
金融規制により、取引記録オブジェクトを保存後5年間、**管理者を含む誰も削除・上書きできない**状態でS3に保管しなければならない。最も適切な構成はどれか。

- A. バージョニング＋MFA Delete
- B. S3 Object Lock（ガバナンスモード）
- C. S3 Object Lock（コンプライアンスモード）
- D. バケットポリシーで削除操作をDeny

> [!success]- 答え：**C**
> Object Lockのコンプライアンスモードは、保持期間中はrootユーザーでも削除・短縮ができない真のWORM。「誰も」が決め手。
> - A：定番のひっかけ。誤削除の**ハードルを上げる**には有効で、その目的なら正解になる。ただしMFAを持つ管理者は削除できるため「誰もできない」は満たさない。
> - B：特別なIAM権限（s3:BypassGovernanceRetention）を持つ管理者にはバイパスを許したい社内運用なら正解になる。規制対応の「例外なし」では不足。
> - C：正解。規制文言（SEC Rule 17a-4等）への対応はコンプライアンスモード。
> - D：ポリシーは管理者が後から変更・削除できるため、保証にならない。一時的な運用ガードとしては実在する手。
> **関連ノート**：[[0205_S3のバージョン管理]] / [[0208_S3バケットポリシー]]

## 第8問（Storage ｜ EFSのコスト最適化）
Amazon EFS上の共有データが増え続けており、分析すると9割のファイルは30日以上アクセスされていない。アプリからのアクセス方法は変えずにストレージコストを削減したい。最も適切な方法はどれか。

- A. DataSyncで古いファイルをS3へ退避する
- B. EFSライフサイクル管理でIA/アーカイブストレージクラスへ自動移行する
- C. EFS One Zoneストレージクラスへ移行する
- D. S3ライフサイクルポリシーを設定する

> [!success]- 答え：**B**
> EFSライフサイクル管理は、指定期間アクセスのないファイルを低コストのEFS IA／アーカイブ階層へ自動移動する。ファイルパスは変わらず、アプリは無変更。
> - A：もうEFS上に置く必要がないコールドデータの**退避**なら正解になる。ただしファイルがEFSから消えるため、アプリのアクセス方法を変えない要件に反する。
> - B：正解。「アクセスされないファイルが多い × アプリ無変更」が決め手。
> - C：可用性要件が緩い開発環境などでコストを下げるなら正解になる（マルチAZ→単一AZ）。アクセス頻度に基づく削減ではなく、既存データの移行も作り直しになる。
> - D：S3のオブジェクトにしか効かない定番の取り違え。EFSにはEFS自身のライフサイクル管理を使う。
> **関連ノート**：[[0214_EFS詳細]] / [[0204_S3のライフサイクル管理]]

## 第9問（Storage ｜ サーバー構成の複製）
チューニング済みのEC2インスタンス（OS設定・ミドルウェア・アプリ導入済み）と同一構成のインスタンスを、別リージョンでも起動できるようにしたい。最も適切な方法はどれか。

- A. AMIを作成して対象リージョンへコピーし、そこから起動する
- B. ルートボリュームのEBSスナップショットを対象リージョンへコピーする
- C. AWS Backupでインスタンスをバックアップする
- D. CloudFormationテンプレートで同一構成を再構築する

> [!success]- 答え：**A**
> AMIはブロックデバイスマッピングや起動情報まで含む「起動可能なテンプレート」で、リージョン間コピーに対応。コピー先でそのまま起動できる。
> - A：正解。「構築済みサーバーの複製起動」はAMI。
> - B：一番きわどい誤答。スナップショット自体はコピーできるが、起動するには結局そこからAMIを登録する手順が要る。ボリューム単体の複製・復元なら正解になる。
> - C：バックアップ・復元の統制（保持ポリシーや復旧）が目的なら正解になる。複製起動のテンプレート配布が主目的ならAMIが直接的。
> - D：構成をコードで管理しており再現構築を自動化したいなら正解になる（むしろベストプラクティス）。ただし本問は「構築済みの状態」を複製したいので、テンプレートを今から書くのは遠回り。
> **関連ノート**：[[0104_スナップショット・AMIとは]]

## 第10問（Storage ｜ 遠隔地からのアップロード高速化）
世界各地の拠点から、東京リージョンのS3バケットへ毎日数GBのファイルをアップロードしている。遠距離拠点ほど転送が遅く不安定である。アプリはS3のAPIをそのまま使い続けたい。最も適切な改善策はどれか。

- A. S3 Transfer Accelerationを有効化する
- B. マルチパートアップロードに切り替える
- C. AWS Snowball Edgeを各拠点に常備する
- D. 各拠点にAWS DataSyncエージェントを導入する

> [!success]- 答え：**A**
> Transfer Accelerationは最寄りのCloudFrontエッジで受けてAWSバックボーン経由でバケットへ運ぶ。バケット名のエンドポイントを変えるだけでS3 APIはそのまま。「遠距離 × S3 APIのまま」が決め手。
> - A：正解。
> - B：大きな単一ファイルの転送効率化・再開性なら正解になる（併用も推奨）。ただし経路はインターネットのままなので、遠距離の遅延・不安定さ自体は解消しない。
> - C：ネットワークが細すぎる拠点からペタバイト級を運ぶなら正解になる。毎日数GBの定常転送に物理輸送は合わない。
> - D：拠点のNASからの定期同期案件なら正解になる。ただしエージェント導入と転送タスク運用が必要で、「S3 APIをそのまま」の要件に合わない。
> **関連ノート**：[[0211_S3パフォーマンス最適化]] / [[1502_DataSync]]
