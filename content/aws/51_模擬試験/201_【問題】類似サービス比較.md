# 【問題】類似サービス比較（4択・全20問）

## この問題集の目的

試験や実務で混同しやすいサービスの「違い」を、要件から選び分けられるようにする。選択肢で迷ったときに「用途の違い」で正解を絞り込む力をつける。

## 対象レベル

基礎〜SAA入門。各問で「なぜ似た他サービスではないか」を必ず確認する。

## 問題数

20問（S3/EBS/EFS・EC2/ECS/Lambda・RDS/Aurora/DynamoDB・ALB/NLB/API GW・CloudWatch/CloudTrail・SG/NACL・IAMユーザー/ロール/ポリシー・Route53/CloudFront・Auto Scaling/ELB・Multi-AZ/リードレプリカ・SQS/SNS/EventBridge・KMS/Secrets Manager/Parameter Store）

## 使い方

正解の根拠だけでなく、**外した似たサービスがどんな場面なら正解になるか**をセットで覚える。比較表を自分で書けるようになるのがゴール。

## 第1問（S3 / EBS / EFS）
3台のEC2（Linux）から同じディレクトリのファイルを同時に読み書きして共有したい。最も適切なストレージはどれか。

- A. 各EC2にEBSを1本ずつ付ける
- B. Amazon EFS
- C. Amazon S3 にファイルを置く
- D. インスタンスストア

> [!success]- 答え：**B**（EFS）
> 「複数EC2から同時マウントしてファイル共有」はEFS（NFSの共有ファイルシステム）の代表用途。
> **他の選択肢が違う理由**
> - A：EBSは原則1台に占有されるブロックストレージ。各台に別々のディスクを付けても「共有」にはならない。
> - C：S3はオブジェクトストレージで、POSIX的なファイル同時読み書き共有には向かない（用途が違う）。
> - D：インスタンスストアは揮発性のローカルディスクで共有不可。
> **覚えるべきキーワード**：EFS＝共有ファイル / EBS＝1台占有ブロック / S3＝オブジェクト
> **関連サービス**：EFS / EBS / S3
> **SAAでの出題イメージ**：「複数台で同時共有」=EFS、「1台のディスク」=EBS、「倉庫」=S3。
> 📎 参照：[[0202_S3とEBS・EFSの違い]]

## 第2問（S3 / EBS / EFS）
1台のEC2で動くデータベースの、低遅延で永続的なデータ保存先が必要。最も適切なストレージはどれか。

- A. Amazon S3
- B. Amazon EBS（io2など）
- C. Amazon EFS
- D. Amazon S3 Glacier

> [!success]- 答え：**B**（EBS）
> 単一EC2のDBデータには、低遅延ブロックストレージのEBS（高IOPSが要るならio2）が適切。
> **他の選択肢が違う理由**
> - A：S3はオブジェクトストレージで、DBのデータファイル用ブロックデバイスにはならない。
> - C：EFSは共有ファイル用途で、単一DBの低遅延ブロック要件には過剰／不向き。
> - D：Glacierはアーカイブ用で即時の読み書きに向かない。
> **覚えるべきキーワード**：ブロックストレージ / 低遅延 / IOPS
> **関連サービス**：EBS / インスタンスストア（揮発・高速）
> **SAAでの出題イメージ**：「単一EC2のDBディスク・低遅延」=EBS。
> 📎 参照：[[0103_EC2のストレージ]] / [[0202_S3とEBS・EFSの違い]]

## 第3問（EC2 / ECS / Lambda）
画像アップロードのたびに数秒で終わるサムネイル生成を、サーバー管理なし・実行回数分だけの課金で動かしたい。最適なコンピューティングはどれか。

- A. Amazon EC2
- B. Amazon ECS（EC2起動タイプ）
- C. AWS Lambda
- D. AWS Elastic Beanstalk

> [!success]- 答え：**C**（Lambda）
> 短時間・イベント駆動・実行時間課金・インフラ管理不要にぴったりなのはLambda。
> **他の選択肢が違う理由**
> - A：EC2は常時起動・起動時間課金でサーバー管理が必要。短い単発処理には過剰。
> - B：ECS（EC2型）はホスト（EC2）管理が残り、イベント単発処理には重い。
> - D：BeanstalkはWebアプリのPaaS的デプロイ基盤で、イベント単発処理向きではない。
> **覚えるべきキーワード**：サーバーレス / イベント駆動 / 15分上限 / 実行時間課金
> **関連サービス**：S3イベント / API Gateway / Step Functions
> **SAAでの出題イメージ**：「短時間・イベント・管理なし」=Lambda。
> 📎 参照：[[0702_Lambda・ECS・RDS・Beanstalk比較]]

## 第4問（EC2 / ECS / Lambda）
Dockerコンテナで作った常時稼働のマイクロサービス群を、ホストのサーバー管理なしで動かしたい。最適な構成はどれか。

- A. AWS Lambda
- B. Amazon ECS on AWS Fargate
- C. 1台の大きなEC2に全コンテナを手動配置する
- D. Amazon EKS を自己管理のEC2ノードグループで運用する

> [!success]- 答え：**B**（ECS on Fargate）
> コンテナの常時稼働オーケストレーションで、かつホスト管理を不要にするならECS（またはEKS）on Fargate。Fargateはホスト（EC2）のプロビジョニング・パッチ・スケールが不要。
> **他の選択肢が違う理由**
> - A：Lambdaは最大15分・イベント駆動向けで、常時稼働の長期サービスには本来不向き。
> - C：単一EC2への手動配置はオーケストレーション（自動再起動・スケール・ローリング更新）がなく、可用性・運用で劣る。
> - D：EKSでも自己管理のEC2ノードはホストのパッチ・スケール管理が残る。ホスト管理をなくすならFargate。
> **覚えるべきキーワード**：コンテナ / オーケストレーション / Fargate（サーバーレス）
> **関連サービス**：ECS / EKS / ECR / Fargate
> **SAAでの出題イメージ**：「コンテナを常時・管理なし」=ECS/EKS on Fargate。
> 📎 参照：[[0901_AWS_コンテナサービス概要]] / [[0902_オーケストレーション_フルマネージド]]

## 第5問（RDS / Aurora / DynamoDB）
MySQL互換で、より高いスループットと可用性（ストレージ自動拡張・高速フェイルオーバー）を、できるだけマネージドに使いたい。最適なDBはどれか。

- A. EC2上に構築したMySQL
- B. Amazon Aurora（MySQL互換）
- C. Amazon DynamoDB
- D. Amazon Redshift

> [!success]- 答え：**B**（Aurora）
> AuroraはMySQL/PostgreSQL互換のクラウド最適化RDB。標準RDSより高性能・高可用（共有ストレージ・高速フェイルオーバー・自動拡張）。
> **他の選択肢が違う理由**
> - A：EC2自前MySQLは運用負荷が大きく、Auroraの高可用機構も得られない。
> - C：DynamoDBはNoSQL（キー値）でMySQL互換ではない。
> - D：Redshiftは分析（OLAP）用DWHで、トランザクション処理用途ではない。
> **覚えるべきキーワード**：MySQL/PostgreSQL互換 / 高性能 / 高速フェイルオーバー
> **関連サービス**：RDS / Aurora Global Database / Aurora Serverless
> **SAAでの出題イメージ**：「RDBで高性能・高可用・互換性」=Aurora。
> 📎 参照：[[0301_RDS]]

## 第6問（RDS / Aurora / DynamoDB）
スキーマが柔軟で、キーによる超低遅延アクセスと予測不能なスパイクへの自動スケールが要る。最適なDBはどれか。

- A. Amazon RDS for PostgreSQL
- B. Amazon Aurora
- C. Amazon DynamoDB
- D. Amazon RDS for MySQL（Multi-AZ）

> [!success]- 答え：**C**（DynamoDB）
> キー値・スキーマ柔軟・1桁ミリ秒・オンデマンドで自動スケールはDynamoDBの独擅場。
> **他の選択肢が違う理由**
> - A・B・D：いずれもリレーショナルDBで、固定スキーマ寄り・スパイク追従はDynamoDBに劣る。可用性は上げられるが「キー値・サーバーレス・急増対応」の最適解ではない。
> **覚えるべきキーワード**：NoSQL / キー値 / オンデマンド / スキーマレス
> **関連サービス**：DAX / グローバルテーブル
> **SAAでの出題イメージ**：「キー値・超低遅延・スパイク・サーバーレス」=DynamoDB。
> 📎 参照：[[0303_DynamoDB]]

## 第7問（ALB / NLB / API Gateway）
極めて高いスループットと超低遅延が必要で、固定IP（静的IP）でTCPトラフィックを負荷分散したい。最適なサービスはどれか。

- A. Application Load Balancer
- B. Network Load Balancer
- C. Amazon API Gateway
- D. Amazon CloudFront

> [!success]- 答え：**B**（NLB）
> NLBはL4（TCP/UDP）の超高性能ロードバランサーで、静的IP（Elastic IP割当）に対応。低遅延・高スループット・固定IP要件はNLB。
> **他の選択肢が違う理由**
> - A：ALBはL7（HTTP）でパスベース等が得意だが、超低遅延・固定IP・非HTTPには不向き。
> - C：API GatewayはREST/HTTP APIの管理用で、L4の高速負荷分散とは目的が違う。
> - D：CloudFrontはCDN。
> **覚えるべきキーワード**：L4 / TCP/UDP / 超低遅延 / 静的IP
> **関連サービス**：ALB（L7） / GWLB（FW挿入） / Global Accelerator
> **SAAでの出題イメージ**：「超低遅延・固定IP・TCP」=NLB、「HTTPパス振り分け」=ALB。
> 📎 参照：[[0503_ALB・NLB・GWLBの違い]]

## 第8問（ALB / NLB / API Gateway）
モバイルアプリ向けのREST APIを公開し、認証・スロットリング（流量制限）・バージョン管理をマネージドに行いたい。最適なサービスはどれか。

- A. Application Load Balancer
- B. Network Load Balancer
- C. Amazon API Gateway
- D. Amazon Route 53

> [!success]- 答え：**C**（API Gateway）
> API GatewayはマネージドなAPIフロント。認証（Cognito/IAM/Lambdaオーソライザ）、スロットリング、ステージ/バージョン管理、Lambda連携などAPI運用機能を備える。
> **他の選択肢が違う理由**
> - A：ALBもLambda/HTTPを裁けるが、APIキー・スロットリング・SDK生成等のAPI管理機能はAPI Gatewayが本職。
> - B：NLBはL4分散でAPI管理機能を持たない。
> - D：Route 53はDNSでAPI管理ではない。
> **覚えるべきキーワード**：REST/HTTP API / スロットリング / オーソライザ / ステージ
> **関連サービス**：Lambda / Cognito / WAF
> **SAAでの出題イメージ**：「マネージドAPI・認証・流量制限」=API Gateway。
> 📎 参照：[[0809_API_Gateway]]

## 第9問（CloudWatch / CloudTrail）
セキュリティ調査のため、「特定のIAMユーザーが先週どのAPI操作を行ったか」を確認したい。使うべきサービスはどれか。

- A. Amazon CloudWatch メトリクス
- B. AWS CloudTrail
- C. Amazon CloudWatch Logs Insights（アプリログ）
- D. AWS Trusted Advisor

> [!success]- 答え：**B**（CloudTrail）
> 「誰がいつどのAPIを呼んだか」はCloudTrailのイベント履歴で追える。
> **他の選択肢が違う理由**
> - A：CloudWatchメトリクスはCPU等の性能数値で、操作者の記録ではない。
> - C：アプリのログ解析であって、AWS API操作の証跡ではない。
> - D：Trusted Advisorはベストプラクティス点検で、操作履歴は見ない。
> **覚えるべきキーワード**：API操作履歴 / 証跡 / 監査
> **関連サービス**：CloudWatch（性能） / Config（構成変更） / Athena（証跡をSQL分析）
> **SAAでの出題イメージ**：「誰が操作したか」=CloudTrail、「性能の数値」=CloudWatch。
> 📎 参照：[[1002_CloudTrail]] / [[1001_CloudWatch]]

## 第10問（Security Group / NACL）
特定のIPアドレス範囲からの通信を、サブネット全体に対してステートレスに「拒否（Deny）」したい。適切な仕組みはどれか。

- A. セキュリティグループのインバウンドで該当IPを拒否
- B. ネットワークACL（NACL）で該当IPをDeny
- C. ルートテーブルで該当IPの経路を削除
- D. IAMポリシーで該当IPを拒否

> [!success]- 答え：**B**（NACL）
> NACLはサブネット単位・ステートレスで、明示的なDenyルールが書ける。特定IPをサブネット全体で遮断する用途に合う。
> **他の選択肢が違う理由**
> - A：セキュリティグループはインスタンス単位・ステートフルで「許可」のみ（Denyルール不可）。
> - C：ルートテーブルは経路制御であり「拒否」の仕組みではない。
> - D：IAMはAWS API認可で、ネットワーク層の通信制御ではない。
> **覚えるべきキーワード**：NACL＝サブネット/ステートレス/Deny可 / SG＝インスタンス/ステートフル/許可のみ
> **関連サービス**：セキュリティグループ / VPC / ルートテーブル
> **SAAでの出題イメージ**：「サブネット全体でIP拒否・ステートレス」=NACL。
> 📎 参照：[[0408_セキュリティグループとNACL]]

## 第11問（Security Group / NACL）
EC2インスタンス単位で「特定ポートのインバウンドを許可」する、最も基本的で推奨される設定はどれか。なお戻りの通信は自動で許可されるべきである。

- A. セキュリティグループのインバウンドルールで許可
- B. NACLでインバウンドとアウトバウンドを個別に許可
- C. ルートテーブルにエントリを追加
- D. IAMポリシーでポートを許可

> [!success]- 答え：**A**（セキュリティグループ）
> SGはインスタンス単位・ステートフル。インバウンドを許可すれば戻りの通信は自動で許可される。基本はSGで制御する。
> **他の選択肢が違う理由**
> - B：NACLはステートレスなので戻り通信も別途許可が必要。インスタンス単位の基本制御はSGが適切。
> - C：ルートテーブルは経路制御でポート許可の仕組みではない。
> - D：IAMはネットワークのポート制御をしない。
> **覚えるべきキーワード**：ステートフル / インスタンス単位 / 戻り自動許可
> **関連サービス**：NACL（サブネット・ステートレス） / VPC
> **SAAでの出題イメージ**：「インスタンス単位・戻り自動許可」=SG。
> 📎 参照：[[0408_セキュリティグループとNACL]]

## 第12問（IAMユーザー / ロール / ポリシー）
EC2上のアプリからS3へアクセスさせる。認証情報（アクセスキー）をEC2内に置かずに権限を渡す、最も安全な方法はどれか。

- A. IAMユーザーのアクセスキーをEC2の環境変数に設定する
- B. EC2にIAMロール（インスタンスプロファイル）をアタッチする
- C. IAMユーザーのアクセスキーをSecrets Managerに保存し、EC2から取得して使う
- D. S3バケットポリシーで、当該EC2のパブリックIPからのアクセスを許可する

> [!success]- 答え：**B**（IAMロール）
> AWSサービス（EC2/Lambda等）への権限付与は、長期キーを置かずにIAMロールで行うのが鉄則。EC2にロールをアタッチすると、一時認証情報が自動発行・自動ローテーションされ、コードにキーを持たせずに済む。
> **他の選択肢が違う理由**
> - A：長期アクセスキーをEC2に置くと漏洩・ローテーション漏れのリスク（アンチパターン）。
> - C：Secrets Managerに入れても結局は長期キーの管理が残る。サービスへの権限は一時認証のIAMロールが安全。
> - D：バケットポリシーのIP許可は送信元IP依存でなりすまし・動的IPに弱く、「誰に権限を与えるか」の認可設計として脆弱。プリンシパルに権限を与えるIAMロールが正道。
> **覚えるべきキーワード**：IAMロール / 一時認証情報 / インスタンスプロファイル
> **関連サービス**：IAMポリシー（権限内容） / STS（一時認証） / IAMユーザー（人間用）
> **SAAでの出題イメージ**：「サービスに権限・キーを置かない」=IAMロール。
> 📎 参照：[[0602_IAMロール詳細]] / [[0601_IAM基礎]]

## 第13問（IAMユーザー / ロール / ポリシー）
別のAWSアカウント（取引先）のユーザーに、自社アカウント内の特定リソースへ一時的にアクセスさせたい。最適な仕組みはどれか。

- A. 取引先用のIAMユーザーを自社アカウントに作り、長期アクセスキーを渡す
- B. クロスアカウントのIAMロールを作成し、AssumeRoleで引き受けてもらう
- C. リソースのリソースベースポリシーで取引先アカウントを直接許可する
- D. 取引先のIAMユーザーARNを自社のIAMグループに追加する

> [!success]- 答え：**B**（クロスアカウントロール）
> 他アカウントへの一時アクセス委譲はIAMロール＋AssumeRole（STS）が定石。相手は自アカウントのロールを「引き受け」て短命の一時認証情報を得る。長期キーを共有せず、信頼ポリシーで相手アカウントを限定できる。
> **他の選択肢が違う理由**
> - A：他社へ長期キーを渡すのは漏洩・失効管理の面でアンチパターン。一時認証のロールが安全。
> - C：S3等のリソースベースポリシーでの他アカウント許可は一部サービスで可能だが、多様なリソースへ最小権限で一時委譲するならクロスアカウントロールが汎用的。
> - D：他アカウントのIAMユーザーを自社のIAMグループには追加できない（IAMはアカウント内で完結）。委譲はロールのAssumeRoleで行う。
> **覚えるべきキーワード**：クロスアカウントロール / AssumeRole / 信頼ポリシー / STS
> **関連サービス**：IAMロール / Organizations / Identity Center
> **SAAでの出題イメージ**：「別アカウントへ一時アクセス委譲」=ロール＋AssumeRole。
> 📎 参照：[[0602_IAMロール詳細]]

## 第14問（Route 53 / CloudFront）
世界中のユーザーへ静的コンテンツを低遅延配信し、オリジンの負荷も下げたい。中心となるサービスはどれか。

- A. Amazon Route 53（レイテンシルーティング）
- B. Amazon CloudFront
- C. AWS Global Accelerator
- D. Elastic Load Balancing

> [!success]- 答え：**B**（CloudFront）
> CloudFrontはエッジでキャッシュして低遅延配信し、オリジン負荷も軽減するCDN。静的コンテンツ配信の主役。
> **他の選択肢が違う理由**
> - A：Route 53は名前解決の最適化（最寄りへ誘導）はするが、コンテンツをキャッシュ配信しない。
> - C：Global Acceleratorは通信経路を高速化するがキャッシュはしない（動的・非HTTP向け）。
> - D：ELBはリージョン内の負荷分散でグローバル配信ではない。
> **覚えるべきキーワード**：CDN / エッジキャッシュ / オリジン負荷軽減
> **関連サービス**：S3（オリジン） / Route 53（DNS） / WAF
> **SAAでの出題イメージ**：「キャッシュして低遅延配信」=CloudFront、「名前解決」=Route 53。
> 📎 参照：[[0405_ネットワークとコンテンツ配信]] / [[0404_Route53]]

## 第15問（Auto Scaling / ELB）
「複数のEC2へトラフィックを振り分ける役割」と「負荷に応じてEC2の台数を増減する役割」。前者を担うのはどれか。

- A. EC2 Auto Scaling
- B. Elastic Load Balancing（ELB）
- C. Amazon CloudWatch
- D. AWS Lambda

> [!success]- 答え：**B**（ELB）
> ELBは受け取ったトラフィックを健全なEC2へ「振り分ける」役割。台数の増減はAuto Scalingの役割で、両者はペアで使う。
> **他の選択肢が違う理由**
> - A：Auto Scalingは台数の増減（スケールアウト/イン）担当で、振り分けはしない。
> - C：CloudWatchは監視・トリガー提供で、振り分けも台数増減も直接は行わない。
> - D：Lambdaは無関係（別の実行基盤）。
> **覚えるべきキーワード**：ELB＝振り分け / Auto Scaling＝台数増減 / 役割分担
> **関連サービス**：ALB/NLB / Auto Scaling Group / CloudWatchアラーム
> **SAAでの出題イメージ**：「振り分け」=ELB、「台数調整」=Auto Scaling（セットで高可用）。
> 📎 参照：[[0501_高可用性とスケーラビリティ]] / [[0504_Auto_Scaling_と_Auto_Scaling_Group]]

## 第16問（RDS Multi-AZ / リードレプリカ）
本番RDSで、AZ障害時に自動でフェイルオーバーして「可用性」を確保したい（読み取り性能の向上は今回の目的ではない）。適切な構成はどれか。

- A. リードレプリカを追加する
- B. Multi-AZ配置にする
- C. インスタンスタイプをスケールアップする
- D. RDSをAuto Scalingグループに入れて台数を増やす

> [!success]- 答え：**B**（Multi-AZ）
> 従来のRDS Multi-AZ DBインスタンス配置は別AZにスタンバイを持ち、障害時に自動フェイルオーバーする「可用性」の構成。スタンバイは読み取りに使えないため、性能向上目的ではない。なお、RDS Multi-AZ DBクラスターは読み取り可能なスタンバイを持つ別方式。
> **他の選択肢が違う理由**
> - A：リードレプリカは主に「読み取り性能」のスケール用。単体では従来のMulti-AZのような自動フェイルオーバー構成そのものではない。
> - C：スケールアップは性能向上で、AZ障害対策（可用性）にはならない。
> - D：RDSのDBインスタンスはEC2 Auto Scalingの対象ではない。台数を増やして可用性を得る仕組みではなく、可用性はMulti-AZで確保する。
> **覚えるべきキーワード**：Multi-AZ＝可用性 / 自動フェイルオーバー / スタンバイ
> **関連サービス**：リードレプリカ（性能） / Aurora / RDS Proxy
> **SAAでの出題イメージ**：「AZ障害で自動切替・可用性」=Multi-AZ。
> 📎 参照：[[0301_RDS]]

## 第17問（RDS Multi-AZ / リードレプリカ）
RDSで、増え続ける読み取り（参照）クエリの負荷を分散し、読み取りスループットをスケールさせたい。適切な構成はどれか。

- A. Multi-AZ配置にする
- B. リードレプリカを追加する
- C. ElastiCacheでクエリ結果をキャッシュする
- D. インスタンスタイプをスケールアップする

> [!success]- 答え：**B**（リードレプリカ）
> リードレプリカは読み取り専用の複製で、参照クエリをオフロードして読み取り性能を水平にスケールさせる（非同期レプリケーション）。
> **他の選択肢が違う理由**
> - A：従来のRDS Multi-AZ DBインスタンス配置は可用性向上が目的で、スタンバイは読み取りに使えない（読み取りスケールではない）。読み取り可能な構成が必要なら、リードレプリカやRDS Multi-AZ DBクラスター/Auroraのリーダーを検討する。
> - C：ElastiCacheは頻出クエリのキャッシュで読み取り負荷を下げる有効策だが、キャッシュできない多様な参照クエリのスループット自体をスケールさせるのはリードレプリカ。
> - D：スケールアップは単一インスタンスの限界内の強化で、水平な読み取りスケールにはならない。
> **覚えるべきキーワード**：リードレプリカ＝読み取りスケール / 非同期レプリケーション
> **関連サービス**：Multi-AZ（可用性） / ElastiCache（キャッシュ） / Aurora Replica
> **SAAでの出題イメージ**：「読み取り負荷をオフロード」=リードレプリカ。可用性はMulti-AZ。
> 📎 参照：[[0301_RDS]]

## 第18問（SQS / SNS / EventBridge）
1件の発注イベントを、複数の独立したシステム（在庫・会計・通知）へ同時に届け、各システムは自分のキューで非同期に処理したい。最適な構成はどれか。

- A. SQSキュー1本に全システムが取りに行く
- B. SNSトピックに発行し、各システムのSQSキューをサブスクライブ（ファンアウト）
- C. EventBridgeのスケジュールでcron実行
- D. API Gatewayで同期呼び出し

> [!success]- 答え：**B**（SNS→複数SQSのファンアウト）
> 「1イベントを複数システムへ同時配信」はSNSのファンアウト。各システムが自前のSQSで受ければ疎結合かつ非同期処理が成立する（SNS＋SQSの定番）。
> **他の選択肢が違う理由**
> - A：単一SQSを複数システムで奪い合うと、各システムが全件を受け取れない（メッセージは基本1コンシューマが取得）。
> - C：EventBridgeスケジュールは定期実行用で、イベントの即時ファンアウト構成ではない（※EventBridgeでもルーティングは可能だが、この設問の最適解はSNS＋SQS）。
> - D：API Gateway同期は疎結合・非同期のバッファにならない。
> **覚えるべきキーワード**：ファンアウト / SNS＋SQS / 疎結合 / 非同期
> **関連サービス**：SNS / SQS / EventBridge
> **SAAでの出題イメージ**：「1→複数へ配信＋各自バッファ」=SNS＋SQS。
> 📎 参照：[[0803_SNS]] / [[0802_SQS]]

## 第19問（SQS / SNS / EventBridge）
複数のAWSサービスやSaaSから来る多様なイベントを、内容に応じたルールで適切な宛先へ振り分けたい。最適なサービスはどれか。

- A. Amazon SQS
- B. Amazon SNS
- C. Amazon EventBridge
- D. Amazon Kinesis Data Firehose

> [!success]- 答え：**C**（EventBridge）
> EventBridgeはイベントの内容に基づくルールでルーティングでき、多数のAWSサービス/SaaSと統合する「イベントバス」。
> **他の選択肢が違う理由**
> - A：SQSは単純なキューで、内容ルールによる振り分け機能はない。
> - B：SNSはトピック購読型の配信で、イベント内容で細かくルール分岐するのはEventBridgeが得意。
> - D：Firehoseはストリームをストレージへ流し込むETL寄りで、イベントのルール振り分けではない。
> **覚えるべきキーワード**：イベントバス / ルールベース / SaaS統合 / スケジュール
> **関連サービス**：SNS / SQS / Lambda / Step Functions
> **SAAでの出題イメージ**：「多様なイベントをルールで振り分け」=EventBridge。
> 📎 参照：[[0804_EventBridge]] / [[0801_アプリケーション統合サービス一覧]]

## 第20問（KMS / Secrets Manager / Parameter Store）
RDSのDB接続パスワードを安全に保管し、定期的に自動ローテーションして、アプリから取得したい。最適なサービスはどれか。

- A. AWS KMS
- B. AWS Secrets Manager
- C. Systems Manager パラメータストア（SecureString）で保存し、自前のLambdaで定期更新する
- D. RDSのIAMデータベース認証に切り替える

> [!success]- 答え：**B**（Secrets Manager）
> Secrets ManagerはDB認証情報の暗号化保管に加え、RDS等と連携した「自動ローテーション」をマネージドに標準提供し、アプリはAPIで取得できる点が決め手。
> **他の選択肢が違う理由**
> - A：KMSは暗号鍵の管理基盤で、シークレット値の保管・自動ローテーション機能そのものは持たない（Secrets Managerが内部でKMSを利用）。
> - C：Parameter StoreのSecureStringでも暗号化保管はできるが、ローテーションは自前実装（運用負荷）が必要。マネージドな自動ローテーションはSecrets Managerが標準提供。
> - D：IAMデータベース認証はパスワードを使わない別方式で安全だが、本問の「パスワードを保管して自動ローテーション」とは前提が異なる（短命トークン・接続数の考慮も要る）。
> **覚えるべきキーワード**：Secrets Manager＝自動ローテーション / Parameter Store＝安価な設定保管 / KMS＝鍵
> **関連サービス**：KMS / Systems Manager / RDS
> **SAAでの出題イメージ**：「自動ローテーション必須」=Secrets Manager、「安価な設定/秘密」=Parameter Store。
> 📎 参照：[[1605_データ保護・暗号化]] / [[1405_Systems Manager]]
