# 【問題】類似サービス比較8（公式サンプル風・全10問）

## この問題集の目的

役割が似ていて混同しやすいサービス/オプションの「使い分け」を要件文で判断する。201〜207で扱っていない比較（Auto Scalingポリシーの種類・S3アーカイブクラスの選択・RDS vs DynamoDB・ALB vs NLB・EC2権限付与の方法・CloudTrail vs CloudWatch vs Config・SQS標準 vs FIFO・EventBridge vs SNS・Kinesis Streams vs Firehose・DMS vs DataSyncなど）を中心に、13分野を横断する。

## 対象レベル

SAA。サービス名は知っている前提で、要件に最も合う1つを選ぶ。

## 問題数

10問

## 第1問（Compute ｜ Auto Scalingポリシーの選択）
EC2 Auto Scalingグループの平均CPU使用率を、常に50%付近に保つよう台数を自動で増減させたい。しきい値ごとの細かなルール設計はせず、目標値を指定するだけで運用したい。最も適切なスケーリングポリシーはどれか。

- A. ターゲット追跡スケーリングポリシー
- B. ステップスケーリングポリシー
- C. シンプルスケーリングポリシー
- D. スケジュールされたスケーリング

> [!success]- 答え：**A**
> ターゲット追跡は目標メトリクス値（例: CPU50%）を指定するだけで、それを維持するよう自動で増減する最も手軽な方式。
> - A：正解。目標値の指定だけで維持でき、設計負荷が小さい。
> - B：メトリクスの段階ごとに増減量を細かく定義する方式で、目標値維持だけが目的なら過剰。
> - C：単一のしきい値で増減し、クールダウン待ちが必要な旧来方式で、目標追従の滑らかさに劣る。
> - D：時刻ベースで台数を変える方式で、CPU使用率に追従する用途ではない。
> **関連ノート**：[[0504_Auto_Scaling_と_Auto_Scaling_Group]]

## 第2問（Storage ｜ S3アーカイブクラスの選択）
数年に一度しかアクセスせず、取り出しに12時間程度かかってもよいコンプライアンス保管用データを、最も低コストで保存したい。最も適切なS3ストレージクラスはどれか。

- A. S3 Glacier Deep Archive
- B. S3 Glacier Flexible Retrieval
- C. S3 Standard-IA
- D. S3 One Zone-IA

> [!success]- 答え：**A**
> Glacier Deep Archiveは最安の長期アーカイブクラスで、取り出しに時間がかかる（数時間〜12時間）代わりに保管コストが最小。
> - A：正解。長期保管かつ取り出し時間に余裕がある要件で最安。
> - B：取り出しが分〜数時間と速い分、Deep Archiveより保管コストが高い。
> - C：低頻度アクセスでも即時取り出し向けで、数年に一度の保管には割高。
> - D：単一AZで安価だが即時取り出し向けで、長期アーカイブの最安解ではなく冗長性も低い。
> **関連ノート**：[[0203_S3ストレージクラス]] / [[0204_S3のライフサイクル管理]]

## 第3問（Database ｜ RDS vs DynamoDB）
予測困難なトラフィックで、ミリ秒未満の応答とほぼ無制限の水平スケールが必要なキー・バリュー中心のアクセスパターンに対応したい。複雑なJOINやトランザクションは不要。最も適切なサービスはどれか。

- A. Amazon DynamoDB
- B. Amazon RDS for MySQL（マルチAZ）
- C. Amazon Redshift
- D. Amazon Aurora（PostgreSQL互換）

> [!success]- 答え：**A**
> DynamoDBはキー・バリュー/ドキュメント型のNoSQLで、ミリ秒未満の応答と自動的なほぼ無制限の水平スケールを提供する。
> - A：正解。スパイクするキー・バリューアクセスの水平スケールに最適。
> - B：リレーショナルで縦スケール中心。予測困難な超大規模スケールには水平拡張のNoSQLが有利。
> - C：分析向けデータウェアハウスで、低レイテンシのトランザクション処理用途ではない。
> - D：高性能なリレーショナルDBだが、キー・バリューのスパイク的な水平スケール特化用途ではDynamoDBが適する。
> **関連ノート**：[[0303_DynamoDB]] / [[0301_RDS]]

## 第4問（Networking / CDN / LB ｜ ALB vs NLB）
毎秒数百万の接続を、極めて低いレイテンシーでL4（TCP/UDP）処理し、ホワイトリスト登録用に変わらない静的IPを公開したい。最も適切なロードバランサーはどれか。

- A. Network Load Balancer（NLB）
- B. Application Load Balancer（ALB）
- C. Gateway Load Balancer（GWLB）
- D. Classic Load Balancer

> [!success]- 答え：**A**
> NLBはL4で超低レイテンシ・超高スループットを処理し、AZごとに静的IP（Elastic IP）を持てる。
> - A：正解。超低レイテンシのL4分散と静的IP公開の要件に合致する。
> - B：L7のHTTP/HTTPS向けで、パスやホストでのルーティングが得意だが静的IPは持たず、超低レイテンシL4要件には不向き。
> - C：サードパーティ仮想アプライアンスへの透過的な振り分け用途で、一般的なL4分散の主目的ではない。
> - D：旧世代で、NLB/ALBが提供する機能・性能・静的IPの要件を満たしにくい。
> **関連ノート**：[[0503_ALB・NLB・GWLBの違い]] / [[0502_Elastic Load Balancing]]

## 第5問（Security / IAM ｜ EC2へのAWS権限付与）
EC2上で動くアプリケーションにS3への権限を与えたい。長期的なアクセスキーを埋め込まず、認証情報を自動でローテーションする最も安全な方法はどれか。

- A. IAMロールをEC2インスタンスにアタッチする
- B. IAMユーザーのアクセスキーをEC2の設定ファイルに保存する
- C. ルートユーザーのアクセスキーを使う
- D. アクセスキーをEC2のユーザーデータに記述する

> [!success]- 答え：**A**
> EC2にIAMロールをアタッチすると、インスタンスメタデータ経由で一時認証情報が自動配布・ローテーションされ、長期キーが不要になる。
> - A：正解。長期キーを埋め込まず、自動ローテーションされる一時認証情報で安全。
> - B：長期アクセスキーの埋め込みは漏えいリスクが高く、ローテーションも手動になる。
> - C：ルート認証情報の利用は最も避けるべきで、最小権限の原則にも反する。
> - D：ユーザーデータは取得しやすく、長期キーの平文保存は安全でない。
> **関連ノート**：[[0602_IAMロール詳細]] / [[0601_IAM基礎]]

## 第6問（Management / Monitoring ｜ CloudTrail vs CloudWatch vs Config）
「誰が」「いつ」「どのAWS API操作を行ったか」を後から監査・追跡したい。セキュリティインシデント調査に使う。最も適切なサービスはどれか。

- A. AWS CloudTrail
- B. Amazon CloudWatch
- C. AWS Config
- D. AWS Trusted Advisor

> [!success]- 答え：**A**
> CloudTrailはアカウント内のAPIコールを記録する監査ログで、操作主体・時刻・内容を追跡できる。
> - A：正解。API操作の証跡（誰が・いつ・何を）の追跡に特化している。
> - B：メトリクス・ログ・アラームによる運用監視が主で、API操作の監査証跡そのものではない。
> - C：リソース構成の変更履歴と準拠評価が主で、API操作主体の監査ログではない。
> - D：コスト・セキュリティ等のベストプラクティス推奨で、操作履歴の追跡ではない。
> **関連ノート**：[[1002_CloudTrail]] / [[1001_CloudWatch]]

## 第7問（Application Integration ｜ SQS標準 vs FIFO）
メッセージを発生した順序のまま厳密に処理し、重複配信も排除したい。1秒あたりの処理量は限られていても許容できる。最も適切なキューはどれか。

- A. Amazon SQS FIFOキュー
- B. Amazon SQS 標準キュー
- C. Amazon SNS 標準トピック
- D. Amazon Kinesis Data Streams

> [!success]- 答え：**A**
> FIFOキューは順序保証と重複排除（exactly-once処理）を提供する代わりに、スループット上限がある。
> - A：正解。厳密な順序と重複排除の要件に合致する。
> - B：高スループットだが順序は最善努力で重複もあり得るため、厳密な順序・重複排除には不向き。
> - C：Pub/Subのプッシュ通知で、順序保証のあるキューイングではない。
> - D：順序付きのストリーム処理は可能だが、SQS FIFOのような重複排除付きのキューとは用途・設計が異なる。
> **関連ノート**：[[0802_SQS]] / [[0803_SNS]]

## 第8問（Serverless ｜ EventBridge vs SNS）
多数のSaaSやAWSサービスから届く多様なイベントを、内容（属性）に基づくルールでフィルタリングし、複数の異なるターゲットへ振り分けたい。最も適切なサービスはどれか。

- A. Amazon EventBridge
- B. Amazon SNS
- C. Amazon SQS
- D. AWS Step Functions

> [!success]- 答え：**A**
> EventBridgeはイベントの内容に基づくルールマッチングと多様なターゲットへのルーティングに特化したイベントバス。
> - A：正解。属性ベースのフィルタリングと多彩なターゲット連携に最適。
> - B：トピック単位のPub/Sub通知で、メッセージ属性での高度なルールルーティングはEventBridgeほど柔軟ではない。
> - C：メッセージのバッファリング（キュー）が主で、内容ベースのルーティング機能ではない。
> - D：ワークフローの状態遷移管理が主で、イベントの内容ベース振り分けハブではない。
> **関連ノート**：[[0804_EventBridge]] / [[0803_SNS]]

## 第9問（Analytics ｜ Kinesis Data Streams vs Firehose）
ストリーミングデータを、独自の処理コードやシャード管理を極力持たずに、S3やRedshiftへ準リアルタイムに変換・ロードしたい。最も適切なサービスはどれか。

- A. Amazon Kinesis Data Firehose
- B. Amazon Kinesis Data Streams
- C. Amazon MSK
- D. Amazon SQS

> [!success]- 答え：**A**
> Firehoseはストリーミングデータをバッファし、変換しつつS3/Redshift/OpenSearch等へ自動ロードするフルマネージド配信サービス。
> - A：正解。シャード管理や独自コンシューマーなしで宛先へ自動ロードできる。
> - B：低レイテンシのストリーム取り込みは可能だが、シャード管理や独自のコンシューマー実装が必要で運用負荷が高い。
> - C：マネージドKafkaで、自前のストリーム処理基盤向け。宛先への自動ロードを最小運用で行う用途ではない。
> - D：メッセージキューで、ストリーミングデータの宛先への自動変換ロードではない。
> **関連ノート**：[[0805_Kinesis]] / [[1103_Amazon Kinesis]]

## 第10問（Migration / Transfer ｜ DMS vs DataSync）
オンプレミスで稼働中のデータベースを、できるだけ停止せずにAWSへ移行し、移行中も継続的にレプリケーションしたい。最も適切なサービスはどれか。

- A. AWS Database Migration Service（DMS）
- B. AWS DataSync
- C. AWS Transfer Family
- D. AWS Snowball Edge

> [!success]- 答え：**A**
> DMSはソースDBを稼働させたまま、最小停止での移行と継続的なレプリケーション（CDC）を提供する。
> - A：正解。稼働中DBの低停止移行と継続レプリケーションに最適。
> - B：ファイル/オブジェクトストレージ間の同期向けで、稼働中DBのレプリケーション移行ではない。
> - C：外部クライアント向けのマネージドSFTP等の受け口で、DB移行サービスではない。
> - D：大容量データの物理オフライン搬送向けで、稼働中DBの継続レプリケーションではない。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[1502_DataSync]]
