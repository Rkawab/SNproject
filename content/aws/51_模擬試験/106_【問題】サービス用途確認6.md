# 【問題】サービス用途確認6（公式サンプル風・全10問）

## この問題集の目的

要件文から主要サービスの役割を選ぶドリル。101〜105で扱っていない切り口（自動スケーリング・長期アーカイブ・キーバリューDB・CDN配信・暗号鍵管理・HTTP API公開・ソースからのコンテナ実行・ストリーミング配信・柔軟な割引コミット・帳票テキスト抽出など）を中心に、13分野を横断する。

## 対象レベル

SAA入口。単語暗記ではなく、要件からサービスを選ぶ練習にする。

## 問題数

10問

## 第1問（Compute ｜ 需要に応じた自動増減）
Webアプリのアクセスが時間帯で大きく変動する。CPU使用率などの指標に応じてEC2の台数を自動で増減させ、需要に追従しつつ過剰なコストを抑えたい。最も適切なサービスはどれか。

- A. Amazon EC2 Auto Scaling
- B. 余裕を持った大きめのEC2を1台、常時起動しておく
- C. AWS Lambda の予約済み同時実行
- D. Elastic IP アドレスを追加で割り当てる

> [!success]- 答え：**A**
> EC2 Auto Scalingは指標やスケジュールに応じてインスタンス数を自動で増減し、可用性とコスト最適化を両立する。
> - A：正解。需要追従の水平スケールに最適。
> - B：固定の垂直スケールでは変動に追従できず、ピーク以外は無駄になる。
> - C：Lambdaの同時実行設定で、EC2台数の自動増減ではない。
> - D：IPの割り当てはスケーリングとは無関係。
> **関連ノート**：[[0504_Auto_Scaling_と_Auto_Scaling_Group]] / [[0501_高可用性とスケーラビリティ]]

## 第2問（Storage ｜ ほとんどアクセスしない長期アーカイブ）
法令対応で7年間の保持が必要なログがある。取り出しは年に数回で、数時間の待ち時間は許容できる。可能な限り低コストで保管したい。最も適切なストレージはどれか。

- A. Amazon S3 Glacier Deep Archive
- B. Amazon S3 標準（Standard）
- C. Amazon EBS gp3 ボリューム
- D. Amazon EFS 標準

> [!success]- 答え：**A**
> S3 Glacier Deep ArchiveはS3で最も低コストのアーカイブクラスで、取り出しに時間を要するが長期保管に最適。
> - A：正解。低頻度アクセス・長期保持・最小コストの要件に合う。
> - B：即時アクセス向けで割高、アーカイブ用途には過剰コスト。
> - C：単一インスタンス向けブロックストレージで、長期アーカイブ用途ではない。
> - D：共有ファイル向けで、コスト面でアーカイブには不向き。
> **関連ノート**：[[0203_S3ストレージクラス]] / [[0204_S3のライフサイクル管理]]

## 第3問（Database ｜ ミリ秒未満のキー・バリューアクセス）
ゲームのプレイヤー状態を、規模に応じてシームレスにスケールさせ、一桁ミリ秒の応答でキー指定の読み書きを行いたい。項目ごとに柔軟なスキーマにしたい。最も適切なサービスはどれか。

- A. Amazon DynamoDB
- B. Amazon RDS for PostgreSQL
- C. Amazon Redshift
- D. Amazon Athena

> [!success]- 答え：**A**
> DynamoDBはフルマネージドのNoSQLで、規模に応じて自動スケールし、一桁ミリ秒のキー・バリューアクセスを提供する。
> - A：正解。柔軟なスキーマと低レイテンシのキー指定アクセスに最適。
> - B：固定スキーマのRDBで、シームレスな大規模スケールには設計が必要。
> - C：分析向けのデータウェアハウスで、低レイテンシのトランザクション用途ではない。
> - D：S3上データへのSQLクエリで、キー・バリューストアではない。
> **関連ノート**：[[0303_DynamoDB]]

## 第4問（Networking / CDN / LB ｜ 静的・動的コンテンツのエッジ配信）
世界中のユーザーへ、画像や動画などの静的コンテンツとAPIの動的レスポンスを、エッジロケーションからキャッシュして高速配信し、オリジンの負荷とレイテンシを下げたい。最も適切なサービスはどれか。

- A. Amazon CloudFront
- B. Amazon Route 53 のみ
- C. AWS Direct Connect
- D. Amazon S3 Transfer Acceleration

> [!success]- 答え：**A**
> CloudFrontはエッジロケーションからコンテンツをキャッシュ配信するCDNで、静的・動的双方の配信を高速化する。
> - A：正解。グローバルなエッジ配信とオリジン負荷軽減に最適。
> - B：DNSの名前解決が主目的で、コンテンツのキャッシュ配信は行わない。
> - C：オンプレとAWS間の専用線で、エンドユーザーへのCDN配信ではない。
> - D：S3アップロードの高速化機能で、配信側のCDNではない。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]]

## 第5問（Security / IAM ｜ 保存データの暗号化鍵の集中管理）
S3・EBS・RDSなど複数サービスの保存データ暗号化に使う鍵を、一元的に作成・管理し、利用をIAMで制御してCloudTrailで監査したい。最も適切なサービスはどれか。

- A. AWS Key Management Service（KMS）
- B. AWS Secrets Manager
- C. AWS Certificate Manager（ACM）
- D. Amazon Macie

> [!success]- 答え：**A**
> KMSは暗号化鍵を一元管理するマネージドサービスで、各AWSサービスと統合し、IAM制御とCloudTrail監査に対応する。
> - A：正解。保存データ暗号化の鍵管理に最適。
> - B：DB認証情報などのシークレット保管・ローテーション向けで、暗号化鍵の管理基盤ではない。
> - C：TLS証明書の管理向けで、保存データの暗号鍵管理ではない。
> - D：S3内の機密データ検出・分類向けで、鍵管理サービスではない。
> **関連ノート**：[[1607_AWS_KMS]] / [[1605_データ保護・暗号化]]

## 第6問（Serverless ｜ HTTP APIの公開とリクエスト管理）
バックエンドのLambda関数を、認証・スロットリング・ステージ管理を備えたHTTP/REST APIとして公開したい。サーバーの管理はしたくない。最も適切なサービスはどれか。

- A. Amazon API Gateway
- B. Amazon Route 53
- C. Classic Load Balancer
- D. AWS Direct Connect

> [!success]- 答え：**A**
> API GatewayはマネージドにAPIを公開し、認証・スロットリング・ステージ管理などを備えてLambda等のバックエンドと統合する。
> - A：正解。サーバーレスなHTTP/REST APIの公開と管理に最適。
> - B：DNSの名前解決が主目的で、APIのリクエスト管理機能は持たない。
> - C：旧世代のL4/L7分散で、APIキー・スロットリング等のAPI管理機能はない。
> - D：専用線接続で、API公開とは無関係。
> **関連ノート**：[[0809_API_Gateway]] / [[0708_Lambda]]

## 第7問（Containers ｜ ソースやイメージから直接Webアプリを実行）
コンテナ化したWebアプリを、ロードバランサーやスケーリングを自分で細かく組まずに、ソースコードまたはコンテナイメージから直接デプロイし、自動スケールさせたい。最小限の運用で公開したい。最も適切なサービスはどれか。

- A. AWS App Runner
- B. Amazon EC2 上で自前でDockerを構築する
- C. AWS Lambda（ZIPデプロイ）
- D. Amazon S3 静的ウェブサイトホスティング

> [!success]- 答え：**A**
> App Runnerはソースコードやコンテナイメージから直接Webアプリをデプロイし、ロードバランシングとスケーリングをマネージドに行う。
> - A：正解。最小運用でコンテナWebアプリを公開・自動スケールできる。
> - B：自前構築は運用負荷が高く、要件の「最小限の運用」に反する。
> - C：常時稼働のWebアプリをコンテナイメージのまま動かす用途には合わない。
> - D：静的サイト配信向けで、コンテナアプリの実行はできない。
> **関連ノート**：[[0901_AWS_コンテナサービス概要]] / [[0902_オーケストレーション_フルマネージド]]

## 第8問（Analytics ｜ ストリーミングデータの取り込みとロード）
アプリやIoTから連続的に発生するデータを、ほぼリアルタイムに受け取り、最小限の運用でバッファリング・変換しながらS3やRedshiftへ自動的にロードしたい。最も適切なサービスはどれか。

- A. Amazon Kinesis Data Firehose
- B. Amazon SQS 標準キュー
- C. AWS Glue のバッチETLジョブのみ
- D. Amazon Athena

> [!success]- 答え：**A**
> Kinesis Data Firehoseはストリーミングデータをバッファリング・変換し、S3・Redshift・OpenSearchへ自動配信するマネージドサービス。
> - A：正解。運用なしでストリームをデータストアへロードする用途に最適。
> - B：メッセージキューで、データストアへの自動ロード機能は持たない。
> - C：定期バッチ向けで、連続ストリームの準リアルタイムロードには合わない。
> - D：クエリエンジンで、データの取り込み・配信は担わない。
> **関連ノート**：[[1103_Amazon Kinesis]] / [[1101_AWS分析サービス一覧]]

## 第9問（Cost Optimization ｜ 柔軟な割引コミット）
今後1〜3年にわたり一定額のコンピュート利用が見込まれる。インスタンスファミリーやリージョン、FargateやLambdaも含めて柔軟に適用できる形で、オンデマンドより大幅な割引を受けたい。最も適切な購入方法はどれか。

- A. Compute Savings Plans
- B. オンデマンド料金のまま利用を続ける
- C. 本番環境をすべてスポットインスタンスで運用する
- D. ハードウェア専有のDedicated Hosts

> [!success]- 答え：**A**
> Compute Savings Plansは時間あたりの利用額をコミットすることで、EC2・Fargate・Lambdaに柔軟に割引を適用する。
> - A：正解。柔軟性を保ちつつ継続利用へ大幅割引を得られる。
> - B：何もコミットしないため割引は得られない。
> - C：スポットは中断され得るため、安定運用が必要な本番全体には不向き。
> - D：物理サーバー専有でコンプライアンス向けの選択肢で、割引コミットの手段ではない。
> **関連ノート**：[[1701_コスト管理サービス一覧]] / [[0102_EC2購入プラン]]

## 第10問（Machine Learning ｜ 帳票からのテキスト・データ抽出）
請求書や申込書をスキャンしたPDFや画像から、単純なOCRだけでなく、フォームのキーと値や表構造を機械学習で抽出したい。モデルの構築はしたくない。最も適切なサービスはどれか。

- A. Amazon Textract
- B. Amazon Rekognition
- C. Amazon Comprehend
- D. Amazon Translate

> [!success]- 答え：**A**
> Textractは書類からテキストに加え、フォームのキー・値や表構造をマネージドに抽出する。
> - A：正解。帳票の構造化データ抽出に最適。
> - B：画像・動画の物体や顔の検出向けで、帳票の表・フォーム抽出が主目的ではない。
> - C：テキストの感情・エンティティ分析向けで、書類からの抽出ではない。
> - D：テキストの言語翻訳サービス。
> **関連ノート**：[[1301_AWS機械学習サービス一覧]]
