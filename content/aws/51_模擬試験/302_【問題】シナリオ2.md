# 【問題】シナリオ ②（4択・全20問）

## この問題集の目的

[[301_【問題】シナリオ]]の続編。短い要件文を読み、適切なAWSサービス・構成を選ぶ練習。1問目では扱わなかったテーマ（限定配信・DR/レプリケーション・疎結合の応用・移行/ハイブリッド・データ保護）を中心に、要件のキーワードから設計を選ぶ感覚を鍛える。

## 対象レベル

SAA入門〜標準。本番より短めの設計シナリオ。1問ごとに「何が決め手か」を意識する。

## 問題数

20問（限定配信・クロスリージョンDR・誤削除/改ざん対策・コストDB・疎結合/スロットリング・順序保証・キャッシュ・ストリーミング蓄積・サーバーレス分析・多VPC接続・プライベート接続・ハイブリッドストレージ・オンライン移行・オフライン移行・Windows共有・バックアップ統合・脅威検知・機密データ検出・会員認証・運用接続 を横断）

## 使い方

要件文から「効いているキーワード（決め手）」を先に拾う癖をつける。回答後は不正解の構成が「どの要件を満たせないか」を確認する。

## 第1問（セキュリティ／限定配信）
S3に置いた有料会員向け動画を、S3を直接公開せず、CloudFront経由かつ許可された会員にだけ期限付きで配信したい。最適な構成はどれか。

- A. S3バケットを公開してURLを会員に配る
- B. CloudFront＋OAC（S3は非公開）＋署名付きURL/Cookie
- C. ALBでS3を配信する
- D. EC2にS3の中身をコピーして配る

> [!success]- 答え：**B**（OAC＋署名付きURL/Cookie）
> S3はOAC（旧OAI）で「CloudFrontからのみ」に限定して非公開を保ち、署名付きURL/Cookieで会員だけに期限付き配信する。直接公開を避けつつ限定配信できる。
> **他の選択肢が違う理由**
> - A：公開＋URL配布はURLが漏れれば誰でもアクセスでき、限定にならない。
> - C：ALBはS3配信の仕組みではない。
> - D：EC2へコピーは整合性・運用・コストで不適切。
> **覚えるべきキーワード**：OAC / 署名付きURL・Cookie / S3非公開
> **関連サービス**：CloudFront / S3 / WAF
> **SAAでの出題イメージ**：「S3を直接公開せず期限付き限定配信」=CloudFront＋OAC＋署名付きURL。
> 📎 参照：[[0405_ネットワークとコンテンツ配信]] / [[0208_S3バケットポリシー]]

## 第2問（DR／クロスリージョン）
規制とDRのため、本番S3バケットのオブジェクトを別リージョンへ自動複製し、リージョン障害時に備えたい。最適な方法はどれか。

- A. S3クロスリージョンレプリケーション（CRR）を有効化
- B. 毎日手動で別リージョンへコピー
- C. 同一リージョン内でバージョニングのみ有効化
- D. EBSスナップショットを取る

> [!success]- 答え：**A**（CRR）
> S3 CRR（クロスリージョンレプリケーション）はオブジェクトを別リージョンへ自動・継続複製する。リージョン障害対策・規制・低遅延アクセスに使える（バージョニング有効化が前提）。
> **他の選択肢が違う理由**
> - B：手動コピーは抜け漏れ・遅延・運用負荷が大きい。
> - C：同一リージョンのバージョニングはリージョン障害に対応しない。
> - D：EBSスナップショットはS3オブジェクトの複製ではない。
> **覚えるべきキーワード**：CRR / 自動複製 / 別リージョン / バージョニング前提
> **関連サービス**：SRR（同一リージョン） / バージョニング / ライフサイクル
> **SAAでの出題イメージ**：「S3を別リージョンへ自動複製」=クロスリージョンレプリケーション。
> 📎 参照：[[0206_S3のレプリケーション]] / [[0205_S3のバージョン管理]]

## 第3問（データ保護／誤削除・改ざん）
監査要件で、S3オブジェクトを保持期間中は「上書き・削除できない（WORM）」ようにし、ランサムウェアや誤削除からも守りたい。最適な方法はどれか。

- A. バケットポリシーで削除をDenyするだけ
- B. S3 Object Lock（バージョニング＋保持期間/リーガルホールド）
- C. ライフサイクルで自動削除する
- D. One Zone-IAに保存する

> [!success]- 答え：**B**（Object Lock）
> S3 Object Lockはバージョニング前提で、保持期間（Retention）やリーガルホールドにより保持中の上書き・削除を不可（WORM）にする。改ざん・誤削除・ランサム対策に有効。
> **他の選択肢が違う理由**
> - A：バケットポリシーのDenyは設定変更で外せてしまい、WORMの強制力に欠ける。
> - C：ライフサイクル自動削除はむしろ消す設定で要件に反する。
> - D：One Zone-IAは単一AZのコスト最適化で改ざん防止とは無関係。
> **覚えるべきキーワード**：Object Lock / WORM / 保持期間 / リーガルホールド / バージョニング
> **関連サービス**：バージョニング / MFA Delete / Glacier（長期保管）
> **SAAでの出題イメージ**：「保持中は消せない・改ざん防止」=S3 Object Lock。
> 📎 参照：[[0205_S3のバージョン管理]] / [[0203_S3ストレージクラス]]

## 第4問（コスト最適化／DB）
1日のうち数時間だけ使われ、それ以外はほぼアイドルな開発用RDBがある。使わない時間のコストを抑えつつ、必要時は自動でスケールさせたい。最適なのはどれか。

- A. 大きめのRDSインスタンスを常時起動
- B. Aurora Serverless v2
- C. DynamoDB プロビジョンドキャパシティ
- D. EC2上のMySQLを常時起動

> [!success]- 答え：**B**（Aurora Serverless v2）
> Aurora Serverless v2は負荷に応じて容量を自動でシームレスに増減し、アイドル時はスケールダウンしてコストを抑える。間欠・変動ワークロードに適する。
> **他の選択肢が違う理由**
> - A：常時起動の固定インスタンスはアイドル時もフル課金で割高。
> - C：DynamoDBはNoSQLで、リレーショナルな開発DBの代替にはデータモデル変更が要る。
> - D：EC2常時起動も同様にアイドルコストと運用負荷が残る。
> **覚えるべきキーワード**：Aurora Serverless / 自動容量増減 / 間欠ワークロード / コスト最適
> **関連サービス**：Aurora / RDS / RDS Proxy
> **SAAでの出題イメージ**：「間欠・変動するRDBでアイドルコスト削減」=Aurora Serverless。
> 📎 参照：[[0301_RDS]]

## 第5問（疎結合／スロットリング対策）
急なトラフィックでLambdaが大量同時起動し、下流のRDSが接続過多で詰まる。下流を保護しつつ処理を取りこぼさないようにしたい。最適な構成はどれか。

- A. リクエストをSQSにため、Lambdaが一定の同時実行数で取り出して処理する
- B. Lambdaの同時実行を無制限にする
- C. RDSをスケールアップして終わりにする
- D. クライアントにリトライを任せる

> [!success]- 答え：**A**（SQSでバッファ＋同時実行制御）
> SQSでリクエストをバッファし、Lambdaのバッチ取得や予約済み同時実行数で流量を制御すれば、下流（RDS）を保護しつつ取りこぼしを防げる（疎結合＋スロットリング）。
> **他の選択肢が違う理由**
> - B：同時実行無制限は下流をさらに圧迫し悪化する。
> - C：スケールアップは一時しのぎで上限・コストの問題が残り、スパイクの根本対策にならない。
> - D：クライアント任せのリトライは取りこぼし・輻輳を招く。
> **覚えるべきキーワード**：SQSバッファ / 予約済み同時実行 / 下流保護 / 負荷平準化
> **関連サービス**：SQS / Lambda / RDS Proxy（接続プール） / DLQ
> **SAAでの出題イメージ**：「スパイクで下流が詰まる」=SQSで受け止め＋同時実行制御。
> 📎 参照：[[0802_SQS]] / [[0302_RDSProxy]]

## 第6問（順序保証・重複排除）
決済処理のメッセージを「送信順に・重複なく一度だけ」処理する必要がある。最適なサービスはどれか。

- A. SQS標準キュー
- B. SQS FIFOキュー
- C. SNS標準トピック
- D. Kinesis Data Firehose

> [!success]- 答え：**B**（SQS FIFOキュー）
> FIFOキューはメッセージグループ内で厳密な順序保持と重複排除（exactly-once処理）を提供する。決済など順序・重複が致命的な処理に適する。
> **他の選択肢が違う理由**
> - A：標準キューはベストエフォート順序で重複配信があり得る（高スループット向け）。
> - C：標準SNSは順序保証なし（FIFOトピックは別だが、ここはキュー処理）。
> - D：Firehoseは配信デリバリーで順序・重複排除の決済処理基盤ではない。
> **覚えるべきキーワード**：FIFO / 順序保持 / 重複排除 / exactly-once
> **関連サービス**：SQS標準（高スループット） / SNS FIFO / Lambda
> **SAAでの出題イメージ**：「順序と重複なしが必須」=SQS FIFO。
> 📎 参照：[[0802_SQS]]

## 第7問（性能／キャッシュ）
読み取りが特定の人気アイテムに集中し、RDSの同じクエリが繰り返されてボトルネックになっている。アプリ改修は許容範囲で、読み取りを高速化しDB負荷を下げたい。最適なのはどれか。

- A. ElastiCacheで頻出クエリ結果をキャッシュ
- B. RDSをMulti-AZにする
- C. EBSをio2にする
- D. NACLで制限する

> [!success]- 答え：**A**（ElastiCacheでキャッシュ）
> 繰り返される参照はElastiCache（Redis/Memcached）にキャッシュし、ヒット時はメモリから返す。DBへの問い合わせを減らし、低遅延と負荷軽減を両立する。
> **他の選択肢が違う理由**
> - B：Multi-AZは可用性向上で、読み取り負荷の軽減・高速化にはならない。
> - C：io2はディスク性能の話で、同一クエリ繰り返しの本質的解決ではない。
> - D：NACLはネットワーク制御で性能改善と無関係。
> **覚えるべきキーワード**：キャッシュ / ヒット率 / DB負荷軽減 / 低遅延
> **関連サービス**：ElastiCache / リードレプリカ（併用） / DAX（Dynamo）
> **SAAでの出題イメージ**：「同じ参照の繰り返しでDBが重い」=ElastiCacheでキャッシュ。
> 📎 参照：[[0304_ElastiCache]] / [[0301_RDS]]

## 第8問（ストリーミング蓄積）
多数のデバイスから届くストリームを、コードを最小限にしてニアリアルタイムでS3へバッファ・配信し、後でAthenaで分析したい。最適なのはどれか。

- A. Kinesis Data Streams＋自前のコンシューマでS3書き込み
- B. Kinesis Data Firehose → S3
- C. SQS → Lambda → S3
- D. SNS → メール

> [!success]- 答え：**B**（Firehose → S3）
> FirehoseはストリームをバッファしてS3等へ自動配信するフルマネージドのデリバリー。コード最小・ニアリアルタイムでデータレイクに蓄積でき、Athena分析につなげやすい。
> **他の選択肢が違う理由**
> - A：Data Streams＋自前コンシューマはS3書き込みを自前実装する必要があり手間が多い（リアルタイム処理が要る場合の選択）。
> - C：SQS→Lambda→S3でも可能だが、ストリーム配信の定番はFirehoseでコードが少ない。
> - D：SNS→メールは蓄積・分析基盤ではない。
> **覚えるべきキーワード**：Firehose / 自動配信 / ニアリアルタイム / S3蓄積
> **関連サービス**：Kinesis Data Streams / S3 / Athena / Glue
> **SAAでの出題イメージ**：「ストリームをコード少なくS3へ蓄積」=Firehose。
> 📎 参照：[[0805_Kinesis]] / [[1109_Amazon Athena]]

## 第9問（サーバーレス分析／低運用）
S3に溜まった大量ログを、クラスタを運用せず・たまにSQLで分析し、結果をダッシュボードで可視化したい。運用負荷を最小にしたい。最適な組み合わせはどれか。

- A. Glue（カタログ）＋ Athena（SQL）＋ QuickSight（可視化）
- B. EMRクラスタを常時起動して集計
- C. RDSに全ログを入れて集計
- D. EC2にHadoopを構築

> [!success]- 答え：**A**（Glue＋Athena＋QuickSight）
> Glueでスキーマをカタログ化し、Athenaでサーバーレスにクエリ、QuickSightで可視化するのが、最も運用負荷の低いサーバーレス分析構成。
> **他の選択肢が違う理由**
> - B：EMR常時起動は運用負荷・コストが高い（大規模分散が要る場合向け）。
> - C：RDSは大量ログのOLAP分析・半構造データに不向き。
> - D：自前Hadoopは構築・運用負荷が極めて高い。
> **覚えるべきキーワード**：データレイク / Glueカタログ / Athena / QuickSight / サーバーレス
> **関連サービス**：S3 / Glue / Athena / QuickSight / Lake Formation
> **SAAでの出題イメージ**：「S3を低運用でSQL分析・可視化」=Glue＋Athena＋QuickSight。
> 📎 参照：[[1107_AWS Glue]] / [[1109_Amazon Athena]]

## 第10問（ネットワーク／多VPC集約）
事業拡大で数十のVPCとオンプレ拠点が増え、相互接続の管理が複雑化している。ハブを中心に一元的にルーティングしたい。最適なのはどれか。

- A. 全VPC間にVPCピアリングをフルメッシュで張る
- B. AWS Transit Gateway を中央ハブにする
- C. 各VPCにNAT Gatewayを置く
- D. 各VPCをインターネット経由で接続する

> [!success]- 答え：**B**（Transit Gateway）
> Transit Gatewayは多数のVPC・VPN/DXを中央ハブで接続し、ルーティングを一元管理する。ピアリングのフルメッシュの複雑さ（接続数の爆発）を解消する。
> **他の選択肢が違う理由**
> - A：フルメッシュのピアリングは接続数がN×Nに増え、管理が破綻する。
> - C：NAT Gatewayは外向き通信用でVPC相互接続ではない。
> - D：インターネット経由はセキュリティ・品質の面で不適。
> **覚えるべきキーワード**：Transit Gateway / ハブ＆スポーク / 一元ルーティング
> **関連サービス**：VPCピアリング / Direct Connect Gateway / VPN
> **SAAでの出題イメージ**：「多数VPC/拠点を一元接続」=Transit Gateway。
> 📎 参照：[[0406_VPC応用機能]]

## 第11問（プライベート接続）
プライベートサブネットのEC2から、NAT/インターネットを経由せずにSQSやKMSなどのAWSサービスへ接続したい。最適なのはどれか。

- A. インターフェース型VPCエンドポイント（PrivateLink）
- B. NAT Gateway
- C. インターネットゲートウェイ
- D. VPCピアリング

> [!success]- 答え：**A**（インターフェース型エンドポイント）
> インターフェース型VPCエンドポイント（PrivateLink）はENI（プライベートIP）経由で多くのAWSサービスへ接続でき、インターネット/NATを経由しない。
> **他の選択肢が違う理由**
> - B：NAT Gatewayはインターネット経由の外向き通信で「非インターネット」に反し、コストもかかる。
> - C：IGWはインターネット出入口で要件に反する。
> - D：VPCピアリングはVPC間接続でAWSサービスへの接続手段ではない。
> **覚えるべきキーワード**：PrivateLink / インターフェース型 / ENI / 非インターネット
> **関連サービス**：ゲートウェイ型（S3・DynamoDB限定・無料） / Route 53
> **SAAでの出題イメージ**：「AWSサービスへプライベート接続」=インターフェース型VPCエンドポイント。
> 📎 参照：[[0406_VPC応用機能]]

## 第12問（ハイブリッドストレージ）
オンプレのアプリから、ローカルと同じ感覚でファイルを扱いつつ、実体はS3に保存して容量を気にせず使いたい（よく使うものはローカルキャッシュ）。最適なのはどれか。

- A. AWS Storage Gateway（File Gateway）
- B. AWS DataSync で一度だけ移行
- C. S3をインターネット経由で直接マウント
- D. EBSをオンプレに接続

> [!success]- 答え：**A**（Storage Gateway / File Gateway）
> Storage GatewayはオンプレからNFS/SMBでアクセスでき、実体をS3に保存しつつローカルキャッシュで低遅延を保つハイブリッドストレージ。日常利用に適する。
> **他の選択肢が違う理由**
> - B：DataSyncは転送（移行・同期）ツールで、日常的なファイルアクセスのゲートウェイではない。
> - C：S3は直接ファイルシステムマウントする日常利用向けではない。
> - D：EBSはEC2用ブロックストレージでオンプレ接続できない。
> **覚えるべきキーワード**：Storage Gateway / NFS/SMB / S3バックエンド / ローカルキャッシュ
> **関連サービス**：DataSync / S3 / FSx
> **SAAでの出題イメージ**：「オンプレからローカル感覚でS3を日常利用」=Storage Gateway。
> 📎 参照：[[1502_DataSync vs Firehose vs Storage Gateway]] / [[1501_移行と転送サービス一覧]]

## 第13問（移行／オンライン）
オンプレNASの数十TBを、十分な回線を使ってAWS（S3/EFS）へオンラインで移行し、以後も増分を定期同期したい。最適なのはどれか。

- A. AWS DataSync
- B. AWS Snowball Edge
- C. S3コンソールから手動アップロード
- D. CloudFrontでアップロード

> [!success]- 答え：**A**（DataSync）
> DataSyncはオンプレ⇄AWSのオンライン転送を高速・自動・暗号化で行い、増分の定期同期もできる。回線があるならNAS移行＋継続同期に最適。
> **他の選択肢が違う理由**
> - B：Snowball Edgeは物理デバイスのオフライン転送で、回線が細い/オフライン要件向け。
> - C：手動アップロードは大量・継続同期に不向き。
> - D：CloudFrontは配信CDNで移行ツールではない。
> **覚えるべきキーワード**：DataSync / オンライン高速転送 / 増分定期同期
> **関連サービス**：Snowball / Storage Gateway / S3 / EFS
> **SAAでの出題イメージ**：「回線ありでオンライン移行・同期」=DataSync。
> 📎 参照：[[1501_移行と転送サービス一覧]] / [[1502_DataSync vs Firehose vs Storage Gateway]]

## 第14問（移行／オフライン）
ネットワーク回線が細い拠点から、ペタバイト級のデータをAWSへ移したい。オンライン転送では数か月かかってしまう。最適なのはどれか。

- A. AWS DataSync
- B. AWS Snowball / Snowmobile
- C. Site-to-Site VPN経由で転送
- D. S3 Transfer Acceleration

> [!success]- 答え：**B**（Snowball / Snowmobile）
> 回線が細く大容量（ペタ級）はSnowball（物理デバイス）やSnowmobile（トラック）でオフライン転送するのが現実的。デバイスにコピーしてAWSへ送付・取り込む。
> **他の選択肢が違う理由**
> - A：DataSyncはオンライン転送で、細い回線・ペタ級ではオンラインの限界に当たる。
> - C：VPN経由も回線帯域に律速されオフライン転送の代替にならない。
> - D：Transfer AccelerationはS3アップロード高速化だが、根本の回線が細い前提では不足。
> **覚えるべきキーワード**：Snowball / オフライン物理転送 / 回線が細い・大容量
> **関連サービス**：DataSync（オンライン） / Snowmobile（超大容量） / S3
> **SAAでの出題イメージ**：「回線細い・大容量・オフライン」=Snowball系。
> 📎 参照：[[1501_移行と転送サービス一覧]]

## 第15問（ストレージ選定／Windows共有）
複数のWindows EC2から、SMBプロトコル・Active Directory統合の共有ファイルストレージを使いたい。最適なのはどれか。

- A. Amazon EFS
- B. Amazon FSx for Windows File Server
- C. Amazon S3
- D. Amazon EBS

> [!success]- 答え：**B**（FSx for Windows File Server）
> FSx for Windows File ServerはSMB対応・AD統合のフルマネージドWindows共有ファイルシステム。Windowsワークロードのファイル共有に適する。
> **他の選択肢が違う理由**
> - A：EFSはNFS（Linux向け）共有で、SMB/AD統合のWindows用途には合わない。
> - C：S3はオブジェクトストレージで、SMBファイル共有としての日常利用には向かない。
> - D：EBSは1台占有のブロックストレージで共有ファイルではない。
> **覚えるべきキーワード**：FSx for Windows / SMB / AD統合 / Windows共有
> **関連サービス**：EFS（Linux/NFS） / FSx for Lustre（HPC） / Storage Gateway
> **SAAでの出題イメージ**：「Windows・SMB・AD統合の共有」=FSx for Windows、「Linux共有」=EFS。
> 📎 参照：[[0202_S3とEBS・EFSの違い]]

## 第16問（バックアップ統合）
EBS・RDS・DynamoDB・EFSなど複数サービスのバックアップを、統一ポリシーでスケジュール・保持・クロスリージョンコピーし、コンプライアンス監査も楽にしたい。最適なのはどれか。

- A. AWS Backup
- B. 各サービスで個別にスナップショットを手動取得
- C. S3にcronでコピー
- D. CloudWatchアラームで通知するだけ

> [!success]- 答え：**A**（AWS Backup）
> AWS Backupはバックアッププランで複数サービスのバックアップを一元管理し、スケジュール・保持・ライフサイクル・クロスリージョン/アカウントコピーを統一適用、監査レポートも提供する。
> **他の選択肢が違う理由**
> - B：個別手動は取り忘れ・運用負荷・統制不足。
> - C：cronコピーは整合性・横断管理・監査の面で不十分。
> - D：通知だけではバックアップ自体を取れない。
> **覚えるべきキーワード**：AWS Backup / バックアッププラン / 横断一元管理 / 監査
> **関連サービス**：EBS/RDS/DynamoDB/EFSスナップショット / Organizations
> **SAAでの出題イメージ**：「複数サービスのバックアップを統一ポリシー＋監査」=AWS Backup。
> 📎 参照：[[0301_RDS]] / [[1501_移行と転送サービス一覧]]

## 第17問（脅威検知）
アカウント内の不審なAPI操作・侵害された認証情報・C2通信などの脅威を、エージェント導入なしに継続的に検知・通知したい。最適なのはどれか。

- A. Amazon GuardDuty
- B. AWS WAF
- C. Amazon Inspector
- D. AWS Config

> [!success]- 答え：**A**（GuardDuty）
> GuardDutyはCloudTrail/VPCフローログ/DNSログを機械学習で継続分析し、不審な挙動を検知・通知する（エージェント不要）。脅威検知の標準解。
> **他の選択肢が違う理由**
> - B：WAFはWebアプリ層のリクエスト遮断で、アカウント全体の挙動検知ではない。
> - C：Inspectorは脆弱性スキャンで、挙動ベースの脅威検知ではない。
> - D：Configは構成準拠チェックで脅威検知ではない。
> **覚えるべきキーワード**：脅威検知 / 機械学習 / ログ分析 / エージェント不要
> **関連サービス**：Security Hub / Detective / EventBridge（自動対応）
> **SAAでの出題イメージ**：「ログを分析して脅威を継続検知」=GuardDuty。
> 📎 参照：[[1603_脅威検知・監査]]

## 第18問（機密データ検出）
S3バケットに保存された大量データの中から、個人情報（PII）やクレジットカード番号などの機密データを自動的に発見・分類したい。最適なのはどれか。

- A. Amazon Macie
- B. Amazon GuardDuty
- C. AWS KMS
- D. Amazon Inspector

> [!success]- 答え：**A**（Macie）
> MacieはS3を機械学習でスキャンし、PII・機密データを自動で発見・分類してリスクを可視化するデータセキュリティサービス。
> **他の選択肢が違う理由**
> - B：GuardDutyは脅威検知（挙動分析）で、S3内の機密データ分類はしない。
> - C：KMSは暗号鍵管理で、機密データの発見・分類機能ではない。
> - D：Inspectorは脆弱性スキャンでデータ分類ではない。
> **覚えるべきキーワード**：Macie / PII検出 / 機密データ分類 / S3
> **関連サービス**：GuardDuty / Security Hub / KMS（暗号化）
> **SAAでの出題イメージ**：「S3内の個人情報・機密を自動検出」=Macie。
> 📎 参照：[[1603_脅威検知・監査]] / [[1605_データ保護・暗号化]]

## 第19問（認証）
モバイルアプリのユーザー登録・ログイン（ソーシャルログイン含む）を管理し、認証済みユーザーにだけバックエンドAPI（API Gateway）へのアクセスを許可したい。最適なのはどれか。

- A. Amazon Cognito（ユーザープール＋API Gatewayオーソライザ）
- B. 各EC2に独自の認証コードを実装
- C. IAMユーザーを会員ごとに発行
- D. S3に認証情報を平文保存

> [!success]- 答え：**A**（Cognito＋オーソライザ）
> Cognitoユーザープールで会員認証を行い、発行されるトークンをAPI Gatewayのオーソライザで検証して、認証済みユーザーだけにAPIアクセスを許可するのがサーバーレスの定番。
> **他の選択肢が違う理由**
> - B：独自実装は車輪の再発明で、セキュリティ・運用負荷が大きい。
> - C：会員ごとにIAMユーザーを発行するのは設計として不適切（IAMはAWS操作者向け）。
> - D：平文保存は論外。
> **覚えるべきキーワード**：Cognito / ユーザープール / API Gatewayオーソライザ / トークン
> **関連サービス**：API Gateway / Lambda / IAM
> **SAAでの出題イメージ**：「アプリ会員認証＋API保護」=Cognito＋API Gateway。
> 📎 参照：[[0810_Cognito]] / [[0809_API_Gateway]]

## 第20問（運用／安全な接続）
プライベートサブネットのEC2に、踏み台サーバー（Bastion）やSSHの22番ポート開放なしで、監査ログ付きで安全にシェル接続したい。最適なのはどれか。

- A. グローバルIPを付けてSSHを開放する
- B. Systems Manager Session Manager
- C. 踏み台EC2を常時起動して経由する
- D. NACLで22番を全許可する

> [!success]- 答え：**B**（Session Manager）
> Systems Manager Session ManagerはSSMエージェント経由でEC2へシェル接続でき、22番ポート開放・踏み台・SSHキー管理が不要。接続はIAMで制御し、CloudTrail/ログで監査できる。
> **他の選択肢が違う理由**
> - A：グローバルIP＋SSH開放は攻撃面が広く危険。
> - C：踏み台EC2は運用負荷・コスト・攻撃面が増える（Session Managerで不要にできる）。
> - D：NACLで22番全許可はセキュリティ上不適切。
> **覚えるべきキーワード**：Session Manager / 踏み台不要 / ポート開放不要 / IAM制御・監査
> **関連サービス**：Systems Manager / IAM / CloudTrail / VPCエンドポイント
> **SAAでの出題イメージ**：「踏み台なし・ポート開放なしで安全にEC2接続」=Session Manager。
> 📎 参照：[[1405_Systems Manager]]
