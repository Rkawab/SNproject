# 【問題】シナリオ3（公式サンプル風・全10問）

## この問題集の目的

複数の要件を同時に満たす構成を選ぶ。スケーリング・コスト最適化・暗号化・サーバーレス・コンテナ・機械学習など13分野を横断する。

## 第1問（Compute ｜ 需要変動への自動スケール）
あるWebアプリケーションは、日中にトラフィックが急増し夜間は大きく減少する。現在は管理者が手動でEC2の台数を調整しているが、対応が追いつかず、過剰プロビジョニングによるコストも問題になっている。手動調整をやめ、需要に応じて自動でスケールしつつ可用性も確保したい。最も適切な構成はどれか。

- A. EC2 Auto Scaling groupをターゲット追跡スケーリングで複数AZに構成し、Application Load Balancer配下に置く
- B. EC2 Auto Scaling groupを単一AZに構成し、スケジュールスケーリングだけで日中の台数を増やす
- C. Application Load Balancer配下に複数AZのEC2を配置し、CPU使用率はCloudWatch Dashboardで可視化する
- D. 既存のEC2をより大きなインスタンスタイプへ変更し、夜間も同じサイズのまま稼働させる

> [!success]- 答え：**A**
> Auto Scaling group＋ALB＋複数AZは、需要に応じた自動スケールと高可用性を両立する定番構成。
> - A：正解。メトリクスに応じて台数を自動増減し、AZ障害にも耐えられる。
> - B：スケジュールが読める負荷には使えるが、単一AZで可用性が不足し、急な変動にも弱い。
> - C：複数AZとALBで可用性は上がるが、可視化だけでは需要に応じた自動増減ができない。
> - D：スケールアップは一定の効果があるが、弾力性がなく過剰プロビジョニングも残る。
> **関連ノート**：[[0504_Auto_Scaling_と_Auto_Scaling_Group]] / [[0501_高可用性とスケーラビリティ]]

## 第2問（Storage ｜ アーカイブのライフサイクル最適化）
監査ログをS3に7年間保管する必要がある。最初の90日はたまに参照するが、それ以降はほとんど参照されず、取り出しには数時間待っても構わない。運用の手間をかけずにストレージコストを最小化したい。取るべきアクションの組み合わせはどれか。（2つ選択）

- A. 90日経過後にS3 Glacier Flexible Retrieval（または Deep Archive）へ移行するライフサイクルルールを設定する
- B. 7年経過後にオブジェクトを失効（削除）するライフサイクルルールを設定する
- C. S3 Intelligent-Tieringを有効化し、保持期限の削除は行わずアクセス階層の自動移動だけに任せる
- D. 90日経過後にS3 Standard-IAへ移行するが、7年経過後の失効ルールは設定しない
- E. S3 Inventoryで古いオブジェクトを月次で一覧化し、担当者が必要に応じて削除する

> [!success]- 答え：**A, B**
> ライフサイクルルールで「古いデータをアーカイブクラスへ移行」「保持期限到来で自動削除」を組み合わせると、運用ゼロでコスト最適化できる。
> - A：正解。参照頻度が下がった時点で低コストなアーカイブクラスへ自動移行できる。
> - B：正解。保持期間到来時の自動削除で不要な保管コストを避けられる。
> - C：アクセス階層の自動最適化はできるが、7年保持後の自動削除がなく、最小コストのアーカイブ移行も明示できない。
> - D：移行だけでは保持期限到来後の削除が自動化されず、不要な保管コストが残る。
> - E：Inventoryは可視化には使えるが、移行・削除を自動化しないため運用の手間が残る。
> **関連ノート**：[[0204_S3のライフサイクル管理]] / [[0203_S3ストレージクラス]]

## 第3問（Database ｜ 読み取り負荷のオフロード）
RDS for MySQLで運用しているアプリケーションで、レポート用の重い読み取りクエリが増え、本番の書き込み性能に影響が出ている。アプリケーションの改修を最小化しつつ、参照系の負荷をオフロードしたい。最も適切な方法はどれか。

- A. リードレプリカを作成し、レポート用などの参照系クエリをレプリカへ向ける
- B. Amazon RDS Proxyを導入し、アプリケーションからの接続をプールして再利用する
- C. Multi-AZのスタンバイインスタンスへ読み取りクエリを向ける
- D. レポート処理の直前に本番DBのスナップショットを取得し、別DBとして復元してから集計する

> [!success]- 答え：**A**
> リードレプリカは参照系トラフィックをプライマリから切り離し、読み取りスケールを実現する。
> - A：正解。接続先を分けるだけで参照負荷をオフロードできる。
> - B：接続数の急増には有効だが、重い読み取りクエリをプライマリから逃がす機能ではない。
> - C：RDSのMulti-AZスタンバイは原則として読み取りに使えない（フェイルオーバー用）。
> - D：本番への影響は減らせる場合があるが、復元待ちが発生し、継続的な読み取りオフロードには向かない。
> **関連ノート**：[[0301_RDS]] / [[0304_ElastiCache]]

## 第4問（Networking / CDN / LB ｜ S3へのプライベートアクセス）
プライベートサブネットのEC2インスタンスが、現在はNAT Gateway経由でS3にアクセスしている。S3向けのデータ転送料とNATの負荷を減らすため、トラフィックをインターネットに出さずVPC内で完結させたい。最も適切な構成はどれか。

- A. S3用のゲートウェイVPCエンドポイントを作成し、プライベートサブネットのルートテーブルに関連付ける
- B. NAT Gatewayを各AZに追加し、S3向けトラフィックも引き続きNAT経由にする
- C. S3用のインターフェイス型VPCエンドポイントを作成し、全S3トラフィックをPrivateLink経由にする
- D. S3用ゲートウェイVPCエンドポイントを作成するが、プライベートサブネットのルートテーブルは変更しない

> [!success]- 答え：**A**
> S3向けゲートウェイVPCエンドポイントを使うと、トラフィックがAWS内部を通り、NAT経由のデータ転送料を回避できる。
> - A：正解。インターネットを経由せずS3へプライベートにアクセスでき、コストも下がる。
> - B：可用性は上がるが、NAT Gatewayのデータ処理料金は残る。
> - C：PrivateLinkでS3へ接続する選択肢もあるが、S3向けの大量通信のコスト最適化ではゲートウェイ型エンドポイントが一般的。
> - D：エンドポイントを作るだけでは経路が変わらず、NAT経由の通信が残る。
> **関連ノート**：[[0409_IGW・NAT Gateway・VPCエンドポイント]]

## 第5問（Security / IAM ｜ 保存データの暗号化と鍵の制御）
コンプライアンス要件により、S3に保存する機密データを暗号化し、暗号鍵の利用を監査ログで追跡し、鍵ポリシーで利用者を制御したい。運用負荷は抑えたい。最も適切な方法はどれか。

- A. AWS KMSのカスタマーマネージドキーでSSE-KMSを有効にする
- B. SSE-S3を有効化し、暗号鍵の管理と使用履歴の監査はAmazon S3に任せる
- C. SSE-Cを使用し、暗号鍵をアプリケーション側で都度指定してS3に保存する
- D. AWS管理キー（aws/s3）によるSSE-KMSを使用し、独自のキーポリシーは設定しない

> [!success]- 答え：**A**
> SSE-KMS＋カスタマーマネージドキーは、保存時暗号化・鍵ポリシーによる利用制御・CloudTrailでの鍵利用監査を満たす。
> - A：正解。鍵の制御と監査、保存時暗号化を運用負荷低く実現できる。
> - B：保存時暗号化はできるが、カスタマー管理キーによる利用者制御や鍵利用監査の要件が弱い。
> - C：鍵の制御はできるが、鍵管理をアプリケーション側で担うため運用負荷が高い。
> - D：KMS連携はできるが、AWS管理キーでは独自のキーポリシーで利用者を細かく制御しにくい。
> **関連ノート**：[[1607_AWS_KMS]] / [[1605_データ保護・暗号化]] / [[0207_S3の暗号化]]

## 第6問（Serverless ｜ サーバーレスAPIバックエンド）
モバイルアプリ向けのREST APIを、サーバー管理なしで構築し、トラフィックに応じて自動スケールさせたい。データはキー・バリュー形式で保存し、運用負荷を最小化したい。最も適切な構成はどれか。

- A. Amazon API Gateway ＋ AWS Lambda ＋ Amazon DynamoDB
- B. Application Load Balancer ＋ EC2 Auto Scaling ＋ Amazon RDSでAPIを構築する
- C. Amazon API Gateway ＋ AWS Lambda ＋ Amazon RDS for MySQLでAPIを構築する
- D. AWS App RunnerでコンテナAPIを実行し、データをEFSにファイルとして保存する

> [!success]- 答え：**A**
> API Gateway＋Lambda＋DynamoDBは、サーバーレスでスケーラブルなAPIバックエンドの定番構成。
> - A：正解。サーバー管理なしで自動スケールし、キー・バリューデータを低運用で扱える。
> - B：スケーラブルにはできるが、EC2とRDSの運用が残り、サーバー管理なしの要件には合わない。
> - C：API GatewayとLambdaは合うが、キー・バリュー形式を低運用で扱うならDynamoDBのほうが適切。
> - D：サーバー管理は抑えられるが、キー・バリューデータの主ストアとしてEFSファイル保存は不向き。
> **関連ノート**：[[0303_DynamoDB]] / [[0708_Lambda]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]]

## 第7問（Containers ｜ コンテナの自動スケール）
ECS on Fargateで稼働するマイクロサービスが、昼のピーク時間帯に処理が詰まり遅延が発生する。サーバー管理を増やさず、需要に応じてタスク数を自動で増減させたい。最も適切な方法はどれか。

- A. Service Auto Scaling（ターゲット追跡）を設定し、CPU使用率などに応じてタスク数を増減する
- B. ECSサービスのスケジュールスケーリングだけを設定し、昼のピーク時間帯に固定数まで増やす
- C. Fargate Spotだけでサービスを構成し、中断時は利用者に再試行してもらう
- D. タスク定義のCPUとメモリを大きくし、タスク数は最小数のまま維持する

> [!success]- 答え：**A**
> ECS Service Auto Scalingは、メトリクスに応じてサービスのタスク数を自動でスケールさせる。
> - A：正解。Fargateのままサーバー管理なしでタスク数を自動調整できる。
> - B：予測可能な時間帯には有効だが、実際の負荷に応じた自動増減ではない。
> - C：コスト削減には有効な場合があるが、中断により通常サービスの可用性を損なう可能性がある。
> - D：垂直方向の余裕は増えるが、需要に応じたタスク数の自動増減にはならない。
> **関連ノート**：[[0904_ECS]] / [[0902_オーケストレーション_フルマネージド]]

## 第8問（Cost Optimization ｜ バッチ処理のコスト削減）
夜間に実行する画像変換バッチは、中断されても安全に再実行でき、完了が多少遅れても許容される。コストを最小化しつつ確実に処理を完了させたい。取るべきアクションの組み合わせはどれか。（2つ選択）

- A. EC2スポットインスタンスを使い、中断された場合は別キャパシティで再試行する
- B. ジョブをSQSキューに入れ、ワーカーがメッセージを取り出して冪等に処理する
- C. AWS Batchをオンデマンドインスタンスのみで構成し、キューは使わず投入順に処理する
- D. Compute Savings Plansでピーク時の最大処理量を1年間コミットし、常時その容量を維持する
- E. 画像ファイルをS3イベント通知でLambdaへ直接渡し、失敗時の再処理設計は行わない

> [!success]- 答え：**A, B**
> スポットで安価に処理しつつ、SQSでジョブを保持して冪等に再処理すれば、中断耐性とコスト最小化を両立できる。
> - A：正解。中断耐性のあるバッチで最大の割引を得られる。
> - B：正解。キューで未処理ジョブを保持し、中断後も取りこぼさず処理を完了できる。
> - C：AWS Batch自体はバッチ管理に有効だが、オンデマンドのみではコスト最小化の要件に届きにくい。
> - D：継続利用の割引にはなるが、変動する中断許容ジョブに常時ピーク容量を維持するのは割高。
> - E：イベント駆動にはできるが、失敗時や中断時にジョブを確実に保持・再処理する仕組みが弱い。
> **関連ノート**：[[0102_EC2購入プラン]] / [[0802_SQS]]

## 第9問（Management / Monitoring ｜ しきい値超過時の自動対応）
EC2インスタンスのCPU使用率が一定時間しきい値を超えたら、運用担当者へ通知しつつ、自動でスケールアウトさせたい。手動監視はやめたい。最も適切な構成はどれか。

- A. CloudWatchアラームでSNSへの通知とAuto Scalingポリシーの両方をトリガーする
- B. CloudWatchアラームでSNS通知だけを行い、Auto Scalingポリシーには関連付けない
- C. EventBridgeでCloudTrailのRunInstancesイベントを検出し、SNSへ通知する
- D. Auto Scalingのスケジュールスケーリングで毎日同じ時刻に台数を増やす

> [!success]- 答え：**A**
> CloudWatchアラームはしきい値超過を検知し、SNS通知とAuto Scalingアクションを自動実行できる。
> - A：正解。通知と自動スケールアウトを無人で実現できる。
> - B：通知はできるが、スケールアウトの自動実行が行われない。
> - C：API監査イベントの通知であり、CPUメトリクスに基づくスケールアウトではない。
> - D：予測可能な負荷には使えるが、CPUしきい値に応じた通知と自動対応ではない。
> **関連ノート**：[[1001_CloudWatch]] / [[0504_Auto_Scaling_と_Auto_Scaling_Group]]

## 第10問（Machine Learning ｜ 問い合わせの感情分析）
コールセンターでテキスト化された大量の問い合わせから、感情（ポジティブ／ネガティブ）やキーフレーズ、エンティティを自動抽出したい。機械学習モデルの構築・訓練は行いたくない。最も適切なサービスはどれか。

- A. Amazon Comprehend
- B. Amazon Translate
- C. Amazon SageMaker
- D. Amazon Kendra

> [!success]- 答え：**A**
> Comprehendは自然言語処理の学習済みAPIで、感情分析・キーフレーズ抽出・エンティティ認識などを提供する。
> - A：正解。モデル訓練なしでテキストの感情・キーフレーズを抽出できる。
> - B：言語翻訳サービスで、感情分析は行わない。
> - C：自前でMLモデルを構築・訓練する基盤で、要件に対して過剰。
> - D：エンタープライズ検索サービスで、感情分析が主目的ではない。
> **関連ノート**：[[1301_AWS機械学習サービス一覧]]
