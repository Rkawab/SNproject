# 【問題】サービス用途確認9（公式サンプル風・全10問）

## この問題集の目的

要件文から主要サービスの役割を選ぶドリル。101〜108で扱っていない切り口（サーバーレスコンテナ実行・S3静的ウェブサイトホスティング・グローバルなNoSQLテーブル・VPCエンドポイント・S3内の機密データ発見・バックアップの一元管理・大量Eメール送信・サーバーレスETL・大容量データの物理搬送移行・テキストの感情分析など）を中心に、13分野を横断する。

## 対象レベル

SAA入口。単語暗記ではなく、要件からサービスを選ぶ練習にする。

## 問題数

10問

## 第1問（Compute ｜ サーバー管理なしでコンテナを実行）
コンテナ化したアプリを動かしたいが、EC2インスタンスのプロビジョニング・パッチ適用・スケール管理はしたくない。タスク単位でコンテナだけを起動し、使った分だけ課金されるのが理想。最も適切なサービスはどれか。

- A. AWS Fargate
- B. Amazon EC2 上に構築した Docker ホスト
- C. AWS Lambda
- D. AWS Batch

> [!success]- 答え：**A**
> FargateはECS/EKSのコンテナを、サーバー（EC2）の管理なしでタスク/ポッド単位に実行するサーバーレスなコンテナ実行基盤。
> - A：正解。インスタンス管理不要でコンテナだけを起動し、消費リソースに応じて課金される。
> - B：コンテナは動かせるが、ホストOSのパッチ・スケール・容量管理を自前で行う必要がある。
> - C：関数（コード）の実行に最適化されており、任意のコンテナイメージを長時間稼働させる主目的の基盤ではない。
> - D：大量のバッチジョブのキューイング実行向けで、常駐型サービスのコンテナ実行基盤ではない。
> **関連ノート**：[[0902_オーケストレーション_フルマネージド]] / [[0701_コンピューティングサービス一覧]]

## 第2問（Storage ｜ 静的ウェブサイトのホスティング）
HTML・CSS・JavaScript・画像だけで構成される静的サイトを、サーバーを一切立てずに低コストで公開したい。可用性が高く、アクセス増にも自動で耐えてほしい。最も適切なサービスはどれか。

- A. Amazon S3 の静的ウェブサイトホスティング
- B. Amazon EC2 上の Web サーバー
- C. Amazon EBS
- D. AWS Storage Gateway

> [!success]- 答え：**A**
> S3はバケットに置いた静的ファイルをそのままWebサイトとして配信でき、サーバー不要・高耐久・自動スケールで低コスト。
> - A：正解。静的コンテンツの公開にサーバーが不要で、運用負荷とコストが最小。
> - B：サーバーの構築・パッチ・スケール管理が必要で、静的サイトには過剰。
> - C：単一EC2にアタッチするブロックストレージで、それ自体がWeb配信を行うわけではない。
> - D：オンプレとAWSをつなぐハイブリッドストレージで、静的サイト公開の用途ではない。
> **関連ノート**：[[0108_EC2とS3のWebサイトの違い]] / [[0201_S3]]

## 第3問（Database ｜ 複数リージョンで読み書きする低遅延NoSQL）
世界各地の利用者に対し、複数リージョンで読み書きでき、各地から低レイテンシでアクセスできるフルマネージドなキー・バリューDBが欲しい。リージョン間のレプリケーションも自動でやってほしい。最も適切なサービスはどれか。

- A. Amazon DynamoDB グローバルテーブル
- B. Amazon RDS for MySQL（シングルリージョン・マルチAZ）
- C. Amazon Redshift
- D. Amazon ElastiCache

> [!success]- 答え：**A**
> DynamoDBグローバルテーブルは複数リージョンにアクティブ-アクティブで複製され、各リージョンで低遅延の読み書きを提供する。
> - A：正解。マルチリージョンの低遅延読み書きと自動レプリケーションを満たす。
> - B：マルチAZは単一リージョン内の冗長化で、複数リージョンでの低遅延な書き込みには対応しない。
> - C：分析向けデータウェアハウスで、低レイテンシのキー・バリュー処理用途ではない。
> - D：インメモリキャッシュで、永続的なマルチリージョンDBそのものではない。
> **関連ノート**：[[0303_DynamoDB]] / [[0301_RDS]]

## 第4問（Networking / CDN / LB ｜ VPCからS3へのプライベート接続）
プライベートサブネットのEC2から、インターネットを経由せずにS3へアクセスしたい。NATゲートウェイの追加コストも避けたい。最も適切な構成はどれか。

- A. S3用のVPCゲートウェイエンドポイントを作成する
- B. NATゲートウェイを追加してS3へ出る
- C. インターネットゲートウェイを各サブネットにアタッチする
- D. S3バケットをパブリック公開する

> [!success]- 答え：**A**
> S3/DynamoDB向けのVPCゲートウェイエンドポイントは、ルートテーブル経由でインターネットを使わずプライベートにアクセスでき、追加料金もかからない。
> - A：正解。インターネット非経由かつ追加コストなしでS3に到達できる。
> - B：S3には到達できるが、インターネット経由でNATの時間・転送課金が発生し要件に反する。
> - C：サブネットをパブリック化するもので、プライベート接続の要件に反する。
> - D：バケットを公開すると重大なセキュリティリスクで、プライベート接続の解にならない。
> **関連ノート**：[[0409_IGW・NAT Gateway・VPCエンドポイント]] / [[0406_VPC応用機能]]

## 第5問（Security / IAM ｜ S3内の機密データの発見と分類）
S3バケットに保存された大量のオブジェクトの中から、クレジットカード番号や個人情報などの機密データを機械学習で自動的に発見・分類し、可視化したい。最も適切なサービスはどれか。

- A. Amazon Macie
- B. Amazon GuardDuty
- C. Amazon Inspector
- D. AWS WAF

> [!success]- 答え：**A**
> MacieはS3上のデータを機械学習でスキャンし、個人情報などの機密データを検出・分類してリスクを可視化するサービス。
> - A：正解。S3内の機密データ（PII等）の発見・分類に特化している。
> - B：アカウント全体の不審な挙動を検知する脅威検知サービスで、S3内のデータ内容の分類ではない。
> - C：EC2やコンテナ等の脆弱性・到達可能性の評価が主で、S3内の機密データ分類ではない。
> - D：L7のWebリクエストをルールで遮断する防御で、データ分類の用途ではない。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[1605_データ保護・暗号化]]

## 第6問（Management / Monitoring ｜ バックアップの一元管理）
EBS・RDS・DynamoDB・EFSなど複数サービスのバックアップを、サービスごとに個別設定するのではなく、共通のポリシー（スケジュール・保持期間）で一元的に管理・自動化したい。最も適切なサービスはどれか。

- A. AWS Backup
- B. Amazon S3 のライフサイクルポリシー
- C. 各サービスで手動スナップショットを取得する
- D. AWS CloudFormation

> [!success]- 答え：**A**
> AWS Backupは複数のAWSサービスのバックアップを、バックアッププランとして一元的にスケジュール・保持管理・自動化できる。
> - A：正解。サービス横断のバックアップを共通ポリシーで集中管理できる。
> - B：S3オブジェクトのストレージクラス移行・削除の自動化で、複数サービス横断のバックアップ管理ではない。
> - C：個別・手動運用となり、一元管理・自動化という要件に反する。
> - D：インフラのプロビジョニング（IaC）が主目的で、バックアップの集中管理サービスではない。
> **関連ノート**：[[1401_マネジメント・ガバナンスサービス一覧]] / [[0104_スナップショット・AMIとは]]

## 第7問（Application Integration ｜ 大量のEメール送信）
アプリから取引確認メールやニュースレターを、高い到達率で大量に送信したい。SMTPまたはAPIで送れ、バウンス・苦情の管理もしたい。最も適切なサービスはどれか。

- A. Amazon SES
- B. Amazon SNS
- C. Amazon SQS
- D. Amazon Pinpoint Voice

> [!success]- 答え：**A**
> SES（Simple Email Service）はトランザクション/マーケティングEメールをAPIやSMTPで大量送信でき、到達率・バウンス・苦情の管理機能を備える。
> - A：正解。大量Eメール送信と到達率・評判管理に最適。
> - B：SMS/プッシュ/HTTP等への汎用プッシュ通知（Pub/Sub）で、Eメール配信の専用サービスではない。
> - C：メッセージのキューイング（バッファ）が主で、メール送信サービスではない。
> - D：音声メッセージ向けで、Eメールの大量送信用途ではない。
> **関連ノート**：[[0813_SES]] / [[0803_SNS]]

## 第8問（Analytics ｜ サーバーレスなETLとデータカタログ）
複数のデータソースからデータを抽出・変換し、分析用に整える処理（ETL）を行いたい。サーバーの管理はせず、メタデータのカタログ化やスキーマ自動検出も任せたい。最も適切なサービスはどれか。

- A. AWS Glue
- B. Amazon Athena
- C. Amazon EMR
- D. Amazon QuickSight

> [!success]- 答え：**A**
> Glueはサーバーレスのデータ統合サービスで、ETLジョブの実行に加え、クローラによるスキーマ検出とデータカタログを提供する。
> - A：正解。サーバーレスETLとデータカタログ（メタデータ管理）に最適。
> - B：S3上データへのアドホックなSQLクエリ用途で、ETLパイプラインとカタログ管理の主体ではない。
> - C：マネージドHadoop/Sparkクラスターで、クラスター運用が前提のためサーバーレスETLという要件に合わない。
> - D：BIダッシュボードによる可視化が主で、ETL処理サービスではない。
> **関連ノート**：[[1107_AWS Glue]] / [[1101_AWS分析サービス一覧]]

## 第9問（Migration / Transfer ｜ 大容量データの物理搬送移行）
ネットワーク回線が細いデータセンターから、数十TBのデータを現実的な期間でAWSへ移したい。オンライン転送では数か月かかってしまう。最も適切な方法はどれか。

- A. AWS Snowball（Snow Family）で物理デバイスに格納して搬送する
- B. インターネット経由で S3 へ直接アップロードする
- C. AWS DataSync でオンライン同期する
- D. AWS Direct Connect を新規敷設してから転送する

> [!success]- 答え：**A**
> Snow Family（Snowball等）は物理ストレージデバイスにデータを格納して物流で搬送し、細い回線でも大容量データを現実的な期間で移行できる。
> - A：正解。回線が細い環境での大容量データ移行に最適なオフライン搬送手段。
> - B：細い回線では数か月かかり、現実的な期間という要件を満たさない。
> - C：オンライン転送のため回線速度に律速され、細い回線では時間がかかりすぎる。
> - D：敷設には時間とコストがかかり、一度きりの大容量移行の手段としては過剰で遅い。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[1502_DataSync]]

## 第10問（Machine Learning ｜ テキストの感情分析・キーフレーズ抽出）
顧客レビューの文章から、感情（ポジティブ/ネガティブ）・キーフレーズ・固有表現を自動抽出したい。機械学習モデルの構築はせず、APIを呼ぶだけで使いたい。最も適切なサービスはどれか。

- A. Amazon Comprehend
- B. Amazon Rekognition
- C. Amazon Transcribe
- D. Amazon Translate

> [!success]- 答え：**A**
> Comprehendは自然言語処理（NLP）のマネージドサービスで、感情分析・キーフレーズ抽出・固有表現抽出などをAPIで提供する。
> - A：正解。学習不要で、テキストの感情・キーフレーズ等をAPIで分析できる。
> - B：画像・動画の物体・顔検出向けで、テキスト分析ではない。
> - C：音声をテキストに変換する用途で、テキストの感情分析ではない。
> - D：テキストの言語間翻訳向けで、感情・キーフレーズの抽出ではない。
> **関連ノート**：[[1301_AWS機械学習サービス一覧]]
