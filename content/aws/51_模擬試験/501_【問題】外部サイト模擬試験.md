外部サイトの模擬問題を、答えの暗記ではなく本番形式（長文シナリオ）で解く練習として収録する。分野は固定せず、コンピュート選定・アプリケーション統合・コスト最適化など、出題された問題を順次このファイルに追加していく。

> [!info] 出題形式
> - シナリオ段落と設問文を分け、です・ます調で記述。
> - 選択肢はサービス名ではなく、具体的な操作・構成で記述。
> - 原則は単一選択（4択 A〜D）。複数選択の問題は5択（A〜E）とし、設問文に「2つ選択」等を明記。

## 第1問（Containers ｜ バッチ処理のコンテナ移行とイベント駆動）
ある企業は、オンプレミスで稼働しているレガシーな在庫管理およびレポート生成バッチを、AWSにリフト＆シフトで移行したいと考えています。現在、Linuxのcronによって「5分おき」「毎時20分」など細かくスケジュールされたジョブが数百本存在しています。それぞれのジョブはBashやPythonスクリプトで構成されており、実行時間は数秒〜30分です。これらのジョブは、在庫データの集計・外部パートナーへのファイル出力・レポート生成などを行っており、可能な限り既存コードを書き換えずにAWSに移行したいと考えています。さらに、将来的には「在庫変動イベント」や「外部トリガー」に応じて、該当ジョブを即時実行できるイベント駆動の柔軟性も求められています。

この要件を最も満たすソリューションはどれですか。

- A. 各ジョブスクリプトをコンテナ化し、ECSでFargate起動タイプを使用して実行する。EventBridge Schedulerでcron式のスケジュールを定義し、各ジョブを定期的にECSタスクとして起動する。
- B. 各ジョブスクリプトをコンテナ化し、AWS Batchのジョブとして実行する。EventBridge Schedulerでcron式のスケジュールを定義し、各ジョブを定期的に実行する。
- C. 各スクリプトをS3バケットに保存し、Athenaでクエリ可能な形式に変換する。EventBridge Schedulerでcron式のスケジュールを定義し、定期的に起動するLambda関数からAthenaを呼び出して処理を実行する。
- D. 各ジョブスクリプトをLambda関数に変換し、EventBridge Schedulerでcron式のスケジュールを定義して定期実行する。

> [!success]- 答え：**A**
> 既存のBash/Pythonスクリプトをコンテナ化すれば最小改修で移行でき、Fargateならサーバー管理不要で実行時間30分も問題ない。EventBridge（Schedulerによるcron＋ルールによるイベント）から `ECS RunTask` を定期・イベントの両方で起動でき、将来のイベント駆動要件も同一構成で満たせる。
> - A：正解。コンテナ化＝最小改修、Fargate＝サーバーレスで30分実行可、EventBridgeで定期＋イベント駆動を統一できる。
> - B：AWS Batchはキュー投入型で大規模・計算集約・大量並列バッチ向け。数百本の細かい短時間cronジョブには起動レイテンシとキュー管理が過剰で、イベントへの即時実行とも相性が悪い。
> - C：Athenaはs3上のデータにSQLを実行するサービスで、Bash/Pythonスクリプトを実行する仕組みではない。完全な作り直しが必要で「書き換えたくない」要件に反する。
> - D：Lambdaは最大実行時間が15分で、最長30分のジョブを実行できない。Bashはネイティブ非対応でハンドラ形式への移植も必要なため、リフト＆シフトにならない。
> **関連ノート**：[[0904_ECS]] / [[0703_AWS_Batch]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]] / [[0804_EventBridge]]
> **詳しい解説**：[[501_【解説】外部サイト模擬試験#第1問（Containers ｜ バッチ処理のコンテナ移行とイベント駆動）]]

## 第2問（Serverless ｜ 軽量・短時間処理の最小運用構成）
あなたは、ウェブアプリケーションのユーザーデータを処理する新しいバックエンドシステムを設計しています。この処理にはデータの変換と軽量な分析が含まれ、完了には最大で5分かかります。処理は1時間に1回のみ実行され、メモリ要件は最大で512MBです。将来の利用者増加に対応するため、最小の運用労力で高いコスト効率を実現したいと考えています。

この要件を満たす最善の選択はどのサービス構成ですか。

- A. AWS Batchを使用してジョブをスケジュールし、必要な計算環境でジョブを実行する。
- B. Amazon ECSを使用してFargateベースのコンテナを利用し、Amazon EventBridgeを使用して1時間ごとに定期的にFargateタスクをトリガーするルールを設定する。
- C. Amazon EC2インスタンスを起動し、必要なソフトウェアをインストールして、Cronジョブを使用して1時間ごとにジョブを実行する。
- D. AWS Lambdaを使用してジョブをコード化し512MBのメモリで構成し、Amazon EventBridgeを使用して1時間ごとに定期的にLambda関数をトリガーするルールを設定する。

> [!success]- 答え：**D**
> 実行5分・メモリ512MBはLambdaの制約（15分・10GB）の内側で、サーバーもコンテナも持たないため運用労力が最小。実行時間ぶんだけの従量課金でコスト効率が最も高く、利用者増加にも自動スケールで対応できる。EventBridgeルールで毎時トリガーする。
> - A：AWS Batchは大規模・計算集約バッチ向けで、1時間に1回の軽量処理にはキューイングとコンピュート環境管理が過剰。
> - B：技術的には可能だが、コンテナイメージ管理やタスク定義の手間があり、短時間ジョブではLambdaより運用負荷もコストも不利。コンテナが必要な要件もない。
> - C：EC2＋cronはOSのパッチ・スケール・監視を自前で抱える運用負荷最大の構成。5分しか使わないのに常時起動で課金され、コスト効率が最低で自動スケールもない。
> - D：正解。15分・10GB以内に収まる短時間・低頻度・軽量処理はLambdaの独擅場で、最小運用・高コスト効率・自動スケールをすべて満たす。
> **関連ノート**：[[0708_Lambda]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]] / [[0804_EventBridge]]
> **詳しい解説**：[[501_【解説】外部サイト模擬試験#第2問（Serverless ｜ 軽量・短時間処理の最小運用構成）]]

## 第3問（Compute ｜ Web層のマルチAZ高可用化）
ある会社では、オンラインストアのウェブシステムをAWS上で運用しています。このシステムのWebサーバーは単一のEC2インスタンスで稼働しており、商品情報や顧客情報を格納するデータベースはAmazon RDS PostgreSQLデータベースをマルチAZ構成で使用しています。同社は、障害発生時やリクエストの急増時に備えてより可用性の高い構成に変更したいと考えています。

この要件を満たす最も適切なアプローチはどれですか。

- A. EC2インスタンスを複数AZにまたがるAuto Scalingグループに配置する。ALBを作成し、Auto Scalingグループをターゲットグループとして設定する。
- B. AWS Direct Connectを使用して、エンドユーザーからウェブサイトへのアクセス速度と可用性を向上させる。
- C. EC2インスタンスを複数起動し、ALBの配下に配置する。Route 53のフェイルオーバールーティングポリシーを構成し、トラフィックをALBにルーティングする。
- D. EC2インスタンスのAMIを定期的に作成し、障害発生時にはAMIをもとに新たなEC2インスタンスを起動する。

> [!success]- 答え：**A**
> DBは既にRDSマルチAZで冗長化済みなので、論点は単一EC2のWeb層をどう高可用化するか。複数AZにまたがるAuto Scalingグループ＋ALBは、AZ障害やインスタンス故障の自動置換（可用性）と、リクエスト急増時の自動増設（スケール）の両方を同時に満たす。
> - A：正解。マルチAZのAuto Scalingグループ＋ALBで「障害」も「急増」も両方カバーする、Web層高可用化の王道。
> - B：Direct Connectはオンプレ⇄AWSを結ぶ専用線で、インターネット越しの一般エンドユーザーのアクセスとは無関係。可用性にもスケールにも寄与しない論点違い。
> - C：「複数起動」は固定台数で、リクエスト急増に自動対応できない（Auto Scalingがない）。さらにRoute 53フェイルオーバーはアクティブ-スタンバイの別エンドポイント切替用で、単一ALB前段では役割が重複・無意味。
> - D：AMIからの起動は手動・事後のリカバリで、復旧までサービスが停止する。常に1台で冗長性がなく、急増にも無力。バックアップは高可用性ではない。
> **関連ノート**：[[0501_高可用性とスケーラビリティ]] / [[0504_Auto_Scaling_と_Auto_Scaling_Group]] / [[0503_ALB・NLB・GWLBの違い]]
> **詳しい解説**：[[501_【解説】外部サイト模擬試験#第3問（Compute ｜ Web層のマルチAZ高可用化）]]

## 第4問（Application Integration ｜ サーバーレスAPIのコスト最適化）
ある企業は、世界中のユーザーが利用するサーバーレスWebアプリケーションをAWS上で構築しています。バックエンドはAWS Lambdaで実装されており、その前段にAmazon API Gatewayを配置してHTTPエンドポイントを公開する予定です。このAPIはLambdaへのシンプルなプロキシ統合が中心で、高度なリクエスト検証やAPIキーによる利用者ごとの細かな課金管理といった機能は求められていません。一方で、このアプリケーションには突発的に大量のリクエストが集中することがあり、リクエストの急増に対応しつつ、API層のコストを可能な限り抑えたいと考えています。

この要件を最も適切に満たすアプローチはどれですか。

- A. 使用量プランを設定せず、Amazon API Gateway の REST API をデプロイしてLambdaと統合する。
- B. Amazon API Gateway でレスポンスキャッシュを有効にし、TTLを長めに設定してバックエンドへのリクエスト数を抑制する。
- C. 使用量プランとAPIキーを設定したうえで、Amazon API Gateway の HTTP API をデプロイしてLambdaと統合する。
- D. 使用量プランを設定せず、Amazon API Gateway の HTTP API をデプロイしてLambdaと統合する。

> [!success]- 答え：**D**
> HTTP API は REST API より低価格（約70%安）・低レイテンシで、シンプルなLambda連携のサーバーレスAPIに最適。使用量プランはリクエストを上限で絞る（スロットリング・課金管理）ための機能で、「急増に対応したい」という要件とは逆方向。不要な機能を足さず HTTP API をそのまま使う D が、急増対応（自動スケール）とコスト最適化を両立する。
> - A：REST APIは多機能だが HTTP API より割高。今回必要な機能はなく、コスト最適化の観点で HTTP API に劣る。
> - B：キャッシュ＋長いTTLは静的・反復的なレスポンス向け。動的でグローバルなアプリでは古いデータを返すリスクがあり、キャッシュ自体にも追加課金が発生するため本質的なコスト削減にならない。
> - C：使用量プラン／APIキーはリクエストを絞る・利用量を管理する機能で「急増に対応」と相反する。そもそも使用量プランは主に REST API の機能であり、不要な制限を足すぶん最適から遠ざかる。
> - D：正解。シンプルなサーバーレスAPIを HTTP API で安価に公開し、余計な使用量プランを付けないことで、急増時の自動スケールとコスト最適化を同時に満たす。
> **関連ノート**：[[0809_API_Gateway]] / [[0708_Lambda]] / [[0801_アプリケーション統合サービス一覧]]
> **詳しい解説**：[[501_【解説】外部サイト模擬試験#第4問（Application Integration ｜ サーバーレスAPIのコスト最適化）]]

## 第5問（Networking / CDN / LB ｜ オンプレ接続の暗号化とトンネル監視）
あるソリューションアーキテクトは、ある会社の企業ネットワークを同社のVPCに接続し、オンプレミスからAWSリソースへのアクセスを許可する必要があります。このソリューションでは、オンプレミスとVPC間のトラフィックをネットワーク層で暗号化して保護し、必要最小限のトラフィックのみを許可するアクセス制御を実装する必要があります。また、VPNトンネルの状態を継続的に監視し、異常が発生した場合にアラームを発報する機能も求められています。

これらの要件を満たす最適なソリューションはどれですか。

- A. Site-to-Site VPNを設定してVPCに接続し、適切なルーティングを構成します。セキュリティグループおよびネットワークACLを活用して必要なトラフィックのみを許可し、AWS ConfigでVPNトンネルの状態を監視してアラームを設定します。
- B. Site-to-Site VPNを設定してVPCに接続し、適切なルーティングを構成します。セキュリティグループおよびネットワークACLを活用して必要なトラフィックのみを許可し、VPNトンネルの状態をCloudWatchで監視してアラームを設定します。
- C. Client VPNエンドポイントを設定してVPCに接続します。クライアント証明書認証を設定し、認証されたユーザーのみにアクセスを許可します。VPNトンネルの状態をCloudWatchで監視してアラームを設定します。
- D. Transit Gatewayを設定し、Direct Connect Gatewayと接続してオンプレミスからVPCへのルーティングを確立します。セキュリティグループおよびネットワークACLを活用して必要なトラフィックのみを許可し、CloudWatchでTransit Gatewayの接続メトリクスを監視してアラームを設定します。

> [!success]- 答え：**B**
> 要件は4つ（①オンプレ⇄VPCの拠点間接続 ②ネットワーク層での暗号化 ③最小限のトラフィック許可 ④VPNトンネル状態の監視・アラーム）。①②はIPsecで暗号化するSite-to-Site VPN、③はSG/NACL、④はVPNトンネルの`TunnelState`メトリクスを持つCloudWatchで満たす。すべてを同時に満たすのはBだけ。
> - A：接続・暗号化・アクセス制御はOKだが、AWS Configは構成変更の記録・コンプライアンス評価用で、VPNトンネルのUP/DOWNをリアルタイムに監視する用途ではない（④を満たせない）。
> - B：正解。Site-to-Site VPN（IPsec）＋SG/NACL＋CloudWatchの`TunnelState`アラームで全要件を満たす。
> - C：Client VPNは個々のユーザー/デバイスをVPCへつなぐリモートアクセス方式。要件は「企業ネットワーク（拠点）まるごと」の接続なので用途が違う。
> - D：Direct Connectは物理専用線で、それ単体ではトラフィックを暗号化しないため②を満たせない（暗号化が必要ならDX＋VPNを重ねる）。単一VPC接続にTransit Gatewayも過剰。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0408_セキュリティグループとNACL]] / [[1401_マネジメント・ガバナンスサービス一覧]]
> **詳しい解説**：[[501_【解説】外部サイト模擬試験#第5問（Networking / CDN / LB ｜ オンプレ接続の暗号化とトンネル監視）]]

## 第6問（Security / IAM ｜ コンテナイメージの脆弱性スキャン）
ある企業は、社内向けの学習管理アプリケーションをAWS上で開発・運用しています。アプリケーションは Amazon ECS の AWS Fargate 起動タイプで稼働しており、コンテナイメージは Amazon ECR に格納されています。セキュリティチームは、ECR に保存されたコンテナイメージに既知の脆弱性が含まれていないかを、継続的に評価したいと考えています。

これらの要件を最も満たすソリューションはどれですか。

- A. AWS Security Hub を有効化し、ECR に保存されたコンテナイメージをスキャンして脆弱性を検出します。
- B. AWS Systems Manager Patch Manager を使用して、ECR に保存されたコンテナイメージをスキャンし、脆弱性を検出します。
- C. AWS Config のコンフォーマンスパックを使用して、コンテナイメージのセキュリティ脆弱性をスキャンします。
- D. Amazon Inspector を使用して、ECR 内のコンテナイメージをスキャンし、脆弱性を検出します。

> [!success]- 答え：**D**
> Amazon Inspector は **EC2・ECRコンテナイメージ・Lambda** の既知の脆弱性（CVE）を自動・継続スキャンする専用サービス。ECR の拡張スキャンを有効にすると、**イメージのプッシュ時に自動スキャンし、新しいCVEが公表されるたびに既存イメージを再スキャン**するため「継続的に評価」をそのまま満たす。
> - A：Security Hub は各種セキュリティサービスの検出結果を**集約・統制**するダッシュボードで、自分ではイメージをスキャンしない（Inspectorの結果を受け取る側）。
> - B：Patch Manager は稼働中のEC2/オンプレOSに**パッチを適用**する自動化ツールで、ECRイメージをスキャンする機能はない。
> - C：Config コンフォーマンスパックはリソースの**設定がルールに準拠**しているかを評価するもので、イメージ内部のCVEは検査しない。
> - D：正解。ECRイメージ・EC2・Lambda の脆弱性を継続スキャンするのは Inspector。
> **関連ノート**：[[1603_脅威検知・監査]] / [[1601_セキュリティサービス一覧]]
> **詳しい解説**：[[501_【解説】外部サイト模擬試験#第6問（Security / IAM ｜ コンテナイメージの脆弱性スキャン）]]

## 第7問（Migration / Transfer ｜ リージョン間EFSの定期同期）
ある大手製薬会社は、離れた地域に2つの主要な研究施設を持っています。それぞれの施設は機密性の制約により、近隣のAWSリージョンだけにアクセスしています。各施設は毎日大量の研究データを生成し、それを各リージョンのAmazon EFSに保存しています。同社は、これらの保存済みデータを毎日同期させ、どちらの施設からも最新の研究データにアクセスできるようにしたいと考えています。

この要件を満たす最も適切なアプローチはどれですか。

- A. AWS Direct Connectを使用して、2つのリージョン間の専用接続を確立する。
- B. AWS Storage Gatewayを使用して、各施設のEFSをオンプレミス経由で相互に共有する。
- C. AWS Lambdaを使用して、毎日のデータ更新をトリガーとし、自前の同期処理でリージョン間のEFSをコピーする。
- D. AWS DataSyncを使用して、2つのリージョンのEFS間でデータ転送タスクをスケジュール実行し、毎日同期する。

> [!success]- 答え：**D**
> 論点は「AWSストレージ（EFS）同士を、リージョンを跨いで・毎日・自動で同期したい」。DataSyncはEFS/FSx/S3/オンプレNFS・SMB間の転送を自動化するマネージドサービスで、リージョン間のEFS⇄EFS転送に正式対応し、スケジュール実行・差分転送・整合性検証まで備えるため要件にそのまま一致する。
> - A：Direct Connectはオンプレ⇄AWSを結ぶ専用線（回線）を提供するだけで、データ同期処理そのものは行わない。リージョン間EFS同期の手段ではない。
> - B：Storage GatewayはオンプレミスとAWSをつなぐハイブリッドストレージの窓口で、AWS内のEFS同士を同期する役割ではない。
> - C：実装可能だが、差分管理・再試行・整合性検証・スケーリングをすべて自前で作り込む必要があり、マネージドなDataSyncがある以上「最も適切」ではない。
> - D：正解。リージョン間EFSの定期同期をマネージドに実現できる唯一の選択肢。
> **関連ノート**：[[1502_DataSync]] / [[1501_移行と転送サービス一覧]] / [[1503_Storage Gateway]]

## 第8問（Security / IAM ｜ DB認証情報の安全管理とローテーション）
あるアプリケーションが、バックエンドのデータベースへアクセスしています。セキュリティチームは、データベースの認証情報をアプリケーションコードや設定ファイルから切り離して安全に管理し、さらに情報漏洩のリスクを下げるために認証情報を定期的にローテーションしたいと考えています。アプリケーションは実行時に最新の認証情報を取得してデータベースに接続する必要があります。

この要件を最も適切に満たす方法はどれですか。

- A. AWS Secrets Manager に認証情報を保管し、自動ローテーションを有効化したうえで、アプリケーションが実行時に取得して認証する。
- B. AWS CloudHSM に保存された鍵を使用して認証する。
- C. AWS KMS を利用して認証する。
- D. AWS KMS のデータキーを使って認証情報を取得する。

> [!success]- 答え：**A**
> 認証情報（シークレット）そのものの暗号化保管・実行時取得・自動ローテーションを一体で提供するのは Secrets Manager。KMS／CloudHSM は「暗号鍵」を管理するサービスで、DBパスワードの保管・ローテーションを行う基盤ではない。
> - A：正解。コードから分離した安全な保管と、マネージドな自動ローテーションを同時に満たす。
> - B：CloudHSM は暗号鍵を専有HSMで自己管理する用途で、DB認証情報そのものの保管・ローテーションではない。
> - C：KMS は暗号鍵の管理サービス。Secrets Manager の裏側で暗号化を担うが、KMS 単体で認証情報を保管・ローテーションする機能はない。
> - D：データキーはデータ暗号化（エンベロープ暗号化）のためのもので、それで認証情報を取得・ローテーションする仕組みは存在しない。
> **関連ノート**：[[1605_データ保護・暗号化]] / [[1607_AWS_KMS]] / [[1601_セキュリティサービス一覧]]
> **詳しい解説**：[[501_【解説】外部サイト模擬試験#第8問（Security / IAM ｜ DB認証情報の安全管理とローテーション）]]

## 第9問（Security / IAM ｜ 国・地域によるアクセス制限）
あるサービスは、Application Load Balancer（ALB）配下のEC2インスタンスでWebコンテンツを公開しています。コンプライアンス上の理由から、このコンテンツへのアクセスを自国内のクライアントだけに限定し、それ以外の国からのアクセスは拒否したいと考えています。

EC2インスタンスへのアクセスを特定の国に制限する方法はどれですか。（2つ選択）

- A. AWS WAF を ALB に関連付け、Web ACL に地理的一致（Geo match）ルールを設定して許可する国を指定する。
- B. ALB へのアクセスを Amazon CloudFront 経由にし、CloudFront の地理的制限（Geographic Restrictions）で許可する国を設定する。
- C. ALB のセキュリティグループのインバウンドルールで、許可する国を指定する。
- D. AWS Shield を ALB に関連付け、Web ACL に許可する国を設定する。
- E. ALB へのアクセスを Amazon API Gateway 経由にし、API Gateway の地理的制限で許可する国を設定する。

> [!success]- 答え：**A ＋ B**
> 国・地域単位でのアクセス制御は、AWS WAF の地理的一致ルールか、CloudFront の地理的制限のいずれかで実現する。
> - A：正解。WAF は ALB に直接関連付けでき、Web ACL の Geo match ルールで送信元IPの国を判定して許可/拒否できる。
> - B：正解。CloudFront には標準で地理的制限機能があり、前段で対象国以外を遮断できる（CloudFront→ALB→EC2）。
> - C：誤り。セキュリティグループは IP/CIDR 単位の制御で、「国」という単位の指定はできない。
> - D：誤り。Web ACL は WAF の機能であり、Shield（DDoS緩和サービス）に国制限のWeb ACLは設定できない。"Web ACL"の語に引っ掛けるひっかけ。
> - E：誤り。API Gateway に「地理的制限」機能は無く、ALB を API Gateway 経由にする構成も不自然。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[1603_脅威検知・監査]]
> **詳しい解説**：[[501_【解説】外部サイト模擬試験#第9問（Security / IAM ｜ 国・地域によるアクセス制限）]]

## 第10問（Analytics ｜ サーバーレスDWHとSQLによる機械学習）
ある小売業者は、Amazon S3 に保存されているデータを定期的にデータウェアハウスにロードし、販売予測や在庫管理に使用する機械学習モデルを SQL コマンドを使用して作成およびトレーニングしたいと考えています。データウェアハウスには、超並列処理（MPP）による高速なデータ処理と列指向ストレージが必要です。また、運用管理の手間を減らすため、可能な限りサーバーレスな AWS サービスを使用したいと考えています。

これらの要件を満たすソリューションはどれですか。

- A. S3 上のデータに対して Amazon Athena で SQL クエリを実行し、機械学習モデルを作成およびトレーニングする。
- B. Aurora Serverless にデータをロードし、Amazon SageMaker AI を使用して機械学習モデルを作成およびトレーニングする。
- C. Amazon Redshift（プロビジョンド）にデータをロードし、Redshift ML を使用して機械学習モデルを作成・トレーニングする。
- D. Amazon Redshift Serverless にデータをロードし、Redshift ML を使用して機械学習モデルを作成・トレーニングする。

> [!success]- 答え：**D**
> 「データウェアハウス・超並列処理（MPP）・列指向」は Redshift、「SQLで機械学習」は Redshift ML、「可能な限りサーバーレス」は Redshift Serverless。3要件すべてに合致するのは D。
> - A：Athena はサーバーレスでSQL実行できるが、DWH（MPP・列指向）という明示要件に応えるサービスではない。
> - B：Aurora は OLTP 向けRDBで、列指向・MPPのデータウェアハウスではない。
> - C：要件の大半を満たすが、プロビジョンド Redshift はクラスタ管理が必要で「可能な限りサーバーレス」に反する。
> - D：正解。DWH・MPP・列指向・SQLベースのML・サーバーレスをすべて満たす。
> **関連ノート**：[[1113_Amazon Redshift]] / [[1101_AWS分析サービス一覧]] / [[1301_AWS機械学習サービス一覧]]
> **詳しい解説**：[[501_【解説】外部サイト模擬試験#第10問（Analytics ｜ サーバーレスDWHとSQLによる機械学習）]]
