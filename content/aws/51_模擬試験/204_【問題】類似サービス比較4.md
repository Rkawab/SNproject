# 【問題】類似サービス比較4（公式サンプル風・全10問）

## この問題集の目的

役割が似ていて混同しやすいサービスの「使い分け」を要件文で判断する。201〜203で扱っていない比較（専有ホスト・DAX・Transit Gateway・FIFO・Kinesis種別・テープ移行など）を中心に13分野を横断する。

## 対象レベル

SAA。サービス名は知っている前提で、要件に最も合う1つを選ぶ。

## 問題数

10問

---

## 第1問（Compute ｜ Dedicated Hosts vs Dedicated Instances vs 共有）
ソケット単位・物理コア単位でライセンスされる既存ソフトウェアを持ち込み（BYOL）で稼働させたい。物理サーバーを専有し、ソケットやコアの可視性とインスタンス配置の制御が必要である。最も適切なオプションはどれか。

- A. Amazon EC2 Dedicated Hosts
- B. Amazon EC2 Dedicated Instances
- C. デフォルトの共有テナンシー
- D. オンデマンドキャパシティ予約

> [!success]- 答え：**A**
> Dedicated Hostsは物理サーバー全体を専有し、ソケット・コアの可視性とインスタンス配置の制御を提供するため、ソケット/コア課金のBYOLに向く。
> - A：正解。物理ホストの専有と配置制御でBYOLライセンス要件を満たせる。
> - B：一番きわどい誤答。物理ホストは専有するが、**ソケット・コア単位の可視性や同一ホストへの配置制御は提供しない**。「専有」まで同じで「可視性・配置制御」で差が付く。
> - C：他テナントと物理ホストを共有し、専有要件を満たさない。
> - D：容量の確保はできるが、テナンシー（専有）とは別の話。「予約」の言葉に釣られない。
> **関連ノート**：[[0102_EC2購入プラン]]

## 第2問（Database ｜ DAX vs ElastiCache vs リードレプリカ）
DynamoDBを使うアプリで、読み取りをマイクロ秒級まで高速化したい。DynamoDBに特化し、アプリの大幅な書き換えなしにキャッシュを導入したい。最も適切なサービスはどれか。

- A. Amazon DynamoDB Accelerator（DAX）
- B. Amazon ElastiCache for Redis
- C. DynamoDBのグローバルセカンダリインデックス
- D. RDSのリードレプリカ

> [!success]- 答え：**A**
> DAXはDynamoDB専用のインメモリキャッシュで、API互換のためアプリ改修を最小化しつつマイクロ秒級の読み取りを実現する。
> - A：正解。DynamoDBに統合され、最小の変更で読み取りを高速化できる。
> - B：汎用キャッシュだが、キャッシュ制御のロジックを自前実装する必要がある。
> - C：クエリの柔軟性向上が目的で、マイクロ秒級キャッシュの仕組みではない。
> - D：RDS用であり、DynamoDBには適用できない。
> **関連ノート**：[[0303_DynamoDB]] / [[0304_ElastiCache]]

## 第3問（Networking / CDN / LB ｜ Transit Gateway vs VPCピアリング）
数十のVPCとオンプレミスを相互接続したい。VPCが増えても接続が複雑化しないよう、中央のハブで集約してルーティングを管理したい。最も適切なサービスはどれか。

- A. AWS Transit Gateway
- B. VPCピアリングのフルメッシュ
- C. VPCピアリングを中継（推移的ルーティング）させて経由接続する
- D. AWS PrivateLink で各VPCのサービスを相互公開する

> [!success]- 答え：**A**
> Transit Gatewayは多数のVPC・オンプレ接続を中央ハブで集約し、フルメッシュの複雑さを避ける。
> - A：正解。多数のネットワークをスケーラブルに集約・ルーティングできる。
> - B：VPCが増えるほど接続数（N×(N-1)/2）が急増し、管理が複雑になる。
> - C：VPCピアリングは推移的ルーティングに対応しておらず、ピアを経由した中継接続はできない。
> - D：特定サービスへのプライベート接続には有効だが、多数のVPC/オンプレを中央集約してルーティングする用途ではない。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0409_IGW・NAT Gateway・VPCエンドポイント]]

## 第4問（Management / Monitoring ｜ Session Manager vs 踏み台ホスト）
プライベートサブネットのEC2へ、SSHポートを開けず、踏み台ホストや鍵管理なしでシェルアクセスし、操作ログも記録したい。最も適切な方法はどれか。

- A. AWS Systems Manager Session Manager
- B. パブリックサブネットに踏み台（bastion）ホストを立てる
- C. EC2 Instance Connect エンドポイントを使う
- D. AWS Client VPN 経由でSSH接続する

> [!success]- 答え：**A**
> Session ManagerはSSHポートや踏み台なしでセキュアにシェルアクセスでき、操作を記録できる。
> - A：正解。インバウンドポートを開けず、鍵管理も不要で操作ログも記録できる。
> - B：踏み台の運用・パッチ・鍵管理の負荷と攻撃面が増える。
> - C：踏み台なしで到達できるが、SSH（鍵）ベースで、Session Managerのようなコマンド単位の操作ログ記録は標準では行わない。
> - D：ネットワーク到達性は得られるが、各EC2でSSHポートと鍵管理が必要で、ポートを開けず鍵管理も避けたい要件に反する。
> **関連ノート**：[[1405_Systems Manager]] / [[0405_ネットワークとコンテンツ配信]]

## 第5問（Application Integration ｜ SQS FIFO vs 標準キュー）
決済処理のメッセージを、送信した順序を厳密に保ち、かつ重複処理を避けて「正確に1回」処理したい。最も適切なサービスはどれか。

- A. Amazon SQS FIFOキュー
- B. Amazon SQS 標準キュー
- C. Amazon SNS 標準トピック
- D. Amazon Kinesis Data Firehose

> [!success]- 答え：**A**
> SQS FIFOキューは順序保証と重複排除（正確に1回の処理）を提供する。
> - A：正解。順序と重複排除が必要な決済処理に最適。
> - B：高スループットだが順序は保証されず、重複配信があり得る。
> - C：ファンアウト通知で、厳密な順序・重複排除のキューではない。
> - D：S3等への配信用途で、順序付きメッセージキューではない。
> **関連ノート**：[[0802_SQS]]

## 第6問（Containers ｜ EKS vs ECS）
チームは既存のKubernetesマニフェストやHelmチャート、エコシステムのツールをそのまま活用し、マルチクラウドでも通用する標準的なKubernetes APIで運用したい。最も適切なサービスはどれか。

- A. Amazon EKS
- B. Amazon ECS
- C. AWS App Runner
- D. AWS Elastic Beanstalk

> [!success]- 答え：**A**
> EKSはマネージドなKubernetesで、標準のKubernetes APIやHelm・既存マニフェスト、エコシステムのツールをそのまま使える。
> - A：正解。「Kubernetesマニフェスト」「Helm」「標準K8s API」の語が出たらEKS確定。マルチクラウド移植性もK8s標準ゆえ。
> - B：きわどい誤答。同じコンテナオーケストレーターだが**AWS独自API**で、K8sマニフェストやHelmは使えず移植性もない。
> - C：コンテナを最も手軽に動かせるが、オーケストレーションのAPIはAWS独自でK8s資産は活かせない。
> - D：アプリのデプロイ自動化サービスで、Kubernetes APIは提供しない。
> **関連ノート**：[[0906_EKS]] / [[0904_ECS]] / [[0709_Beanstalk・CloudFormation・AppRunnerの違い]]

## 第7問（Analytics ｜ Kinesis Data Streams vs Firehose）
ストリーミングデータを、複数のカスタムコンシューマーがサブ秒単位でリアルタイム処理し、必要に応じて一定期間内のデータを再処理（リプレイ）できるようにしたい。最も適切なサービスはどれか。

- A. Amazon Kinesis Data Streams
- B. Amazon Kinesis Data Firehose
- C. Amazon SQS 標準キュー
- D. Amazon Athena

> [!success]- 答え：**A**
> Kinesis Data Streamsはシャードに保持したデータを複数コンシューマーが低レイテンシで処理でき、保持期間内のリプレイにも対応する。
> - A：正解。カスタムなリアルタイム処理とリプレイの要件に合う。
> - B：S3/Redshift等への配信が主目的で、任意の再処理やサブ秒のカスタム処理には向かない。
> - C：メッセージ取得後は基本的に削除され、ストリームのリプレイには適さない。
> - D：保存済みデータへのクエリエンジンで、ストリーム処理ではない。
> **関連ノート**：[[1103_Amazon Kinesis]] / [[0805_Kinesis]]

## 第8問（Migration / Transfer ｜ Storage Gateway の種類）
オンプレミスの物理テープによるバックアップ運用を、既存のバックアップソフトの仮想テープライブラリ（VTL）インターフェースを保ったままクラウドへ移行したい。最も適切なサービスはどれか。

- A. AWS Storage Gateway（Tape Gateway）
- B. AWS Storage Gateway（File Gateway）
- C. AWS Storage Gateway（Volume Gateway）
- D. AWS Snowball Edge

> [!success]- 答え：**A**
> Tape Gatewayは仮想テープライブラリ（VTL）として振る舞い、既存のバックアップソフトを変えずにテープバックアップをクラウドへ移行できる。
> - A：正解。VTLインターフェースを保ったままクラウドへ移行できる。
> - B：NFS/SMBのファイルアクセス向けで、VTL用途ではない。
> - C：iSCSIブロックボリュームをオンプレへ提供するタイプで、テープ（VTL）インターフェースの移行用途ではない。
> - D：一度きりの大容量オフライン移送向けで、継続的なテープバックアップ運用には合わない。
> **関連ノート**：[[1503_Storage Gateway]] / [[1502_DataSync]] / [[1501_移行と転送サービス一覧]]

## 第9問（Cost Optimization ｜ Savings Plans vs RI vs On-Demand）
インスタンスファミリーやサイズ、リージョンを今後変更する可能性がある定常的なコンピューティング利用に対し、1年または3年のコミットで柔軟に割引を受けたい。最も適切なオプションはどれか。

- A. Compute Savings Plans
- B. スタンダードリザーブドインスタンス
- C. オンデマンドインスタンス
- D. スポットインスタンス

> [!success]- 答え：**A**
> Compute Savings Plansは一定額のコミットで、インスタンスファミリー・サイズ・リージョン・Fargate/Lambdaを横断して柔軟に割引を適用できる。
> - A：正解。構成変更の柔軟性を保ちつつ割引を得たい場合に最適。
> - B：割引率は高いが特定の構成に紐づきやすく、柔軟性で劣る。
> - C：割引のない基準料金。
> - D：中断され得るため、定常的な利用のベースラインには不向き。
> **関連ノート**：[[0102_EC2購入プラン]]

## 第10問（Machine Learning ｜ Transcribe vs Translate vs Polly）
会議やコールセンターの録音音声を、自動的に文字起こし（テキスト化）したい。話者の区別やタイムスタンプも欲しい。最も適切なサービスはどれか。

- A. Amazon Transcribe
- B. Amazon Translate
- C. Amazon Polly
- D. Amazon Comprehend

> [!success]- 答え：**A**
> Transcribeは音声をテキストへ変換（STT）し、話者分離やタイムスタンプにも対応する。
> - A：正解。音声の文字起こし用途に最適。
> - B：テキストの言語翻訳サービス。
> - C：テキストを音声に変換する（TTS）逆方向のサービス。
> - D：テキストの分析（感情・エンティティ等）向けで、音声の文字起こしではない。
> **関連ノート**：[[1301_AWS機械学習サービス一覧]]
