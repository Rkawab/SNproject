# 【問題】類似サービス比較4（ロードバランサー・配信編・全10問）

## この問題集の目的

第2版（2026-07-14全面新作）。ELB 3種・CloudFront・Global Accelerator・エッジ処理・セッション管理の使い分けを要件文で判断する。誤答はすべて実在の構成で、**別の要件なら正解になるもの**だけを置く。

## 第1問（Networking / CDN / LB ｜ L4の超高性能ロードバランシング）
金融取引システムで、毎秒数百万のTCP接続を超低レイテンシで分散したい。また、取引先企業のファイアウォール許可リストに登録するため、**固定IPアドレス**が必要である。最も適切なロードバランサーはどれか。

- A. Network Load Balancer
- B. Application Load Balancer
- C. Gateway Load Balancer
- D. AWS Global Accelerator＋ALB

> [!success]- 答え：**A**
> NLBはL4で動作し、毎秒数百万接続・ミリ秒未満のレイテンシ・AZごとの固定IP（EIP割当可）を提供する。「超高性能TCP × 固定IP」が決め手。
> - A：正解。
> - B：パスやホストで振り分けるHTTP/HTTPSアプリなら正解になる。L7処理の分レイテンシが増え、IPアドレスも変動する。
> - C：セキュリティアプライアンスを透過挿入する別ジャンルの装置。トラフィック分散の主役ではない。
> - D：**複数リージョン**に展開して世界中から固定IPで受けたいなら正解になる。単一リージョン内の要件には過剰で、GA分の料金も上乗せされる。
> **関連ノート**：[[0503_ALB・NLB・GWLBの違い]] / [[0502_Elastic Load Balancing]]

## 第2問（Networking / CDN / LB ｜ L7ルーティング）
1つのエンドポイントで受けて、`/api/*` はECSコンテナへ、`/reports/*` はLambda関数へ、それ以外はEC2グループへ、というように**パスに応じて異なるターゲットへ**振り分けたい。最も適切なサービスはどれか。

- A. Network Load Balancer
- B. Application Load Balancer
- C. Amazon API Gateway
- D. Amazon CloudFront

> [!success]- 答え：**B**
> ALBはパス・ホスト・ヘッダー等のL7情報でルーティングでき、ターゲットにEC2・ECS・Lambda・IPを混在させられる。「パスで振り分け × ターゲット多様」が決め手。
> - A：L4のため、URLパスを見る振り分けはできない。TCPレベルの高性能分散なら正解になる。
> - B：正解。
> - C：一番きわどい誤答。Lambda中心のAPIで認証・スロットリング・APIキー管理まで欲しいなら正解になる。EC2/ECSへの汎用L7負荷分散が主目的ならALBが本筋。
> - D：ビヘイビア（パスパターン）でオリジンを出し分けられるため一見できそうだが、CloudFrontはキャッシュ・配信の層。ターゲットのヘルスチェックや動的な負荷分散は背後のLBの仕事。
> **関連ノート**：[[0503_ALB・NLB・GWLBの違い]] / [[0809_API_Gateway]]

## 第3問（Networking / CDN / LB ｜ 検査アプライアンスの挿入）
セキュリティ要件により、サードパーティ製のIPS/ファイアウォール仮想アプライアンスを、VPCの全トラフィックに**透過的に**通したい。アプライアンス自体も負荷に応じてスケール・冗長化させたい。最も適切なサービスはどれか。

- A. Network Load Balancerの背後にアプライアンスを並べる
- B. Gateway Load Balancer
- C. AWS Network Firewall
- D. 各サブネットのルートテーブルで単一のアプライアンスEC2へ向ける

> [!success]- 答え：**B**
> GWLBはGENEVEカプセル化で全パケットをアプライアンス群へ透過転送し、スケール・ヘルスチェックを面倒みる。「サードパーティ製アプライアンス × 透過挿入」専用の装置。
> - A：アプライアンスがプロキシ型（明示的に宛先になる）なら成立する構成。しかし送信元/宛先を保ったままの**バンプ・イン・ザ・ワイヤ**挿入はGENEVE対応のGWLBでないとできない。
> - B：正解。
> - C：一番きわどい誤答。**AWSマネージド**のFW/IPSでよいなら正解になる。本問は「サードパーティ製品を使う」指定があるため受け皿が必要。
> - D：小規模なら動くが、その1台が単一障害点かつ性能ボトルネックになる。
> **関連ノート**：[[0503_ALB・NLB・GWLBの違い]] / [[1604_ネットワーク防御]]

## 第4問（Networking / CDN / LB ｜ 非HTTPのグローバル高速化）
世界中のプレイヤーが参加する対戦ゲームで、**UDP**ベースの通信を各地から最寄りのAWSエッジで受けてバックボーン経由で複数リージョンへ届けたい。クライアント設定簡素化のため**静的なIPアドレス2つ**で公開し、リージョン障害時は数秒で切り替えたい。最も適切なサービスはどれか。

- A. Amazon CloudFront
- B. AWS Global Accelerator
- C. Route 53 レイテンシーベースルーティング
- D. 各リージョンのNLBにElastic IPを割り当てて公開

> [!success]- 答え：**B**
> Global Acceleratorはエニーキャストのグローバル静的IP2つで受け、AWS網で最寄りリージョンへ運ぶ。TCP/UDP対応・秒オーダーのフェイルオーバー。「UDP × 静的IP × 即時切替」の3点でこれ一択。
> - A：**HTTP/HTTPSのコンテンツ配信・キャッシュ**なら正解になる。UDPの任意プロトコルは扱えず、IPも固定されない。
> - B：正解。
> - C：リージョン振り分け自体はできるが、DNSキャッシュ（TTL）のせいで切替に時間がかかり、IPも増減する。DNSだけで安く済ませたい場合の実在解。
> - D：単一リージョンなら正解になる。複数リージョンではIPがリージョン数×2に増え、切替の仕組みも別途必要。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]]

## 第5問（Networking / CDN / LB ｜ 一時的なアップロード許可）
モバイルアプリから、認証済みユーザーにのみ**S3バケットへ画像を直接アップロード**させたい。許可は数分間だけ有効にし、サーバーを経由させたくない。最も適切な方法はどれか。

- A. CloudFront署名付きURLを発行する
- B. S3署名付きURL（PUT）を発行する
- C. API Gateway＋Lambda経由で画像を受け取りS3へ保存する
- D. アップロード用IAMユーザーのアクセスキーをアプリに埋め込む

> [!success]- 答え：**B**
> S3署名付きURLは、発行者の権限で特定オブジェクトへの期限付き操作（PUT/GET）を許可する。「S3へ直接 × 一時的 × アップロード」が決め手。
> - A：一番きわどい誤答。CloudFront**配信**の会員限定コンテンツ（動画のダウンロード等）なら正解になる。アップロード許可の用途ではない。
> - B：正解。
> - C：実在の構成で、受信時にバリデーションや変換を挟みたいなら正解になる。ただしペイロード上限（API Gatewayは10MB）とLambdaコストがかかり、「サーバーを経由させない」要件に反する。
> - D：長期認証情報をクライアントに配る構成は漏えいした瞬間に終わる。試験でも実務でも常に不正解の選択肢として覚えてよい。
> **関連ノート**：[[0208_S3バケットポリシー]]

## 第6問（Networking / CDN / LB ｜ オリジンの保護）
CloudFront＋S3で静的サイトを配信している。利用者がCloudFrontを迂回してS3バケットへ直接アクセスできないようにしたい。現在推奨される方法はどれか。

- A. オリジンアクセスコントロール（OAC）を設定し、バケットポリシーでCloudFrontからのみ許可する
- B. CloudFrontのIPレンジをバケットポリシーで許可する
- C. S3ブロックパブリックアクセスを有効化するだけ
- D. 利用者にCloudFront署名付きURLを配布する

> [!success]- 答え：**A**
> OACはCloudFrontディストリビューションをプリンシパルとしてバケットポリシーで許可する現行推奨方式（旧OAIの後継）。「CloudFront経由のみ」＝OAC。
> - A：正解。
> - B：実在した旧手法だが、IPレンジは変動し管理が破綻しやすい。他のAWS利用者も同じIP帯から来るため厳密な制限にもならない。
> - C：直アクセスは塞がるが、**CloudFrontにも読ませる許可**が無いので配信自体が壊れる。OACとの併用部品としては必須。
> - D：「特定の会員だけが見られる」ようにするユーザー側の認可の仕組み。オリジン迂回の防止とは層が違う（OACと併用はする）。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0208_S3バケットポリシー]]

## 第7問（Serverless ｜ エッジでの処理の使い分け）
CloudFront配信で、オリジンへ転送する前に**外部の認証APIを呼び出して**トークンを検証し、結果に応じてリクエスト本文を書き換えたい。最も適切なサービスはどれか。

- A. CloudFront Functions
- B. オリジンのEC2でリクエスト受信時に処理する
- C. Lambda@Edge
- D. API Gateway Lambdaオーソライザー

> [!success]- 答え：**C**
> Lambda@Edgeはビューワー/オリジンリクエスト等の4フェーズで動き、**ネットワーク呼び出し・本文アクセス**が可能な本格的なエッジ実行環境。「外部API呼び出し × 本文加工」が決め手。
> - A：一番きわどい誤答。ヘッダー書き換え・URL正規化・簡易リダイレクトのような**サブミリ秒の軽量処理**なら正解で、コストも約1/6。ネットワーク呼び出しと本文アクセスができない。
> - B：エッジで弾けず全リクエストがオリジンまで到達するため、防御にも高速化にもならない。エッジ要件がなければ普通の実装。
> - C：正解。
> - D：API Gatewayの入口で認可するAPI構成なら正解になる。CloudFront配信のエッジ処理という土俵ではない。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0708_Lambda]]

## 第8問（Networking / CDN / LB ｜ セッション管理）
Auto Scaling配下のWebサーバー群で、スケールイン（台数削減）が起きるたびに一部ユーザーのログイン状態が消える。**どのサーバーに振り分けられてもセッションを維持**できる、スケーラブルな解決策はどれか。

- A. ALBのスティッキーセッションを有効化する
- B. ElastiCacheへセッションを外部化する
- C. 各インスタンスのEBSにセッションを保存する
- D. Auto Scalingの終了ポリシーを最古インスタンス優先に変更する

> [!success]- 答え：**B**
> セッションをインスタンスの外（ElastiCache/DynamoDB）へ出せば、どのサーバーが応えても、どのサーバーが消えても状態が残る。ステートレス化の定石。
> - A：定番のひっかけ。改修ゼロで「同じユーザーを同じサーバーへ」固定でき、**一時しのぎ**としては実在解。ただしそのサーバーが消えればセッションも消え、負荷も偏る。根本解決にならない。
> - B：正解。
> - C：EBSはインスタンス付属のストレージなので、状況は何も変わらない。
> - D：どのインスタンスもいずれ終了するため、消えるタイミングが変わるだけ。
> **関連ノート**：[[0505_スティッキーセッション]] / [[0304_ElastiCache]]

## 第9問（Networking / CDN / LB ｜ 障害検知と切替の層）
東京リージョンで稼働するサービスに対し、**リージョン全体の障害**時には大阪リージョンの待機環境へ自動で切り替えたい。切替を担うべき仕組みはどれか。

- A. Route 53ヘルスチェック＋フェイルオーバールーティング
- B. ALBのヘルスチェック
- C. Auto Scalingのヘルスチェック
- D. CloudWatchアラーム＋SNS通知

> [!success]- 答え：**A**
> リージョンをまたぐ切替はDNS層でしかできない。Route 53のヘルスチェックがプライマリの異常を検知し、セカンダリのレコードへ自動で切り替える。
> - A：正解。「リージョン間」が出たらDNS＝Route 53。
> - B：定番のひっかけ。**同一リージョン内**で異常なターゲットを切り離すのがALBの守備範囲。リージョンごと落ちればALBも一緒に落ちる。
> - C：異常なインスタンスを**置き換える**仕組み。切替先を選ぶ機能ではない。
> - D：人へ知らせる仕組みとしては実在解だが、「自動で切り替え」にはならない（通知を受けて手動対応）。
> **関連ノート**：[[0506_ヘルスチェックの種類と検知できる異常]] / [[0404_Route53]]

## 第10問（Networking / CDN / LB ｜ 静的サイトのHTTPS配信）
S3の静的ウェブサイトホスティングで社外向けサイトを公開しているが、セキュリティ監査で**独自ドメインのHTTPS必須**を指摘された。既存のS3バケットを活かした最小変更の対応はどれか。

- A. S3静的ウェブサイトエンドポイントの設定でHTTPSを有効化する
- B. CloudFrontを前段に置き、ACM証明書を割り当てる
- C. AWS Amplifyホスティングへサイトを移行する
- D. EC2＋nginxへサイトを移設しLet's Encryptを設定する

> [!success]- 答え：**B**
> S3のウェブサイトエンドポイントは**HTTPS非対応**。HTTPSと独自ドメイン証明書はCloudFront＋ACM（バージニア北部で発行）で被せるのが定石。
> - A：定番のひっかけ。そのような設定は存在しない。S3ウェブサイトエンドポイントはHTTPのみ。
> - B：正解。キャッシュによる高速化もおまけで付く。
> - C：CI/CD込みでフロントエンドの開発・デプロイを一新するなら正解になる。既存バケットの「最小変更」ではなく、デプロイフローの作り直しになる。
> - D：静的サイトのためにサーバー運用（パッチ・証明書更新・冗長化）を抱え込むのは退化。動的処理が必要になったときの選択肢。
> **関連ノート**：[[0108_EC2とS3のWebサイトの違い]] / [[0405_ネットワークとコンテンツ配信]]
