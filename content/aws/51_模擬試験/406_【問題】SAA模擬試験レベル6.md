# 【問題】SAA模擬試験レベル6（公式サンプル寄せ改訂版・全10問）

## この問題集の目的

旧406とは別問題。SAA-C03公式サンプル問題に寄せて、分析、移行、アプリケーション統合、サーバーレスの判断を横断的に確認する。

> [!info] 出題形式
> - シナリオ段落と設問文を分け、です・ます調で記述。
> - 選択肢はサービス名だけでなく、具体的な操作・構成で記述。
> - 複数選択問題は5択（A〜E）、単一選択問題は4択（A〜D）。

## 第1問（Analytics ｜ ストリームの再処理と複数コンシューマー）
ある決済サービスは、取引イベントをリアルタイムに処理しています。不正検知チーム、集計チーム、監査チームが同じイベントをそれぞれ独立して読み取り、必要に応じて過去数時間分のイベントを再処理できる必要があります。イベントは取引IDごとの順序を保つ必要があります。

これらの要件を満たす最も適切なソリューションはどれですか。

- A. Amazon Kinesis Data Firehoseにイベントを送信し、各チームの処理先へ直接配信する。
- B. Amazon SQS標準キューにイベントを送信し、各チームのコンシューマーが同じキューからメッセージを取得する。
- C. Amazon Kinesis Data Streamsにイベントを送信し、取引IDをパーティションキーにして、各チームが独立したコンシューマーとして読み取る。
- D. Amazon EventBridgeイベントバスにイベントを送信し、すべてのイベントを24時間後に再配信するルールを作成する。

> [!success]- 答え：**C**
> Kinesis Data Streamsは保持期間内の再読み取り、複数コンシューマー、パーティションキー単位の順序性に対応する。
> - A：Firehoseは配信先へのロードに向くが、複数チームが独立して再処理するストリーム用途ではData Streamsが合う。
> - B：SQS標準キューでは複数コンシューマーが同じメッセージを独立して再読する用途に向かず、順序性も限定的。
> - C：正解。複数コンシューマーと再処理、パーティションキー単位の順序性を満たす。
> - D：イベントルーティングには有効だが、ストリーム保持と任意時点からの再処理の主用途ではない。
> **関連ノート**：[[1103_Amazon Kinesis]] / [[1102_データ分析の超基礎・用語集]]

## 第2問（Analytics ｜ S3データレイクへの配信とSQL分析）
あるIoT企業は、数百万台のデバイスから送信されるセンサーイベントを継続的に受信しています。企業は、サーバーを管理せずにイベントをAmazon S3へ保存し、列指向形式へ変換したうえで、データサイエンティストが必要なときにSQLで分析できるようにしたいと考えています。分析基盤の常時クラスター運用は避けたい方針です。

要件を満たすアクションの組み合わせはどれですか。（2つ選択）

- A. Amazon Kinesis Data Firehoseを使用し、必要に応じてデータ形式変換を行いながらAmazon S3へ配信する。
- B. Amazon Kinesis Data Streamsを使用し、EC2上の常駐コンシューマーでS3へ書き込む。
- C. Amazon MSKクラスターを作成し、Kafka Connectを自己管理してS3へ配信する。
- D. AWS Glue Data CatalogでS3データのスキーマを管理し、Amazon AthenaでSQLクエリを実行する。
- E. Amazon OpenSearch Serviceを唯一の保存先にし、S3にはイベントを保存しない。

> [!success]- 答え：**A, D**
> FirehoseはストリームをS3へマネージドに配信でき、AthenaはS3上のデータをサーバーレスにSQL分析できる。
> - A：正解。サーバー管理なしでS3への配信と形式変換を実現しやすい。
> - B：可能だが、コンシューマー運用が必要で、運用負荷を避ける要件に反する。
> - C：Kafkaが必要な要件ではなく、クラスターとConnectorの運用負荷が増える。
> - D：正解。S3データレイクに対して必要時だけSQL分析できる。
> - E：検索用途には有効な場合があるが、S3データレイクとサーバーレスSQL分析の要件を満たさない。
> **関連ノート**：[[1103_Amazon Kinesis]] / [[1109_Amazon Athena]]

## 第3問（Migration / Transfer ｜ 異種データベースの最小停止移行）
ある企業は、オンプレミスのOracleデータベースをAmazon Aurora PostgreSQL互換エディションへ移行しようとしています。データベースエンジンが異なるため、スキーマや一部のSQLコードを変換する必要があります。アプリケーション停止時間は、最終切り替え時の短時間に抑える必要があります。

これらの要件を満たすために使用すべきサービスと機能はどれですか。（2つ選択）

- A. Oracle Data PumpでエクスポートしたダンプをAmazon S3へアップロードし、カットオーバー時にAuroraへ一括ロードする。
- B. AWS Schema Conversion Tool（SCT）を使用して、OracleのスキーマとコードをPostgreSQL向けに変換する。
- C. AWS BackupでOracleデータベースのバックアップを取得し、Aurora PostgreSQLへ直接復元する。
- D. Amazon RDSリードレプリカを作成し、OracleからAurora PostgreSQLへ昇格する。
- E. AWS Database Migration Service（DMS）のフルロードと継続的レプリケーション（CDC）を使用する。

> [!success]- 答え：**B, E**
> 異種エンジン移行ではSCTでスキーマ変換を行い、DMSのCDCで移行中の変更を追従して停止時間を短縮する。
> - A：一括ロードでは移行中の変更を追従できず、停止時間が長くなりやすい。
> - B：正解。異なるDBエンジン間のスキーマ・コード変換に使う。
> - C：AWS Backupは異種DBエンジンへの直接変換復元サービスではない。
> - D：RDSリードレプリカは異種エンジン変換と昇格の仕組みではない。
> - E：正解。初期ロード後もCDCで同期し、短い停止時間で切り替えられる。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[0307_Aurora詳細]]

## 第4問（Migration / Transfer ｜ オンプレミスのファイル共有をS3へ拡張）
ある製造業の工場では、オンプレミスのWindowsアプリケーションがSMBファイル共有へ図面ファイルを書き込んでいます。会社は、ファイルをAmazon S3に保存して耐久性を高めたい一方、工場内のアプリケーションからは引き続きSMB共有としてアクセスしたいと考えています。頻繁に使うファイルはローカルにキャッシュし、WAN越しのレイテンシを抑える必要があります。

最も適切なソリューションはどれですか。

- A. AWS Transfer FamilyのSFTPエンドポイントを作成し、WindowsアプリケーションをSFTPクライアントへ変更する。
- B. Amazon EFSファイルシステムを作成し、オンプレミスからインターネット経由でNFSマウントする。
- C. AWS DataSyncタスクを1時間ごとに実行し、オンプレミス共有からS3へ一方向同期する。
- D. AWS Storage GatewayのFile Gatewayをオンプレミスに配置し、SMB共有として公開して、バックエンドにS3バケットを使用する。

> [!success]- 答え：**D**
> File GatewayはオンプレミスにNFS/SMB共有を提供し、バックエンドとしてS3を使いながらローカルキャッシュも利用できる。
> - A：外部転送プロトコルとしては有効だが、既存アプリのSMB共有アクセスを維持する要件に合わない。
> - B：EFSはAWS内や接続された環境から使えるが、S3バックエンドのSMB共有とローカルキャッシュ要件ではFile Gatewayが適切。
> - C：同期には有効だが、アプリケーションがリアルタイムにSMB共有として使う要件を満たさない。
> - D：正解。既存のSMBアクセスを保ちながらS3へ保存し、ローカルキャッシュでレイテンシを抑えられる。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[1502_DataSync]]

## 第5問（Migration / Transfer ｜ オンプレミスファイルの定期差分同期）
ある研究機関は、オンプレミスのNFSファイルサーバーに保存された解析結果を毎晩Amazon S3へ同期しています。ファイル数は多く、変更差分だけを効率よく転送し、転送後の検証も自動化したいと考えています。外部ユーザーへSFTPを提供する要件はありません。

最も適切なソリューションはどれですか。

- A. AWS Transfer FamilyでSFTPサーバーを作成し、研究機関内のバッチ処理からSFTPでS3へアップロードする。
- B. オンプレミスにAWS DataSyncエージェントを配置し、NFSからS3へのスケジュール同期タスクを作成する。
- C. AWS Snowball Edgeを毎晩発送し、差分ファイルを物理搬送する。
- D. Amazon S3 Transfer Accelerationを有効化し、独自スクリプトで全ファイルを毎晩再アップロードする。

> [!success]- 答え：**B**
> DataSyncはオンプレミスのNFS/SMBなどとAWSストレージ間の大規模データ転送を、スケジュール、差分検出、検証付きで実行できる。
> - A：SFTP提供には有効だが、内部NFSサーバーの差分同期と検証にはDataSyncが適切。
> - B：正解。変更差分の効率的な同期と検証をマネージドに扱える。
> - C：初回の超大容量移行には有効だが、毎晩の差分同期には現実的でない。
> - D：転送高速化の可能性はあるが、差分検出や検証を自前実装する必要がある。
> **関連ノート**：[[1502_DataSync]] / [[1501_移行と転送サービス一覧]]

## 第6問（Application Integration ｜ 複数ステップ処理の可視化とリトライ）
ある旅行予約サービスは、予約リクエストを受け付けた後、在庫確認、支払い、予約確定メール送信という複数ステップを順番に実行しています。各ステップは失敗する可能性があり、失敗時にはステップごとにリトライや補償処理を行いたいと考えています。サーバーの運用は避け、ワークフローの実行状態を確認できる必要があります。

要件を満たすアクションの組み合わせはどれですか。（2つ選択）

- A. AWS Step Functionsでステートマシンを定義し、各ステップのリトライ、分岐、補償処理を管理する。
- B. Amazon SQS標準キューを1つだけ作成し、すべてのステップを同じメッセージ本文のフラグで管理する。
- C. 各処理ステップをAWS Lambda関数またはコンテナタスクとして実装し、Step Functionsから呼び出す。
- D. Amazon EventBridgeのスケジュールルールで、5分ごとに未処理予約をまとめて処理する。
- E. 予約処理を1つの巨大なLambda関数にまとめ、失敗時は最初からすべて再実行する。

> [!success]- 答え：**A, C**
> Step Functionsは複数ステップの順序、分岐、リトライ、状態可視化に適しており、LambdaやECSタスクなどを各ステップとして呼び出せる。
> - A：正解。ワークフロー制御と実行状態の可視化を満たす。
> - B：キューは疎結合には有効だが、複雑な状態遷移や補償処理の可視化はStep Functionsが適切。
> - C：正解。処理単位を分け、Step Functionsでオーケストレーションできる。
> - D：定期実行ではリアルタイムな予約フローとステップごとの状態管理が弱い。
> - E：失敗箇所の特定や部分リトライが難しく、要件に反する。
> **関連ノート**：[[0807_Step_Functions]] / [[0708_Lambda]]

## 第7問（Application Integration ｜ SaaSイベントのルーティング）
ある企業は、複数のAWSアカウントで運用するマイクロサービスへイベントを配信しています。今後、外部SaaSから発生する契約更新イベントも受け取り、イベントの種類に応じてLambda関数、SQSキュー、Step Functionsへルーティングしたいと考えています。イベントの発行元と処理先を疎結合にし、ルールベースで配信先を増やせる必要があります。

最も適切なソリューションはどれですか。

- A. Amazon SNSトピックを1つ作成し、すべてのイベントを全サブスクライバーへ同じ形式でファンアウトする。
- B. Amazon SQSキューを1つ作成し、すべての処理サービスが同じキューからメッセージを競合して取得する。
- C. AWS CloudTrailログをS3へ配信し、S3イベント通知で各サービスを起動する。
- D. Amazon EventBridgeイベントバスを使用し、イベントパターンに基づくルールで各ターゲットへルーティングする。

> [!success]- 答え：**D**
> EventBridgeはAWSサービス、カスタムアプリ、SaaSイベントをイベントバスで受け取り、ルールに基づいて多様なターゲットへルーティングできる。
> - A：単純なファンアウトには有効だが、SaaS連携やイベントパターンによる柔軟なルーティングではEventBridgeが自然。
> - B：単一キューでは処理先が競合し、イベント種類ごとのルーティングに向かない。
> - C：監査ログ起点の処理であり、アプリケーションイベントルーティングの標準構成ではない。
> - D：正解。発行元と処理先を疎結合にし、ルールで配信先を増やせる。
> **関連ノート**：[[0806_EventBridge]]

## 第8問（Serverless ｜ スパイクするモバイルAPI）
ある小売企業は、モバイルアプリ向けのREST APIを新規構築しています。キャンペーン開始直後にリクエストが急増しますが、通常時はトラフィックが少ない見込みです。ユーザープロファイルはユーザーIDで参照され、サーバー管理を避け、利用量に応じた課金にしたいと考えています。

最も適切な構成はどれですか。

- A. Application Load Balancerの背後にEC2 Auto Scalingグループを配置し、Amazon RDS for MySQLを最小構成で使用する。
- B. Amazon API Gateway、AWS Lambda、Amazon DynamoDBオンデマンドキャパシティを組み合わせる。
- C. Amazon ECS on EC2でAPIコンテナを実行し、DynamoDBをプロビジョンドキャパシティのピーク値で固定する。
- D. Amazon CloudFrontからS3静的Webサイトへ直接APIリクエストを書き込む。

> [!success]- 答え：**B**
> API Gateway、Lambda、DynamoDBオンデマンドは、サーバー管理なしで急なスパイクに追従し、リクエストや使用量に応じた課金にできる。
> - A：有効なWeb構成だが、サーバー管理やDB容量管理が残り、利用量課金の要件に弱い。
> - B：正解。キーアクセスのプロファイル管理とスパイクに適したサーバーレス構成。
> - C：EC2ノード管理とピーク固定のキャパシティが残り、通常時のコスト効率が悪い。
> - D：S3静的サイトはAPIの動的処理先ではない。
> **関連ノート**：[[0809_API_Gateway]] / [[0708_Lambda]] / [[0303_DynamoDB]]

## 第9問（Analytics ｜ 検索とログ分析の専用基盤）
あるSaaS企業は、アプリケーションログをほぼリアルタイムに検索し、エラー文字列、ユーザーID、リクエストIDで絞り込み、ダッシュボードで傾向を可視化したいと考えています。ログ量は大きく、全文検索や集計を継続的に行う専用の検索基盤が必要です。

最も適切なソリューションはどれですか。

- A. Amazon RDS for MySQLにログを保存し、LIKE検索でエラー文字列を検索する。
- B. Amazon S3にログを保存し、年に数回だけAmazon Athenaで監査クエリを実行する。
- C. Amazon OpenSearch Serviceにログを取り込み、検索、集計、ダッシュボードを構成する。
- D. Amazon SQSにログを保存し、必要なときにキューからメッセージを読み戻す。

> [!success]- 答え：**C**
> OpenSearch Serviceは大規模ログの全文検索、集計、可視化に適したマネージド検索・分析サービス。
> - A：リレーショナルDBのLIKE検索は大規模ログの全文検索・分析基盤としては不向き。
> - B：低頻度のアドホック分析には有効だが、ほぼリアルタイム検索とダッシュボード用途ではOpenSearchが合う。
> - C：正解。検索、集計、可視化の継続運用に向いている。
> - D：SQSはメッセージキューであり、ログの検索・分析ストアではない。
> **関連ノート**：[[1102_データ分析の超基礎・用語集]]

## 第10問（Analytics ｜ ETLと列指向データの作成）
ある企業は、複数システムから出力されるCSVとJSONファイルをAmazon S3に集約しています。分析チームは、データを定期的にクレンジングし、列指向形式に変換し、日付でパーティション分割して、AthenaやRedshift Spectrumから効率よく参照したいと考えています。サーバー管理は避けたい方針です。

最も適切なソリューションはどれですか。

- A. AWS GlueのクローラーとETLジョブを使用し、S3上のデータをカタログ化してParquet形式へ変換し、パーティションを作成する。
- B. Amazon EMRクラスターを常時稼働させ、手動でSparkジョブを実行する。
- C. AWS DMSを使用し、CSVファイルをAmazon RDSへ継続的にレプリケーションする。
- D. Amazon Kinesis Data Streamsを使用し、既存のS3ファイルをシャードへ分割して保存する。

> [!success]- 答え：**A**
> Glueはサーバーレスなデータ統合サービスとして、S3データのカタログ化、ETL、Parquet変換、パーティション管理に利用できる。
> - A：正解。AthenaやRedshift Spectrumで効率よく読むためのデータ整形に合う。
> - B：Spark処理は可能だが、常時クラスター運用が必要になり要件に反する。
> - C：DMSはDB移行・レプリケーション向けで、S3ファイルのETL変換の主役ではない。
> - D：Data Streamsはリアルタイムストリーム向けで、既存S3ファイルのETL変換には向かない。
> **関連ノート**：[[1102_データ分析の超基礎・用語集]] / [[1109_Amazon Athena]]
