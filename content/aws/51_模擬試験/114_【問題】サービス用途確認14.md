## 第1問（Compute ｜ ゴールデンイメージの自動作成）
企業は、最新パッチと社内セキュリティ設定を適用したEC2用AMIを定期的に作成し、テスト後に複数アカウントへ配布する処理を自動化したいと考えています。
最も適したサービスはどれですか。

- A. EC2 Image Builder
- B. AWS Systems Manager Patch Manager
- C. AWS CloudFormation
- D. AWS CodeDeploy

> [!success]- 答え：**A**
> EC2 Image Builderは、VMイメージやコンテナイメージの構築・テスト・配布パイプラインを自動化します。
> - A：正解。安全で更新済みのゴールデンイメージを繰り返し作成できる。
> - B：稼働中ノードへのパッチ適用を管理する機能で、AMI作成パイプラインそのものではない。
> - C：AWSリソースをテンプレートから構築するIaCサービス。
> - D：アプリケーションコードを実行環境へデプロイするサービス。
> **着眼点（SAA）**：「AMIを定期作成・テスト・配布」ならEC2 Image Builder。
> **関連ノート**：[[0701_コンピューティングサービス一覧]] / [[1405_Systems Manager]]
> **詳しい解説**：[[114_【解説】サービス用途確認14#第1問（Compute ｜ ゴールデンイメージの自動作成）]]

## 第2問（Compute ｜ オンプレミスでAWS基盤を実行）
データ所在地と極低遅延の要件により、一部のワークロードを自社データセンター内で実行する必要があります。AWSと同じAPI・運用モデルを使い、AWSが管理するハードウェアをオンプレミスに設置したいと考えています。
最も適したサービスはどれですか。

- A. AWS Outposts
- B. AWS Local Zones
- C. AWS Wavelength
- D. AWS Direct Connect

> [!success]- 答え：**A**
> AWS Outpostsは、AWSのインフラストラクチャと一部サービスを顧客のオンプレミス拠点へ拡張します。
> - A：正解。AWSが管理するハードウェアを自社施設に設置して一貫したハイブリッド運用を実現する。
> - B：AWSが都市圏近くに設置するインフラであり、顧客データセンター内ではない。
> - C：通信事業者の5Gネットワークエッジ向け。
> - D：オンプレミスとAWSリージョンを結ぶ専用線であり、計算基盤を設置しない。
> **着眼点（SAA）**：「自社施設内にAWS管理ハードウェア」ならOutposts。
> **関連ノート**：[[0705_AWS_Outposts]]
> **詳しい解説**：[[114_【解説】サービス用途確認14#第2問（Compute ｜ オンプレミスでAWS基盤を実行）]]

## 第3問（Compute ｜ 大都市圏での低レイテンシー）
動画制作会社は、AWSリージョンから遠い大都市圏の利用者に対し、1桁ミリ秒台の低レイテンシーが必要なアプリケーションをEC2で実行したいと考えています。設備を自社施設に置く必要はありません。
最も適した選択肢はどれですか。

- A. AWS Local Zones
- B. AWS Outposts
- C. AWS Wavelength
- D. Amazon CloudFront

> [!success]- 答え：**A**
> AWS Local Zonesは、コンピューティングやストレージなどを大都市圏の利用者に近い場所で実行できるAWSインフラです。
> - A：正解。リージョンから離れた都市のレイテンシー重視ワークロードに適する。
> - B：顧客のオンプレミス拠点へ設置する。
> - C：5Gモバイル端末向けに通信事業者ネットワーク内で実行する。
> - D：コンテンツ配信やエッジ処理には適するが、一般的なEC2ワークロードの配置先ではない。
> **着眼点（SAA）**：「特定都市の近く」「EC2を配置」「5Gの指定なし」ならLocal Zones。
> **関連ノート**：[[0701_コンピューティングサービス一覧]]
> **詳しい解説**：[[114_【解説】サービス用途確認14#第3問（Compute ｜ 大都市圏での低レイテンシー）]]

## 第4問（Compute ｜ 5G端末向け超低遅延）
通信事業者の5Gネットワークに接続された車両やモバイル端末向けに、ネットワークの端にEC2ワークロードを配置して超低レイテンシーを実現したいと考えています。
最も適したサービスはどれですか。

- A. AWS Wavelength
- B. AWS Local Zones
- C. AWS Outposts
- D. AWS Global Accelerator

> [!success]- 答え：**A**
> AWS Wavelengthは、対応する通信事業者の5Gネットワーク内にワークロードを配置します。
> - A：正解。5G端末とアプリケーション間のネットワーク遅延を最小化する。
> - B：都市圏に近いAWSインフラだが、5G通信事業者ネットワーク内という要件にはWavelengthが合う。
> - C：顧客施設内のAWSインフラ。
> - D：AWSグローバルネットワーク経由でTCP/UDP通信を高速化するが、5GエッジへEC2を配置しない。
> **着眼点（SAA）**：「5G」「通信事業者のエッジ」ならWavelength。
> **関連ノート**：[[0706_AWS_Wavelength]]
> **詳しい解説**：[[114_【解説】サービス用途確認14#第4問（Compute ｜ 5G端末向け超低遅延）]]

## 第5問（Analytics ｜ ストリームのリアルタイム処理）
Kinesis Data Streamsに到着するクリックイベントを、ウィンドウ集計や状態を持つ処理によってリアルタイム分析したいと考えています。Apache Flinkをマネージドで実行する必要があります。
最も適したサービスはどれですか。

- A. Amazon Managed Service for Apache Flink
- B. Amazon Data Firehose
- C. Amazon Athena
- D. AWS Glue Data Catalog

> [!success]- 答え：**A**
> Amazon Managed Service for Apache Flinkは、Apache Flinkによる状態を持つストリーム処理をマネージドで実行します。
> - A：正解。連続的な集計、変換、リアルタイム分析に適する。
> - B：ストリームをS3などの宛先へ配信するサービスで、複雑な状態付き処理が主目的ではない。
> - C：主にS3上の保存済みデータへSQLクエリを実行する。
> - D：メタデータを管理するカタログ。
> **着眼点（SAA）**：「Apache Flink」「状態を持つリアルタイム処理」ならManaged Service for Apache Flink。
> **関連ノート**：[[1101_AWS分析サービス一覧]] / [[1103_Amazon Kinesis]]

## 第6問（Analytics ｜ クラスター管理不要のビッグデータ処理）
データ分析チームは、Apache SparkやApache Hiveを使うジョブを実行したいと考えています。EMRクラスターのプロビジョニングや容量管理は行わず、ジョブ実行時に使用したリソース分だけ支払う必要があります。
最も適したサービスはどれですか。

- A. Amazon EMR Serverless
- B. Amazon EMR on EC2
- C. AWS Lambda
- D. Amazon Redshift provisioned cluster

> [!success]- 答え：**A**
> EMR Serverlessは、クラスターを構築・管理せずにSparkやHiveなどの分析フレームワークを実行できます。
> - A：正解。ワーカー容量を自動管理し、ジョブで消費したリソースに基づいて課金される。
> - B：EC2ベースのクラスター構成と容量を利用者が選択・管理する。
> - C：短時間のイベント処理向けで、一般的な大規模Sparkジョブの実行基盤ではない。
> - D：SQL分析向けDWHであり、Spark/Hiveジョブ実行サービスではない。
> **着眼点（SAA）**：「Spark/Hive」「クラスター管理不要」ならEMR Serverless。
> **関連ノート**：[[1108_Amazon EMR]]
> **詳しい解説**：[[114_【解説】サービス用途確認14#第6問（Analytics ｜ クラスター管理不要のビッグデータ処理）]]

## 第7問（Analytics ｜ データレイクの権限管理）
企業はS3上のデータレイクをAthena、Redshift、EMRから利用します。テーブル、列、行などのきめ細かなアクセス許可を中央管理し、データカタログと監査を統合したいと考えています。
最も適したサービスはどれですか。

- A. AWS Lake Formation
- B. AWS Glue ETL
- C. Amazon S3 ACL
- D. Amazon QuickSight

> [!success]- 答え：**A**
> AWS Lake Formationは、S3データレイクの構築ときめ細かなデータアクセス制御を一元化します。
> - A：正解。Glue Data Catalogを利用し、複数分析サービスにまたがる権限を管理できる。
> - B：データの抽出・変換・ロード処理が中心。
> - C：S3オブジェクト単位の旧来のアクセス制御で、表や列の権限管理には向かない。
> - D：BIダッシュボードを作成・共有するサービス。
> **着眼点（SAA）**：「データレイク」「表・列・行レベル権限」ならLake Formation。
> **関連ノート**：[[1106_AWS Lake Formation]] / [[1107_AWS Glue]]
> **詳しい解説**：[[114_【解説】サービス用途確認14#第7問（Analytics ｜ データレイクの権限管理）]]

## 第8問（Migration / Transfer ｜ オンプレミス環境の移行調査）
大規模なデータセンター移行を計画しています。移行前に、オンプレミスサーバーの構成、使用率、稼働状況、サーバー間依存関係を収集して移行計画に利用したいと考えています。
最も適したサービスはどれですか。

- A. AWS Application Discovery Service
- B. AWS Application Migration Service
- C. AWS Database Migration Service
- D. AWS DataSync

> [!success]- 答え：**A**
> Application Discovery Serviceは、オンプレミスのサーバー構成・利用状況・依存関係を収集して移行計画を支援します。
> - A：正解。移行前の発見・評価フェーズで使う。
> - B：サーバーをAWSへリホストする実行フェーズのサービス。
> - C：データベースの移行と継続レプリケーション用。
> - D：ファイルやオブジェクトデータのオンライン転送用。
> **着眼点（SAA）**：「移行前」「棚卸し」「依存関係の把握」ならApplication Discovery Service。
> **関連ノート**：[[1501_移行と転送サービス一覧]]

## 第9問（Migration / Transfer ｜ 移行状況の一元追跡）
複数のAWS移行ツールとパートナーツールを使って、多数のアプリケーションを段階的に移行します。各アプリケーションの移行状況を1つの画面で追跡したいと考えています。
最も適したサービスはどれですか。

- A. AWS Migration Hub
- B. AWS Control Tower

> - A：正解。移行ポートフォリオ全体の進捗を一元表示する。
> - B：マルチアカウントのランディングゾーンを構築・統制する。
> - C：AWSリソースとノードの運用管理サービス。
> - D：承認済みの製品をカタログとして提供する。
> **着眼点（SAA）**：「複数ツールの移行進捗を一元追跡」ならMigration Hub。
> **関連ノート**：[[1501_移行と転送サービス一覧]]

## 第10問（Cost Optimization ｜ 異常なコストの検知）
通常の利用傾向を機械学習で学習し、予想外のコスト増加をサービスやアカウントなどの単位で自動検知して担当者へ通知したいと考えています。
最も適した機能はどれですか。

- A. AWS Cost Anomaly Detection
- B. AWS Budgets
- C. AWS Compute Optimizer
- D. AWS Cost Explorer

> [!success]- 答え：**A**
> Cost Anomaly Detectionは、過去の支出パターンを基に異常なコストを検出し、通知します。
> - A：正解。固定しきい値だけでは見つけにくい異常な支出パターンを監視する。
> - B：設定した予算額や使用量のしきい値に基づいて通知する。
> - C：リソースの利用状況からライトサイジングを推奨する。
> - D：コストと使用量を可視化・分析する。
> **着眼点（SAA）**：「通常傾向から外れた支出を自動検知」ならCost Anomaly Detection。
> **関連ノート**：[[1701_コスト管理サービス一覧]]
