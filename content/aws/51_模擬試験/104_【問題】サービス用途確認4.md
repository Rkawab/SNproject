# 【問題】サービス用途確認4（公式サンプル風・全10問）

## この問題集の目的

要件文から主要サービスの役割を選ぶドリル。101〜103で扱っていない切り口（デプロイ自動化・Windows共有・IaC・BI・予算管理など）を中心に、13分野を横断する。

## 対象レベル

SAA入口。単語暗記ではなく、要件からサービスを選ぶ練習にする。

## 問題数

10問

---

## 第1問（Compute ｜ デプロイの自動化）
開発者は、アプリケーションコードをアップロードするだけで、プロビジョニング・ロードバランシング・オートスケール・監視までを自動で構成してWebアプリを実行したい。基盤の細部は管理したくないが、必要に応じて配下のEC2にもアクセスできる状態を保ちたい。最も適切なサービスはどれか。

- A. AWS Elastic Beanstalk
- B. AWS Lambda
- C. AWS App Runner
- D. AWS CloudFormation

> [!success]- 答え：**A**
> Elastic Beanstalkはコードを渡すだけで実行環境一式を自動構成し、配下のリソースにもアクセスできる。
> - A：正解。インフラ管理を抑えつつ、Webアプリを素早くデプロイできる。配下のEC2にも触れる。
> - B：イベント駆動の関数実行向けで、常時稼働Webアプリ一式の自動構成とは異なる。
> - C：コンテナ/Webアプリをフルマネージドで実行できるが、基盤が完全に抽象化され配下のEC2には触れられない。
> - D：リソースを宣言的に定義する基盤で、「コードを上げるだけで自動構成」ではなく利用者が構成を記述する必要がある。
> **関連ノート**：[[0708_Lambda]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]] / [[0701_コンピューティングサービス一覧]]

## 第2問（Storage ｜ Windows向け共有ファイル）
Windowsベースのアプリケーション群が、SMBプロトコルとNTFSの権限、Active Directory統合を必要とする共有ファイルストレージを求めている。最も適切なサービスはどれか。

- A. Amazon FSx for Windows File Server
- B. Amazon EFS
- C. Amazon S3
- D. Amazon EBS

> [!success]- 答え：**A**
> FSx for Windows File ServerはSMB・NTFS・Active Directory統合に対応したフルマネージドWindowsファイル共有。
> - A：正解。Windowsアプリの共有ファイル要件にそのまま合致する。
> - B：Linux向けのNFS共有で、SMB/NTFS要件には合わない。
> - C：オブジェクトストレージで、ファイル共有プロトコルを直接は提供しない。
> - D：単一EC2向けブロックストレージで、共有ファイル用途ではない。
> **関連ノート**：[[0202_S3とEBS・EFSの違い]]

## 第3問（Database ｜ 断続的・予測困難な負荷）
開発・検証用のデータベースは、利用されない時間帯が長く、負荷も予測しづらい。アイドル時のコストを抑えつつ、必要なときに自動でキャパシティをスケールさせたい。最も適切なサービスはどれか。

- A. Amazon Aurora Serverless
- B. Amazon RDS（常時起動のプロビジョンドインスタンス）
- C. Amazon Redshift
- D. Amazon DynamoDB（プロビジョンドキャパシティ固定）

> [!success]- 答え：**A**
> Aurora Serverlessは負荷に応じて自動的に容量を増減し、アイドル時のコストを抑えられる。
> - A：正解。断続的・予測困難なワークロードのコスト最適化に向く。
> - B：常時起動のため、利用がない時間帯もコストが発生する。
> - C：分析用データウェアハウスで、用途が異なる。
> - D：固定キャパシティでは断続的な負荷に対するコスト最適化が難しい。
> **関連ノート**：[[0301_RDS]] / [[0307_Aurora詳細]]

## 第4問（Networking / CDN / LB ｜ DNSフェイルオーバー）
プライマリサイトが障害になった場合に、ヘルスチェックの結果に基づいて自動的にスタンバイサイトへDNSレベルで切り替えたい。最も適切な構成はどれか。

- A. Amazon Route 53 フェイルオーバールーティング
- B. Amazon CloudFront
- C. Elastic Load Balancing
- D. Amazon VPCピアリング

> [!success]- 答え：**A**
> Route 53はヘルスチェックと連動し、異常時にDNS応答をスタンバイへ切り替えるフェイルオーバールーティングを提供する。
> - A：正解。DNSレベルでのヘルスチェック連動フェイルオーバーに最適。
> - B：コンテンツのキャッシュ配信が主目的。
> - C：単一リージョン内のターゲット分散で、サイト全体のDNS切替は行わない。
> - D：VPC間接続であり、フェイルオーバールーティングの機能ではない。
> **関連ノート**：[[0404_Route53]]

## 第5問（Security / IAM ｜ Webアプリ層の保護）
公開WebアプリケーションをSQLインジェクションやクロスサイトスクリプティングなどの一般的な攻撃から、レイヤー7で保護したい。既知の攻撃パターンに対するマネージドルールも利用したい。最も適切なサービスはどれか。

- A. AWS WAF
- B. Amazon GuardDuty
- C. AWS Shield Advanced
- D. AWS Network Firewall

> [!success]- 答え：**A**
> AWS WAFはHTTP/HTTPSリクエストを検査し、SQLi・XSSなどのレイヤー7攻撃をマネージドルールで防御できる。
> - A：正解。Webアプリ層の攻撃フィルタリングに最適。
> - B：脅威検知サービスで、攻撃のインライン遮断は行わない。
> - C：L3/L4を中心としたDDoS攻撃の保護に特化し、SQLi/XSSなどHTTPリクエスト内容の検査・遮断は行わない。
> - D：VPCのネットワークトラフィックをフィルタリングするが、Webアプリ層（L7）のSQLi/XSSをマネージドルールで検査・遮断する用途ではない。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[1603_脅威検知・監査]]

## 第6問（Management / Monitoring ｜ Infrastructure as Code）
インフラ構成をコードとして記述し、複数の環境（開発・本番など）へ再現性をもって同一構成をプロビジョニング・更新したい。最も適切なサービスはどれか。

- A. AWS CloudFormation
- B. AWS Config
- C. AWS CloudTrail
- D. AWS CodePipeline

> [!success]- 答え：**A**
> CloudFormationはテンプレートでインフラを宣言的に定義し、スタックとして再現性高くプロビジョニング・更新できる。
> - A：正解。IaCによる再現可能なインフラ管理に最適。
> - B：構成の評価・準拠チェックが主目的で、プロビジョニングツールではない。
> - C：API監査証跡で、インフラの作成は行わない。
> - D：CI/CDのパイプラインをオーケストレーションするサービスで、インフラ構成そのものを宣言的に定義・プロビジョニングするIaCツールではない。
> **関連ノート**：[[1402_CloudFormation]]

## 第7問（Serverless ｜ 定期実行ジョブ）
毎晩決まった時刻に集計スクリプトを実行したい。専用サーバーは常時起動したくなく、運用負荷も抑えたい。最も適切な構成はどれか。

- A. Amazon EventBridge（スケジュールルール）でAWS Lambdaを起動する
- B. Amazon EC2を常時起動し、cronでスクリプトを実行する
- C. AWS Batch のジョブを毎晩スケジュール実行する
- D. Amazon ECSのスケジュールされたタスク（Fargate）で実行する

> [!success]- 答え：**A**
> EventBridgeのスケジュールルールでLambdaを定刻起動すれば、サーバーを常時起動せずに定期処理を実行できる。
> - A：正解。サーバーレスで軽量な定期ジョブを実現でき、運用負荷が低い。
> - B：常時起動のEC2管理コストとパッチ運用が発生し、運用負荷が高い。
> - C：コンテナ化した大量・長時間バッチ向けで、単純な集計スクリプトの定期実行にはオーバースペック。
> - D：コンテナで定期実行はできるが、軽量なスクリプト1本にもコンテナ定義の用意が必要で、Lambda起動より重い。
> **関連ノート**：[[0804_EventBridge]] / [[0708_Lambda]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]]

## 第8問（Analytics ｜ BIダッシュボード）
経営層向けに、複数のデータソースの集計をインタラクティブなダッシュボードとして可視化し、組織内で共有したい。サーバー管理は避けたい。最も適切なサービスはどれか。

- A. Amazon QuickSight
- B. Amazon Athena
- C. AWS Glue
- D. Amazon EMR

> [!success]- 答え：**A**
> QuickSightはサーバーレスのBIサービスで、ダッシュボードの作成・共有・インタラクティブな可視化に向く。
> - A：正解。可視化・ダッシュボード共有の要件に最適。
> - B：SQLクエリエンジンで、可視化そのものは主目的ではない。
> - C：ETLサービスで、ダッシュボード作成機能ではない。
> - D：大規模分散処理基盤で、BI可視化用途とは異なる。
> **関連ノート**：[[1111_Amazon QuickSight]] / [[1102_データ分析の超基礎・用語集]]

## 第9問（Migration / Transfer ｜ ハイブリッドストレージ）
オンプレミスのアプリケーションから、低レイテンシのローカルキャッシュを保ちつつ、バックアップやファイルをAWSのS3へシームレスに保存・連携したい。最も適切なサービスはどれか。

- A. AWS Storage Gateway
- B. AWS Snowball Edge
- C. AWS DataSync
- D. AWS Snowmobile

> [!success]- 答え：**A**
> Storage GatewayはオンプレとAWSストレージを橋渡しし、ローカルキャッシュを保ちながらS3等へシームレスに連携する。
> - A：正解。ハイブリッドな低レイテンシアクセスとクラウド保存を両立できる。
> - B：一度きりの大容量オフライン移送向けで、継続的なハイブリッドアクセス用途ではない。
> - C：オンプレとS3間のデータ転送・同期は自動化できるが、オンプレアプリへ低レイテンシのローカルキャッシュアクセスを提供する仕組みではない。
> - D：エクサバイト級の超大容量移送向けで、要件に対して過剰。
> **関連ノート**：[[1503_Storage Gateway]] / [[1502_DataSync]] / [[1501_移行と転送サービス一覧]]

## 第10問（Cost Optimization ｜ 予算アラート）
月間のAWS利用額が設定したしきい値を超えそうになった時点で通知を受け取り、コストの超過を早期に把握・管理したい。最も適切なサービスはどれか。

- A. AWS Budgets
- B. AWS Cost Anomaly Detection
- C. AWS Cost Explorer
- D. Amazon CloudWatch 請求アラーム

> [!success]- 答え：**A**
> AWS Budgetsはコストや使用量の予算を設定し、しきい値到達・超過予測時にアラート通知できる。
> - A：正解。予算しきい値に基づくプロアクティブな通知に最適。
> - B：機械学習で異常な支出を検知する用途で、任意の予算しきい値への到達を事前通知する仕組みが主目的ではない。
> - C：コストの可視化・分析向けで、しきい値到達時の自動通知の仕組みではない。
> - D：実績の見積もり請求額がしきい値を超えた時に通知できるが、使用量予算や予測ベースのアラートには対応しない。
> **関連ノート**：[[1701_コスト管理サービス一覧]]
