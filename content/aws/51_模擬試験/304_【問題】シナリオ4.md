# 【問題】シナリオ4（公式サンプル風・全10問）

## この問題集の目的

複数の要件を同時に満たす構成を選ぶ。301〜303で扱っていない切り口（HPC配置・誤削除保護・多層防御・パッチ運用・順序処理・パーソナライズなど）を中心に13分野を横断する。

> [!info] 出題形式（2026-07-05 本番形式へ改訂）
> - シナリオ段落と設問文を分け、です・ます調で記述（公式サンプル準拠）。
> - 選択肢は実在の構成で記述し、「どこが要件に合わないか」を考えさせる。
> - 複数選択問題は5択（A〜E）、単一選択問題は4択（A〜D）。

## 第1問（Compute ｜ 密結合HPCの低遅延配置）
ある研究機関は、ノード間でMPIによる頻繁な通信を行う密結合のHPCシミュレーションを、多数のEC2インスタンスで実行しています。ジョブの完了時間を短縮するために、インスタンス間のネットワークレイテンシを最小化し、高いスループットを確保する必要があります。

この要件を満たす構成はどれですか。

- A. クラスタープレイスメントグループにインスタンスを配置し、拡張ネットワーキング（ENA）を有効化する
- B. 複数AZにまたがるスプレッドプレイスメントグループに分散配置する
- C. パーティションプレイスメントグループに配置し、ラック単位の障害分離を優先する
- D. オンデマンドキャパシティ予約を同一リージョンで作成し、起動容量を事前に確保する

> [!success]- 答え：**A**
> クラスター配置＝「同じラックの近くに固めて置く」。ノード間通信が速くなる代わりに障害は共倒れしやすい、というトレードオフ。
> - A：正解。単一AZ内で物理的に近接配置され、低レイテンシ・高帯域のノード間通信を実現する。
> - B：スプレッドは**わざと離して置く**配置（可用性重視）。低遅延要件とは真逆。
> - C：パーティションはHadoop等の**大規模分散**向け（ラック障害の影響範囲を区切る）。「密結合・低遅延最優先」ならクラスター。
> - D：キャパシティ予約は「起動できること」の保証で、配置の近さやレイテンシとは無関係。
> **着眼点（SAA）**：プレイスメントグループ3種は目的で切る——速く=クラスター／守る=スプレッド／大規模分散=パーティション。「MPI」「HPC」「ノード間通信」はクラスターの合図。
> **関連ノート**：[[0106_プレイスメントグループ]]

## 第2問（Storage ｜ 誤削除・改ざんからの保護）
ある企業は、コンプライアンス上重要なオブジェクトをAmazon S3に保管しています。監査部門は、運用ミスや不正操作による削除・上書きに備えて以前のバージョンを保持すること、さらにバージョンの完全削除には追加の認証を要求することを求めています。

取るべきアクションの組み合わせはどれですか。（2つ選択）

- A. S3バージョニングを有効化する
- B. MFA Deleteを有効化し、バージョンの完全削除に多要素認証を要求する
- C. S3クロスリージョンレプリケーションを有効化し、別リージョンにもコピーを作成する
- D. S3ライフサイクルルールで旧バージョンを30日後に削除する
- E. S3イベント通知で削除イベントを検出し、SNSで管理者に通知する

> [!success]- 答え：**A, B**
> 「前の版を残す」＝バージョニング、「完全削除に追加認証」＝MFA Delete。要件の2文がそのまま2つの機能に対応する。
> - A：正解。上書き・削除しても以前のバージョンに戻せる（削除は「削除マーカー」が付くだけ）。
> - B：正解。バージョンの恒久削除とバージョニング停止にMFAを要求できる。
> - C：きわどい誤答。レプリケーションは**DR（リージョン障害）対策**。削除操作は複製先にも伝播しうるため、誤削除対策の代わりにならない。
> - D：旧バージョンを30日で消すと、復元できる猶予がむしろ**短くなる**（要件と逆行）。
> - E：通知は「気づく」だけで、防ぐことも復元することもできない。
> **着眼点（SAA）**：S3の保護系は要件の動詞で選ぶ——「戻せる」=バージョニング／「消させない」=Object Lock・MFA Delete／「別の場所にも」=レプリケーション。
> **関連ノート**：[[0205_S3のバージョン管理]]

## 第3問（Database ｜ メンテナンス時の可用性）
ある企業の基幹システムはRDSデータベースを使用しています。定期的なパッチ適用やインフラ障害の際にもダウンタイムを最小化し、自動でフェイルオーバーさせる必要があります。また、アプリケーション側の接続先設定は変更したくありません。

この要件を満たす構成はどれですか。

- A. RDSをマルチAZ配置に変更する
- B. シングルAZのまま、毎晩スナップショットを取得する
- C. リードレプリカを1台追加する
- D. DBインスタンスをより大きいインスタンスタイプへ変更する

> [!success]- 答え：**A**
> マルチAZは同期スタンバイへ**同じエンドポイントのまま**自動フェイルオーバーする。「接続先を変えたくない」の一文まで含めて要件どおり。
> - A：正解。パッチ適用時はスタンバイ側から先に適用する動きでダウンタイムも短縮される。
> - B：スナップショットは「あとで復元できる」だけ。復元には時間がかかり、自動フェイルオーバーではない。
> - C：きわどい定番。リードレプリカは非同期の読み取り用で、**自動**フェイルオーバーはしない（手動昇格は可能だが接続先も変わる）。
> - D：性能の話であって可用性の話ではない。
> **着眼点（SAA）**：「自動フェイルオーバー」「ダウンタイム最小」→マルチAZ。「読み取りスケール」→リードレプリカ。この対はSAA最頻出の混同ペア。
> **関連ノート**：[[0301_RDS]] / [[0307_Aurora詳細]]

## 第4問（Security / IAM ｜ 公開Webの多層防御）
ある企業は、インターネットに公開したApplication Load Balancer配下でWebアプリケーションを運用しています。セキュリティチームは、大規模なDDoS攻撃と、SQLインジェクションなどのアプリケーション層攻撃の両方からシステムを保護することを求めています。運用はマネージドサービスで負荷を抑える方針です。

取るべきアクションの組み合わせはどれですか。（2つ選択）

- A. AWS WAFをALBに関連付け、マネージドルールでアプリケーション層攻撃を遮断する
- B. AWS Shield Advancedを有効化し、高度なDDoS保護と緩和を利用する
- C. AWS Shield Standardのみを利用し、L7リクエストの検査は行わない
- D. AWS Network Firewallをパブリックサブネットに配置し、SQLインジェクション検査も任せる
- E. セキュリティグループでALBへの443番ポートだけを許可し、WAFルールは設定しない

> [!success]- 答え：**A, B**
> 攻撃のレイヤーで担当が分かれる——アプリ層（SQLi/XSS）＝WAF、DDoS＝Shield。「両方から守る」なら両方入れる。
> - A：正解。SQLi・XSS・不正なリクエストパターンをマネージドルールで遮断できる。
> - B：正解。Standardを超える大規模・高度なDDoSへの保護と、専門チーム（SRT）の支援を受けられる。
> - C：Shield Standardは全員に自動適用される基本保護のみ。アプリ層攻撃はそもそも守備範囲外。
> - D：きわどい誤答。Network Firewallは**VPCのネットワーク層**の防御。ALB宛てHTTPSの中身（アプリ層）の検査はWAFの領分。
> - E：ポートを絞っても、**許可した443番から入ってくる攻撃リクエスト**は素通りする。
> **着眼点（SAA）**：防御系は層で切る——L3/4 DDoS=Shield／L7アプリ攻撃=WAF／VPC内トラフィック=Network Firewall／インスタンス=SG。「SQLインジェクション」の一語が出たらWAF確定。
> **関連ノート**：[[1604_ネットワーク防御]] / [[1601_セキュリティサービス一覧]]

## 第5問（Management / Monitoring ｜ フリートのパッチ運用）
ある企業は、数百台のEC2インスタンスに対してOSパッチを定期的かつ一貫した基準で適用し、適用状況をコンプライアンスレポートとして提出する必要があります。手作業での適用や各インスタンスへの個別ログインは廃止したいと考えています。

この要件を満たすサービスはどれですか。

- A. AWS Systems Manager Patch Managerでパッチベースラインとメンテナンスウィンドウを設定する
- B. AWS Systems Manager Run Commandで更新コマンドを都度実行し、ベースラインやレポートは使わない
- C. AWS CloudTrailでパッチ適用の操作履歴を記録する
- D. Amazon Inspectorで脆弱性を検出し、自動的にパッチを適用させる

> [!success]- 答え：**A**
> Patch Manager＝「パッチの基準（ベースライン）＋適用スケジュール＋適用状況レポート」の3点セット。
> - A：正解。定期・一貫・レポートの3要件をすべてマネージドに満たす。
> - B：きわどい誤答。Run Commandは「一斉にコマンドを打つ」道具で実行はできるが、**基準管理と準拠レポート**が手作りになる。
> - C：CloudTrailは「誰が何のAPIを呼んだか」の記録。パッチの適用も適用状況の評価もしない。
> - D：Inspectorは脆弱性の**検出**まで。「見つける人」と「直す人」の混同を突く定番。
> **着眼点（SAA）**：Systems Manager系は動詞で切る——適用する=Patch Manager／実行する=Run Command／入る=Session Manager／記録する=CloudTrail（SSM外）。
> **関連ノート**：[[1405_Systems Manager]]

## 第6問（Application Integration ｜ 順序と重複排除が必要な処理）
ある決済システムでは、同一注文に関するイベントを厳密に発生順で処理し、二重課金を防ぐために「正確に1回」だけ処理する必要があります。注文ごとの順序が守られれば、注文をまたいだ並列処理は許容されます。

この要件を満たす構成はどれですか。

- A. Amazon SQS FIFOキューを使い、注文IDをメッセージグループIDとして順序と重複排除を行う
- B. Amazon SQS標準キューで高スループットを優先し、重複は許容する
- C. Amazon SNS FIFOトピックでイベントを配信し、順序と重複の管理はサブスクライバー側で個別に実装する
- D. Amazon EventBridgeイベントバスに送信し、ルールで複数ターゲットへ振り分ける

> [!success]- 答え：**A**
> FIFOキューのメッセージグループIDは「グループ内は順番どおり、グループ間は並列」を実現する仕組み。問題文の要件そのもの。
> - A：正解。注文ID＝グループIDにすれば、注文内の順序保証と重複排除（正確に1回）を両立できる。
> - B：標準キューは順序も「1回だけ」も保証しない。二重課金のリスクが残る。
> - C：きわどい誤答。SNS FIFOは**順序付きの配信**はできるが、「処理を1回に制御する」のはキューを持つSQS FIFO側の仕事。通常はSNS FIFO→SQS FIFOと組み合わせる。
> - D：EventBridgeは振り分けの仕組みで、順序保証・重複排除の要件を満たさない。
> **着眼点（SAA）**：「順序」「exactly once／正確に1回」→SQS FIFO。「注文ごとに順序、注文間は並列」まで書かれていたらメッセージグループIDの理解を問うている。
> **関連ノート**：[[0802_SQS]]

## 第7問（Serverless ｜ サーバーレスな定期バッチ）
ある企業は、毎日深夜に複数の外部APIからデータを取得して集計し、結果をS3へ保存する処理を実行しています。処理時間は数分程度です。企業は常時稼働するサーバーを持たず、運用負荷と実行コストを最小化したいと考えています。

この要件を満たす構成はどれですか。

- A. Amazon EventBridge Schedulerで定刻にAWS Lambda関数を起動し、結果をS3へ保存する
- B. Amazon ECS on EC2で常時稼働クラスターを作成し、cronコンテナで深夜処理を実行する
- C. AWS BatchをオンデマンドEC2環境で構成し、最小vCPUを維持したままジョブを投入する
- D. AWS Step Functionsの状態マシンを作成し、日次のトリガーは外部の運用スクリプトで管理する

> [!success]- 答え：**A**
> 「定刻に短時間だけ動く」＝スケジュール＋Lambdaの黄金パターン。動いた数分しか課金されない。
> - A：正解。サーバーゼロ・実行時のみ課金・スケジュール管理もマネージド。
> - B：深夜数分のためにクラスターを24時間動かすのは本末転倒。
> - C：最小vCPUを維持する構成では待機コストが発生し、数分のジョブには過剰。
> - D：Step Functions自体はサーバーレスだが、トリガーを**外部スクリプト**にした時点で管理すべきサーバー／運用が復活してしまう（EventBridgeで起動すべき）。
> **着眼点（SAA）**：定期実行は「EventBridge（Scheduler）→Lambda」を第一候補に。15分超・大容量メモリならFargateやBatchへ切り替える、という時間軸での使い分けも押さえる。
> **関連ノート**：[[0804_EventBridge]] / [[0708_Lambda]]

## 第8問（Analytics ｜ 近リアルタイムの運用ダッシュボード）
ある企業は、大量のアプリケーションログを継続的に取り込み、障害の予兆を近リアルタイムで検索・可視化できる運用ダッシュボードを構築したいと考えています。サーバーの管理は最小化する方針です。

この要件を満たす構成はどれですか。

- A. Kinesis Data FirehoseでログをAmazon OpenSearch Serviceへ配信し、OpenSearch Dashboardsで可視化する
- B. CloudWatch Logs Insightsで必要時にクエリし、運用担当者が都度検索する
- C. Amazon S3へログを集約し、Amazon Athenaで日次バッチのSQL集計を行う
- D. Amazon Redshiftのプロビジョンドクラスターへログをロードし、夜間に集計クエリを実行する

> [!success]- 答え：**A**
> 「ログの全文検索＋リアルタイム可視化ダッシュボード」はOpenSearch（旧Elasticsearch）の主戦場。Firehoseが配管を担う。
> - A：正解。取り込み（Firehose）→検索・可視化（OpenSearch Dashboards）が近リアルタイムでつながる。
> - B：都度検索は「ダッシュボードで常時可視化」の要件を満たさない（Insightsは調査ツール）。
> - C：Athenaはアドホック分析に強いが、「日次バッチ」では近リアルタイムに届かない。
> - D：DWHは集計分析向けで、ログの全文検索・常時可視化にはOpenSearchが適する。クラスター管理も残る。
> **着眼点（SAA）**：「近リアルタイム＋検索＋ダッシュボード」→OpenSearch。「たまにSQLで分析」→Athena。「大規模定型集計」→Redshift。鮮度と用途で3つを切り分ける。
> **関連ノート**：[[1110_Amazon OpenSearch Service]] / [[1103_Amazon Kinesis]] / [[1109_Amazon Athena]]

## 第9問（Migration / Transfer ｜ 段階移行中のハイブリッドアクセス）
ある企業は、オンプレミスのNASに保存されたファイルデータを段階的にAWSへ移行しています。移行期間中も、オンプレミスのアプリケーションからは低レイテンシのローカルアクセスを維持する必要があります。一方で、データの実体はAmazon S3に置き、オンプレミスのストレージ容量を削減したいと考えています。

この要件を満たすサービスはどれですか。

- A. AWS Storage Gateway（S3 File Gateway）を導入し、NFS/SMBでアクセスさせる
- B. AWS Snowball Edgeでデータを一括移送する
- C. AWS DataSyncでオンプレミスNASとS3を定期同期し、利用者は引き続きオンプレNASを参照する
- D. Amazon FSx for Windows File ServerをAWS上に作成し、オンプレミスからVPN経由でアクセスさせる

> [!success]- 答え：**A**
> File Gateway＝「実体はS3、手元にはよく使う分のキャッシュ」というハイブリッドの窓口。
> - A：正解。オンプレアプリはNFS/SMBのまま低レイテンシでアクセスでき、実体はS3にあるため容量も削減できる。
> - B：一括移送の道具で、移行期間中の**継続的なローカルアクセス**を提供しない。
> - C：きわどい誤答。同期はできるが、利用者がオンプレNASを参照し続ける限り**オンプレの容量は減らない**（実体をS3へ移せていない）。
> - D：ファイル共有はできるが実体はFSx側にあり、「実体はS3」の要件に合わない。VPN越しのアクセスは低レイテンシでもない。
> **着眼点（SAA）**：「オンプレから今まで通り使いたい＋実体はS3＋キャッシュで低レイテンシ」の3点セット→File Gateway。「一度きりの大量輸送」→Snowball、「継続的な転送・同期」→DataSyncと役割で切る。
> **関連ノート**：[[1503_Storage Gateway]] / [[1502_DataSync]] / [[1501_移行と転送サービス一覧]]

## 第10問（Machine Learning ｜ パーソナライズドな推薦）
あるECサイトの企業は、ユーザーの閲覧・購入履歴に基づいてパーソナライズされた商品レコメンドを表示したいと考えています。社内に機械学習の専門知識を持つ人材はおらず、レコメンドモデルの構築・学習・運用はマネージドサービスに任せる方針です。

この要件を満たすサービスはどれですか。

- A. Amazon Personalizeに行動データを渡し、レコメンドAPIを利用する
- B. Amazon Rekognitionで商品画像を解析し、類似商品を提示する
- C. Amazon Comprehendでレビューのテキストを分析し、人気商品を抽出する
- D. Amazon SageMakerでレコメンドモデルを一から構築・訓練する

> [!success]- 答え：**A**
> Personalize＝「Amazon.comの推薦技術の貸し出し」。行動データを渡せば推薦モデルを自動で作ってくれる。
> - A：正解。専門知識なしでパーソナライズされたレコメンドを実現できる。
> - B：画像解析でできるのは「見た目が似ている」まで。**個人の行動履歴に基づく**パーソナライズにはならない。
> - C：テキスト分析で「人気」は分かっても、**ユーザーごと**の推薦にはならない（パーソナライズの要件を外す）。
> - D：「専門知識がない・任せたい」と明記された時点で、一からの構築は消える。
> **着眼点（SAA）**：「レコメンド／推薦」の一語でPersonalize。ML系の問題は要件の名詞（推薦・文字起こし・翻訳…）とサービスの1対1対応で即答してよい。
> **関連ノート**：[[1301_AWS機械学習サービス一覧]]
