# 【問題】シナリオ4（公式サンプル風・全10問）

## この問題集の目的

複数の要件を同時に満たす構成を選ぶ。301〜303で扱っていない切り口（HPC配置・誤削除保護・多層防御・パッチ運用・順序処理・パーソナライズなど）を中心に13分野を横断する。

## 問題数

10問

## 第1問（Compute ｜ 密結合HPCの低遅延配置）
ある研究チームは、ノード間でMPIによる頻繁な通信を行う密結合のHPCシミュレーションを多数のEC2インスタンスで実行している。ジョブ完了時間を縮めるため、インスタンス間のネットワークレイテンシを最小化し、高い帯域を確保したい。最も適切な構成はどれか。

- A. クラスタープレイスメントグループに配置し、拡張ネットワーキング（ENA）を有効化する
- B. 複数AZのスプレッドプレイスメントグループに分散配置する
- C. パーティションプレイスメントグループに配置し、ラック単位の障害分離を優先する
- D. オンデマンドキャパシティ予約を同一リージョンで作成し、起動容量を事前に確保する

> [!success]- 答え：**A**
> クラスタープレイスメントグループは単一AZ内でインスタンスを近接配置し、低レイテンシ・高帯域の通信を実現する。拡張ネットワーキングと併用すると効果が高い。
> - A：正解。密結合HPCのノード間通信を高速化できる。
> - B：スプレッドは可用性重視で、ノードを物理的に離すためHPCの低遅延要件に反する。
> - C：大規模分散処理の障害分離には向くが、密結合HPCの低レイテンシ最優先ではクラスタープレイスメントが合う。
> - D：容量確保には役立つが、インスタンス間の近接配置や低レイテンシ通信を保証するものではない。
> **関連ノート**：[[0106_プレイスメントグループ]]

## 第2問（Storage ｜ 誤削除・改ざんからの保護）
コンプライアンス上重要なオブジェクトをS3に保管している。運用ミスや不正による削除・上書きに備え、以前のバージョンを保持しつつ、バージョンの完全削除には追加の認証を要求したい。取るべきアクションの組み合わせはどれか。（2つ選択）

- A. S3バージョニングを有効化する
- B. MFA Deleteを有効化し、バージョンの完全削除に多要素認証を要求する
- C. S3クロスリージョンレプリケーションを有効化し、別リージョンにもコピーを作成する
- D. S3ライフサイクルルールで旧バージョンを30日後に削除する
- E. S3イベント通知で削除イベントを検出し、SNSで管理者に通知する

> [!success]- 答え：**A, B**
> バージョニングで過去バージョンを保持し、MFA Deleteで完全削除に多要素認証を課すと、誤削除・不正削除への耐性が高まる。
> - A：正解。上書き・削除後も以前のバージョンへ復元できる。
> - B：正解。バージョンの恒久削除に追加認証を要求し、不正・誤操作を防げる。
> - C：別リージョンへの複製はDRには有効だが、削除・上書き時の過去バージョン保持とMFA Deleteの要件を直接満たさない。
> - D：旧バージョンを短期で削除すると、誤削除・改ざんから復元できる期間が短くなる。
> - E：検出と通知はできるが、削除や上書きを防いだり過去バージョンを保持したりはできない。
> **関連ノート**：[[0205_S3のバージョン管理]]

## 第3問（Database ｜ メンテナンス時の可用性）
基幹システムのRDSデータベースで、パッチ適用やインフラ障害の際にもダウンタイムを最小化し、自動でフェイルオーバーさせたい。アプリケーションの接続先は変えたくない。最も適切な構成はどれか。

- A. RDSをMulti-AZ配置にする
- B. シングルAZのまま、毎晩スナップショットを取得する
- C. リードレプリカを1台追加する
- D. インスタンスタイプを大きくする

> [!success]- 答え：**A**
> Multi-AZは別AZの同期スタンバイへ自動フェイルオーバーし、同じエンドポイントのままダウンタイムを最小化する。
> - A：正解。パッチ適用や障害時の可用性を高められる。
> - B：スナップショットは復旧手段だが、自動フェイルオーバーではない。
> - C：リードレプリカは読み取りスケール用で、自動フェイルオーバーの主目的ではない。
> - D：スケールアップは可用性向上の手段ではない。
> **関連ノート**：[[0301_RDS]] / [[0307_Aurora詳細]]

## 第4問（Security / IAM ｜ 公開Webの多層防御）
公開しているApplication Load Balancer配下のWebアプリを、大規模なDDoS攻撃と、SQLインジェクションなどのアプリ層攻撃の両方から保護したい。マネージドで運用負荷を抑えたい。取るべきアクションの組み合わせはどれか。（2つ選択）

- A. AWS WAFをALBに関連付け、マネージドルールでアプリ層攻撃を遮断する
- B. AWS Shield Advancedを有効化し、高度なDDoS保護と緩和を利用する
- C. AWS Shield Standardのみを利用し、ALBへのL7リクエスト検査は行わない
- D. AWS Network Firewallをパブリックサブネットに配置し、HTTPリクエスト本文のSQLインジェクション検査も任せる
- E. セキュリティグループでALBへの443番ポートだけを許可し、WAFルールは設定しない

> [!success]- 答え：**A, B**
> WAFでアプリ層攻撃を遮断し、Shield AdvancedでDDoSを緩和すると、多層防御をマネージドに実現できる。
> - A：正解。SQLi/XSSなどレイヤー7の攻撃をマネージドルールで防げる。
> - B：正解。大規模DDoSに対する高度な保護と緩和を提供する。
> - C：DDoSの基本保護はあるが、SQLインジェクションなどのアプリ層攻撃を検査・遮断できない。
> - D：ネットワーク層の制御には有効だが、ALB宛てHTTPリクエストのアプリ層攻撃対策はWAFが適している。
> - E：ポート制御は必要だが、許可されたHTTPSリクエスト内の攻撃を判定できない。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[1603_脅威検知・監査]]

## 第5問（Management / Monitoring ｜ フリートのパッチ運用）
数百台のEC2インスタンスに対し、OSのパッチ適用を定期的かつ一貫して行い、適用状況をレポートしたい。手作業や個別ログインは避けたい。最も適切なサービスはどれか。

- A. AWS Systems Manager Patch Manager
- B. AWS Systems Manager Run Commandで任意のタイミングに更新コマンドを実行し、パッチベースラインやコンプライアンスレポートは使わない
- C. AWS CloudTrailでパッチ状況を記録する
- D. Amazon Inspectorで自動的にパッチを適用する

> [!success]- 答え：**A**
> Systems Manager Patch Managerはパッチ適用をスケジュール・自動化し、コンプライアンス状況をレポートできる。
> - A：正解。大規模フリートのパッチ運用を一元化・自動化できる。
> - B：一斉コマンド実行には使えるが、定期的なパッチ基準管理や適用状況レポートまで含めるならPatch Managerが適切。
> - C：CloudTrailは監査証跡で、パッチ適用は行わない。
> - D：Inspectorは脆弱性評価で、パッチ適用そのものは行わない。
> **関連ノート**：[[1405_Systems Manager]]

## 第6問（Application Integration ｜ 順序と重複排除が必要な処理）
決済イベントを処理するシステムで、メッセージの処理順序を厳密に保ち、二重課金を避けるために「正確に1回」処理する必要がある。最も適切な構成はどれか。

- A. Amazon SQS FIFOキューを使い、メッセージグループ単位で順序と重複排除を行う
- B. Amazon SQS 標準キューで高スループットを優先する
- C. Amazon SNS FIFOトピックでイベントを配信し、サブスクライバー側で順序と重複を個別に管理する
- D. Amazon EventBridgeイベントバスに送信し、複数ターゲットへルールで振り分ける

> [!success]- 答え：**A**
> SQS FIFOキューは順序保証と重複排除を提供し、決済のような厳密性が必要な処理に向く。
> - A：正解。順序と「正確に1回」の処理要件を満たせる。
> - B：標準キューは順序保証がなく、重複配信もあり得る。
> - C：FIFOトピックは順序付き配信に使えるが、コンシューマーが自分のペースで取り出すキューとしての処理制御はSQS FIFOが合う。
> - D：イベントルーティングには有効だが、同一注文内の厳密な順序処理と重複排除をキューとして制御する要件とは異なる。
> **関連ノート**：[[0802_SQS]]

## 第7問（Serverless ｜ サーバーレスな定期バッチ）
毎日深夜に、複数のAPIからデータを取得して集計し、結果をS3へ保存する処理を実行したい。常時稼働サーバーは持ちたくなく、実行時間も比較的短い。最も適切な構成はどれか。

- A. Amazon EventBridge Schedulerで定刻にAWS Lambdaを起動し、結果をS3へ保存する
- B. Amazon ECS on EC2で常時稼働クラスターを作成し、cronコンテナで深夜処理を実行する
- C. AWS BatchをオンデマンドEC2コンピューティング環境で構成し、最小vCPUを維持したまま毎晩ジョブを投入する
- D. Step Functionsの状態マシンを作成し、日次実行のトリガーとS3保存処理は外部の運用スクリプトで管理する

> [!success]- 答え：**A**
> EventBridge SchedulerでLambdaを定刻起動すれば、サーバーレスかつ低運用で定期バッチを実行できる。
> - A：正解。常時稼働サーバーなしで定期処理を自動化できる。
> - B：コンテナ化はできるが、EC2クラスターの常時運用が残り、短時間処理には重い。
> - C：AWS Batchはバッチ処理に有効だが、短時間処理でEC2容量を維持すると常時稼働サーバーを持ちたくない要件に合いにくい。
> - D：ワークフロー管理には使えるが、定刻起動とS3保存をマネージドに完結する構成ではない。
> **関連ノート**：[[0804_EventBridge]] / [[0708_Lambda]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]]

## 第8問（Analytics ｜ 近リアルタイムの運用ダッシュボード）
大量のアプリケーションログを継続的に取り込み、サーバー管理を最小化しつつ、近リアルタイムで検索・可視化できる運用ダッシュボードを構築したい。最も適切な構成はどれか。

- A. Kinesis Data FirehoseでログをAmazon OpenSearch Serviceへ配信し、OpenSearch Dashboardsで可視化する
- B. CloudWatch Logs Insightsで必要時にクエリし、ダッシュボードは作らず運用担当者が都度検索する
- C. Amazon S3へログを集約し、Amazon Athenaで日次バッチのSQL集計を行う
- D. Amazon Redshiftのプロビジョンドクラスターへログをロードし、夜間に集計クエリを実行する

> [!success]- 答え：**A**
> FirehoseでOpenSearchへ配信し、OpenSearch Dashboardsで可視化すると、サーバー管理を抑えつつ近リアルタイムの運用ダッシュボードを実現できる。
> - A：正解。継続取り込みと近リアルタイムの検索・可視化に合う。
> - B：ログ検索には有効だが、継続取り込みからダッシュボード可視化までを近リアルタイムに整える構成としては弱い。
> - C：アドホック分析には向くが、日次集計では近リアルタイムの検索・可視化に届かない。
> - D：分析基盤としては使えるが、常時クラスター管理が必要で、ログ全文検索・運用ダッシュボード用途ではOpenSearchが合いやすい。
> **関連ノート**：[[1103_Amazon Kinesis]] / [[1110_Amazon OpenSearch Service]]

## 第9問（Migration / Transfer ｜ 段階移行中のハイブリッドアクセス）
オンプレミスのNASを段階的にAWSへ移行している間も、オンプレ側アプリからは低レイテンシのローカルアクセスを維持しつつ、データの実体はS3に置きたい。最も適切なサービスはどれか。

- A. AWS Storage Gateway（File Gateway）
- B. AWS Snowball Edgeで一括移送する
- C. AWS DataSyncでオンプレミスNASとS3を定期同期し、利用者は引き続きオンプレNASだけを参照する
- D. Amazon FSx for Windows File ServerをAWS上に作成し、オンプレ側からVPN経由で直接アクセスする

> [!success]- 答え：**A**
> File Gatewayはローカルキャッシュでオンプレアプリへ低レイテンシアクセスを提供しつつ、データの実体をS3に保存する。
> - A：正解。段階移行中のハイブリッドな低レイテンシアクセスに最適。
> - B：一括移送向けで、移行期間中の継続的なローカルアクセスは提供しない。
> - C：同期には有効だが、S3をバックエンドにしたローカルキャッシュ付きファイル共有を提供するわけではない。
> - D：マネージドなWindowsファイル共有には有効だが、データの実体をS3に置きオンプレ側でキャッシュする要件とは異なる。
> **関連ノート**：[[1503_Storage Gateway]] / [[1502_DataSync]] / [[1501_移行と転送サービス一覧]]

## 第10問（Machine Learning ｜ パーソナライズドな推薦）
ECサイトで、ユーザーの行動履歴に基づくパーソナライズされた商品レコメンドを提供したい。社内に機械学習の専門知識は乏しく、レコメンドモデルの構築・運用はマネージドに任せたい。最も適切なサービスはどれか。

- A. Amazon Personalize
- B. Amazon Rekognition
- C. Amazon Comprehend
- D. Amazon SageMakerでモデルを一から構築する

> [!success]- 答え：**A**
> Personalizeは行動データからレコメンドモデルをマネージドに構築・提供する、専門知識が乏しくても使える推薦サービス。
> - A：正解。パーソナライズドな推薦をマネージドに実現できる。
> - B：画像・動画の解析向けで、推薦用途ではない。
> - C：テキスト分析向けで、推薦エンジンではない。
> - D：一からの構築は専門知識と運用負荷が必要で、要件に反する。
> **関連ノート**：[[1301_AWS機械学習サービス一覧]]
