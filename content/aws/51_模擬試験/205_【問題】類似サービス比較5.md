# 【問題】類似サービス比較5（公式サンプル風・全10問）

## この問題集の目的

役割が似ていて混同しやすいサービスの「使い分け」を要件文で判断する。201〜204で扱っていない比較（FSx Lustre・Aurora Global DB・NLB・Macie・SES・Lambda同時実行・OpenSearch・Compute Optimizer・Lexなど）を中心に13分野を横断する。

## 対象レベル

SAA。サービス名は知っている前提で、要件に最も合う1つを選ぶ。

## 問題数

10問

---

## 第1問（Storage ｜ FSx for Lustre vs EFS vs S3）
機械学習の学習やHPCで、S3上の大規模データセットに対して、数百ギガバイト/秒級のスループットとサブミリ秒のレイテンシを持つ高性能な並列ファイルシステムが必要である。最も適切なサービスはどれか。

- A. Amazon FSx for Lustre
- B. Amazon EFS（標準）
- C. Amazon S3 を直接マウント
- D. Amazon EBS gp3

> [!success]- 答え：**A**
> FSx for LustreはHPC・ML向けの高性能並列ファイルシステムで、S3と連携し高スループット・低レイテンシを提供する。
> - A：正解。HPC/MLの高性能ファイルアクセス要件に最適。
> - B：汎用の共有ファイルシステムで、HPC級スループットには及ばない。
> - C：オブジェクトストレージで、並列ファイルシステムのレイテンシ特性は持たない。
> - D：単一EC2向けブロックストレージで、多数ノードからの並列高スループット用途ではない。
> **関連ノート**：[[0202_S3とEBS・EFSの違い]] / [[0109_EBSボリュームタイプ]]

## 第2問（Database ｜ Aurora Global Database vs Multi-AZ vs リードレプリカ）
ミッションクリティカルなリレーショナルDBで、リージョン障害に備えたクロスリージョンの災害対策を行いたい。RPOを秒単位、RTOを分単位に抑え、別リージョンで読み取りも提供したい。最も適切な機能はどれか。

- A. Amazon Aurora Global Database
- B. Amazon RDS Multi-AZ
- C. 単一リージョンのリードレプリカ
- D. 手動のスナップショットコピー

> [!success]- 答え：**A**
> Aurora Global Databaseはクロスリージョンに低遅延でレプリケートし、リージョン障害時の短いRPO/RTOと別リージョンでの読み取りを実現する。
> - A：正解。クロスリージョンDRと読み取り提供の要件に合う。
> - B：単一リージョン内のAZ間高可用性で、リージョン障害対策にはならない。
> - C：同一リージョン内ではリージョン障害に耐えられない。
> - D：手動コピーはRPO/RTOが大きく、ミッションクリティカル要件に不足。
> **関連ノート**：[[0301_RDS]] / [[0307_Aurora詳細]]

## 第3問（Networking / CDN / LB ｜ NLB vs ALB）
毎秒数百万のリクエストを、極めて低いレイテンシで処理するTCPベースのサービスがある。クライアントに提示する固定IPアドレスが必要で、ソースIPの保持も求められる。最も適切なロードバランサーはどれか。

- A. Network Load Balancer
- B. Application Load Balancer
- C. Gateway Load Balancer
- D. Classic Load Balancer

> [!success]- 答え：**A**
> NLBはレイヤー4で超低レイテンシ・高スループットを提供し、静的IPやソースIP保持に対応する。
> - A：正解。固定IP・低レイテンシ・大量TCPの要件に最適。
> - B：レイヤー7のHTTP/HTTPS向けで、固定IPやL4超低レイテンシ要件には合わない。
> - C：仮想アプライアンスを透過的に挟む用途。
> - D：旧世代で、新規設計では機能・性能面で推奨されない。
> **関連ノート**：[[0503_ALB・NLB・GWLBの違い]] / [[0502_Elastic Load Balancing]]

## 第4問（Security / IAM ｜ Macie vs GuardDuty vs Inspector）
S3バケットに保存されたデータの中から、個人情報（PII）などの機密データを機械学習で自動的に検出・分類し、保護状況を可視化したい。最も適切なサービスはどれか。

- A. Amazon Macie
- B. Amazon GuardDuty
- C. Amazon Inspector
- D. AWS Config

> [!success]- 答え：**A**
> MacieはS3内のデータを機械学習で分析し、PIIなどの機密データを検出・分類してリスクを可視化する。
> - A：正解。S3の機密データ検出・分類に特化している。
> - B：アカウント全体の脅威検知が主目的で、S3内データの分類は行わない。
> - C：EC2やコンテナの脆弱性評価向け。
> - D：構成の準拠評価で、データ内容の機密性分類は行わない。
> **関連ノート**：[[1603_脅威検知・監査]] / [[1605_データ保護・暗号化]]

## 第5問（Application Integration ｜ SES vs SNS）
ユーザーへの注文確認メールやニュースレターなど、リッチな本文を持つ大量のEメールを、高い到達率で送受信したい。最も適切なサービスはどれか。

- A. Amazon Simple Email Service（SES）
- B. Amazon SNS
- C. Amazon SQS
- D. Amazon EventBridge

> [!success]- 答え：**A**
> SESはトランザクションメールやマーケティングメールの大量送受信に特化したEメールサービス。
> - A：正解。リッチなEメールの大量送信・到達率管理に最適。
> - B：プッシュ通知・SMS・シンプルな通知向けで、本格的なEメール送信には不向き。
> - C：メッセージキューであり、Eメール送信サービスではない。
> - D：イベントルーティングで、Eメール配信が主目的ではない。
> **関連ノート**：[[0813_SES]] / [[0803_SNS]]

## 第6問（Serverless ｜ プロビジョンド同時実行 vs オンデマンド）
レイテンシに敏感なAPIをLambdaで提供しているが、急なリクエスト増加時にコールドスタートによる遅延が問題になっている。安定した低レイテンシ応答を確保したい。最も適切な方法はどれか。

- A. Lambdaのプロビジョンド同時実行を構成する
- B. オンデマンドの同時実行に任せたままにする
- C. メモリ割り当てを最小値に下げる
- D. 関数のタイムアウトを短くする

> [!success]- 答え：**A**
> プロビジョンド同時実行は関数の実行環境を事前に初期化・準備し、コールドスタートを回避して安定した低レイテンシを提供する。
> - A：正解。レイテンシ要件の厳しいAPIでコールドスタートを抑えられる。
> - B：何もしなければコールドスタートの遅延は解消しない。
> - C：メモリ削減はむしろ性能低下を招く可能性がある。
> - D：タイムアウト短縮はコールドスタートの解決にならない。
> **関連ノート**：[[0708_Lambda]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]]

## 第7問（Containers ｜ Fargate vs Lambda）
コンテナイメージで提供される処理を、1回あたり数十分かかる長時間実行のジョブとして動かしたい。サーバー管理は避けたいが、Lambdaの実行時間上限には収まらない。最も適切なサービスはどれか。

- A. AWS Fargate
- B. AWS Lambda
- C. Amazon EC2（自己管理）
- D. Amazon Lightsail

> [!success]- 答え：**A**
> Fargateはサーバーレスにコンテナを実行でき、Lambdaの実行時間制限を超える長時間ジョブにも対応する。
> - A：正解。サーバー管理なしで長時間のコンテナジョブを実行できる。
> - B：実行時間に上限があり、数十分の長時間ジョブには収まらない場合がある。
> - C：サーバー管理が必要になる。
> - D：簡易VPS向けで、コンテナのサーバーレス実行用途ではない。
> **関連ノート**：[[0904_ECS]] / [[0902_オーケストレーション_フルマネージド]] / [[0708_Lambda]]

## 第8問（Analytics ｜ OpenSearch vs Athena vs CloudWatch Logs）
大量のアプリケーションログを集約し、全文検索とリアルタイムに近い可視化ダッシュボードで、障害調査時に素早く深掘りしたい。最も適切なサービスはどれか。

- A. Amazon OpenSearch Service
- B. Amazon Athena
- C. Amazon QuickSight
- D. AWS Glue

> [!success]- 答え：**A**
> OpenSearch Serviceはログの全文検索とダッシュボード（OpenSearch Dashboards）による可視化・深掘りに向く。
> - A：正解。ログの検索・分析・可視化をリアルタイムに近い形で行える。
> - B：S3上データへのアドホックSQL向けで、全文検索ダッシュボードの即時深掘りには不向き。
> - C：BIダッシュボード向けで、ログの全文検索エンジンではない。
> - D：ETLサービスで、検索・可視化用途ではない。
> **関連ノート**：[[1110_Amazon OpenSearch Service]]

## 第9問（Cost Optimization ｜ Compute Optimizer vs Cost Explorer vs Trusted Advisor）
稼働中のEC2やAuto Scalingグループ、Lambdaの実際の使用状況メトリクスを分析し、オーバープロビジョニングを是正する具体的なリソースサイズの推奨を得たい。最も適切なサービスはどれか。

- A. AWS Compute Optimizer
- B. AWS Cost Explorer
- C. AWS Budgets
- D. AWS CloudTrail

> [!success]- 答え：**A**
> Compute Optimizerは使用状況メトリクスを機械学習で分析し、適切なインスタンスサイズなどのライトサイジング推奨を提示する。
> - A：正解。リソースの適正化推奨を得る用途に最適。
> - B：コストの可視化・分析向けで、具体的なリソースサイズ推奨が主目的ではない。
> - C：予算しきい値の通知が主目的。
> - D：API監査証跡で、ライトサイジング推奨は行わない。
> **関連ノート**：[[1701_コスト管理サービス一覧]] / [[1401_マネジメント・ガバナンスサービス一覧]]

## 第10問（Machine Learning ｜ Lex vs Polly vs Comprehend）
自然言語で対話するカスタマーサポート用のチャットボット／音声ボットを構築したい。意図（インテント）認識とスロット入力の管理を、マネージドに行いたい。最も適切なサービスはどれか。

- A. Amazon Lex
- B. Amazon Polly
- C. Amazon Comprehend
- D. Amazon Translate

> [!success]- 答え：**A**
> Lexは音声・テキストの対話インターフェース（チャットボット）を構築でき、インテント認識とスロット管理を提供する。
> - A：正解。会話ボットの構築に最適。
> - B：テキストを音声に変換するTTSで、対話管理は行わない。
> - C：テキスト分析向けで、対話フロー管理の機能ではない。
> - D：言語翻訳サービス。
> **関連ノート**：[[1301_AWS機械学習サービス一覧]]
