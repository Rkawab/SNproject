# 【問題】類似サービス比較5（セキュリティ・IAM編・全10問）

## この問題集の目的

第2版（2026-07-14全面新作）。暗号化・証明書・秘密情報・検知系・権限統制の「似た者同士」を要件文で切り分ける。誤答はすべて実在の構成で、**別の要件なら正解になるもの**だけを置く。

## 対象レベル

SAA。セキュリティサービスの守備範囲（何を・どの層で守るか）を問う。

## 問題数

10問

## 第1問（Security / IAM ｜ S3暗号化方式の選択）
S3の機密データについて、①**鍵の利用履歴をCloudTrailで監査**できる、②鍵を無効化すれば全オブジェクトを読み取り不能にできる、③アプリ改修は最小限、の3要件を満たす暗号化方式はどれか。

- A. SSE-S3
- B. SSE-KMS（カスタマーマネージドキー）
- C. SSE-C
- D. クライアントサイド暗号化

> [!success]- 答え：**B**
> SSE-KMSはKMSキーで暗号化するため、キー利用がCloudTrailに記録され、キーの無効化＝データ全体の読み取り遮断ができる。ヘッダー指定だけでアプリ改修も最小。
> - A：監査・無効化の要件が**ない**なら正解になる（無料でデフォルト有効）。鍵はS3任せで、利用履歴も個別に追えない。
> - B：正解。「監査」「鍵の無効化」が出たらKMS。
> - C：鍵を**自社でしか持ちたくない**規制要件なら正解になる。ただし全リクエストで鍵を送る改修と鍵管理の重責を負う。
> - D：AWSに一切平文を渡したくない最高機密なら正解になる。暗号化・復号ロジックをアプリに組み込む大改修が要る。
> **関連ノート**：[[0207_S3の暗号化]] / [[1607_AWS_KMS]]

## 第2問（Security / IAM ｜ 鍵管理基盤の選択）
全社の暗号鍵管理基盤を選定する。多数のAWSサービス（S3・EBS・RDS等）との**ネイティブ統合**、IAMポリシーと連動したアクセス制御、エンベロープ暗号化を、最小の運用負荷で使いたい。最も適切なサービスはどれか。

- A. AWS KMS
- B. AWS CloudHSM
- C. AWS Secrets Manager
- D. AWS Certificate Manager

> [!success]- 答え：**A**
> KMSはAWSサービス統合・IAM連携・監査を備えたマネージド鍵管理。特段の規制がなければ鍵管理はKMSが既定解。
> - A：正解。
> - B：一番きわどい誤答。FIPS 140-2レベル3の**専有ハードウェア**が規制で必須、鍵を完全に自社管理したい場合は正解になる。ただしAWSサービスとの統合は薄く、クラスタ運用も自前。
> - C：管理するのは「鍵」ではなく「秘密情報（パスワード・APIキー）」。DB認証情報のローテーションなら正解になる。
> - D：管理するのはTLS**証明書**。暗号鍵基盤の代わりにはならない。
> **関連ノート**：[[1607_AWS_KMS]] / [[1605_データ保護・暗号化]]

## 第3問（Security / IAM ｜ TLS証明書の管理）
ALBとCloudFrontで使うパブリックTLS証明書を、**無料で発行し、期限切れ前に自動更新**したい。最も適切なサービスはどれか。

- A. サードパーティ証明書を購入しIAMにインポートする
- B. AWS Certificate Manager（ACM）
- C. AWS Private CA
- D. Secrets Managerに証明書と秘密鍵を保存して手動更新する

> [!success]- 答え：**B**
> ACMはパブリック証明書を無料発行し、ALB/CloudFront/API Gateway等に紐づけて自動更新まで面倒をみる。「無料 × 自動更新」が決め手。
> - A：EV証明書など特定の発行元が必要な場合の実在解。ただし自動更新はされず、期限管理は自前。
> - B：正解。
> - C：**社内システム間のmTLS**など、プライベートな認証局が必要なら正解になる。月額固定費が高く、パブリックサイト用ではない。
> - D：保管はできるが、それだけでは配布も更新もされない。証明書のライフサイクル管理はACMの仕事。
> **関連ノート**：[[1605_データ保護・暗号化]]

## 第4問（Security / IAM ｜ 設定値・秘密情報の保管）
アプリケーションの接続先URL・機能フラグなどの設定値と、暗号化したいパラメータを**階層構造で一元管理**したい。DBパスワードの自動ローテーションは不要で、**追加コストをかけたくない**。最も適切なサービスはどれか。

- A. AWS Systems Manager Parameter Store
- B. AWS Secrets Manager
- C. AWS KMSに設定値を直接保存する
- D. S3バケット（SSE-KMS）に設定ファイルを置く

> [!success]- 答え：**A**
> Parameter Storeは階層パス管理・SecureString（KMS暗号化）・バージョン管理を**標準枠なら無料**で提供する。「ローテーション不要 × 無料」ならこちら。
> - A：正解。
> - B：一番きわどい誤答。**自動ローテーション**（RDS等と連携）やクロスアカウント共有が必要なら正解になる。シークレット1件ごとに月額課金がある。
> - C：定番の誤解。KMSが保管するのは「鍵」であり、任意の値の保管庫ではない（暗号化APIはあるが保存場所は別途必要）。
> - D：実在する手だが、取得API・キャッシュ・変更履歴・細粒度の権限管理を自作することになる。専用サービスがある領域。
> **関連ノート**：[[1405_Systems Manager]] / [[1605_データ保護・暗号化]]

## 第5問（Security / IAM ｜ 検出結果の集約）
マルチアカウント環境で、GuardDuty・Inspector・Macie等の検出結果を**1か所に集約**し、CIS AWS Foundationsなどのセキュリティ基準への準拠状況を継続チェックしたい。最も適切なサービスはどれか。

- A. Amazon GuardDuty
- B. AWS Security Hub
- C. AWS Config
- D. Amazon Detective

> [!success]- 答え：**B**
> Security Hubは各セキュリティサービスの検出結果（Findings）を標準形式で集約し、セキュリティ基準のスコアリングを行う「まとめ役」。「集約」「基準チェック」が決め手。
> - A：脅威を**検出する側**のサービス。検出はしても、他サービスの結果をまとめる役ではない。
> - B：正解。
> - C：リソース**構成**の記録と準拠評価なら正解になる。セキュリティ検出結果の集約は守備範囲外（ConfigのデータはSecurity Hubが利用する側）。
> - D：検出後の**深掘り調査**（原因分析・影響範囲の可視化）なら正解になる。集約・スコアリングの機能ではない。
> **関連ノート**：[[1606_コンプライアンス]] / [[1603_脅威検知・監査]]

## 第6問（Security / IAM ｜ 脅威の継続検知）
アカウント内で、①漏えいした認証情報による不審なAPI呼び出し、②EC2から暗号通貨マイニングプールへの通信、③普段と異なる地域からのアクセス、を**機械学習ベースで継続検知**したい。最も適切なサービスはどれか。

- A. Amazon GuardDuty
- B. Amazon Inspector
- C. Amazon Macie
- D. AWS WAF

> [!success]- 答え：**A**
> GuardDutyはCloudTrail・VPCフローログ・DNSログを機械学習で分析し、進行中の脅威（不正利用・侵害兆候）を検知する。「振る舞いの異常検知」が守備範囲。
> - A：正解。
> - B：**攻撃される前の弱点**（OS/アプリのCVE脆弱性、意図しないネットワーク露出）のスキャンなら正解になる。進行中の脅威は見ない。
> - C：**S3内の機密データ（PII）**の発見・分類なら正解になる。データの中身が対象で、振る舞いは見ない。
> - D：L7の攻撃リクエストを**ブロックする**防御装置。検知・分析のサービスではない。
> **関連ノート**：[[1603_脅威検知・監査]]

## 第7問（Security / IAM ｜ 組織全体の禁止事項）
AWS Organizations配下の全アカウントで、**ルートユーザーを含む全プリンシパル**に対し、承認リージョン以外でのリソース作成を禁止したい。最も適切な仕組みはどれか。

- A. 全アカウントへ同一のIAMポリシーを配布する
- B. サービスコントロールポリシー（SCP）
- C. アクセス許可境界（Permission Boundary）
- D. AWS Config Ruleで非準拠リソースを検出する

> [!success]- 答え：**B**
> SCPは組織/OU単位で権限の上限を定め、メンバーアカウントの**ルートユーザーにも効く**唯一のガードレール。「組織全体 × ルートにも強制」が決め手。
> - A：各アカウントの管理者が編集・削除できてしまい、強制にならない。単一アカウント内の権限設計なら本来の道具。
> - B：正解。
> - C：**特定のIAMユーザー/ロール個別**の上限設定なら正解になる。アカウント全体やルートユーザーには適用できない。
> - D：**事後検出**の仕組みで、作成自体は止められない。検出＋自動修復の監査運用なら正解になる。
> **関連ノート**：[[1403_Organizations]] / [[0602_IAMロール詳細]]

## 第8問（Security / IAM ｜ 権限委任の上限）
開発チームに、自分たちが使うIAMロールを**自ら作成する権限を与えたい**。ただし、作成されるロールがチーム管理者の想定を超える権限（例：IAM管理権限）を持てないよう、**上限を強制**したい。最も適切な仕組みはどれか。

- A. サービスコントロールポリシー（SCP）
- B. IAMグループ
- C. アクセス許可境界（Permission Boundary）
- D. ロールの信頼ポリシー

> [!success]- 答え：**C**
> アクセス許可境界は「作成するロール/ユーザーにこの境界を必ず付けること」を条件（iam:PermissionsBoundary）にすることで、委任先が作る権限の**上限**を強制できる。権限委任の定番パターン。
> - A：**アカウント全体**の禁止事項なら正解になる。チーム単位・エンティティ単位の細かい上限には粒度が粗い。
> - B：ユーザーへのポリシーまとめ付け機能。作成されるロールの上限を縛る仕組みではない。
> - C：正解。「自分でロールを作らせつつ上限を課す」のキーワードで反射したい。
> - D：「**誰がこのロールを引き受けられるか**」の制御なら正解になる。ロールが持てる権限の上限とは別物。
> **関連ノート**：[[0602_IAMロール詳細]] / [[0601_IAM基礎]]

## 第9問（Security / IAM ｜ アプリ利用者への一時認証情報）
モバイルアプリで、Googleアカウントでログインした**エンドユーザー**に、各自専用のS3プレフィックスへ直接アクセスできる**一時的なAWS認証情報**を払い出したい。最も適切な構成はどれか。

- A. Cognito IDプール（ユーザープールと連携）
- B. Cognito ユーザープール単体
- C. IAM Identity Center
- D. ユーザーごとにIAMユーザーを発行しアクセスキーを配布

> [!success]- 答え：**A**
> IDプール（フェデレーテッドアイデンティティ）は、認証済みユーザーへIAMロールベースの**一時AWS認証情報**を発行する仕組み。「エンドユーザーがAWSリソースへ直接アクセス」＝IDプール。
> - A：正解。認証はユーザープールやGoogleが担い、AWS認証情報への交換をIDプールが担う。
> - B：定番のひっかけ。サインイン・サインアップとJWT発行（**認証**）なら正解になる。ただしユーザープールが返すのはトークンであり、AWS認証情報は発行しない。
> - C：**従業員**のマルチアカウントSSOなら正解になる。アプリの一般利用者向けの仕組みではない。
> - D：ユーザー数分のIAMユーザーは上限にも管理にも耐えず、長期キー配布は漏えいリスクの塊。
> **関連ノート**：[[0810_Cognito]] / [[1602_アイデンティティ系]]

## 第10問（Security / IAM ｜ ネットワーク防御の層）
VPC内の全ワークロードからの**アウトバウンド通信を、許可したドメイン（FQDN）のみに制限**し、あわせて既知の不正シグネチャを検査（IPS）したい。最も適切なサービスはどれか。

- A. セキュリティグループ
- B. AWS WAF
- C. AWS Network Firewall
- D. Amazon GuardDuty

> [!success]- 答え：**C**
> Network FirewallはVPCの境界に置くマネージドFWで、FQDNベースの送信フィルタリングとIPS（Suricata互換ルール）を提供する。「ドメインで出口制限 × IPS」が決め手。
> - A：定番のひっかけ。IP・ポート単位の制御なら基本装備として正解だが、**FQDN指定ができない**（IPは変わる）。
> - B：守る向きが逆。**インバウンドのWebリクエスト**（SQLi/XSS）をCloudFront/ALBの前で守るなら正解になる。
> - C：正解。
> - D：不審な通信を**検知**はするが、通信を**遮断**する装置ではない。検知して通知したい要件なら正解になる。
> **関連ノート**：[[1604_ネットワーク防御]] / [[0408_セキュリティグループとNACL]]
