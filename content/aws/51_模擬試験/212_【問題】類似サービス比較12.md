# 【問題】類似サービス比較12（セキュリティ・運用・ガバナンス）

## この問題集の目的

セキュリティ、監査、運用、マルチアカウント管理で混同しやすいサービスを、決め手となる要件で比較する。

## 対象レベル

SAA基礎〜標準。似たサービスの役割境界を1問1論点で確認する。

## 問題数

10問

---

## 第1問（Security / IAM ｜ Inspector と GuardDuty）
EC2インスタンスにインストールされたパッケージの既知の脆弱性を継続的に検出し、修正の優先順位を付けたいと考えています。
最も適したサービスはどれですか。

- A. Amazon Inspectorでワークロードを継続スキャンする
- B. Amazon GuardDutyでCloudTrailやVPC Flow Logsを分析する
- C. Amazon Detectiveで検出結果の関係を可視化する
- D. AWS Security Hub CSPMでセキュリティ標準を有効化する

> [!success]- 答え：**A**
> Inspectorは「弱点を探す」、GuardDutyは「不審な活動を探す」サービスです。
> - A：正解。CVEなどのソフトウェア脆弱性とネットワーク露出を評価する。
> - B：侵害の兆候や異常な活動を検知するが、パッケージの脆弱性スキャンではない。
> - C：検知済みの事象を調査する段階で使う。
> - D：検出結果とセキュリティ態勢の集約に使う。
> **着眼点（SAA）**：脆弱性・CVEはInspector、不審なAPI・通信はGuardDuty。
> **関連ノート**：[[1603_脅威検知・監査]]
> **詳しい解説**：[[212_【解説】類似サービス比較12#第1問（Security / IAM ｜ Inspector と GuardDuty）]]

## 第2問（Security / IAM ｜ Security Hub と Detective）
複数アカウントのGuardDutyやInspectorから届く検出結果を標準形式で集約し、セキュリティ標準への準拠状況も一元確認したいと考えています。
最も適したサービスはどれですか。

- A. AWS Security Hub CSPM
- B. Amazon Detective
- C. Amazon GuardDuty
- D. AWS Audit Manager

> [!success]- 答え：**A**
> Security Hub CSPMは「検出結果を集めて全体を見る」、Detectiveは「個別事象を深掘りする」という違いがあります。
> - A：正解。複数サービス・アカウントの検出結果とセキュリティ態勢を集約する。
> - B：検出結果に関連するユーザー、リソース、通信をたどって根本原因を調査する。
> - C：脅威を検知する情報源の一つ。
> - D：監査フレームワークに沿って証拠を収集・評価する。
> **着眼点（SAA）**：集約・態勢管理はSecurity Hub、根本原因調査はDetective。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[1603_脅威検知・監査]]
> **詳しい解説**：[[212_【解説】類似サービス比較12#第2問（Security / IAM ｜ Security Hub と Detective）]]

## 第3問（Management / Monitoring ｜ CloudTrail と Config）
セキュリティグループのインバウンドルールが、先週のある時点でどのような設定だったかを確認し、現在の構成が社内ルールに準拠しているか評価したいと考えています。
最も適したサービスはどれですか。

- A. AWS Config
- B. AWS CloudTrail
- C. Amazon CloudWatch
- D. AWS X-Ray

> [!success]- 答え：**A**
> Configは「リソースがどう設定されていたか」、CloudTrailは「誰がどのAPIを呼んだか」を記録します。
> - A：正解。構成履歴の参照とConfig Rulesによる準拠評価ができる。
> - B：変更操作を行った主体やAPI呼び出しの追跡には使えるが、構成状態の評価が主目的ではない。
> - C：メトリクス・ログ・アラームによる運用監視サービス。
> - D：分散アプリケーションのリクエスト追跡サービス。
> **着眼点（SAA）**：構成の履歴・準拠はConfig、操作の監査証跡はCloudTrail。
> **関連ノート**：[[1401_マネジメント・ガバナンスサービス一覧]] / [[1603_脅威検知・監査]]
> **詳しい解説**：[[212_【解説】類似サービス比較12#第3問（Management / Monitoring ｜ CloudTrail と Config）]]

## 第4問（Security / IAM ｜ Artifact と Audit Manager）
監査担当者から、AWSが取得しているSOCレポートとPCI関連文書の提出を求められました。自社リソースの設定証拠を新たに収集する必要はありません。
最も適したサービスはどれですか。

- A. AWS ArtifactからAWSのコンプライアンス文書を取得する
- B. AWS Audit Managerで自社環境の評価を作成する
- C. AWS Configで構成スナップショットを配信する
- D. AWS Security Hub CSPMで標準チェックを実行する

> [!success]- 答え：**A**
> Artifactは「AWS側の監査文書を受け取る窓口」、Audit Managerは「自社AWS利用の監査証拠を集める道具」です。
> - A：正解。SOCやPCIなどAWSのコンプライアンスレポートをオンデマンドで取得できる。
> - B：自社環境の証拠収集と評価を自動化する場合に使う。
> - C：自社リソースの構成情報を記録する。
> - D：セキュリティベストプラクティスへの準拠状況を集約する。
> **着眼点（SAA）**：AWSの報告書はArtifact、自社監査の証拠はAudit Manager。
> **関連ノート**：[[1606_コンプライアンス]]
> **詳しい解説**：[[212_【解説】類似サービス比較12#第4問（Security / IAM ｜ Artifact と Audit Manager）]]

## 第5問（Security / IAM ｜ Firewall Manager と WAF）
Organizations配下にある新規・既存アカウントのCloudFrontディストリビューションへ、共通のAWS WAFルールを自動で適用し、非準拠リソースを中央で確認したいと考えています。
最も適したサービスはどれですか。

- A. AWS Firewall Managerで組織向けポリシーを作成する
- B. 各アカウントでAWS WAFのWeb ACLを手動作成する
- C. AWS Network Firewallを各VPCに配置する
- D. AWS Shield Standardを無効化してShield Advancedだけを使う

> [!success]- 答え：**A**
> WAFは個々のWebリソースを保護し、Firewall ManagerはWAFなどの設定を組織全体へ中央展開します。
> - A：正解。複数アカウントへ共通ポリシーを適用し、準拠状況も管理できる。
> - B：少数リソースなら可能だが、組織全体への自動適用と中央管理の要件を満たしにくい。
> - C：VPCのネットワークトラフィック検査用で、CloudFrontへのWeb ACL配布ではない。
> - D：Shield Standardは自動提供され、WAFルールの中央配布とは無関係。
> **着眼点（SAA）**：個別のL7ルールはWAF、Organizations横断の配布はFirewall Manager。
> **関連ノート**：[[1604_ネットワーク防御]] / [[1403_Organizations]]
> **詳しい解説**：[[212_【解説】類似サービス比較12#第5問（Security / IAM ｜ Firewall Manager と WAF）]]

## 第6問（Management / Monitoring ｜ Session Manager と 踏み台）
プライベートサブネットのEC2へ管理者が接続します。SSH鍵の管理、踏み台EC2のパッチ適用、ポート22の開放をすべてなくし、IAMで接続権限を制御したいと考えています。
最も適した構成はどれですか。

- A. Systems Manager Session Managerを使用する
- B. パブリックサブネットに踏み台EC2を配置する
- C. 各EC2へElastic IPを割り当てる
- D. Client VPN経由で各EC2へSSH接続する

> [!success]- 答え：**A**
> Session Managerは管理用通信をアウトバウンドで確立し、SSH鍵や踏み台なしでIAM制御のセッションを提供します。
> - A：正解。インバウンドポートを開けず、操作ログの記録も構成できる。
> - B：接続経路にはなるが、踏み台自体の可用性・パッチ・鍵を管理する必要が残る。
> - C：攻撃対象を増やし、要件と逆行する。
> - D：プライベート接続は可能だが、SSH鍵とSSHポートの管理が残る。
> **着眼点（SAA）**：「踏み台・ポート・鍵を廃止」ならSession Manager。
> **関連ノート**：[[1204_SSM_Session_Manager]]
> **詳しい解説**：[[212_【解説】類似サービス比較12#第6問（Management / Monitoring ｜ Session Manager と 踏み台）]]

## 第7問（Management / Monitoring ｜ Control Tower と Organizations）
新規AWSアカウントを標準化された方法で払い出し、ログアーカイブや監査アカウントを含むランディングゾーンを自動構成し、統制ルールを継続適用したいと考えています。
最も適したサービスはどれですか。

- A. AWS Control Tower
- B. AWS Organizationsのみ
- C. AWS IAM Identity Centerのみ
- D. AWS Resource Access Manager

> [!success]- 答え：**A**
> Organizationsはマルチアカウント管理の土台で、Control Towerはその上にベストプラクティスのランディングゾーンを組み立てます。
> - A：正解。アカウントファクトリーとコントロールによりセットアップと統制を自動化する。
> - B：OUやSCPなどは提供するが、ランディングゾーン一式の自動構成には追加実装が必要。
> - C：従業員のSSOを管理する機能で、アカウント環境全体を構築しない。
> - D：対応リソースをアカウント間で共有する。
> **着眼点（SAA）**：組織構造・SCPはOrganizations、ランディングゾーン自動化はControl Tower。
> **関連ノート**：[[1404_Control Tower]] / [[1403_Organizations]]
> **詳しい解説**：[[212_【解説】類似サービス比較12#第7問（Management / Monitoring ｜ Control Tower と Organizations）]]

## 第8問（Management / Monitoring ｜ Service Catalog と CloudFormation）
開発者が、中央IT部門の承認済み構成だけを一覧から選んでセルフサービスで起動できるようにし、利用可能なインスタンスタイプなども制限したいと考えています。
最も適したサービスはどれですか。

- A. AWS Service Catalog
- B. AWS CloudFormationのみ
- C. AWS Systems Manager Automation
- D. AWS Config

> [!success]- 答え：**A**
> CloudFormationは構築エンジン、Service Catalogは承認済み製品を統制付きで配布するカタログです。
> - A：正解。CloudFormation製品をポートフォリオにまとめ、起動制約やアクセス権を設定できる。
> - B：リソース構築はできるが、承認製品の発見・配布・制約管理を単独では提供しない。
> - C：運用手順の自動化に使う。
> - D：構成の記録と準拠評価に使う。
> **着眼点（SAA）**：作る仕組みはCloudFormation、承認済み構成を配る仕組みはService Catalog。
> **関連ノート**：[[1203_CloudFormationとServiceCatalog]] / [[1402_CloudFormation]]
> **詳しい解説**：[[212_【解説】類似サービス比較12#第8問（Management / Monitoring ｜ Service Catalog と CloudFormation）]]

## 第9問（Cost Optimization ｜ Compute Optimizer と Cost Explorer）
CPUやメモリなどの利用状況を分析し、特定のEC2インスタンスをどのインスタンスタイプへ変更すべきか具体的な推奨を得たいと考えています。
最も適したサービスはどれですか。

- A. AWS Compute Optimizer
- B. AWS Cost Explorer
- C. AWS Budgets
- D. AWS Cost Anomaly Detection

> [!success]- 答え：**A**
> Compute Optimizerはリソース構成の最適化を推奨し、Cost Explorerは支出の傾向を可視化します。
> - A：正解。メトリクスを分析して具体的なライトサイジング候補を示す。
> - B：サービス別・タグ別などでコストと使用量を分析する。
> - C：予算しきい値の監視と通知に使う。
> - D：通常パターンから外れた支出を検知する。
> **着眼点（SAA）**：「どのサイズに変えるか」はCompute Optimizer、「いくら使ったか」はCost Explorer。
> **関連ノート**：[[1202_Compute_Optimizer]] / [[1701_コスト管理サービス一覧]]

## 第10問（Networking / CDN / LB ｜ AWS RAM と PrivateLink）
ネットワーク管理アカウントが所有するTransit Gatewayを、Organizations内の複数アカウントから同じリソースとして利用させたいと考えています。
最も適したサービスはどれですか。

- A. AWS Resource Access Manager（AWS RAM）
- B. AWS PrivateLink
- C. VPCピアリング
- D. AWS Transit Gatewayを各アカウントに個別作成する

> [!success]- 答え：**A**
> AWS RAMは対応リソースそのものを共有し、PrivateLinkはサービスへのプライベートな接続口を公開します。
> - A：正解。中央所有のTransit Gatewayを他アカウントと共有できる。
> - B：NLBなどの背後のサービスをインターフェイスエンドポイント経由で公開する用途。
> - C：VPC同士のネットワーク接続であり、Transit Gatewayの所有権を共有しない。
> - D：重複作成を避けるという要件に反する。
> **着眼点（SAA）**：リソース共有はRAM、サービス公開はPrivateLink。
> **関連ノート**：[[0406_VPC応用機能]] / [[1403_Organizations]]
> **詳しい解説**：[[212_【解説】類似サービス比較12#第10問（Networking / CDN / LB ｜ AWS RAM と PrivateLink）]]
