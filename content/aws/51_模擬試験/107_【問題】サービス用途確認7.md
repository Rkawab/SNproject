# 【問題】サービス用途確認7（公式サンプル風・全10問）

## この問題集の目的

要件文から主要サービスの役割を選ぶドリル。101〜106で扱っていない切り口（アプリ環境の自動構築・遠隔地アップロード高速化・MySQL互換マネージドDB・プライベートサブネットのアウトバウンド・アプリ利用者の認証・複数アカウント統制・既存メッセージブローカー移行・Kubernetes互換基盤・ログ全文検索・予算超過通知など）を中心に、13分野を横断する。

## 対象レベル

SAA入口。単語暗記ではなく、要件からサービスを選ぶ練習にする。

## 問題数

10問

## 第1問（Compute ｜ アプリ実行環境の自動構築）
開発者がアプリのコードをアップロードするだけで、容量プロビジョニング・ロードバランシング・自動スケーリング・モニタリングまで含む実行環境を自動で用意してほしい。インフラの細部は管理したくないが、必要なら下層のEC2にもアクセスしたい。最も適切なサービスはどれか。

- A. AWS Elastic Beanstalk
- B. 各リソースを手動でEC2上に構築する
- C. Amazon S3 静的ウェブサイトホスティング
- D. AWS Lambda のみ

> [!success]- 答え：**A**
> Elastic Beanstalkはコードをアップロードするだけで実行環境一式（容量・LB・スケーリング・監視）を自動構築し、下層リソースにもアクセスできる。
> - A：正解。最小運用で環境一式を自動構築でき、必要なら下のEC2も触れる。
> - B：手動構築は要件の「自動で用意」に反し、運用負荷が高い。
> - C：静的コンテンツ配信向けで、アプリ実行環境の自動構築ではない。
> - D：関数単位の実行で、LB・容量管理を含む環境一式の自動構築ではない。
> **関連ノート**：[[0702_Lambda・ECS・RDS・Beanstalk比較]] / [[0701_コンピューティングサービス一覧]]

## 第2問（Storage ｜ 遠隔地からの大容量アップロード高速化）
世界各地の拠点から、大容量ファイルを単一リージョンのS3バケットへアップロードしているが、距離に起因する遅延で転送が遅い。アプリ側の改修は最小限に、CloudFrontのエッジ経由でアップロードを高速化したい。最も適切な機能はどれか。

- A. Amazon S3 Transfer Acceleration
- B. S3 標準のまま並列アップロード数だけ増やす
- C. AWS Direct Connect
- D. Amazon EFS

> [!success]- 答え：**A**
> S3 Transfer AccelerationはCloudFrontのエッジロケーション経由で、長距離のS3アップロードを高速化する。
> - A：正解。距離由来の遅延をエッジ経由で緩和し、アプリ改修も最小で済む。
> - B：並列化だけでは距離由来の遅延の根本解決にならない。
> - C：オンプレ専用線で、世界各地の拠点からの汎用アップロード高速化用途ではない。
> - D：共有ファイルシステムで、S3への遠隔アップロード高速化機能ではない。
> **関連ノート**：[[0211_S3パフォーマンス最適化]] / [[0201_S3]]

## 第3問（Database ｜ MySQL/PostgreSQL互換の高性能マネージドDB）
既存のMySQL互換アプリを大きく改修せずに、商用DB級の高スループットと、ストレージの自動拡張、最大15のリードレプリカによる読み取りスケールを得たい。マネージドで運用負荷を抑えたい。最も適切なサービスはどれか。

- A. 自前でEC2上にMySQLを構築する
- B. Amazon Aurora（MySQL互換）
- C. Amazon DynamoDB
- D. Amazon Redshift

> [!success]- 答え：**B**
> AuroraはMySQL/PostgreSQL互換のクラウドネイティブDBで、高スループット・ストレージ自動拡張・最大15リードレプリカに対応する。
> - A：自前構築は運用負荷が高く、自動拡張や高可用性を自分で作り込む必要がある。
> - B：正解。互換性を保ちつつ高性能・自動拡張・読み取りスケールを得られる。
> - C：キー・バリューのNoSQLで、MySQL互換のリレーショナル用途ではない。
> - D：分析向けデータウェアハウスで、OLTPのMySQL互換用途ではない。
> **関連ノート**：[[0307_Aurora詳細]] / [[0301_RDS]]

## 第4問（Networking / CDN / LB ｜ プライベートサブネットのアウトバウンド）
プライベートサブネットのEC2から、OSパッチ取得などインターネットへのアウトバウンド通信を行いたい。ただしインターネット側からの新規インバウンド接続は受け付けたくない。マネージドで可用性の高い仕組みがよい。最も適切なものはどれか。

- A. NAT ゲートウェイ
- B. インターネットゲートウェイにEC2を直接公開する
- C. ゲートウェイ型VPCエンドポイント
- D. AWS Direct Connect

> [!success]- 答え：**A**
> NATゲートウェイはプライベートサブネットからのアウトバウンドのみを許可し、外部からの新規インバウンドは受けないマネージドな仕組み。
> - A：正解。アウトバウンド限定で可用性が高く、要件に合致する。
> - B：パブリック公開となり外部からのインバウンドを受けてしまうため要件に反する。
> - C：S3/DynamoDB等への私設経路で、インターネット全般へのアウトバウンドではない。
> - D：オンプレ専用線で、インターネットへのアウトバウンド用途ではない。
> **関連ノート**：[[0409_IGW・NAT Gateway・VPCエンドポイント]]

## 第5問（Security / IAM ｜ アプリ利用者の認証）
モバイル/Webアプリに、利用者のサインアップ・サインイン機能と、GoogleやAppleなど外部IDプロバイダー連携、トークン発行をマネージドで追加したい。アプリ側で認証基盤を自作したくない。最も適切なサービスはどれか。

- A. 利用者ごとにAWS IAMユーザーを作成する
- B. Amazon Cognito
- C. AWS KMS
- D. AWS Secrets Manager

> [!success]- 答え：**B**
> Cognitoはアプリ利用者のサインアップ/サインイン、外部IdP連携、トークン発行をマネージドで提供する。
> - A：IAMユーザーはAWS運用者向けのIDで、アプリ利用者の大量サインアップ管理には適さない。
> - B：正解。エンドユーザー認証基盤をマネージドに追加でき、外部IdP連携も可能。
> - C：暗号鍵管理サービスで、ユーザー認証基盤ではない。
> - D：シークレット保管/ローテーション向けで、エンドユーザー認証ではない。
> **関連ノート**：[[0810_Cognito]] / [[0602_IAMロール詳細]]

## 第6問（Management / Monitoring ｜ 複数アカウントの一元管理）
部門ごとに増えたAWSアカウントを一元管理したい。一括請求でボリューム割引を受けつつ、サービスコントロールポリシー（SCP）で利用可能なサービスを組織全体に制限したい。最も適切なサービスはどれか。

- A. すべてを単一アカウントのIAMグループで分離する
- B. AWS Organizations
- C. AWS Config
- D. AWS IAM Identity Center だけを使う

> [!success]- 答え：**B**
> Organizationsは複数アカウントの一元管理・一括請求・SCPによるガードレールを提供する。
> - A：単一アカウント内のグループ分離では、アカウント分割・一括請求・SCPの要件を満たせない。
> - B：正解。アカウント統制・一括請求・組織全体のサービス制限を満たす。
> - C：構成変更の記録・準拠評価が主目的で、アカウント統制・一括請求ではない。
> - D：SSOによるアクセス管理が主で、SCPや一括請求の土台はOrganizations側にある。
> **関連ノート**：[[1403_Organizations]] / [[1401_マネジメント・ガバナンスサービス一覧]]

## 第7問（Application Integration ｜ 既存メッセージブローカーの移行）
オンプレで稼働するRabbitMQ/Apache ActiveMQベースのアプリを、メッセージングプロトコル（AMQP/MQTT/STOMP等）を変えずにAWSへ移行したい。アプリ側のコード改修を最小化したい。最も適切なサービスはどれか。

- A. Amazon SQS に書き換える
- B. Amazon MQ
- C. Amazon SNS
- D. Amazon Kinesis Data Streams

> [!success]- 答え：**B**
> Amazon MQは業界標準プロトコル互換のマネージドメッセージブローカーで、既存資産をほぼ無改修で移行できる。
> - A：SQSはAWS独自APIで、既存プロトコル資産をそのまま使えずコード改修が必要。
> - B：正解。既存ブローカーのプロトコル互換で、最小改修の移行に最適。
> - C：Pub/Sub通知向けで、既存ブローカープロトコル互換の移行用途ではない。
> - D：ストリーミングデータ向けで、メッセージブローカーの置き換えではない。
> **関連ノート**：[[0806_Amazon_MQ]] / [[0801_アプリケーション統合サービス一覧]]

## 第8問（Containers ｜ Kubernetes互換のマネージド基盤）
すでにKubernetesで運用しているワークロードを、エコシステム（kubectl・Helm等）をそのまま使いつつ、コントロールプレーンの運用をAWSに任せたい。最も適切なサービスはどれか。

- A. Amazon ECS（AWS独自オーケストレーター）
- B. EC2上に自前でKubernetesクラスタを構築する
- C. Amazon EKS
- D. AWS Lambda

> [!success]- 答え：**C**
> EKSはマネージドなKubernetesコントロールプレーンを提供し、既存のKubernetesエコシステムをそのまま使える。
> - A：AWS独自のオーケストレーターで、Kubernetes APIエコシステムはそのまま使えない。
> - B：自前構築はコントロールプレーン運用の負荷が高く、要件の「AWSに任せる」に反する。
> - C：正解。Kubernetes互換で運用を任せられ、既存資産を活かせる。
> - D：関数実行基盤で、Kubernetesワークロードの実行基盤ではない。
> **関連ノート**：[[0906_EKS]] / [[0902_オーケストレーション_フルマネージド]]

## 第9問（Analytics ｜ ログの全文検索とダッシュボード）
大量のアプリログを集約し、ほぼリアルタイムに全文検索・分析し、ダッシュボードで可視化したい。マネージドな検索/分析エンジンを使いたい。最も適切なサービスはどれか。

- A. Amazon OpenSearch Service
- B. Amazon S3 Glacier
- C. Amazon RDS for PostgreSQL
- D. AWS Backup

> [!success]- 答え：**A**
> OpenSearch Serviceはログの全文検索・準リアルタイム分析とダッシュボード可視化に適したマネージドサービス。
> - A：正解。全文検索・分析・可視化を一体で満たす。
> - B：長期アーカイブ向けで、全文検索・可視化の用途ではない。
> - C：トランザクション向けRDBで、ログ全文検索エンジンとしては不向き。
> - D：バックアップ集中管理サービスで、検索・可視化用途ではない。
> **関連ノート**：[[1110_Amazon OpenSearch Service]] / [[1101_AWS分析サービス一覧]]

## 第10問（Cost Optimization ｜ 予算超過の検知と通知）
月次のコストや使用量に上限を設定し、しきい値に近づいた／超えた際にメールで通知を受け取りたい。予測値に基づくアラートも欲しい。最も適切なサービスはどれか。

- A. AWS Cost Explorer のグラフを毎日目視で確認する
- B. AWS Budgets
- C. AWS Trusted Advisor
- D. Amazon CloudWatch のEC2メトリクスアラーム

> [!success]- 答え：**B**
> AWS Budgetsはコスト/使用量に予算と通知しきい値を設定でき、予測ベースのアラートにも対応する。
> - A：分析・可視化は得意だが、予算しきい値による自動通知の仕組みではない。
> - B：正解。任意の予算しきい値で自動通知でき、予測アラートも出せる。
> - C：ベストプラクティス助言が主で、任意の予算しきい値通知の仕組みではない。
> - D：リソースメトリクス監視向けで、コスト予算の超過通知が主目的ではない。
> **関連ノート**：[[1701_コスト管理サービス一覧]]
