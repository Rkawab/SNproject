# 【問題】外部サイト模擬試験 2（複数分野・全10問）

## この問題集の目的

外部サイトの模擬問題を、答えの暗記ではなく本番形式（長文シナリオ）で解く練習として収録する。`501` が10問で満杯になったため、その続きを本ファイルに蓄積する。分野は固定せず、出題された問題を順次追加していく。

> [!info] 出題形式
> - シナリオ段落と設問文を分け、です・ます調で記述。
> - 選択肢はサービス名ではなく、具体的な操作・構成で記述。
> - 原則は単一選択（4択 A〜D）。複数選択の問題は5択（A〜E）とし、設問文に「2つ選択」等を明記。

> [!note] 運用ルール
> 外部サイト模擬試験は分野横断で1ファイルに蓄積する。**1ファイルが10問を超えたら `503_…` 以降に新規ファイルを作成**して分割する（解説側 `61_模擬問題解説/` も同じ番号で対応させる）。

## 第1問（Security / IAM ｜ プライベート閉域でのS3アクセスと一時認証情報）
ある企業が、プライベートサブネットで動作するWebアプリケーションを運用しています。このアプリケーションは既に Amazon Cognito ユーザープールと統合されており、ユーザー認証が実装されています。今後、認証されたユーザーが個人ドキュメントを Amazon S3 バケットにアップロードする機能を追加する必要があります。なお、プライベートサブネットから S3 バケットへのトラフィックはインターネットを経由せず、AWSネットワーク内で完結させる必要があります。

アプリケーションから S3 バケットへ安全にアクセスさせるために必要な手順の組み合わせはどれですか。（2つ選択）

- A. VPC に NAT ゲートウェイを作成してパブリックサブネットに配置し、プライベートサブネットのルートテーブルを更新する。S3 バケットポリシーで NAT ゲートウェイの Elastic IP からのアクセスのみを許可する。
- B. Cognito ID プールを設定し、認証されたユーザーに対して IAM ロールを引き受けるための一時的な AWS 認証情報を提供する。
- C. VPC 内に Amazon S3 バケットへの接続用の VPC エンドポイント（ゲートウェイ型）を作成し、プライベートサブネットのルートテーブルに関連付ける。
- D. VPC 内に AWS Storage Gateway を配置し、プライベートサブネットのアプリケーションから NFS または SMB プロトコルで S3 バケットにアクセスできるようにする。
- E. Cognito アダプティブ認証を有効化し、多要素認証を適用することで認証成功後のユーザーに S3 バケットへのアクセス権限を付与する。

> [!success]- 答え：**B ＋ C**
> 「認証ユーザーにS3操作の認可（一時認証情報）」＝Cognito IDプール、「インターネット非経由でAWS内完結」＝S3用のゲートウェイ型VPCエンドポイント。認可と閉域経路の2軸を別々に満たす。
> - A：NAT ゲートウェイはインターネット（パブリック経路）を経由するため、「インターネット非経由・AWS内完結」要件に反する。
> - B：正解。ユーザープール（認証）に対し、IDプールが認証済みユーザーへIAMロールの一時認証情報を発行し、S3への認可を安全に与える。
> - C：正解。S3はゲートウェイ型VPCエンドポイントに対応し、ルートテーブル関連付けでインターネットを通らずVPC内で完結してアクセスできる。
> - D：Storage Gateway はオンプレ↔AWSのハイブリッドストレージ用で、VPC内アプリからS3への直接アクセス手段ではない。
> - E：アダプティブ認証・MFAは認証強度の話で、S3への権限付与や通信経路の要件とは無関係。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[0409_IGW・NAT Gateway・VPCエンドポイント]] / [[0810_Cognito]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第1問（Security / IAM ｜ プライベート閉域でのS3アクセスと一時認証情報）]]

## 第2問（Serverless ｜ Lambdaの大容量依存ライブラリ共有ストレージ）
ある企業が、VPC内でデータ変換ジョブを実行するサーバーレスアプリケーションを構築しています。アプリケーションが動作する Lambda 関数は、大規模な機械学習ライブラリを含む依存パッケージにアクセスする必要がありますが、その依存パッケージのサイズが Lambda レイヤーのクォータを超えています。依存パッケージは複数の Lambda 関数から同時にアクセスされ、Lambda 関数が読み込むデータは転送中に暗号化してセキュリティを確保する必要があります。

運用上のオーバーヘッドを最小限に抑えながら、これらの要件を満たすソリューションはどれですか。

- A. 依存パッケージを Amazon S3 バケットに保存する。Lambda 関数が実行されるたびに、署名付き URL を介して S3 バケットからファイルをダウンロードする。
- B. 依存パッケージを Amazon EBS ボリュームに格納し、Lambda 関数から EBS ボリュームをマウントして実行時にファイルを読み込む。
- C. 依存パッケージを EC2 インスタンス上の NFS サーバーに格納し、Lambda 関数の実行時にネットワーク経由でファイルを読み込む。
- D. 依存パッケージを Amazon EFS ファイルシステムに格納し、Lambda 関数から EFS ファイルシステムをマウントして実行時にファイルを読み込む。

> [!success]- 答え：**D**
> 「Lambdaレイヤーのクォータ超え」＋「複数Lambdaから同時アクセス」＋「転送中暗号化」＋「運用オーバーヘッド最小」＝**Amazon EFS をLambdaにマウント**。EFSはNFSv4.1で複数関数から同時共有でき、転送中暗号化（TLS）に対応し、マネージドなので運用負荷が低い。
> - A：S3はオブジェクトストレージで、実行のたびに大容量MLライブラリをダウンロードするのは起動が遅く非効率。署名付きURLの発行・管理も運用オーバーヘッドになる。マウントして直接importする使い方には不向き。
> - B：EBSは**単一AZ・基本1インスタンスにアタッチ**するブロックストレージで、**Lambdaからマウントできない**。「複数Lambdaから同時アクセス」も満たせない。
> - C：自前のEC2上にNFSサーバーを構築・運用するのはパッチ・可用性・スケールを自分で見る必要があり、**運用オーバーヘッドが最大**。マネージドのEFSがあるため不適。
> - D：正解。EFSはLambdaにマウント可能（Lambda + EFS統合）、複数関数から同時共有でき、転送中暗号化（TLS）も標準対応。レイヤークォータの制約も回避できる。
> **関連ノート**：[[0202_S3とEBS・EFSの違い]] / [[0708_Lambda]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第2問（Serverless ｜ Lambdaの大容量依存ライブラリ共有ストレージ）]]

## 第3問（Management / Monitoring ｜ 設定コンプライアンスのリアルタイム検出）
ある金融系企業は、Amazon EMR を使用して大規模なデータ分析処理を実行しています。同社のセキュリティポリシーでは、すべての新しい EMR クラスターにおいて、クラスター起動時に転送中の暗号化を有効にしたセキュリティ設定を必ず使用することが義務付けられています。セキュリティチームは、ポリシーに準拠していない設定で起動されたクラスターを即座に検出し、通知を受け取れるようにしたいと考えています。

この要件を最も適切に満たすアーキテクチャはどれですか。

- A. 新しい EMR クラスターの起動イベントを CloudTrail 経由で S3 バケットに保存し、定期的に Athena を使用して転送中の暗号化が無効な起動イベントを抽出する。ポリシー違反が検出された場合は、セキュリティチームに通知するように設定する。
- B. Amazon Inspector を有効化し、転送中の暗号化が有効になっていない EMR クラスターを検出する。検出結果に基づき、ポリシー違反があればセキュリティチームへ通知するように設定する。
- C. 新しい EMR クラスターの起動イベントを Amazon Data Firehose のストリームに取り込み、Data Firehose から Lambda 関数を呼び出して各レコードの暗号化設定を検証する。ポリシー違反があればセキュリティチームへ通知するように設定する。
- D. 新しい EMR クラスターの起動イベントを検知する Amazon EventBridge ルールを設定し、そのイベントごとに Lambda 関数を起動する。Lambda 関数で対象クラスターの暗号化設定を検証し、ポリシーに違反している場合はセキュリティチームに通知するように設定する。

> [!success]- 答え：**D**
> 「起動イベントを即座に検出」はイベント駆動の合図。EMR起動イベントを EventBridge ルールで捉え、その都度 Lambda で暗号化設定を検証し、非準拠なら通知する構成がリアルタイム性・運用性ともに最適。
> - A：CloudTrail＋Athenaは定期抽出（バッチ）で、「即座に検出」という要件を満たせない。タイムラグが生じる。
> - B：Inspectorは脆弱性診断（EC2/ECR/Lambda等のCVE・到達可能性評価）サービスで、EMRクラスターの暗号化設定のコンプライアンス検証には使えない。
> - C：Data Firehoseは大量データのストリーム配信サービス。起動イベントをトリガーに検証する仕組みとして不自然で回りくどく、イベント検知の主役ではない。
> - D：正解。EventBridgeがEMR起動イベントを即キャッチしLambdaを起動、暗号化設定を検証して非準拠なら通知。イベント駆動でリアルタイム、サーバーレスで運用負荷も低い。
> **関連ノート**：[[0804_EventBridge]] / [[1003_AWS_Config]] / [[1001_CloudWatch]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第3問（Management / Monitoring ｜ 設定コンプライアンスのリアルタイム検出）]]

## 第4問（Management / Monitoring ｜ マルチアカウント・マルチリージョンの構成可視化）
ある企業では、複数の開発部門がそれぞれ異なる AWS アカウントを利用してシステムを運用しています。また、各部門は可用性や利用地域の要件に応じて、複数の AWS リージョンにリソースを展開しています。各部門では利用中の AWS リソースを個別に管理していますが、管理方法や記録内容は統一されていません。そのため、企業全体の各ワークロードの構成や、リソース間の関係性を横断的に把握できていません。この企業は、複数アカウント・複数リージョンにまたがる AWS 環境について、ワークロードを構成するリソース間の関係性も含めて可視化し、一元的に管理したいと考えています。

この要件を最も運用効率よく満たすソリューションはどれですか。

- A. AWS でのワークロード検出（Workload Discovery on AWS）を導入し、複数の AWS アカウントおよび AWS リージョンに分散したリソースを収集する。ワークロードを構成するリソース間の関係を可視化し、アーキテクチャ図を生成する。
- B. AWS Application Discovery Service を使用して、各環境のサーバー情報と依存関係を収集し、アプリケーション構成を可視化する。
- C. Amazon QuickSight を使用して、各アカウントから収集したリソース情報をダッシュボードで可視化し、ワークロード全体の構成情報を確認する。
- D. AWS X-Ray を使用してアプリケーションの動作状況と依存関係を可視化し、その情報を基にワークロードのアーキテクチャ図を作成する。

> [!success]- 答え：**A**
> 「複数アカウント・複数リージョン横断で、リソース間の関係も含めて可視化＋アーキテクチャ図を生成」は Workload Discovery on AWS（旧 AWS Perspective）の役割そのもの。
> - A：正解。AWS内に既存のリソースをアカウント・リージョン横断で収集し、関係性を可視化してアーキテクチャ図を自動生成する。要件に直結。
> - B：Application Discovery Service は「オンプレ→AWS移行」の発見・計画ツールで、オンプレのサーバー情報・依存関係を集めるもの。既存AWS環境の横断可視化が目的ではない。"Discovery"の語による混同狙い。
> - C：QuickSight はBIダッシュボード。数値データの可視化はできるが、リソース間の関係性やアーキテクチャ構成を描くものではない。
> - D：X-Ray は分散トレーシング（リクエストがどのサービスを通ったか追跡）で、アプリの動作追跡用。アカウント横断のリソース構成図を作る道具ではない。
> **関連ノート**：[[1401_マネジメント・ガバナンスサービス一覧]] / [[1501_移行と転送サービス一覧]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第4問（Management / Monitoring ｜ マルチアカウント・マルチリージョンの構成可視化）]]

## 第5問（Application Integration ｜ SQSによる疎結合とスケーラブルな移行構成）
ある企業はオンプレミスで多層構成の Web アプリケーションを運用しています。アプリケーションのフロントエンドは静的コンテンツで構成されており、コンテンツ内のアンケートフォームにデータが入力されると、リクエストがアプリケーションへ送信されます。AWS へ移行するにあたって、Web アプリケーションのデカップリング（疎結合化）を実現した上で、スケーラブルなシステムにしたいと考えています。

次のうち最も適切なソリューションはどれですか。

- A. フロントエンドに S3 の静的 Web サイトホスティングを使用する。アンケートフォームにデータが入力されると SNS トピックにメッセージを登録する。Auto Scaling に所属した EC2 インスタンスが SNS トピックのメッセージを購読してアプリケーションを実行する。SNS からの通知に応じてスケーリングする。
- B. フロントエンドに静的コンテンツ用の EC2 インスタンスを配置する。アンケートフォームの入力データから SQS キューにリクエストを投入する。Auto Scaling グループに所属したアプリケーション用の EC2 インスタンスで処理を実行し、SQS キューのメッセージ数に応じてスケーリングする。
- C. フロントエンドに S3 の静的 Web サイトホスティングを使用し、アンケートフォームの入力データを S3 バケットに保存する。Auto Scaling グループに所属した EC2 インスタンスで、S3 イベント通知をトリガーにアプリケーションを実行する。
- D. フロントエンドに S3 の静的 Web サイトホスティングを使用する。アンケートフォームの入力データを Amazon API Gateway へ送信し、SQS キューにリクエストを投入する。Auto Scaling グループに所属した EC2 インスタンスでアプリケーションを実行し、SQS キューのメッセージ数に応じてスケーリングする。

> [!success]- 答え：**D**
> 疎結合＝コンポーネント間にSQSキューを挟む、スケーラブル＝キューの未処理メッセージ数でAuto Scaling。静的コンテンツはS3、フォーム受付はAPI Gateway。各部品が役割どおりに噛み合うDが最適。
> - A：SNSは発行/購読（ファンアウト）で処理待ちをためるバッファではない。「SNS通知でスケール」はバックログ基準のスケールにならず、疎結合バッファとしてはSQSが適切。
> - B：SQSで疎結合・キュー長スケールは妥当だが、静的コンテンツをEC2で配信しており、S3で済む所にサーバーを使う無駄がある（スケーラビリティ・効率でDに劣る）。
> - C：フォームデータをS3に保存しS3イベントでEC2起動は回りくどく、キューによる疎結合・バックログ基準スケールになっていない。デカップリングの意図に合わない。
> - D：正解。S3で静的配信、API Gatewayで受け、SQSで疎結合、キュー長でスケール。最も効率的でスケーラブル。
> **関連ノート**：[[0802_SQS]] / [[0809_API_Gateway]] / [[0108_EC2とS3のWebサイトの違い]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第5問（Application Integration ｜ SQSによる疎結合とスケーラブルな移行構成）]]

## 第6問（Serverless ｜ マネージドなWebSocket構成とDynamoDBキャパシティ選定）
ある会社は、WebSocket を使用したアプリケーションを開発しています。アプリケーションへは毎秒 500 件のリクエストが発生すると予測されます。なお、データベースは Amazon DynamoDB を使用します。

AWS のマネージドサービスを使用して実装するには、どのソリューションが最も適切ですか。

- A. Auto Scaling グループを作成し、EC2 インスタンスを実行する。DynamoDB をオンデマンドモードに設定する。
- B. Amazon API Gateway と AWS Lambda を組み合わせる。DynamoDB をオンデマンドモードに設定する。
- C. Auto Scaling グループを作成し、EC2 インスタンスを実行する。DynamoDB をプロビジョンドモードに設定する。
- D. Amazon API Gateway と AWS Lambda を組み合わせる。DynamoDB をプロビジョンドモードに設定する。

> [!success]- 答え：**D**
> 「マネージドで実装」→ EC2ではなく API Gateway（WebSocket API）＋ Lambda のサーバーレス。「毎秒500件と予測できる安定負荷」→ DynamoDB はプロビジョンドモードがコスト効率的。両軸を満たすのがD。
> - A：コンピュートがEC2（自前管理寄り）で「マネージド」に反し、予測可能負荷にオンデマンドはコスト非効率。二重に外れる。
> - B：API Gateway＋Lambdaは正しいが、予測できる負荷にオンデマンドはプロビジョンドより割高。コスト効率でDに劣る。
> - C：DynamoDBプロビジョンドはOKだが、コンピュートがEC2＋Auto Scalingで「マネージドで実装」に反する。
> - D：正解。API Gateway（WebSocket）＋Lambda（フルマネージド）＋DynamoDBプロビジョンド（予測可能負荷に最適）。
> **関連ノート**：[[0303_DynamoDB]] / [[0809_API_Gateway]] / [[0708_Lambda]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第6問（Serverless ｜ マネージドなWebSocket構成とDynamoDBキャパシティ選定）]]

## 第7問（Migration / Transfer ｜ 災害復旧（DR）でのデータ統一と迅速復旧）
ある会社はオンプレミスで Web アプリケーションを運用しています。同社はデータセンターの自然災害時の影響を考慮して、AWS を利用した災害復旧（DR：Disaster Recovery）を計画しています。災害発生時に AWS のバックアップ環境でサービスを継続するために、オンプレミスで利用しているデータと AWS のバックアップデータを統一したいと考えています。なお、オンプレミスから AWS へのフェイルオーバーは Amazon Route 53 のフェイルオーバールーティングポリシーを利用します。

どのようなソリューションの組み合わせが、もっともダウンタイム（障害によりアクセスできない時間）を少なくできますか。（2つ選択）

- A. 障害発生時に AWS CloudFormation テンプレートを実行して、ALB と EC2 インスタンスを作成する。
- B. AWS Storage Gateway のボリュームゲートウェイ（ストアドボリューム）を設定する。
- C. AWS Storage Gateway のボリュームゲートウェイ（キャッシュボリューム）を設定する。
- D. AWS Direct Connect でバックアップ環境の VPC とオンプレミスをセキュアに接続する。
- E. ALB 配下の EC2 インスタンスを停止状態にしておき、障害発生時に起動する。

> [!success]- 答え：**B ＋ E**
> ダウンタイム最小化は「データが即使える」＋「コンピュートが即立ち上がる」で決まる。ストアドボリュームでオンプレ全データをAWSへバックアップしデータを統一（B）、停止状態のEC2を災害時に起動するパイロットライトで復旧を高速化（E）。事前にデータも基盤も用意しておくことが要。
> - A：災害時にCloudFormationでALB/EC2をゼロから作成するのはプロビジョニングに時間がかかり、ダウンタイムが増える。Eの「停止EC2を起動」の方が速い。
> - B：正解。ストアドボリュームはプライマリデータをオンプレに保持しつつAWSへ非同期バックアップ。「オンプレ運用＋AWSに控え」でデータを統一できる。
> - C：キャッシュボリュームはプライマリデータがAWS（S3）側。オンプレ運用＋AWSバックアップという前提と逆になり不適。
> - D：Direct Connectは安全・安定な接続だが、フェイルオーバーの復旧速度＝ダウンタイムを直接縮める手段ではない。切替はRoute 53が担う前提で、ダウンタイム最小化の決め手にならない。
> - E：正解。停止状態の構築済みEC2を災害時に起動するだけ（パイロットライト）なので、ゼロから作るより圧倒的に速く、ダウンタイムを最小化できる。
> **関連ノート**：[[1503_Storage Gateway]] / [[0404_Route53]] / [[0102_EC2購入プラン]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第7問（Migration / Transfer ｜ 災害復旧（DR）でのデータ統一と迅速復旧）]]

## 第8問（Cost Optimization ｜ S3ライフサイクルによるログ保管のコスト最適化）
あるクラウドベースのITサービス管理会社は、顧客のインフラストラクチャが生成するログファイルを S3 Standard バケットに保存して解析作業を行っています。これらのログファイルには、エラーログ、アクセスログ、パフォーマンスメトリクスが含まれ、常時行うシステム監視と問題解決に不可欠となっています。生成後30日が経過するとこれらのログファイルへのアクセスは減少しますが、ログの種類や取得時期によってアクセスにばらつきがあります。また規則により最低180日は保存する必要がありますが、それ以降は削除できます。ログファイルはサイズが大きいため、より効率的なアップロードのためにマルチパートアップロード機能を使用していますが、バージョニング機能は使用していません。ログファイルの安全な保管に留意しつつ、ストレージにかかるコストも考慮に入れ、無駄な出費を避けたいと考えています。

この会社の求める要件を満たす組み合わせは次のうちどれですか。（3つ選択）

- A. 180日後にログファイルに対してオブジェクトロックを有効にする。
- B. 有効期限切れのオブジェクト削除マーカーを削除する S3 ライフサイクルを構成する。
- C. 30日後にログファイルを S3 One Zone-IA に移動する S3 ライフサイクルを構成する。
- D. 不完全なマルチパートアップロードを削除する S3 ライフサイクルを構成する。
- E. 180日後にログファイルを削除する S3 ライフサイクルを構成する。
- F. 30日後にログファイルを S3 Intelligent-Tiering に移動する S3 ライフサイクルを構成する。

> [!success]- 答え：**D ＋ E ＋ F**
> アクセスが予測できない（ばらつく）→ Intelligent-Tiering（F）、マルチパートの断片課金を排除（D）、最低180日の後は失効させて保持コストを断つ（E）。安全性（多AZ）を保ちつつ無駄な出費を避けられる。
> - A：オブジェクトロックはWORM（改ざん防止）用でコスト最適化ではなく、バージョニングが前提。本問（バージョニング未使用）と不整合。
> - B：「期限切れ削除マーカーの削除」はバージョニングが有効なバケットでのみ意味を持つ（削除マーカーはバージョニング時に生成）。本問はバージョニング未使用のため効果がなく、要件を満たさない罠。
> - C：One Zone-IAは単一AZで耐久・安全性が低く、取り出し料金・最低保管期間もある。アクセスがばらつく・安全な保管という要件に不適。
> - D：正解。中断した不完全なマルチパートアップロードの断片を自動削除し、無駄なストレージ課金を防ぐ。
> - E：正解。規則の最低180日経過後に失効させ、不要な保持コストを断つ（無駄な出費を避ける）。
> - F：正解。アクセスが予測不能なログにはIntelligent-Tieringが最適。取り出し料金なしで自動的に階層を最適化し、多AZで安全。
>
> ※外部アプリの正解表示は **B・D・F** だが、Bはバージョニング未使用では効果がない。SAA本番の観点では **D・E・F** が妥当と判断し本ノートではこちらを正解とする。
> **関連ノート**：[[0204_S3のライフサイクル管理]] / [[0203_S3ストレージクラス]] / [[0205_S3のバージョン管理]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第8問（Cost Optimization ｜ S3ライフサイクルによるログ保管のコスト最適化）]]

## 第9問（Management / Monitoring ｜ マルチアカウントのガバナンスと継続的セキュリティ準拠）
ある企業では、複数の AWS アカウントを一元的に管理するために Organizations を導入しており、今後はサービス制御ポリシー（SCP）の適用や共通のセキュリティガードレールの整備を進めていく方針です。セキュリティチームは、すべてのアカウントが AWS の推奨するセキュリティ基準（AWS Foundational Security Best Practices）に準拠しているかどうかを継続的に確認できる仕組みを求めています。また、これらのセキュリティ基準は、新たに作成されるアカウントにも自動的に適用されることが求められており、日常的な運用負荷はできる限り抑えたいと考えています。

この要件を満たす最も適切な対応はどれですか。

- A. 組織の管理アカウントに Control Tower を導入し、Security Hub を有効化して全アカウントに適用する。Control Tower Account Factory で新規アカウントをプロビジョニングする。
- B. Organizations の統合機能を利用し、GuardDuty と Config を全アカウントに有効化し、準拠性評価を行う。新規アカウントについても組織への追加時に自動的に有効化されるように設定する。
- C. Organizations 内の特定のメンバーアカウントに Control Tower を展開し、その環境に Security Hub を有効化する。Control Tower Account Factory を用いて新規アカウントにも同様の設定を適用する。
- D. CloudFormation のスタックセットを使って Security Hub の設定をあらかじめ作成し、全アカウントに一括展開するとともに、新規アカウントについても同じスタックセットを使用してテンプレートを適用する。

> [!success]- 答え：**A**
> 「AWS Foundational Security Best Practices（FSBP）への継続準拠チェック」はSecurity Hubのセキュリティ標準、「マルチアカウントのガードレール＋新規アカウントへの自動適用＋低運用」はControl Tower（Account Factory）。導入先は組織の管理アカウント。両者の組み合わせが定番解。
> - A：正解。管理アカウントにControl Tower＋Security Hub（FSBP評価）＋Account Factoryで新規も自動適用。継続評価・自動適用・低運用の3要件を満たす。
> - B：GuardDuty（脅威検知）＋Configでは「FSBP準拠の継続評価」に直結しない。FSBPはSecurity Hubのセキュリティ標準であり、サービスの役割が異なる。
> - C：Control Towerは組織の「管理アカウント」に展開するもの。特定のメンバーアカウントに展開する構成が誤り。
> - D：CloudFormation StackSetsでも配れるが、テンプレートの管理・展開を自前で回す必要があり運用負荷が高い。Control Tower Account Factoryのマネージドな自動化に劣る。
> **関連ノート**：[[1404_Control Tower]] / [[1403_Organizations]] / [[1601_セキュリティサービス一覧]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第9問（Management / Monitoring ｜ マルチアカウントのガバナンスと継続的セキュリティ準拠）]]

## 第10問（Security / IAM ｜ マネージドDDoS防御の選定）
あるスタートアップ企業は、顧客向けWebアプリケーションをAWS上に構築しています。アプリケーションはプライベートサブネット上のEC2インスタンスで稼働し、リクエストはCloudFrontディストリビューションを経由してALBに転送されます。DNS管理にはRoute 53を使用しています。同社はセキュリティの専門知識が限られているため、DDoS攻撃を自動で検知し、必要に応じて自動緩和できるマネージド型の防御策を、最小限の運用負荷で導入したいと考えています。

最も適切な選択肢はどれですか。

- A. AWS Shield Advancedを有効化し、保護対象としてCloudFrontディストリビューション、ALB、Route 53のホストゾーンを登録する。
- B. AWS WAFを有効化し、不正リクエストやDDoS攻撃を検出して自動で防御するルールを設定したWeb ACLを、CloudFrontディストリビューションとALBに関連付ける。
- C. Amazon InspectorとAWS Systems Managerを連携し、脆弱性検出時にパッチ適用や設定変更を自動化して、DDoSを含む外部攻撃に対する防御を実現する。
- D. AWS Network Firewallを用いてVPC内の通信をフィルタリングし、外部からの不正トラフィックを自動遮断してDDoS攻撃を防止する。

> [!success]- 答え：**A**
> 「DDoS攻撃の自動検知・自動緩和」「マネージド」「最小の運用負荷」という3点はShield Advancedのためにある要件。Shield AdvancedはCloudFront・ALB・Route 53・Global Accelerator・Elastic IPを保護対象に登録でき、L3/L4の大規模DDoSを自動検知・自動緩和する。
> - A：正解。CloudFront・ALB・Route 53という今回の構成要素をそのまま保護対象に登録でき、24時間のDRT支援や課金保護も付く、DDoS特化のマネージドサービス。
> - B：WAFはL7（HTTP）の不正リクエストをルールで弾くサービスで、L3/L4の大規模DDoSを自動緩和する仕組みではない。レートベースルールで一部緩和はできるが、ルール設計を自前で行う必要があり「専門知識が限られ・最小運用」に反する。最大の罠。
> - C：Inspectorは脆弱性スキャン、Systems Managerは運用自動化で、どちらもDDoS検知・緩和のサービスではない。要件と役割が噛み合わない。
> - D：Network FirewallはVPC内の通信を検査するL3〜L7のファイアウォールで、ステートフルなトラフィック制御が用途。DDoS自動緩和を主目的とするマネージドサービスではない。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[1603_脅威検知・監査]]
> **詳しい解説**：[[502_【解説】外部サイト模擬試験2#第10問（Security / IAM ｜ マネージドDDoS防御の選定）]]
