# 【問題】類似サービス比較11（50番台の混同ペア対策）

## この問題集の目的

外部サイト模擬（**50番台**）で混同しやすかった「似たサービス／構成の使い分け」を、短い比較ドリルで固める。50番台の応用シナリオを解く**前提となる二択・三択の見極め**を、1問1論点で練習する。

## 対象レベル

SAA基礎〜標準。長いシナリオではなく「A と B、どちらを選ぶ？なぜ？」の判断練習。

## 問題数

蓄積中（10問を超えたら212へ分割）

---

## 第1問（Security / IAM ｜ Shield と WAF の使い分け）
Webアプリに対する**大規模なDDoS攻撃（大量トラフィックによるL3/L4の飽和）**を、マネージドに自動検知・自動緩和したいと考えています。
最も適したサービスはどれですか。

- A. AWS Shield（重要システムなら Shield Advanced）
- B. AWS WAF
- C. Amazon GuardDuty
- D. ネットワークACL（NACL）

> [!success]- 答え：**A**
> **DDoS（量の攻撃）＝Shield**、**SQLi/XSSなどL7の不正リクエスト＝WAF**。役割が別物。
> - A：正解。ShieldはL3/L4の大規模DDoSを自動緩和するマネージドサービス。
> - B：WAFはHTTP（L7）の中身を見て不正リクエストを弾く。DDoSの物量緩和は担当外。
> - C：GuardDutyは脅威検知（気づく）で、緩和（防ぐ）はしない。
> - D：NACLは静的なIP/ポート制御で、大規模DDoSの自動緩和には無力。
> **着眼点（SAA）**：**DDoS→Shield／SQLi・XSS・Geo・レート制限→WAF**。両者は併用が定番。「WAFでDDoS対策」は頻出ひっかけ。
> **関連ノート**：[[1601_セキュリティサービス一覧]] / [[1604_ネットワーク防御]]

## 第2問（Security / IAM ｜ リソースベース と IDベース ポリシー）
アカウントB が所有する SQS キュー（や S3 バケット）を、**別のアカウントA のプリンシパルに使わせたい**と考えています。リソースの所有者（B）側が取るべき設定はどれですか。

- A. リソース側のポリシー（リソースベースポリシー）で、アカウントA のプリンシパルを許可する
- B. アカウントB のIAMポリシー（IDベース）を変更する
- C. アカウントA にアカウントB のルート認証情報を渡す
- D. リソースをパブリック公開する

> [!success]- 答え：**A**
> 「**モノ側に貼る許可証**（リソースベース）」と「**人側に持たせる許可証**（IDベース）」の違い。他アカウントに使わせるのはモノ側。
> - A：正解。SQS/S3/KMS等はリソースポリシーを持ち、`Principal`に相手アカウントを指定して許可できる。
> - B：IDベースは「自アカウント内のプリンシパルが何をできるか」で、他アカウントのリソースに権限は与えられない。
> - C：ルート認証情報の共有は重大なアンチパターン。
> - D：パブリック公開は過剰かつ危険。
> **着眼点（SAA）**：**クロスアカウントで自分のリソースを相手に使わせる→リソースベースポリシー**（S3/SQS/KMS/Lambda/SNS/ECR等）。IDベースとの取り違えが頻出。
> **関連ノート**：[[0601_IAM基礎]] / [[0801_アプリケーション統合サービス一覧]]

## 第3問（Database ｜ マルチAZインスタンス / クラスター / リードレプリカ）
RDS で、**障害時の自動フェイルオーバー（高可用性）**に加えて、**レポート用の読み取りを本番の書き込みと分離**したいと考えています。最も適した構成はどれですか。

- A. マルチAZ DB クラスター（ライター＋リーダー2台）
- B. マルチAZ DB インスタンス（本番＋スタンバイ）
- C. リードレプリカのみ（単一インスタンス＋レプリカ）
- D. 単一AZの単一インスタンス

> [!success]- 答え：**A**
> 3つは似て非なるもの。**インスタンス＝スタンバイは読めない（高可用性のみ）／レプリカ＝読めるが自動FOなし／クラスター＝両取り**。
> - A：正解。リーダーEPで読み取り分離＋ライター障害時に自動フェイルオーバー。
> - B：スタンバイは待機専用で読み取りに使えない（高可用性のみ）。
> - C：読み取り分離はできるが、レプリカは自動フェイルオーバー対象外で高可用性が別途必要。
> - D：冗長性なし。
> **着眼点（SAA）**：「**自動フェイルオーバー＋読み取りスケール**」＝マルチAZ DB**クラスター**。「マルチAZ」の語だけで飛びつくとインスタンス（読めない）を選ぶ罠。
> **関連ノート**：[[0301_RDS]] / [[0501_高可用性とスケーラビリティ]]

## 第4問（Networking / CDN / LB ｜ NATインスタンス と NATゲートウェイ）
プライベートサブネットからの外向き通信を、**帯域が自動でスケールし、運用負荷が小さく、高可用**な構成にしたいと考えています。最も適した対応はどれですか。

- A. 各AZに NATゲートウェイを1つずつ配置する
- B. NATインスタンス2台を同一AZに配置する
- C. NATインスタンスをAuto Scalingグループに入れる
- D. インターネットゲートウェイを直接使う

> [!success]- 答え：**A**
> **NATインスタンス＝自前EC2（帯域・パッチ・冗長を自己管理）**、**NATゲートウェイ＝マネージド（自動スケール・低運用）**。高可用性は**AZごとに1つずつ**が鉄則。
> - A：正解。各AZにNATゲートウェイを置き、各サブネットは同一AZのGWへルーティング。
> - B：同一AZに2台では、そのAZ障害で全滅（高可用性にならない）。
> - C：NATインスタンスの継続運用で管理負荷が残る。
> - D：プライベートサブネットにIGWを直接向けるとプライベートでなくなる。
> **着眼点（SAA）**：「**NATインスタンス＋帯域/高可用/低運用が課題**」→ NATゲートウェイへ移行。NATゲートウェイの冗長化は**各AZに1つずつ**。
> **関連ノート**：[[0406_VPC応用機能]] / [[0405_ネットワークとコンテンツ配信]]

## 第5問（Networking / CDN / LB ｜ CloudFront と Global Accelerator）
**UDP通信**を使うグローバル向けアプリを、**固定IPアドレス**でアクセスでき、世界中のユーザーから**最寄りエッジ経由で低遅延**にしたいと考えています。フロントに置くべきサービスはどれですか。

- A. AWS Global Accelerator
- B. Amazon CloudFront
- C. Amazon Route 53 のみ
- D. Application Load Balancer（ALB）

> [!success]- 答え：**A**
> **CloudFront＝HTTPコンテンツのキャッシュ配信**、**Global Accelerator＝TCP/UDPアプリの高速化＋静的IP**。プロトコルと固定IPで見分ける。
> - A：正解。TCP/UDP対応・静的Anycast IPを2つ提供・エッジ経由で低遅延。
> - B：CloudFrontはHTTP/HTTPS用でUDP不可・固定IPなし。
> - C：Route 53はDNSで、通信の高速化・固定IPの役割ではない。
> - D：ALBはHTTP用でUDP非対応。
> **着眼点（SAA）**：「**UDP／非HTTP＋固定IP＋世界中で低遅延**」→ Global Accelerator。「**画像・動画・静的サイトのキャッシュ配信**」→ CloudFront。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0503_ALB・NLB・GWLBの違い]]

## 第6問（Management / Monitoring ｜ ARC と FIS）
本番で**リージョン障害が起きたときに確実に切り替えたい**（切替先の準備も保証したい）。一方で、平常時に**わざと障害を起こして耐障害性をテスト**するのは別サービスです。前者（本番の確実な切替）に使うのはどれですか。

- A. Amazon Application Recovery Controller（ARC）
- B. AWS Fault Injection Service（FIS）
- C. AWS CloudFormation
- D. AWS Config

> [!success]- 答え：**A**
> **ARC＝本番の確実なフェイルオーバー＋切替先の準備保証**、**FIS＝障害をわざと起こすテスト（カオスエンジニアリング）**。目的が正反対。
> - A：正解。Readiness Checkとルーティングコントロールで確実に切り替える。
> - B：FISは障害注入で耐性を「試す」道具。本番切替の制御ではない。
> - C：CloudFormationはIaC（構築）で、切替制御ではない。
> - D：Configは構成評価で無関係。
> **着眼点（SAA）**：「**確実な本番フェイルオーバー／Readiness**」→ ARC。「**障害シミュレート／耐性テスト／カオス**」→ FIS。
> **関連ノート**：[[1406_Application_Recovery_Controller]] / [[1407_Fault_Injection_Service]]

## 第7問（Application Integration ｜ API Gatewayキャッシュ と 使用量プラン）
API Gateway の背後のバックエンドに、**同じデータへの重複リクエストが集中して過負荷**です。ETLの効率を保ちつつ、負荷を根本的に減らしたいと考えています。最も適した対応はどれですか。

- A. API Gateway のキャッシュを有効化する
- B. 使用量プラン（スロットル／クォータ）でリクエストを制限する
- C. WebSocket API に切り替える
- D. リージョンを追加してデプロイする

> [!success]- 答え：**A**
> **キャッシュ＝重複を手前で消す（到達数を減らす）**、**使用量プラン＝流量を絞る（正当なリクエストも制限）**。過負荷の原因が「重複」ならキャッシュ。
> - A：正解。同一キーの繰り返しをバックエンド未到達で返し、負荷を根本削減。
> - B：スロットルは絞るだけで、重複取得は減らず効率も落ちる。
> - C：WebSocketは双方向リアルタイム用で無関係。
> - D：リージョン追加は負荷の原因（重複）を解決しない。
> **着眼点（SAA）**：「**同じデータの繰り返しで過負荷**」→ キャッシュ。「使用量プランで保護」に見せる**流量制限の罠**に注意。
> **関連ノート**：[[0809_API_Gateway]] / [[0304_ElastiCache]]

## 第8問（Security / IAM ｜ Cognito と IAM Identity Center）
自社の**従業員**が、複数のAWSアカウントの**マネジメントコンソール**に、社内IDで**シングルサインオン**したいと考えています。適したサービスはどれですか。（対して、Webアプリの一般エンドユーザーのサインアップ／サインインは別サービス）

- A. AWS IAM Identity Center（旧 AWS SSO）
- B. Amazon Cognito
- C. AWS Directory Service のみ
- D. アカウントごとにIAMユーザーを量産する

> [!success]- 答え：**A**
> **IAM Identity Center＝従業員のAWSアカウント/コンソールへのSSO**、**Cognito＝アプリ（Web/モバイル）のエンドユーザー認証**。誰がどこにログインするかで見分ける。
> - A：正解。Organizations配下の全アカウントにSSO、外部IdP/AD連携も可能。
> - B：Cognitoはアプリのエンドユーザー（顧客）向けで、従業員のコンソールSSOではない。
> - C：Directory Serviceは AD 連携の土台だが、SSOの受付はIAM Identity Center。
> - D：IAMユーザー量産は一元管理と真逆。
> **着眼点（SAA）**：**従業員のAWSログイン＝IAM Identity Center／アプリの利用者認証＝Cognito**。この取り違えが頻出。
> **関連ノート**：[[1602_アイデンティティ系]] / [[1403_Organizations]]

## 第9問（Migration / Transfer ｜ DR戦略の使い分け（RTOで選ぶ））
災害復旧（DR）で、**RTO（復旧目標時間）が数分**という要件があります。セカンダリリージョンに**縮小版の本番を常時稼働**させておき、障害時に即座に切り替えたいと考えています。この戦略の名前はどれですか。

- A. ウォームスタンバイ
- B. バックアップ＆リストア
- C. パイロットライト
- D. マルチサイト アクティブ/アクティブ

> [!success]- 答え：**A**
> DRは**RTOが短いほど“温めておく”必要があり高コスト**：バックアップ&リストア（遅・安）→ パイロットライト → **ウォームスタンバイ** → マルチサイト（速・高）。
> - A：正解。縮小版を常時稼働＝ウォームスタンバイ。切替後にスケールアウト。
> - B：バックアップから復元＝最も時間がかかる（RTO長）。
> - C：パイロットライトは核だけ起動、サーバーは障害時に起動＝ウォームより遅い。
> - D：マルチサイトは常時フル稼働で最速だが最も高コスト（「縮小版」ではない）。
> **着眼点（SAA）**：「**数分以内・常時稼働の縮小版**」→ ウォームスタンバイ。「起動/復元してから」系はRTOを食う。RTO/RPOの要件で戦略を選ぶ。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[0501_高可用性とスケーラビリティ]]
