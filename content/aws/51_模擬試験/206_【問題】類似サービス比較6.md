# 【問題】類似サービス比較6（公式サンプル風・全10問）

## この問題集の目的

役割が似ていて混同しやすいサービスの「使い分け」を要件文で判断する。201〜205で扱っていない比較（コンピュート選択・S3ストレージクラス・Redis vs Memcached・セキュリティグループ vs NACL・Secrets Manager vs Parameter Store・CloudWatch/CloudTrail/Config・SQS vs Kinesis・DMS vs DataSync・Step Functions Standard vs Express・SageMaker vs 事前学習済みAIなど）を中心に、13分野を横断する。

## 対象レベル

SAA。サービス名は知っている前提で、要件に最も合う1つを選ぶ。

## 問題数

10問

---

## 第1問（Compute ｜ EC2 vs Lambda vs Fargate）
数分で完了する不定期なイベント処理を、サーバーの起動・パッチ・スケーリングを一切管理せずに実行し、実行時間にのみ課金されるようにしたい。コードは関数単位で十分小さい。最も適切なサービスはどれか。

- A. AWS Lambda
- B. Amazon EC2（オンデマンドで常時起動）
- C. AWS Fargate（常時起動タスク）
- D. Amazon Lightsail

> [!success]- 答え：**A**
> Lambdaはイベント駆動の短時間処理をサーバー管理なしで実行し、実行時間に応じて課金される。
> - A：正解。小さな不定期処理を最小運用・従量課金で動かせる。
> - B：常時起動はアイドル時もコストがかかり、サーバー管理も必要。
> - C：常時起動タスクはアイドルコストが発生し、不定期処理には過剰。
> - D：簡易VPS向けで、イベント駆動の従量実行用途ではない。
> **関連ノート**：[[0701_コンピューティングサービス一覧]] / [[0702_Lambda・ECS・RDS・Beanstalk比較]]

## 第2問（Storage ｜ S3ストレージクラスの選択）
アクセス頻度が月単位で予測できないオブジェクト群を、取り出し遅延なく即時アクセスできる状態を保ちつつ、アクセスパターンに応じて自動で最適なクラスへ移動させてコストを最小化したい。最も適切なストレージクラスはどれか。

- A. S3 Intelligent-Tiering
- B. S3 標準（Standard）固定
- C. S3 Glacier Flexible Retrieval
- D. S3 One Zone-IA

> [!success]- 答え：**A**
> S3 Intelligent-Tieringはアクセスパターンを監視し、即時アクセスを保ったまま自動で適切な階層へ移動してコストを最適化する。
> - A：正解。アクセス予測が難しいデータの自動コスト最適化に最適。
> - B：固定では低頻度データに割高なコストを払い続ける。
> - C：取り出しに遅延があり、即時アクセス要件に反する。
> - D：単一AZで可用性が低く、アクセス頻度が既知の場合向け。
> **関連ノート**：[[0203_S3ストレージクラス]]

## 第3問（Database ｜ ElastiCache Redis vs Memcached）
セッションやリーダーボードを、永続化・レプリケーション・自動フェイルオーバー、さらに並べ替え集合やPub/Subといった高度なデータ構造とともにインメモリで扱いたい。最も適切なサービスはどれか。

- A. Amazon ElastiCache for Redis
- B. Amazon ElastiCache for Memcached
- C. Amazon DynamoDB（オンデマンド）
- D. Amazon RDS for MySQL

> [!success]- 答え：**A**
> ElastiCache for Redisは永続化・レプリケーション・フェイルオーバーや高度なデータ構造（ソート済み集合・Pub/Sub等）に対応する。
> - A：正解。高可用性と豊富なデータ構造を要するインメモリ用途に最適。
> - B：シンプルでマルチスレッドだが、永続化・レプリケーション・高度な構造は持たない。
> - C：低レイテンシのNoSQLだが、インメモリのソート集合やPub/Subの要件には合わない。
> - D：ディスクベースのRDBで、インメモリキャッシュ用途ではない。
> **関連ノート**：[[0304_ElastiCache]]

## 第4問（Networking / CDN / LB ｜ セキュリティグループ vs NACL）
サブネット全体に対して、特定のIP範囲からのインバウンド通信を明示的に「拒否」するルールを、ステートレスに適用したい。インスタンス単位ではなくサブネット境界で制御したい。最も適切な機能はどれか。

- A. ネットワークACL（NACL）
- B. セキュリティグループ
- C. IAMポリシー
- D. ルートテーブル

> [!success]- 答え：**A**
> NACLはサブネット境界でステートレスに動作し、許可ルールに加えて明示的な拒否ルールを設定できる。
> - A：正解。サブネット単位・ステートレス・明示的拒否の要件に合う。
> - B：インスタンス単位・ステートフルで、許可ルールのみで明示的拒否は設定できない。
> - C：AWS APIの権限制御で、ネットワークトラフィックの制御ではない。
> - D：宛先へのルーティング定義で、トラフィックの許可・拒否制御ではない。
> **関連ノート**：[[0408_セキュリティグループとNACL]]

## 第5問（Security / IAM ｜ Secrets Manager vs Parameter Store）
データベースの認証情報を安全に保管したうえで、定期的な自動ローテーションをマネージドに行いたい。最も適切なサービスはどれか。

- A. AWS Secrets Manager
- B. AWS Systems Manager Parameter Store（standard）
- C. 環境変数に平文で保存する
- D. Amazon S3 にテキストファイルとして保存する

> [!success]- 答え：**A**
> Secrets Managerはシークレットの暗号化保管に加え、認証情報の自動ローテーションをマネージドに提供する。
> - A：正解。保管とローテーションの自動化を一体で満たす。
> - B：暗号化保管はできるが、マネージドな自動ローテーション機能は標準で備えない。
> - C：平文保存は重大なセキュリティリスクで論外。
> - D：アクセス制御や暗号化・ローテーションの仕組みを自前で作る必要があり不適切。
> **関連ノート**：[[1405_Systems Manager]] / [[1605_データ保護・暗号化]]

## 第6問（Management / Monitoring ｜ CloudWatch vs CloudTrail vs Config）
「誰が・いつ・どのAWS APIを呼び出したか」を記録し、不正操作の調査やコンプライアンス監査に使いたい。最も適切なサービスはどれか。

- A. AWS CloudTrail
- B. Amazon CloudWatch
- C. AWS Config
- D. AWS Trusted Advisor

> [!success]- 答え：**A**
> CloudTrailはアカウント内のAPI呼び出しを証跡として記録し、操作の監査・調査に用いる。
> - A：正解。API操作の「誰が・いつ・何を」を記録する用途に最適。
> - B：メトリクス・ログ・アラームによる性能監視が主目的で、API監査が主眼ではない。
> - C：リソース構成の変更履歴と準拠評価が主目的で、API呼び出しの証跡ではない。
> - D：コスト・セキュリティ等のベストプラクティス助言で、API監査ログではない。
> **関連ノート**：[[1002_CloudTrail]] / [[1001_CloudWatch]] / [[1003_AWS_Config]]

## 第7問（Application Integration ｜ SQS vs Kinesis Data Streams）
複数のコンシューマーが同じイベントストリームを、順序を保ったまま独立して読み取り、一定期間さかのぼって再処理できるようにしたい。シャードでスループットをスケールさせたい。最も適切なサービスはどれか。

- A. Amazon Kinesis Data Streams
- B. Amazon SQS 標準キュー
- C. Amazon SQS FIFOキュー
- D. Amazon SNS（通知のみ）

> [!success]- 答え：**A**
> Kinesis Data Streamsはレコードを一定期間保持し、複数コンシューマーが順序を保って独立に読み取り・再処理でき、シャードでスケールする。
> - A：正解。複数読み取り・再処理・シャードスケールの要件に合う。
> - B：メッセージは1コンシューマーが受信後に削除され、さかのぼり再処理はできない。
> - C：順序保証はあるが、同一メッセージの複数コンシューマー独立読み取り・再処理の用途ではない。
> - D：通知のファンアウト向けで、保持・再処理可能なストリームではない。
> **関連ノート**：[[0805_Kinesis]] / [[0802_SQS]] / [[1103_Amazon Kinesis]]

## 第8問（Migration / Transfer ｜ DMS vs DataSync）
オンプレミスのOracleデータベースを、最小限のダウンタイムでAWSのAuroraへ継続的にレプリケートしながら移行したい。異なるDBエンジンへのスキーマ変換も視野に入れている。最も適切なサービスはどれか。

- A. AWS Database Migration Service（DMS）
- B. AWS DataSync
- C. AWS Storage Gateway（ファイルゲートウェイ）
- D. Amazon S3 Transfer Acceleration

> [!success]- 答え：**A**
> DMSはデータベースを継続レプリケーションして最小ダウンタイムで移行でき、SCTと併用すれば異種エンジンへのスキーマ変換も行える。
> - A：正解。異種DB移行と継続レプリケーションの要件に最適。
> - B：ファイル・オブジェクトの転送向けで、DBの論理レプリケーションは行わない。
> - C：オンプレとAWS間のハイブリッドストレージ接続で、DB移行ツールではない。
> - D：S3アップロードの高速化機能で、DB移行とは無関係。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[1502_DataSync]]

## 第9問（Serverless ｜ Step Functions Standard vs Express）
毎秒大量に発生するイベント駆動の短時間ワークフローを、高スループットかつ低コストで処理したい。1回の実行は数秒で完結し、実行履歴の完全な可視化よりも処理量とコスト効率を優先する。最も適切なワークフロー種別はどれか。

- A. Step Functions Express ワークフロー
- B. Step Functions Standard ワークフロー
- C. 1つのLambda関数にすべての処理を詰め込む
- D. Amazon SWF（Simple Workflow Service）

> [!success]- 答え：**A**
> Express ワークフローは大量・短時間のイベント処理向けに最適化され、高スループットと低コストを実現する。
> - A：正解。毎秒大量の短時間ワークフローをコスト効率よく処理できる。
> - B：長時間・厳密一回実行・完全な実行履歴向けで、超高頻度の短時間処理にはコスト効率で劣る。
> - C：単一関数への詰め込みは状態管理や可視性・再試行制御を失い、ワークフローの利点を活かせない。
> - D：旧世代のワークフローで、新規設計ではStep Functionsが推奨される。
> **関連ノート**：[[0807_Step_Functions]]

## 第10問（Machine Learning ｜ SageMaker vs 事前学習済みAIサービス）
自社固有のデータで独自の機械学習モデルを構築・学習・チューニングし、本番エンドポイントとしてデプロイ・運用したい。アルゴリズムやハイパーパラメータも自分で制御したい。最も適切なサービスはどれか。

- A. Amazon SageMaker
- B. Amazon Rekognition
- C. Amazon Comprehend
- D. Amazon Polly

> [!success]- 答え：**A**
> SageMakerはモデルの構築・学習・チューニング・デプロイまでを一貫して行え、アルゴリズムやハイパーパラメータを制御できる。
> - A：正解。独自データでのカスタムモデル開発・運用に最適。
> - B：事前学習済みの画像・動画解析API で、独自モデルの学習・制御はできない。
> - C：事前学習済みのテキスト分析APIで、カスタムモデルの構築基盤ではない。
> - D：事前学習済みの音声合成APIで、モデルの学習・チューニングは行わない。
> **関連ノート**：[[1301_AWS機械学習サービス一覧]]
