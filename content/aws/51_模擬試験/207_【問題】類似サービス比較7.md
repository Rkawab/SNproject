# 【問題】類似サービス比較7（公式サンプル風・全10問）

## この問題集の目的

役割が似ていて混同しやすいサービス/オプションの「使い分け」を要件文で判断する。201〜206で扱っていない比較（プレイスメントグループの種類・EBSボリュームタイプ・Multi-AZ vs リードレプリカ・VPN vs Direct Connect・WAF vs Shield vs SG・API Gatewayの種類・Amazon MQ vs SQS/SNS・Savings Plansの種類・Systems Managerの機能・DataSync vs Transfer Familyなど）を中心に、13分野を横断する。

## 対象レベル

SAA。サービス名は知っている前提で、要件に最も合う1つを選ぶ。

## 問題数

10問

---

## 第1問（Compute ｜ プレイスメントグループの種類）
HPC計算ノード群を同一AZ内に密集配置し、ノード間で最大のネットワークスループットと最小のレイテンシーを得たい。最も適切なプレイスメントグループはどれか。

- A. クラスタープレイスメントグループ
- B. スプレッドプレイスメントグループ
- C. パーティションプレイスメントグループ
- D. Elastic Fabric Adapter（EFA）を有効化する（プレイスメントグループは指定しない）

> [!success]- 答え：**A**
> クラスタープレイスメントグループは単一AZ内にインスタンスを密集させ、低レイテンシー・高スループットの相互通信に最適化する。
> - A：正解。密集配置でノード間通信の遅延を最小化できる。
> - B：少数の重要インスタンスを別ハードに分散させ可用性を高める用途で、密集低遅延とは逆。
> - C：分散配置で大規模分散処理の障害分離向け。最小レイテンシーの密集配置ではない。
> - D：EFAはHPC向けの高速ネットワークインターフェースだが、物理的な密集配置を保証するものではなく、低レイテンシーにはクラスタープレイスメントグループとの併用が前提。
> **関連ノート**：[[0106_プレイスメントグループ]]

## 第2問（Storage ｜ EBSボリュームタイプの選択）
数百GBの大容量ログを順次（シーケンシャル）に書き込み・読み出すバッチ処理で、スループット重視かつコストを抑えたい。ランダムIOPS性能は重要でない。最も適切なEBSボリュームタイプはどれか。

- A. プロビジョンドIOPS SSD（io2）
- B. 汎用SSD（gp3）
- C. スループット最適化HDD（st1）
- D. Cold HDD（sc1）

> [!success]- 答え：**C**
> st1は大容量・シーケンシャル・スループット重視のワークロード向けに最適化された低コストのHDD。
> - A：高IOPS・低レイテンシーが要るミッションクリティカルDB向けで、本要件にはコストが過剰。
> - B：汎用バランス型で悪くないが、大容量シーケンシャルのスループット単価ではHDDに劣る。
> - C：正解。スループット重視・コスト抑制のシーケンシャル処理に合う。
> - D：最も低コストだが低頻度アクセス向けで、定常的なバッチ処理のスループットには不足しがち。
> **関連ノート**：[[0109_EBSボリュームタイプ]] / [[0103_EC2のストレージ]]

## 第3問（Database ｜ Multi-AZ vs リードレプリカ）
RDSで、AZ障害時に自動フェイルオーバーして可用性を確保したい。読み取り負荷の分散が目的ではなく、あくまで耐障害性が目的である。最も適切な構成はどれか。

- A. RDS マルチAZ配置
- B. RDS リードレプリカを複数作成する
- C. 単一AZのインスタンスサイズを大きくする
- D. リードレプリカを別リージョンに作成する

> [!success]- 答え：**A**
> マルチAZ配置は別AZの同期スタンバイへ自動フェイルオーバーし、AZ障害時の可用性を確保する。
> - A：正解。自動フェイルオーバーによる耐障害性の確保に最適。
> - B：読み取りスケール向けで、自動フェイルオーバーによる可用性確保が主目的ではない。
> - C：垂直スケールは性能向上であり、AZ障害への耐障害性はもたらさない。
> - D：DRや遠隔読み取り向けで、同一システムの自動フェイルオーバーとは目的が異なる。
> **関連ノート**：[[0301_RDS]]

## 第4問（Networking / CDN / LB ｜ VPN vs Direct Connect）
オンプレミスとAWS間に、帯域が安定し一貫した低レイテンシーの接続を確立し、大容量データを継続的にやり取りしたい。インターネットを経由しない物理専用線がよい。最も適切なサービスはどれか。

- A. AWS Site-to-Site VPN
- B. AWS Direct Connect
- C. AWS Client VPN
- D. VPCピアリング

> [!success]- 答え：**B**
> Direct Connectは専用線による一貫した帯域・低レイテンシーの接続を提供し、大容量の継続転送に適する。
> - A：インターネット経由のIPsec VPNで、手早いが帯域・レイテンシーは経路に依存し一貫性に欠ける。
> - B：正解。専用線で安定帯域・低レイテンシーを実現し、要件に合致する。
> - C：個々のユーザー端末からVPCへのリモートアクセス向けで、拠点間専用接続ではない。
> - D：AWS内のVPC同士の接続で、オンプレとの接続手段ではない。
> **関連ノート**：[[0405_ネットワークとコンテンツ配信]] / [[0406_VPC応用機能]]

## 第5問（Security / IAM ｜ 防御層の使い分け）
Webアプリに対するSQLインジェクションやクロスサイトスクリプティングなど、アプリ層（L7）の不正なHTTPリクエストをルールで検査・ブロックしたい。最も適切なサービスはどれか。

- A. AWS Shield Standard
- B. AWS WAF
- C. セキュリティグループ
- D. ネットワークACL

> [!success]- 答え：**B**
> WAFはL7のHTTPリクエストを検査し、SQLi/XSSなどをルールでブロックするWebアプリケーションファイアウォール。
> - A：主にL3/L4のDDoS緩和向けで、L7のSQLi/XSSをルールで止めるものではない。
> - B：正解。アプリ層の不正リクエストの検査・遮断に最適。
> - C：ポート/プロトコル/IP単位のステートフル制御で、HTTPペイロードの検査はできない。
> - D：サブネット境界のIP/ポート制御で、L7アプリ攻撃の検査はできない。
> **関連ノート**：[[1604_ネットワーク防御]] / [[0408_セキュリティグループとNACL]]

## 第6問（Serverless ｜ API Gatewayの種類）
Lambdaバックエンドへの単純なリクエスト/レスポンス型のHTTP APIを、最小コスト・低レイテンシーで公開したい。詳細なリクエスト検証や使用量プランなどの高度な機能は不要。最も適切なものはどれか。

- A. API Gateway HTTP API
- B. API Gateway REST API
- C. API Gateway WebSocket API
- D. Application Load BalancerにLambdaをターゲットとして登録し、HTTPリクエストを転送する

> [!success]- 答え：**A**
> HTTP APIは軽量・低コスト・低レイテンシーで、単純なHTTPプロキシ用途に最適化されている。
> - A：正解。高度な機能が不要な単純APIを最小コストで公開できる。
> - B：詳細なリクエスト検証・使用量プラン等を備えるが、その分コストは高めで単純用途には過剰。
> - C：双方向の常時接続（チャット等）向けで、単純なリクエスト/レスポンスには不要。
> - D：L7分散はできるが、APIキーやスロットリング等のAPI管理機能は持たない。
> **関連ノート**：[[0809_API_Gateway]] / [[0708_Lambda]]

## 第7問（Application Integration ｜ Amazon MQ vs SQS/SNS）
新規開発のクラウドネイティブなアプリで、ほぼ無制限にスケールするシンプルなキューイングを、運用負荷ゼロ・従量課金で使いたい。既存メッセージングプロトコルへの互換は不要。最も適切なサービスはどれか。

- A. Amazon MQ
- B. Amazon SQS
- C. Amazon SNS
- D. Amazon Kinesis Data Streams

> [!success]- 答え：**B**
> SQSはフルマネージドでほぼ無制限にスケールし、運用負荷ゼロ・従量課金のキューイングを提供する。
> - A：既存のAMQP/MQTT等プロトコル互換が要るときに有効だが、互換不要の新規開発ではスケール・運用面でSQSが有利。
> - B：正解。クラウドネイティブな新規開発のシンプルキューに最適。
> - C：1対多のプッシュ通知（Pub/Sub）向けで、コンシューマーが自分のペースで取り出すシンプルなキューイングとは異なる。
> - D：ストリーミングデータの取り込み・順序処理向けで、運用ゼロのシンプルなキューとしてはオーバースペック。
> **関連ノート**：[[0806_Amazon_MQ]] / [[0802_SQS]] / [[0803_SNS]]

## 第8問（Cost Optimization ｜ Savings Plansの種類）
今後1年、特定のインスタンスファミリーを特定リージョンで使い続けることが確実で、その範囲で最大限の割引率を得たい。Fargate・Lambdaや他ファミリーへの柔軟な適用は不要。最も適切な購入方法はどれか。

- A. EC2 Instance Savings Plans
- B. Compute Savings Plans
- C. オンデマンド
- D. スポットインスタンス

> [!success]- 答え：**A**
> EC2 Instance Savings Plansはファミリー・リージョンを固定する代わりに、Compute型より高い割引率を得られる。
> - A：正解。柔軟性を捨てる分、対象範囲で最大の割引を得られる。
> - B：ファミリーやFargate/Lambdaにも柔軟に適用できる分、割引率はEC2 Instance型より低い。
> - C：コミットしないため割引はない。
> - D：中断され得るため、確実な継続利用に対する割引コミット手段ではない。
> **関連ノート**：[[1701_コスト管理サービス一覧]] / [[0102_EC2購入プラン]]

## 第9問（Management / Monitoring ｜ Systems Managerの機能）
多数のEC2/オンプレサーバーへ、SSHやRDPの受信ポートを開けず・踏み台も用意せずに、監査ログ付きのシェルアクセスを行いたい。最も適切な機能はどれか。

- A. Session Manager
- B. Run Command
- C. Patch Manager
- D. Parameter Store

> [!success]- 答え：**A**
> Session Managerは受信ポートを開けず踏み台なしで、監査可能なシェルセッションを確立できる。
> - A：正解。ポート開放・踏み台不要で、操作ログも残せる安全なアクセス手段。
> - B：多数のサーバーへコマンドを一括実行する機能で、対話的シェルアクセスが主目的ではない。
> - C：OSパッチ適用の自動化機能で、シェルアクセス手段ではない。
> - D：設定値/シークレットの保管機能で、サーバーへのアクセス手段ではない。
> **関連ノート**：[[1405_Systems Manager]]

## 第10問（Migration / Transfer ｜ DataSync vs Transfer Family）
外部の取引先が、既存のSFTPクライアントを使って定期的にファイルをS3へアップロードしてくる。こちらはマネージドなSFTPエンドポイントを提供し、サーバーの運用はしたくない。最も適切なサービスはどれか。

- A. AWS Transfer Family（SFTP）
- B. AWS DataSync
- C. AWS Database Migration Service（DMS）
- D. AWS Snowball Edge

> [!success]- 答え：**A**
> Transfer FamilyはマネージドなSFTP/FTPS/FTPエンドポイントを提供し、外部クライアントからのS3連携を運用レスで実現する。
> - A：正解。既存のSFTPクライアントをそのまま使わせつつ、サーバー運用を不要にできる。
> - B：AWSと自社ストレージ間の大容量同期向けで、外部のSFTPクライアントの受け口ではない。
> - C：データベースの移行・レプリケーション向けで、ファイル受信用途ではない。
> - D：物理アプライアンスでのオフライン一括移送で、定期的なSFTP受信には不向き。
> **関連ノート**：[[1501_移行と転送サービス一覧]] / [[1502_DataSync]]
