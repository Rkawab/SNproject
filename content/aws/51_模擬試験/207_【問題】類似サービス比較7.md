# 【問題】類似サービス比較7（アプリケーション統合編・全10問）

## この問題集の目的

第2版（2026-07-14全面新作）。SQS/SNS/EventBridge/MQ/Step Functions/API系の「つなぎ方」の使い分けを要件文で判断する。誤答はすべて実在の構成で、**別の要件なら正解になるもの**だけを置く。

## 対象レベル

SAA。疎結合パターン（キュー・Pub/Sub・イベントバス・ワークフロー）の選定軸を問う。

## 問題数

10問

---

## 第1問（Application Integration ｜ イベントのルーティング）
外部SaaS（Zendesk・Datadog等）と自社アプリのイベントを1か所で受け、**イベント内容（属性）に基づくルール**で、Lambda・Step Functions・SQSなど異なるターゲットへ振り分けたい。最も適切なサービスはどれか。

- A. Amazon SNS
- B. Amazon SQS
- C. Amazon EventBridge
- D. Amazon Kinesis Data Streams

> [!success]- 答え：**C**
> EventBridgeはSaaSパートナー統合を持つイベントバスで、イベントの中身をルールで評価して多彩なターゲットへルーティングする。「SaaS統合 × 内容ベースのルール」が決め手。
> - A：一番きわどい誤答。メッセージ属性によるフィルタは可能なので、**自社システム内の単純なファンアウト**なら正解になる。SaaSからの直接受信やターゲットの多彩さでEventBridgeに劣る。
> - B：1対1で処理を平準化する**バッファ**。振り分けの頭脳ではなく、ルーティング先として使う部品。
> - C：正解。
> - D：ストリームの**分析・リプレイ**（大量データの時系列処理）なら正解になる。イベントルーティングの用途ではない。
> **関連ノート**：[[0804_EventBridge]] / [[0803_SNS]]

## 第2問（Application Integration ｜ ファンアウト）
注文確定イベントを、在庫・請求・配送の3システムがそれぞれ**自分のペースで確実に**処理したい。あるシステムが一時停止しても他に影響せず、復旧後に未処理分を処理できる必要がある。最も適切な構成はどれか。

- A. SNSトピックから3つのSQSキューへファンアウトし、各システムが自分のキューを処理する
- B. SNSトピックから3システムへ直接HTTP配信する
- C. 1本のSQSキューを3システムでポーリングする
- D. EventBridgeから3システムのAPIを直接呼び出す

> [!success]- 答え：**A**
> SNS＋SQSのファンアウトは「同報」と「バッファ」の合わせ技。各キューが独立の受信箱になるため、止まっても他に波及せず、復旧後に積み残しを消化できる。
> - A：正解。疎結合の最重要パターン。
> - B：受信側が常時稼働で、多少の取りこぼしを許容できる通知なら成立する。停止中の配信は再試行が尽きれば失われ、「復旧後に未処理分」を満たせない。
> - C：定番のひっかけ。SQSの1メッセージは**1つのコンシューマー**しか受け取れないため、3システム全員には届かない（早い者勝ちになる）。同種ワーカーの負荷分散なら正解になる。
> - D：ルーティングとしては現代的だが、API直接呼び出しでは受信側停止時のバッファがない。ターゲットをSQSにすれば実質Aと同じ構成になる。
> **関連ノート**：[[0803_SNS]] / [[0802_SQS]]

## 第3問（Application Integration ｜ キューの種類）
IoTデバイスからのログを毎秒数十万件キューイングする。**順序は問わず**、稀な重複はコンシューマー側の冪等処理で吸収できる。スループットを制限なくスケールさせたい。最も適切なサービスはどれか。

- A. Amazon SQS FIFOキュー
- B. Amazon SQS 標準キュー
- C. Amazon SNS
- D. Amazon MQ

> [!success]- 答え：**B**
> 標準キューは**ほぼ無制限のスループット**を提供する（少なくとも1回配信・ベストエフォート順序）。「順序不問 × 重複は冪等で吸収 × 超高スループット」なら標準一択。
> - A：**順序保証と重複排除が必須**（決済・在庫）なら正解になる。その代償としてスループットに上限がある。
> - B：正解。
> - C：複数の購読者へ**同報**したいなら正解になる。メッセージを溜めて順に処理するキューではない。
> - D：既存のJMS/AMQPアプリを改修せず移行するなら正解になる。ブローカーの性能上限があり、新規の超高スループット用途には選ばない。
> **関連ノート**：[[0802_SQS]]

## 第4問（Application Integration ｜ 失敗メッセージの扱い）
SQSキューのワーカーが特定の不正なメッセージの処理に毎回失敗し、そのメッセージが**何度もキューに戻って**リソースを浪費している。これを自動で隔離し、後から分析できるようにしたい。最も適切な方法はどれか。

- A. 可視性タイムアウトを延長する
- B. 遅延キューを設定する
- C. ロングポーリングを有効化する
- D. デッドレターキュー（DLQ）とmaxReceiveCountを設定する

> [!success]- 答え：**D**
> DLQは規定回数（maxReceiveCount）失敗したメッセージを隔離キューへ自動移動する。「毒メッセージの隔離と事後分析」の専用機構。
> - A：処理に時間がかかって**処理中に再配信されてしまう**問題なら正解になる。失敗し続けるメッセージは、時間を延ばしても戻り続ける。
> - B：送信直後の**初回配信を遅らせたい**（後続システムの準備待ち等）場合の機能。失敗後の扱いは変わらない。
> - C：空受信を減らしてAPIコストを下げる機能。失敗メッセージ対策とは無関係の改善。
> - D：正解。
> **関連ノート**：[[0802_SQS]]

## 第5問（Application Integration ｜ 既存メッセージ基盤の移行）
オンプレのApache ActiveMQに依存する基幹アプリ群（JMS・AMQPプロトコル使用）をクラウドへ移行する。**アプリのメッセージング部分は改修しない**方針である。最も適切なサービスはどれか。

- A. Amazon MQ
- B. Amazon SQS
- C. Amazon SNS
- D. Amazon MSK

> [!success]- 答え：**A**
> Amazon MQはActiveMQ/RabbitMQ互換のマネージドブローカー。JMS・AMQP・MQTT等の標準プロトコルをそのまま話せる。「既存プロトコル互換 × 無改修移行」が決め手。
> - A：正解。
> - B：**新規開発**ならこちらが第一候補（無制限スケール・運用レス）。ただし独自APIのため既存JMSアプリの無改修移行はできない。
> - C：Pub/Subの同報が要件なら候補になるが、こちらも独自API。
> - D：既存資産が**Apache Kafka**ならこちらが正解になる互換サービス。ActiveMQとはプロトコルが別物。
> **関連ノート**：[[0806_Amazon_MQ]] / [[0802_SQS]]

## 第6問（Serverless ｜ ワークフローの種類）
IoTデータの取り込み変換パイプラインをStep Functionsで組む。**毎秒10万実行以上**の高頻度で、1実行は5分以内に終わる。実行あたりのコストを最小化したい。最も適切なワークフロータイプはどれか。

- A. Standardワークフロー
- B. Expressワークフロー
- C. SQS＋Lambdaで自前オーケストレーション
- D. Amazon SWF

> [!success]- 答え：**B**
> Expressは最長5分・実行回数課金で、大量・短時間・高頻度の処理に特化した廉価版。「高頻度 × 5分以内 × コスト最小」が決め手。
> - A：**人の承認待ちを含む長時間処理**（最長1年）や、厳密に1回だけの実行保証が必要なら正解になる。状態遷移ごとの課金が高頻度実行では高くつく。
> - B：正解。少なくとも1回実行のセマンティクスは冪等設計で吸収する。
> - C：実在の構成だが、リトライ・分岐・タイムアウト管理を自作することになる。単純な直列処理なら十分正解になる。
> - D：Step Functionsの前身にあたるレガシー。新規採用の理由はほぼない（既存SWF資産の維持のみ）。
> **関連ノート**：[[0807_Step_Functions]] / [[0808_SWF]]

## 第7問（Serverless ｜ APIのスタイル選択）
モバイルアプリ向けバックエンドで、①複数のデータソース（DynamoDB・Aurora・既存API）を**1回のリクエストで**まとめて取得、②データ更新の**リアルタイム購読（サブスクリプション）**、を実現したい。最も適切なサービスはどれか。

- A. API Gateway REST API
- B. API Gateway WebSocket API
- C. AWS AppSync
- D. Application Load Balancer

> [!success]- 答え：**C**
> AppSyncはマネージドGraphQL。複数データソースを1クエリで束ね、サブスクリプションによるリアルタイム配信も組み込みで持つ。「複数ソースを1リクエスト × リアルタイム購読」＝GraphQL＝AppSync。
> - A：エンドポイント単位のRESTでよいなら王道の正解。複数ソースの集約は呼び出し側かLambda内で自作することになる。
> - B：双方向通信だけが要件なら正解になる。データ集約の仕組みは持たず、接続・ルーティング管理を自作する。
> - C：正解。
> - D：コンテナ/EC2への負荷分散装置。APIマネジメント（認証・スロットリング・集約）の層ではない。
> **関連ノート**：[[0811_AppSync]] / [[0809_API_Gateway]]

## 第8問（Serverless ｜ 双方向通信の実装）
既存のREST中心のサーバーレス構成に、**サーバー側から任意のタイミングでクライアントへ通知を送る**チャット機能を追加したい。GraphQLは採用していない。最も適切なサービスはどれか。

- A. API Gateway WebSocket API
- B. API Gateway HTTP API＋クライアントの定期ポーリング
- C. AWS AppSync
- D. Application Load Balancer＋EC2でWebSocketサーバーを自前運用

> [!success]- 答え：**A**
> API GatewayのWebSocket APIは永続接続を張り、サーバー側から接続IDに向けて任意のタイミングでプッシュできる。サーバーレスの双方向通信の定番。
> - A：正解。
> - B：実在する簡易解で、更新頻度が低く多少の遅延が許されるなら正解になり得る。リアルタイム性と無駄なリクエストのトレードオフで劣る。
> - C：サブスクリプションで同じことができるが、**GraphQL採用が前提**。REST資産のままという本問の制約に合わない。
> - D：WebSocketはALBも通せるので実在の構成。ただし常時稼働サーバーの運用が復活し、サーバーレス構成の利点を失う。
> **関連ノート**：[[0809_API_Gateway]] / [[0811_AppSync]]

## 第9問（Application Integration ｜ SaaSデータの取り込み）
SalesforceのデータをS3とRedshiftへ**定期的に、コードを書かずに**転送したい。フィールドのマッピングや簡単なフィルタも設定だけで済ませたい。最も適切なサービスはどれか。

- A. AWS DataSync
- B. Amazon AppFlow
- C. AWS Glue
- D. AWS Transfer Family

> [!success]- 答え：**B**
> AppFlowはSaaS（Salesforce・SAP・Slack等）とAWS間のデータフローをノーコードで定義できる。「SaaS × ノーコード転送」が決め手。
> - A：転送対象が**ファイルストレージ**（NFS/SMB/S3/EFS間）なら正解になる。SaaSのオブジェクトデータには接続できない。
> - B：正解。
> - C：複雑な変換・結合を伴う**本格ETL**なら正解になる。ジョブ開発（コード）が前提で、「コードを書かずに」の要件に反する。
> - D：取引先からの**SFTP/FTPS受信**の受け口。SaaS連携の道具ではない。
> **関連ノート**：[[0812_AppFlow]] / [[1502_DataSync]]

## 第10問（Serverless ｜ 定期実行）
毎日深夜3時にLambda関数を起動して日次レポートを生成したい。**サーバーレスで最小の構成**で実現する方法はどれか。

- A. EC2のcronからLambdaを呼び出す
- B. Step FunctionsのWaitステートで24時間待機するループを組む
- C. EventBridgeスケジュール（cron式）でLambdaを起動する
- D. SQS遅延キューで24時間後に配信する

> [!success]- 答え：**C**
> EventBridgeのスケジュールルール（またはEventBridge Scheduler）はcron式で任意のターゲットを起動できる。サーバーレス定期実行の既定解。
> - A：cronサーバーが**既に存在して他の用途でも使う**なら実在解。定期実行のためだけにEC2を常時稼働させるのは本末転倒。
> - B：ワークフローの**途中**の待機（例：3日後にリマインド）なら正解になる。無限ループの常駐実行は実行期間上限（1年）と可読性の面で筋が悪い。
> - C：正解。
> - D：遅延キューの上限は**15分**。24時間の遅延はそもそも設定できない。短時間の遅延投入なら正解になる。
> **関連ノート**：[[0804_EventBridge]] / [[0708_Lambda]]
