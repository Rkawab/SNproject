# AWS X-Ray（リクエストの旅を追跡する）

## TL;DR

> [!info] 超かんたんに言うと
> **X-Ray = アプリの「荷物追跡サービス」**。1つのリクエストが複数のサービスを渡り歩く様子を追跡し、「**どこで遅くなったか・どこでエラーが起きたか**」を特定する。
> [[1001_CloudWatch]] が「サーバーが元気か」を見るのに対し、X-Rayは「**リクエスト1件1件の旅路**」を見る。

---

## たとえ話：宅配便の荷物追跡

ネット通販で買った荷物が届かないとき、追跡番号で調べると「いつ・どの拠点を通ったか」が分かる。

| 宅配便 | X-Ray |
| --- | --- |
| 荷物に貼られる追跡番号 | **トレースID**（リクエストに付与される） |
| 「東京倉庫 → 横浜中継所 → 配達店」の経路 | リクエストが通るサービス（API Gateway → Lambda → DynamoDB） |
| 各拠点での滞在時間の記録 | **セグメント**（各サービスでの処理時間） |
| 拠点内の「仕分け→積込」の内訳 | **サブセグメント**（DB呼び出し・外部API等の内訳） |
| 「横浜で3日止まってた」と分かる | 「**Lambda内のDynamoDB呼び出しが遅い**」と分かる |
| 配送網全体の地図 | **サービスマップ**（構成と健康状態を図で可視化） |

マイクロサービス化したアプリは「荷物が10拠点を経由する」ようなもの。**どこか1か所が遅いと全体が遅くなる**が、各サーバーのCPUを見ても（[[1001_CloudWatch]]では）原因が分からない。そこでリクエスト自体に追跡番号を付けて旅を記録するのがX-Ray。

## 仕組みと用語

```
ユーザー
  │ ①リクエストにトレースIDを付与
  ▼
API Gateway ──→ Lambda ──→ DynamoDB
  │セグメント     │セグメント   │(サブセグメントとして記録)
  └──────┬───────┘
         ▼ ②各サービスがX-Rayへ計測データ送信
     X-Ray サービス
         ▼ ③可視化
   サービスマップ / トレース詳細（ウォーターフォール表示）
```

| 用語 | 意味 |
| --- | --- |
| **トレース（Trace）** | 1リクエストの最初から最後までの記録全体（追跡番号1個分） |
| **セグメント（Segment）** | 1つのサービス・リソースでの処理記録 |
| **サブセグメント** | セグメント内の細かい内訳（SQLクエリ、外部HTTP呼び出し等） |
| **サービスマップ** | サービス間の呼び出し関係を自動で図にしたもの。遅延・エラー箇所が色で分かる |
| **アノテーション** | トレースに付ける**検索可能な**ラベル（例: `userId=123`）。フィルタ式で絞り込める |
| **メタデータ** | トレースに付ける補足情報。**検索には使えない**（アノテーションとの違いが問われる） |
| **サンプリング** | 全リクエストでなく一部だけ記録する仕組み（デフォルト: 毎秒最初の1件＋以降の5%）。コスト・性能への影響を抑える |

### データの送り方

- アプリに **X-Ray SDK** を組み込む（コードに計測を仕込む）
- EC2/ECS/オンプレでは **X-Rayデーモン** を常駐させ、SDKからのデータをまとめてAWSへ転送（ECSでは**サイドカーコンテナ**として配置）
- **Lambda** はツールバー的に「アクティブトレース」を有効化するだけでOK（デーモン内蔵）
- 送信にはIAM権限（`AWSXRayDaemonWriteAccess` 等）が必要

### 主な対応サービス

API Gateway / Lambda / ECS / EKS / EC2 / Elastic Beanstalk / ELB / SQS / SNS / Step Functions など。サーバーレス構成（[[0809_API_Gateway]] → Lambda → DynamoDB）との組み合わせが定番。

## CloudWatch・CloudTrailとの使い分け（SAA最頻出）

| 質問 | 答え |
| --- | --- |
| サーバーのCPU・メモリは大丈夫か？ | [[1001_CloudWatch]] |
| **誰が**APIを呼んでリソースを変更したか？（監査） | [[1002_CloudTrail]] |
| リソースの設定は基準どおりか？ | [[1003_AWS_Config]] |
| **マイクロサービスのどこで遅延・エラーが起きているか？** | **X-Ray** ✅ |

> [!warning] 試験のキーワード反応
> 問題文に「**分散アプリケーション**」「**マイクロサービス**」「**ボトルネックの特定**」「**リクエストのトレース（追跡）**」「**エンドツーエンドの可視化**」とあれば、ほぼX-Rayが正解。
> 逆に「メトリクス監視・アラーム」ならCloudWatch、「API操作の記録・監査」ならCloudTrail。

## SAA頻出シナリオ集

### ① サーバーレスアプリ（API Gateway + Lambda + DynamoDB）の遅延原因を特定したい
**答え**：X-Rayを有効化（Lambdaはアクティブトレース、API Gatewayはトレース有効化）。サービスマップとトレースで遅い区間を特定

### ② マイクロサービス間の依存関係を可視化したい
**答え**：X-Rayの**サービスマップ**

### ③ EC2上のアプリでX-Rayを使いたい
**答え**：アプリにX-Ray SDKを組み込み、**X-Rayデーモン**をインスタンスで実行。インスタンスロールに書き込み権限を付与

### ④ 特定ユーザーのリクエストだけトレースを検索したい
**答え**：**アノテーション**として記録する（メタデータは検索不可）

### ⑤ トレースのコストを抑えたい
**答え**：**サンプリングルール**を調整（全件記録しない）

## 料金の勘所

- 記録・取得・スキャンしたトレース数に応じた従量課金（無料枠: 月10万トレース記録）
- サンプリングがコスト調整弁。全リクエスト記録は高くつく
