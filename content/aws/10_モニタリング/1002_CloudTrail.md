# CloudTrail（AWSアカウントの監視カメラ・入退室記録）

## TL;DR

> [!info] ひとことで
> **CloudTrail = AWSアカウント内で行われた「すべてのAPI呼び出し」の操作記録サービス**。
> 誰が・いつ・どこから・何をしたかを残す監査ログの基盤。

CloudWatchが「**リソースの体調**」を診るのに対し、CloudTrailは「**人間・システムの操作履歴**」を残す。両者の役割分担はSAA頻出。

---

## たとえ話：オフィスビルの入退室管理システム

CloudTrailはビルの**ICカードリーダー＆受付ログ**に近い。

| ビルでの役割 | AWSで対応するもの |
| --- | --- |
| ICカードによる入退室記録（誰が・いつ・どこに入ったか） | **CloudTrail** のイベントログ |
| 監視カメラの映像（建物の状況をリアルタイム監視） | **CloudWatch** |
| 入室許可カードの発行・権限設定 | **IAM** |
| 「金庫室を5回連続で開閉した」→警備員通報 | CloudTrail → **EventBridge → SNS/Lambda** |
| 記録は倉庫に保管して10年後も見返せる | CloudTrail → **S3バケットに証跡保存** |

つまりCloudTrailは「**操作の事実を残す台帳**」。リアルタイムに止めるのではなく、「あとから誰が何をしたか追える」ことに価値がある。

---

## 記録される内容

CloudTrailは AWS の **ほぼすべてのAPI呼び出し** を記録する。

イベント1件には以下が含まれる：

- **誰が**（IAMユーザー / ロール / ルート）
- **いつ**（タイムスタンプ）
- **どこから**（送信元IP）
- **何のサービスに**（例：EC2、S3）
- **何のAPIを**（例：RunInstances、PutObject）
- **どのリソースに対して**（例：i-xxxx、bucket名）
- **成功/失敗**と**エラー内容**

> マネジメントコンソールでのクリック操作も、裏ではAPIが叩かれているのですべて記録される。

---

## 3種類のイベント（SAA頻出）

### 1. 管理イベント（Management Events）

AWSリソースの **制御系操作** を記録。

- 例：EC2起動/停止、IAMユーザー作成、VPC作成、S3バケット作成
- **デフォルトで有効**、最近90日分は「イベント履歴」で無料閲覧可能
- 読み取りAPIと書き込みAPIを別々に有効化できる

### 2. データイベント（Data Events）

リソース内部の **データ操作** を記録。発生件数が膨大。

- 例：**S3オブジェクトのGetObject/PutObject/DeleteObject**、**Lambda関数のInvoke**、DynamoDB のアイテム操作
- **デフォルトOFF**（有料）
- S3バケットの不正アクセス調査などで有効化する

> [!warning] よくある誤解
> 「S3にアクセスがあったか調べたい」→ ❌ 管理イベントだけでは不十分。
> ✅ **データイベントを明示的に有効化**する必要がある。

### 3. インサイトイベント（Insights Events）

通常と異なる**異常なAPI呼び出しパターン** を機械学習で検知。

- 例：普段1日10回しか呼ばれないAPIが急に1000回呼ばれた
- 有効化が必要（追加料金あり）

---

## イベント履歴 vs 証跡（Trail）

| 観点 | イベント履歴 | 証跡(Trail) |
| --- | --- | --- |
| デフォルト | 自動で有効 | 自分で作成する必要あり |
| 保存期間 | **直近90日** | **無期限**（S3次第） |
| 保存先 | CloudTrail内部 | **S3バケット** |
| 対象 | 管理イベントのみ | 管理 + データ + インサイト |
| 料金 | 無料 | S3保存料金＋データイベントは有料 |

> [!tip] SAA対策
> 「**90日以上の監査ログを保存したい**」→ 証跡を作成してS3へ保存。
> 「Glacierにアーカイブしてコスト削減」→ S3ライフサイクルポリシーを設定。

---

## 証跡の種類

### リージョン証跡 vs 全リージョン証跡

- **All Regions（推奨）**：すべてのリージョンを1つの証跡で網羅。**将来追加される新リージョンも自動カバー**
- **単一リージョン**：そのリージョンのみ

### 組織証跡（Organization Trail）

- **AWS Organizations** 配下の **全アカウントの操作を一括収集**
- 子アカウントからは削除・無効化できない（管理アカウント専権）
- 大規模環境の統一監査に使う

---

## セキュリティ強化機能

### ログファイル整合性検証（Log File Validation）

- CloudTrailが出力したログファイルが**改ざんされていないか**を検証可能にする
- ハッシュをデジタル署名付きで別ファイル保存
- **証跡作成時にONにするのが推奨**（追加料金なし）

### S3バケットの保護

- ログ保存先S3は **バケットポリシーでCloudTrail以外からのPutObjectを拒否**
- **MFA Delete** や **オブジェクトロック** で改ざん・削除を防止
- 別アカウントの「ログ専用アカウント」に保存するのがベストプラクティス

### KMS暗号化

- ログを **SSE-KMS** で暗号化し、復号権限もIAMで制御
- 「ログを読めるのは監査チームだけ」という分離が可能

## 連携サービス

| 連携先 | できること |
| --- | --- |
| **S3** | 長期保存・分析の基盤 |
| **CloudWatch Logs** | リアルタイムにログ転送 → メトリクスフィルタで検知 |
| **EventBridge** | 特定APIが呼ばれたら自動アクション（Lambda起動・SNS通知） |
| **Athena** | S3に保存したログをSQLで検索 |
| **Security Hub / GuardDuty** | セキュリティ脅威の自動検知 |

> [!example] 典型構成
> CloudTrail → S3（証跡保存） → Athena（SQL検索）
> CloudTrail → CloudWatch Logs → アラーム → SNS（即時通知）

## CloudWatch との明確な使い分け（SAA最重要）

### そもそも「ログ」の意味が2つ違う

**「どっちもログでしょ?」という混乱の根本原因** は、ログの定義が異なること：

| | CloudTrail | CloudWatch Logs |
| --- | --- | --- |
| 何のログ? | **誰が何の操作をしたか**（API呼び出しの記録） | **アプリやサーバーが吐き出した文章**（出力ログ） |
| 書く人 | **AWSが自動で書く**（あなたは書けない） | **アプリ/サーバー/Lambdaが自分で書く** |
| 形式 | 固定（誰が・いつ・どこから・何を） | 自由なテキスト（何でも書ける） |
| 例 | 「田中さんが10:03にEC2を削除した」 | 「ERROR: DB接続失敗」「ユーザーAがログイン」 |

### たとえ話：会社で考える

- **CloudTrail ＝ 入退室カードの記録**。「誰が・いつ・どこの部屋に入った」を**自動で**記録する。社員が手書きするわけじゃなく、カードリーダーが勝手に残す。形式は決まっている。
- **CloudWatch Logs ＝ 各部署の業務日報を集める箱**。中身は部署が自由に書く（「在庫が切れた」「機械が止まった」など何でも）。**書くのは現場(アプリ)で、CloudWatch Logsは集めて保管・検索する箱**。

つまり：
- 「**誰がやったか**」を追いたい → **CloudTrail**（監査・セキュリティ）
- 「**アプリが何を出力したか**（エラーログ等）」を見たい → **CloudWatch Logs**

**CloudTrail は「操作の台帳」、CloudWatch Logs は「出力文の保管箱」。役割が全然違うんです。**

### 具体的な使い分け比較表

| 観点 | CloudTrail | CloudWatch |
| --- | --- | --- |
| 監視対象 | **API呼び出し**（操作の事実） | **リソース状態**（メトリクス・ログ） |
| 主な用途 | **監査・セキュリティ・原因追跡** | **性能監視・障害検知** |
| 「誰がEC2削除した？」 | ✅ CloudTrail | ❌ |
| 「EC2のCPU使用率は？」 | ❌ | ✅ CloudWatch |
| 「Lambdaの実行回数は？」 | ❌（呼び出した主体は分かる） | ✅ メトリクス |
| 「不審なIPからのアクセスは？」 | ✅ CloudTrail（送信元IP記録） | ❌ |
| デフォルト保存 | 90日（イベント履歴） | メトリクス15か月 / ログ無期限設定可 |

> [!question] SAA出題パターン
> - 「**不正操作の追跡**」「**監査ログ**」「**コンプライアンス**」→ **CloudTrail**
> - 「**リソースの異常検知**」「**閾値アラート**」→ **CloudWatch**
> - 「**両方使う**」→ CloudTrail で「誰が何した」、CloudWatch で「結果どうなった」

## CloudTrail と CloudWatch Logs の「連携」とは何か

### 問題点：CloudTrailだけでは「リアルタイム警報」ができない

CloudTrailは記録は得意ですが、**単体だと「リアルタイムに条件で警報を鳴らす」のが苦手**です。

標準では以下の2つの保存モードしかありません：
- **イベント履歴**（直近90日、後から検索する用）
- **S3への証跡保存**（長期保管・後から分析する用）

→ 「**ログイン失敗が3回来た瞬間に通知**」というリアルタイム判定の仕組みがない

### 解決策：CloudWatch Logs へのストリーミング

**CloudTrailの記録の流れを、CloudWatch Logsという箱に「流し込む」設定をする。これが「連携」。**

連携すると、CloudTrailが記録するそばから、その内容がCloudWatch Logsに**ストリーム（垂れ流し）**されます。

CloudWatch Logsに入ってきたテキストに対しては：
- **メトリクスフィルタ** = 「"ログイン失敗"という文字が来たらカウント+1」という見張りを仕掛けられる
- **アラーム** = 「カウントが3を超えたらSNSでメール」と判定できる

これでようやく「3回失敗→即通知」が実現します。

### 連携のイメージ（パイプでつなぐ）

```
CloudTrail（操作を記録）
   │  ← ここで「CloudWatch Logsに流す」設定＝連携
   ▼
CloudWatch Logs（流れてきた記録を受け止める箱）
   │  ← メトリクスフィルタで「失敗」だけ数える
   ▼
CloudWatch アラーム（3回超えたら発火）
   │
   ▼
SNS → メール通知
```

CloudTrailは「**録画する人**」、CloudWatch Logsは「**録画を受け取って、特定シーンが映ったら数える人**」。1人では完結しないので**バトンを渡す＝連携**するわけです。

## なぜS3じゃダメだったか（補足）

CloudTrailの定番保存先はS3ですが、**S3は「倉庫」なのでリアルタイム判定ができません**。

- **長期保管・後でSQL分析** → CloudTrail → **S3**（→ Athena）
- **リアルタイムに条件検知して通知** → CloudTrail → **CloudWatch Logs** → アラーム → SNS

この使い分けもSAA頻出です。

## 料金の感覚

- **イベント履歴**：無料（90日まで）
- **管理イベント**：最初の証跡コピーは無料、2つ目以降は有料
- **データイベント / インサイトイベント**：有料（1リクエスト課金）
- **S3保存料**：保存量に応じて
- **CloudWatch Logs転送**：データ取り込み量に応じて

> [!warning] 注意
> S3バケットに大量のオブジェクト操作があるとデータイベントは大量発生する。安易に全バケット有効化すると料金爆発する可能性あり。

---

## ベストプラクティス

1. **全リージョン証跡** を1つ作成（管理イベントを永続保存）
2. **AWS Organizations** 配下なら組織証跡で一括管理
3. **ログファイル整合性検証** を必ずON
4. ログ保存用S3は **別アカウント** または **MFA Delete + バージョニング**
5. 重要バケット・関数のみ **データイベント** を有効化
6. **CloudWatch Logs と EventBridge** で異常検知の自動化
