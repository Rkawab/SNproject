# AWS Config（リソース設定の健康診断・タイムマシン）

## TL;DR

> [!info] ひとことで
> **AWS Config = AWSリソースの「設定状態」を継続記録し、ルール準拠を評価するサービス**。
> 「今この瞬間どういう設定か」と「過去にどう変わってきたか」を時系列で追える。

CloudTrailが「**操作の事実（誰が何をAPI呼んだか）**」を残すのに対し、Configは「**結果としてリソースがどんな状態か**」を残す。

---

## たとえ話：建物の定期点検記録簿

| 建物管理での役割 | AWSで対応するもの |
| --- | --- |
| 入退室ログ（誰が・いつ・どこに入ったか） | **CloudTrail** |
| 防犯カメラ映像（リアルタイムの様子） | **CloudWatch** |
| 各部屋の定期点検記録（鍵・窓・消火器の状態） | **AWS Config**（リソース設定の記録） |
| 「全部屋の鍵は閉めること」等の社内規則 | **Config Rules** |
| 規則違反を見つけたら自動施錠する係 | **Remediation**（SSM Automation連携） |
| 過去の点検記録を全部読み返す | **Configuration History**（S3保存） |
| 複数ビルの点検結果を本社で一括確認 | **Aggregator**（マルチアカウント/リージョン） |

つまりConfigは「**設定の事実を点で記録し続け、ルールと照らし合わせて◯×をつける**」サービス。

---

## 何を記録するのか

リソースごとに以下を保存：

- **設定項目**（例：EC2のインスタンスタイプ、SGのインバウンドルール、S3バケットポリシー）
- **関連リソース**（このEC2が使っているSG・EBS・サブネット等の依存関係）
- **タグ**
- **設定変更のタイムスタンプ**
- **変更前→変更後の差分**

> 結果として「2026-05-01時点でこのSGはどんなルールだったか」が一発で分かる。

## 主な構成要素

### 1. Configuration Recorder

リソース設定を継続的に記録するエンジン。

- **リージョン単位**で有効化
- 対象リソースタイプを選択可能（全部 or 一部）
- **グローバルリソース（IAM等）**は1リージョンで記録すれば十分

### 2. Configuration History / Snapshot

- **History**：個別リソースの設定変更を時系列で保存
- **Snapshot**：ある時点の全リソース設定の一括写し
- どちらも **S3バケットに保存**

### 3. Config Rules（設定ルール）

「望ましい設定」を定義してリソースを評価。

- **マネージドルール**：AWS提供の組み込みルール（100種類以上）
- **カスタムルール**：Lambda関数で自作 or Guard構文（IaC的に書ける）

評価タイミング：

- **設定変更時**（イベントトリガー）
- **定期的**（24時間ごと等）

### 4. Conformance Packs

複数ルールをパッケージ化してデプロイ。

- 例：**PCI-DSS、HIPAA、CIS Benchmarks、AWS Foundational Security Best Practices**
- YAMLテンプレートで管理・配布可能
- 日本語では **コンフォーマンスパック** と表記されることがある

> [!tip] SAAひっかけ
> コンフォーマンスパックは「設定ルール集」を配る仕組み。例：S3公開禁止、EBS暗号化必須、root MFA必須など。
> EC2/ECR/Lambdaの中にある既知脆弱性（CVE）をスキャンする機能ではない。脆弱性スキャンは [[1603_脅威検知・監査|Amazon Inspector]]。

### 5. Aggregator（アグリゲータ）

複数アカウント・複数リージョンの評価結果を **1か所に集約** 表示。

- AWS Organizations 連携で全アカウント自動追加
- セキュリティ部門の統一監査ダッシュボードに使う

### 6. Remediation（自動修復）

非準拠リソースを **SSM Automation Document** で自動修復。

- 例：「暗号化されていないEBSを発見 → スナップショット作って暗号化版を作成」
- 手動修復もマネジメントコンソールから実行可能

## よく使うマネージドルール（SAA頻出）

| ルール名 | チェック内容 |
| --- | --- |
| `s3-bucket-public-read-prohibited` | S3バケットが公開読み取り不可か |
| `s3-bucket-public-write-prohibited` | S3バケットが公開書き込み不可か |
| `s3-bucket-server-side-encryption-enabled` | S3がSSE有効か |
| `encrypted-volumes` | EBSが暗号化されているか |
| `restricted-ssh` | SGが0.0.0.0/0からのSSHを許可していないか |
| `root-account-mfa-enabled` | rootアカウントでMFA有効か |
| `iam-password-policy` | IAMパスワードポリシーが基準を満たすか |
| `rds-storage-encrypted` | RDSストレージが暗号化されているか |
| `cloud-trail-enabled` | CloudTrailが有効か |

> [!tip] SAA対策
> 「**Sグループが0.0.0.0/0でSSH許可していないか継続的にチェック**」→ Config Rule `restricted-ssh`
> 「**S3公開を検知して通知**」→ Config Rule + EventBridge → SNS

## CloudTrail / CloudWatch / Config の使い分け（最重要）

| 観点 | CloudTrail | CloudWatch | **AWS Config** |
| --- | --- | --- | --- |
| 監視対象 | **API呼び出し** | **メトリクス・ログ** | **リソースの設定状態** |
| 視点 | 「誰が何をしたか」 | 「今、調子はどうか」 | **「今、どういう設定になっているか」** |
| 時間軸 | イベント（点） | 時系列（線） | **状態のスナップショット＋履歴** |
| 例1 | 誰がSGを変更したか | EC2のCPU使用率 | **SGが0.0.0.0/0を許可していないか** |
| 例2 | 誰がS3を公開設定にしたか | S3リクエスト数 | **S3が公開されていないか** |
| 例3 | 誰がEBSを作成したか | EBSのIOPS | **EBSが暗号化されているか** |

> [!question] SAA出題パターン
> - 「**コンプライアンス準拠を継続的にチェック**」→ **AWS Config**
> - 「**操作した人物を追跡**」→ **CloudTrail**
> - 「**性能異常を検知**」→ **CloudWatch**
> - 「**3つ組み合わせ**」→ Configで違反検知 → CloudTrailで誰がやったか調査 → CloudWatchで影響度確認

## 典型構成パターン

### パターン1：違反検知と通知

```
リソース変更
  → Config Recorder（記録）
  → Config Rule評価
  → 非準拠
  → EventBridge
  → SNS（メール通知）
```

### パターン2：自動修復

```
非準拠検知
  → Remediation Action（SSM Automation）
  → 自動でリソース修正
  → 再評価して準拠状態に
```

### パターン3：マルチアカウント統合監査

```
複数アカウントのConfig
  → Aggregator（集約）
  → Security Hub / Audit Manager
  → 統合ダッシュボード
```

## 料金の感覚

- **記録された設定項目1件あたり**で課金
- **Config Rule評価1回あたり**で課金
- **Conformance Packも評価数で課金**
- リソースが多いと地味に高くなるため、不要なリソースタイプは記録対象から外すのも手

> [!warning] 注意
> 全リージョンで全リソース記録ONにすると料金が膨らむ。本番リージョンに絞る・必要なリソースタイプに絞るのが基本。

---

## ベストプラクティス

1. **本番リージョン中心**にConfig Recorderを有効化
2. **グローバルリソース（IAM等）は1リージョンのみ**記録（重複防止）
3. **Conformance Pack** で業界標準ベンチマークを一括適用
4. **重大ルール違反は EventBridge → SNS** で即通知
5. **Remediation** で繰り返し違反は自動修復
6. **Aggregator** で組織全体の準拠率を可視化
7. **Security Hub** と連携して総合的なセキュリティスコア化

## CloudTrailとConfigの「合わせ技」

SAAでよく出る問題パターン：

> 「セキュリティグループが意図せず変更されていた。**誰が・いつ・どんな設定に変更したか** を確認したい」

→ **Config** で設定変更履歴（前→後の差分）を確認
→ **CloudTrail** で変更APIを呼んだIAMユーザー・時刻・送信元IPを確認

**両方ないと完全な追跡はできない**のがポイント。
