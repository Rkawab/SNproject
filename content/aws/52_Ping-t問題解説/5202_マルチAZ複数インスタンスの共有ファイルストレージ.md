# マルチAZ・複数インスタンスの共有ファイルストレージ

> Ping-t の問題解説。汎用知識は [[0202_S3とEBS・EFSの違い]] / [[0304_ElastiCache]] に統合。ここは問題単位の解き方メモ。

## 問題

ある会社は、数十台のEC2インスタンスで実行されるバッチ処理用のアプリケーションを開発している。EC2インスタンスはALB配下のAuto Scalingグループに所属し、マルチAZで起動する。バッチ処理に使用するファイルは各インスタンスで共有し、大量の読み込み/書き込みが行われるので、ACID特性のあるストレージに保存したい。
これらの要件を満たす方法は、次のうちどれか。

- ① Amazon EBS にファイルを保存し、各EC2インスタンスからアタッチする
- ② **Amazon EFS にファイルを保存し、各EC2インスタンスからマウントする** ✅
- ③ Memcached型の Amazon ElastiCache にファイルを保存し、各EC2インスタンスから参照/更新する
- ④ Amazon S3 One Zone-IA にファイルを保存し、各EC2インスタンスから使用する

## 正解：②

## 要件の読み解き

| 要件 | 何が必要か |
|------|-----------|
| 数十台のEC2＋マルチAZで起動 | AZをまたいで使える共有ストレージ |
| ファイルを各インスタンスで共有 | 同時マウントできる共有ファイルシステム |
| 大量の読み込み/書き込み | スループットが伸びるストレージ |
| ACID特性 | ファイルロック・一貫性のある書き込み |

→ 「複数インスタンスから同時マウントできる共有ファイルシステム」＝ **EFS**。

## たとえ話：ストレージの「形」の違い

| サービス | たとえ | 同時に何台使える？ |
|----------|--------|------------------|
| EBS | 1台のPCに挿すUSB SSD | 基本1台・同一AZ |
| EFS | みんなで使うネットワーク共有フォルダ（NAS） | 何台でも・複数AZから同時に ✅ |
| S3 | 倉庫に箱を出し入れするオブジェクト保管庫 | マウントするファイルシステムではない |
| ElastiCache | 計算結果を覚える付箋メモ（高速・揮発） | ファイル保存先ではない |

## 各選択肢の検討

- **① EBS**：原則1インスタンスにアタッチ＋単一AZ固定。マルチAZ数十台の共有には不向き（Multi-Attachは io1/io2・同一AZ限定の特殊用途で本要件に合わず）。
- **② EFS**：NFSベースの共有ファイルシステム。マルチAZの複数インスタンスから同時マウントでき、ファイルロックで一貫性も担保。要件に完全合致。✅
- **③ Memcached型 ElastiCache**：インメモリキャッシュでファイルストレージではない。永続性・ACIDなし。用途違い（MemcachedはRedisより永続化・複雑構造に弱い）。
- **④ S3 One Zone-IA**：オブジェクトストレージでマウント用途ではない。One Zone＝単一AZでマルチAZ要件にも反する。

## 学びポイント

> [!important]
> - 「複数インスタンス / 共有 / マルチAZ / ファイル」が揃ったら → **EFS**
> - 1インスタンス専用の高速ブロックストレージ → EBS
> - 「One Zone」は単一AZ＝マルチAZ要件では即除外
> - ElastiCache は「ファイル保存先」ではない（キャッシュ）
