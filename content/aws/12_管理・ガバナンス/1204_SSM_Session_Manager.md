# SSM Session Manager と プライベートEC2への安全な接続方法

## 問題設定：プライベートサブネットのEC2に管理者だけが接続する

**前提**
- **プライベートサブネット** … VPC（AWS上の自分専用ネットワーク）の中で、**インターネットに直接つながっていない区画**
- **パブリックIPが無い／インバウンド不許可** … 外から名指しで呼べる住所を持たず、外からの接続を一切受け付けない状態
- **SSH（Secure Shell）** … Linuxサーバーに遠隔ログインして、シェルでコマンドを打つ標準方式
  - 通常は**サーバー側で22番ポートを開けて待ち受ける**必要がある = インバウンドを許可することになる

**課題**：隔離されたサーバーへ「踏み台なし・ポート開放なし・セッション内のすべてのコマンドを監査可能」に接続したい

---

## 4つの接続方法の比較

| 方法 | 仕組み | 開け方 | コマンド監査 | 運用 | 最適な用途 |
|---|---|---|---|---|---|
| **A. Client VPN** | 管理者のPCとVPCの間に暗号化トンネルを張り、PCを仮想的にVPCの中に居る状態にする | VPN接続＋インバウンド許可 | **ローカルシェル履歴**（本人が消せる） | VPN構築・証明書管理の負荷 | 恒常的なリモートアクセス |
| **B. EC2 Instance Connect** | VPC内に置くAWS管理の中継口。踏み台を自分で建てずにプライベートEC2へSSHできる | インバウンド許可（ただしインスタンスコネクトエンドポイント経由） | **CloudTrailのみ**（APIの操作記録、SSHセッション内のコマンドは記録されない） | 中程度 | 開発・デバッグ |
| **C. 踏み台（Bastion）ホスト** | パブリックサブネットに中継用EC2を自分で建て、まずそこにSSHし、そこから奥のEC2へ乗り継ぐ | 2段階のインバウンド許可 | CloudWatch Agentで取得可能（設定複雑） | 踏み台のパッチ・可用性管理が必要 | 構成がシンプルな環境 |
| **D. SSM Session Manager** | EC2に入れたSSM Agent（常駐ソフト）が**外向き**にAWSへ繋ぎに行き、その通り道を借りてシェルを開く | **アウトバウンド許可のみ**（サーバー側から迎えに来てもらう） | **CloudWatch Logs**に丸ごと記録可能 | インバウンド・踏み台・パッチ不要 | **セキュリティ重視・監査重視** |

---

## 正解：SSM Session Manager（D）が選ばれる理由

### 1. **接続方向が逆転**
- A・B・C … **外から中へ入っていく（インバウンド）** → どこかで扉（ポート）を開ける必要がある
- D … **EC2の中のSSM Agentが自分から外のAWSへ繋ぎに行く（アウトバウンド）**
  - 管理者はAWS側の窓口（AWS Systems Manager）に依頼を出す
  - Agentが開けた通り道に相乗りする
  - **扉を1つも開けずに済む** = 攻撃面が最小

### 2. **セッション内のすべてのコマンドが記録できる**
Session Managerは**セッション（ターミナル操作）そのものをAWSが仲介する**ので：
- 打ったコマンドと出力を丸ごと**CloudWatch Logsに記録**できる
- **ローカルシェル履歴ではない** = ユーザーが消したり改ざんできない
- 一元的・改ざん不可能な監査証跡になる

### 3. **MFA強制が可能**
- IAMポリシーの条件（`aws:MultiFactorAuthPresent`）で、Session Manager接続にMFA必須を強制できる
- 接続認可はIAMで効く

### 4. **運用オーバーヘッドがない**
- インバウンドのポート開放が不要
- パブリックIPが不要
- **踏み台の構築・パッチ・可用性管理という運用がまるごと消える**

---

## 各方法がなぜ不正解か

### A. Client VPN × 
**問題：シェルコマンドの完全監査ができない**
- VPN接続は実現できるが、その後のSSHセッションはローカルマシンとサーバー間の直接通信
- コマンドは**ローカルのシェル履歴**に保持されるだけ
- 問題：
  - シェル履歴は本人が消せる・改ざん可能
  - 端末ごとにバラバラで一元的な監査証跡にならない
- VPN構築・証明書管理という運用負荷も増える

### B. EC2 Instance Connect エンドポイント × 
**問題：CloudTrailではSSHセッション内のコマンドが記録されない（最大の罠）**
- 接続手段としては新しく正当な方法
- CloudTrailが記録するのは「**AWS APIの操作**」
  - 「誰がいつ接続を開始したか」は残る
  - しかし**SSHでログイン後に打った `rm -rf` や `cat 機密ファイル` の中身は一切記録されない**
  - SSHセッションの中身はAWSから見えない素のLinux操作だから
- **「すべてのシェルコマンドを記録」要件を満たせない**

### C. 踏み台（Bastion）ホスト × 
**問題：攻撃面増加＆運用負荷**
- インターネットから踏み台サーバーを晒す時点で攻撃面が増える
- 踏み台のパッチ当て・セキュリティ更新・可用性管理という**運用がまるごと乗る**
- CloudWatch Agentでセッションログを取る構成も設定が煩雑で確実性に欠ける
- 「**運用オーバーヘッド最小**」要件と逆行

## 記録系サービスの役割

### CloudTrail
- **記録対象**：「**誰がいつAWSのAPIを叩いたか**」の監査ログ
- **記録対象外**：サーバー（EC2）の中での操作（SSHセッション内のコマンド）
- **用途**：AWS側の操作履歴、コンプライアンス
- **このセッション管理の観点では**：接続「開始」の記録のみ

### CloudWatch Logs
- **機能**：ログを集めて保存・検索する置き場
- **このセッション管理の観点では**：**Session Managerのセッション記録の送り先**
- **記録内容**：打ったコマンド、出力、タイムスタンプ全て

### CloudWatch エージェント
- **機能**：サーバー（EC2）に入れてOS内部のログ（syslog等）を CloudWatch Logs に送るソフト
- **このセッション管理の観点では**：補助的（踏み台構成の場合）

## SAA 試験の着眼点

### ポイント1：「プライベートEC2に踏み台なし・ポート開放なしで安全に接続」
→ **SSM Session Manager**。定番中の定番。

### ポイント2：「実行したコマンド／セッションの操作を記録・監査」
→ **Session Managerのセッションログ + CloudWatch Logs**
→ **CloudTrailは「APIの操作記録」でセッション内のシェルコマンドは残らない** ← この線引きが決定的なひっかけ

### ポイント3：「接続時のMFA強制」
→ **IAMポリシーの条件** (`aws:MultiFactorAuthPresent`)

### ポイント4：Instance Connect / Client VPN / 踏み台は「接続はできる」
→ しかし**セッション内コマンドの完全監査までは標準で担保しない**
→ 監査要件が強い問題ではSession Managerを選ぶ

---

## 出題パターン

1. **「プライベートEC2にセキュアに接続＋コマンド監査」** → Session Manager
2. **「CloudTrailで何が記録されるか」** → AWS APIの操作（セッション内のコマンドは対象外）
3. **「踏み台との違いは」** → ポート開放・踏み台管理・可用性管理が不要
4. **「CloudWatch Logsの役割」** → Session Managerのセッションログ保存先
