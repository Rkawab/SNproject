# Amazon ECS Anywhere

**ECSのコントロールプレーンはAWSに置いたまま、コンテナ本体をオンプレ・エッジ・他クラウド等のAWS外サーバーで動かせる**サービス（2021年〜提供）。

通常の[[0904_ECS]]は「AWSの中のEC2かFargate」でしかコンテナを動かせないが、ECS Anywhereを使うと**自社データセンターのサーバーや工場PC、別クラウドの仮想マシンなど、SSM AgentとECS Agentが入れられる場所ならどこでも**ECSタスクとして起動できる。

---

## たとえ話：本部直営の宅配ピザチェーン

| 概念 | ピザチェーンのたとえ |
|---|---|
| ECSコントロールプレーン | 本部（メニュー・配車指示・売上管理） |
| 通常のECS (EC2/Fargate) | 本部所有の直営店舗 |
| **ECS Anywhere** | **個人経営の店舗が本部とフランチャイズ契約し、本部のレシピ・指示でピザを焼く** |
| 外部サーバー | フランチャイズ店の厨房 |
| ECS Agent + SSM Agent | 本部とつながるための専用タブレット |

本部（AWS）の指示書（タスク定義）はそのままに、**厨房（実行場所）だけ自前**にできるのがポイント。

---

## アーキテクチャ

```
[AWS側]                              [ユーザー側（オンプレ/他クラウド）]
ECSコントロールプレーン ───────►  外部インスタンス
  (タスクスケジューラ)               ├── ECS Agent
                                      ├── SSM Agent
                                      └── Docker
        ▲                                   │
        │     SSMで登録・通信              │
        └───────────────────────────────────┘
```

### 登録の流れ

1. ECSクラスターを作成
2. SSM Activationを発行（アクティベーションコード取得）
3. オンプレサーバーで `ssm-agent` と `ecs-agent` をインストール、コードで登録
4. 外部インスタンスとしてクラスターに参加
5. タスク定義の起動タイプ = **EXTERNAL** でタスクを実行

## 主なユースケース

| ケース | 理由 |
|---|---|
| オンプレ既存資産を活かしたい | サーバー買い替えせずコンテナ運用に移行 |
| データ主権・規制でクラウドに置けない | 医療・金融・公共データを自社内に留めながらECSで統一管理 |
| 工場・店舗の**エッジ**で低遅延処理 | 現地マシンでコンテナ実行、管理だけクラウドから |
| ハイブリッド運用の管理コンソール統一 | ECSのAPI/CLIでAWS内外を同じ仕組みで扱える |

## 通常ECS / Fargate / ECS Anywhere 比較

| 項目 | ECS on EC2 | ECS on Fargate | **ECS Anywhere** |
|---|---|---|---|
| コントロールプレーン | AWS | AWS | AWS |
| 実行ノード | AWSのEC2 | AWSが完全管理 | **ユーザーの外部サーバー** |
| ノード管理責任 | ユーザー | AWS | **ユーザー（OS・Docker・パッチ）** |
| ネットワーキング | VPC | VPC | ユーザー側NW（awsvpcモード非対応） |
| 課金 | EC2料金 | vCPU/メモリ秒 | **外部インスタンス1台あたり時間課金（少額）** |
| ロードバランサー | ALB/NLB | ALB/NLB | **ELB非対応**（自前ロードバランサ） |

## 制限事項

> [!warning] ECS Anywhere には通常ECSと異なる制限がある
> - **タスクのネットワークモードは `bridge` / `host` のみ**（`awsvpc` 非対応）
> - **AWSのロードバランサー（ALB/NLB）と直結できない**
> - **Service Discovery（AWS Cloud Map）非対応**
> - **永続ストレージはEBS/EFS非対応**（ローカルディスクのみ）
> - インターネット経由でAWS APIに到達できる必要あり（SSM/ECSエンドポイント）

## EKS Anywhere との違い

| | ECS Anywhere | EKS Anywhere |
|---|---|---|
| コントロールプレーン | **AWS側に存在** | **ユーザー側に存在**（オンプレで動く） |
| AWS接続必須か | はい（API通信が必要） | いいえ（完全オフライン運用も可能） |
| 互換性 | ECS API | Kubernetes標準 |
| 管理方法 | AWSコンソール/CLIで一元管理 | ローカルのkubectl |

> [!tip] 「ECS Anywhere」はあくまでAWS主導のハイブリッド運用、「EKS Anywhere」は完全オンプレK8sにも対応できるのが大きな差。

## SAA試験 頻出ポイント

| シナリオ | 正解 |
|---|---|
| オンプレのコンテナワークロードをAWSと同じ仕組みで一元管理したい | **ECS Anywhere** |
| オンプレでKubernetesをAWSと統一管理したい | **EKS Anywhere** |
| データを自社内に留めたままコンテナ管理だけAWSで行いたい | **ECS / EKS Anywhere** |
| サーバーレスでコンテナを動かしたい | ECS + Fargate |
| エッジ拠点に少数のコンテナを置きたい | ECS Anywhere（または[[AWS Outposts]]） |

> [!info] 「Anywhere = コントロールプレーンはAWS、データプレーンは外部」と覚えればEKS Anywhereとの対比もスッキリ。

## 料金イメージ

- 外部インスタンス1台あたり時間課金（2026年現在、$0.01024/時間程度）
- 別途、登録・通信に使うSSMやCloudWatch Logs等の料金が発生
- Fargateのような秒単位課金ではない点に注意
