# SSM Agent / ECS Container Agent / ECS コントロールプレーン

[[0905_ECS_Anywhere]]の登場人物3種を整理する。役割が全く違うので混同しがちだが、**「司令塔・通信窓口・現場監督」**の3層構造で捉えるとスッキリする。

---

## たとえ話：宅配ピザのフランチャイズ店

| 登場人物 | 役割 | AWSでの対応 |
|---|---|---|
| **本部** | メニュー決定・注文受付・全店舗の在庫管理 | **ECSコントロールプレーン** |
| **店舗の固定電話＆ID証** | 本部と確実に連絡を取り、店舗の存在を本部に知らせる | **SSM Agent** |
| **店長（現場監督）** | 本部の指示を受けて実際にピザを焼く・スタッフに指示する | **ECS Container Agent** |

3つは協力関係で、欠けると店舗（外部サーバー）はECSクラスターに参加できない。

---

## 1. ECS コントロールプレーン（司令塔・AWS側）

**役割：「どのタスクを、どこで、いくつ動かすか」を決める頭脳**

- AWSが完全マネージドで運用する
- ユーザーからは見えない（AWSリージョン内のどこかで動いている）
- APIエンドポイント（`ecs.ap-northeast-1.amazonaws.com` など）として公開

### 主な仕事

| 仕事 | 説明 |
|---|---|
| タスクスケジューリング | 「タスク定義Aを3個起動して」という指示を受け、配置先ノードを決める |
| クラスター状態管理 | クラスター内のノード一覧・タスクの稼働状況を保持 |
| サービス維持 | 「常に3個」を維持するため落ちたタスクを再起動指示 |
| デプロイ管理 | ローリング更新・Blue/Greenの制御 |

> [!info] Kubernetesでいう「マスターノード（kube-apiserver, scheduler, controller-manager）」に相当。ECSではこれを**全部AWSが代行**してくれる。

---

## 2. SSM Agent（通信窓口・ノード側）

**役割：オンプレやEC2を「AWSから操作可能な状態」にする汎用エージェント**

正式名称：**AWS Systems Manager Agent**

- ECS専用ではなく、**Systems Manager（SSM）全体の足回り**
- インストールするとサーバーがAWSコンソール/CLIから操作できるようになる
- ECS Anywhereでは「外部サーバーをAWSに登録する」役を担う

### 主な仕事

| 仕事 | 説明 |
|---|---|
| マネージドインスタンス登録 | アクティベーションコードでサーバーをAWSに紐付け |
| リモートコマンド実行 | Run Commandでスクリプト配信 |
| パッチ適用 | Patch Managerからの指示受付 |
| Session Manager接続 | SSHキーなしでブラウザからシェル接続 |

### ECS Anywhereでの具体的な役割

```
オンプレサーバーの起動
   ↓
SSM Agent が起動
   ↓
SSM Activation Code でAWSに登録
   ↓
AWSコンソールに「マネージドインスタンス」として現れる
   ↓
ECS Anywhereの土台が完成（ECS Agentが上に乗れる状態）
```

> [!tip] SSM Agentは「AWSへの裏口を開ける鍵」。ECSに限らず、オンプレをAWSから管理したい時の**ほぼ必須エージェント**。

---

## 3. ECS Container Agent（現場監督・ノード側）

**役割：そのサーバー上で実際にDockerコンテナを起動・停止する常駐プログラム**

- 各ノード（EC2 or 外部サーバー）に1つずつ常駐
- Go言語製のオープンソース（`amazon-ecs-agent` GitHubで公開）
- コントロールプレーンと**常時通信**して指示を受ける

### 主な仕事

| 仕事 | 説明 |
|---|---|
| タスク受信 | コントロールプレーンから「このタスク定義を起動せよ」を受信 |
| Docker制御 | ローカルのDocker daemonに `docker run` 相当を発行 |
| ヘルス報告 | タスクの状態（RUNNING/STOPPED）を定期的にAWSへ送信 |
| ログ転送 | コンテナのstdoutを[[1001_CloudWatch]] Logsへ転送 |
| イメージpull | [[0903_Amazon_ECR]]などから必要なDockerイメージを取得 |

### 動作モデル

```
[コントロールプレーン(AWS)]
        ↕  ロングポーリング（HTTPS）
[ECS Container Agent]  ← 各ノードに常駐
        ↕  Unix Socket
[Docker daemon]
        ↓
   コンテナ起動
```

> [!info] Kubernetesでいう「kubelet」に相当。ノードの代理人として中央(マスター)とやり取りする。

## 3つの関係図（ECS Anywhere構成）

```
┌─────────────────────────────────────┐
│      ECS コントロールプレーン        │  AWSが管理
│       （タスクスケジューラ）         │
└─────────────────────────────────────┘
          ▲              ▲
   登録/状態報告      タスク指示
   (SSMチャネル)    (ECSチャネル)
          │              │
┌─────────┴──────────────┴──────────┐
│         オンプレサーバー            │
│  ┌──────────┐    ┌──────────────┐ │
│  │ SSM Agent│    │ ECS Container│ │  両方必要
│  │(登録窓口) │    │   Agent      │ │
│  └──────────┘    │ （現場監督） │ │
│                  └──────┬───────┘ │
│                         │          │
│                  ┌──────▼───────┐ │
│                  │   Docker     │ │
│                  │   コンテナ群 │ │
│                  └──────────────┘ │
└────────────────────────────────────┘
```

## 比較表（一発理解用）

| 項目 | コントロールプレーン | SSM Agent | ECS Container Agent |
|---|---|---|---|
| 所在 | AWS内部 | 各ノード | 各ノード |
| 役割 | 司令塔 | AWSへの登録窓口 | コンテナ実行担当 |
| 管理者 | AWS | ユーザー（自動更新可） | ユーザー（自動更新可） |
| ECS専用？ | ECS専用 | **汎用**（SSM全体で使う） | ECS専用 |
| 通常ECS(EC2)で必要？ | あり | 不要（ECS Anywhereで必須） | 必要 |
| Fargateで必要？ | あり | 不要 | AWSが代行（ユーザー操作なし） |
| K8s類似 | kube-apiserver等 | （該当なし） | kubelet |

## SAA頻出ポイント

> [!warning] 試験ではエージェント名の取り違えが選択肢に混ざる
> - 「SSM Agent = AWSとの**通信路**を開く」
> - 「ECS Agent = **コンテナ起動**を実行する」
> - 「コントロールプレーン = **指示する側**（AWS管理）」

### よく出る設問パターン

| シナリオ | 正解 |
|---|---|
| オンプレサーバーをECS Anywhereで使うために必要なものは？ | **SSM Agent + ECS Container Agent + Docker** |
| AWSコンソールからオンプレにシェル接続したい | SSM Agent + Session Manager |
| ECSのマスターノードを自分でパッチ管理する必要があるか？ | **不要**（AWSが管理） |
| Fargateの利点は？ | コントロールプレーンも**ノードも**AWSが管理 |
