# オーケストレーションとフルマネージド

AWSのサービスを理解するうえで頻出する2つの概念。特にコンテナ・データベース・分析サービス等で繰り返し登場する。

---

## オーケストレーション（Orchestration）

**多数のコンポーネントを自動的に調整・制御する仕組み**のこと。

> [!info] 語源はオーケストラの「指揮者（コンダクター）」。楽器奏者全員に演奏を指示してまとめる役割から来ている。

### たとえ話：オーケストラの指揮者

| オーケストラ | コンテナ環境 |
|------------|------------|
| 楽器奏者 | コンテナ（アプリ） |
| 演奏タイミング | コンテナの起動・停止タイミング |
| ステージ配置 | どのサーバーで動かすか |
| 演奏者が倒れたら代役手配 | コンテナがクラッシュしたら自動再起動 |
| 指揮者 | オーケストレーター（ECS / EKS） |

指揮者がいないと全員バラバラに演奏してしまうように、オーケストレーターがいないとコンテナ管理は手動の地獄になる。

### オーケストレーターが担う仕事

```
コンテナが増えてくると管理が必要になること
├── どのサーバーで動かすか（スケジューリング）
├── クラッシュしたら自動再起動
├── アクセス増加時にコンテナを自動増加（スケールアウト）
├── デプロイ時に順番よく入れ替える（ローリングアップデート）
└── ヘルスチェックと状態監視
```

### AWSにおけるオーケストレーション例

| サービス | 何をオーケストレーションするか |
|---------|------------------------------|
| ECS | Dockerコンテナ |
| EKS | Kubernetesワークロード |
| Step Functions | Lambdaや各種AWSサービスの処理フロー |
| EMR | Hadoopクラスターの分散処理 |

---

## フルマネージド（Fully Managed）

**インフラの構築・運用・保守をAWSが全て担ってくれる**サービス形態のこと。

> [!tip] 「マネージド」だけのサービスもある。フルマネージドとの違いは管理の範囲。

### たとえ話：賃貸マンション vs 持ち家

| | 持ち家（セルフマネージド） | 賃貸（フルマネージド） |
|--|--------------------------|----------------------|
| 建物管理 | 自分で修繕 | 管理会社が対応 |
| 設備故障 | 自分で業者手配 | 管理会社が手配 |
| 初期コスト | 高い | 低い（月額払い） |
| カスタマイズ | 自由にリフォーム可 | 制限あり |
| 手間 | 大きい | ほぼゼロ |

### 管理レベルの比較

```
自分が管理する範囲
│
├─ セルフマネージド（EC2上に自前でDB構築など）
│   └─ OS / DB / パッチ / バックアップ / HA構成 → 全部自分
│
├─ マネージド（RDS）
│   └─ OS / パッチ / バックアップ → AWSが管理
│       インスタンスタイプ / スケール → 自分が管理
│
└─ フルマネージド（DynamoDB / Aurora Serverless / Fargate）
    └─ ほぼ全部AWSが管理。自分はデータ・クエリ・設定のみ
```

### AWSにおけるフルマネージドの例

| サービス | フルマネージドの内容 |
|---------|---------------------|
| ECS + Fargate | サーバー・OS・コンテナランタイムを全てAWS管理 |
| DynamoDB | サーバー・スケーリング・レプリケーションを全てAWS管理 |
| S3 | ストレージの物理管理・冗長化をAWS管理 |
| Lambda | サーバー・スケーリング・OS全てAWS管理 |

---

## 2つの概念の関係

```
フルマネージド ＋ オーケストレーション ＝ ECS（Fargate）

  フルマネージド：「AWSがインフラを管理してくれる」
  オーケストレーション：「コンテナの配置・管理・調整を自動化してくれる」
```

> [!warning] ECSはどちらの性質も持つ。一方だけを覚えると試験で混乱しやすい。

## SAA試験 頻出ポイント

### オーケストレーション関連

- 「コンテナを自動スケールさせたい」→ ECS や EKS のオーケストレーション機能
- 「複数Lambdaを順番に実行したい」→ Step Functions（Lambdaのオーケストレーション）
- 「Kubernetesのオーケストレーション管理をAWSに任せたい」→ EKS

### フルマネージド関連

- 「運用負荷を最小化したい」という選択肢が出たら**フルマネージドサービスを選ぶ**のが正解パターン
- EC2上に自前構築 vs フルマネージドサービスの比較問題は頻出
- フルマネージドはカスタマイズ性が下がる点がトレードオフ（問われることがある）

### よく出る選択問題の軸

| シナリオ | フルマネージド（推奨） | セルフマネージド |
|---------|----------------------|----------------|
| 運用コスト削減 | ◎ | △ |
| 細かいOS設定が必要 | △ | ◎ |
| 初期構築の速さ | ◎ | △ |
| 特殊なソフトウェア要件 | △ | ◎ |
