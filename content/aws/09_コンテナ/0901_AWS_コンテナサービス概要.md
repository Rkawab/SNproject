# AWS コンテナサービス完全ガイド

> [!info] SAA試験対策と実務理解のためのまとめノート。
> たとえ話を交えて、コンテナの基礎から AWS の主要サービスまで一気通貫で整理する。

---

## 1. そもそもコンテナとは？

### たとえ話：引っ越しの段ボール
アプリケーションを「引っ越し」と考える。

- **物理サーバー**: 家具を全部バラして運び、新居でまた組み立てる。新居の壁紙やコンセントの形が違うと使えない。
- **仮想マシン (VM)**: 家ごと持ち運ぶイメージ。確実だが、重くて遅い。
- **コンテナ**: 必要な家具と「壁紙・コンセント込み」の段ボールに梱包して運ぶ。どの家でも段ボールを開ければそのまま使える。

つまりコンテナは「アプリ + ライブラリ + 設定」を**ひとつのパッケージ**にして、どこでも同じように動かす技術。

### VM とコンテナの違い

| 観点 | VM | コンテナ |
|---|---|---|
| 含むもの | OS + アプリ | アプリのみ（OSはホストと共有） |
| 起動時間 | 分単位 | 秒単位 |
| サイズ | GB単位 | MB単位 |
| 例 | EC2インスタンス | Dockerコンテナ |

> [!note] Docker の登場で、コンテナはアプリ配布の標準形式になった。AWS はこの Docker コンテナを動かすための一連のサービスを提供している。

## 2. AWS コンテナサービスの3層構造

AWS のコンテナ関連サービスは、役割で3層に分かれる。

```
┌────────────────────────────────────────┐
│ ① レジストリ層（イメージ保管庫）         │
│   → ECR                                 │
├────────────────────────────────────────┤
│ ② オーケストレーション層（管理者）       │
│   → ECS / EKS                           │
├────────────────────────────────────────┤
│ ③ コンピューティング層（実行場所）       │
│   → EC2 / Fargate                       │
└────────────────────────────────────────┘
```

### たとえ話：宅配サービス
- **レジストリ (ECR)** = 倉庫。商品（コンテナイメージ）を保管。
- **オーケストレーション (ECS/EKS)** = 配車管制センター。「どの荷物をどのトラックに、いつ何台必要か」を管理。
- **コンピューティング (EC2/Fargate)** = 実際のトラック（実行環境）。

---

## 3. レジストリ層：ECR

### Amazon ECR (Elastic Container Registry)
コンテナイメージを保管・配布するためのプライベートレジストリ。
Docker Hub のプライベート版 + AWS 統合版だと思えば OK。

**特徴**
- IAM で細かいアクセス制御
- イメージの脆弱性スキャン機能
- ECS / EKS と密接に統合
- **ECR Public**: 公開リポジトリ（Docker Hub の代替）

**SAAポイント**
- イメージのプル元は同一リージョンが基本（リージョン間レプリケーションも可）
- ライフサイクルポリシーで古いイメージを自動削除可能

関連: [[0601_IAM基礎]] / [[0201_S3]]（イメージ実体は裏で S3 に保存される）

## 4. オーケストレーション層：ECS vs EKS

### Amazon ECS (Elastic Container Service)
AWS 独自のコンテナオーケストレーター。**シンプル・AWS 統合が深い**。

- AWS サービス（ALB, IAM, CloudWatch）と最初から繋がっている
- 学習コストが低い
- AWS 外への移植性は低い

### Amazon EKS (Elastic Kubernetes Service)
業界標準の Kubernetes (k8s) をマネージドで提供。

- マルチクラウド / オンプレに移植可能
- k8s エコシステム（Helm、Istio 等）が使える
- 学習コスト高、運用も複雑

### たとえ話：会社の組織編成
- **ECS** = 自社専用の業務フロー。シンプルで自社サービスとの連携抜群、でも転職時に通用しにくい。
- **EKS** = ISO 規格の業務フロー。世界標準で他社にも通用、でも覚えることが多い。

### SAA 試験での判断軸

| 要件 | 選ぶべきもの |
|---|---|
| AWS にロックインしてもOK、シンプルが正義 | **ECS** |
| 既に k8s を使っている / マルチクラウド戦略 | **EKS** |
| Docker を単発で動かしたいだけ | EC2 に Docker 直接 |

## 5. コンピューティング層：EC2 vs Fargate

オーケストレーター (ECS/EKS) に対して「**コンテナをどこで動かすか**」を指定する選択肢。

### 起動タイプ：EC2
- 自分で EC2 インスタンスを管理（パッチ、スケーリング）
- インスタンスタイプを細かく選べる
- **コスト安**だが運用負荷あり
- GPU や特殊なインスタンスを使いたい時に必須

### 起動タイプ：Fargate
- **サーバーレス**：EC2 を意識しない
- CPU / メモリだけ指定すれば AWS が裏で動かしてくれる
- 運用ゼロ、ただし単価はやや高め
- 短時間ジョブやイベント駆動に最適

### たとえ話：レンタカー vs タクシー
- **EC2 起動タイプ** = レンタカー。安いが運転も給油も自分。
- **Fargate** = タクシー。乗るだけ。少し高いが手間ゼロ。

### SAA 頻出パターン

| シナリオ | 推奨 |
|---|---|
| 運用負荷を下げたい | **Fargate** |
| トラフィックが予測不能、急増する | **Fargate** |
| 長時間・大量・コスト最優先 | **EC2 起動タイプ** |
| GPU / 特殊ハードが必要 | **EC2 起動タイプ** |

関連: [[0101_EC2やVPCとは]] / [[0703_AWS_Batch]]

## 6. その他の関連サービス

### AWS App Runner
**Web アプリ専用の超簡単サービス**。ソースコードかコンテナイメージを指定するだけで、HTTPS エンドポイント込みで公開できる。
- 内部は Fargate ベース
- 「Heroku みたいに使いたい」需要に対応

### AWS Copilot
ECS へのデプロイを簡単にする CLI ツール。`copilot init` で雛形作成からデプロイまで一気通貫。

### AWS Proton
コンテナ・サーバーレスのテンプレート管理。大企業のプラットフォームチーム向け。

### Amazon Lightsail Containers
シンプル & 固定料金でコンテナを動かせる入門サービス。検証や小規模利用向け。

## 7. 全体の選択フロー (SAA向け)

```
コンテナを動かしたい
│
├─ とにかく簡単に公開したい
│   └→ App Runner
│
├─ オーケストレーションが必要
│   │
│   ├─ k8s でなくてもいい？
│   │   └→ ECS
│   │       ├─ 運用ラクに → Fargate
│   │       └─ コスト最優先 → EC2
│   │
│   └─ k8s 必須（マルチクラウド等）
│       └→ EKS
│           ├─ Fargate
│           └─ EC2
│
└─ イメージ保管
    └→ ECR
```

## 8. SAA 試験で押さえる定番ポイント

> [!tip] 試験のお決まりパターン
> - 「運用負荷を最小化」「サーバー管理したくない」→ **Fargate**
> - 「Kubernetes を使い続けたい」「オンプレからの移行」→ **EKS**
> - 「最も簡単に Web アプリをデプロイ」→ **App Runner**
> - 「コンテナイメージの脆弱性スキャン」→ **ECR**
> - 「既存 ECS タスクを別リージョンへ」→ **ECR のレプリケーション + ECS 再デプロイ**

> [!warning] よくある引っかけ
> - 「k8s 互換が必須」と書いてあれば ECS は不正解 → **EKS**
> - 「EC2 のパッチ管理から開放」とあれば → **Fargate**
> - 「Lambda で動かせる」は 15 分超や常駐用途では不適 → コンテナサービスを選ぶ
