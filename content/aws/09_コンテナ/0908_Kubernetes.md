# Kubernetes（K8s）入門

Kubernetes（クバネティス / 略称K8s）とは、**大量のコンテナを自動で管理してくれる司令塔システム**。Googleが社内で使っていたBorgをOSS化したもので、現在はCNCF（Cloud Native Computing Foundation）が管理する**コンテナオーケストレーションのデファクトスタンダード**。

---

## なぜ必要になったのか

### Dockerだけだと困ること

| 規模 | 何が起こるか |
|------|-------------|
| コンテナ1〜数個 | 手動で `docker run` でも問題ない |
| コンテナ数十〜数百個 | 落ちたら？増やすには？どこに置く？が破綻 |

K8sは「コンテナを大量に・自動で・宣言した状態に保つ」ためのレイヤー。

---

## たとえ話：宅配ピザチェーン店の本部

| 概念 | ピザチェーンのたとえ |
|------|---------------------|
| **コンテナ** | ピザ職人1人（仕事を実行する単位） |
| **Pod** | 「職人＋助手」のチーム（一緒に動く最小単位） |
| **Node** | 1店舗（職人が働く場所＝サーバー） |
| **クラスター** | チェーン全体（全店舗の集合） |
| **コントロールプレーン** | 本部の指令室 |
| **Deployment** | 「マルゲリータ職人を常に10人配置せよ」という方針書 |
| **Service** | 注文窓口（お客さんはどの職人が作るか意識しない） |
| **マニフェスト（YAML）** | 本部に出す「こうあるべき」計画書 |

→ あなたは本部にYAMLで「マルゲリータ職人10人欲しい」と渡すだけ。  
   本部（K8s）が「どの店舗に何人配置するか」「倒れたらすぐ補充」を**自動でやる**。

---

## 最大の特徴：「宣言的（Declarative）」な仕組み

### 命令的（普通のスクリプト）
```bash
# 1. サーバーAでコンテナ起動
# 2. 落ちたら検知して再起動
# 3. 負荷上がったら別サーバーでも起動
# ... 全部自分で書く
```

### 宣言的（Kubernetes）
```yaml
# 「Webアプリのコンテナを常に3個動かせ」
replicas: 3
image: my-web:v1.0
```
→ あとは**K8sが現在の状態を観察し、3個になるよう自動調整し続ける**

> [!tip] これを「**Desired State（あるべき状態）**を宣言し、K8sが現実をそれに合わせる」と言う。**コントロールループ**という仕組みで常時監視・補正している。

---

## 主要パーツ

### コントロールプレーン（本部の指令室）

| 部品 | 役割 |
|------|------|
| **API Server** | 全ての司令塔。kubectlからの命令を受ける窓口 |
| **etcd** | 「あるべき状態」と「現在の状態」を保存するキーバリューDB |
| **Scheduler** | 「新しいPodをどのNodeに置くか」を決定 |
| **Controller Manager** | 状態を監視して足りなければ補充・余れば削除 |

### ノード（実際にコンテナが動く場所）

| 部品 | 役割 |
|------|------|
| **kubelet** | ノード上の管理エージェント（本部からの指示を実行） |
| **kube-proxy** | ネットワーク通信・サービス検出を制御 |
| **コンテナランタイム** | 実際にコンテナを動かす（containerd / CRI-O 等） |

---

## 中核オブジェクト

| オブジェクト | 役割 |
|--------------|------|
| **Pod** | 1つ以上のコンテナをまとめた最小実行単位 |
| **ReplicaSet** | Podを指定数維持する |
| **Deployment** | ReplicaSetをラップしてローリングアップデート等を提供 |
| **Service** | Podへの安定したアクセスポイント（IP・DNS） |
| **Ingress** | 外部からのHTTP/HTTPS L7ルーティング |
| **ConfigMap / Secret** | 設定値・機密情報をPodに注入 |
| **PersistentVolume** | 永続ストレージ |
| **Namespace** | クラスター内の論理分割 |
| **DaemonSet** | 全Nodeに1つずつPodを配置（ログ収集等） |
| **Job / CronJob** | バッチ処理・定期実行 |

---

## K8sができること（自動で）

1. **自動再起動（Self-Healing）** — コンテナが落ちたら即座に新Pod起動
2. **オートスケール（HPA）** — CPU/メモリ使用率に応じてPod数を自動増減
3. **ローリングアップデート** — サービスを止めずにバージョン更新
4. **ロードバランシング** — 複数Podへ自動で負荷分散
5. **自己修復** — Nodeごと落ちても別Nodeで再スケジュール
6. **シークレット管理** — パスワード・トークンを安全に注入
7. **サービスディスカバリ** — Pod同士が名前で通信できる

## DockerとKubernetesの関係

```
Docker / containerd       ← コンテナを「動かす」エンジン
   ↑
Kubernetes                ← コンテナを「大量に管理する」司令塔
   ↑
EKS / GKE / AKS           ← Kubernetesを「マネージド提供」するサービス
```

> [!info] Docker単体だとPC1台分の話。Kubernetesは「サーバー何十台にまたがるコンテナ群」を扱うレイヤー。

## マニフェスト例（Deployment）

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3                  # ← 常に3個維持
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: my-web:v1.0
        ports:
        - containerPort: 80
```

```bash
kubectl apply -f deployment.yaml   # ← 宣言を投入
kubectl get pods                   # ← 3つのPodが立ち上がる
```

## なぜマネージドK8s（EKS等）が必要か

Kubernetes本体は強力だが、**自前運用は重い**：

- コントロールプレーンをHA構成で組む（EC2 3台でetcdクラスター…）
- 内部TLS証明書の自動更新
- 年3〜4回出るK8sバージョンアップへの追従
- ノードのAMI更新・パッチ管理

→ AWSがコントロールプレーンを完全代行してくれるのが [[0906_EKS]]。

## マネージドK8sサービス比較

| 提供元 | サービス名 |
|--------|-----------|
| AWS | **Amazon EKS** |
| Google Cloud | GKE（Google Kubernetes Engine） |
| Azure | AKS（Azure Kubernetes Service） |

> [!tip] K8s自体は標準化されているため、マニフェストは大体どのクラウドでも動く。これが「マルチクラウド対応」の根拠。
