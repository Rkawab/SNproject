# Amazon EKS（Elastic Kubernetes Service）

Amazon EKSとは、**AWSがフルマネージドで提供するKubernetes（K8s）サービス**。Kubernetesのコントロールプレーン（マスターノード）部分をAWSが完全に管理してくれるため、ユーザーはコンテナワークロードの設計・運用だけに集中できる。

---

## たとえ話：マンションの管理

| 概念 | マンション運営のたとえ |
|------|------------------------|
| Kubernetes | マンション全体の運営ルール・仕組み |
| コントロールプレーン | 管理人室（誰がどの部屋に住み、トラブル対応する） |
| ノード（ワーカー） | 各部屋（実際に人＝コンテナが住む場所） |
| Pod | 部屋の中の家族（一緒に暮らすコンテナのグループ） |
| EKS | **管理人を完全外注**したマンション |
| EKSセルフマネージドK8s | 自分で管理人もやる従来運用 |

→ EKSは「管理人の手配・採用・教育・夜勤シフトを全部やってくれるサービス」というイメージ。

---

## Kubernetesとは（前提知識）

Kubernetesは**コンテナオーケストレーションのOSSデファクトスタンダード**。Googleが社内で使っていたBorgをOSS化したもの。

- 多数のコンテナを「どこに配置し、何個動かし、障害時どう復旧するか」を自動制御
- マニフェスト（YAML）で「あるべき状態」を宣言 → K8sがその状態を維持
- どのクラウド・オンプレでも動くため**ベンダーロックインを避けられる**

> [!info] K8sは強力だが、自前運用は「コントロールプレーンHA構成・etcdバックアップ・バージョンアップグレード・証明書更新」など運用負荷が極めて高い。これをAWSが代行するのがEKS。

## ECSとの違い（SAA頻出）

| 比較項目 | Amazon ECS | Amazon EKS |
|----------|-----------|-----------|
| オーケストレーター | AWS独自実装 | Kubernetes（OSS標準） |
| 学習コスト | 低い | 高い（K8s知識必須） |
| マルチクラウド移行 | 困難（AWS依存） | 容易（K8sはどこでも動く） |
| クラスター料金 | 無料 | $0.10/時間 |
| エコシステム | AWSサービス連携重視 | Helm/Istio/ArgoCD等OSS資産が豊富 |
| 適したケース | AWSのみで完結する案件 | 既存K8s資産あり・マルチクラウド |

> [!tip] **SAA選択キーワード**
> - 「Kubernetesの既存資産」「マルチクラウド」「ハイブリッド」「OSS準拠」 → **EKS**
> - 「シンプル」「AWSに閉じた運用」「学習コスト低く」 → **ECS**

関連: [[0904_ECS]] / [[0901_AWS_コンテナサービス概要]]

## 起動オプション

| 種類 | ノード管理 | 特徴・適したケース |
|------|------------|--------------------|
| **EKS on EC2（マネージドノードグループ）** | AWSがEC2のプロビジョニング・ライフサイクル管理 | バランス型・最も標準的 |
| **EKS on Fargate** | 完全サーバーレス | ノード意識ゼロ・Podごとに課金 |
| **セルフマネージドノードグループ** | 自分でEC2を構築・登録 | 最大の柔軟性（カスタムAMI/GPU等） |
| **EKS Anywhere** | オンプレ実行 | データセンター内でEKS互換 |
| **EKS Distro** | OSSとして配布 | 任意の場所で構築可能 |

> [!tip] Fargateを選べばコントロールプレーンもノードもAWS管理になり、運用負荷はほぼゼロになる。ただしDaemonSetや特権Pod等、Fargateで制限される機能があるため要件確認が必要。

関連: [[0905_ECS_Anywhere]] / [[0902_オーケストレーション_フルマネージド]]

## コントロールプレーンとノードの責任範囲

| サービス | コントロールプレーン | ノード（サーバー） |
|----------|----------------------|--------------------|
| ECS + Fargate | AWS管理 | AWS管理 |
| ECS + EC2 | AWS管理 | **自分が管理** |
| **EKS + Fargate** | AWS管理 | AWS管理 |
| **EKS + マネージドノードグループ** | AWS管理 | AWS部分管理（AMI更新は自分） |
| **EKS + セルフマネージド** | AWS管理 | **自分が管理** |
| EC2に自前K8s構築 | **自分が管理** | **自分が管理** |

EKSの最大の価値は **「Kubernetesのコントロールプレーンを絶対にAWSが管理する」** という一点。これが自前運用との決定的な違い。

## 主要コンポーネント（K8s共通）

| 用語 | 説明 |
|------|------|
| **クラスター** | EKSの管理単位。コントロールプレーン+ノード群の集合 |
| **Pod** | コンテナの最小実行単位（1つ以上のコンテナをまとめたもの） |
| **Node** | Podを動かすワーカー（EC2 or Fargate） |
| **Deployment** | Podの数・ローリングアップデート方法を管理 |
| **Service** | Podへのアクセス入口（ClusterIP / NodePort / LoadBalancer） |
| **Ingress** | L7ルーティング（ALB Ingress Controllerと連携可） |
| **kubectl** | クラスターを操作するCLIツール |
| **マニフェスト（YAML）** | 「あるべき状態」を宣言する設定ファイル |

## AWSサービスとの連携

| 連携先 | 役割 |
|--------|------|
| **IAM Roles for Service Accounts（IRSA）** | Pod単位でIAMロールを付与 |
| **AWS Load Balancer Controller** | Ingress→ALB、Service→NLB を自動構築 |
| **Amazon ECR** | コンテナイメージのレジストリ |
| **VPC CNI** | PodにVPCのENIを直接割り当て（IP管理） |
| **CloudWatch Container Insights** | メトリクス・ログ収集 |
| **AWS Fargate** | サーバーレスでPod実行 |

## いつEKSを選ぶか（判断基準）

✅ **EKS推奨**
- 既にK8sの知識・マニフェスト資産がある
- オンプレ・GCP・Azureとマルチクラウドで揃えたい
- Helm / Istio / ArgoCD / Prometheus等のK8sエコシステムを活用したい
- ベンダーロックインを避けたい

❌ **EKSは過剰**
- AWSに閉じた小〜中規模のWebアプリ → **ECS + Fargate**で十分
- K8sの学習コストを払えない / 払う必要がない
- クラスター料金（$0.10/h × クラスター数）が無視できないコスト感

---

## SAA試験で押さえるポイント

1. **「マルチクラウド」「既存K8s資産」「OSS準拠」** のキーワード → EKS
2. **コントロールプレーンはAWSが必ず管理** する（自前運用との差）
3. **Fargate起動タイプ** を選べばノードも完全マネージド
4. **EKS Anywhere** = オンプレでEKS互換運用
5. **IRSA** = Pod単位のIAM権限（セキュリティ問題で出題）
6. **ECSとの料金差**：EKSはクラスター単位で課金される
