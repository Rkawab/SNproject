# Amazon ECR（Elastic Container Registry）

> [!info] SAA試験対策と実務理解のためのまとめノート。
> コンテナイメージ管理の基礎から、ECRの主要機能・SAA頻出パターンまで整理する。

---

## 1. ECR が必要な理由

**ECR(Elastic Container Registry)とは、コンテナイメージを保管・配布するためのAWSマネージドなプライベートレジストリ(保管庫)。** なぜ専用の保管庫が要るのかを、料理にたとえて見てみる。

### たとえ話：料理レシピと冷蔵庫

- コンテナイメージ = **料理のレシピ＋食材セット**（Dockerfileをビルドしたもの）
- コンテナ実行環境 (ECS/EKS) = **キッチン**（料理を実際に作る場所）
- **ECR = 冷蔵庫**。レシピと食材を鮮度よく保管し、どのキッチンからでも取り出せる

キッチン（ECS/EKS）は「冷蔵庫（ECR）からイメージを pull して料理（コンテナ起動）する」という流れ。

---

## 2. 基本構造

```
┌─────────────────────────────────────────┐
│           Amazon ECR                    │
│                                         │
│  ┌──────────────────────────────────┐   │
│  │  リポジトリ（Repository）         │   │
│  │  例: myapp/backend               │   │
│  │                                  │   │
│  │  ┌──────────┐ ┌──────────┐       │   │
│  │  │ v1.0.0   │ │ v1.1.0   │ ...   │   │
│  │  │(イメージ)│ │(イメージ)│       │   │
│  │  └──────────┘ └──────────┘       │   │
│  └──────────────────────────────────┘   │
│                                         │
│  ※イメージの実体は裏で S3 に保管される   │
└─────────────────────────────────────────┘
```

| 用語 | 意味 | たとえ |
|---|---|---|
| レジストリ | ECR全体（アカウント単位） | 図書館全体 |
| リポジトリ | イメージの種別ごとの棚 | 「料理本」コーナー |
| イメージ | ビルド済みのコンテナ | 1冊の本 |
| タグ | バージョン識別子（`latest`、`v1.0`等） | 版番号（第1版・第2版） |

## 3. 基本的な使い方の流れ

```
①ビルド   ②タグ付け   ③プッシュ   ④プル（ECS等から）
docker    docker       docker       docker
build     tag          push         pull
  ↓         ↓            ↓             ↓
ローカル →→→→→→→→→→→→ ECR →→→→→→→→ ECS/EKS
```

**コマンド例：**

```bash
# 1. ECRへの認証（トークン有効期限: 12時間）
aws ecr get-login-password --region ap-northeast-1 | \
  docker login --username AWS --password-stdin \
  <account_id>.dkr.ecr.ap-northeast-1.amazonaws.com

# 2. イメージをビルド
docker build -t myapp .

# 3. ECR用にタグ付け
docker tag myapp:latest \
  <account_id>.dkr.ecr.ap-northeast-1.amazonaws.com/myapp:latest

# 4. プッシュ
docker push <account_id>.dkr.ecr.ap-northeast-1.amazonaws.com/myapp:latest
```

## 4. 主要機能

### 4-1. 認証・アクセス制御（IAM）

**たとえ話：社員証とセキュリティゲート**

ECRへのアクセスはIAMで完全制御される。2種類のポリシーがある。

| ポリシー種別 | 何を制御するか | 用途 |
|---|---|---|
| **IAMポリシー** | 「誰がECRを操作できるか」（push/pull権限など） | 基本のアクセス制御 |
| **リポジトリポリシー** | 「このリポジトリに誰がアクセスできるか」 | クロスアカウントアクセスなど |

> [!tip] SAA頻出
> 別AWSアカウントのECSからECRのイメージをpullしたい
> → **リポジトリポリシー** でクロスアカウントアクセスを許可する

認証トークンは **12時間有効**（docker loginで取得するトークン）。

---

### 4-2. イメージスキャン（脆弱性検出）

**たとえ話：食品の品質検査**

コンテナイメージに危険なライブラリ（CVE）が含まれていないかスキャンする。

| スキャン種別 | タイミング | 特徴 |
|---|---|---|
| **ベーシックスキャン** | push時 or 手動 | OSレベルのCVEを検出（無料） |
| **拡張スキャン** | push時 + 継続監視 | OS＋アプリライブラリのCVEを検出（Amazon Inspector連携・有料） |

> [!tip] SAA頻出
> 「セキュリティ要件でコンテナイメージの脆弱性を自動検出したい」
> → **ECRの拡張スキャン（Amazon Inspector連携）**

---

### 4-3. ライフサイクルポリシー（古いイメージの自動削除）

**たとえ話：冷蔵庫の賞味期限管理**

古いイメージを放置するとストレージコストが積み上がる。ルールを設定して自動削除できる。

```json
{
  "rules": [{
    "rulePriority": 1,
    "description": "30日以上前のuntaggedイメージを削除",
    "selection": {
      "tagStatus": "untagged",
      "countType": "sinceImagePushed",
      "countUnit": "days",
      "countNumber": 30
    },
    "action": { "type": "expire" }
  }]
}
```

**よくある設定パターン：**
- `latest` タグ以外を10個まで保持
- 90日経過したイメージを削除
- `prod-` タグのついたイメージは削除しない

---

### 4-4. クロスリージョンレプリケーション

**たとえ話：チェーン店の在庫共有**

本社倉庫（東京リージョン）にある商品を、大阪や海外倉庫に自動コピーする。DR構成やグローバル展開に使う。

```
東京 ECR → （自動レプリケーション） → 大阪 ECR
                                       → バージニア ECR
```

> [!tip] SAA頻出
> 「DR構成でマルチリージョンにコンテナを展開したい」
> → **ECRのクロスリージョンレプリケーション**

---

### 4-5. ECR Public（公開リポジトリ）

**Docker Hub の AWS 版**。OSSツールや公式イメージを公開リポジトリに置くことができる。

| | ECR Private | ECR Public |
|---|---|---|
| アクセス | IAM認証が必要 | 誰でも pull 可能 |
| 用途 | 社内・自社アプリ用 | OSS公開、ベースイメージ配布 |
| URL | `<account>.dkr.ecr.<region>.amazonaws.com` | `public.ecr.aws` |

---

## 5. コスト感

| 費用対象 | 概要 |
|---|---|
| **ストレージ** | 50GB/月まで無料枠あり、超過分は $0.10/GB |
| **データ転送（IN）** | 無料（push） |
| **データ転送（OUT）** | 同一リージョン内の ECS/EKS へはほぼ無料 |

> [!note] 同一リージョンの ECS から pull する限り転送コストはほぼかからない設計になっている。

## 6. 他サービスとの連携

```
S3           ← ECRのイメージ実体が裏側で保存されている（意識不要）
IAM          ← ECRへのアクセス制御
CloudTrail   ← ECR操作の監査ログ
CloudWatch   ← push/pullのメトリクス監視
Amazon Inspector ← 拡張スキャンのバックエンド
ECS / EKS    ← イメージを pull して実行
CodeBuild    ← CI/CDでイメージをビルドし ECR に push
CodePipeline ← ビルド→push→デプロイの自動化パイプライン
```

**CI/CDの典型構成：**
```
GitHub → CodeBuild（docker build & push to ECR）→ ECS（新イメージでデプロイ）
```

## ECR と AWS CodeArtifact の違い

名前が似ているが、保管するものが違う。

| サービス | 保管するもの | 典型用途 |
|---|---|---|
| **Amazon ECR** | Docker/OCIコンテナイメージ | ECS/EKS/Fargateがpullして実行 |
| **AWS CodeArtifact** | npm / pip / Maven などのパッケージ | アプリのビルド時に依存ライブラリを取得 |

> [!tip] SAAひっかけ
> 「コンテナイメージをECSタスク定義から参照」なら **ECR**。
> CodeArtifactはパッケージリポジトリで、ECSがコンテナイメージを直接pullするレジストリではない。

---

## 7. SAA試験 頻出パターン

> [!tip] 試験のお決まりパターン
> - 「コンテナイメージを保管・管理したい」→ **ECR**
> - 「イメージの脆弱性を自動検出したい」→ **ECR 拡張スキャン（Inspector連携）**
> - 「古いイメージを自動削除したい」→ **ECR ライフサイクルポリシー**
> - 「別アカウントのECSからpullしたい」→ **ECR リポジトリポリシー（クロスアカウント）**
> - 「マルチリージョンでイメージを使いたい」→ **ECR クロスリージョンレプリケーション**
> - 「CI/CDにpush権限を付与したい」→ **IAMロール（ecr:PutImage 等）**

> [!warning] よくある引っかけ
> - イメージスキャンを「CloudWatch で設定する」は誤り → **ECR または Inspector**
> - 認証トークンの有効期限を「永続」と思い込まない → **12時間**
> - ECR Public と Private の違いを混同しない
