# CloudFront と Global Accelerator ★SAA超頻出

> [!note] このドキュメントについて
> AWS SAA 模擬試験の出題から、CloudFront vs Global Accelerator の違い、複数オリジン設定による Webサイト配信の実装パターンをまとめた解説です。

---

## 概要：CloudFront vs Global Accelerator

**同じ「グローバル低遅延」だが、使い道は全く違う。**

| 観点 | **CloudFront** | **Global Accelerator** |
|---|---|---|
| **役割** | **HTTP/HTTPSコンテンツ配信・キャッシュ** | **TCP/UDPアプリの高速化・固定IP** |
| **何をキャッシュ？** | ✅ Webサイト・画像・動画・HTML | ❌ キャッシュしない（パススルー） |
| **オリジンの例** | S3、ALB、EC2、オンプレ | EC2、ALB、オンプレのアプリ |
| **適用対象** | **Webサイト、API、ストリーミング** | **ゲーム、IoT、金融トレーディング** |
| **固定IP** | ❌ エッジロケーションは可変 | ✅ 固定IP提供 |
| **試験での出現** | **超頻出**（Webサイト配信） | **頻出**（非HTTPアプリ） |

---

## CloudFront 完全ガイド

### 「CloudFront＝世界各地に置いた支店」

> **たとえ：** ニュースサイトを世界中の読者に届ける出版社を想像してください。
>
> - **CloudFront＝世界各地に置いた"支店（エッジロケーション）"**
>   - **静的コンテンツ（画像・動画＝S3）**：人気ページは支店にコピーを置いて即配る（キャッシュ）。本社まで取りに行かない＝速い。
>   - **動的コンテンツ（記事＝ALB/EC2）**：毎回変わるのでコピーはできないが、読者は近くの支店に入り、支店から本社へ AWS 専用の高速回線で取り次ぐ。公衆網より速い＝動的も加速。
>   - この「静的はキャッシュ、動的は加速取次」を**1つのCloudFront**に、**オリジンをS3とALBの2つ**登録するだけで実現。

### CloudFront の3つの仕事

#### 1. **キャッシュ** — 静的コンテンツを世界中に複製

```
[ユーザー（東京）] 
   ↓ リクエスト
[CloudFront エッジ（東京）]
   ├─ キャッシュあり？ → 即座に返す ✅ 高速
   └─ キャッシュなし？ → オリジン（S3）に取りに行く
```

- TTL（Time To Live）を設定して「何秒キャッシュを保つか」を制御
- S3、ALB、EC2 など複数のオリジンから同じリクエストに対して返す内容を振り分け可能

#### 2. **加速取次** — 動的コンテンツを高速に取り次ぐ

```
[ユーザー（シドニー）]
   ↓ リクエスト
[CloudFront エッジ（シドニー）]
   ├─ キャッシュ対象外（動的）
   ├─ AWS 専用の高速ネットワーク経由で本社へ
   └─ オリジン（ALB/EC2）からレスポンス取得
```

- CloudFront は単なるキャッシュではなく、**動的コンテンツを高速に運ぶ運送業者**
- 公衆インターネットより遅延が少ない専用回線を使用

#### 3. **DDoS対策・セキュリティ** — AWS WAF、CloudFormation と組み合わせ

- CloudFront の前に **AWS WAF** をアタッチして不正アクセスをブロック
- オリジン（S3、ALB）への**直アクセスを制限**して、必ず CloudFront 経由にする

## 複数オリジン構成 — Web サイト配信の教科書

### パターン：CloudFront + S3 + ALB

```
          [Route 53]
         (独自ドメイン)
              ↓
          [CloudFront]
              ↓
         ┌────┴────┐
         ↓         ↓
      [S3]      [ALB]
     静的        動的
   (画像/CSS)  (HTML/API)
```

### オリジンの使い分けルール

| オリジン | 役割 | 設定例 |
|---|---|---|
| **S3** | 画像・CSS・JavaScript などの**静的ファイル** | パス例：`/images/*`、`/css/*` |
| **ALB** | 動的な HTML・API など**生成コンテンツ** | パス例：`/api/*`、`/article/*` |

### CloudFront に複数オリジンを登録する手順（概要）

1. **CloudFront ディストリビューションを作成**
2. **Origin 1**：S3 バケット を指定
   - キャッシュ有効（TTL = 86400秒 など）
   - パスパターン：`/images/*`、`/css/*`
3. **Origin 2**：ALB を指定
   - キャッシュなし or 短い TTL
   - パスパターン：`/api/*`、`/article/*`
4. **Route 53 でエイリアスレコード** を CloudFront CNAME に向ける
   - 例：`example.com` → CloudFront ディストリビューションドメイン
5. **完成**：ユーザーが `example.com` にアクセス
   - `/images/logo.png` → S3 経由（キャッシュ有効）
   - `/api/users` → ALB 経由（動的・キャッシュなし）

## Global Accelerator とは何か

### 「Global Accelerator＝非 HTTP アプリの高速受付カウンター」

> **たとえ：** 高速トレーディングアプリを使う金融ディーラーを想像。
>
> - **固定の受付IPを提示**：いつもここに送ればいい（IP切り替え不要）
> - **世界のどこからでも、最短の専用ルートで本社へ**：TCP/UDP ベースのアプリ通信を加速
> - **キャッシュはしない**：毎回オリジンに問い合わせ（リアルタイム性が必須）

### CloudFront との決定的違い

- **CloudFront**：HTTP/HTTPS のコンテンツを **キャッシュして配信**
  - 向き先：**Web サイト・画像・動画・CDN 用途**
- **Global Accelerator**：TCP/UDP アプリを**キャッシュせず加速**
  - 向き先：**ゲーム・IoT・金融・リアルタイム双方向通信**
  - 固定 IP が必要なシーン

### Global Accelerator の主な特徴

| 特徴 | 説明 |
|---|---|
| **固定IP** | グローバルに固定 IP 2 個を提供。世界どこからアクセスしても同じ IP |
| **キャッシュなし** | パススルー。常にオリジンに問い合わせ |
| **TCP/UDPサポート** | HTTP/HTTPS 限定でない |
| **複数リージョン対応** | オリジンを複数リージョンの EC2/ALB に分散可能 |

## SAA 頻出パターン：正解の見分け方

### ✅ **正解：CloudFront の複数オリジン**

> **シーン**：「Webサイト（HTML+CSS+画像）を世界中へ、低遅延で配信したい」

**正解の選択肢の特徴：**
- **CloudFront**が登場
- **オリジンが複数**（S3 + ALB の組み合わせ）
- **Route 53 で CloudFront へ向ける**
- 「単一の CDN で静的・動的を統合」という構成

**例：**
```
Route 53 → CloudFront（S3オリジン + ALBオリジン）
```

### ❌ **誤り：Global Accelerator を CloudFront の前に置く**

> **シーン**：SAA 試験で見かける**多段の罠**
> 
> 「CloudFront のさらに前に Global Accelerator を置く」構成

**なぜ誤りか：**
- CloudFront 自身が世界中のエッジロケーションを持つ
- その前に Global Accelerator を重ねる = **無駄な二重化**
- Global Accelerator はキャッシュしないので、CloudFront の高速キャッシュの価値を活かせない
- 「サービス名をたくさん並べた豪華そうな選択肢」の典型的な罠

```
❌ 誤り：
[Global Accelerator] → [CloudFront] → [S3/ALB]
      （不要な前置き）
```

### ❌ **誤り：ALB/EC2 を世界各地に展開**

> **シーン**：「複数リージョンに ALB を展開 + CRR + 地理的近接性ルーティング」

**なぜ誤りか：**
- **莫大なコスト**（各リージョンでインスタンス起動）
- **運用負荷**が非常に高い
- CloudFront なら 1 つの設定で世界中対応できるのに、過剰設計

## 実装上のベストプラクティス

### 1. **Route 53 + CloudFront 設定**

```
# Route 53 にエイリアスレコードを作成
ドメイン：example.com
タイプ：A レコード（CloudFront）
ターゲット：cloudfront-xxx.cloudfront.net
```

### 2. **CloudFront でパスベースのルーティング**

```
CloudFront ディストリビューション内：

Behavior 1：
  パスパターン：/images/*
  オリジン：S3
  キャッシュ有効

Behavior 2：
  パスパターン：/api/*
  オリジン：ALB
  キャッシュなし（または短い TTL）

Behavior 3（デフォルト）：
  パスパターン：/*
  オリジン：ALB
  キャッシュなし
```

### 3. **S3 オリジンの注意点**

- **CloudFront を使わない場合**：S3 ウェブサイトホスティングを許可
- **CloudFront を使う場合**：S3 への直アクセスを制限し、CloudFront 経由のみにする
  - Origin Access Identity (OAI) または Origin Access Control (OAC) を設定

## まとめ表：SAA 試験の着眼点

| 試験で問われる局面 | 正解 | 理由 |
|---|---|---|
| **「Webサイト(静的+動的)を世界中へ、低遅延」** | CloudFront（複数オリジン） | HTTP コンテンツのキャッシュ・加速が目的 |
| **「UDP ゲームアプリを固定 IP で加速」** | Global Accelerator | キャッシュ不要、固定 IP 必要 |
| **「API の DDoS 対策も含めて」** | CloudFront + AWS WAF | Web トラフィックの保護 |
| **「複数リージョンに ALB 展開」** | ❌ 過剰設計 | CDN で足りるのに不要 |
| **「CloudFront の前に GA」** | ❌ 無駄な二重化 | CloudFront は既に世界中エッジを持つ |

---

> [!note] つながり
> **505 第3問（UDP・固定IP → Global Accelerator）** と対で覚えるのが強いです。
>
> **同じ「グローバル低遅延」でも**
> - **HTTP Webコンテンツの配信** = **CloudFront**
> - **UDP・固定IPのアプリ高速化** = **Global Accelerator**
>
> 中身が「Webコンテンツか非HTTPアプリか」で分岐します。これが SAA の超重要分岐点。
