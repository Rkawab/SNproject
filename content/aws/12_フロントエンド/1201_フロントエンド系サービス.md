# AWS フロントエンド系サービス

> [!info] 前提：「フロントエンド」とは
> 「ユーザーが直接触る部分」のこと。スマホアプリの画面、Webサイトの見た目、ボタンを押したときの反応など、お客さんから見える側を**フロント（前）**、データベースや計算処理などの裏方を**バックエンド（後ろ）**と呼ぶ。
> ここで扱う4つは、フロント側を**作る・テストする・ユーザーに届ける**のを助けるサービス。

---

## 1. AWS Amplify 🏗️ — アプリ開発の「お任せパック」

### 何が困るのか？（Amplifyが無い世界）

「ToDoアプリ」を普通に作ろうとすると…

1. ログイン機能を作る（パスワード管理、メール認証…）
2. データを保存するデータベースを用意する
3. アプリとDBをつなぐAPIを書く
4. 完成したらサーバーに置いて公開する
5. コードを直すたびに、また手作業で公開し直す

…やることが多すぎる。一つひとつにAWSの別サービス（Cognito、DynamoDB、API Gateway…）の知識が必要で、初心者には壁が高い。

### Amplifyがやってくれること

> **たとえ：** 引っ越しで「家具付き・電気ガス開通済み・ネット工事済み」の部屋を借りるようなもの。自分は荷物（アプリの中身）を持ち込むだけ。

- **「ログイン機能つけたい」** → コマンド一発で、裏で認証サービス（[[0810_Cognito]]）が用意される
- **「データ保存したい」** → 裏でデータベース（DynamoDB）とAPIが自動生成される
- **「公開したい」** → GitHubとつないでおけば、コードを保存（push）するたびに**自動で公開**してくれる

「インフラの細かいことは知らないけど、とにかくアプリを早く形にしたい」人の味方。

> [!tip] 一言で
> 「フロントもバックも、まとめて面倒みるよ」パック。VercelやFirebaseのAWS版だと思えばOK。

## 2. AWS Device Farm 📱 — 実機テストの「レンタルスマホ部屋」

### 何が困るのか？

> 「自分のiPhoneでは動くけど、古いAndroidやGalaxyでもちゃんと動くの…？」

スマホは機種ごとに画面サイズもOSバージョンもバラバラ。**特定の機種だけ表示が崩れる・落ちる**ことがよくある。でも確認のために何十台もスマホを買うのは非現実的。

### Device Farmがやってくれること

> **たとえ：** AWSのデータセンターに「あらゆる機種の実機スマホがずらっと並んだ部屋」があって、それをネット越しに借りて触れる。

- **本物の端末**（iPhone各種・Android各種、古い機種も）で動作確認できる
- ブラウザから**遠隔で実機を操作**できる（画面をタップしたり）
- テストを自動化して、**何十台もの端末で一気に検証**できる

> [!tip] 一言で
> 「全機種そろえるのは無理だから、AWSの実機を借りてテストする」サービス。

## 3. Amazon Pinpoint 📨 — 「お知らせ配信＆効果測定」担当

### 何が困るのか？

アプリ運営ではユーザーにお知らせを送りたくなる。

- 「セール始まりました！」（プッシュ通知）
- 「久しぶりにログインしませんか？」（メール）
- 「予約確認です」（SMS）

ただ送るだけでなく、**誰に・どの手段で・送った結果どうだったか**まで考えたい。

### Pinpointがやってくれること

> **たとえ：** お客さんごとに、最適なタイミングで・最適な方法で・お知らせを送り分けて、反応も記録してくれるマーケティング担当者。

- **複数の手段**で送れる：プッシュ通知 / メール / SMS / 音声
- **狙い撃ち配信**：「30日間ログインしてない人だけに通知」などの条件指定ができる
- **効果測定**：開封率やクリック率を記録して、次の配信に活かせる

> [!note] 似たサービス「SES」との違い（よく混同する）
> - **SES** = メールを大量に送るための**エンジン（土管）**。送ることだけが仕事。
> - **Pinpoint** = **誰に・いつ・何を**送るかを設計し、結果も分析する**マーケ層**。実際の送信は裏でSESなどに任せる。
>
> → 「送信の仕組み」がSES、「配信の戦略・分析」がPinpoint。

## 4. Amazon API Gateway 🚪 — APIの「正面玄関＆警備員」★最重要

SAA試験で超頻出。特に丁寧に。

### まず「API」って何？

APIは「アプリ同士の窓口・受付」のこと。

> **たとえ：** レストランの「注文窓口」。お客さん（アプリ）は厨房（バックエンド）に直接入れない。代わりに窓口に「ハンバーグください」と注文すると料理が出てくる。この**注文の取り次ぎ口**がAPI。

### 何が困るのか？（API Gatewayが無い世界）

スマホアプリが裏のプログラム（Lambdaなど）を直接呼ぶと…

- **誰でも呼べてしまう**（認証がない＝不審者も厨房に入れる）
- **大量アクセスで裏が潰れる**（同時に1万人が殺到したら厨房パンク）
- **窓口がバラバラ**で管理しづらい

### API Gatewayがやってくれること

> **たとえ：** すべての注文を一手に引き受ける「正面玄関の受付＋警備員」。

```
[スマホ/Web]  →  [API Gateway]  →  [Lambda]  →  [DynamoDB]
  お客さん          受付・警備      厨房        食材倉庫
```

| 仕事 | 何をする？ | たとえ |
|---|---|---|
| 取り次ぎ | リクエストを正しい裏方へ振り分ける | 注文を適切な厨房へ通す |
| 認証・認可 | 「入っていい人か」チェック | 警備員が会員証を確認 |
| 流量制限（スロットリング） | アクセスが多すぎたら制限して裏方を守る | 「ただいま満席です」と入場制限 |
| キャッシュ | 同じ問い合わせは記録から即答 | よくある質問は受付が即答（厨房に聞かない） |

### APIの種類（3つ）

| 種類 | 特徴 | 使いどころ |
|---|---|---|
| **REST API** | 多機能・高機能 | 細かい制御が必要なとき |
| **HTTP API** | シンプル・安くて速い | シンプルなAPIならこれで十分 |
| **WebSocket API** | 双方向通信（送り合える） | チャット・リアルタイム通知 |

> [!important] SAA試験で問われるポイント
> - **サーバーレスの王道構成** = **API Gateway + Lambda + DynamoDB**。この3点セットは丸暗記レベル。
> - **DDoS対策・不正アクセス対策** = API Gateway の前に **AWS WAF** や **CloudFront** を置く。
> - **リアルタイムな双方向通信が欲しい** → **WebSocket API** を選ぶ。
> - **裏のシステムを過負荷から守りたい** → **スロットリング（流量制限）** を設定する。

## 全体まとめ：4つはどこで使う？

アプリ開発の流れに沿って並べると、居場所がハッキリする。

```
①作る           ②テストする        ③公開する         ④運営する
Amplify   →   Device Farm   →   （Amplifyで公開）  →   Pinpoint
（土台作り）    （実機検証）                            （お知らせ配信）

そして、アプリと裏方の間にはいつも…
        🚪 API Gateway（玄関で受付・警備）
```

| サービス | 役割 | 試験的な重要度 |
|---|---|---|
| **Amplify** | アプリの土台を丸ごと用意 | 中（名前と役割） |
| **Device Farm** | 実機でテスト | 低〜中（名前と役割） |
| **Pinpoint** | ユーザーへ配信＆分析 | 中（SESとの違い） |
| **API Gateway** | APIの玄関・警備員 | **高（最重要・構成を覚える）** |
