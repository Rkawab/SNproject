# Amazon Application Recovery Controller（ARC）（SAA ★★）

> [!info] 一言で
> **マルチAZ/マルチリージョンのフェイルオーバーを「切替先がちゃんと動く」と保証しながら、確実かつマネージドに実行する**サービス。

## たとえ話
2拠点（リージョン）で営業している会社。本社が被災したら全業務を支社へ移したい。

- **Route 53（ヘルスチェック/フェイルオーバー）** ＝ 電話を支社へ転送する仕組み。**転送はできるが、支社が全社分を捌けるかは保証しない**。
- **ARC** ＝ 「支社の受け入れ準備」を毎日点検する本部＋安全な非常用スイッチ。
  - **支社は今、全社分の席・人員・在庫があるか？** を継続点検（＝Readiness Check）
  - **点検済みの状態で、確実に切り替える**（＝ルーティングコントロール）

「切替できる」だけでなく「**切替先が確実に処理できる状態**」まで担保するのがARCの存在意義。

## 2つの主要機能

| 機能 | 役割 |
|---|---|
| **Readiness Check（準備状況チェック）** | 待機側リージョン/AZの**容量・構成・サービスクォータ・スケーリング設定**が本番と同等で、「今フェイルオーバーしても処理できる状態か」を**継続的に検証**する |
| **ルーティングコントロール（Routing Control）** | DNSの伝播の不確実さに頼らず、**ON/OFFのスイッチで安全・確実にトラフィックを切り替える**仕組み（Route 53のヘルスチェックと連動） |

> [!note] Readiness Check が見るもの
> 「エンドポイントが応答するか」だけを見るRoute 53ヘルスチェックと違い、ARCは**待機側に十分なリソース（EC2台数・クォータ・設定）が揃っているか**まで確認する。だから「フェイルオーバー先が確実に処理できる状態」という要件を満たせる。

## SAAでの狙われ方
- 「**マルチリージョン／マルチAZのフェイルオーバー**」＋「**切替先が確実に処理できる状態を保証**」＋「**低運用**」→ **ARC**。
- キーワード「**Readiness（準備状況）**」が出たらARCを疑う。
- 比較：
  - **Route 53 ヘルスチェック/フェイルオーバールーティング** … 振り分け・切り離しはするが、切替先の容量保証はしない。[[0404_Route53]]
  - **AWS Fault Injection Service（FIS）** … 障害を“テスト”するもので本番切替の制御はしない。[[1407_Fault_Injection_Service]]
  - **自前でヘルスチェック＋定期テスト＋DNS更新を作り込む** … 同じことはできるが運用負荷が高く、ARCがある場面では負ける。
