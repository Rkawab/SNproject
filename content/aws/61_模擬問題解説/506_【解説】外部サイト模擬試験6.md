## 第1問（Cost Optimization ｜ ECSタスクの右サイジング（Compute Optimizer））

> [!question] 問題（再掲）
> 会社のアプリケーションチームは、大規模な機械学習処理を実行するために Fargate 利用の Amazon ECS を使用している。処理は CPU とメモリを大量に消費し、現在十分な CPU とメモリが割り当たっていないタスクがあるため、パフォーマンスが低下している。コストを最適化しつつ処理パフォーマンスを改善したい。最も適切なソリューションは？
> - A. AWS Auto Scaling を使用して、ECSサービスの望ましいタスク数を設定しリソースを自動的に調整する。
> - B. AWS Cost Explorer を使用して、過去のリソース使用量と費用を分析し、適切なリソース設定を特定する。
> - C. AWS Compute Optimizer を使用して、ECSタスク定義のCPUとメモリ設定に関する推奨事項を取得する。
> - D. AWS CloudWatch を使用して、ECSタスクのメトリクスをモニタリングしボトルネックを特定する。

> [!success]- 答え：**C**
> 「タスク数を増やす」話ではなく、**1タスクあたりの CPU / メモリ設定をちょうどよくしたい**話。ECS on Fargate の CPU / メモリが過小または過大かを見て、推奨サイズを出せるのは **AWS Compute Optimizer**。

この問題の肝は、スケーリングではなく**右サイジング**です。

「処理が重いから増やす」と聞くと Auto Scaling を選びたくなりますが、問題文は「タスクに十分な CPU とメモリが割り当たっていない」と言っています。つまり、足りないのは**タスクの個数**ではなく、**タスク定義に書かれた CPU / メモリの大きさ**です。

Compute Optimizer は CloudWatch メトリクスなどの利用状況を分析し、ECS サービス on Fargate に対して、現在のタスクサイズと推奨タスクサイズ、CPU / メモリの過不足、性能リスク、想定コスト差を提示できます。だから「コストを最適化しつつ、パフォーマンスも改善」という両方の要件に合います。

### たとえ話：厨房の人数ではなく、調理台の大きさを直す

ECSタスクを「料理人」、CPU/メモリを「調理台と火力」だとします。

- **Auto Scaling** は、料理人の人数を増やす仕組み。
- **Compute Optimizer** は、各料理人に割り当てる調理台と火力が適切かを診断する仕組み。
- **Cost Explorer** は、今月いくら食材費・人件費がかかったかを見る会計表。
- **CloudWatch** は、火力・待ち時間・作業量を測る温度計やメーター。

今回の問題は「料理人が使っている調理台が小さすぎて作業が遅い。無駄なコストも避けたい」です。料理人の人数を増やしても、1人あたりの調理台が小さいままだと根本解決になりません。だから、各タスクの CPU / メモリを適正化する Compute Optimizer が最適です。

### 各選択肢をズバッと

- **C：正解** … Compute Optimizer は利用状況を分析して、ECS on Fargate の推奨 CPU / メモリサイズを提示する。過小ならサイズアップ、過大ならサイズダウンの判断ができ、性能とコストのバランスを取りやすい。
- **A：誤り** … Auto Scaling は ECS サービスのタスク数を増減する水平スケール。1つのタスクに割り当てる CPU / メモリが不足している問題には直接答えていない。
- **B：誤り** … Cost Explorer は費用の可視化・分析に強いが、ECS タスク定義の CPU / メモリ推奨値を出すサービスではない。コスト傾向を見る道具であって、右サイジング提案の主役ではない。
- **D：誤り** … CloudWatch はメトリクス監視・アラーム・ダッシュボードの主役。ボトルネックの発見には使えるが、「どの CPU / メモリ設定にすればよいか」という推奨までは Compute Optimizer の役割。

> [!tip] SAA着眼点
> - **CPU / メモリ / インスタンスタイプを適正化したい** → Compute Optimizer。
> - **数を増減したい** → Auto Scaling。
> - **費用の推移・サービス別コストを見たい** → Cost Explorer。
> - **メトリクスを監視し、しきい値で通知したい** → CloudWatch。
> - 「コスト最適化」と「パフォーマンス改善」が同時に出ても、文中に**右サイジング、推奨設定、過小/過大プロビジョニング**の匂いがあれば Compute Optimizer を優先する。

## 第2問（Security / IAM ｜ IAM Identity Centerと外部SAML IdPのRBAC）

> [!question] 問題（再掲）
> Organizations配下の複数AWSアカウントと複数リージョンを使う企業が、既存のSAML対応IdPでユーザーアクセスを一元管理したい。IdP側ではユーザーが地理的位置（東京、シンガポールなど）と職務（開発者、管理者）に基づくグループに分類されている。IdPグループごとに適切なロールを割り当て、地理的位置と職務の組み合わせでRBACを行い、各チームは担当リージョンのリソースのみにアクセスできるようにしたい。IdP側MFAともシームレスに統合したい。最適なソリューションは？
> - A. IAM Identity Centerで既存IdPと統合し、地理的位置と職務の組み合わせごとに権限セットを作成する。IdPグループを適切な権限セットに割り当て、権限セットごとにリージョン固有のアクセス権限を設定する。
> - B. Cognitoユーザープールを作成し、既存IdPと統合する。地理的位置と職務ごとにCognitoグループを作成し、各グループにIAMロールを割り当てる。
> - C. Organizationsで既存IdPと統合し、地理的位置と職務の組み合わせごとにOUを作成する。OUごとにSCPでリージョン固有のアクセス権限を設定する。
> - D. Directory Serviceを使用して、既存IdPとIAM Identity Centerをフェデレーションする。地理的位置と職務の組み合わせごとにIAMロールを作成し、IdPグループを適切なロールに割り当てる。

> [!success]- 答え：**A**
> **従業員のAWSアカウントアクセス**を、**既存SAML IdP** と連携して、**Organizations配下の複数アカウントへ一元的に割り当てる**なら IAM Identity Center。IdPグループを権限セットに割り当て、権限セットのIAMポリシーで職務別・リージョン別の制御を行う。

この問題は、キーワードを拾うとかなり素直です。

- **Organizations配下の複数AWSアカウント**：マルチアカウントへの一元アクセス管理が必要。
- **既存のSAML対応IdP**：社内ID基盤とフェデレーションしたい。
- **地理的位置 × 職務のグループ**：ユーザーやグループ単位で権限を割り当てたい。
- **ロールベースで制御**：IAM Identity Centerの権限セットが、各アカウントにロールとして展開される。
- **IdP側MFA**：認証は外部IdPに任せ、AWS側はフェデレーションされた結果を使う。

つまり答えは **IAM Identity Center + 外部SAML IdP + グループ割り当て + 権限セット** です。

### たとえ話：本社の社員証で、各国支店の担当フロアだけに入る

既存SAML IdPは、本社が管理している**社員証システム**です。社員証には「東京の開発者」「シンガポールの管理者」のような所属グループが入っています。

IAM Identity Centerは、AWS側の**総合受付**です。本社の社員証システムと連携し、社員がいつもの社員証とMFAで本人確認を済ませると、AWS側の受付が「この人は東京開発者グループだから、東京支店の開発者フロアだけ入れる」と判断します。

この「どの支店・どのフロアへ入れるか」を定義する入館ルールが **権限セット** です。権限セットは、AWSアカウント内ではIAM Identity Center管理のIAMロールとして展開されます。

### 実務イメージ

たとえば、次のようなグループと権限セットを作るイメージです。

| IdPグループ | IAM Identity Centerの権限セット | 制御内容 |
|---|---|---|
| Tokyo-Developers | TokyoDeveloperAccess | 開発者権限 + 東京リージョンのみ |
| Tokyo-Admins | TokyoAdminAccess | 管理者権限 + 東京リージョンのみ |
| Singapore-Developers | SingaporeDeveloperAccess | 開発者権限 + シンガポールリージョンのみ |
| Singapore-Admins | SingaporeAdminAccess | 管理者権限 + シンガポールリージョンのみ |

リージョン制限は、権限セットに含めるIAMポリシーで `aws:RequestedRegion` などの条件キーを使って実装するのが典型です。SAAでは細かいJSONより、「**権限セット/IAMポリシーでリージョン固有の権限にする**」と判断できれば十分です。

> [!note] IAMロールと権限セットの関係
> 選択肢では「IAMロールを割り当てる」と表現されることがありますが、IAM Identity Centerの正確な考え方は**権限セットをユーザー/グループとAWSアカウントに割り当てる**です。割り当てると、各AWSアカウント側にIAM Identity Center管理のIAMロールが作られます。試験では「IAM Identity Center + 権限セット + グループ割り当て」の構図を見抜けばOKです。

### 各選択肢をズバッと

- **A：正解** … IAM Identity Centerは外部SAML IdPと連携でき、IdP側のユーザー/グループをAWSアカウントアクセスへ割り当てられる。権限セットで職務別ポリシーを定義し、リージョン制限もIAMポリシー条件で表現できる。MFAは外部IdP側の認証フローをそのまま使える。
- **B：誤り** … Cognitoは自社Web/モバイルアプリの利用者ログイン基盤。B2Cアプリ利用者には強いが、従業員がOrganizations配下のAWSアカウントへSSOする用途ではIAM Identity Centerが主役。
- **C：誤り** … Organizations/SCPはアカウントやOUに対する**権限の上限**を決めるガードレール。ユーザー認証、SAML IdP連携、IdPグループ単位のロール割り当てを行う仕組みではない。SCPだけでは「東京開発者だけ東京リージョンへ」のようなユーザーグループ別SSOは成立しない。
- **D：誤り** … Directory ServiceはMicrosoft ADをAWSで使う/オンプレADへ橋渡しするためのサービス。今回の軸は「既存SAML IdP」と「AWSアカウントアクセスの一元管理」なので、主役はIAM Identity Center。Directory Serviceを使うのは、IDソースがADで、そのADと接続する文脈。

> [!tip] SAA着眼点
> - **従業員が複数AWSアカウントへSSO** → IAM Identity Center。
> - **既存SAML IdP / Okta / Microsoft Entra ID / 社内IdP** → IAM Identity Centerの外部IdP連携。
> - **グループごとにAWSアカウント権限を割り当てる** → ユーザー/グループ + AWSアカウント + 権限セット。
> - **Cognito** → アプリのエンドユーザー認証。AWSコンソール/CLIへ従業員を入れる話なら外す。
> - **Organizations/SCP** → アカウント全体のガードレール。認証やグループ割り当ての主役ではない。
> - **Directory Service** → Microsoft AD連携。SAML IdP一般とのSSO管理そのものはIAM Identity Center。

## 第3問（Networking / CDN / LB ｜ S3への閉域・低コストアクセス（Gateway型VPCエンドポイント））

> [!question] 問題（再掲）
> 金融サービス会社が、顧客情報のレポート処理システムをAWSへ移行する。大量の取引データを処理して日次レポートを生成するアプリケーションをEC2インスタンスで動かし、処理後の日次レポートをS3バケットに保存する。レポートは厳格なセキュリティとプライバシー要件に従って管理する必要があり、データの外部露出を最小限に抑えたい。データの保存と取得に関するコスト削減も必要。選ぶべき手段は？
> - A. EC2インスタンスのあるVPCに、ゲートウェイ型VPCエンドポイントを作成する。エンドポイントへのルーティングをルートテーブルに設定し、S3バケットへアクセスする。
> - B. パブリックサブネットにNATゲートウェイを作成する。NATゲートウェイへのルーティングをルートテーブルに設定し、S3バケットへアクセスする。
> - C. EC2インスタンスのあるVPCに、PrivateLinkを作成する。エンドポイントへのルーティングをルートテーブルに設定し、S3バケットへアクセスする。
> - D. EC2インスタンスのあるVPCに、インターネットゲートウェイを作成する。インターネットゲートウェイへのルーティングをルートテーブルに設定し、S3バケットへアクセスする。

> [!success]- 答え：**A**
> キーワードは **EC2 → S3**、**外部露出を最小化**、**保存・取得コスト削減**。S3へのVPC内アクセスでこの3つを同時に満たす定番は **S3用ゲートウェイ型VPCエンドポイント**。

まず、この問題は「S3を暗号化する方法」ではなく、**EC2からS3へ行くネットワーク経路をどれにするか**を聞いています。

日次レポートはS3に保存されます。EC2がS3にアクセスする道はいくつかありますが、金融サービス会社なので外部露出を抑えたい。さらに大量データなので、NAT Gatewayのようなデータ処理料がかかる道は避けたい。そこで、S3専用の無料直通ルートであるゲートウェイ型VPCエンドポイントを選びます。

### たとえ話：S3倉庫へ行く4つの道

VPCを「会社ビル」、EC2を「社員」、S3を「AWS内の巨大倉庫」だとします。

| 選択肢 | たとえ | 何が問題か |
|---|---|---|
| **A. Gateway型VPCエンドポイント** | ビル内からS3倉庫へ行ける**社内専用通路** | 正解。外の道路を通らず、追加料金もない |
| **B. NAT Gateway** | 受付に頼んで外の道路経由で荷物を運ぶ**代理外出窓口** | 通れるが、窓口手数料がかかる |
| **C. PrivateLink / Interface型** | S3倉庫への**有料の内線電話**をビルに引く | 通れる場合もあるが、VPC内EC2→S3だけなら有料で過剰 |
| **D. Internet Gateway** | ビルの**正面玄関**から外の道路へ出る | 外部露出を抑えたい要件に合わない |

この問題の感覚としては、「S3に行きたいだけなのに、わざわざ有料タクシー（NAT）や正面玄関（IGW）を使う必要ある？」です。S3には無料の社内専用通路（Gateway型VPCエンドポイント）が用意されています。

### EC2からS3への経路を図で見る

正解のイメージはこうです。

```text
EC2（VPC内）
  |
  | ルートテーブル:
  | S3のプレフィックスリスト → vpce-xxxx
  v
Gateway型VPCエンドポイント
  |
  v
Amazon S3
```

ポイントは、ルートテーブルに「S3宛てはこのエンドポイントへ」という道案内を足すことです。`0.0.0.0/0` のように全インターネット向けを出すのではなく、**S3宛てだけをS3専用の道に流す**イメージです。

> [!note] プレフィックスリストとは
> S3のIPアドレス範囲をAWSがまとめて管理している「S3行き住所録」のようなものです。Gateway型エンドポイントをルートテーブルに関連付けると、S3宛て通信がその住所録を使ってエンドポイントへ向かいます。試験では「ルートテーブルに設定」と読めれば十分です。

### なぜAがコスト削減になるのか

NAT Gatewayは便利ですが、時間課金とデータ処理料がかかります。大量のレポートを毎日S3へ保存・取得すると、NATを通るデータ量が増えてコストも増えます。

一方、S3のGateway型VPCエンドポイント自体には追加料金がありません。つまり、EC2からS3への大量通信をNAT経由からGateway型エンドポイント経由に切り替えると、NATのデータ処理料を避けられます。これがこの問題の「コスト削減」の正体です。

### 各選択肢をズバッと

- **A：正解** … S3用Gateway型VPCエンドポイントをVPCに作成し、EC2がいるサブネットのルートテーブルに関連付ける。S3宛て通信がインターネット/NATを通らず、追加料金なしでS3へ向かう。
- **B：誤り** … NAT Gatewayはプライベートサブネットから外へ出るための片方向出口。S3へは到達できるが、NAT Gatewayの時間課金・データ処理料が発生する。大量データの保存・取得ではコスト削減になりにくい。
- **C：誤り** … PrivateLink、つまりInterface型エンドポイントは、ENIを作ってプライベートIPでAWSサービスに接続する方式。S3でも使えるが有料。オンプレからS3へプライベート接続したい等の追加要件がない限り、VPC内EC2→S3ではGateway型が優先。
- **D：誤り** … Internet GatewayはVPCからインターネットへ出る正面玄関。外部露出を最小化したい金融系シナリオでは選びにくい。さらにプライベートサブネットのEC2が使うにはパブリックIPやルート設計も絡む。

> [!tip] SAA着眼点
> - **EC2 in VPC → S3 / DynamoDB**、かつ **低コスト・インターネット非経由** → Gateway型VPCエンドポイント。
> - **S3 / DynamoDB以外のAWSサービス**（SQS、SNS、KMS、CloudWatch、ECRなど）へプライベート接続 → Interface型エンドポイント（PrivateLink）。
> - **プライベートサブネットから外部WebやOS更新へ出たい** → NAT Gateway。
> - **外部から公開Webへ入れる/インターネットへ直接出る** → Internet Gateway。
> - 「S3へ大量アクセス」「NATコスト削減」が出たら、ほぼ **Gateway型VPCエンドポイント** を疑う。

## 第4問（Cost Optimization ｜ AWS Budgets ActionsとSCPによる予算超過時の自動制御）

> [!question] 問題（再掲）
> 国際的な非営利団体が、世界中の様々なプロジェクトで使用するために複数のAWSアカウントを運用している。各プロジェクトに異なる予算を割り当て、予算超過を未然に防ぎながら、進捗に応じてリソースを自動調整したい。特定プロジェクトの予算が超過した場合には、新たなリソースのプロビジョニングを自動的に停止したい。AWS Budgetsを活用する場合、どの設定をすべきか。（2つ選択）
> - A. 各プロジェクトの予算アラートをAWS Budgetsで設定する。予算超過時のアクションを実行するための権限を持つBudgets用のIAMロールを作成し適用する。
> - B. Organizationsで組織に新たなリソースのプロビジョニングを停止するSCPを設定する。予算のしきい値を超えた場合、AWS BudgetsがこのSCPを適用し、関連するAWSアカウントでの追加コストの発生を防ぐ。
> - C. 特定プロジェクトに対して管理者がアラートを受け取れるようAWS Budgetsで設定する。そのアラートに基づき、管理者が手動でリソース停止などの必要な措置を行う。
> - D. AWS Budgetsでプロジェクトごとの予算アラートを設定する。予算を超過したプロジェクトのみにAWS CloudTrailログを有効化し、該当プロジェクトの新たなリソースのプロビジョニングを停止する。
> - E. AWS Budgetsでプロジェクトごとの予算を設定した後に、AWS Cost and Usage Reportsと連携させて使用状況をモニタリングする。リソースを停止するLambda関数のトリガーに、予算上限の超過を設定する。

> [!success]- 答え：**A, B**
> AWS Budgetsは「通知するだけ」のサービスではなく、**Budgets Actions**でしきい値超過時にIAMポリシーやSCPを適用できる。アクション実行にはAWS Budgetsが使うサービスロールが必要。複数AWSアカウントで新規リソース作成を止めたいなら、予算超過時にSCPを適用する構成が最も素直。

この問題で混乱しやすいのは、AWS Budgetsを「予算を超えたらメールが来るだけのサービス」と思ってしまう点です。

AWS Budgetsには **Budget Actions** があります。予算のしきい値を超えたとき、事前に決めたアクションを自動または承認後に実行できます。公式に用意されている代表的なアクションは、IAMポリシーの適用、SCPの適用、特定のEC2/RDSインスタンスの対象化です。今回の要件は「複数アカウントの特定プロジェクトで、新規リソース作成を止めたい」なので、OrganizationsのSCPと組み合わせるのがきれいです。

### たとえ話：プロジェクトの小遣い帳と自動ブレーキ

プロジェクトごとの予算を、子どもごとの小遣い帳だとします。

- **AWS Budgets** は、小遣い帳を見て「今月もう90%使った」「上限を超えた」と知らせる係です。
- **Budgets用IAMロール** は、その係が実際にブレーキをかけるための委任状です。
- **SCP** は、「これ以上、新しい買い物は禁止」という家全体のルールです。

ただ通知するだけなら、係が「使いすぎです」と叫ぶだけです。今回ほしいのは「使いすぎたら新しい買い物を止める」なので、係に委任状を持たせ、家のルールを自動で適用させる必要があります。

### 何をどう設定するのか

ざっくり流れはこうです。

```text
1. プロジェクト/アカウント/タグ単位でAWS Budgetsを作る
2. 予算しきい値を設定する
3. AWS Budgetsがアクションを実行するためのサービスロールを用意する
4. 新規リソース作成を制限するSCPをOrganizations側で用意する
5. しきい値超過時にBudgets ActionsでSCPを対象アカウントへ適用する
```

これで、予算超過時に「管理者が気づいて手で止める」のではなく、AWS側の仕組みとして追加コストを増やしにくくできます。

### 各選択肢をズバッと

- **A：正解** … Budgets Actionsを動かすには、AWS Budgetsがユーザーの代わりにアクションを実行するためのサービスロールが必要。予算アラートだけでなく、アクション実行権限まで用意するのがポイント。
- **B：正解** … OrganizationsのSCPで「新規リソース作成を禁止/制限」するガードレールを用意し、予算しきい値超過時にBudgets Actionsで適用する。複数アカウントの追加コスト抑止に合う。
- **C：誤り** … アラートを受けて管理者が手動対応する方式。問題文は「自動的に停止」と言っているので弱い。通知だけでは未然防止にも自動制御にもならない。
- **D：誤り** … CloudTrailは「誰が何をしたか」を記録する監査ログ。ログを有効化しても、新規リソースのプロビジョニングは止まらない。監視・証跡と制御を混同した選択肢。
- **E：誤り** … Cost and Usage Reportsは詳細なコストデータをS3へ出す分析用レポート。Lambdaで自作すれば何かはできるかもしれないが、予算超過時の制御はBudgets Actionsでネイティブにできる。試験では「マネージドな既存機能」があるなら自作Lambdaは負けやすい。

> [!tip] SAA着眼点
> - **予算超過を通知したい** → AWS Budgetsのアラート。
> - **予算超過時に自動で制限したい** → AWS Budgets Actions。
> - **Budgets Actionsを実行する** → AWS Budgets用のサービスロールが必要。
> - **複数アカウントで新規リソース作成を止めたい** → OrganizationsのSCPを適用。
> - **CloudTrail** は監査ログ、**CUR** は詳細コストレポート。どちらも「新規リソース作成を止めるブレーキ」ではない。

## 第5問（Security / IAM ｜ サードパーティCA証明書のACMインポートとCloudFront）

> [!question] 問題（再掲）
> 企業ポリシーにより特定のパブリックなサードパーティ認証局（CA）の使用が義務付けられている会社が、CloudFrontでHTTPS通信を提供するため、そのサードパーティCAの証明書を利用する必要がある。適切な手段は？
> - A. ACMで証明書を作成し、サードパーティCAに署名してもらう。その証明書をCloudFrontのカスタムSSL/TLS証明書として設定する。
> - B. ACMでサードパーティCAによる証明書を発行する。その証明書をCloudFrontのカスタムSSL/TLS証明書として設定する。
> - C. ローカルマシンでサードパーティCAから証明書を取得し、us-east-1のACMにインポートする。その証明書をCloudFrontのカスタムSSL/TLS証明書として設定する。
> - D. サードパーティCAが発行した証明書をus-east-1のSecrets Managerに保存する。その証明書をCloudFrontのカスタムSSL/TLS証明書として設定する。

> [!success]- 答え：**C**
> ACMが発行する証明書は必ず**Amazon自身のCA**（Amazon Trust Services）署名。特定のサードパーティCAの証明書が必要なら、ACMでの発行は不可能で、**外部で取得してACMにインポート**する一択。さらにCloudFront用の証明書は**us-east-1のACM**に置く必要がある。

知らないと解けない知識問題だが、覚えるポイントは実質2つだけ。**①ACM発行＝Amazon署名（発行元は選べない）**、**②CloudFrontの証明書はus-east-1のACM**。この2つが頭にあれば、Cが一瞬で残る。

### たとえ話：印鑑証明の「発行元」は選べない

ACMを**市役所の証明書発行窓口**だと思うと分かりやすい。

- 市役所（ACM）に頼めば、**市役所の印鑑が押された**証明書がタダで出てくる。自動更新までしてくれて便利。
- でも会社の規定で「**〇〇法務局の印鑑**が押された証明書じゃなきゃダメ」と言われたら？ 市役所の窓口でいくら頼んでも**法務局の印鑑は押せない**。発行元が違う。
- できるのは、**自分で法務局に行って証明書をもらい、市役所の保管庫に預ける（＝インポート）**こと。預ければ市役所経由で各所（CloudFront）に提示できる。

「ローカルマシンを使用して」とは、自分のPCで秘密鍵とCSR（証明書署名要求）を作り、サードパーティCAに署名してもらって証明書を受け取る——「自分で法務局に行く」部分のこと。

### 各選択肢をズバッと

- **A：誤り** … ACMで作った証明書のCSRを取り出して外部CAに署名させる、という流れが**ACMに存在しない**。ACM発行＝Amazon署名で完結し、途中に他人を挟めない。
- **B：誤り** … 一見それっぽいが、**ACMは他社CAの証明書を「発行」できない**。発行元は常にAmazon。サードパーティのものが欲しければ発行ではなく**インポート**。
- **C：正解** … 外部CAから取得→**us-east-1のACMにインポート**→CloudFrontのカスタムSSL/TLS証明書に設定。唯一の正攻法で、us-east-1指定も正しい。
- **D：誤り** … Secrets ManagerはDBパスワードやAPIキーの**金庫**。CloudFrontは**Secrets Managerから証明書を読み込めない**。「us-east-1」というそれっぽい単語で釣る罠。

> [!tip] SAA着眼点
> - 「**特定の／社内指定のCAの証明書**」→ ACM発行は不可、**外部取得→ACMインポート**。
> - 「**CloudFrontのカスタムSSL証明書**」→ **必ずus-east-1のACM**（他リージョンの証明書は選べない）。
> - ACM発行証明書は**自動更新される**が、**インポート証明書は自動更新されない**（期限前に手動再インポート）——ここも頻出。
> - Secrets Manager / Parameter Storeは「秘密の金庫」であって**TLS証明書の配布元にはならない**。

## 第6問（Database ｜ ベクトル検索の低コスト実装（RDS pgvector））

> [!question] 問題（再掲）
> ある企業が、生成AIを活用した検索機能をWebアプリケーションに統合しようとしている。ユーザーからの問い合わせ内容と過去のFAQデータを**ベクトル形式で比較**し、最も類似した回答を検索する必要がある。ベクトル検索をサポートし、**コスト効率の高い**方法で構築可能なデータベースソリューションはどれか。
> - A. RDS for PostgreSQL で pgvector 拡張を有効化し、FAQベクトルデータをベクトルテーブルに保存して検索機能を実装する。
> - B. ElastiCache を使用し、FAQデータをインメモリで管理してベクトル検索を実装する。
> - C. S3バケットにFAQデータのベクトルファイルを保存し、Athena で検索クエリを実行する。
> - D. OpenSearch Service を使用し、FAQデータのベクトルインデックスを管理するクラスターを構築する。

> [!success]- 答え：**A**
> pgvector は PostgreSQL に「ベクトル型のカラム」と「近い順に並べる検索」を追加する**無料の拡張機能**。RDS のインスタンス1台で済むため、FAQ検索規模なら最も低コストでベクトル検索を実現できる。

この問題は、まず「ベクトル検索とは何か」が分からないと選択肢が読めない。そこから行きます。

### そもそもベクトル検索とは

生成AIの世界では、文章を**数百個の数字の並び（ベクトル）**に変換します（埋め込み／エンベディングと呼ぶ）。ポイントは、**意味が近い文章ほど数字の並びも近くなる**こと。

- 「パスワードを忘れた」→ `[0.21, -0.53, 0.88, …]`
- 「ログインできない」→ `[0.19, -0.49, 0.85, …]` ← 数字が近い＝意味が近い
- 「送料はいくら？」→ `[-0.72, 0.11, -0.30, …]` ← 全然違う

**ベクトル検索**＝ユーザーの質問をベクトルに変換し、保存済みFAQベクトルの中から**数字の並びが一番近いもの**を探すこと。キーワードが一致しなくても「意味が似ている」回答を見つけられるのが、生成AI検索（RAG）の土台です。つまりこの問題は「**類似度の近い順に並べる検索ができて、安いDBはどれ？**」と聞いています。

### たとえ話：図書館の「感想が似てる本」探し

10万冊の本それぞれに「感想の座標」（ベクトル）が付いていて、「この本と読後感が似た本を探して」と頼まれたとします。

- **pgvector入りPostgreSQL**：普通の図書館の台帳に「座標が近い順に並べる」機能を**無料で後付け**したもの。今ある設備で済む。
- **OpenSearch**：座標検索専門の**巨大検索センターを新設**。高性能だが建物の維持費（クラスター常時稼働費）が高い。
- **ElastiCache**：机の上に本を全部広げて探す。速いが**机（メモリ）が超高級品**で、本棚代わりにするには高すぎる。
- **S3 + Athena**：倉庫に座標メモを山積みして、集計係に「近い順に並べて」と頼む。集計係（Athena）は足し算や絞り込みは得意だが、**「座標の近さ順」という探し方を知らない**。

### 各選択肢をズバッと

- **A：正解** … pgvector は PostgreSQL の拡張機能で、`CREATE EXTENSION vector` で有効化するだけ。ベクトル型カラムにFAQの埋め込みを保存し、類似度の近い順に検索できる。既存のRDS運用の延長で済み、専用クラスター費用が不要なので「コスト効率」で勝つ。
- **B：誤り** … ElastiCache はインメモリキャッシュ。メモリはディスクの何十倍も高価で、データ保管・検索の主役にするのはコスト要件と真逆。Redis系にベクトル検索機能自体はあるが、「安く作る」勝負では負ける。
- **C：誤り（最大の罠）** … 「S3は安い」に釣られやすいが、Athena は SQL で集計・絞り込みをするクエリサービスで、**ベクトルの類似度順検索ができない**。どんなに安くても要件（ベクトル検索）を満たさない時点で脱落。
- **D：誤り** … OpenSearch Service はベクトル検索（k-NN）に**対応している**ので機能面では合格。ただしクラスターを常時稼働させる費用が高く、「**最も**コスト効率よく」で pgvector に負ける。大規模RAG基盤なら本命になる、惜しい選択肢。

> [!tip] SAA着眼点
> - 「**ベクトル検索 ＋ コスト効率／既存RDB活用**」→ **RDS/Aurora PostgreSQL + pgvector**。
> - 「**ベクトル検索 ＋ 大規模・全文検索・ログ分析も**」→ **OpenSearch Service**（k-NN対応）。
> - **Athena はベクトル検索不可**。「S3が安い」というだけで選ばない。機能を満たさない安さは無意味。
> - **ElastiCache** は「読み取りの高速化（キャッシュ）」の文脈で選ぶもの。データ保管の主役にした瞬間コストで×。
> - 生成AI系の問題は今後増える傾向。「埋め込み／ベクトル／類似検索／RAG」という単語を見たら、候補は **pgvector・OpenSearch・Kendra・Bedrock ナレッジベース** あたりに絞られる。

## 第7問（Migration / Transfer ｜ Transfer Family SFTPのエンドポイントタイプとIP制限）

> [!question] 問題（再掲）
> 複数の顧客が**インターネット経由**でAWSストレージにアクセスできるSFTPサーバーを提供したい。アクセスは**承認されたIPアドレスのみ**に限定し、転送時・保存時の暗号化を行う。高い拡張性と可用性のあるAWSストレージを使いたい。最も適切なソリューションは？
> - A. 暗号化S3バケット + Storage Gateway S3ファイルゲートウェイで各顧客のオンプレとAWSを接続。バケットポリシーでアクセス許可。
> - B. 暗号化S3バケット + Transfer FamilyのSFTPサーバー（**内部向けVPCエンドポイント**）。承認IPをセキュリティグループで許可。
> - C. 暗号化EFS + Transfer FamilyのSFTPサーバー（**インターネットからアクセス可能なVPCエンドポイント**）。承認IPをセキュリティグループで許可。
> - D. 暗号化S3バケット + Transfer FamilyのSFTPサーバー（**パブリックエンドポイント**）。承認IPをセキュリティグループで許可。

> [!success]- 答え：**C**
> 「インターネットから顧客が入れる」＋「セキュリティグループでIP制限できる」を**両方**満たすのは、Transfer Familyの3種のエンドポイントのうち**インターネット向けVPCエンドポイント**だけ。

この問題の本質は、ストレージ選び（S3 vs EFS）**ではなく**、Transfer Familyの**エンドポイントタイプの使い分け**です。B・DはストレージがS3でも問題なし。落ちる理由はエンドポイントにあります。

### 前提：AWS Transfer Familyとは

SFTP（SSH経由のファイル転送プロトコル）などのサーバーを、EC2を立てずにマネージドで提供するサービスです。受け取ったファイルの保存先として**S3またはEFS**を選べます。古いシステムや取引先が「SFTPでしか送れない」場合の定番解です。

### 肝：エンドポイント3タイプの使い分け

Transfer FamilyのSFTPサーバーには「どこに窓口を置くか」の選択肢が3つあります。

| タイプ | インターネットから届く？ | セキュリティグループ使える？ |
|---|---|---|
| パブリック | ○ | **×（使えない）** |
| VPC・内部向け（internal） | **×** | ○ |
| VPC・インターネット向け | ○ | ○ |

要件は「インターネット経由 ＋ 承認IPのみ」。表を見れば、両方○なのは**インターネット向けVPCエンドポイント**だけです。

### たとえ話：店をどこに構えるか

SFTPサーバーを「受付窓口」だとします。

- **パブリックエンドポイント**＝ショッピングモール内の店舗。誰でも来られるが、**入口はモール（AWS）の共有設備**なので、自前の警備員（セキュリティグループ）を立たせられない。
- **内部向けVPCエンドポイント**＝社員専用ビルの中の店舗。警備員は置けるが、**そもそも外のお客さんが入館できない**。
- **インターネット向けVPCエンドポイント**＝**自社の土地（VPC）に建てた路面店**。外から誰でも来られる住所（Elastic IP）を持ちつつ、入口は自分の土地なので**警備員（セキュリティグループ）に「承認リストにある人だけ通せ」と指示できる**。

「外から来られる」×「入口を自分で警備できる」の両立が、路面店＝インターネット向けVPCエンドポイントです。

### 各選択肢をズバッと

- **A：誤り** … Storage Gatewayファイルゲートウェイは、**オンプレ側にNFS/SMBの窓口を置いてS3と同期する**ためのもの。SFTPサーバーは一切登場しない。「顧客にSFTPを提供する」という問題の入口から外れている。各顧客のオンプレに機器を置くのも非現実的。
- **B：誤り** … 「内部向け（internal）」は**VPC内・VPN・Direct Connect経由の身内専用**。インターネット経由の顧客は到達できない。セキュリティグループが使えても、客が来られなければ意味がない。
- **C：正解** … インターネット向けVPCエンドポイントで外部公開しつつ、セキュリティグループで承認IPのみ許可。EFSは暗号化を有効化でき、拡張性・可用性の要件も満たす。
- **D：誤り（最大の罠）** … パブリックエンドポイントもインターネットから届くが、**AWS管理の共有基盤上にあるためセキュリティグループを適用できない**。「セキュリティグループで許可する」という記述自体が実現不可能。Cとの違いはここ1点。

> [!tip] SAA着眼点
> - 「**SFTP / FTPS / FTPでAWSストレージへ**」→ **AWS Transfer Family**（保存先はS3またはEFS）。
> - 「**インターネット公開 ＋ 送信元IP制限**」→ **インターネット向けVPCエンドポイント**（EIP＋セキュリティグループ）。
> - **パブリックエンドポイントはセキュリティグループ不可**——Transfer Family問題の最頻出ひっかけ。
> - 「**社内・VPN経由のみ**」なら内部向けVPCエンドポイント。「誰が来るか」で窓口を選ぶ。
> - Storage Gatewayは「**オンプレ側に窓口**」、Transfer Familyは「**AWS側に窓口**」。方向が逆。

## 第8問（Networking / CDN / LB ｜ グローバル3層Webアプリ（CloudFront・ECS Fargate・Aurora Global Database））

> [!question] 問題（再掲）
> グローバル企業が、AWS上で注文処理を行う3層Webアプリケーションを新規開発する。フロントエンドは画像やJavaScriptなどの静的コンテンツを配信し、カート投入、在庫照会、注文確定は動的APIとしてバックエンドで処理する。商品情報と在庫情報はリレーショナルデータベースに格納され、アプリケーションはSQLで参照・更新する。世界中の顧客に低レイテンシーで提供し、リージョン全体の障害が発生しても継続できる可用性を確保し、運用オーバーヘッドも最小化したい。最適な選択肢は？
> - A. CloudFront + S3で静的コンテンツをグローバル配信。動的APIはECSをFargate起動タイプで各リージョンに配置。DBはAurora Global Database。
> - B. S3静的ウェブサイトホスティングで静的コンテンツを配信。API Gateway + Lambdaで動的処理。DBはAurora Serverless。
> - C. CloudFront + S3で静的コンテンツをグローバル配信。動的APIは各リージョンのEC2で実行。DBはRDS for MySQL + DMS継続レプリケーション。
> - D. Global Acceleratorで単一リージョンのALBへ転送。動的APIは単一リージョンのECS Fargate。静的コンテンツは同リージョンのEC2から配信。DBはRDS for MySQLのマルチAZ DBクラスター。

> [!success]- 答え：**A**
> 静的コンテンツ、動的API、リレーショナルDBの3層を、それぞれグローバル・低運用・リージョン障害対応の観点で選ぶ問題。**CloudFront + S3**、**ECS Fargateを複数リージョン配置**、**Aurora Global Database** の組み合わせが要件に最もきれいに対応する。

この問題は、サービス名を暗記するよりも「3層Webアプリを3つに分解」すると楽になります。

```text
1. 静的コンテンツ層: 画像 / JavaScript / CSS
2. 動的API層: カート投入 / 在庫照会 / 注文確定
3. データベース層: 商品情報 / 在庫情報 / 注文情報（SQL）
```

それぞれの層で求められていることが違います。

| 層 | 要件 | 選ぶもの |
|---|---|---|
| 静的コンテンツ | 世界中に低レイテンシー配信 | CloudFront + S3 |
| 動的API | 複数リージョンで動かし、サーバー運用を減らす | ECS Fargate |
| DB | SQL、リレーショナル、リージョン障害対応 | Aurora Global Database |

### たとえ話：世界展開する通販会社

画像やJavaScriptは、カタログやチラシのようなものです。世界中の顧客に毎回本社から送るより、各国の近い拠点に置いておく方が速い。これが **CloudFront + S3** です。

カート投入や注文確定は、注文受付スタッフの仕事です。世界中に顧客がいるなら、注文受付も各地域に置いた方が低レイテンシーです。ただしサーバー管理はしたくないので、EC2ではなく **ECS Fargate** を使います。

商品情報や在庫情報は、会社の台帳です。世界中で同じ台帳を見たいし、ひとつの地域が落ちても別地域で続けたい。SQLで扱うリレーショナルDBなので、**Aurora Global Database** が合います。

### なぜAが強いのか

- **CloudFront + S3**：静的コンテンツをエッジロケーションから配信し、世界中の顧客の表示を速くする。
- **ECS Fargate**：コンテナを実行するが、EC2インスタンス管理が不要。動的APIを複数リージョンに置きやすい。
- **Aurora Global Database**：Aurora DBクラスターを複数リージョンにまたがって構成でき、グローバルな読み取り低レイテンシーとリージョン障害時の復旧性を得られる。

この3つが、問題文の「低レイテンシー」「リージョン全体の障害でも継続」「運用オーバーヘッド最小」をすべて拾っています。

### 各選択肢をズバッと

- **A：正解** … 静的はCloudFront + S3、動的APIは各リージョンのECS Fargate、DBはAurora Global Database。3層それぞれに対して、低レイテンシー・高可用性・低運用を満たす。
- **B：誤り** … S3静的ウェブサイトホスティングだけでは、CloudFrontのように世界中のエッジから低レイテンシー配信する構成ではない。API Gateway + Lambdaは低運用だが、Aurora Serverless単体では「リージョン全体障害でも継続」のグローバルDB要件が弱い。
- **C：誤り** … CloudFront + S3は良いが、動的APIをEC2で運用するためオーバーヘッドが大きい。DMS継続レプリケーションでRDSを自前連携するより、Aurora Global Databaseの方がグローバルDBのマネージド解として適切。
- **D：誤り** … Global Acceleratorはグローバル入口として使えるが、単一リージョンのALBへ集約しているため、リージョン障害で止まる。RDS Multi-AZはAZ障害対策であって、リージョン全体の障害対策ではない。

> [!tip] SAA着眼点
> - **静的コンテンツを世界中へ低レイテンシー配信** → CloudFront + S3。
> - **コンテナAPIを低運用で実行** → ECS Fargate。
> - **SQL/RDB + グローバル低レイテンシー読み取り + リージョン障害対策** → Aurora Global Database。
> - **Multi-AZ** は同一リージョン内のAZ障害対策。**リージョン障害**にはマルチリージョン構成が必要。
> - **DMS継続レプリケーション** は移行/同期の道具。グローバルDBの可用性設計の主役としてはAurora Global Databaseに負ける。

## 第9問（Storage ｜ SMB共有が必要なCOTSアプリ（FSx for Windows File Server））

> [!question] 問題（再掲）
> 企業がAWS上にCOTSアプリケーションを導入する。COTSアプリケーションは、SMBを使用して共有ストレージからデータを取り込む必要がある。共有ストレージはCOTSアプリケーション以外にも、複数のEC2インスタンスからSMBでアクセスする。運用コストを可能な限り最小限にして要件を満たすには？
> - A. FSx for Windowsファイルサーバーをデプロイする。COTSアプリケーションとEC2インスタンスから共有する。
> - B. EFSファイルシステムをデプロイする。COTSアプリケーションとEC2インスタンスから共有する。
> - C. 複数のEC2インスタンスにアタッチされたEBSボリュームを、COTSアプリケーションに共有する。
> - D. S3バケットをデプロイする。IAMでアクセス権限を設定し、共有ファイルサーバーとして使用できるようにする。

> [!success]- 答え：**A**
> 問題文の決定打は **SMB** と **複数EC2から共有アクセス**。SMBのマネージド共有ファイルサーバーは **Amazon FSx for Windows File Server**。

これはストレージ問題の中でもかなり分かりやすい部類です。難しく見えるのは「COTS」という言葉が混ざっているからですが、実際に見ればよいのは **SMBを使う共有ストレージ** という一点です。

### COTSとは

COTS（Commercial Off-The-Shelf）は、市販の既製品アプリケーションのことです。自社で自由に作り替えるアプリではなく、ベンダーが作ったパッケージ製品を導入するイメージです。

そのため、アプリ側が「SMB共有フォルダからデータを読む」と決まっているなら、AWS側もその前提に合わせる必要があります。アプリをS3 API対応に書き換える、NFSに変える、という発想は基本的に要件から外れます。

### たとえ話：アプリが「Windows共有フォルダしか読めない」

COTSアプリを、昔ながらの業務ソフトだと考えてください。

そのソフトは「`\\fileserver\share` みたいな共有フォルダにファイルを置いてください」と決まっています。なら、AWS側で用意すべきなのは、同じようにSMBでアクセスできる共有フォルダです。

- **FSx for Windows File Server**：Windows共有フォルダそのもの。アプリが期待する形に合う。
- **EFS**：Linux向けのNFS共有。似ているがプロトコルが違う。
- **EBS**：EC2に挿すディスク。共有フォルダサービスではない。
- **S3**：巨大なオブジェクト倉庫。ファイルサーバーではない。

つまり、「共有ストレージ」と聞いても、**何のプロトコルで共有するか**を必ず見るのがコツです。今回はSMBなのでFSx for Windows File Serverです。

### ストレージ4択の見分け方

| サービス | ざっくり役割 | プロトコル/使い方 | 今回 |
|---|---|---|---|
| **FSx for Windows File Server** | Windows共有ファイルサーバー | **SMB** / NTFS / AD連携 | **正解** |
| **EFS** | Linux向け共有ファイルシステム | NFS | SMB要件に合わない |
| **EBS** | EC2に付けるディスク | ブロックストレージ | 複数EC2の共有ファイルサーバーではない |
| **S3** | オブジェクトストレージ | API/HTTPでオブジェクト操作 | SMB共有ではない |

### 各選択肢をズバッと

- **A：正解** … FSx for Windows File Serverは、SMBプロトコルでアクセスできるフルマネージドWindowsファイルサーバー。複数EC2やCOTSアプリから共有フォルダとして使える。自前でWindowsファイルサーバーをEC2に立てるより運用負荷が低い。
- **B：誤り** … EFSは共有ファイルシステムではあるが、主役はNFS。Linux系アプリの共有ファイル置き場なら候補になるが、SMBを要求するCOTSアプリには合わない。
- **C：誤り** … EBSはブロックストレージ。EC2にディスクとしてアタッチして使うもので、複数EC2がSMBで共有アクセスするファイルサーバーではない。Multi-Attachも特殊用途で、共有ファイルシステムを提供する機能ではない。
- **D：誤り** … S3はオブジェクトストレージ。IAMで権限管理はできるが、SMB共有フォルダとしてマウントして既存COTSアプリがそのまま読む用途ではない。

> [!tip] SAA着眼点
> - **SMB / Windows共有 / NTFS / Active Directory連携 / COTS Windowsアプリ** → FSx for Windows File Server。
> - **NFS / Linux複数EC2共有 / Lambdaから共有ファイル** → EFS。
> - **EC2のローカル永続ディスク** → EBS。
> - **安価・高耐久なオブジェクト保管** → S3。
> - 「運用コスト最小」でも、アプリのプロトコル要件を満たさない安い選択肢は落とす。

## 第10問（Migration / Transfer ｜ Transfer Family内部SFTPとManaged Workflows）

> [!question] 問題（再掲）
> 多地域の工場から集めた製造データをオンプレミスで処理している。製造データはSFTPで不定期に転送され、転送直後に簡単な分析ジョブが実行されている。これらをAWSへ移行したい。すでに各工場からAWSまではVPN接続が可能で、新しいシステムは既存の製造管理システムとの統合、高いセキュリティ、冗長性が必要。
> - A. Transfer FamilyのインターナルSFTPサーバーを設置し、2つのAZとエンドポイントを接続する。保存先はS3。Lambda関数を作成し、Transfer FamilyのマネージドワークフローでLambdaを呼び出す。
> - B. Transfer Familyのインターネット向けSFTPサーバーを設置し、1つのAZとエンドポイントを接続する。保存先はS3。Lambda関数を作成し、Transfer FamilyのマネージドワークフローでLambdaを呼び出す。
> - C. Transfer FamilyのインターナルSFTPサーバーを設置し、1つのAZとエンドポイントを接続する。保存先はS3。EC2インスタンスを設置し、スケジュールされたジョブでデータ処理を行う。
> - D. Transfer Familyのインターネット向けSFTPサーバーを設置し、2つのAZとエンドポイントを接続する。保存先はS3。Lambda関数を作成し、S3イベントでLambdaを呼び出す。

> [!success]- 答え：**A**
> 決め手は **VPN接続済み** と **転送直後に分析**。VPNでAWSへ入れるならSFTPサーバーはインターネット公開せず、VPC内のインターナルエンドポイントにする。転送直後の処理はTransfer Family Managed WorkflowsでLambdaを呼び出す。

この問題はネットワーク用語が多く見えますが、実際に聞かれているのはかなり素直です。

```text
工場
  |
  | VPN
  v
VPC内のTransfer Family SFTP（内部エンドポイント・複数AZ）
  |
  v
S3
  |
  v
Managed Workflows -> Lambdaで分析
```

まず、工場からAWSまではすでにVPNでつながっています。VPNは「工場とAWSの間にある専用の通路」のようなものです。せっかく専用通路があるのに、SFTPサーバーをインターネットに公開する必要はありません。

だから入口は **Transfer FamilyのインターナルSFTPサーバー** にします。これは、VPC内のプライベートIPで到達するSFTP入口です。AWS公式ドキュメントでも、VPCホスト型エンドポイントは内部用にでき、プライベートIPでアクセスする構成を取れます。

次に、冗長性です。1つのAZだけに置くと、そのAZに障害が起きたときに入口が弱くなります。問題文に「高いセキュリティと冗長性」とあるので、**2つのAZにエンドポイントを接続**する選択肢が強くなります。

最後に、転送直後の分析です。データは不定期に来ます。なら「毎時0分にEC2で処理」のようなスケジュール型より、ファイルが届いた瞬間に処理を始めるイベント型の方が自然です。Transfer Familyには **Managed Workflows** があり、SFTPなどのファイル転送後にリアルタイムで処理を開始し、Lambdaを呼び出せます。

### たとえ話：工場から本社倉庫への専用道路

工場から本社倉庫へ荷物を運ぶとします。

- VPN：工場と本社を結ぶ専用道路
- インターナルSFTP：専用道路の先にある社内用の搬入口
- インターネット向けSFTP：誰でも見える大通り沿いの搬入口
- 2つのAZ：搬入口を2か所に分けておく
- S3：大きくて壊れにくい倉庫
- Managed Workflows + Lambda：荷物が届いた瞬間に検品担当が自動で動く仕組み

この問題では、すでに専用道路があるので、社内用の搬入口を使うのが自然です。さらに搬入口を2か所に分け、荷物が届いた瞬間に検品処理を走らせます。これがAです。

### 各選択肢をズバッと

- **A：正解** … VPN経由で到達する内部SFTP、複数AZによる冗長化、S3保存、Managed Workflowsによる転送直後のLambda起動をすべて満たす。
- **B：誤り** … Managed WorkflowsとLambdaは良いが、インターネット向けSFTPで公開範囲が広い。さらに1つのAZなので冗長性も弱い。
- **C：誤り** … インターナルSFTPは良いが、1つのAZだけなので冗長性が不足する。EC2のスケジュール処理も「転送直後」の要件に合いにくく、サーバー運用も増える。
- **D：誤り** … 2つのAZとLambdaは良いが、インターネット向けSFTPが不要に公開範囲を広げている。S3イベントでもLambda起動はできるが、Transfer Familyの転送後処理としてはManaged Workflowsの方がこの問題文に直結する。

> [!tip] SAA着眼点
> - **SFTPをAWSで受ける** → AWS Transfer Family。
> - **既にVPN / Direct Connectがある** → インターネット公開ではなく、VPC内部エンドポイントを疑う。
> - **承認済みの外部顧客がインターネットから来る** → インターネット向けVPCエンドポイントやパブリックエンドポイントを検討する。
> - **高可用性・冗長性** → 複数AZ。
> - **転送直後に処理** → Transfer Family Managed Workflows、Lambda、イベント駆動。
> - **定期処理** と **直後処理** は別物。問題文に「直後」とあればスケジュールジョブは弱い。
