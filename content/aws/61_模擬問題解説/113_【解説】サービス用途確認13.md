## 第1問（Storage ｜ 高性能コンピューティング向けファイルシステム）

> [!question] 問題（再掲）
> S3に保存した大量のデータを、機械学習やHPCのLinuxワークロードから高スループット・低レイテンシーのPOSIXファイルシステムとして処理したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon FSx for Lustre
> - B. Amazon EFS
> - C. Amazon FSx for Windows File Server
> - D. Amazon S3 Glacier Deep Archive

> [!success]- 答え：**A（Amazon FSx for Lustre）**
> 大量データを多数の計算ノードから並列処理するための高性能ファイルシステムを提供する。

### 深掘り解説

EFSが「皆で使える一般倉庫」なら、FSx for Lustreは「大量の部品を多数の作業台へ同時供給する高速物流センター」。S3を耐久性の高い保管場所、Lustreを処理用の高速層として組み合わせられる。

- **A**：HPC、機械学習、並列I/Oに合う。
- **B**：Linux向け汎用NFS共有に向く。
- **C**：Windows SMB向け。
- **D**：低コスト長期保管で、即時ファイル処理用ではない。

「HPC・Lustre・S3連携・高スループット」がまとまればFSx for Lustre。

## 第2問（Storage ｜ NetApp互換ストレージ）

> [!question] 問題（再掲）
> オンプレミスで利用しているNetApp ONTAP環境をAWSへ移行します。NFS、SMB、iSCSI、スナップショット、重複排除など、使い慣れたONTAP機能との互換性を維持したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon FSx for NetApp ONTAP
> - B. Amazon EFS
> - C. Amazon FSx for OpenZFS
> - D. AWS Storage Gateway Volume Gateway

> [!success]- 答え：**A（Amazon FSx for NetApp ONTAP）**
> ONTAPの管理機能とNFS・SMB・iSCSIのマルチプロトコルをAWS上で利用できる。

### 深掘り解説

これは「引っ越し先でも同じメーカーの設備と操作体系を使いたい」という要件。単にファイル共有ができるだけでなく、ONTAP互換が明記されていることが決め手になる。

- **A**：NetAppワークロードの移行に直接対応する。
- **B**：汎用NFSだがSMB・iSCSI・ONTAP機能は持たない。
- **C**：ZFSベースのファイルサーバー向け。
- **D**：オンプレミスからクラウドストレージへつなぐゲートウェイ。

製品名が要件に出る問題では、一般カテゴリより互換サービスを優先して確認する。

## 第3問（Storage ｜ S3オブジェクトの一括処理）

> [!question] 問題（再掲）
> S3バケット内にある数億個の既存オブジェクトに対して、タグ付け、コピー、復元開始などの処理を一括で実行し、進捗と完了レポートを確認したいと考えています。
> 最も適した機能はどれですか。
>
> - A. S3 Batch Operations
> - B. S3 Transfer Acceleration
> - C. S3 Select
> - D. S3イベント通知

> [!success]- 答え：**A（S3 Batch Operations）**
> マニフェストのオブジェクト一覧に対して管理操作を実行し、ジョブ進捗と完了レポートを提供する。

### 深掘り解説

Batch Operationsは「住所録に載った全家庭へ同じ作業を行う巡回計画」。イベント通知は「新しい来客が来た瞬間だけベルを鳴らす」仕組みなので、過去の全対象をさかのぼる用途とは違う。

- **A**：既存の大量オブジェクトへの反復処理に合う。
- **B**：クライアントとS3間の長距離転送を高速化する。
- **C**：単一オブジェクト内部の一部データを抽出する。
- **D**：PUTなど今後のイベントを契機に処理する。

「大量」「既存オブジェクト」「同じ操作」「ジョブ報告」はBatch Operationsのセット。

## 第4問（Storage ｜ S3利用状況の可視化）

> [!question] 問題（再掲）
> 企業は多数のアカウントとリージョンにS3バケットを所有しています。ストレージ使用量、オブジェクト数、暗号化状況、コスト効率などを組織横断のダッシュボードで可視化したいと考えています。
> 最も適した機能はどれですか。
>
> - A. S3 Storage Lens
> - B. S3 Inventory
> - C. AWS Cost Explorer
> - D. Amazon CloudWatch Logs Insights

> [!success]- 答え：**A（S3 Storage Lens）**
> S3固有の利用・保護・コスト最適化メトリクスを組織横断で集約する。

### 深掘り解説

Storage Lensは「倉庫群の経営ダッシュボード」、Inventoryは「各棚に何があるかを列挙した台帳」。問題が求めるのは個別明細ではなく、全体傾向と改善点である。

- **A**：多数アカウント・バケットを集約表示する。
- **B**：オブジェクト単位のメタデータ一覧を出す。
- **C**：AWS全体の費用分析で、S3保護状況などは主目的外。
- **D**：ログに対してクエリする。

「ダッシュボード・組織横断・S3の最適化」はStorage Lens、「CSV/Parquetのオブジェクト一覧」はInventory。

## 第5問（Database ｜ データベース接続のプール）

> [!question] 問題（再掲）
> 多数のLambda関数がAmazon RDSへ短時間に接続するため、接続数が急増してデータベースが不安定になります。アプリケーションの変更を抑えながら接続をプールし、フェイルオーバー時の復旧も速めたいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon RDS Proxy
> - B. Amazon ElastiCache
> - C. Amazon API Gateway
> - D. AWS Database Migration Service

> [!success]- 答え：**A（Amazon RDS Proxy）**
> アプリケーションの多数の論理接続を、より少ないDB接続へ集約・再利用する。

### 深掘り解説

RDS Proxyは「大量の来客を少数の窓口へ整理する受付係」。ElastiCacheは「よく聞かれる答えを受付前に掲示するキャッシュ」で、接続そのものを整理する役割ではない。

- **A**：接続急増とフェイルオーバー影響を軽減する。
- **B**：データの再利用でクエリ数を減らす。
- **C**：HTTP APIの入口を管理する。
- **D**：DBを別環境へ移行する。

「Lambda→RDS」「接続枯渇」「プール」はRDS Proxyの典型キーワード。

## 第6問（Database ｜ グラフデータベース）

> [!question] 問題（再掲）
> ソーシャルネットワークにおけるユーザー同士の関係や、不正取引に関係する口座のつながりを多段で高速に探索したいと考えています。
> 最も適したデータベースサービスはどれですか。
>
> - A. Amazon Neptune
> - B. Amazon DocumentDB（MongoDB互換）
> - C. Amazon DynamoDB
> - D. Amazon Timestream

> [!success]- 答え：**A（Amazon Neptune）**
> ノードとエッジで関係を保存し、複数段のつながりをグラフクエリで探索する。

### 深掘り解説

通常の表が「住所録」なら、グラフDBは「人物と線を結んだ相関図」。誰と誰が何段先でつながるかをたどる処理に向く。

- **A**：関係性探索のためのグラフDB。
- **B**：JSON文書の柔軟な保存・検索向け。
- **C**：キーで高速に値を取得するNoSQL。
- **D**：時間を軸とする測定データ向け。

「友人関係・推薦・不正ネットワーク・知識グラフ」からNeptuneを連想する。

## 第7問（Database ｜ MongoDB互換ドキュメントDB）

> [!question] 問題（再掲）
> 既存のMongoDB向けアプリケーションをAWSへ移行し、JSON形式のドキュメントを扱えるマネージドデータベースを利用したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon DocumentDB（MongoDB互換）
> - B. Amazon Neptune
> - C. Amazon Keyspaces（Apache Cassandra向け）
> - D. Amazon RDS for PostgreSQL

> [!success]- 答え：**A（Amazon DocumentDB）**
> MongoDB互換APIでJSONドキュメントワークロードを実行する。

### 深掘り解説

DocumentDBは「項目数が少しずつ違う書類を、書類の形のまま保管する棚」。リレーショナルDBの固定表や、Neptuneの相関図とはデータモデルが異なる。

- **A**：MongoDB互換という明示要件に合う。
- **B**：グラフ関係の探索用。
- **C**：Cassandra互換のワイドカラム用。
- **D**：SQLの関係モデル用。

試験では互換元の名前が強い決め手になる。MongoDBはDocumentDB、CassandraはKeyspaces。

## 第8問（Application Integration ｜ SaaSデータ連携）

> [!question] 問題（再掲）
> Salesforceに保存された顧客データを、カスタム連携コードをできるだけ書かずにS3へ定期転送して分析に利用したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon AppFlow
> - B. AWS DataSync
> - C. Amazon EventBridge
> - D. AWS Transfer Family

> [!success]- 答え：**A（Amazon AppFlow）**
> 対応SaaSとAWSサービスの間に、設定ベースのデータフローを作る。

### 深掘り解説

AppFlowは「SaaSとAWSの間に用意された専用の搬送ベルト」。API連携コードを毎回自作せず、接続・フィールド変換・実行条件を設定できる。

- **A**：SalesforceなどSaaSデータの取り込みに合う。
- **B**：NFS/SMBやオブジェクトストレージ間のファイル転送向け。
- **C**：小さなイベント通知をルーティングする。
- **D**：SFTP/FTPS/FTPクライアントとのファイル授受向け。

「SaaSの業務データ本体をS3やRedshiftへ」はAppFlow、「SaaSのイベント」はEventBridgeという違いも押さえる。

## 第9問（Containers ｜ サービスディスカバリ）

> [!question] 問題（再掲）
> ECS上のマイクロサービスが動的に増減します。各サービスを論理名で登録し、他のサービスがDNS名やAPIを使って正常なエンドポイントを発見できるようにしたいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Cloud Map
> - B. Amazon Route 53 Resolver
> - C. AWS App Mesh
> - D. Elastic Load Balancing

> [!success]- 答え：**A（AWS Cloud Map）**
> 動的なクラウドリソースを名前空間へ登録し、DNSまたはAPIで発見できるようにする。

### 深掘り解説

Cloud Mapは「店名から現在営業中の支店住所を引く電話帳」。App Meshは「支店間の道路を観測・制御する交通管制」で、登録・発見と通信制御は役割が違う。

- **A**：サービス名から動的エンドポイントを発見する。
- **B**：オンプレミスとVPCなどのDNS問い合わせを転送する。
- **C**：サービス間通信の可視化・ルーティングを行う。
- **D**：固定した入口で複数ターゲットへ負荷分散する。

「サービスレジストリ」「名前で発見」「ECSサービスディスカバリ」ならCloud Map。

## 第10問（Application Integration ｜ マネージドGraphQL API）

> [!question] 問題（再掲）
> モバイルアプリ向けにGraphQL APIを提供し、複数のデータソースへのアクセス、リアルタイム更新、オフライン同期をマネージドで実装したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS AppSync
> - B. Amazon API Gateway REST API
> - C. Amazon MQ
> - D. AWS Step Functions

> [!success]- 答え：**A（AWS AppSync）**
> GraphQLスキーマ、リゾルバー、購読によるリアルタイム更新をマネージドで提供する。

### 深掘り解説

AppSyncは「利用者が欲しい項目だけ注文できるGraphQL専用窓口」。API GatewayはREST、HTTP、WebSocketなどの汎用API玄関である。

- **A**：GraphQL、リアルタイム購読、オフライン同期に合う。
- **B**：REST/HTTP/WebSocket APIを公開する場合の中心選択肢。
- **C**：既存メッセージブローカー互換用。
- **D**：バックエンド処理の手順を管理する。

「GraphQL」が明示されたらAppSyncを第一候補にし、REST/WebSocket単独ならAPI Gatewayを検討する。
