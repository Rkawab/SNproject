## 第8問（Management / Monitoring ｜ 踏み台なしの運用接続）

> [!question] 問題（再掲）
> ある企業の本番環境のAmazon EC2インスタンスは、プライベートサブネットで稼働しています。運用担当者はトラブルシューティングのためにシェル接続を必要としますが、セキュリティポリシーにより踏み台（bastion）サーバーの運用やSSHポートの公開は禁止されています。また、誰がいつ接続したかの操作ログも記録する必要があります。
>
> これらの要件を満たす方法はどれですか。
>
> - A. AWS Systems Manager Session Managerを使用して、インバウンドポートを開けずにEC2インスタンスへ接続し、操作をログに記録する。
> - B. EC2 Instance Connect Endpointを作成し、SSHでプライベートサブネットのEC2へ接続するが、操作ログの集中記録は行わない。
> - C. AWS Client VPNでVPCへ接続し、プライベートIP宛てにSSHする運用にする。
> - D. 踏み台EC2をSystems Managerで管理し、踏み台から各プライベートEC2へSSHする。

> [!success]- 答え：**A（Systems Manager Session Manager）**
> SSMエージェントが内側から外向きに接続するため、インバウンドポートも踏み台も不要。IAMで接続権限を制御し、セッション操作をログに残せるので監査要件も満たす。

### 要件を分解する

| 要件 | 言い換え | 効く部品 |
|---|---|---|
| 踏み台（bastion）の運用が禁止 | **踏み台サーバーを置かない**接続方式 | Session Manager（踏み台レス） |
| SSHポートの公開が禁止 | **インバウンド22番を開けず**に接続 | Session Manager（inbound不要） |
| 誰がいつ接続したか記録 | **接続・操作の監査ログ** | IAM＋セッションログ（S3 / CloudWatch Logs）＋CloudTrail |

### たとえ話：「裏口の合鍵小屋」 vs 「社員証の遠隔解錠」

- **踏み台（bastion）方式**＝ビルの裏口に**合鍵を預けた警備小屋**を建て、そこを経由して各部屋に入る。合鍵の管理と、開けっ放しの裏口（SSHポート）そのものが攻撃対象になり、運用も増える。
- **Session Manager**＝部屋の中の担当者（**SSMエージェント**）が**内側からインターホンを押して管理室（AWS）につなぐ**方式。建物の外側には穴（インバウンドポート）を一切開けない。入室できるのは**社員証（IAM）**を持つ人だけで、いつ誰がどう操作したかは**管理室の入退室ログ（S3 / CloudWatch Logs / CloudTrail）に自動記録**される。

### なぜ A が正解か

Session Manager は、EC2に常駐する **SSMエージェントが外向き（アウトバウンド）でSSMエンドポイントへ接続**して通信路を張る。だから：

- **インバウンドポートを1つも開けない**（SSH 22の公開不要）→ セキュリティポリシー適合。
- **踏み台サーバーが不要**→ 踏み台禁止ポリシー適合。
- 接続可否は **IAM** で制御、**セッションのコマンド履歴を S3 / CloudWatch Logs に記録**、API操作は **CloudTrail** に残る → 「誰がいつ接続したか」の監査要件を満たす。

> [!tip] プライベートサブネットでの注意
> エージェントの外向き通信先（SSM / SSMMessages / EC2Messages）への経路が必要。インターネットに出さないなら**インターフェイス型VPCエンドポイント**、出してよいなら**NAT Gateway**で到達させる。いずれもインバウンドは開けないままにできる。

### 各選択肢の正誤

- **A：正解** … 踏み台不要・SSH非公開・監査記録の3要件をすべて満たす。
- **B：誤り** … EC2 Instance Connect Endpoint も「inbound 22不要・踏み台不要」でプライベートEC2へSSHできるが、**選択肢自身が「操作ログの集中記録は行わない」と明言**しており、監査要件を満たさない。
- **C：誤り** … Client VPNで接続元は絞れるが、**SSHポート運用と鍵管理が残り**、操作ログ記録も別途用意が必要。要件をまとめて満たせない。
- **D：誤り** … Systems Managerで管理しても**踏み台EC2が残る**ため、踏み台サーバー禁止のポリシーに正面から反する。

### 紛らわしい「踏み台レス接続」の整理（SAA頻出）

| 方式 | インバウンドポート | SSH鍵管理 | 操作ログ | ひとこと |
|---|---|---|---|---|
| **Session Manager** | **不要** | **不要（IAM制御）** | **S3 / CW Logsに記録◎** | 監査が要るならこれが定番 |
| EC2 Instance Connect Endpoint | 不要 | 一時鍵を都度発行 | 集中記録は標準でない | SSHクライアントを使いたい時 |
| 踏み台（bastion） | SSH 22を公開 | 必要 | 別途仕込みが必要 | 運用・攻撃面が残り非推奨 |

### SAA的な着眼点

- 「**踏み台なし ＋ SSHポート開けない ＋ 操作ログ（監査）**」の3点セットが出たら、**Session Manager** がほぼ即答。
- inboundを開けない理由＝**エージェントがアウトバウンドでSSMへ接続**する仕組み、を押さえる。
- ひっかけ：Instance Connect Endpoint も「inbound不要・踏み台不要」で正しそうに見えるが、**監査ログ要件で差がつく**（本問は選択肢が「集中記録しない」と自白している）。
- 「Systems Managerで管理」という語に釣られない。**踏み台EC2が残る構成は踏み台禁止に反する**（D）。
