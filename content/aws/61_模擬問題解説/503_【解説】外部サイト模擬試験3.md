## 第1問（Containers ｜ Kubernetes運用のマネージド化（EKS Auto Mode））

> [!question] 問題（再掲）
> コンテナ化したアプリをAWSで大規模デプロイ。オープンソースのKubernetesを積極活用したい。インフラ管理の負担を最小限にし、開発チームがアプリ開発に集中できる環境にしたい。要件を満たすソリューションは？
> - A. EKSクラスターにデプロイし、EKSマネージドノードグループを使用する。
> - B. EC2上でセルフマネージドKubernetesクラスターを構築し、デプロイする。
> - C. ECSクラスターでコンテナアプリをデプロイし、AWS Fargateで実行する。
> - D. EKSクラスターにデプロイし、EKS Auto Modeを使用する。

> [!success]- 答え：**D**
> 2軸で決まる。**①「Kubernetesを使いたい」→ EKS**（ECSは非Kubernetesで除外）。**②「インフラ管理を最小・開発に集中」→ EKS Auto Mode**（ノードの調達・スケール・パッチ・OS管理までAWSが自動運用）。両方を満たすのはD。

論点を割ると、①でC（ECS）が落ち、②でB（セルフマネージド＝負担最大）とA（ノード管理が残る）が落ちて、Dが残る。

### たとえ話：レンタル農地の「手のかかり具合」
Kubernetesという畑（EKS）を借りるとして——
- **セルフマネージド（B）** ＝ 土地の開墾もトラクター整備も全部自分。重労働。
- **マネージドノードグループ（A）** ＝ 畑は貸してもらえるが、**畝づくり・水やり・農機の更新は自分**。だいぶ楽だが手は残る。
- **EKS Auto Mode（D）** ＝ **耕運・水やり・農機メンテまで農園側が全自動**。あなたは作物（アプリ）に専念するだけ。

### 各選択肢をズバッと
- **A：惜しいが誤り** … EKSでKubernetesはOKだが、ノードの構成・AMI更新・スケーリング設計がこちらに残る。「**インフラ管理を最小**」でDに負ける。
- **B：誤り** … セルフマネージドK8sはコントロールプレーンもノードも全部自前＝**管理負担が最大**。要件と真逆。
- **C：誤り** … Fargateで運用は楽だが、**ECSはKubernetesではない**。「オープンソースのKubernetes」要件を満たさない。
- **D：正解** … EKS（Kubernetes）＋Auto Modeでノード〜スケール〜パッチまで自動。インフラ最小・開発集中を両立。

### SAA着眼点
- 「**Kubernetes／オープンソースのK8sを使いたい**」→ **EKS**。**ECSは独自仕様（非Kubernetes）**なので、Kubernetes指定の問題では外す。
- コンテナ運用の「**手放し度**」の階段：**セルフマネージド(EC2) ＜ EKSマネージドノードグループ ＜ EKS Fargate / EKS Auto Mode**（右ほど低運用・手離れ）。
- 「**インフラ管理を最小・開発に集中**」が最優先なら、ノードまで自動化する **EKS Auto Mode**（用途次第でFargateも）。マネージドノードグループより上。
- 補足：**EKS Fargate** はサーバーレスでノードレス、**EKS Auto Mode** はノードのプロビジョニング〜運用をAWSが自動化する比較的新しいモード。どちらも「ノード管理を手放す」方向の選択肢。

## 第2問（Management / Monitoring ｜ コンソールログイン失敗の検知と通知）

> [!question] 問題（再掲）
> AWSマネジメントコンソールへの不正ログインを監視したい。サインインに3回以上失敗したら、管理用メーリングリスト宛にメール通知が届くようにしたい。必要な設定は？
> - A. CloudWatch Logs Insights のクエリにログインイベントを指定。CloudWatch のアラームにメール通知。
> - B. CloudTrail と EventBridge を連携し、ログインイベントをメトリクスとして登録。EventBridge のアクションにメール通知。
> - C. EventBridge の監視対象にログインイベントを設定。EventBridge のアクションにメール通知。
> - D. CloudTrail と CloudWatch Logs を連携し、ログインイベントのフィルタを設定。CloudWatch のアラームにメール通知。

> [!success]- 答え：**D**
> コンソールのサインイン記録は **CloudTrail** が取り、**CloudWatch Logs のメトリクスフィルタ**で「失敗」だけを抽出してカウント、**CloudWatch アラーム→SNS**で「3回以上」を判定してメール通知する。回数のしきい値判定と通知を成立させるのはこの組み合わせだけ。

肝は2つ。**①コンソールサインインの記録元は CloudTrail**（`ConsoleLogin` イベント）。**②「○回以上」という回数のしきい値を数えて通知できるのは CloudWatch のメトリクスフィルタ＋アラーム**。この2点で D 以外は全部落ちる。

### たとえ話：会社の入口の監視カメラと警備員
- **CloudTrail＝入口の監視カメラ**。誰がいつサインインしたかを録画し続ける記録係。ただし自分から警報は鳴らさない。
- **CloudWatch Logs＝録画の保管庫＋仕分け係**。映像（ログ）を溜め、**メトリクスフィルタ**で「ログイン失敗」だけ拾って回数を数える。
- **CloudWatch アラーム＝警備員**。「失敗が3回を超えたら笛を吹け（SNSでメール）」と見張る。

「**録画→失敗だけ抽出してカウント→3回超えたら通報**」——これがAWSでログイン失敗を監視する鉄板の流れ。

### 各選択肢をズバッと
- **A：誤り** … Logs Insights は溜まったログを**人が後から検索・分析**する事後調査向け。クエリ結果にリアルタイムでアラームを直接張る仕組みではなく、常時監視に不向き。
- **B：誤り（最大の罠）** … EventBridge は「イベント発生で何かを起動する」交通整理役で、**イベントを“メトリクスとして登録・集計”する機能を持たない**。「3回以上」という回数判定はできない。"メトリクスとして登録" の一語が誤り。
- **C：誤り** … EventBridge は単体でコンソールログインを直接監視対象にできず（ソースは CloudTrail 経由）、しかも**回数を数えてしきい値判定できない**。1回で即発火はできても「3回以上」が表現できない。
- **D：正解** … CloudTrail（記録）→CloudWatch Logs メトリクスフィルタ（失敗抽出・カウント）→CloudWatch アラーム→SNS（メール）。要件を全部満たす唯一の構成。

### SAA着眼点
- 「**コンソールのサインイン／APIコールを監視**」→ まず **CloudTrail**（記録係）を疑う。
- 「**○回以上／しきい値を超えたら通知**」→ **CloudWatch メトリクスフィルタ＋アラーム＋SNS**。回数を数えるのは CloudWatch の仕事。
- **EventBridge は「イベント駆動で起動」役**。"回数を数える・メトリクス化する" は守備範囲外——という切り分けが頻出ひっかけ。
- 補足：EventBridge も CloudTrail を介してコンソールログイン等のイベントに反応できるが、それは「1回起きたら即アクション」型。**しきい値（3回）を伴う集計監視は CloudWatch メトリクス**が定石。

## 第3問（Security / IAM ｜ DB認証情報のローテーションとコード最小化）

> [!question] 問題（再掲）
> ある会社では Amazon Aurora を利用したWebアプリを運用。Auroraへの認証情報がアプリコードにハードコーディングされており、より安全に管理したい。なお既存コードの変更や追加コードの記述はできる限り避けたい。最も適切なアプローチは？
> - A. Systems Manager Parameter Store に資格情報を保存し、ローテーション用のLambda関数を作成して定期実行する。
> - B. Amazon Cognito でユーザー認証を管理し、認証ユーザーにのみAuroraアクセスを許可。Cognitoユーザープールの機能で資格情報をローテーションする。
> - C. AWS Secrets Manager に資格情報を保存し、Secrets Manager の自動ローテーション機能でローテーションする。
> - D. AWS Secrets Manager に資格情報を保存し、ローテーション用のLambda関数を作成してEventBridgeで定期実行する。

> [!success]- 答え：**C**
> 「コードから分離して保管」＋「定期ローテーション」＋「コードの追加・変更は最小」を同時に満たすのは Secrets Manager の自動ローテーションだけ。Aurora/RDS とのネイティブ統合により、ローテーションの仕組みをAWS側が提供するため自前コードが要らない。

この問題の**勝敗は「追加コードを書きたくない」という一文**で決まる。保管＋ローテーション自体は複数の選択肢でできてしまうので、「自前Lambdaを書く必要があるか」で振り落とす設計になっている。

### たとえ話：金庫の鍵番号を誰が回すか
社員（アプリ）が毎日倉庫（Aurora）を開ける必要があり、その鍵番号（パスワード）を扱う。

- **付箋に鍵番号を書いて貼る** ＝ いまの「コードにハードコーディング」。漏れたら一発アウト。
- **Secrets Manager（自動ローテーション）** ＝ 鍵番号を金庫にしまい、**管理会社が定期的に番号を作り替え、倉庫の鍵穴側も同時に更新**してくれる。社員は「今の番号を金庫に聞く」だけ。**番号を作り替える作業（Lambda）は一切書かなくていい**。
- **Parameter Store＋Lambda／Secrets Manager＋Lambda＋EventBridge** ＝ 金庫はあるが「番号を作り替える係を自分で雇い、手順書（Lambdaコード）を書く」必要がある。「追加コードは避けたい」に反する。
- **Cognito** ＝ そもそも倉庫の鍵の話ではなく、**ビルの入館証（来訪者の本人確認）**を発行する係。土俵が違う。

### 各選択肢をズバッと
- **A：誤り** … Parameter Store には**自動ローテーション機能が無い**ため、ローテーション用Lambdaの自作が必須。「追加コードを書きたくない」に反する。
- **B：誤り（最大の罠）** … Cognito は**アプリのエンドユーザー認証**（サインアップ／サインイン）の仕組み。アプリ↔DB間の接続資格情報を保管・ローテーションする機能は無い。役割が完全に別物。
- **C：正解** … コードからの分離・実行時のAPI取得・マネージドな自動ローテーションを一体で提供。Aurora/RDSとはネイティブ統合で、追加コードも最小。
- **D：誤り** … Secrets Manager を使う点は正しいが、ローテーションを**自前Lambda＋EventBridgeで実装**している。組込みの自動ローテーションを使えば不要な手間で、「最も適切」ではない。

> [!tip] SAA着眼点
> - 「**DB認証情報の保管＋定期ローテーション**」→ **Secrets Manager**（自動ローテーション）が即答パターン。
> - **Parameter Store と Secrets Manager の最頻出比較**：Parameter Store は設定値・パラメータ保管が主で**ローテーションは非対応**（やるなら自前Lambda）。Secrets Manager は**シークレット特化＋マネージド自動ローテーション**（RDS/Aurora/Redshift/DocumentDBはネイティブ対応）。
> - 「**コードを変えたくない／運用を最小化**」の一文があれば、**自前Lambda系の選択肢を落とす**合図。
> - **Cognito ＝ エンドユーザー認証**。アプリ↔DB間の資格情報とは無関係、という典型ひっかけ。

## 第4問（Database ｜ 3層Webアプリの高可用DB・セッション・キャッシュ構成）

> [!question] 問題（再掲）
> あるEコマース会社が、オンプレの3層Webアプリ（Web／アプリ／PostgreSQL DB）をAWSへ移行。高可用性とスケーラビリティ、将来のトラフィック急増に対応したい。Web・アプリ層は3AZにまたがるAuto Scalingで稼働予定。さらにユーザーのセッション情報を効率的に管理し、キャッシュでパフォーマンスを上げたい。要件を満たすソリューションは？
> - A. DBをAurora for PostgreSQLに移行。セッション管理にDynamoDB、キャッシュにRDS Proxyを使用。
> - B. DBをマルチAZ配置のRDS for PostgreSQLに移行。セッション管理とキャッシュにElastiCache for Memcachedを使用。
> - C. DBをマルチAZ DBクラスター配置のRDS for PostgreSQLに移行。セッション管理とキャッシュにElastiCache for Memcachedを使用。
> - D. DBをマルチAZ DBクラスター配置のRDS for PostgreSQLに移行。セッション管理とキャッシュにElastiCache for Redisを使用。

> [!success]- 答え：**D**
> 3つの判断軸で削っていく。**①高可用性＋読み取りスケール＋将来の急増 → マルチAZ DBクラスター**（リーダー2台が読める）。**②セッション管理 → Redis**（レプリケーション・永続化・フェイルオーバーがあり、ノードが落ちてもセッションを失わない）。**③キャッシュもRedisに一本化**。全部満たすのはD。

論点を割ると、まずキャッシュに **RDS Proxy** を当てているAが即落ち（Proxyはキャッシュではない）。次にセッションを **Memcached** に置くB・Cが落ち（落ちるとセッション消失）、さらにBはDBが"マルチAZインスタンス"で読み取りスケールに劣る。残るDが正解。

### たとえ話：人気レストランの厨房を増床する
オンラインストア＝繁盛レストラン。

- **DB＝厨房本体**。
  - **マルチAZ "インスタンス"** ＝ 予備の厨房はあるが**普段は締め切り（待機専用で読めない）**。本番が倒れた時だけ稼働。可用性はあるが**読み取りを分散できない**。
  - **マルチAZ "DBクラスター"** ＝ **3AZに厨房を配置し、予備2つも「読み取り客」をさばける**。倒れても即切替＋普段から読み負荷を分散＝**可用性とスケーラビリティの両取り**。将来の急増にはこちら。
- **セッション管理＝客の注文票**（消えると大事故）。
  - **Memcached＝使い捨てメモ帳**。速いが**レプリケーション・永続化なし**。ノードが落ちると注文票が消える＝強制ログアウト多発。
  - **Redis＝複製付きの注文台帳**。レプリケーション・フェイルオーバー・永続化があり、障害でも注文票が残る。セッションストアの定番。
- **キャッシュ＝よく出る料理の作り置き**。RedisでもMemcachedでも可だが、②でRedisが要るのでRedisに統一するのが素直。

### 各選択肢をズバッと
- **A：誤り（最大の罠）** … 「キャッシュに **RDS Proxy**」が決定的にNG。**RDS Proxyはコネクションプーリング（DB接続の使い回し・枯渇防止）**の道具で、データをキャッシュするサービスではない。「キャッシュ」と書いてあったら即除外。
- **B：誤り** … マルチAZ"インスタンス"は**待機系が読めず読み取りスケールに劣る**。加えてセッションを**Memcached**に置くのは永続性・冗長性の面で不適。二重に弱い。
- **C：誤り** … DBはマルチAZ DBクラスターで良いが、**セッションをMemcached**に置く点がNG（ノード障害でセッション消失）。あと一歩。
- **D：正解** … マルチAZ DBクラスター（高可用＋読み取りスケール）＋Redis（セッションを安全に保持＆キャッシュ）。3軸すべて満たす。

> [!tip] SAA着眼点
> - **セッション管理 → Redis**。Memcachedは**永続化・レプリケーション・フェイルオーバーが無い**ため、落ちるとセッションが消えてセッションストアに不向き。Redis/Memcached比較の最頻出ポイント。
> - **「高可用性＋読み取りスケール＋将来の急増」→ マルチAZ DBクラスター**（リーダーが読み取り可）。単なる**マルチAZインスタンス**は「待機系は読めない＝可用性専用」と覚える。
> - **RDS Proxy ＝ コネクションプーリング**であってキャッシュではない。Lambda等からの大量接続でDB接続が枯渇するのを防ぐ用途。「キャッシュ」と紐づけたら誤り。
> - Memcachedが向くのは「単純・水平スケールするシンプルなキャッシュ」。永続化・複製・高度なデータ構造（セッション、ランキング、Pub/Sub等）が要るならRedis。

> [!note] Redis と Memcached の使い分け（補助）
> | 観点 | ElastiCache for Redis | ElastiCache for Memcached |
> |---|---|---|
> | レプリケーション／フェイルオーバー | あり（高可用） | なし |
> | 永続化 | あり | なし（揮発） |
> | データ構造 | 豊富（List/Set/Sorted Set/Pub/Sub等） | 単純なKey-Valueのみ |
> | セッションストア | 適（消えない） | 不向き（落ちると消える） |
> | 向く用途 | セッション・ランキング・複雑なキャッシュ | 単純で大規模な揮発キャッシュ |

## 第5問（Security / IAM ｜ クロスアカウントでのSQSアクセス許可）

> [!question] 問題（再掲）
> ある大手製造業は、部門ごとに異なるAWSアカウントを運用している。**マーケティング部門**は顧客フィードバックを処理するアプリを持ち、データを **SQSキュー** に格納している。**データ分析部門**は、自部門のAWSリソースから、マーケ部門が管理するSQSキューに **ポーリングして取得** したい。マーケ部門はどうSQSへのアクセスを提供すべきか。
> - A. SQSキューのアクセスポリシーを設定して、データ分析部門のAWSリソースからのアクセスを許可する。
> - B. データ分析部門のアカウント内に新しいIAMユーザーを作成し、クロスアカウント設定でアクセスを許可する。
> - C. マーケティング部門のIAMポリシーを設定して、データ分析部門のリソースからのアクセスを許可する。
> - D. データ分析部門のアカウント内にSNSトピックを作成し、SQSのアクセスポリシーでSNS→SQSを許可する。

> [!success]- 答え：**A**
> キューを持っているのはマーケ部門。だから「誰に貸すか」は**所有者がSQSのアクセスポリシー（リソースベースポリシー）で宣言**するのが筋。

肝は **「IDベースポリシー」と「リソースベースポリシー」のどちらでクロスアカウントを開けるか**。あなたが選んだ C はこの2つを取り違えた**最大の罠**です。

### たとえ話：貸し倉庫のカギ
SQSキューを「マーケ部門が借りている貸し倉庫」だと思ってください。データ分析部門が中身を取りに来たい。

- **A＝倉庫のドアに“この会社の人は入っていい”と許可リストを貼る**（リソースベースポリシー）。倉庫の持ち主（マーケ部門）が、入れる相手（データ分析部門のアカウント）を名指しで許可する。これがクロスアカウントの正攻法。
- **C＝マーケ部門が“自分の社員証で何ができるか”の規則をいじる**（IDベースポリシー）。でも社員証の規則をどういじっても、**他社（データ分析部門）の人の入館許可にはならない**。そもそも誰に権限を付けているのか対象が違う。
- **B＝他社の中に新しい社員証を発行する**。発行しても、マーケ倉庫のドアの許可リスト（リソースポリシー）がなければ入れない。手間が増えるだけ。
- **D＝わざわざ放送設備（SNS）を別棟に建てて中継させる**。でも今回は「自分から取りに行く（ポーリング）」話。配信（プッシュ）の仕組みは要らない過剰設計。

**ポイント：クロスアカウントは「相手のID側」と「自分のリソース側」の両方が噛み合って初めて通る。SQS・S3・KMSのように“リソースにポリシーを貼れる”サービスは、所有者がリソースポリシーで相手を許可するのが定番。**

### 各選択肢をズバッと
- **A：正解** … SQSはリソースベースポリシー（キューのアクセスポリシー）を持てる。`Principal` にデータ分析部門のアカウント/ロールを指定して `sqs:ReceiveMessage` 等を許可すれば、クロスアカウントのポーリングが通る。所有者が出すべき設定はこれ。
- **B：誤り** … データ分析側にIAMユーザーを新設しても、マーケ側キューの許可は**所有側のリソースポリシーが別途必要**。さらに長期認証情報のIAMユーザー乱立は管理・セキュリティ面でベストプラクティス外（本来はロール）。
- **C：誤り（最大の罠）** … IDベースポリシー（IAMポリシー）は「**そのアカウント内のプリンシパルが何をできるか**」を定めるもの。マーケ部門のIAMポリシーをいじっても、**別アカウント（データ分析部門）のプリンシパルに権限は与えられない**。IDベース↔リソースベースの典型的な取り違え。
- **D：誤り** … SNSを挟むのは**プッシュ型のファンアウト**構成。要件は「データ分析側から**ポーリングして取得**」なので方向が逆で、不要なSNSを足す過剰設計。

> [!tip] SAA着眼点
> - 「**クロスアカウントで自分のリソースを相手に使わせる**」→ **リソースベースポリシー**（SQS/S3/KMS/Lambda/SNS/ECR 等が対応）。所有者が `Principal` で相手を許可。
> - **IDベース vs リソースベース**：IDベース＝「このプリンシパルは何ができるか（自アカウント内）」、リソースベース＝「このリソースに誰がアクセスできるか（他アカウントの`Principal`も指定可）」。クロスアカウントは**両側の許可が揃って通る**（リソース側で相手アカウントを許可＋相手側IAMでも許可）。
> - 「ポーリングして取得」＝**プル型（SQS）**。SNSが出てきたら「これはプッシュ型では？要件と向きが合うか」を疑う。

## 第6問（Management / Monitoring ｜ IAMリソースの構成変更検知）

> [!question] 問題（再掲）
> 監査チームの通達でIAM（ユーザー/グループ/ロール）の運用を見直し。以降、IAMリソースの設定（構成）が変更されたら検知したい。適切なアクションは？
> - A. CloudWatchでIAMメトリクスを監視。変更時にLambdaで検知し、Lambda→SNS FIFOトピックへ通知。
> - B. CloudTrailでIAM運用を監視し、Lambdaで設定変更を検知。SQSキューからLambdaへ通知。
> - C. Trusted AdvisorでIAM運用を監視。設定変更時にSNSトピックへ通知。
> - D. IAMリソースを監視するAWS Configルールを作成。設定変更時にSNSトピックから通知。

> [!success]- 答え：**D**
> 「リソースの**設定（構成）が変更されたら検知**」は AWS Config の定義そのもの。Configルールで IAM リソースを監視し、構成変更時に SNS で通知する。

この問題は **CloudTrail と AWS Config の役割分担**が分かっていれば一発。「**設定変更を検知**」というキーワードが Config を名指ししている。

> [!info] ここを押さえる
> **CloudTrail＝「誰が・いつ・どのAPIを叩いたか」（操作の証跡・監査）／ AWS Config＝「設定が今どうなっていて、どう変わったか」（構成の記録・評価・準拠チェック）**。「**構成変更の検知**」と来たら Config。

### たとえ話：防犯カメラ（CloudTrail）と 設備点検簿（Config）
- **CloudTrail＝防犯カメラ** … 「誰が部屋に入ったか」を録画。犯人特定は得意だが、「**模様替えされたか**」を判定する係ではない。
- **AWS Config＝設備点検簿** … 部屋のレイアウト（設定）を常に記録し、**変わった瞬間に「ここが変更されました」と知らせる**。今回ほしいのはこっち。

### 各選択肢をズバッと
- **A：誤り** … IAMの設定変更は**メトリクス（数値）ではない**。CloudWatchメトリクス監視では構成変更を捉えられない。SNS FIFOも無関係。
- **B：誤り** … CloudTrailでも変更イベントは拾えるが、本問の直球は構成変更を評価する **Config**。さらに「**SQSキューからLambdaへ通知**」という配線も不自然（通常はイベント→Lambda→SNS）。回りくどく最適でない。
- **C：誤り** … Trusted Advisor は**ベストプラクティス診断**（コスト・セキュリティの推奨）。リアルタイムの設定変更検知ツールではない。
- **D：正解** … AWS Config ルールでIAMリソースを監視 → **構成変更を検知 → SNSで通知**。要件に直結。

### SAA着眼点
- 「**設定／構成が変更されたら検知・記録・準拠評価**」→ **AWS Config**（即答ワード）。
- 「**誰がいつ何のAPIを実行したか（操作の証跡・監査）**」→ **CloudTrail**。この切り分けが頻出ひっかけ。
- **Trusted Advisor＝推奨チェック／ CloudWatch＝メトリクス・ログの監視**。どちらも「構成変更検知」の主役ではない。
- 合わせ技も定番：**CloudTrail（記録）＋Config（構成評価）＋EventBridge/SNS（通知）**。

## 第7問（Serverless ｜ スパイク＋アイドルなワークロードのサーバーレス化）

> [!question] 問題（再掲）
> テキスト翻訳WebアプリをEC2で運用（S3保存→EC2翻訳→S3保存→DL）。過去3か月で、翻訳が遅延するほどの大量リクエストが突発する一方、長時間使われない期間もある（スパイク＋アイドル）。リソース効率化とコスト削減の提案は？
> - A. 翻訳処理をEC2→Lambdaに変更。SQSとLambdaを連携してリクエストをキューに投入。
> - B. Auto Scalingで動的スケーリング、増加分はスポットインスタンスを契約。
> - C. CloudFrontを配置し、S3のデータをキャッシュして配信。
> - D. リクエストのない期間はインスタンスを停止し、S3にファイルがアップされたら起動するEventBridgeを作成。

> [!success]- 答え：**A**
> 負荷の形が「**突発スパイク＋長いアイドル**」＋コスト削減＝**サーバーレスの独壇場**。Lambdaが需要に応じて自動スケールし**従量課金（アイドル時ほぼ無料）**、SQSがスパイクを**緩衝**して非同期処理する。

この問題は**ワークロードの「形」が答えを決める**。「跳ねる＋暇が長い＋安くしたい」が揃ったら、常時起動のEC2系より **SQS＋Lambda** が素直に最適。

### たとえ話：常駐スタッフ vs 出来高払いの応援
- **今のEC2** ＝ **正社員を常時雇用**。客ゼロでも給料発生（アイドルコスト）。急に行列ができると人手不足で遅延。
- **Lambda＋SQS** ＝ **客が来た瞬間だけ応援を呼ぶ出来高払い**。行列（スパイク）は**整理券（SQS）**でさばき、暇なら誰も雇わない＝払わない。

### 各選択肢をズバッと
- **A：正解** … Lambdaで自動スケール＋従量課金、SQSでスパイク吸収。アイドル時コストゼロで効率・コスト両取り。
- **B：不適** … Auto Scaling＋スポットは固定台数よりマシだが、**EC2の土台コスト・管理が残り、アイドルでも最低台数ぶん課金**。長いアイドルにはサーバーレスが有利。
- **C：誤り** … CloudFrontは**配信のキャッシュ**。翻訳処理（コンピュート）のスパイク・コストには無関係。論点違い。
- **D：不適** … 停止EC2をEventBridgeで起動は、**起動の待ち時間**・運用の複雑さが残り、依然EC2管理。サーバーレス化のAに劣る。

### SAA着眼点
- 「**突発スパイク＋アイドルが長い＋コスト削減**」→ **Lambda（従量・自動スケール）**。常時起動EC2は暇な時間に払い続けるので不利。
- 「**急なリクエストを取りこぼさず非同期処理**」→ **SQSで緩衝＋Lambdaで処理**（疎結合）。
- EC2前提の選択肢（Auto Scaling／停止起動）は、**サーバーレス化が選べる文脈ではたいてい次点**。「サーバー管理を無くす一手」があるか探す。
- CloudFrontは**配信高速化・キャッシュ**で、バックエンドの**計算負荷・コスト**は解決しない（論点すり替えの罠）。

## 第8問（Application Integration ｜ SQSバックログに基づくAuto Scaling）

> [!question] 問題（再掲）
> 分析システムは「変換アプリ」と「分析アプリ」の2段で、各々Auto ScalingのEC2群で動く。変換は1件1分以内、分析は数分。各処理でデータ損失が出ないよう、Auto Scalingに適切なスケーリングポリシーを設定したい。最も適切な提案は？
> - A. 変換用・分析用にSQSキューを作成。SQSキューからの通知に基づくスケーリング。アプリはポーリング処理。
> - B. 変換用・分析用にSNSトピックを作成。SNSの通知に基づくスケーリング。アプリはSNSから受け取って処理。
> - C. 変換用・分析用にSQSキューを作成。SQSキューのバックログに基づくスケーリング。アプリはポーリング処理。
> - D. 変換アプリは処理後5分待機して分析へ送信。各EC2のCPU使用率でスケーリング。

> [!success]- 答え：**C**
> キュー処理ワーカーのAuto Scalingの正しい指標は「**SQSのバックログ（未処理メッセージ数／インスタンスあたりのバックログ）**」。溜まったら増やすので、捌ききれず溢れて**データ損失する前にスケールアウト**できる。

AとCは「2段それぞれSQS＋ポーリング」まで同じで、**違いはスケーリング指標が「通知」か「バックログ」か**だけ。ここだけで勝負が決まる。

### たとえ話：レジの増やし方
- **C：行列の長さ（バックログ）で増やす** … 「**待ち客が◯人を超えたらレジを開ける**」。溜まり具合で増やすので、捌けず客が帰る（データ損失）前に対応できる。正解。
- **A：「通知」を待つ** … でもSQSという行列は**自分から「混んでます」と放送しない**（プル型＝店員が見に行く）。「通知でレジを増やす」は仕組み上できない。
- **D：レジ係の疲労（CPU）で増やす** … 行列が伸びてもCPUが低ければ増えない＝**行列＝負荷を反映しない**。「5分待ってから渡す」も無意味な遅延。

### 各選択肢をズバッと
- **A：誤り** … SQSは**プル型で“通知”しない**。「SQSキューからの通知に基づくスケーリング」という指標が成立しない。Cの劣化版。
- **B：誤り** … SNSは**プッシュ型のファンアウト**で処理待ちを溜めない。バックログで測れず、データ損失防止にも不向き。キュー処理はSQS。
- **C：正解** … 各段をSQSで疎結合し、**バックログ（未処理数）でスケール**。溢れる前に増やしデータ損失を防ぐ。
- **D：誤り** … **CPUは行列の長さを反映しない**（待ち件数が多くてもCPU低なら増えず取りこぼす）。「5分待機」も不要なハック。

### SAA着眼点
- **「キュー処理のワーカーをAuto Scaling」→ 指標は“キューのバックログ（ApproximateNumberOfMessages／backlog per instance）”** が公式の定石。**CPUではない**。
- **SQS＝プル型（ポーリング、通知しない）／SNS＝プッシュ型（通知・ファンアウト）**。「キューからの通知」という表現はほぼ罠。
- **データ損失を防ぐ＝溜められる耐久バッファ（SQS）＋溜まりで増やす**。SNSは溜めないので不適。
- 多段パイプラインは**各段に独立のキュー**を置いて疎結合＋独立スケール。

## 第9問（Security / IAM ｜ EBS暗号化の予防ガードレールと自動修復）

> [!question] 問題（再掲）
> EC2に接続されたEBSのデータが暗号化されていることを確実にしたい。①既存の未暗号化EBSを自動検出して暗号化するプロセスが欲しい。②未暗号化EBSが新規作成されるのも防止したい。最適なアプローチは？（2つ選択）
> - A. セキュリティグループを設定して、未暗号化EBSの作成を防ぐ。
> - B. IAMポリシーを設定し、未暗号化EBSの作成を防ぐ。
> - C. Macieで未暗号化EBSを特定し、SSM Automationで暗号化修正。
> - D. GuardDutyで未暗号化EBSを検出し、Session Managerで修正して自動暗号化。
> - E. Configで未暗号化EBSを自動検出し、SSM Automationで暗号化修正。

> [!success]- 答え：**B ＋ E**
> 要件は「**新規作成を防ぐ（予防）**」＋「**既存を検出して直す（検出＆修復）**」の2本立て。予防は **IAMポリシーの条件**でB、検出＆自動修復は **AWS Config＋SSM Automation**でE。

問題文が「既存を直す」「新規を防ぐ」と2つ要求しているので、答えも2方向。予防＝B、検出＆修復＝E。残りA・C・Dは全部**サービスの役割違い**。

### たとえ話：金庫の「入口チェック」と「巡回点検」
- **B（IAMポリシー）＝入口の警備員** … 「**暗号化してない箱は持ち込み禁止**」と入口で弾く（新規作成を防ぐ）。
- **E（Config＋SSM Automation）＝巡回点検＋自動補修** … 既に置いてある箱を見回り（Config）、未暗号化の箱を見つけたら**その場で自動的に鍵付き箱へ入れ替える**（SSM Automation）。

### 各選択肢をズバッと
- **A：誤り** … セキュリティグループは**ネットワークのファイアウォール**。EBSの作成可否・暗号化に無関係。
- **B：正解** … IAMポリシーの**条件（`ec2:Encrypted`がfalseなら拒否）**で未暗号化ボリュームの作成APIを拒否（予防ガードレール）。
- **C：誤り** … **MacieはS3の機密データ（PII等）検出**。EBSの暗号化状態は対象外。
- **D：誤り** … **GuardDuty＝脅威検知**でコンプラ検出はしない。**Session Manager＝シェル接続**で自動修復ツールではない。2つとも用途違い。
- **E：正解** … **Configで検出（encrypted-volumesルール）＋SSM Automationで自動修復**。既存の未暗号化EBSを直す王道。

### SAA着眼点
- 「**未準拠リソースの新規作成を防ぐ（予防）**」→ **IAMポリシーの条件**（または SCP）で作成APIを拒否。**SGはネットワークなので無関係**。
- 「**既存の未準拠を検出して自動修復**」→ **AWS Config（検出）＋ Systems Manager Automation（修復アクション）**の鉄板コンビ。
- 役割取り違えの罠：**Macie＝S3の機密データ／GuardDuty＝脅威検知／Session Manager＝シェル接続**。いずれも「設定コンプラの検出・修復」役ではない。
- 補足：「EBS暗号化を組織全体で強制」なら **アカウントの“EBSデフォルト暗号化”有効化**も定番（本問の選択肢には無いが頻出）。

## 第10問（Serverless ｜ エッジでの動的画像処理（Lambda@Edge））

> [!question] 問題（再掲）
> ストックフォト企業が **CloudFront** で世界中に高解像度画像を配信。本体は **ALB＋EC2**、素材は **S3**。プレビュー時に **ユーザーIDとアクセス日を右下に埋め込む透かし（ウォーターマーク）を動的合成**して返したい。世界各地から **毎秒数百件**、**オリジン負荷を抑え運用オーバーヘッド最小**にしたい。最適な構成は？
> - A. EC2に画像処理ライブラリを入れ、リクエスト毎にS3からDLした素材へ透かし合成して返す。
> - B. CloudFront Response Headers Policyで、ユーザーID・アクセス日をHTTPヘッダーに追加して透かし合成する。
> - C. API Gatewayのエッジ最適化エンドポイントで受け、単一リージョンの通常Lambdaに転送。LambdaがS3原本を取得し合成してAPIレスポンスで返す。
> - D. CloudFrontでLambda@Edgeを実行し、S3から取得した素材に画像処理ライブラリで透かしを合成して返す。

> [!success]- 答え：**D**（あなたの回答も D ＝正解）
> 「CloudFront配信」＋「画像をその場で加工（計算が必要）」＋「世界中から毎秒数百件」＋「オリジン負荷・運用を最小」が揃ったら、ユーザー近接のエッジで本格処理を走らせる **Lambda@Edge** が定番解。

肝は **「エッジでどこまでの処理ができるか」のグラデーション**。CloudFrontまわりの“処理を挿し込む部品”は機能と重さが段階的に違い、今回は「画像のピクセルを加工する」という重めの処理なので、一番パワーのある Lambda@Edge を選ぶ。

### たとえ話：高速道路のサービスエリアで加工する
画像配信を「全国に荷物を届ける物流」だと思ってください。S3が中央倉庫、CloudFrontのエッジが各地の**サービスエリア（SA）**です。

- **D＝各SAに小さな加工工房（Lambda@Edge）を置く**。客が来たら、中央倉庫から取り寄せた素材にその場で名入れ（透かし）して渡す。客の近くで完結するので**中央倉庫（オリジン）に毎回行かずに済み、待ち時間も短い**。工房はサーバーレスで人を常駐させる必要もない（運用最小）。
- **A＝中央倉庫の隣の1工場（EC2）に全部やらせる**。全国の客の名入れ依頼が**1拠点に集中**し、工場はパンクしがち。増設・監視も全部自前（運用重い）。
- **B＝SAの掲示板に貼り紙を足すだけ（Response Headers Policy）**。荷物の伝票（HTTPヘッダー）に文字は足せるが、**商品そのもの（画像のピクセル）には一切手を加えられない**。透かし合成は原理的に無理。
- **C＝SAでは受け付けるが、加工は遠い1か所の工場（単一リージョンLambda）に送って戻す**。エッジで受けた意味が薄れ、**遠距離往復＋1拠点集中**でレイテンシも負荷も不利。

### 各選択肢をズバッと
- **D：正解** … Lambda@Edgeはエッジロケーションで**画像処理ライブラリを使う重めの動的処理**を実行できる（実体はus-east-1で作成し各エッジへ複製実行）。S3原本を取得→透かし合成→返却を**ユーザー近接**でこなし、オリジン負荷を抑えつつサーバーレスで運用最小。
- **A：誤り** … 技術的には可能だが、合成負荷が**オリジン（EC2/S3）に集中**し、世界規模トラフィックへのスケールと監視・増設の**運用負荷が大きい**。「オリジン負荷を抑え運用最小」に真っ向から反する。
- **B：誤り** … Response Headers Policyは**レスポンスのHTTPヘッダーを足し引きするだけ**。画像のバイナリ加工はできず、透かし合成は不可能。「ヘッダーに足す」を「画像に足す」と錯覚させるひっかけ。
- **C：誤り** … 単一リージョンのLambdaに集約すると、世界各地のユーザーから遠く、毎秒数百件が**1リージョンに集中**してレイテンシ・負荷で不利。エッジ実行のLambda@Edgeに明確に劣る。

> [!tip] SAA着眼点
> - **CloudFrontの処理挿し込み部品を3段で覚える**：①**Response Headers Policy**＝ヘッダーを足し引き（コード不要・設定のみ）。②**CloudFront Functions**＝超軽量JS（ヘッダー書換・URLリライト・簡易認証/リダイレクト。サブミリ秒・激安・ネットワーク/ファイルシステム不可）。③**Lambda@Edge**＝重い処理OK（画像加工・S3等への**ネットワークアクセス**・大きめの計算。CloudFront Functionsより高機能だが高レイテンシ・高コスト）。
> - 「**エッジで画像加工/本格的な動的処理/外部アクセスが要る**」→ **Lambda@Edge**。「**ヘッダー書換・URLリライトだけを最速最安で**」→ **CloudFront Functions**（cf. [[209_【問題】類似サービス比較9#第8問（Serverless ｜ CloudFront Functions vs Lambda@Edge）]]）。
> - 「**オリジン負荷を抑える＋世界中から大量アクセス＋動的**」というキーワードの組はエッジコンピューティング（Lambda@Edge）を疑う。
