## 第1問（Security / IAM ｜ Inspector と GuardDuty）

> [!question] 問題（再掲）
> EC2インスタンスにインストールされたパッケージの既知の脆弱性を継続的に検出し、修正の優先順位を付けたいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon Inspectorでワークロードを継続スキャンする
> - B. Amazon GuardDutyでCloudTrailやVPC Flow Logsを分析する
> - C. Amazon Detectiveで検出結果の関係を可視化する
> - D. AWS Security Hub CSPMでセキュリティ標準を有効化する

> [!success]- 答え：**A（Amazon Inspector）**
> パッケージのCVEとワークロードの露出を評価する。

### 比較の核心

Inspectorは「壊れやすい鍵を点検する」、GuardDutyは「不審者の行動を監視する」。脆弱性は弱点、脅威検知は悪用の兆候である。

- **A**：既知のソフトウェア脆弱性という要件に一致。
- **B**：API、DNS、ネットワークなどから侵害兆候を検知。
- **C**：検知後の根本原因調査。
- **D**：結果集約と態勢評価。

SAAではCVE・ECR・EC2パッケージはInspector、不審API・暗号資産採掘・異常通信はGuardDuty。

## 第2問（Security / IAM ｜ Security Hub と Detective）

> [!question] 問題（再掲）
> 複数アカウントのGuardDutyやInspectorから届く検出結果を標準形式で集約し、セキュリティ標準への準拠状況も一元確認したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Security Hub CSPM
> - B. Amazon Detective
> - C. Amazon GuardDuty
> - D. AWS Audit Manager

> [!success]- 答え：**A（AWS Security Hub CSPM）**
> 結果の集約とセキュリティ態勢管理を行う。

### 比較の核心

Security Hubは「警報一覧を集める司令室」、Detectiveは「一件の事件を調べる捜査班」。全体を俯瞰するか、個別を深掘りするかで分ける。

- **A**：集約・標準準拠の両要件を満たす。
- **B**：ユーザーやリソースの関係をたどる調査用。
- **C**：検出結果を作るサービスの一つ。
- **D**：監査証拠をフレームワークに沿って整理する。

「単一画面・複数サービス・複数アカウント」はSecurity Hub、「なぜ起きたか」はDetective。

## 第3問（Management / Monitoring ｜ CloudTrail と Config）

> [!question] 問題（再掲）
> セキュリティグループのインバウンドルールが、先週のある時点でどのような設定だったかを確認し、現在の構成が社内ルールに準拠しているか評価したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Config
> - B. AWS CloudTrail
> - C. Amazon CloudWatch
> - D. AWS X-Ray

> [!success]- 答え：**A（AWS Config）**
> リソース構成の履歴を保持し、Config Rulesで準拠状況を評価する。

### 比較の核心

Configは「部屋のレイアウトを時点ごとに撮った写真」、CloudTrailは「誰がいつ家具を動かしたかの作業記録」。状態と操作を分けて考える。

- **A**：過去構成と現在の準拠評価に合う。
- **B**：変更APIの実行者や時刻を調べる場合に使う。
- **C**：性能やログを監視する。
- **D**：アプリのリクエスト経路を追跡する。

構成履歴・準拠はConfig、API監査・誰が変更したかはCloudTrail。両方を併用する問題も多い。

## 第4問（Security / IAM ｜ Artifact と Audit Manager）

> [!question] 問題（再掲）
> 監査担当者から、AWSが取得しているSOCレポートとPCI関連文書の提出を求められました。自社リソースの設定証拠を新たに収集する必要はありません。
> 最も適したサービスはどれですか。
>
> - A. AWS ArtifactからAWSのコンプライアンス文書を取得する
> - B. AWS Audit Managerで自社環境の評価を作成する
> - C. AWS Configで構成スナップショットを配信する
> - D. AWS Security Hub CSPMで標準チェックを実行する

> [!success]- 答え：**A（AWS Artifact）**
> AWS自身の監査報告書や契約文書をオンデマンドで取得する。

### 比較の核心

Artifactは「建物メーカーが発行した品質証明書を受け取る場所」。Audit Managerは「自社が建物を正しく運用している証拠を集める担当」である。

- **A**：AWS側のSOC・PCI文書の取得に合う。
- **B**：自社AWS利用の証拠収集を自動化する場合。
- **C**：構成状態は証拠の一部にすぎない。
- **D**：セキュリティ標準の自動チェック用。

問題文の主語を確認する。AWSの証明はArtifact、自社の統制評価はAudit Manager。

## 第5問（Security / IAM ｜ Firewall Manager と WAF）

> [!question] 問題（再掲）
> Organizations配下にある新規・既存アカウントのCloudFrontディストリビューションへ、共通のAWS WAFルールを自動で適用し、非準拠リソースを中央で確認したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Firewall Managerで組織向けポリシーを作成する
> - B. 各アカウントでAWS WAFのWeb ACLを手動作成する
> - C. AWS Network Firewallを各VPCに配置する
> - D. AWS Shield Standardを無効化してShield Advancedだけを使う

> [!success]- 答え：**A（AWS Firewall Manager）**
> WAF等のポリシーを組織全体へ配布し、適用状況を中央管理する。

### 比較の核心

WAFは「一つの入口の検査装置」、Firewall Managerは「全拠点に検査装置のルールを配る本部」。組織スケールの管理が追加されたらFirewall Managerになる。

- **A**：既存・新規アカウントへの自動適用に合う。
- **B**：手動運用で非準拠や設定漏れが生じやすい。
- **C**：VPC内トラフィックの検査用。
- **D**：Shield Standardは自動提供で、問題の解決にならない。

「Organizations・中央適用・非準拠管理」はFirewall Managerの決め手。

## 第6問（Management / Monitoring ｜ Session Manager と 踏み台）

> [!question] 問題（再掲）
> プライベートサブネットのEC2へ管理者が接続します。SSH鍵の管理、踏み台EC2のパッチ適用、ポート22の開放をすべてなくし、IAMで接続権限を制御したいと考えています。
> 最も適した構成はどれですか。
>
> - A. Systems Manager Session Managerを使用する
> - B. パブリックサブネットに踏み台EC2を配置する
> - C. 各EC2へElastic IPを割り当てる
> - D. Client VPN経由で各EC2へSSH接続する

> [!success]- 答え：**A（Session Manager）**
> インバウンド管理ポートと鍵を使わず、IAM認可でセッションを開始する。

### 比較の核心

踏み台は入口を集約するが、その入口のOS・鍵・可用性を守り続ける必要がある。Session Managerは管理面をサービス化し、その運用対象を減らす。

- **A**：列挙された不要化要件をすべて満たす。
- **B**：踏み台管理が残る。
- **C**：公開範囲と攻撃面を増やす。
- **D**：経路は非公開になるがSSH鍵とポートは残る。

「最小運用」「鍵なし」「ポートなし」「IAM」はSession Manager。

## 第7問（Management / Monitoring ｜ Control Tower と Organizations）

> [!question] 問題（再掲）
> 新規AWSアカウントを標準化された方法で払い出し、ログアーカイブや監査アカウントを含むランディングゾーンを自動構成し、統制ルールを継続適用したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Control Tower
> - B. AWS Organizationsのみ
> - C. AWS IAM Identity Centerのみ
> - D. AWS Resource Access Manager

> [!success]- 答え：**A（AWS Control Tower）**
> Organizations等を組み合わせ、標準ランディングゾーンを自動構築する。

### 比較の核心

Organizationsは「会社の組織図と全社規則」、Control Towerは「新部署の標準オフィス・監査・ログまでセットで開設する仕組み」。後者は前者を基盤として使う。

- **A**：アカウントファクトリーとコントロールを提供する。
- **B**：OU・SCP・一括請求の土台だが環境一式の自動構築は別途必要。
- **C**：人のSSOを管理する。
- **D**：リソース共有を管理する。

ランディングゾーン、アカウントファクトリー、コントロールならControl Tower。

## 第8問（Management / Monitoring ｜ Service Catalog と CloudFormation）

> [!question] 問題（再掲）
> 開発者が、中央IT部門の承認済み構成だけを一覧から選んでセルフサービスで起動できるようにし、利用可能なインスタンスタイプなども制限したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Service Catalog
> - B. AWS CloudFormationのみ
> - C. AWS Systems Manager Automation
> - D. AWS Config

> [!success]- 答え：**A（AWS Service Catalog）**
> 承認済みCloudFormation製品をポートフォリオとして統制付きで配布する。

### 比較の核心

CloudFormationは「レシピを実行して料理を作る」、Service Catalogは「許可されたレシピだけを社内メニューに載せる」。作成エンジンと配布統制を分ける。

- **A**：カタログ、セルフサービス、制約に合う。
- **B**：テンプレートの実行はできるが製品ポートフォリオ管理がない。
- **C**：運用手順の自動実行用。
- **D**：構成の準拠評価用。

「承認済み製品」「ポートフォリオ」「起動制約」はService Catalog。

## 第9問（Cost Optimization ｜ Compute Optimizer と Cost Explorer）

> [!question] 問題（再掲）
> CPUやメモリなどの利用状況を分析し、特定のEC2インスタンスをどのインスタンスタイプへ変更すべきか具体的な推奨を得たいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Compute Optimizer
> - B. AWS Cost Explorer
> - C. AWS Budgets
> - D. AWS Cost Anomaly Detection

> [!success]- 答え：**A（AWS Compute Optimizer）**
> 構成と利用メトリクスから具体的なライトサイジング推奨を生成する。

### 比較の核心

Compute Optimizerは「使い方に合う機械サイズを勧める技術者」、Cost Explorerは「支出内訳を分析する会計担当」。何を知りたいかで選ぶ。

- **A**：変更先候補という要件に合う。
- **B**：費用の推移やサービス別内訳を可視化する。
- **C**：予算超過を知らせる。
- **D**：普段と異なる支出を知らせる。

具体的なリソースサイズはCompute Optimizer、金額分析はCost Explorer。

## 第10問（Networking / CDN / LB ｜ AWS RAM と PrivateLink）

> [!question] 問題（再掲）
> ネットワーク管理アカウントが所有するTransit Gatewayを、Organizations内の複数アカウントから同じリソースとして利用させたいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Resource Access Manager（AWS RAM）
> - B. AWS PrivateLink
> - C. VPCピアリング
> - D. AWS Transit Gatewayを各アカウントに個別作成する

> [!success]- 答え：**A（AWS RAM）**
> 中央所有の対応リソースを他アカウントへ共有する。

### 比較の核心

RAMは「同じ会議室を複数部署で共有」、PrivateLinkは「ある部署のサービス窓口を他部署へ公開」。リソースそのものを共同利用するか、サービスへ接続させるかが違う。

- **A**：Transit Gatewayのクロスアカウント共有に合う。
- **B**：特定サービスをインターフェイスエンドポイント経由で公開する。
- **C**：VPC間のIP接続でありリソース共有ではない。
- **D**：重複作成を避ける要件に反する。

共有対象がTransit Gateway、サブネット、Route 53 Resolver RulesなどならRAMを確認する。
