## 第1問（Management / Monitoring ｜ マルチリージョンの確実なフェイルオーバー（Application Recovery Controller））

> [!question] 問題（再掲）
> 動画配信企業が**2つのAWSリージョンにまたがるアクティブ/アクティブ構成**で稼働。いずれかで大規模障害が起きても利用者リクエストを**正常なリージョンへ切替**えたい。フェイルオーバー時、**切替先が確実にリクエストを処理できる状態**である必要がある。**運用負荷は抑えたい**。最適なのは？
> - A. Route 53のレイテンシーベースルーティング＋ヘルスチェックで、低レイテンシー振り分け＆障害リージョンの切り離し。
> - B. Route 53ヘルスチェックで監視＋障害時に切替。EventBridge Schedulerで定期テストして切替先が処理できる状態か確認。
> - C. AWS Fault Injection Serviceでリージョン障害をシミュレートし、検出時にRoute 53のDNSを更新して切替。
> - D. Amazon Application Recovery Controllerのマルチリージョンリカバリ機能を使用する。

> [!success]- 答え：**D**
> 「**切替先が“確実に処理できる状態”であることを保証**」＋「**低運用**」を両取りできるのは、その保証を専用機能（Readiness Check）で持つ **Application Recovery Controller（ARC）** だけ。

肝は要件の一文「**フェイルオーバー先が確実にリクエストを処理できる状態である必要がある**」。ただ切り替えるだけならRoute 53で足りますが、この問題は「**切替先がちゃんと受け止められるかを保証せよ**」と言っています。これを**マネージドに**やるのがARCで、あなたが選んだBは**それを自前で作ろうとして運用負荷が増える**から負けます。

### たとえ話：本社が止まったとき、支社に「今すぐ全業務を回せるか」を保証できるか
2拠点（リージョン）の両方で営業中（アクティブ/アクティブ）。本社が被災したら全部支社に回す——ここまでは電話転送（Route 53）でできます。問題は「**支社は本当に今、全社分の業務を捌けるのか？**」です。

- **ARC（D）＝“支社の受け入れ準備”を毎日点検してくれる本部＋安全なスイッチ**。
  - **Readiness Check（準備状況チェック）**＝支社の席数・人員・在庫（容量・構成・サービスクォータ）が本社と同等かを**継続的に点検**し、「今フェイルオーバーしても大丈夫」を保証する。
  - **ルーティングコントロール**＝点検済みの状態で、**安全に確実に切り替えられる“非常用スイッチ”**（DNSの不確実さに頼らずON/OFFできる）。
- **電話転送だけ（A・B）＝支社に転送はできるが、支社が受け止められるかは保証しない**。Bは「毎晩自分で支社にテスト電話する仕組みを手作り」＝**ARCのReadiness Checkを自前で再発明**しているだけで運用が重い。
- **避難訓練（C＝FIS）＝わざと障害を起こして耐性を試す訓練**。訓練は大事だが、**本番の切替を制御する装置ではない**。

### 各選択肢をズバッと
- **D：正解** … ARCのReadiness Checkで「切替先が処理できる状態」をマネージドに**保証**し、ルーティングコントロールでフェイルオーバーを安全実行。要件の核心（確実性＋低運用）に直球。
- **A：誤り** … レイテンシーベースルーティング＋ヘルスチェックは「平常時の最適振り分け」「障害リージョンの切り離し」まで。**切替先の処理可能性を保証する仕組みがない**。
- **B：誤り（あなたの回答）** … ヘルスチェックは「エンドポイントが応答するか」だけで、**容量やクォータは見ない**。EventBridge Schedulerで定期テストを自作＝**ARCのReadiness Checkを手作りで再現**しているだけ。運用負荷が高く「低運用」に反する。
- **C：誤り** … FISは**レジリエンスを“テスト”する**ためのカオスエンジニアリング用。**本番のフェイルオーバーを制御する道具ではない**。DNS自作更新も運用負荷大。

> [!note] 選択肢に出てきた用語ミニ解説（ご要望ぶん）
> - **Route 53 レイテンシーベースルーティング**：ユーザーを**一番低遅延のリージョン**へ振り分けるRoute 53のルーティングポリシー。速さ最適化が目的で、可用性保証の仕組みではない。
> - **Route 53 ヘルスチェック**：エンドポイントが**正常応答するか**を定期監視し、異常なら**フェイルオーバールーティング等で切り離す**。見るのは「生きているか」だけで、容量までは見ない。
> - **EventBridge Scheduler**：cron/レート式で**定期的に処理を起動**するスケジューラ。汎用の“タイマー”で、それ自体はDR保証機能ではない（自作テストの足回りに使われる）。
> - **AWS Fault Injection Service（FIS）**：意図的に障害（インスタンス停止・遅延・AZ/リージョン障害等）を注入して、システムの**回復力を検証するテストサービス**（カオスエンジニアリング）。本番の切替制御はしない。
> - **Amazon Application Recovery Controller（ARC）**：マルチAZ/マルチリージョンの**フェイルオーバーを確実に・マネージドに**行う。**Readiness Check**＝切替先が処理可能な状態かを継続検証／**ルーティングコントロール**＝DNSに依存せず安全に切替できるON/OFFスイッチ。

> [!tip] SAA着眼点
> - 「**マルチリージョンのフェイルオーバーで“切替先が確実に処理できる状態を保証”＋低運用**」→ **Application Recovery Controller（ARC）**。“Readiness（準備状況）”という語が出たらARC。
> - **Route 53のヘルスチェック/ルーティングは「振り分け・切り離し」までで、切替先の“受け入れ容量”は保証しない**。容量・構成・クォータの準備保証はARCの仕事。
> - **FIS＝障害をわざと起こして耐性をテスト**（本番制御ではない）。「シミュレート」「テスト」「カオス」の語が要件の主目的なら正解候補だが、“本番フェイルオーバーの制御”を問われたら不正解。
> - 「**自前でヘルスチェック＋定期テスト＋DNS更新を作り込む**」系は、マネージド代替（ARC）がある場面では**運用負荷が高く負ける**。

## 第2問（Security / IAM ｜ WAFの保護対象（NLBは非対応）とL7攻撃対策）

> [!question] 問題（再掲）
> ゲーム会社がリアルタイム多人数ゲームを運営。NLB配下のAuto ScalingのEC2で稼働、低レイテンシTCP/UDP。ユーザーはAPI Gateway経由でEC2にアクセス。XSSやSQLインジェクションなどの脆弱性攻撃からアプリを保護したい。最もセキュリティ効果が高い方法は？
> - A. AWS WAFを使用して、Amazon API Gatewayを保護する。
> - B. AWS Shield Advancedを使用して、NLBを保護する。
> - C. AWS Shield Advancedを使用して、Amazon API Gatewayを保護する。
> - D. AWS WAFを使用して、NLBを保護する。

> [!success]- 答え：**A**
> XSS/SQLインジェクションは**L7（Webアプリ層）の攻撃**なので対策は**WAF**（ShieldはDDoS対策で対象外）。さらに**WAFはL7リソースにしか関連付けできず**、ユーザーの入口は**API Gateway**なので、WAFをAPI Gatewayに適用するのが最適。

### 前提整理：ロードバランサーとAPI Gateway
- **ロードバランサー**＝大勢のアクセスを複数のEC2に振り分ける受付。**ALB＝L7（HTTPの中身が分かる・WAF付けられる）／ NLB＝L4（TCP/UDPで中身は見ない・超低遅延・WAF付けられない）**。ゲームの低レイテンシTCP/UDPはNLBが定番。
- **API Gateway**＝APIの正面玄関（受付カウンター）。クライアントはここにHTTPリクエストを出し、認証・流量制限をして裏（NLB→EC2）へ渡す。**L7のマネージド入口でWAFを付けられる**。
- 構成：**ユーザー → API Gateway（L7の受付）→ NLB（L4で振り分け）→ EC2（ゲーム本体）**。

### 2段階で決まる
1. **XSS・SQLi＝L7のWebアプリ攻撃 → WAF**（Shieldは**DDoS＝量の攻撃**で対象外）。これでB・Cが落ちる。
2. **WAFはL7の中身を読む道具 → L7が分かる入口にしか付けられない**（CloudFront/ALB/API Gateway/AppSync/Cognito等）。**NLBはL4なのでWAF不可** → Dは成立しない。入口のAPI Gatewayに付けるAが正解。

### たとえ話：郵便の仕分け（L4）と、中身を読む窓口（L7）
- **NLB（L4）＝郵便の仕分け係**。宛先の住所（IP）と部屋番号（ポート）だけ見て高速に流す。**手紙の中身は読まない**ので、「これはXSS攻撃だ」と気づけない＝WAFの出番がない。
- **API Gateway（L7）＝手紙を開けて読む受付**。中身が読めるから、そこに**WAF（内容を検査する警備員）**を立てて、XSS/SQLiの手紙を門前払いできる。

### 各選択肢をズバッと
- **A：正解** … L7攻撃(XSS/SQLi)はWAF、WAFを付けられて実際の入口でもあるAPI Gatewayに適用。
- **B：誤り** … Shieldは**DDoS対策**でXSS/SQLi非対応。対象もNLB。
- **C：誤り** … Shieldは**DDoS対策**。XSS/SQLiには効かない（対象がAPI GatewayでもNG）。
- **D：誤り** … **WAFはNLB（L4）に関連付けできない**。設定自体が成立しない（頻出ひっかけ）。

### SAA着眼点
- **XSS・SQLインジェクション（Webアプリ層の攻撃）→ WAF**。**DDoS（量）→ Shield**。まずこの切り分け。
- **WAFを付けられる対象＝CloudFront / ALB / API Gateway / AppSync / Cognito / App Runner**。**NLB・EC2直・RDSには付けられない**（NLBはL4なので特に頻出ひっかけ）。
- **ALB＝L7（中身が分かる・WAF可）／ NLB＝L4（TCP/UDP・超低遅延・WAF不可）**。ゲームや低レイテンシTCP/UDPはNLB。
- 保護は**ユーザーの入口（今回はAPI Gateway）**に置くのが基本。

## 第3問（Networking / CDN / LB ｜ Global AcceleratorとNLB（UDP・静的IP・エッジ低遅延））

> [!question] 問題（再掲）
> 株価をリアルタイム配信するアプリ。OSカーネルをカスタマイズしたLinux上で動作、UDP通信。世界中のユーザーに提供し、最寄りのAWSエッジロケーションにルーティングして低遅延にしたい。さらに固定IPアドレスでアクセスできる必要がある。要件に合う構成は？
> - A. Global Accelerator → NLB → Auto ScalingのEC2。
> - B. Global Accelerator → ALB → Auto ScalingのEC2。
> - C. CloudFront → ALB → Auto ScalingのEC2。
> - D. CloudFront → NLB → Auto ScalingのLambda。

> [!success]- 答え：**A**
> UDPなのでLBは**NLB**（L4・TCP/UDP対応。ALBはHTTP専用でUDP不可）。世界中で最寄りエッジから低遅延＋固定IPが要るのでフロントは**AWS Global Accelerator**（TCP/UDP対応・**静的Anycast IPを2つ**提供）。CloudFrontはHTTPコンテンツのキャッシュ配信用で、UDPも固定IPも要件に合わない。

### 3つのキーワードで削る
1. **UDP → NLB**（ALB＝HTTP専用でUDP不可、CloudFront＝HTTP専用）。これでB（ALB）・C（ALB）・D（CloudFront）が落ちる方向。
2. **最寄りエッジへ低遅延＋世界中 → Global Accelerator or CloudFront**。だが…
3. **固定IP＋UDP → Global Accelerator**（静的Anycast IP・TCP/UDP対応）。CloudFrontはHTTPキャッシュ用で不適。

→ 残るA（Global Accelerator＋NLB）。

### 核心：CloudFront と Global Accelerator の違い（超頻出）
| | CloudFront | Global Accelerator |
|---|---|---|
| 主目的 | **コンテンツのキャッシュ配信** | **TCP/UDPアプリの高速化・可用性** |
| プロトコル | HTTP/HTTPS | **TCP/UDP（非HTTPもOK）** |
| キャッシュ | する | しない |
| 固定IP | ✗ | **✓ 静的Anycast IP 2つ** |
| 向く用途 | 画像・動画・静的サイト配信 | **ゲーム・IoT・UDP・固定IP・高速フェイルオーバー** |

### たとえ話
- **CloudFront** ＝ 各地に**コピー品を置く売店**。人気商品（コンテンツ）を近くの売店にコピーして素早く渡す。ただし**Web商品（HTTP）専用**。
- **Global Accelerator** ＝ 世界中に**専用高速レーンの入口（固定の番地＝静的IP）**を用意し、**AWSの高速道路**で本社（NLB→EC2）へ最短で運ぶ。**荷物の種類（UDPでも）を問わない**。

### 各選択肢をズバッと
- **A：正解** … Global Accelerator（エッジ＋固定IP）→**NLB（UDP対応）**→EC2。UDP・固定IP・低遅延を全部満たす。
- **B：誤り** … Global Acceleratorは正しいが、**ALBはUDP非対応**。UDP要件で落ちる。
- **C：誤り** … CloudFront（HTTP用でUDP不可・固定IPなし）＋ALB（UDP不可）。二重に不適。
- **D：誤り** … CloudFront（UDP不可）。バックエンドがLambdaでリアルタイムUDP配信にも合わない。

### SAA着眼点
- **UDP／非HTTPのL4通信 → NLB**（ALB＝HTTP専用、CloudFront＝HTTP専用）。「UDP」が出たら**ALB・CloudFrontを消す**。
- **静的IP（固定IP）＋世界中で低遅延＋TCP/UDP → AWS Global Accelerator**（2つの静的Anycast IP）。
- **CloudFront＝コンテンツのキャッシュ配信（HTTP）／ Global Accelerator＝TCP/UDPアプリの高速化・可用性・固定IP**。使い分けが頻出。
- ゲーム・金融リアルタイム・IoTなど「**低遅延＋UDP＋固定IP**」は Global Accelerator＋NLB が鉄板。

## 第4問（Storage ｜ HPC向け高性能ファイル（FSx for Lustre×S3統合））

> [!question] 問題（再掲）
> 気候研究機関が大量の気象データをS3に保存。シミュレーションのためHPC環境を導入。HPCからS3上のデータへ低レイテンシー・高スループットでアクセスしたい。最も効果的な選択肢は？
> - A. FSx for Lustreを設定し、S3データを手動でインポート。HPCからFSにアクセス。
> - B. S3 Transfer Accelerationでアクセス速度を向上させ、HPCからのアクセスを高速化。
> - C. FSx for Lustreを設定し、既存S3バケットと統合してHPCから直接FSにアクセス。
> - D. CloudFrontでS3データへのアクセスをキャッシュし、HPCからのアクセスを高速化。

> [!success]- 答え：**C**
> HPCの低レイテンシ・高スループットなファイルアクセスは、**HPC/ML向けに設計された高性能並列ファイルシステム Amazon FSx for Lustre** が最適。FSx for Lustreは**S3とネイティブ統合**でき、既存バケットにリンクすればS3オブジェクトをファイルとして直接・高速に扱える（手動コピー不要）。

AとCはどちらもFSx for Lustreで方向は正しい。差は**「S3を手動インポート（A）」か「S3とネイティブ統合して直接アクセス（C）」か**だけ。手間のないCが「最も効果的」。

### たとえ話：巨大書庫（S3）と、直結した高速閲覧席（FSx for Lustre）
- **S3** ＝ 蔵書が全部ある**巨大書庫**。保管は最強だが、1冊ずつ取りに行くと遅い。
- **FSx for Lustre** ＝ 書庫と**直結した超高速の閲覧席**。研究者（HPC）が座ると**必要な本が書庫から自動で手元に届く**（S3統合＝直接アクセス）。
- **A（手動インポート）** ＝ 使う前に**自分で本を全部運んでくる**方式。動くが手間。Cの自動連携がスマート。

### 各選択肢をズバッと
- **A：次点** … FSx for Lustreは正しいが、**S3を手動インポート**する運用が余計。ネイティブ統合のCに劣る。
- **B：誤り** … **S3 Transfer Accelerationは“遠距離のS3アップロード/ダウンロードを高速化”**する機能。HPCの低遅延ファイルアクセスとは別物。
- **C：正解** … FSx for LustreをS3にネイティブ統合し、HPCから直接高速アクセス。低遅延・高スループット・手間なし。
- **D：誤り** … **CloudFrontはHTTPコンテンツのキャッシュ配信**。HPCのファイルシステムアクセス高速化には使えない。

### SAA着眼点
- **「HPC／機械学習／大規模解析＋低レイテンシ・高スループットなファイルアクセス」→ Amazon FSx for Lustre**。
- **FSx for Lustre は S3 とネイティブ統合**（バケットにリンク→オブジェクトをファイルとして遅延ロード）。「**S3データをHPC/MLで高速に使う**」の定番解。
- ひっかけ：**S3 Transfer Acceleration＝遠距離のS3転送高速化／ CloudFront＝HTTPキャッシュ配信**。どちらも**HPCのファイルアクセス性能**の話ではない。
- **Windows共有→FSx for Windows File Server／ Linux・HPC・MLの高性能並列FS→FSx for Lustre**、と用途で選ぶ。

## 第5問（Analytics ｜ ネスト文書のDynamoDB保存とMacieによるPII定期スキャン）

> [!question] 問題（再掲）
> カスタマーサポートの**問い合わせチケット管理**システム。各チケットは**スレッド一覧・添付メタデータ・顧客プロフィール**を**ネスト文書構造**で保存、**大規模でも低レイテンシ応答**が必要。さらに顧客が誤って入力する**PII（住所・電話番号等）を毎月全データスキャン**し、検出時は**コンプラ部門へメール通知**したい。取るべき組み合わせは？（2つ選択）
> - A. Macie有効化、月次の分類ジョブ**完了イベントをEventBridge→Lambda**に送信。
> - B. **S3にJSONで保存しAthenaでクエリ**。
> - C. Macie有効化、**月次スケジュールの分類ジョブを設定してスキャン**。完了イベントをEventBridge→**SNSトピック**に転送。
> - D. **DynamoDBにネスト構造で保存**し、**エクスポート機能で毎月S3に出力**。
> - E. **DynamoDB Streams→Lambda→Comprehend**でPII検出しSNS通知。

> [!success]- 答え：**C ＋ D**（あなたの選択 A・B はどちらも誤り）
> ①ネスト文書＋低レイテンシ＋大規模＝**DynamoDB**（D）、②毎月全データPIIスキャン＋メール通知＝**Macieの月次ジョブ＋SNS**（C）。この2つが噛み合う。

肝は要件が**2つ**あること。「**保存方式（①）**」と「**PII検出＋通知（②）**」を別々に満たす選択肢を1つずつ選ぶ。ポイントは、**MacieはS3しかスキャンできない**——だから「DynamoDBに置くだけ」では検出できず、**毎月S3へエクスポート**して橋渡しする必要がある。C（Macie）とD（DynamoDB＋S3エクスポート）がセットで初めて成立する。

### たとえ話：受付台帳（DynamoDB）と、月次の個人情報点検（Macie）
- **D＝受付台帳を“折りたたみ式のカルテ”（ネスト文書）でデジタル管理**。問い合わせが殺到しても瞬時に開ける（低レイテンシ・大規模）。ただし点検業者はこの台帳を直接読めないので、**毎月コピーを資料室（S3）に出しておく**。
- **C＝月に一度、個人情報の専門点検業者（Macie）が資料室（S3）を巡回**。住所や電話番号（PII）を見つけたら、**受付の内線（EventBridge）から警備室（SNS）へ連絡→担当にメール**。
- **なぜ両方要るか**：点検業者（Macie）は**資料室（S3）しか見ない**。台帳（DynamoDB）に置きっぱなしでは点検できないので、Dの「毎月S3へ出す」が点検（C）の前提になる。

### 各選択肢をズバッと
- **C：正解** … Macieの**月次分類ジョブ**でS3をスキャン（＝「毎月全データスキャン」に一致）、完了イベントを**EventBridge→SNS**でコンプラ部門へ**メール通知**。要件②に直球。
- **D：正解** … DynamoDBは**ネスト（JSON）文書を低レイテンシ・大規模**に扱える（要件①）。**S3エクスポート**でMacieのスキャン対象を用意する橋渡し役も兼ねる。
- **A：誤り** … 分類ジョブの**スケジュール（月次スキャンの実行）を設定していない**うえ、通知先が**Lambda**（メール送信は追加実装が要る）。Cの劣化版。
- **B：誤り** … **S3＋JSON＋Athena**は「低レイテンシなチケット応答（ネスト文書の高速読み書き）」に不向き。Athenaは分析クエリ用で要件①を満たさない。
- **E：誤り** … DynamoDB Streams＋Comprehendは**書き込みごとのリアルタイム検出**で、「**毎月全データを一括スキャン**」という要件と方式が違う。自前実装で運用負荷も高い（MacieというマネージドPII検出があるのに再発明）。

> [!tip] SAA着眼点
> - **PII/機密データの検出・分類＝Macie**（対象は**S3のみ**）。「個人情報」「PII」「機密データをスキャン」と来たらMacie。[[1603_脅威検知・監査]]
> - **Macieの通知**：分類ジョブ→**EventBridgeイベント→SNS**でメール通知が定番。「メールで通知」＝SNS（Lambda経由は不要な自作）。
> - **ネスト（JSON）文書・低レイテンシ・大規模＝DynamoDB**。「スキーマ柔軟／ネスト構造／ミリ秒応答」はDynamoDBの合図。
> - **DynamoDBのデータをMacie/Athena/分析に回す**には、まず**S3へエクスポート**（DynamoDBは直接Macieでスキャンできない）。この“橋渡し”が2択問題の肝。
> - **Comprehend**もPII検出はできるが、**ストリーム/レート単位のリアルタイム処理**向け。「**毎月一括スキャン**」の要件ではMacieの分類ジョブが適合。

## 第6問（Migration / Transfer ｜ Application Migration Service（MGN）によるVM移行（3つ選択））

> [!question] 問題（再掲）
> オンプレVMで販売管理システムを運用。コスト最適化でAWS移行を検討し、まず影響の少ないバッチ系から移行開始。コードやシステム構成の変更は最小限、短期間で、ダウンタイムも最小にしたい。適切な移行手順の組み合わせは？（3つ選択）
> - A. VMの初期レプリケーション完了後、テストインスタンスを起動して動作確認する。
> - B. DataSyncでVMのディスクイメージをS3にコピーし、AMIを作成してEC2を起動する。
> - C. Systems Manager Patch Managerを有効化し、移行対象VMのデータを収集する。
> - D. Application Migration Serviceを使うために、移行対象VMにAWS Replication Agentをインストールする。
> - E. Elastic Disaster Recoveryを使うために、移行対象VMにAWS Replication Agentをインストールする。
> - F. カットオーバー時にオンプレVMへの書き込みを停止し、カットオーバーインスタンスを起動して最終データ同期を実施する。

> [!success]- 答え：**A ＋ D ＋ F**
> 「**変更最小・短期間・低ダウンタイムでVMをリフト＆シフト**」は **AWS Application Migration Service（MGN）** の定番。標準手順は **①D：Replication Agent導入（継続的ブロック複製を開始）→ ②A：テストインスタンスで検証 → ③F：カットオーバー（書き込み停止→最終同期→切替）**。継続複製＋一瞬の切替でダウンタイムを最小化する。

「エージェント導入 → テスト → カットオーバー」の3点セット。これがMGNの型。あなたが選んだ **B（DataSync）** は、正しくは **F（カットオーバー）** の位置。

### たとえ話：引っ越しを「生活しながら」進める
- **D（エージェント導入）** ＝ 新居に**自動荷造りロボット**を設置。旧居の荷物を**裏で少しずつ新居に複製**し続ける（継続レプリケーション）。
- **A（テストインスタンス）** ＝ 引っ越し前に新居で**試泊**して電気・水道が動くか確認。
- **F（カットオーバー）** ＝ 引っ越し当日、**旧居の使用を止めて最後の差分だけ運び、本移動**。旧居生活を止める時間が最短。
- **B（DataSyncでコピー）** ＝ 全荷物を一度倉庫に運んでから積み替える方式。手間も休業時間も大きい。

### 各選択肢をズバッと
- **A：正解** … MGNのテスト工程。本番切替前にテストインスタンスで動作確認。
- **B：誤り** … DataSyncは**ファイル/オブジェクトのデータ転送**でVM全体の移行手段ではない。手作業が多く継続複製もなくダウンタイムも大きい。
- **C：誤り** … Patch Managerは**パッチ適用**の道具で移行の発見・収集用ではない（発見はApplication Discovery Service）。
- **D：正解** … MGNの起点。VMにAWS Replication Agentを入れて継続的レプリケーションを開始。
- **E：誤り** … Elastic Disaster Recovery（DRS）は**災害復旧**サービス。エージェントは似るが、今回は「移行」なので使うのは**MGN**。用途違いの罠。
- **F：正解** … MGNのカットオーバー工程。書き込み停止→最終同期→カットオーバーインスタンス起動で本番切替。

### SAA着眼点
- **「オンプレVMを変更最小・短期間・低ダウンタイムでAWSへリフト＆シフト」→ AWS Application Migration Service（MGN）**。手順は**①Replication Agent導入 → ②テストインスタンス検証 → ③カットオーバー**。
- **MGN（移行）と Elastic Disaster Recovery＝DRS（災害復旧）は別サービス**。どちらもReplication Agentを使うが、**“移行”ならMGN**。用途語で選ぶ。
- **DataSync＝ファイル/オブジェクトのデータ転送**（VM丸ごと移行ではない）。**Application Discovery Service＝移行前の発見/インベントリ**（Patch Managerではない）。
- 低ダウンタイム移行のキモは**継続的レプリケーション＋一瞬のカットオーバー**。

## 第7問（Application Integration ｜ API Gatewayのキャッシュで重複参照を削減）

> [!question] 問題（再掲）
> 金融企業のETLジョブが、顧客プロファイルや取引条件などの**参照データ**を**オンプレの参照データサービス**から取得。市場が活発な時間帯に**同じ参照データへの多数リクエストが集中**し、データサービスが**過負荷**。ETLの効率を保ちつつ**参照データサービスの負荷を軽減**、**管理コスト最小**にしたい。最適なのは？
> - A. API Gateway REST API＋**使用量プランでスロットル/クォータ**。VPCリンクでオンプレ接続して取得。
> - B. API Gateway **WebSocket API**で持続的接続を維持し、接続確立オーバーヘッドを削減。
> - C. API Gateway **HTTP API＋Lambda**で取得。自動スケールアウトで負荷分散。
> - D. API Gateway REST API＋**顧客IDや商品コードをキャッシュキー**にして**キャッシュ機能を有効化**。

> [!success]- 答え：**D**
> 問題の本質は「**同じ参照データへの重複リクエスト**がバックエンドに殺到」。API Gatewayの**キャッシュ**なら、2回目以降は手前で即応答し、**オンプレまでリクエストが届かない**＝重複取得そのものが消える。

肝は「何が過負荷の原因か」。原因は**同じデータの取り直し**です。だから効く対策は「**繰り返しを手前で肩代わりする＝キャッシュ**」の一点。あなたが選んだAの**スロットル（流量制限）は“蛇口を絞る”だけ**で、絞られたETLの効率が落ちるうえ、同じデータの重複取得は減らないので根本解決になりません。

### たとえ話：人気商品の問い合わせを、受付が代わりに答える
オンプレの参照データサービス＝**奥にいる専門スタッフ1人**。ETL＝ひっきりなしに来る客。市場が活発だと**「あの商品の情報教えて」と同じ質問が殺到**し、奥のスタッフがパンクします。

- **D（キャッシュ）＝受付（API Gateway）が“よくある質問の答え”をメモして貼っておく**。同じ質問（同じ顧客ID・商品コード＝キャッシュキー）が来たら**受付がその場で即答**し、奥のスタッフに取り次がない。**質問の重複が奥に届かない**ので過負荷が解消。
- **A（スロットル）＝受付が「1分◯人まで」と入場制限する**。奥は守れるが、**客（ETL）が待たされ効率が落ちる**。しかも同じ質問の重複は減っていない。
- **B（WebSocket）＝受付とスタッフを内線でつなぎっぱなしにする**。接続の手間は減るが、**質問の数（重複）は減らない**。そもそも用途が双方向リアルタイム向け。
- **C（HTTP API＋Lambda＋自動スケール）＝受付を増員して客をどんどん奥へ流す**。流せば流すほど**奥のスタッフ（オンプレ）への問い合わせが増え**、過負荷が悪化。

### 各選択肢をズバッと
- **D：正解** … キャッシュキー（顧客ID/商品コード）ごとにレスポンスをAPI Gatewayが保持し、TTL内の同一キーは**バックエンド未到達で即応答**。重複を根本から削減、マネージドで低管理コスト。ETL側も速くなり効率維持。
- **A：誤り（あなたの回答）** … 使用量プランのスロットル/クォータは**リクエスト数の上限制限**。バックエンドは守れても**ETLの効率が落ち**、同一データの重複取得は残る。過負荷の原因（重複）に手を打っていない。
- **B：誤り** … WebSocketは**双方向リアルタイム通信**用。参照データ取得はリクエスト/レスポンス型で、持続接続にしても**リクエスト数＝重複は減らない**。用途違い。
- **C：誤り** … 自動スケールは**さばく能力を増やす**だけ。スケールするほど**オンプレへのアクセスが増えて過負荷を悪化**させる。原因と逆方向。

> [!tip] SAA着眼点
> - 「**同じデータへの繰り返しリクエストで backend が過負荷**」→ **キャッシュ**で重複を手前で吸収。API Gatewayなら**REST APIのキャッシュ**（キャッシュキーで区別）。より汎用のデータキャッシュは **ElastiCache**、CDN層なら **CloudFront**。
> - **API Gatewayの“キャッシュ”は REST API の機能**（HTTP API/WebSocketには無い）。「キャッシュしたい」＝REST APIを選ぶ手がかり。
> - **使用量プラン（スロットル/クォータ）＝流量を絞る/課金管理**。「バックエンド保護」に見えるが**重複取得は減らせず、正当なリクエストまで絞る**ので「効率を保ちつつ負荷軽減」には不適。頻出の引っかけ。
> - **自動スケール（Lambda/HTTP API）はバックエンドの負荷を減らさない**（むしろ増やす）。ボトルネックがバックエンド側なら、スケールではなくキャッシュ/キューで“到達数”を減らす発想。
> - API Gateway種別：**REST（多機能・キャッシュ/使用量プラン）／ HTTP（安い・速い・シンプル）／ WebSocket（双方向リアルタイム）**。

## 第8問（Networking / CDN / LB ｜ オンプレからS3への閉域アクセス（Direct Connect＋インターフェイスエンドポイント））

> [!question] 問題（再掲）
> S3のデータに、自社AWSリージョン内のシステムとオンプレミス環境の双方から安全にアクセスしたい。企業ポリシーで、すべての通信はインターネットを経由せず閉域ネットワーク経由が必須。最適なソリューションは？
> - A. Site-to-Site VPNのIPsec暗号化通信を利用し、仮想プライベートゲートウェイ経由でS3にアクセス。
> - B. Transit GatewayとDirect Connectゲートウェイを設定し、各VPCでゲートウェイ型エンドポイント経由でS3にアクセス。
> - C. Direct Connect接続を確立し、S3にゲートウェイ型エンドポイントを作成してオンプレからのアクセスを許可。
> - D. Direct Connect接続を確立し、S3にインターフェイス型エンドポイントを構成し、プライベートIPによるアクセスを実現。

> [!success]- 答え：**D**
> **閉域接続＝Direct Connect**（Site-to-Site VPNはインターネット経由なので不可）。**オンプレからS3をプライベートに叩く＝インターフェイス型エンドポイント（PrivateLink）**（VPC内にプライベートIP＝ENIを持つのでDX越しに到達可）。ゲートウェイ型はVPC内からのみ有効でオンプレからは到達できない。

この問題の分かれ目は2つ。**①閉域→VPNかDXか**、**②S3エンドポイントはGateway型かInterface型か**。特に②が最頻出。

### 核心：Gateway型 vs Interface型（オンプレから使えるかが違う）
| エンドポイント | 実体 | オンプレ（DX/VPN経由）から使える？ |
|---|---|---|
| **Gateway型**（S3/DynamoDB） | **ルートテーブルのroute**（VPC内でだけ有効） | ❌ **使えない**（VPC内専用） |
| **Interface型**（PrivateLink） | **VPC内にプライベートIPを持つENI** | ✅ **使える**（DX/VPN経由で到達可能） |

- **Gateway型は“VPCの中からS3へ”専用**。オンプレからDX越しには到達できない → B・Cが落ちる。
- **Interface型はVPC内にプライベートIPを持つ**ので、オンプレからDX越しにそのIP宛でS3にアクセスできる → D。

### たとえ話：本社（オンプレ）から倉庫（S3）への専用ルート
- **Direct Connect** ＝ 本社と倉庫を結ぶ**専用の私道**（公道＝インターネットを通らない）。
- **Gateway型エンドポイント** ＝ 倉庫への**社内限定の近道**。**AWSの敷地（VPC）の中からしか通れない**。本社（オンプレ）からは入れない。
- **Interface型エンドポイント** ＝ 倉庫の**私設の受付窓口（内線番号＝プライベートIP付き）**。本社からも私道（DX）経由でこの内線に繋げる。

### 各選択肢をズバッと
- **A：誤り** … Site-to-Site VPNは**インターネット経由**。「インターネット非経由」ポリシーに反する。
- **B：誤り** … 閉域接続は良いが、S3を**Gateway型エンドポイント**で受けており、**オンプレからGateway型には到達できない**。
- **C：誤り** … 同じく**Gateway型はVPC内専用**で、オンプレからのアクセスを許可できない。
- **D：正解** … Direct Connect（閉域）＋**Interface型エンドポイント（プライベートIP）**で、VPCからもオンプレからもインターネットを通らずS3へ。

### SAA着眼点
- **「オンプレミスからS3/DynamoDBへプライベートにアクセス」→ インターフェイス型エンドポイント（PrivateLink）**。**Gateway型はVPC内からのみ**でオンプレからは使えない、が最頻出の分かれ目。
- **閉域（インターネット非経由）でオンプレ↔AWS → Direct Connect**。**Site-to-Site VPNはインターネット経由**（暗号化しても通り道はネット）なので「非経由」要件では選ばない。
- Gateway型＝**無料・ルートテーブル方式・S3/DynamoDB専用・VPC内限定**／ Interface型＝**ENIにプライベートIP・多数サービス対応・オンプレからも到達可（有料）**。
- 「VPC内からS3を無料でプライベートアクセス」だけならGateway型でOK。**オンプレが絡んだらInterface型**。

## 第9問（Security / IAM ｜ マルチアカウント統制とAD連携シングルサインオン）

> [!question] 問題（再掲）
> 多国籍企業が、**事業部ごとに個別のAWSアカウント**を使うマルチアカウント構成へ移行。ユーザー認証は**本社のオンプレMicrosoft Active Directory**を用いて**全AWSアカウントを一元管理**したい。有効なアクションの組み合わせは？（2つ選択）
> - A. **IAM Identity Center**を設定し、**AWS Directory Service**を介して本社のADと連携する。
> - B. 各アカウントに**個別のIAMユーザー**を作成し、ADから一時認証情報を配布する。
> - C. **AWS Organizations**で全機能を有効にした組織を作成し、事業部用アカウントを追加する。
> - D. 本社ADに**Cognito**を接続し、Cognitoで全AWSアカウントのユーザー認証を行う。
> - E. Organizationsで組織を作成し、**認証メカニズムにDirectory Serviceを直接使う**よう組織を設定する。

> [!success]- 答え：**A ＋ C**（あなたの選択は A・E で、Eが誤り）
> 要件は2つ。①**マルチアカウントを束ねる土台**＝Organizations（C）、②**オンプレADで全アカウントの認証を一元化（SSO）**＝IAM Identity Center＋Directory Service（A）。器（C）と鍵の管理（A）がセットで成立する。

肝は「**アカウントの“入れ物”**」と「**ログインの“一元管理”**」の2役を別々に選ぶこと。Cが入れ物（Organizations）、Aが鍵（AD連携SSO）。あなたが選んだEは**“それっぽい造語”の罠**で、Organizationsに「認証メカニズムを設定する」機能は存在しません。

### たとえ話：グループ企業の「本社ビル群」と「共通の社員証システム」
- **C（Organizations 全機能有効）＝グループ各社のビル（AWSアカウント）を1つの本社管理下に束ねる**。事業部ごとにビルを増やせて、経費もルールも一元管理。マルチアカウントの“入れ物”。
- **A（IAM Identity Center＋Directory Service）＝どのビルにも“本社の社員証（オンプレAD）”でそのまま入れる共通ゲート**。IAM Identity CenterがSSOの受付、Directory Serviceが本社ADとの連携パイプ。社員は1枚の社員証（既存AD ID）で全ビルにSSOできる。
- この2つが揃って初めて「マルチアカウント＋AD一元認証」が完成する。

### 各選択肢をズバッと
- **A：正解** … IAM Identity Center（旧AWS SSO）はOrganizations配下**全アカウントへのSSO**を提供。**Directory Service**でオンプレADと連携すれば、**既存のADユーザーのまま**各アカウントにログインできる。
- **C：正解** … Organizationsの**全機能を有効化**した組織を作り、事業部ごとにアカウントを追加＝マルチアカウント統制の土台（SCP等のガードレールもここ）。
- **B：誤り** … アカウントごとに**IAMユーザーを個別作成**は「一元管理」と真逆で管理が爆発。AD一元認証にもならない（IAMユーザーは長期認証情報でSSOでもない）。
- **D：誤り** … **Cognitoはアプリ（Web/モバイル）のエンドユーザー認証**用。**従業員のAWSアカウント/コンソールへのSSO**の道具ではない。役割違いの定番ひっかけ。
- **E：誤り（あなたの選択）** … Organizationsに「**認証メカニズムにDirectory Serviceを直接使う**」という設定は**存在しない**。AD連携SSOを担うのはIAM Identity Center。もっともらしい造語の罠。

> [!tip] SAA着眼点
> - 「**複数アカウント（Organizations）＋オンプレAD/外部IdPで全アカウントにSSO**」→ **IAM Identity Center**（旧AWS SSO）が定番解。ADとの橋渡しは**Directory Service（AD Connector / AWS Managed Microsoft AD）**。
> - **マルチアカウントの器＝Organizations（全機能有効でSCP等が使える）**。「事業部/部門ごとにアカウント分割・一元統制」はOrganizations。
> - **Cognito＝アプリのエンドユーザー認証（BtoC/サインアップ）**。**従業員のAWSログイン（SSO）はIAM Identity Center**。この2つの取り違えが頻出。
> - **IAMユーザーを各アカウントに量産**は「一元管理」要件では常に不正解寄り（管理煩雑・長期認証情報）。一元認証はロール＋Identity Centerで。

## 第10問（Security / IAM ｜ セキュリティグループのSG参照（ALB経由のみ許可））

> [!question] 問題（再掲）
> ALB配下のEC2でWebコンテンツを公開。インターネットからEC2への直接アクセスを防ぎ、ALB経由のアクセスのみ許可したい。どんなネットワークアクセス制御をすべき？
> - A. EC2のSGで、ALBのプライベートIPアドレスからの通信を許可する。
> - B. EC2のSGで、インターネットからの通信を拒否する。
> - C. ALBのSGで、EC2のSGへの通信を許可する。
> - D. EC2のSGで、ALBのSGからの通信を許可する。

> [!success]- 答え：**D**
> **セキュリティグループは許可（allow）ルールのみで拒否ルールが書けない**。EC2側SGのインバウンドで**送信元にALBのSGを指定して許可**すれば、ALB経由の通信だけ通り、インターネットからの直アクセスは**自動的に遮断**される（SG参照/チェイン）。ALBのIP変動にも追従できるベストプラクティス。

### 核心①：SGには「拒否ルール」が書けない
> [!warning] SGは「許可のみ」（ホワイトリスト）
> セキュリティグループは**許可ルールしか書けない**。「〜を拒否する」は存在せず、**許可した通信以外は自動で全遮断**。だからB「インターネットからの通信を拒否する」は**SGでは成立しない**。拒否を書けるのは**NACL**。

拒否を書く必要すらない——「ALBだけ許可」すれば残り（インターネット直アクセス）は自動で落ちる。

### 核心②：ベストプラクティスは「SG参照（チェイン）」
EC2のSGのインバウンドで**送信元＝ALBのセキュリティグループ**を許可する。ALBを通った通信だけEC2に入れ、**ALBのIPが増減・変動しても追従**する（IP直指定より堅牢）。

### たとえ話：会員制ラウンジの入口
- **EC2＝ラウンジ**。入口（EC2のSG）で**「ALBというコンシェルジュを通した客だけ入れる」**と決める＝D。
- **B（お断りの札）** … SGという入口には**“お断り札（拒否ルール）”を貼る機能がない**。「コンシェルジュ経由だけ通す」と書けば他は自動で入れない。
- **A（コンシェルジュの今日の車のナンバーを控える）** … 毎日違う車（IP変動）で来るので破綻しやすい。**コンシェルジュ本人（SG）**を指定すべき。

### 各選択肢をズバッと
- **A：非推奨** … ALBのプライベートIP許可は**IP変動で壊れる**。SG参照が正道。
- **B：誤り** … **SGに拒否ルールは書けない**（許可のみ）。設定として成立しない。
- **C：誤り** … 制御する場所が逆（ALB側SGではなく**EC2側SGのインバウンド**で決める）。
- **D：正解** … EC2のSGで**送信元にALBのSGを指定して許可**。ALB経由のみ通り直アクセスは自動遮断、IP変動にも強い。

### SAA着眼点
- **セキュリティグループは「許可のみ（ホワイトリスト）・ステートフル」。拒否ルールは書けない**（拒否は**NACL**）。「〜を拒否するSG」という選択肢はほぼ罠。
- **「ALB配下のEC2をALB経由のみに絞る」→ EC2のSGのインバウンドで“送信元＝ALBのSG”を許可（SG参照/チェイン）**。IP直指定より堅牢で定番。
- 「許可した以外は自動で遮断」なので、**わざわざ拒否を書く必要はない**。必要なものだけ許可する発想。
- SG＝**インスタンス単位・ステートフル・許可のみ**／ NACL＝**サブネット単位・ステートレス・許可と拒否**。
