## 第1問（Compute ｜ Lambda と Fargate）

> [!question] 問題（再掲）
> 既存のDockerコンテナで動く動画変換処理があります。1回の処理は45分かかり、OSライブラリとCPU・メモリの割り当てを細かく制御する必要があります。サーバーは管理したくありません。
> 最も適した実行基盤はどれですか。
>
> - A. Amazon ECS on AWS Fargate
> - B. AWS Lambda
> - C. Amazon EC2の常時稼働インスタンス1台
> - D. Amazon API Gateway

> [!success]- 答え：**A（ECS on Fargate）**
> サーバーを管理せず、長時間の任意コンテナを指定したCPU・メモリで実行する。

### 比較の核心

Lambdaは「短い用事ごとに呼ぶ作業員」、Fargateは「必要な間だけ専用作業台でコンテナを動かす」。実行時間上限と実行環境の制御が決め手。

- **A**：45分、Docker、リソース指定を満たす。
- **B**：短時間イベント処理向けで、単一実行の時間上限に抵触する。
- **C**：柔軟だがOSと容量を管理する。
- **D**：API入口であり計算処理を実行しない。

イベント駆動で短時間ならLambda、長時間コンテナやOS依存が強いならFargate。

## 第2問（Compute ｜ AWS Batch と Step Functions）

> [!question] 問題（再掲）
> 数万件の独立した科学計算ジョブをキューに入れ、優先度に応じてEC2 Spotを含む計算環境へスケジュールし、必要な計算容量を自動で増減させたいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Batch
> - B. AWS Step Functions
> - C. Amazon EventBridge Scheduler
> - D. AWS Lambda

> [!success]- 答え：**A（AWS Batch）**
> ジョブキュー、スケジューラー、EC2/Fargate計算環境をまとめて管理する。

### 比較の核心

Batchは「大量作業を計算機へ割り振る工場長」、Step Functionsは「複数工程の順番と失敗時の動きを決める工程表」。計算資源を用意するか、処理の流れを管理するかが違う。

- **A**：数万ジョブとSpot計算環境に合う。
- **B**：Batchジョブを含むワークフローの前後関係には使える。
- **C**：時刻で起動するが、計算環境を管理しない。
- **D**：短い個別処理向け。

大量の計算ジョブはBatch、サービスをまたぐ分岐・再試行はStep Functions。

## 第3問（Application Integration ｜ EventBridge Pipes と Step Functions）

> [!question] 問題（再掲）
> SQSキューのメッセージをフィルタリングし、入力形式を変換してStep Functionsへ渡したいと考えています。独自のポーリング用Lambdaコードは作成せず、単一の送信元と単一の送信先を直接接続する必要があります。
> 最も適したサービスはどれですか。
>
> - A. Amazon EventBridge Pipes
> - B. AWS Step Functionsだけを使用する
> - C. Amazon SNSトピックを作成する
> - D. Amazon Kinesis Data Streamsへ移行する

> [!success]- 答え：**A（EventBridge Pipes）**
> 1つの送信元をポーリングし、フィルタ・変換・エンリッチメント後に1つの送信先へ渡す。

### 比較の核心

Pipesは「一本の配管にフィルターと変換器を付ける」、Step Functionsは「複数の作業場所を順番に回る工程表」。ポイントツーポイント統合か、状態を持つワークフローかで選ぶ。

- **A**：SQSからStep Functionsへの直接接続に合う。
- **B**：受け取った後の複数工程には使えるが、SQSポーリング接続は別途必要。
- **C**：プッシュ型ファンアウト用。
- **D**：再読可能なストリームが必要な要件ではない。

単一送信元→単一送信先、フィルタ・変換、統合コード削減ならPipes。

## 第4問（Application Integration ｜ SNS と EventBridge）

> [!question] 問題（再掲）
> 自社アプリケーションが発行する業務イベントを、イベント内容の属性に応じて異なるAWSアカウントのターゲットへルーティングし、必要に応じてイベントをアーカイブして再送したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon EventBridge
> - B. Amazon SNS
> - C. Amazon SQS FIFOキュー1つ
> - D. Amazon MQ

> [!success]- 答え：**A（Amazon EventBridge）**
> イベントパターンによるルーティング、クロスアカウントイベントバス、アーカイブとリプレイを提供する。

### 比較の核心

SNSは「登録者へ一斉通知する放送局」、EventBridgeは「内容を見て適切な部署へ仕分け、必要なら記録を再送する交換局」。単純ファンアウトか高度なイベントルーティングかを見る。

- **A**：属性ルール、クロスアカウント、再送に合う。
- **B**：同じメッセージを多数の購読先へ低遅延プッシュする場合に向く。
- **C**：一つの作業列であり全ターゲットへの配信ではない。
- **D**：既存ブローカー互換アプリ向け。

イベントバス、コンテンツベースルール、アーカイブならEventBridge。通知・モバイルプッシュならSNS。

## 第5問（Application Integration ｜ Kinesis Data Streams と SQS）

> [!question] 問題（再掲）
> センサーイベントを順序付きのストリームとして数日間保持し、複数の独立した分析アプリケーションがそれぞれ同じイベントを繰り返し読み取れるようにしたいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon Kinesis Data Streams
> - B. Amazon SQS Standardキュー
> - C. Amazon SNSから単一SQSキューへ配信する
> - D. AWS Step Functions

> [!success]- 答え：**A（Kinesis Data Streams）**
> 保持期間内のストリームを複数コンシューマーが独立した読み取り位置から再読できる。

### 比較の核心

Kinesisは「一定期間残る映像テープを複数担当がそれぞれ再生」、SQSは「一件の仕事を誰か一人が受け取って完了後に捨てる作業箱」。再読と消費モデルが違う。

- **A**：順序、保持、複数独立読者に合う。
- **B**：処理の分散とバッファリング向け。
- **C**：単一キューでは全分析アプリに同じコピーを渡さない。
- **D**：状態遷移を管理する。

時系列ログの再処理・複数分析はKinesis、非同期ジョブの受け渡しはSQS。

## 第6問（Compute ｜ Outposts と Local Zones）

> [!question] 問題（再掲）
> 工場設備との通信遅延とデータ所在地の要件により、AWS計算基盤を自社工場の建物内に設置する必要があります。AWSがハードウェアを管理し、リージョンと一貫したAPIを使いたいと考えています。
> 最も適した選択肢はどれですか。
>
> - A. AWS Outposts
> - B. AWS Local Zones
> - C. AWS Wavelength
> - D. AWS Direct Connect

> [!success]- 答え：**A（AWS Outposts）**
> 顧客が指定するオンプレミス拠点へAWS管理インフラを設置する。

### 比較の核心

OutpostsとLocal Zonesはどちらもリージョン外へAWS機能を近づけるが、所有場所が違う。自社建物ならOutposts、AWSが用意した都市拠点ならLocal Zones。

- **A**：工場内という要件に一致。
- **B**：都市圏のAWS拠点であり任意の工場内ではない。
- **C**：5G通信事業者網内。
- **D**：専用線で接続するだけで計算基盤を設置しない。

配置場所を図にして読むと誤りにくい。

## 第7問（Compute ｜ Local Zones と Wavelength）

> [!question] 問題（再掲）
> 5G接続のARアプリケーションで、モバイル端末から通信事業者ネットワーク内の計算処理までの遅延を最小にする必要があります。
> 最も適した選択肢はどれですか。
>
> - A. AWS Wavelength
> - B. AWS Local Zones
> - C. AWS Outposts
> - D. Amazon CloudFront Functions

> [!success]- 答え：**A（AWS Wavelength）**
> 5Gネットワークのエッジへ対応ワークロードを配置する。

### 比較の核心

Local Zonesは「都市の近く」、Wavelengthは「携帯網の中」。どちらも低遅延だが、5G端末から通信事業者網内という経路がWavelengthを決める。

- **A**：5G ARの要件に直接合う。
- **B**：都市圏の一般低遅延ワークロード向け。
- **C**：顧客施設内。
- **D**：HTTPリクエストに対する軽量エッジ処理用。

「5G」が出たらWavelengthを検討し、単なる都市の低遅延ならLocal Zonesを検討する。

## 第8問（Migration / Transfer ｜ Application Discovery Service と Migration Hub）

> [!question] 問題（再掲）
> 移行プロジェクトの計画段階で、オンプレミスサーバーのCPU使用率、構成、実行中プロセス、依存関係を収集したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Application Discovery Service
> - B. AWS Migration Hub
> - C. AWS Application Migration Service
> - D. AWS Database Migration Service

> [!success]- 答え：**A（Application Discovery Service）**
> オンプレミス環境を発見・計測して移行計画の入力データを作る。

### 比較の核心

Discovery Serviceは「現場調査員」、Migration Hubは「全体の進捗ボード」。調査データを集めるか、複数移行の状態をまとめるかで分ける。

- **A**：構成・使用率・依存関係収集に合う。
- **B**：アプリケーション単位の移行追跡に使う。
- **C**：サーバーの継続複製とリホスト用。
- **D**：DB移行用。

移行ライフサイクルを「発見→計画→実行→追跡」に分けて選ぶ。

## 第9問（Analytics ｜ Managed Service for Apache Flink と Data Firehose）

> [!question] 問題（再掲）
> クリックストリームに対して、過去10分間の状態を保持したウィンドウ集計と異常パターン検出をリアルタイムで行いたいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon Managed Service for Apache Flink
> - B. Amazon Data Firehose
> - C. Amazon S3 Transfer Acceleration
> - D. AWS Glue Data Catalog

> [!success]- 答え：**A（Managed Service for Apache Flink）**
> 状態と時間ウィンドウを持つ複雑なストリーム計算を継続実行する。

### 比較の核心

Flinkは「流れを見ながら計算し続ける分析装置」、Firehoseは「流れてくるデータをまとめて保存先へ運ぶ配送管」。計算か配信かで選ぶ。

- **A**：ウィンドウ集計と状態付き異常検出に合う。
- **B**：S3、Redshift、OpenSearch等への配信に合う。
- **C**：S3アップロード高速化。
- **D**：メタデータ管理。

複雑なリアルタイム処理はFlink、最低限の変換と配信ならFirehose。

## 第10問（Cost Optimization ｜ Savings Plans と Reserved Instances）

> [!question] 問題（再掲）
> 企業は、リージョンやインスタンスファミリーが変わる可能性のあるEC2利用に加えて、FargateとLambdaも継続利用します。一定額のコンピューティング使用量を1年または3年コミットして、広い範囲へ柔軟に割引を適用したいと考えています。
> 最も適した購入オプションはどれですか。
>
> - A. Compute Savings Plans
> - B. EC2 Standard Reserved Instances
> - C. EC2 Dedicated Hosts
> - D. EC2 Spot Instancesだけを使用する

> [!success]- 答え：**A（Compute Savings Plans）**
> 1時間当たりの使用額をコミットし、EC2・Fargate・Lambdaへ柔軟に割引を適用する。

### 比較の核心

Compute Savings Plansは「複数交通手段で使える定額回数券」、Standard RIは「特定路線向けの割引定期券」。柔軟性と最大割引のどちらを重視するかで変わる。

- **A**：サービス・リージョン・ファミリーをまたぐ柔軟性に合う。
- **B**：EC2の安定した特定構成では候補だが、Fargate/Lambdaをカバーしない。
- **C**：物理ホスト専有やライセンス持ち込み向け。
- **D**：中断可能で価格が変動する余剰容量で、コミット割引とは別物。

EC2だけでなくFargate/Lambdaが出たらCompute Savings Plansが強い決め手。
