## 第1問（Compute ｜ EBSスナップショット と AMI）

> [!question] 問題（再掲）
> 同じOS、ミドルウェア、起動設定を持つEC2インスタンスを複数リージョンで繰り返し起動したいと考えています。
> 最も適した方法はどれですか。
>
> - A. EC2インスタンスからAMIを作成し、必要なリージョンへコピーする
> - B. データ用EBSボリュームのスナップショットだけをコピーする
> - C. 稼働中インスタンスのメモリ内容をS3へ保存する
> - D. AWS BackupでRDSのバックアッププランを作成する

> [!success]- 答え：**A（AMIを作成・コピー）**
> AMIはEC2起動に必要なルートボリュームと起動設定をまとめた再利用可能なイメージである。

### 比較の核心

EBSスナップショットは「ディスクの写真」、AMIは「そのディスクを使ってサーバーを起動する設計図」。起動可能な複製を作るならAMIを使う。

- **A**：同じEC2を別リージョンで起動できる。
- **B**：ボリューム復元はできるが、EC2起動情報が不足する。
- **C**：通常のAMIはRAM状態を保存しない。
- **D**：RDSバックアップは無関係。

「データだけ」はスナップショット、「OS込みのEC2を起動」はAMI、「RAMも保持して再開」は休止と整理する。

## 第2問（Storage ｜ EFS と FSx for Lustre）

> [!question] 問題（再掲）
> 数百台のLinux計算ノードが同じデータセットへ並列アクセスし、HPC処理で最大限のスループットを得る必要があります。処理元データはS3にあります。
> 最も適したストレージはどれですか。
>
> - A. Amazon FSx for Lustre
> - B. Amazon EFS General Purpose
> - C. Amazon S3 Glacier Flexible Retrieval
> - D. Amazon FSx for Windows File Server

> [!success]- 答え：**A（FSx for Lustre）**
> 大規模並列I/Oを行うHPCワークロード向けに設計されている。

### 比較の核心

EFSは「一般業務で共有するファイルサーバー」、Lustreは「多数の計算ノードが同時に読み書きする高速並列倉庫」。Linux共有という共通点だけでEFSを選ばない。

- **A**：HPCとS3データ連携に合う。
- **B**：Webコンテンツやホームディレクトリなどの汎用NFS向け。
- **C**：アーカイブでありマウント型高速処理には向かない。
- **D**：Windows SMB向け。

性能要件がHPC・機械学習・並列処理ならLustre、一般共有ならEFS。

## 第3問（Storage ｜ FSx for Windows と FSx for ONTAP）

> [!question] 問題（再掲）
> Windowsアプリケーションが、Active Directory統合されたSMB共有とWindows ACLを必要としています。NetApp固有機能やNFS/iSCSIは不要です。
> 最も適したストレージはどれですか。
>
> - A. Amazon FSx for Windows File Server
> - B. Amazon FSx for NetApp ONTAP
> - C. Amazon EFS
> - D. Amazon FSx for Lustre

> [!success]- 答え：**A（FSx for Windows File Server）**
> WindowsネイティブのSMB、AD統合、Windows ACLを提供する。

### 比較の核心

FSx for Windowsは「Windows専用の社内ファイルサーバー」、FSx for ONTAPは「NetAppの管理機能と複数プロトコルを持つ統合ストレージ」。必要以上の機能を選ばない。

- **A**：Windows固有要件へ最短で合う。
- **B**：NetApp移行、NFSとSMB併用、ONTAP機能が必要な場合。
- **C**：Linux NFS向け。
- **D**：HPC向け。

AD・SMB・Windows ACLならFSx for Windows、ONTAP互換やマルチプロトコルならFSx for ONTAP。

## 第4問（Storage ｜ S3 Inventory と Storage Lens）

> [!question] 問題（再掲）
> 特定のS3バケットに保存された全オブジェクトについて、キー、サイズ、ストレージクラス、暗号化状態、レプリケーション状態をCSVまたはParquetの一覧として毎日出力したいと考えています。
> 最も適した機能はどれですか。
>
> - A. S3 Inventory
> - B. S3 Storage Lens
> - C. S3 Batch Operations
> - D. AWS Config

> [!success]- 答え：**A（S3 Inventory）**
> オブジェクト単位のメタデータ明細を定期レポートとして出力する。

### 比較の核心

Inventoryは「商品一つずつの在庫台帳」、Storage Lensは「店舗全体の在庫量や管理状態を示す経営グラフ」。CSV/Parquetの明細という出力形式も強い手掛かり。

- **A**：キー、サイズ、暗号化などの一覧に合う。
- **B**：集約メトリクスとダッシュボード用。
- **C**：オブジェクトへ一括操作する実行機能。
- **D**：AWSリソース構成を追跡する。

明細はInventory、傾向はStorage Lens、明細を使って操作するのがBatch Operations。

## 第5問（Database ｜ RDS Proxy と ElastiCache）

> [!question] 問題（再掲）
> Lambda関数からRDSへ同時に多数の接続が発生し、データベース接続数の上限に達しています。SQL結果のキャッシュではなく、接続自体を再利用したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon RDS Proxy
> - B. Amazon ElastiCache
> - C. Amazon DynamoDB Accelerator（DAX）
> - D. Amazon API Gatewayのキャッシュ

> [!success]- 答え：**A（Amazon RDS Proxy）**
> 多数のアプリ接続をDBへの少数接続へ多重化する。

### 比較の核心

RDS Proxyは「窓口への行列を整理する受付」、ElastiCacheは「同じ質問への回答を手前に置く掲示板」。接続とデータという対象の違いを読む。

- **A**：コネクションプールの要件に一致。
- **B**：読み取りデータのキャッシュ。
- **C**：DynamoDB専用キャッシュ。
- **D**：HTTP APIレスポンスのキャッシュ。

Lambdaの同時実行とDB接続枯渇はRDS Proxyの定型。読み取り負荷ならElastiCacheも検討する。

## 第6問（Database ｜ DAX と ElastiCache）

> [!question] 問題（再掲）
> DynamoDBを利用するアプリケーションで、コード変更を最小限に抑えながら、結果整合性でよい読み取りをマイクロ秒単位に高速化したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon DynamoDB Accelerator（DAX）
> - B. Amazon ElastiCache for Redis
> - C. Amazon RDS Proxy
> - D. DynamoDB Global Tables

> [!success]- 答え：**A（DAX）**
> DynamoDB互換APIを備え、反復読み取りをインメモリから返す専用キャッシュである。

### 比較の核心

DAXは「DynamoDB専用の受付前コピー」、ElastiCacheは「用途を自由に設計する汎用メモリ棚」。専用品は変更量を抑えやすい。

- **A**：DynamoDB、マイクロ秒、最小変更、結果整合性に合う。
- **B**：柔軟だがキャッシュアサイドや無効化を自分で設計することが多い。
- **C**：RDS接続のプール。
- **D**：マルチリージョン複製。

強い整合性読み取りや毎回異なるデータではDAXの利点が薄い点にも注意する。

## 第7問（Database ｜ Neptune と DocumentDB）

> [!question] 問題（再掲）
> 不正取引の調査で、口座、端末、IPアドレス、取引の関係を複数段たどり、共通する関連ノードを高速に発見したいと考えています。
> 最も適したデータベースはどれですか。
>
> - A. Amazon Neptune
> - B. Amazon DocumentDB（MongoDB互換）
> - C. Amazon Timestream
> - D. Amazon Keyspaces（Apache Cassandra向け）

> [!success]- 答え：**A（Amazon Neptune）**
> エンティティ間の関係をエッジとして保存し、多段探索を効率化する。

### 比較の核心

Neptuneは「相関図を線に沿ってたどる」、DocumentDBは「JSON書類の条件検索」。同じNoSQLでも得意な問い合わせ形が違う。

- **A**：関連ノードの複数段探索に合う。
- **B**：MongoDB互換の文書保存向け。
- **C**：時系列測定値向け。
- **D**：Cassandra互換の大規模ワイドカラム向け。

問題文の名詞より動詞に注目し、「関係をたどる」ならグラフDBを選ぶ。

## 第8問（Database ｜ Aurora Global Database と DynamoDB Global Tables）

> [!question] 問題（再掲）
> 複雑なSQL、トランザクション、外部キーを使う既存のMySQL互換アプリケーションを、複数リージョンへ展開します。主リージョン障害時のリージョン切り替えと、他リージョンでの低遅延読み取りが必要です。
> 最も適した構成はどれですか。
>
> - A. Amazon Aurora Global Database
> - B. Amazon DynamoDB Global Tables
> - C. RDSリードレプリカを同一AZに作成する
> - D. Amazon ElastiCache Global Datastoreだけを使用する

> [!success]- 答え：**A（Aurora Global Database）**
> リレーショナルモデルを維持しつつ、複数リージョンの読み取りとDRを実現する。

### 比較の核心

どちらも「グローバル」だが、データモデルが決定的に違う。AuroraはSQLの表とトランザクション、DynamoDBはキー・バリューと各リージョン書き込みに強い。

- **A**：MySQL互換と外部キーを維持できる。
- **B**：NoSQLへ再設計でき、マルチリージョン書き込みが必要なら候補。
- **C**：同一AZではリージョンDRにならない。
- **D**：キャッシュだけでは永続DBを置き換えられない。

グローバル要件より先に「関係DBかNoSQLか」を確定させる。

## 第9問（Storage ｜ S3 Batch Operations と イベント通知）

> [!question] 問題（再掲）
> バケットに既に存在する5億個のオブジェクトへ、新しいタグを一括付与したいと考えています。今後追加されるオブジェクトだけでなく、既存データ全体が対象です。
> 最も適した方法はどれですか。
>
> - A. S3 Batch Operationsで一括ジョブを実行する
> - B. S3イベント通知からLambdaを呼び出す
> - C. S3 Transfer Accelerationを有効化する
> - D. S3のマルチパートアップロードを使用する

> [!success]- 答え：**A（S3 Batch Operations）**
> 既存オブジェクト一覧を対象に大規模な管理操作を実行する。

### 比較の核心

Batch Operationsは「過去の名簿全員へ一斉作業」、イベント通知は「これから受付に来た人へ都度作業」。時間軸が違う。

- **A**：既存5億件という決め手に合う。
- **B**：今後のPUTなどを処理する場合。
- **C**：アップロード経路の高速化。
- **D**：大きな一ファイルを分割アップロードする。

既存分はBatch Operations、将来分はイベント通知という併用構成も実務では自然。

## 第10問（Analytics ｜ Logs Insights と OpenSearch Service）

> [!question] 問題（再掲）
> CloudWatch Logsに集約済みのLambdaログについて、障害時に数時間分を一時的に絞り込み、集計クエリを実行したいと考えています。専用検索クラスターや常設ダッシュボードは不要です。
> 最も適した方法はどれですか。
>
> - A. CloudWatch Logs Insightsで対話的にクエリする
> - B. Amazon OpenSearch Serviceへ常時取り込む
> - C. Amazon AthenaでRDSへ直接クエリする
> - D. AWS X-Rayだけでログ全文を検索する

> [!success]- 答え：**A（CloudWatch Logs Insights）**
> CloudWatch Logs内のログへオンデマンドで検索・集計を行う。

### 比較の核心

Logs Insightsは「保管中のログを必要な時だけ調べる検索窓口」、OpenSearchは「検索・可視化を常設する専用資料室」。要件の継続性と運用規模を見る。

- **A**：一時的な障害調査と追加基盤不要の要件に合う。
- **B**：大量ログの継続全文検索や常設ダッシュボードに向く。
- **C**：Athenaは主にS3データを対象とする。
- **D**：トレースのサービス間経路を見るもので、ログ全文検索の代替ではない。

短期調査はLogs Insights、検索分析プラットフォームが必要ならOpenSearch。
