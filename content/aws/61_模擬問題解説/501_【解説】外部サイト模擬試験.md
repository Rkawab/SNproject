# 【解説】外部サイト模擬試験（[[501_【問題】外部サイト模擬試験|501]] 対応）

> [!info] このノートの位置づけ
> [[501_【問題】外部サイト模擬試験]] の各問を、答えの暗記ではなく「なぜそうなるか」で理解するための深掘り解説。たとえ話とSAA試験での着眼点を添える。問題文は各問の冒頭に再掲する。
> 見出し（`## 第N問（…）`）は問題ファイルと**完全に一致**させてあるので、Obsidianの見出しリンクで問題⇄解説を相互に飛べる。
>
> 第1〜3問は **コンピュート選定の3兄弟**。第1問（30分→ECS）と第2問（5分→Lambda）は「実行時間が15分上限を超えるか」で答えが逆転する対の問題、第3問は「高可用性＝冗長化であってバックアップではない」を問う。第4問は分野が変わり、**サーバーレスAPIのコスト最適化（HTTP API か REST API か・使用量プランの是非）** を扱う。第5問はさらに分野が変わり、**オンプレ⇄VPC接続の方式選定（Site-to-Site VPN／Client VPN／Direct Connect）とトンネル監視（CloudWatch か Config か）** を扱う。第6問は **コンテナイメージ（ECR）の脆弱性スキャン** で、**Inspector／Security Hub／Patch Manager／Config の役割の違い** を問う。第7問は **リージョン間EFSの定期同期** で、**DataSync／Direct Connect／Storage Gateway の役割の違い**（同期は誰がやるのか）を問う。第8問は **DB認証情報の安全管理とローテーション** で、**「シークレット管理（Secrets Manager）」と「鍵管理（KMS／CloudHSM）」の取り違え** を問う。

## 第1問（Containers ｜ バッチ処理のコンテナ移行とイベント駆動）

> [!question] 問題（再掲）
> ある企業は、オンプレミスで稼働しているレガシーな在庫管理およびレポート生成バッチを、AWSにリフト＆シフトで移行したい。Linuxのcronで「5分おき」「毎時20分」など細かくスケジュールされたジョブが数百本あり、各ジョブはBash/Pythonスクリプト、実行時間は数秒〜30分。可能な限り既存コードを書き換えずに移行したい。さらに将来は「在庫変動イベント」「外部トリガー」に応じて該当ジョブを即時実行できるイベント駆動の柔軟性も求められる。最も満たすソリューションはどれか。
>
> - A. コンテナ化し、ECS（Fargate起動タイプ）で実行。EventBridge Schedulerのcronで定期的にECSタスクとして起動。
> - B. コンテナ化し、AWS Batchのジョブとして実行。EventBridge Schedulerのcronで定期実行。
> - C. スクリプトをS3に保存しAthenaでクエリ可能な形式に変換。Lambda経由でAth-enaを呼び出して実行。
> - D. 各スクリプトをLambda関数に変換し、EventBridge Schedulerのcronで定期実行。

> [!success]- 答え：**A**
> コンテナ化で最小改修移行、Fargateでサーバー管理なし・30分実行OK、EventBridgeで「定期＋イベント」を同一構成で起動。A以外は「論点違い（C）」か「制約超過（D）」か「用途違い（B）」。

### 要件を分解する

| 要件 | 言い換え | 効く部品 |
|---|---|---|
| 極力書き換えず移行（リフト＆シフト） | 既存Bash/Pythonをそのまま動かす | **コンテナ化**（C・Dを弾く） |
| 実行時間 数秒〜**30分** | 15分を超える | **Lambda不可**（Dを弾く） |
| 数百本の細かいcron | スケジュール定期実行 | EventBridge Scheduler |
| 将来は**イベントで即時実行** | 待ち時間なく起動 | EventBridgeルール → ECS RunTask |

### たとえ話：作り置き厨房（Batch）と オンデマンド調理（ECSタスク）

- **AWS Batch** = 大型の作り置き厨房。素材をキューに積んで、空いた設備でまとめて大規模調理。重い仕込みを大量にさばくのは得意だが、**「いま来た1件を今すぐ」は順番待ちになる**。
- **ECS on Fargate のタスク** = 注文が入るたびに専用コンロを1つ立てて1品すぐ作る方式。**毎時の仕込み（スケジュール）にも飛び込み注文（イベント）にも即応**でき、終わればコンロを片付ける（タスク終了で課金停止）。

今回は「数百種の小皿を、決まった時間にも・急な注文にも出したい」→ コンロを都度立てる **ECS** が正解。大型厨房（Batch）はオーバースペックで即応性に欠ける。

### なぜ A か（B との分かれ目が本命）

A も B も「コンテナ化＋EventBridge Scheduler」で入口はそっくり。差は **ECSタスク vs Batchジョブ**。

- **ECS(Fargate)** … タスクを直接・即時に起動。EventBridgeルールがイベントを拾って `RunTask` を低レイテンシで叩ける。スケジュールもイベントも同じ仕組みで統一。
- **AWS Batch** … ジョブをキューに積み、コンピュート環境を確保・スケールしながら順番にさばくモデル。大規模・計算集約・大量並列向け。5分おき×数百本の軽量cronには **キュー管理と起動レイテンシが過剰**で、イベントの即時実行とも噛み合わない。

### 各選択肢の正誤

- **A：正解** … コンテナ化＝最小改修、Fargate＝サーバーレスで30分実行可、EventBridge（Scheduler＋ルール）で定期＋イベント駆動を1アーキに統一。
- **B：誤り** … Batchはキュー投入型・大規模バッチ向け。細かい短時間ジョブには重く、即時イベント実行に不向き。「コンテナ化」で釣る引っかけ。
- **C：誤り** … Athenaは **S3上のデータにSQLを投げる**サービスで、スクリプトを実行する仕組みではない。「スクリプトをクエリ可能な形式に変換」自体が意味不明で完全な作り直し。リフト＆シフトに真っ向から反する論点違い。
- **D：誤り** … **Lambdaは最大15分**で最長30分のジョブを実行できない。Bashはネイティブ非対応（カスタムランタイム要）でハンドラ移植も必要＝書き換え発生。

### SAA的な着眼点

- 「**既存スクリプトを極力書き換えず移行**」→ ほぼ **コンテナ化（ECS/Fargate）** が定石。
- 「**実行15分超 or 長時間/不明**」→ **Lambdaを消す**。今回の「30分」はそのための数字。
- **ECS vs Batch**：Batchは大規模・計算集約・大量並列・キュー許容（HPC的）。任意コンテナを即時オンデマンド起動したい／イベント駆動なら **ECS(Fargate)**。
- **EventBridge Scheduler＝cron/rate定期**、**EventBridgeルール＝イベント起動**。どちらも `ECS RunTask` をターゲットにでき、定期＋イベントを統一できる。

## 第2問（Serverless ｜ 軽量・短時間処理の最小運用構成）

> [!question] 問題（再掲）
> ウェブアプリのユーザーデータを処理する新しいバックエンドを設計している。処理はデータ変換＋軽量分析で、完了に最大5分。1時間に1回のみ実行、メモリ要件は最大512MB。将来の利用者増加に対応し、最小の運用労力で高コスト効率を実現したい。最善の構成はどれか。
>
> - A. AWS Batchでジョブをスケジュールし、必要な計算環境で実行。
> - B. ECS（Fargateベース）を使い、EventBridgeで1時間ごとにFargateタスクをトリガー。
> - C. EC2を起動しソフトをインストール、cronで1時間ごとに実行。
> - D. Lambdaでコード化し512MBで構成、EventBridgeで1時間ごとにLambdaをトリガー。

> [!success]- 答え：**D**
> 5分・512MBはLambdaの制約内。サーバーもコンテナも持たず運用労力が最小、実行時間ぶんだけの従量課金でコスト効率最高、利用者増加にも自動スケール。前問（30分→ECS）と「実行時間が15分を超えるか」で答えが逆転する対の問題。

### 要件を分解する

| 要件 | 数値 | 効き方 |
|---|---|---|
| 実行時間 | 最大**5分** | Lambdaの15分上限の**内側** → Lambda OK |
| メモリ | 最大**512MB** | Lambda上限(10GB)内 → 余裕 |
| 頻度 | **1時間に1回**だけ | 常時起動は無駄。イベント起動向き |
| 方針 | **最小運用・高コスト効率**・将来増加 | フルサーバーレスが最適 |

### たとえ話：タクシー（Lambda）と マイカー常時待機（EC2）

- **Lambda** = 必要なときだけ呼ぶタクシー。**乗った5分ぶんだけ**払えばよく、車庫も整備も自分で持たない（運用ゼロ）。客が増えても会社が車を増やす（自動スケール）。
- **EC2＋cron** = 1日5分しか乗らないのに**マイカーを24時間エンジンかけて待機**させる方式。駐車場代・車検・整備（パッチ・監視）を全部自分で抱え、稼働率が最低でコストが最悪。
- **ECS(Fargate)** = レンタカー。タクシーよりは手配（コンテナ管理）の手間があり、短時間のちょい乗りならタクシー（Lambda）の方が安くて楽。

### なぜ D か（前問との対比で覚える）

| | 第1問 | 第2問 |
|---|---|---|
| 実行時間 | 30分（>15分） | 5分（<15分） |
| 既存資産 | Bash/Pythonスクリプト多数 | 新規バックエンド（コンテナ要件なし） |
| 答え | **ECS(Fargate)** | **Lambda** |

「**Lambdaの制約（15分・10GBメモリ）に収まるか？**」が ECS と Lambda の最初の分岐点。収まる今回はより軽量・低コスト・低運用の **Lambda** を選ぶ。

### 各選択肢の正誤

- **A：誤り** … Batchは大規模・計算集約バッチ向け。1時間に1回の軽量処理にはキューイングとコンピュート環境管理が完全に過剰。
- **B：誤り** … 動きはするが、イメージ管理・タスク定義の手間があり、短時間ジョブではLambdaより運用負荷もコストも不利。コンテナが必要な要件もない。
- **C：誤り** … **最悪の選択**。OSパッチ・スケール・監視を自前で抱え運用負荷最大。5分しか使わないのに常時起動で課金され続けコスト効率最低、自動スケールもない。
- **D：正解** … 15分・10GB以内の短時間・低頻度・軽量処理はLambdaの独擅場。最小運用・高コスト効率・自動スケールをすべて満たす。

### SAA的な着眼点

- **まずLambdaの制約に収まるか判定**：①実行15分以内 ②メモリ10GB以内 ③`/tmp`やランタイム制約。収まれば短時間・イベント駆動・低運用ではLambdaが第一候補。
- 「**最小の運用労力（least operational overhead）**」「**コスト効率**」が出たら、**サーバーレス（Lambda）＞ Fargate ＞ EC2** の優先順位を思い出す。
- **EC2＋cronは「運用負荷大・常時課金」**で、運用を増やしたくない設問のほぼ確実な当て馬。
- **EventBridgeルールの定期トリガー**はLambda/ECS共通の部品で決め手にならない。実行時間・運用・コストの軸で選ぶ。

## 第3問（Compute ｜ Web層のマルチAZ高可用化）

> [!question] 問題（再掲）
> オンラインストアのWebシステムをAWSで運用。Webサーバーは**単一EC2**、DBは**RDS PostgreSQLのマルチAZ**。障害発生時やリクエスト急増時に備え、より可用性の高い構成にしたい。最も適切なアプローチはどれか。
>
> - A. EC2を複数AZにまたがるAuto Scalingグループに配置。ALBを作成しASGをターゲットグループに設定。
> - B. AWS Direct Connectでエンドユーザーからのアクセス速度と可用性を向上。
> - C. EC2を複数起動しALB配下に配置。Route 53フェイルオーバールーティングでトラフィックをALBへ。
> - D. EC2のAMIを定期作成し、障害時にAMIから新EC2を起動。

> [!success]- 答え：**A**
> DBは既にRDSマルチAZで冗長化済み。論点は**単一EC2のWeb層の高可用化**だけ。キーワードは「**障害（＝可用性）**」と「**リクエスト急増（＝スケール）**」の2つで、両方を同時に満たすのはマルチAZのAuto Scalingグループ＋ALBだけ。

### 要件を分解する

| キーワード | 必要なもの | 満たす選択肢 |
|---|---|---|
| 障害発生時に止まらない | 複数AZでの**冗長化＋自動置換** | A |
| **リクエストの急増**に耐える | **Auto Scaling**による自動増設 | A のみ（C・Dは固定 or 1台） |

### たとえ話：レジの増設（ASG＋ALB）と バックアップDVD（AMI）

- **マルチAZ ASG＋ALB** = 複数フロアにレジ（EC2）を並べ、**警備員（ALB）が客を空いているレジへ誘導**し、故障レジは即閉鎖。混雑時は**自動でレジを増設**（Auto Scaling）、1フロア停電（AZ障害）でも他フロアで営業継続。
- **AMI定期作成＋障害時起動（D）** = レジ構成を**DVDにバックアップ**しておき、壊れたら新品を箱から出してセットアップ。**復旧までレジは0台で店は閉店**、しかも増設はできない。これは「復旧」であって「止まらない（可用性）」ではない。

### なぜ A か。誤答の急所

- **B：Direct Connect** … オンプレ拠点⇄AWSを結ぶ**専用線（ハイブリッド接続）**。インターネット越しの一般エンドユーザーのアクセスとは無関係で、可用性にもスケールにも寄与しない。**用途を取り違えさせる論点違い**の当て馬。
- **C：複数EC2＋ALB＋Route 53フェイルオーバー** … ALBまでは合っているが2点ダメ。①「複数**起動**」は**固定台数**で**Auto Scalingがない**＝リクエスト急増に自動対応できない。②**Route 53フェイルオーバーはプライマリ／セカンダリの2エンドポイントを切り替えるアクティブ-スタンバイ用**（本来は別サイト・別リージョンへのDR切替）。単一ALB前段では切替先がなく、インスタンス故障の切り離しは**既にALBのヘルスチェックが担う**ので役割が重複・無意味。
- **D：AMI定期作成＋障害時起動** … **「バックアップ≠高可用性」の典型トラップ**。手動・事後のリカバリで復旧までサービス停止（RTOが悪い）、常に1台で冗長性ゼロ、自動スケールもなく急増に無力、AMIも陳腐化リスクあり。

> [!tip] ALB と Route 53フェイルオーバー の違い
> - **ALB** = 交差点の警官。リアルタイムで複数レーン（インスタンス）に振り分け、事故ったレーンを即閉鎖。**1サイト内**の分散・切替。
> - **Route 53フェイルオーバー** = 高速入口の電光掲示板「本線通行止め→**迂回路（別サイト）へ**」。**サイトまるごとA→B**を切り替えるDNSレベルの仕組み。
> 今回必要なのは1サイト内の台数増減＆故障切り離し＝**警官（ALB＋ASG）**であって、別サイトへの迂回看板ではない。

### SAA的な着眼点

- **高可用性(HA)＝冗長化（複数AZで同時に複数稼働）＋自動置換**。**バックアップ/復元（AMI再起動）はHAではない** → D系は不正解の定番。
- **「リクエストの急増」が出たらAuto Scaling必須**。固定台数（Cの「複数起動」）は弾く。
- **Web層HAの定番＝「マルチAZ ASG ＋ ALB」**。まずこの形を探す。
- **Route 53フェイルオーバールーティング＝アクティブ-スタンバイのDR切替（別エンドポイント/別リージョン）**。単一ALB前段に置く部品ではない。
- **Direct Connect＝オンプレ⇄AWS専用線**。一般エンドユーザーのWeb高速化やHA文脈で出たらほぼ当て馬。

## 第4問（Application Integration ｜ サーバーレスAPIのコスト最適化）

> [!question] 問題（再掲）
> ある企業は、世界中のユーザーが利用するサーバーレスWebアプリをAWSで構築。バックエンドはLambdaで、その前段にAPI Gatewayを置いてHTTPエンドポイントを公開する。APIはLambdaへのシンプルなプロキシ統合が中心で、高度なリクエスト検証やAPIキーによる利用者ごとの課金管理は不要。突発的に大量のリクエストが集中することがあり、急増に対応しつつAPI層のコストを可能な限り抑えたい。最も適切なアプローチはどれか。
>
> - A. 使用量プランなしで API Gateway **REST API** をデプロイしてLambdaと統合。
> - B. API Gateway でレスポンスキャッシュを有効にし、TTLを長く設定してリクエスト数を抑制。
> - C. 使用量プランとAPIキーを設定し、API Gateway **HTTP API** をデプロイしてLambdaと統合。
> - D. 使用量プランなしで API Gateway **HTTP API** をデプロイしてLambdaと統合。

> [!success]- 答え：**D**
> 論点は2軸だけ。①コストを抑える→**HTTP API**（REST APIより約70%安・低レイテンシ）。②急増に対応→**使用量プランを付けない**（付けると逆に絞ってしまう）。この2軸で機械的に切るとDだけが残る。

### 要件を分解する

| 要件 | 言い換え | 効く判断 |
|---|---|---|
| コストを抑えたい | シンプルなAPIを安く公開 | **HTTP API**（A・Bの REST API系を弾く） |
| リクエスト急増に対応 | スロットリングしたくない | **使用量プランなし**（Cを弾く） |
| 高度な検証・課金管理は不要 | リッチな機能はいらない | REST APIの優位点が活きない |

### たとえ話：フルコース（REST API）と アラカルト（HTTP API）

- **REST API** = 全部入りのフルコース。APIキー・使用量プラン・リクエスト検証・キャッシュ・WAF連携まで揃うが、そのぶん**お高い**。
- **HTTP API** = 必要な料理だけのアラカルト。Lambdaを呼ぶだけのシンプルなAPIを**安く・速く**出せる。
- 今回は「Lambdaを叩くだけのAPIをとにかく安く」なので、余計な皿のないアラカルト（HTTP API）が正解。フルコースは食べきれない機能に課金しているようなもの。

そして **使用量プラン＝「1秒/1日に何回まで」と客の入店数に上限をかける整理券**。本来は有料APIを枠で売ったり濫用を防ぐための仕組みで、「お客さんが急に増えても全員さばきたい」という今回の願いとは真逆。整理券を配った瞬間に、せっかく来た急増トラフィックを追い返してしまう。

### なぜ D か（誤答の急所）

- **A：誤り** … REST APIは多機能だが HTTP API より割高。今回その多機能を使う要件がなく、コスト最適化では HTTP API に明確に劣る。
- **B：誤り** … キャッシュ＋長いTTLは「同じ問い合わせに同じ答えを返す静的寄りAPI」向け。**世界中のユーザーが使う動的アプリで長TTLは古いデータ（陳腐化）を返すリスク**。さらにキャッシュ自体が有料（REST APIの機能）で、コスト削減策として本質的でない。
- **C：誤り（あなたの誤答）** … 使用量プラン/APIキーは**リクエストを上限で絞る・課金管理する**機能で、「急増に対応」と相反する。加えて使用量プラン/APIキーは伝統的に **REST API の機能**で、HTTP API＋使用量プランという組み合わせ自体が前提のズレた**二重に不適切**な選択肢。
- **D：正解** … シンプルなサーバーレスAPIを HTTP API で安く公開し、余計な使用量プランを付けない。API Gateway も Lambda も**自動スケール**するので、プランで絞らなければ急増にもそのまま追従する。

### SAA的な着眼点

> [!tip] 覚え方
> - **「コスト最適化 × シンプルなサーバーレスAPI（Lambdaプロキシ）」→ HTTP API**。REST APIより約30%の価格、低レイテンシ。リッチ機能（APIキー/使用量プラン/リクエスト検証/エッジ最適化）が要るときだけ REST API。
> - **使用量プラン／APIキー＝「使わせすぎを防ぐ・枠で売る」スロットリング系機能**。「急増に対応・スケールさせたい」要件では**むしろNG**（逆方向のひっかけ選択肢として頻出）。
> - **キャッシュ＋長いTTL**は静的・反復レスポンス向け。動的・グローバルなアプリでは陳腐化リスクで不正解にされやすい。
> - 発想の核：**「コストを抑える＝機能を盛る」ではなく「余計なものを削る（HTTP API・プランなし）」**。

## 第5問（Networking / CDN / LB ｜ オンプレ接続の暗号化とトンネル監視）

> [!question] 問題（再掲）
> あるソリューションアーキテクトは、会社の企業ネットワークをVPCに接続し、オンプレミスからAWSリソースへのアクセスを許可する必要がある。オンプレ⇄VPC間のトラフィックを**ネットワーク層で暗号化**して保護し、**必要最小限のトラフィックのみ許可**するアクセス制御を実装する必要がある。さらに**VPNトンネルの状態を継続的に監視**し、異常時にアラームを出す機能も求められる。最適なソリューションはどれか。
>
> - A. Site-to-Site VPNで接続・ルーティング構成。SG/NACLで必要なトラフィックのみ許可し、**AWS Config**でVPNトンネルの状態を監視してアラーム設定。
> - B. Site-to-Site VPNで接続・ルーティング構成。SG/NACLで必要なトラフィックのみ許可し、VPNトンネルの状態を**CloudWatch**で監視してアラーム設定。
> - C. **Client VPN**エンドポイントで接続。クライアント証明書認証で認証ユーザーのみ許可。トンネル状態をCloudWatchで監視してアラーム設定。
> - D. **Transit Gateway＋Direct Connect Gateway**でオンプレ→VPCのルーティング確立。SG/NACLで必要なトラフィックのみ許可し、CloudWatchでTGWの接続メトリクスを監視してアラーム設定。

> [!success]- 答え：**B**
> 4つの要件を1つずつ部品に割り当てると機械的に決まる。「拠点まるごと接続＋ネットワーク層の暗号化」＝Site-to-Site VPN（IPsec）、「最小限の許可」＝SG/NACL、「VPNトンネルの状態監視」＝CloudWatch（`TunnelState`）。この3点セットが噛み合うのはBだけ。Aは監視サービス違い、Cは接続単位違い、Dは暗号化欠落で脱落する。

### 要件を分解する

| 要件 | 言い換え | 効く部品 |
|---|---|---|
| 企業ネットワークをVPCに接続 | 拠点（サブネット）まるごと相互接続 | **Site-to-Site VPN**（Client VPNを弾く） |
| ネットワーク層で暗号化 | L3で**IPsec**暗号化 | **VPN（IPsec）**（Direct Connect単体を弾く） |
| 必要最小限のトラフィックのみ許可 | きめ細かなアクセス制御 | **SG / NACL** |
| トンネル状態を継続監視＋アラーム | UP/DOWNを**リアルタイム**検知 | **CloudWatch（`TunnelState`）**（Configを弾く） |

### たとえ話：装甲トンネル と 地下専用線、そして「建物ごと」か「個人通行証」か

- **Site-to-Site VPN** = 公道（インターネット）の上に張った**装甲付きの連絡トンネル**。会社の建物（オンプレ拠点）と倉庫（VPC）を**まるごと**つなぎ、中身はIPsecで暗号化されているので公道を通っても覗かれない。
- **Client VPN** = 社員が**1人ずつ通行証（証明書）をかざして**トンネルに入る方式。建物ごとではなく「個人」を通す、在宅ワーカー向けのリモートアクセス。
- **Direct Connect** = 会社と倉庫を結ぶ**物理的な地下専用線**。速くて安定だが、線そのものは暗号化していない（物理的に閉じている前提）。暗号化が要るなら、この専用線の中にさらにVPNトンネルを通す必要がある。

今回は「**建物（企業ネットワーク）まるごと**を、**暗号化して**つなぐ」なので、装甲トンネル＝Site-to-Site VPN が正解。個人通行証（Client VPN）でも、暗号化なしの地下線（DX単体）でもない。

### なぜ B か（本命は A との分かれ目 ＝ Config か CloudWatch か）

A と B は「Site-to-Site VPN＋SG/NACL」まで**まったく同じ**で、違いは監視を **AWS Config** でやるか **CloudWatch** でやるかの一点だけ。ここが本問の急所。

- **CloudWatch** = 設備の**バイタルモニター**。VPNトンネルの `TunnelState`（1=UP / 0=DOWN）や `TunnelDataIn/Out` を**メトリクスとしてリアルタイムに計測**し、`TunnelState < 1` でアラーム→SNS通知ができる。「状態を継続監視してアラーム」はまさにこれ。
- **AWS Config** = 設備の**設定変更の監査台帳**。「VPN接続の構成がいつ変わったか」「あるべき設定（ルール）に準拠しているか」を記録・評価する。**トンネルが今UPかDOWNかという運用状態のリアルタイム監視は守備範囲外**。

「異常（DOWN）をすぐ検知してアラーム」という運用監視の要件には、台帳（Config）ではなくモニター（CloudWatch）。

### 各選択肢の正誤

- **A：誤り** … 接続・暗号化・アクセス制御は満たすが、**監視がConfig**なのが致命傷。Configは構成変更・コンプライアンス用で、トンネルのUP/DOWNを継続監視・即アラームする用途ではない。「VPNまで合っているので正しそう」に見せる引っかけ。
- **B：正解** … Site-to-Site VPN（IPsec暗号化）＋SG/NACL（最小許可）＋CloudWatch（`TunnelState`アラーム）で、4要件をすべて過不足なく満たす。
- **C：誤り** … Client VPNは**個々のユーザー/デバイス**をVPCにつなぐリモートアクセス用。要件は「企業ネットワーク＝拠点まるごと」の接続なので接続単位が違う。証明書認証の話に気を取られると選びがち。
- **D：誤り** … **Direct Connectは専用線で、それ単体ではトラフィックを暗号化しない**ため「ネットワーク層で暗号化」を満たせない（満たすにはDXの上にVPNを重ねる）。単一VPCの接続にTransit Gatewayを持ち出すのも過剰。「TGW＋DXで本格的＝正しそう」に見せる過剰構成の罠。

### 紛らわしい接続部品の整理

| 部品 | 接続する単位 | 経路 | 暗号化 | 主な用途 |
|---|---|---|---|---|
| **Site-to-Site VPN** | 拠点（ネットワーク）まるごと | インターネット | **あり（IPsec）** | オンプレ⇄VPCを安く即時に暗号化接続 |
| **Client VPN** | 個々のユーザー/デバイス | インターネット | あり | 在宅ワーカー等のリモートアクセス |
| **Direct Connect** | 拠点（ネットワーク） | 専用線 | **なし**（要VPN併用） | 一貫帯域・低レイテンシ・大容量転送 |
| **Transit Gateway** | 多数のVPC/オンプレを集約 | ハブ | 経路次第 | 大規模なネットワークハブ |

### SAA的な着眼点

> [!tip] キーワード→定番解
> - 「**拠点（企業ネットワーク）をまるごと接続**」→ **Site-to-Site VPN**。「**個々のユーザー/在宅から**」→ **Client VPN**。接続単位で割る。
> - 「**ネットワーク層で暗号化が必須**」→ **VPN（IPsec）**。**Direct Connectは暗号化されない**ので、暗号化要件があればDX単体は×（DX＋VPNが必要）。
> - 「**VPNトンネルの状態監視・アラーム**」→ **CloudWatch（`TunnelState`）**。**AWS Configは構成変更・コンプライアンス監査**であって運用状態のリアルタイム監視ではない（この対比は頻出ひっかけ）。
> - SG/NACL は「必要最小限のトラフィックのみ許可」という**最小権限のアクセス制御**の定番表現。

## 第6問（Security / IAM ｜ コンテナイメージの脆弱性スキャン）

> [!question] 問題（再掲）
> ある企業が、社内向けの学習管理アプリを **ECS on Fargate** で運用し、コンテナイメージを **ECR** に格納している。セキュリティチームは、ECRに保存されたコンテナイメージに既知の脆弱性が含まれていないかを**継続的に評価**したい。最も満たすソリューションはどれか。
>
> - A. Security Hub を有効化し、ECRのコンテナイメージをスキャンして脆弱性を検出。
> - B. Systems Manager Patch Manager で、ECRのコンテナイメージをスキャンして脆弱性を検出。
> - C. Config コンフォーマンスパックで、コンテナイメージのセキュリティ脆弱性をスキャン。
> - D. Amazon Inspector で、ECR内のコンテナイメージをスキャンして脆弱性を検出。

> [!success]- 答え：**D**
> 「**ECRイメージ／EC2／Lambda の脆弱性（CVE）を継続スキャン**」と来たら問答無用で **Amazon Inspector**。残り3つは「集約役（Security Hub）」「パッチ適用（Patch Manager）」「設定準拠チェック（Config）」で、いずれもイメージ中身のCVEは検査しない。

### 要件を分解する

| 要件 | 言い換え | 効く部品 |
|---|---|---|
| 対象は ECR のコンテナイメージ | イメージの中身を見る | **Inspector**（ECR拡張スキャン） |
| 脆弱性が含まれていないか | 既知のCVEを検出 | Inspector |
| **継続的に**評価 | プッシュ時＋新CVE公表時に再スキャン | Inspectorの自動再スキャン |

### たとえ話：レントゲン技師・総合ダッシュボード・薬局・構造検査官

- **Inspector** = 中身をスキャンする「レントゲン技師」。イメージの中身を継続的に撮影して病気（脆弱性）を見つける。
- **Security Hub** = 各検査結果を集める「病院の総合ダッシュボード」。自分では撮影せず、各クリニック（Inspector/GuardDuty/Macie…）の結果を一画面に集約する。
- **Patch Manager** = 薬を処方する「薬局」。すでに動いている患者（稼働中サーバー）に治療（パッチ）を施す。検査はしない。
- **Config** = 「建物の構造検査官」。設定がルール通りかを見るだけで、中身の病気は診ない。

### なぜ D か

Inspector は **EC2・ECRコンテナイメージ・Lambda** を対象とする脆弱性管理サービス。ECRで**拡張スキャン（Enhanced scanning）**を有効にすると、①**イメージのプッシュ時に自動スキャン**、②**新しいCVEが公表されるたびに既存イメージを自動で再スキャン**する。これがそのまま「継続的に評価したい」を満たす。ECS on Fargate＋ECR構成での脆弱性可視化の定番解。

### 各選択肢の正誤

- **D：正解** … ECRイメージ・EC2・Lambda の脆弱性を継続スキャンする専用サービス。
- **A：誤り** … Security Hub は検出結果の**集約・統制（セキュリティ態勢管理）**。Inspector等の結果を受け取って一元表示するが、自分でイメージをスキャンしない。「有効化すればスキャンしてくれそう」に見せる役割取り違えの罠。
- **B：誤り** … Patch Manager は **Systems Manager の管理対象インスタンス（EC2/オンプレ）にパッチを適用**する仕組み。ECRのイメージを走査する機能はそもそも無い。
- **C：誤り** … Config（コンフォーマンスパック）は**リソースの設定が準拠ルールに合っているか**を評価するコンプライアンス用。イメージ内部のCVEという「中身」は守備範囲外。

### 紛らわしいセキュリティサービスの整理

| サービス | 役割 | 一言 |
|---|---|---|
| **Inspector** | 脆弱性スキャン | EC2/ECRイメージ/Lambda の穴を継続検出 |
| **Security Hub** | 検出結果の集約・統制 | 各サービスの結果を1画面に |
| **GuardDuty** | 脅威（振る舞い）検知 | 攻撃されてる？をログから検知 |
| **Macie** | データ保護 | S3の個人情報を検出 |
| **Patch Manager** | パッチ適用 | OS/アプリにパッチを当てる（検査ではない） |
| **Config** | 設定の準拠評価 | 構成がルール通りかを監査 |

### SAA的な着眼点

> [!tip] キーワード→定番解
> - 「**脆弱性スキャン / CVE**」＋「**EC2・ECRイメージ・Lambda**」→ **Inspector**。
> - 「複数サービスの検出結果を**集約・横断確認**／セキュリティ態勢管理」→ **Security Hub**。
> - 「**パッチ適用**の自動化」→ **Patch Manager**（スキャンではない）。
> - 「リソース**設定の準拠チェック**」→ **Config**（中身のCVEは見ない）。
> - **GuardDuty＝振る舞い検知（攻撃されてる？）／Inspector＝脆弱性検知（穴が空いてる？）** の対比も頻出。

## 第7問（Migration / Transfer ｜ リージョン間EFSの定期同期）

> [!question] 問題（再掲）
> ある大手製薬会社は、離れた地域に2つの主要な研究施設を持つ。各施設は機密性の制約で**近隣のAWSリージョンだけ**にアクセスし、毎日大量の研究データを生成して**各リージョンのAmazon EFS**に保存している。同社はこれらのデータを**毎日同期**させ、どちらの施設からも最新データにアクセスできるようにしたい。最も適切なアプローチはどれか。
>
> - A. AWS Direct Connectで2つのリージョン間の専用接続を確立する。
> - B. AWS Storage Gatewayで各施設のEFSをオンプレミス経由で相互共有する。
> - C. AWS Lambdaで毎日のデータ更新をトリガーとし、自前の同期処理でEFSをコピーする。
> - D. AWS DataSyncで2つのリージョンのEFS間の転送タスクをスケジュール実行し、毎日同期する。

> [!success]- 答え：**D**
> 「**AWSストレージ（EFS）同士を・リージョンを跨いで・毎日・自動で同期**」と来たら **DataSync**。残り3つは「回線を引くだけ（DX）」「オンプレ⇄AWSの窓口（Storage Gateway）」「車輪の再発明（Lambda自作）」で、いずれも"同期そのもの"をマネージドにやってはくれない。

### 要件を分解する

| 要件 | 言い換え | 効く部品 |
|---|---|---|
| 別リージョンのEFS同士 | AWSストレージ間のオンライン転送 | **DataSync** |
| 毎日同期 | スケジュール実行＋差分転送 | DataSyncの組み込み機能 |
| 手間をかけたくない | 自前のサーバ・コードを持たない | DataSync（フルマネージド） |
| 整合性を担保したい | 転送後の検証 | DataSyncの整合性チェック |

### たとえ話：2つの支店をつなぐ「専門の定期宅配便」

東京支店と大阪支店が、それぞれ毎日大量の書類（研究データ）を作る。お互い最新の書類を共有したい——という場面。

- **DataSync（正解）** = 「毎晩、増えたぶんの書類だけ箱詰めして相手支店へ届け、中身が壊れていないか検品までやる専門宅配業者」。差分・スケジュール・検品を全部おまかせ。
- **Direct Connect** = 「2支店をつなぐ専用高速道路を敷く」だけ。道路は荷物を運んでくれない。しかも本来オンプレ⇄AWS用で、リージョン間EFS同期の手段ではない。
- **Storage Gateway** = 「自社倉庫（オンプレ）とAWS倉庫をつなぐ受付窓口」。AWS内のEFS同士を結ぶ役ではない。
- **Lambda自作** = 「アルバイトに毎日手作業でコピーさせる」。動きはするが、差分管理・再試行・検品・大量データのスケーリングを全部自前で実装する羽目に。マネージドなDataSyncがあるのに再発明。

### なぜ D か

DataSyncは **EFS／FSx／S3／オンプレNFS・SMB** の間でデータ転送を自動化するマネージドサービス。今回の要件のど真ん中で、
- **EFS ⇄ EFS（リージョン跨ぎ）転送に正式対応**
- **スケジュール実行**（毎日）＋**差分転送**（増えたぶんだけ）
- **暗号化・整合性検証**まで込み

を満たす。「定期的に・自動で・マネージドに・AWSストレージ間を同期」という要件にそのまま一致する。

### 各選択肢の正誤

- **D：正解** … リージョン間EFSの定期同期をマネージドに実現できる唯一の選択肢。
- **A：誤り** … Direct Connectは「回線（接続）」を提供するだけで、データ同期処理は行わない。用途もオンプレ⇄AWS。「接続を確立」という言葉に釣られない。
- **B：誤り** … Storage Gatewayはオンプレ⇄AWSのハイブリッド窓口で、AWS内EFS同士の同期役ではない。
- **C：誤り** … 実装可能だが、差分・再試行・整合性検証・スケーリングを自作する車輪の再発明で「最も適切」ではない。

### 紛らわしい移行・転送サービスの整理

| サービス | 向き/対象 | 役割 |
|---|---|---|
| **DataSync** | AWSストレージ間／オンプレ⇄AWS | データの**同期・転送**（差分・スケジュール・検証） |
| **Direct Connect** | オンプレ⇄AWS | **専用線（回線）** を提供（同期はしない） |
| **Storage Gateway** | オンプレ⇄AWS | オンプレからAWSストレージを使う**ハイブリッド窓口** |
| **Snowball** | オンプレ→AWS（初回） | 超大容量の**物理搬送**（オフライン） |

### SAA的な着眼点

> [!tip] キーワード→定番解
> - 「**AWSストレージ間／オンプレ⇄AWS を “定期的に同期・転送”**」→ **DataSync**。
> - **Direct Connect ＝「回線・接続」の話**（同期はしない）。「接続を確立」に釣られない頻出ひっかけ。
> - **Storage Gateway ＝ オンプレ⇄AWSのハイブリッド**（AWS内同士の同期ではない）。
> - **マネージドサービスがあるのに Lambda で自前実装＝原則×**（再発明）。
> - 組み合わせ問題：初回の超大容量は **Snowball で物理搬送**、その後の**差分同期は DataSync**、がセットで頻出。

## 第8問（Security / IAM ｜ DB認証情報の安全管理とローテーション）

> [!question] 問題（再掲）
> あるアプリケーションがバックエンドのDBへアクセスしている。DBの認証情報をアプリのコードや設定ファイルから切り離して安全に管理し、情報漏洩のリスクを下げるため定期的にローテーションしたい。アプリは実行時に最新の認証情報を取得してDBに接続する必要がある。最も適切な方法はどれか。
>
> - A. AWS Secrets Manager に認証情報を保管し、自動ローテーションを有効化してアプリが実行時に取得して認証する
> - B. AWS CloudHSM に保存された鍵を使用して認証する
> - C. AWS KMS を利用して認証する
> - D. AWS KMS のデータキーを使って認証情報を取得する

> [!success]- 答え：**A**
> 「認証情報（シークレット）そのものの保管・取得・自動ローテーション」を一体で担えるのは Secrets Manager だけ。B・C・D はいずれも「暗号鍵（KMS/CloudHSM）」の話で、論点がズレている。

### 要件を分解する

| 要件 | 言い換え | 効く部品 |
|---|---|---|
| 認証情報を**コード/設定から分離** | 埋め込まず外部で管理 | Secrets Managerに保管し実行時取得 |
| **安全に管理** | 暗号化保管＋アクセス制御 | Secrets Manager（裏でKMSが暗号化、IAMで取得制御） |
| **定期的にローテーション** | パスワードを自動入れ替え | **Secrets Managerの自動ローテーション**（RDS等にマネージド対応） |

### たとえ話：貸金庫（中身）と 鍵工房（カギ）

- **Secrets Manager** ＝ **貸金庫＋中身の自動入れ替え係**。お金（＝DBパスワード）を金庫に預かり、必要なときだけ取り出させ、定期的に**中の札束を新札に交換**してくれる。
- **KMS** ＝ **金庫のカギを作る鍵工房**。カギ（暗号鍵）は作るが、**金庫の中身（パスワード）そのものは預からない**。
- **CloudHSM** ＝ 自社専用の**超頑丈な鍵保管室**。「鍵を専有ハードで自己管理（FIPS 140-2 Lv3）」という厳格要件のときだけ登場。

今回ほしいのは「**中身（認証情報）を預けて・取り出して・自動で入れ替える**」こと → 貸金庫である Secrets Manager。鍵工房（KMS）や鍵保管室（CloudHSM）は役割違い。

### なぜ A か（C＝KMSとの分かれ目が本命）

選びがちな C（KMS）と A（Secrets Manager）は、**「鍵管理」か「シークレット管理」か**で別物。

- **KMS** … 暗号鍵の生成・保管・利用制御をするサービス。Secrets Manager の裏側で「保管した認証情報を暗号化する」役割は担うが、**KMS 単体ではパスワードを保管できず、ローテーションもできない**。そもそも「KMSで認証する」という機能が存在しない。
- **Secrets Manager** … 認証情報そのものを暗号化保管し、アプリに実行時取得させ、**RDS等と連携して自動ローテーション**する。3要件をまとめて満たす。

### 各選択肢の正誤

- **A：正解** … コードからの分離・暗号化保管・実行時取得・マネージドな自動ローテーションを一体で満たす。
- **B：誤り** … CloudHSM は**暗号鍵を専有HSMで自己管理**するサービス。DBパスワードそのものの保管・ローテーションが目的ではない（鍵管理であってシークレット管理ではない）。
- **C：誤り** … KMS は**鍵の管理**。Secrets Manager の暗号化を裏で支えるが、認証情報の保管も**ローテーションもできない**。「鍵」と「中身」を混同させる頻出ひっかけ。
- **D：誤り** … データキーは**データ暗号化（エンベロープ暗号化）**用のもの。それで「認証情報を取得・ローテーションする」仕組みは存在しない。もっともらしいが操作として成立しない。

### 紛らわしい部品の整理

| サービス | 管理するもの | 主用途 |
|---|---|---|
| **Secrets Manager** | パスワード・APIキー等の**シークレット（中身）** | 保管・実行時取得・**自動ローテーション** |
| **Parameter Store（SecureString）** | 設定値・小さなシークレット | 保管・取得（**自動ローテーションは標準では非対応**） |
| **KMS** | **暗号鍵** | 鍵の生成・保管・利用制御（マルチテナント基盤） |
| **CloudHSM** | **暗号鍵（専有）** | FIPS 140-2 Lv3・鍵の完全自己管理 |

### SAA的な着眼点

> [!tip] キーワード→定番解
> - 「**DB認証情報 / パスワード / APIキー ＋ ローテーション**」→ ほぼ即 **Secrets Manager**。
> - 「**暗号化の鍵を管理**」→ **KMS**。「**鍵を専有ハードで自己管理（FIPS Lv3）**」→ **CloudHSM**。
> - 頻出ひっかけ：**鍵管理サービス（KMS/CloudHSM）を「シークレット管理」の答えに混ぜてくる**。「鍵」と「中身」を切り分けて読む。
> - Parameter Store でも保管はできるが、**マネージドな自動ローテーションが要るなら Secrets Manager** が定番。

## 第9問（Security / IAM ｜ 国・地域によるアクセス制限）

> [!question] 問題（再掲）
> あるサービスは、ALB配下のEC2インスタンスでWebコンテンツを公開している。自国内のクライアントだけにアクセスを限定したい。EC2インスタンスへのアクセスを特定の国に制限する方法はどれか。（2つ選択）
> - A. AWS WAF を ALB に関連付け、Web ACL の地理的一致（Geo match）ルールで許可する国を指定する。
> - B. ALB へのアクセスを CloudFront 経由にし、CloudFront の地理的制限（Geographic Restrictions）で許可する国を設定する。
> - C. ALB のセキュリティグループのインバウンドルールで許可する国を指定する。
> - D. AWS Shield を ALB に関連付け、Web ACL に許可する国を設定する。
> - E. ALB へのアクセスを API Gateway 経由にし、API Gateway の地理的制限で許可する国を設定する。

> [!success]- 答え：**A ＋ B**
> 国・地域単位のアクセス制御は、**AWS WAF の地理的一致ルール**か **CloudFront の地理的制限**で実現する。これが「国で弾く」の二大定番。

### 要件を分解する
| 要件 | 言い換え | 効く部品 |
|---|---|---|
| 自国だけ許可 | 送信元の**国**で許可/拒否 | WAF Geo match / CloudFront 地理的制限 |
| それ以外を拒否 | 国の許可リスト方式 | 同上（許可国リスト or ブロック国リスト） |
| ALB配下のEC2が対象 | 前段（エッジ）で遮断したい | WAFはALBに直付け／CloudFrontはALB手前に配置 |

### たとえ話：入口で「身分証の住所」を確認する係員
ビル（EC2/ALB）に「自国の人だけ通したい」とき、入口で**住所（国）を見て通す/帰す**係が要る。

- **AWS WAF** ＝ 入口に立ち、来訪者のIP→国を判定して入館可否を決める**審査係員**。Web ACL に「○○国は許可、他は拒否」という**地理ルール**を書ける。ALBに直接配置できる。
- **CloudFront の地理的制限** ＝ ビル手前の**玄関フロント（CDN）**が持つ国別入場制限。ここで弾けば奥のEC2まで届かない。
- **Shield** ＝ 押し寄せる暴徒（DDoS）を止める**警備**。住所で個人を選り分ける仕事はしない。

### なぜ A・B がセットで正解か
どちらも「**送信元IPから国を判定して通す/弾く**」機能を持つ。WAFはALBに**直付け**、CloudFrontはALBの**前段**に置く、という配置の違いがあるだけで、いずれも「国で制限」を正しく実現できる。設問は「方法を2つ」なので、この2つが解答ペアになる。

### 各選択肢の正誤
- **A：正解** … WAFはALB/CloudFront/API Gateway/AppSyncに関連付け可能。Web ACLのGeo matchで国を判定し許可/拒否できる。
- **B：正解** … CloudFrontには標準で地理的制限（許可/拒否する国のリスト）機能がある。前段で対象国以外を遮断。
- **C：誤り** … セキュリティグループは**IP/CIDR単位**の制御で、「国」という単位の指定はできない（対象の粒度違い）。
- **D：誤り** … **Web ACL は WAF の機能**。Shield（DDoS緩和）に国制限のWeb ACLは作れない。"Web ACL"の語をShieldに結びつける**ひっかけ**。
- **E：誤り** … API Gatewayに「地理的制限」機能は存在せず（地理制限はCloudFront側）、ALBをAPI Gateway経由にする構成も不自然。

### 紛らわしい部品の整理
| 部品 | 制御の単位/役割 | 国でのアクセス制限 |
|---|---|---|
| **WAF（Web ACL / Geo match）** | L7のルール審査（IP→国・SQLi・レート等） | ✅ できる |
| **CloudFront 地理的制限** | エッジでの配信可否 | ✅ できる |
| **Security Group** | IP/CIDR・ポート | ❌ 国指定不可 |
| **Shield** | DDoS（L3/L4・量）緩和 | ❌ 国制御の機能なし |
| **API Gateway** | API公開・スロットリング等 | ❌ 地理的制限機能なし |

### SAA的な着眼点
- 「**国・地域でアクセス制限**」というキーワード → **WAF（Geo match）か CloudFront（地理的制限）**の2択がセットの王道。
- 「**Web ACL**」という語が出たら主語は**WAF**。Shield/SG/API Gateway に "Web ACL" をくっつける選択肢はほぼ誤り。
- **Security Group は IPベース、国指定は不可**。「国・地域」が条件ならSGは即除外。
- **Shield = DDoS（量・L3/L4）／ WAF = L7のルール審査（SQLi・XSS・Geo・レート制限）**。役割を混同させる出題が頻出。

## 第10問（Analytics ｜ サーバーレスDWHとSQLによる機械学習）

> [!question] 問題（再掲）
> ある小売業者は、S3 のデータを定期的にデータウェアハウスにロードし、販売予測や在庫管理に使う機械学習モデルを **SQLコマンドで**作成・トレーニングしたい。DWHには**超並列処理（MPP）と列指向ストレージ**が必要。運用の手間を減らすため**可能な限りサーバーレス**にしたい。これらを満たすソリューションはどれか。
> - A. S3 上のデータに Athena で SQL クエリを実行し、MLモデルを作成・トレーニングする。
> - B. Aurora Serverless にロードし、SageMaker AI で MLモデルを作成・トレーニングする。
> - C. Redshift（プロビジョンド）にロードし、Redshift ML で作成・トレーニングする。
> - D. Redshift Serverless にロードし、Redshift ML で作成・トレーニングする。

> [!success]- 答え：**D**
> 「DWH・MPP・列指向」＝Redshift、「SQLでML」＝Redshift ML、「可能な限りサーバーレス」＝Redshift Serverless。3つの要件キーワードがすべて Redshift Serverless に集約される。

### 要件を分解する
| 要件 | 言い換え | 効く部品 |
|---|---|---|
| 超並列処理（MPP）・列指向のDWH | 分析特化の列指向倉庫 | **Amazon Redshift** |
| SQLコマンドでMLを作成・訓練 | データを出さずSQLだけでML | **Redshift ML**（裏でSageMaker） |
| 可能な限りサーバーレス | クラスタ管理をしたくない | **Redshift Serverless** |

### たとえ話：巨大倉庫＋伝票だけでモデル＋全自動
- **Redshift** ＝ 商品を**列ごとに整然と並べ（列指向）**、**大勢が同時に棚卸し（MPP）**する分析専用の巨大倉庫＝データウェアハウス。
- **Redshift ML** ＝ 倉庫の中で**伝票（SQL）を書くだけ**で予測モデルを作ってくれる仕組み。データを外へ出さない。
- **Serverless** ＝ 倉庫の管理人（クラスタ運用）を雇わず、**使った分だけ・自動運用**。

### なぜ C ではなく D か（本命の分かれ目）
C と D は「DWH・MPP・列指向・SQLでML」をどちらも満たす。違いは**運用形態だけ**。

- **C（プロビジョンド Redshift）** … ノード数・サイズを自分で決め、**クラスタを管理**する必要がある。
- **D（Redshift Serverless）** … クラスタ管理が不要で、**使った分だけ自動スケール**。「**可能な限りサーバーレス**」という要件があるため D が上位互換。

### 各選択肢の正誤
- **A：誤り** … Athena はサーバーレスでS3にSQLを実行できるが、**「DWH／MPP／列指向が必要」という明示要件**に応えるサービスではない（DWHへロードする前提とも噛み合わない）。
- **B：誤り** … Aurora は**OLTP向けRDB**で、列指向・MPPのデータウェアハウスではない。SQLだけでML完結する構成でもない。
- **C：不適** … 要件の大半は満たすが、**プロビジョンドはクラスタ管理が必要**で「可能な限りサーバーレス」に反する。
- **D：正解** … DWH・MPP・列指向・SQLベースのML・サーバーレスをすべて満たす。

### 紛らわしい部品の整理
| サービス | 種別 | MPP/列指向DWH | SQLでML |
|---|---|---|---|
| **Redshift / Redshift Serverless** | データウェアハウス | ✅ | ✅（Redshift ML） |
| **Athena** | サーバーレスのクエリサービス | ❌（DWHではない） | △（SageMaker連携） |
| **Aurora / RDS** | OLTP向けRDB | ❌（行指向） | ❌ |

### SAA的な着眼点
- 「**データウェアハウス／超並列処理（MPP）／列指向**」→ **Redshift** ほぼ確定（Athena・Aurora・RDS は除外）。
- 「**SQLコマンドで機械学習**」→ **Redshift ML**。
- 「**可能な限りサーバーレス**」が付いたら **Redshift Serverless**（プロビジョンドより上位）。C/Dの分かれ目はここだけ。
- 頻出ひっかけ：Athena も「サーバーレスでSQL」だが、**“DWH／MPP／列指向”の語が出たら Athena ではなく Redshift**。
