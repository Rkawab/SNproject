## 第1問（Networking / CDN / LB ｜ 特定ドメインへのアウトバウンド制限（Network Firewall））

> [!question] 問題（再掲）
> プライベートサブネットのEC2でデータ分析アプリを運用。アプリは**インターネット上の外部APIのURL**に定期アクセスして気象データを取得。ネットワークセキュリティ強化のため、インターネット通信は**このAPIとEC2アプリ間のみ許可**したい。満たす方法は？
> - A. **Shield Advanced**で、API通信以外の全トラフィックをブロックするルールを作成。
> - B. **Firewall Manager**でVPC出入りの全トラフィックを検査。指定URLとIGW間の通信のみ許可するファイアウォール設定。
> - C. **Network Firewall**で、API通信のみ許可する**ドメイン名フィルタリング**ルールを作成。プライベートサブネットのアウトバウンドがNetwork Firewall経由になるようルートテーブルを修正。
> - D. EC2の**セキュリティグループ**で、APIの**URL**とのインバウンド/アウトバウンド通信のみ許可。

> [!success]- 答え：**C**
> 決め手は「**URL（ドメイン名）で許可先を絞る**」。**セキュリティグループやNACLはIPアドレス（CIDR）でしか書けず、ドメイン名を条件にできない**。ドメイン名（FQDN）でアウトバウンドを許可/遮断できるのは**AWS Network Firewall**。プライベートサブネットのアウトバウンドをルートテーブルでFirewall経由に通し、許可ドメイン以外を落とす。

あなたが選んだDが落ちる理由がこの問題の核心であり、SAAで超頻出の知識です。**「セキュリティグループにURL/ドメイン名は書けない」**——これを知っているかどうかだけの問題とも言えます。

### 最重要：SG/NACLは「IPアドレス」でしか制御できない
セキュリティグループとネットワークACLのルールに書けるのは、**IPアドレス（CIDRブロック）**、または（SGの場合）**別のセキュリティグループ**だけです。**`api.weather.example.com` のようなドメイン名（URL）を許可先に指定することはできません**。

外部APIは「URL」で指定されており、その裏のIPアドレスは**CDNやロードバランサーで複数あり、時間で変わることもあります**。だからそもそもSGに固定IPで書くのも現実的でない。**「ドメイン名で許可したい」と出たらSG/NACLは選べない**、と即断できます。

### ではドメイン名で絞れるのは何か → Network Firewall
**AWS Network Firewall**は、VPCに配置するマネージドなファイアウォールで、**ドメイン名（FQDN）フィルタリング**ができます。「`api.weather.example.com` へのアウトバウンドだけ許可、他は全部ブロック」という**ドメインベースの許可リスト**を作れる。

構成のポイントは2つ：
1. Network Firewallに「このドメインのみ許可」ルールを作る。
2. **ルートテーブルを修正**し、プライベートサブネットからのアウトバウンドが**Network Firewallを必ず経由**するようにする（検査ポイントを通す）。

### たとえ話：出口に立つ「行き先チェックの警備員」
- **SG/NACL＝入館証の“番号”しか見ない受付**。「社員番号（IPアドレス）が名簿にあれば通す」方式。**“どの会社へ行くか（ドメイン名）”では判断できない**。
- **Network Firewall＝出口に立って“行き先”を尋ねる警備員**。「気象APIさんの所へ？ ならOK。それ以外へ？ 通さない」と**訪問先（ドメイン名）で判断**する。今回ほしいのはこの警備員。
- そして**ルートテーブルの修正＝全員が必ずこの警備員のいる出口を通るよう動線を変える**こと。抜け道があると検査の意味がないので、経路をFirewall経由に固定する。

### 各選択肢をズバッと
- **A：誤り** … Shield AdvancedはDDoS（大量攻撃）からの防御サービス。**アウトバウンドを特定ドメインに絞る道具ではない**。要件と用途が全く違う。
- **B：誤り** … Firewall Managerは、複数アカウント/リソースにまたがってWAFやSG、Network Firewallのルールを**一元管理する“管理者”サービス**。それ自体がドメイン名でトラフィックを検査・許可する実体ではない。「URLとIGW間のみ許可」という記述も仕組みとして不正確。
- **C：正解** … Network Firewallのドメイン名フィルタリングで特定APIのみ許可し、ルートテーブルでアウトバウンドをFirewall経由に通す。ドメインベースのアウトバウンド制御の王道。
- **D：誤り（あなたの回答・最大の罠）** … セキュリティグループのルールは**IP（CIDR）かSGしか書けず、URL/ドメイン名を指定できない**。外部APIのIPは可変でもあり、SGでこの要件は満たせない。「URLで許可」という文言を見た瞬間にSGを消せる。

> [!tip] SAA着眼点
> - **「ドメイン名／URL／FQDNで通信を許可・遮断」→ AWS Network Firewall（またはプロキシ）**。**SG・NACLはIPアドレスのみ**で、ドメイン名フィルタは不可。この線引きは頻出。
> - **SG＝ステートフル（戻りは自動許可）・許可ルールのみ**、**NACL＝ステートレス・許可/拒否両方・サブネット単位**、どちらも**条件はIP/ポート/プロトコル**。
> - **アウトバウンドを検査ポイントに通す＝ルートテーブルでFirewall/NAT/エンドポイント経由に向ける**。「経路を強制する」発想がセットで問われる。
> - Shield＝DDoS防御、WAF＝L7（SQLi/XSS等のWebリクエスト）、Firewall Manager＝ルールの一元管理、Network Firewall＝VPCのL3/L4〜ドメインフィルタ。**役割の住み分け**を押さえる。

## 第2問（Security / IAM ｜ マルチアカウント構成とオンプレADによる認証一元化）

> [!question] 問題（再掲）
> 多国籍企業が、事業部ごとに個別AWSアカウントを使う**マルチアカウント構成**へ移行。ユーザー認証は本社の**オンプレMicrosoft AD**を使い、**全AWSアカウントを一元管理**したい。有効なアクションの組み合わせは？（2つ選択）
> - A. Organizationsで組織作成し、**認証メカニズムにDirectory Serviceを直接使用**するよう組織を設定。
> - B. 本社ADに**Cognito**を接続し、Cognitoで全AWSアカウントのユーザー認証。
> - C. **各AWSアカウントごとに個別のIAMユーザー**を作成し、ADから一時認証情報を配布。
> - D. **Organizationsですべての機能を有効**にした組織を作成し、各事業部用のAWSアカウントを追加。
> - E. **IAM Identity Center**を設定し、**AWS Directory Service経由で本社ADと連携**。

> [!success]- 答え：**D, E**
> この手の「マルチアカウント＋一元認証」は**2つの層をセットで組む**のが定石。**①器（アカウントを束ねる土台）＝Organizations**（D）。**②認証（誰がログインできるか）＝IAM Identity Center＋AD連携**（E）。DとEは片方だけでは要件を満たさず、**必ずペア**で選ぶ。

複数選択問題は「**2つで1つの構成を完成させる**」ことが多く、この問題はまさにそれ。**器（D）と認証（E）は役割が違うので競合せず、両方必要**です。あなたはE（正しい認証）は選べていて、もう片方でC（個別IAMユーザー）を選んでしまった——ここが分かれ目です。

### 2つの層で考える
| 層 | 役割 | 使うもの |
|---|---|---|
| 器（アカウント統制） | 事業部ごとのアカウントを1つの組織に束ね、一括管理 | **AWS Organizations（すべての機能）** ＝ D |
| 認証（ログインの一元化） | オンプレADの資格情報で全アカウントへSSO | **IAM Identity Center ＋ AD連携** ＝ E |

「全AWSアカウントを一元管理」には**この2階建て**が要る。Dだけだとアカウントは束ねられるが認証がバラバラ。Eだけだと認証はできてもアカウントを束ねる器がない。だから**D＋E**。

### なぜC（あなたの回答）が間違いか──一元管理の真逆
C「各AWSアカウントごとに個別のIAMユーザーを作成」は、一見ADから認証情報を配るので繋がって見えますが、**アカウントの数だけ個別ユーザーを作る**時点で破綻します。

- アカウントが10個あれば、同じ人のIAMユーザーを10個作る羽目に。**追加・削除・棚卸しが爆発的に増える**。
- これは「**一元管理**」の真逆＝**分散管理のアンチパターン**。
- ADと繋ぐなら、個別ユーザーを作らず**フェデレーション（IAM Identity Center）で一度連携すれば全アカウントにSSO**できる。これがEの発想。

つまり**Cの目的（ADで認証）はEが正しく実現する**ので、CとEは“同じ狙いの下位互換と正解”の関係。正解Eがある以上Cは選ばない。

### たとえ話：本社の社員証で全支社に入れるようにする
- **Organizations（D）＝グループ会社を1つの「持ち株会社」の下に束ねる**。各支社（事業部アカウント）が勝手バラバラでなく、本社の統制下に入る。
- **IAM Identity Center＋AD連携（E）＝本社の社員証システム（AD）を、全支社の入館ゲートに繋ぐ**。社員は**いつもの社員証（AD資格情報）**で、どの支社にも入れる（SSO）。
- **C＝支社ごとに別々の入館証を発行して配る**。支社が増えるたび全社員に新しい証を配り直し。管理が地獄で、一元管理の理念に反する。

### 各選択肢をズバッと
- **A：誤り** … Organizationsは複数アカウントを束ねる器で、**それ自体は認証メカニズムを持たない**。ADと認証を繋ぐのはIAM Identity Centerの仕事。「組織に認証を直接設定」は仕組みとして誤り。
- **B：誤り** … Cognitoは**Web/モバイルアプリのエンドユーザー**認証向け。**従業員がAWSアカウント/コンソールへSSO**する用途の主役ではない。
- **C：誤り（あなたの回答）** … 各アカウントに個別IAMユーザーを作るのは**一元管理と真逆**でスケールしない。ADと繋ぐならフェデレーション（E）にすべき。
- **D：正解** … Organizations（すべての機能）で組織を作り事業部アカウントを追加＝**マルチアカウント統制の土台**。
- **E：正解** … IAM Identity CenterをDirectory Service経由で本社ADと連携＝**オンプレADの資格情報で全アカウントへSSO**する認証の一元化。

> [!tip] SAA着眼点
> - **「複数AWSアカウントを一元管理」→ AWS Organizations（＋統制したいならSCP）**。**「オンプレAD/外部IdPの資格情報で全アカウントにSSO」→ IAM Identity Center**。この2つは**セットで問われる**定番ペア。
> - **オンプレMicrosoft ADと繋ぐ**ときは、間に**AWS Directory Service（AD Connector / AWS Managed Microsoft AD）**を挟んでIAM Identity Centerと連携。
> - **Cognito＝アプリのエンドユーザー**、**IAM Identity Center＝従業員のAWSアクセス**。この住み分けを混同しない。
> - **複数選択問題は「1つの構成を2ピースで完成」させる視点**。器（Organizations）と認証（Identity Center）のように**役割が別なら両立**、同じ役割の劣化版（個別IAMユーザー）は切る。

## 第3問（Security / IAM ｜ IPを変える大量リクエストへのWAFレートベース防御）

> [!question] 問題（再掲）
> ALBの背後で動くWebサービスに、応答遅延や障害が出るレベルの負荷が発生。調べると、**IPアドレスを変えながら送られる不審なリクエスト**が大量に来ていた。ソリューションアーキテクトの適切な対応は？
> - A. AWS WAF を使用し、**レートベースのルール**を設定する。
> - B. **ネットワークACL**にルールを追加し、特定のIPアドレスからのリクエストをブロックする。
> - C. **Amazon Inspector** を使用し、リクエストの不正な挙動を検出したら自動的にアクセスを制限する。
> - D. AWS WAF を使用し、特定の国や地域からのアクセスを制限する**地理的一致（Geo match）ルール**を設定する。

> [!success]- 答え：**A**
> この問題の決め手は「**攻撃者がIPを変えてくる**」の一言。**事前にIPを名指しできない**攻撃なので、IPを手で登録する方式（B）は全滅。**回数（レート）で自動判定して弾く**AWS WAFのレートベースルールが正解。

この問題は「どうやって弾くか」の**手段の型**を選ばせる問題です。IPが変わるので **①IP名指しはムダ（Bを消す）→ ②弾く仕事ができるサービスか（Cを消す）→ ③何を基準に弾くか＝回数か国か（DでなくA）** の順で3つ落とせます。

### 用語ミニ辞書（ここが分かれば解ける）
- **AWS WAF**（ワフ） … Webサイトの前に立つ門番。届いたHTTPリクエストを1件ずつ見て「通す／弾く」を判断する仕組み。
- **レートベースのルール** … WAFのルールの一種。**送信元IPごとに直近5分間のリクエスト数を数え**、決めたしきい値（例:5分で2000回）を**超えたIPを自動でブロック**する。回数が落ち着けばまた通す。→ 攻撃元IPを事前に知らなくても、叩きすぎているIPを勝手に締め出せるのが肝。
- **ネットワークACL** … サブネット（ネットワークの区画）の出入口に置く通行リスト。許可/拒否を**IPアドレスで手動登録**する静的な方式。
- **Amazon Inspector** … **脆弱性スキャナ**。EC2やコンテナの中に古いソフトや設定ミスが無いかを点検して**報告する**ツール。リクエストをリアルタイムに弾く役ではない。
- **地理的一致（Geo match）ルール** … WAFのルールの一種。**国・地域**を基準にアクセスを許可／ブロックする（IPの回数ではなく「どこの国から来たか」で見る）。

### 各選択肢をズバッと（不正解がなぜダメか）
- **A：正解** … レートベースルールが「IPごとの回数」でしきい値超過を自動ブロック。**IPが変わっても「今まさに大量に叩いてきたIP」を都度捕まえる**ので、IPローテーション攻撃にちょうど噛み合う。
- **B：誤り** … ネットワークACLは**ブロックするIPを人間が手作業で1つずつ登録**する方式。ところが攻撃者はIPを変えてくる＝登録しても次は別IPで来る。**イタチごっこで永久に追いつけない**。「IPを変えて」という条件と真正面から噛み合わない。
- **C：誤り（最大の罠）** … Inspectorは**脆弱性を点検して報告するだけ**のサービスで、そもそも**リクエストを弾く機能を持っていない**。「不正な挙動を検出して自動制限」という選択肢の説明自体がInspectorの実際の役割と違う（できないことをできると書いた誤答）。役割を知らないと選んでしまう。
- **D：誤り** … WAFを使う点はAと同じでも、**弾く基準が「国・地域」**なのがズレ。問題文には「特定の国から攻撃されている」とは書いていないので国で切る根拠がなく、**同じ国の中でIPを変えられれば素通り**。逆にその国の正規ユーザーまで巻き添えにする。「WAFだから」で飛びつくと引っかかる。

> [!tip] SAA着眼点
> - **「IPを変えて大量リクエスト」「DDoS的な過剰アクセス」→ AWS WAFのレートベースルール** が定番解。
> - **「攻撃元IPが変わる／可変」と書いてあったら、IPを手で指定する防御（ネットワークACL・セキュリティグループ）は全部×**。追いつけないから。
> - **Inspector＝脆弱性の点検役**。「アクセスをリアルタイムに弾く／制限する」文脈で出てきたら基本ひっかけ。**防御して弾くのはWAF（L7の不正リクエスト）／Shield（DDoS）**。
> - WAFの中でも **回数で弾く＝レートベース**、**国で弾く＝地理的一致（Geo match）**。問題文が「回数」の話か「国」の話かで選び分ける。

## 第4問（Compute ｜ 単一EC2のハードウェア障害からの迅速復旧（自動復旧））

> [!question] 問題（再掲）
> スタートアップが重要アプリを開発。**予算制約で単一のEC2インスタンス**で運用予定だが、**ハードウェア障害によるサービス停止は最小限**にしたい。インスタンスの障害から**迅速に回復する最も効率的**な方法は？
> - A. **EventBridge**からインスタンス復旧を行う**Lambda関数**を呼び出して復旧する。
> - B. **定期的なバックアップ**を作成し、障害時に**スナップショットから復元**する。
> - C. **ELB**でインスタンスの健全性を監視し、障害が発生したインスタンスを**交換**する。
> - D. **CloudWatchアラームとインスタンス復旧アクション**を設定し、ヘルスチェック異常時に**自動的に復旧**する。

> [!success]- 答え：**D**
> 決め手は「**単一インスタンス**」「**ハードウェア障害**」「**最も効率的**」の3つ。単一のまま、ハード障害を検知して**同じインスタンスを別の正常なホストで自動的に立て直す**のがEC2の自動復旧＝D。データもIPも引き継ぎ、追加コストも自作コストもなく最速。

この問題は「復旧の手段」を選ばせる問題で、**「単一インスタンス」という条件で候補が一気に絞れます**。交換・置き換え（C）は複数台前提なので単一とは噛み合わず真っ先に消え、残りは「自作(A)」「バックアップ復元(B)」「純正の自動復旧(D)」の効率比べになります。

### 用語ミニ辞書（ここが分かれば解ける）
- **EC2の自動復旧（Auto Recovery）** … CloudWatchアラームに「復旧アクション」を仕込む純正機能。インスタンスの載っている**物理ホストが壊れた**とき、**同じインスタンスを別の正常なホスト上で自動再起動**する。**インスタンスID・プライベートIP・Elastic IP・アタッチ済みEBS（データ）はそのまま引き継ぐ**。
- **システムステータスチェック** … AWS側の土台（ホスト・電源・ネットワーク）が正常かの点検。これが失敗＝ハードウェア障害。自動復旧はこれをトリガーにする。（対になる「インスタンスステータスチェック」はOS内部の問題を見る）
- **ELB（ロードバランサー）** … 複数インスタンスに負荷を振り分ける係。「壊れた1台を外して別の1台に流す＝交換」には**裏に複数台＋Auto Scalingがある前提**。
- **スナップショット復元** … EBSのバックアップから**新しいボリューム/インスタンスを作り直す**復旧。作成に時間がかかり、前回バックアップ以降のデータは失われる。

### 各選択肢をズバッと（不正解がなぜダメか）
- **D：正解** … システムステータスチェック失敗（＝ハード障害）を検知し、同じインスタンスを別ホストで自動復旧。**ID・IP・EBSを維持**するのでアプリ設定もデータもそのまま、単一構成のまま最速・低コストで戻せる。3条件すべてに噛み合う。
- **C：誤り（あなたの回答・最大の罠）** … ELBの「交換」は、**裏に予備インスタンスやAuto Scalingグループ（＝複数台）がある前提**の考え方。本問は**予算制約で単一インスタンス**なので、そもそも交換する相手がいない。さらに「交換」は新しいインスタンスへの置き換え＝**元のローカルデータやインスタンスIDは失われる**。「単一インスタンス」の一言でCは消せる。
- **A：誤り** … EventBridge＋Lambdaで復旧処理を自作する案。**動きはするが、AWSが標準で用意している自動復旧をわざわざLambdaで作り込む**ぶん、開発・保守の手間（運用オーバーヘッド）が増える。「最も効率的」という条件で純正のDに負ける典型的な"車輪の再発明"。
- **B：誤り** … 定期バックアップ＋スナップショット復元は、復元＝**新規作成に時間がかかり**、しかも**前回バックアップ以降のデータは消える**。バックアップは大事だが「**迅速な復旧手段**」ではない。「迅速に回復」に反する。

> [!tip] SAA着眼点
> - **「単一インスタンス」＋「ハードウェア障害から素早く復旧」→ EC2自動復旧（CloudWatchアラームの復旧アクション）** が定番解。
> - **「単一インスタンス」と書いてあったら、ELB／Auto Scalingでの"交換・増減"は基本×**（複数台前提だから）。裏返しに「複数台で可用性を確保」ならELB＋ASGが正解になる。
> - **自動復旧＝同じインスタンスを別ホストで復活（ID・IP・EBS維持）**。**Auto Scalingの置き換え＝新しいインスタンスに交換（中身は引き継がない）**。この違いが選択肢の切り分け。
> - **スナップショット復元は「バックアップ」であって「迅速復旧」ではない**、というひっかけに注意。
> - Lambda等での自作は「動くが最適でない（運用オーバーヘッド大）」誤答の定番パターン。純正のマネージド機能があるなら基本そちらが正解。

## 第5問（Management / Monitoring ｜ ハイブリッド環境のパッチ適用と構成情報収集（SSM））

> [!question] 問題（再掲）
> EC2 と**オンプレミスの Linux サーバーを併用**して業務システムを運用。セキュリティ部門から、**全サーバーの OS の種類・バージョン・インストール済みパッケージ等の構成情報を自動収集・可視化**することと、**定期的なパッチ適用の自動化**が求められている。**最小限の管理労力**で実現し、コンプライアンス基準を満たしたい。最も適したソリューションは？
> - A. **Systems Manager Patch Manager とインベントリ機能**で、全 EC2 およびオンプレサーバーを対象にパッチ適用と構成情報の自動収集を行う。
> - B. **cron で定期実行されるシェルスクリプト**で各サーバーに SSH 接続し、夜間バッチで構成情報の収集とパッチ適用を実行する。
> - C. **Security Hub** でサーバーのインベントリ情報とパッチ適用状況を可視化し、未適用パッチがあればアップデートを定期実行する。
> - D. 既存サーバーを **Elastic Beanstalk 環境として登録**し、プラットフォーム更新機能でパッチ適用とインベントリ収集を行う。

> [!success]- 答え：**A**
> 決め手は「**EC2＋オンプレの混在（ハイブリッド）**」「**構成情報の収集・可視化**」「**パッチ自動化**」「**最小限の管理労力**」の4条件が全部そろっていること。この4つを1つのマネージドサービスでまとめて満たせるのが **AWS Systems Manager**（インベントリ＋Patch Manager）で、これ以外の選択肢はどれか1つ以上の条件と衝突する。

この問題は「Systems Manager が何屋さんか」を知っていれば一撃で、知らないと C（Security Hub）や B（自作スクリプト）に流れる作りです。要件を分解すると、**①構成情報を集めて見える化 → SSM インベントリ**、**②パッチを定期自動適用 → SSM Patch Manager**、**③オンプレも対象 → SSM のハイブリッドアクティベーション**、**④管理労力は最小 → マネージドサービスに任せる**——4つの矢印がすべて Systems Manager を指します。

### 用語ミニ辞書
- **AWS Systems Manager（SSM）** … サーバー群を AWS 側から遠隔でまとめて運用管理する「管理室」的サービス。サーバーに **SSM Agent**（常駐の小さなソフト）を入れると、そのサーバーが管理室から操作できる「**マネージドノード**」になる。
- **ハイブリッドアクティベーション** … オンプレミス（自社機械室など AWS の外）のサーバーにも SSM Agent を入れて、**EC2 と同じ画面・同じ仕組みで管理対象に登録**する仕組み。「EC2 もオンプレも一元管理」の鍵。
- **インベントリ（Inventory）** … マネージドノードから **OS の種類・バージョン、インストール済みソフト、ネットワーク設定などの構成情報を自動で定期収集**し、コンソールで一覧・検索できるようにする SSM の機能。まさに「棚卸し表」。
- **Patch Manager** … SSM の機能の1つ。**パッチベースライン**（どの種類のパッチをいつまでに当てるかの基準）と**メンテナンスウィンドウ**（適用する時間帯）を決めておくと、**スキャン→適用→準拠状況（当たってる/当たってない）のレポート**まで自動でやってくれる。
- **コンプライアンス** … ここでは「決めた基準（パッチ適用済みであること等）を満たしているか」の準拠状態のこと。Patch Manager はノードごとの準拠/非準拠をダッシュボードで見せてくれる。
- **Security Hub** … 各セキュリティサービス（GuardDuty・Inspector・Config 等）が見つけた**検出結果（findings）を1か所に集約して表示する掲示板**。自分で情報を収集したり、パッチを当てたりする「実行役」ではない。
- **Elastic Beanstalk** … アプリのコードを渡すと、実行環境（EC2・LB・Auto Scaling）を**新規に組み立ててくれる**デプロイサービス。既にあるサーバーを後から「登録」して管理する仕組みではない。

### 各選択肢をズバッと
- **A：正解** … SSM Agent＋ハイブリッドアクティベーションで **EC2 もオンプレも同じマネージドノード**として扱える。**インベントリが構成情報（OS・パッケージ等）を自動収集・可視化**し、**Patch Manager がベースライン＋メンテナンスウィンドウで定期パッチを自動適用し、準拠状況までレポート**。すべてマネージドなので自前の作り込みがなく「最小限の管理労力」に合致。4条件を1サービスで完答できる。
- **B：誤り** … cron（Linux の定期実行の仕組み）＋SSH の自作スクリプトでも**技術的には動く**。しかしスクリプトの保守、全サーバーへの SSH 鍵の配布・管理、失敗時のリトライや通知、収集結果の可視化画面まで**全部自前で作って面倒を見る**ことになる。問題文の「**最小限の管理労力**」と真正面から衝突する、「動くが最適でない」典型。SAA では自作スクリプト運用はほぼ常に「運用負荷が高い」側の誤答。
- **C：誤り（最大の罠）** … 「可視化」「コンプライアンス」という単語の響きが Security Hub と重なるので釣られやすい。しかし Security Hub は**他のサービスが検出した結果を集約して見せる掲示板**であって、**サーバーからインベントリ情報を収集する機能も、パッチを適用（アップデートを実行）する機能も持っていない**。「未適用パッチがあればアップデートを定期実行する」という記述自体が、Security Hub にできないことをできると書いた誤り。集める・当てるの実働部隊はあくまで SSM。
- **D：誤り** … Elastic Beanstalk は**新しいアプリを載せる環境を丸ごと作る**サービスで、**既存の EC2 やオンプレサーバーを後から「Beanstalk 環境として登録」する仕組みは存在しない**。プラットフォーム更新機能も「Beanstalk が自分で作った環境」の OS・ミドルウェアを更新するもの。オンプレサーバーが混ざっている時点で前提から成立しない。

> [!tip] SAA着眼点
> - **「EC2＋オンプレを一元管理」「ハイブリッド環境の運用管理」→ Systems Manager** が定番解。オンプレ側は SSM Agent＋ハイブリッドアクティベーションで取り込む。
> - SSM は機能名で切り分ける：**構成情報を集める＝インベントリ／パッチを当てる＝Patch Manager／コマンドを一斉実行＝Run Command／サーバーに入る＝Session Manager**。問題文の動詞と対応させる。
> - **Security Hub は「集約・表示」専門**。「収集する」「適用する」「修復する」と書かれていたらひっかけを疑う（実行役は SSM や各サービス）。
> - **「最小限の管理労力／運用負荷を減らす」とあったら、cron・SSH・自作スクリプト系は基本×**。マネージドサービスの純正機能を選ぶ。

## 第6問（Security / IAM ｜ ブルー/グリーン切替中のDDoS検知（Shield Advanced保護グループ））

> [!question] 問題（再掲）
> ALB背後の複数EC2で本番アプリを稼働中。ALBは**Shield Advancedで保護済み**、Route 53で独自ドメインからルーティング。新バージョン導入で、現行環境と新環境を並行稼働させつつトラフィックを新ALBへ切り替える**ブルー/グリーンデプロイ**を実施する。**切り替え中もアプリケーション全体として一貫したトラフィックベースラインに基づいてDDoS検知・保護**を行いたい。最も適切なソリューションは？
> - A. 現行ALBと新ALBを**同一の保護グループ**にまとめ、集計タイプ **Sum（合計）** を設定。
> - B. 現行ALBと新ALBを**それぞれ別の保護グループ**に登録し、集計タイプ Sum（合計）を設定。
> - C. 現行ALBと新ALBを**同一の保護グループ**にまとめ、集計タイプ **Mean（平均）** を設定。
> - D. **CloudFrontを新規作成**して新ALBをオリジンに登録し、CloudFrontと新ALBを同一の保護グループに含めて集計タイプ **Max（最大）** を設定。

> [!success]- 答え：**A**
> 決め手は「**切り替え中（トラフィックが旧→新へ移る）**」＋「**アプリ全体として一貫したベースライン**」。旧が減り新が増えても**2本の合計＝アプリ全体は一定**なので、**同一保護グループ＋Sum（合計）**でグループ全体を1本のベースラインとして見るのが正解。

この問題は「保護グループにまとめるか分けるか」と「集計タイプ（Sum/Mean/Max）の選び分け」の2段構え。まとめる判断（A/C/D）までは合っていても、**集計タイプで落とす**作りになっています。あなたはグループ化は正解、Mean（平均）を選んだ点だけがズレでした。

### 用語ミニ辞書（ここが分かれば解ける）
- **DDoS攻撃** … 大量のリクエストを浴びせてサービスを潰す攻撃。「普段よりトラフィックが異常に多い」ことを手がかりに検知する。
- **Shield Advanced** … AWSのDDoS防御サービス（上位版）。**平常時のトラフィック量（ベースライン）を学習しておき、そこからの逸脱で攻撃を判定**する。やっていることは「**1本の数字のグラフを見張る**」こと。
- **トラフィックベースライン** … 「このアプリは普段これくらいのアクセス量」という**基準線**。検知はこの基準線との比較で行われる。
- **保護グループ（protection group）** … Shield Advancedの機能。複数リソース（ALB等）を**1つのまとまりとして扱い、グループ全体でベースラインを持たせる**仕組み。
- **集計タイプ** … ALBが2本あると数字も2つ出るが、Shieldが見張れるグラフは1本。**「複数の数字をどう1つに合体させるか」の計算方法**が集計タイプ。ただの算数で、旧400・新600のとき：**Sum＝足し算（400+600=1,000）**／**Mean＝平均（÷台数で500）**／**Max＝大きい方だけ（600）**。
- **ブルー/グリーンデプロイ** … 現行環境（ブルー）と新環境（グリーン）を並行稼働させ、トラフィックを**徐々に新側へ切り替える**リリース手法。切り替え中は旧ALBが減り新ALBが増える＝**1本ずつ見ると両方とも「異常な変動」に見える**のがこの問題の罠。

### 数字で見る：切り替え中に各集計タイプが指す値

アプリ全体の平常トラフィックを毎秒1,000とすると：

| 時点 | 旧ALB | 新ALB | **Sum** | **Mean** | **Max** |
|---|---|---|---|---|---|
| 切替前 | 1,000 | 0 | **1,000** | 500 | 1,000 |
| 切替中 | 400 | 600 | **1,000** | 500 | 600 |
| 切替後 | 0 | 1,000 | **1,000** | 500 | 1,000 |

**Sumだけが常に「1,000」＝アプリの実際の総量を指し続ける**。これが「アプリケーション全体として一貫したトラフィックベースライン」の正体。学習済みの基準線（このアプリは平常1,000）とグループの数字がそのまま繋がる。

### 各選択肢をズバッと（不正解がなぜダメか）
- **A：正解** … 同一グループ＋**Sum**。切り替え中に旧が減って新が増えても**合計は一定**＝ベースラインが乱れず誤検知しない。本物のDDoS（合計の異常増）はちゃんと検知できる。「アプリ全体として一貫」を最も素直に実現する。
- **C：誤り（あなたの回答）** … グループ化までは正しいが、Mean（平均）がダメな理由は3つ。①**答えている質問が違う**：Sumは「アプリ**全体で**どれだけ浴びているか」、Meanは「**1台あたり平均**どれだけか」の数字で、要件の「アプリ全体として」に対応するのはSum。②**平均が意味を持つのは全員がほぼ均等に分担しているときだけ**（10台が100ずつなら「平均100」は各台の実態を表す）。ブルー/グリーンは配分が**100:0→40:60→0:100と偏り続ける**ので、「平均500」はどの瞬間のどのALBの実態も表さない数字になる。③**メンバーの出入りで数字が勝手に跳ぶ**：実運用では新ALB追加→切替→旧ALB削除と台数が1→2→1と変わる。Meanは÷台数なので、トラフィックは何も変わっていないのに**追加の瞬間に半減・削除の瞬間に倍増**し、基準線がガタガタになる。Sumは足し算なので台数が変わっても実トラフィックのまま。
- **B：誤り** … **別々の保護グループ**では、それぞれが自分のベースラインを持つだけで「アプリ全体で1本」にならない。切り替え中は旧グループ（激減）も新グループ（激増）も**個別に異常と判定されうる**。グループにまとめる意味がなくなる、グループ化の趣旨と真逆の選択肢。
- **D：誤り** … **CloudFrontの新設**は要件にないアーキテクチャ変更で、ブルー/グリーンの保護という論点に関係ない工事を持ち込んでいる。さらに**Max（最大）**は「一番トラフィックが多いメンバーの値」でグループを代表させる方式。上の表のとおり切替中は600しか見えず合計1,000を見失う。Maxが活きるのは**同じリクエストが多段を通る**構成（CloudFront→ALBは1アクセスが両方でカウントされ、足すと二重計上になるので「大きい方＝外から来た本当の量」を採る）で、横並びの2本のALBには合わない。二重にズレた過剰設計。

> [!tip] SAA着眼点
> - **「ブルー/グリーン切り替え中もDDoS検知を一貫させたい」→ Shield Advanced 保護グループ＋Sum（合計）** が定番解。
> - 集計タイプの選び分け：**Sum＝全体で1つのまとまりとして動く（ブルー/グリーン・フェイルオーバー）**／**Mean＝各リソースが均等に分担**／**Max＝1つが全体を代表（CloudFront＋オリジンなどトラフィックが重複する構成）**。
> - 「切り替え・移行中に個別リソースのトラフィックが大きく揺れる」問題文では、**個別に見る案（別グループ）は誤検知の元**として切る。
> - 要件にないリソース新設（今回のCloudFront）は「動くかもしれないが過剰」の定番誤答パターン。

## 第7問（Cost Optimization ｜ 大量データ配布の転送コスト最小化（既存Direct Connect活用））

> [!question] 問題（再掲）
> EC2で稼働する勤怠管理アプリ。既存の**AWS Direct Connect回線**あり。毎月、全社員分の勤怠データを作り、**暗号化して社員に配布**する。1件平均10MB、月合計**約500GB以上**。**データ転送コストを削減**する最適なソリューションは？
> - A. **Direct Connect経由**でオンプレのファイルサーバーへアップロード。社員はそこからダウンロード。
> - B. 社員が**インターネット経由でEC2アプリにHTTPS接続**してダウンロード。
> - C. **S3バケットにサーバー側暗号化でアップロード**。社員はインターネット経由でS3からダウンロード。
> - D. **SFTPでNATゲートウェイ経由**でオンプレのファイルサーバーへアップロード。社員はそこからダウンロード。

> [!success]- 答え：**A**
> **要するに**「毎月500GB超を社員に配る、一番安い配り方は？」という純粋なコスト問題。決め手は**データ転送アウトの単価**で、**Direct Connect経由（約0.041 USD/GB）はインターネット経由（約0.114 USD/GB）より安い**。同社は既にDirect Connectを契約済みなので、それを使ってオンプレのファイルサーバーへ配るのが最安。

> [!warning] ここが直感に反する（一度誤答した所）
> 「大量配布ならS3/インターネットが最安」と思いがちだが**逆**。AWSの料金では **Direct Connectのデータ転送アウト（約0.041/GB）＜ インターネットへのアウト（約0.114/GB）**。だから「**既存のDirect Connectがある**」は撒き餌ではなく、**答えに直結するヒント**だった。

### 用語ミニ辞書（ここが分かれば解ける）
- **データ転送アウト（egress）** … AWSは**外へ出ていく下りデータに課金**（受信は原則無料）。同じ外向きでも**出口によって単価が違う**のが肝。詳しくは [[0411_データ転送料金]]。
- **インターネットへのアウト** … 約 **0.114 USD/GB**。S3からのインターネット配信も、EC2からの配信もこの単価。
- **Direct Connect経由のアウト** … 約 **0.041 USD/GB**（インバウンドは無料）。**インターネットより安い**。専用線なので大容量・安定でもある。
- **NATゲートウェイ** … 時間料金＋**通したデータ量（GB）課金（約0.062/GB）**が上乗せ。大量データを通すと高くつく。

### 単価で並べると（安い順）
Direct Connect経由（0.041）＜ NAT（0.062）＜ インターネット／S3配信（0.114）。同じ500GBを配るなら、DC経由が一番安い。

### 各選択肢をズバッと（不正解がなぜダメか）
- **A：正解** … 既存のDirect Connectでオンプレのファイルサーバーへ配る。**DCのアウト単価はインターネットより安く**、追加契約も不要。大量配布の転送コストを最小化できる。
- **B：誤り** … **EC2からインターネットへの下り（約0.114/GB）は割高**。DC経由（0.041）より高い。EC2が500GB/月の配信を全部捌く負荷面でも不利。
- **C：誤り（私が最初に誤って正解とした選択肢）** … S3に置いても、社員が**S3からインターネット経由でダウンロードする分はインターネット料金（約0.114/GB）**。DC経由（0.041）より高い。SSEで暗号化はできるが「転送コスト最小」ではDCに負ける。**「大量ならS3が最安」という思い込みの罠**。
- **D：誤り** … **NATゲートウェイ**は処理データ量（GB）課金（約0.062/GB）が上乗せされ、500GB超では割高。既にDirect Connectがあるのに、わざわざNATを挟む必然性もない。

> [!tip] SAA着眼点
> - **「既にDirect Connectがある × 大量のegress（配布・転送）」→ Direct Connect経由が最安**。DCのアウト単価はインターネットより安い、が核心の知識。
> - **「既存のDirect Connectがある」はヒント**（撒き餌ではない）。コスト削減問題で既存インフラが提示されたら、まずそれを活かせないか考える。
> - **NATゲートウェイはGB課金**。大量トラフィックで出てきたら「高い」と反応。
> - 逆に「DCが無い／小規模」なら、インターネット/S3配信やCloudFrontが素直な答えになることもある。**前提（既存DCの有無・データ量）で最適解が変わる**。

## 第8問（Management / Monitoring ｜ 承認済み環境のセルフサービス払い出し（Service Catalog））

> [!question] 問題（再掲）
> 複数アプリ・複数開発グループを持つ会社。システム管理者は、**構築済みのAWS環境と同一の環境を、各開発者がセルフサービスで開発・テスト用に構築**できるようにしたい。かつ、**共有するAWS環境のバージョンや使用履歴を一元管理**する必要がある。使うべきAWSの機能は？
> - A. **AWS Service Catalog** の製品
> - B. **AWS CloudFormation** のテンプレート
> - C. **AWS CloudTrail** の証跡
> - D. **Amazon CloudWatch** のメトリクス

> [!success]- 答え：**A**
> **要するに**「管理者が承認した"環境の型"を、開発者が自分で払い出せて、かつ誰がどのバージョンを使ったか管理者が握れるようにしたい」という問題。この「**セルフサービスで払い出す**」＋「**バージョン・使用履歴を一元管理**」の2語が Service Catalog の役割そのもの。

> [!note] 出題頻度は低め（キーワード判定だけでOK）
> Service Catalog はSAAの管理・ガバナンス領域で**たまに出る**程度（0〜1問）。細かい設定は不要で、下の「キーワード→Service Catalog」の反射だけ作れば十分。

### 用語ミニ辞書（これだけ）
- **CloudFormation テンプレート** … AWS環境を**コードで組み立てる設計図**そのもの。「この構成を作れ」という中身。
- **Service Catalog** … 管理者が承認した構成を**「製品」としてカタログに並べ、開発者がボタン一つで払い出せる**ようにする仕組み。中身は CloudFormation テンプレートだが、その上に**「誰に使わせるか・どのバージョンか・誰がいつ使ったか」の管理（ガバナンス）**の層をかぶせる。
- つまり **Service Catalog は CloudFormation の"上に乗る"**。設計図（CloudFormation）を、社員食堂のメニュー表（Service Catalog）に載せて配膳できるようにするイメージ。

### 各選択肢をズバッと（不正解がなぜダメか）
- **A：正解** … 承認済み環境をカタログ化 → 開発者がセルフサービスで構築 → 管理者がバージョン・利用履歴を一元管理。要件の2語に完全一致。
- **B：誤り（あなたの回答・最大の罠）** … CloudFormation テンプレートは環境を作る設計図としては正しいが、**それ単体には「開発者向けのセルフサービス窓口」も「バージョン・使用履歴の一元管理」も無い**。要件の後半2つを満たせない。**Service Catalog の"部品"であって、要件が求めた管理層が足りない**。
- **C：誤り** … CloudTrail の証跡は「誰がどのAPIを叩いたか」の**監査ログ**。環境の払い出し機能ではない。用途違い。
- **D：誤り** … CloudWatch のメトリクスはリソースの**監視・数値計測**。払い出しやバージョン管理とは無関係。

> [!tip] SAA着眼点
> - **「承認済み構成を、利用者にセルフサービスで払い出す」「バージョン/権限/使用履歴のガバナンス」→ Service Catalog**。
> - **CloudFormation との違い**：CloudFormation＝作る設計図そのもの。Service Catalog＝それをカタログ化して**統制付きで配る**上位の仕組み。「セルフサービス」「一元管理」の語が出たら Service Catalog を疑う。

## 第9問（Serverless ｜ 高頻度・短時間ワークフローのコスト最適化（Step Functions Express））

> [!question] 問題（再掲）
> 数十万台のIoTデバイスを管理するプラットフォーム。センサー値・イベントを大量送信し、**ピーク時1秒に10万回以上**の呼び出し。受信データを**ワークフローとして自動化**、**実行時間は5分以内**。大量リクエストの**実行コストが問題**で、できるだけコストを抑えて同時実行数をさばきたい。**高いイベントレート・大規模拡張・コスト効率重視**の場合、最適な選択肢は？
> - A. **標準ワークフロー**でStep Functionsを使用。
> - B. **標準とExpressを組み合わせて**Step Functionsを使用。
> - C. **Expressワークフロー**でStep Functionsを使用。
> - D. **予約済み同時実行**を設定した複数のLambda関数を使用。

> [!success]- 答え：**C**
> **要するに**「大量（1秒10万回超）・短時間（5分以内）のワークフローを、できるだけ安くさばきたい」という問題。決め手はStep Functionsの2つの型（標準 vs Express）のどちらが"大量・短時間・安い"か。

### 用語ミニ辞書（ここが分かれば解ける）
Step Functions（複数のAWS処理を順番につなぐ"ワークフロー"サービス）には2つの型があり、**課金の仕組みが根本的に違う**のが肝。

| | 標準（Standard） | **Express** |
|---|---|---|
| 最大実行時間 | 1年 | **5分**まで |
| 課金 | **状態遷移1回ごと**に課金 | **実行回数＋時間＋メモリ**で課金 |
| 大量イベント時のコスト | **高い**（遷移が積み上がる） | **安い** |
| 向く用途 | 長時間・厳密な監査が要る処理 | **高頻度・短時間・大量**の処理 |

問題文の「1秒10万回超」「5分以内」「コスト効率重視」は、Expressの得意分野にそのまま一致する。

### 各選択肢をズバッと（不正解がなぜダメか）
- **C：正解** … Expressは高頻度・短時間・大量向けで、**状態遷移ごと課金でないため大量イベントでも安い**。5分以内という条件もExpressの上限（5分）にぴったり。3条件に完全一致。
- **B：誤り（あなたの回答）** … 標準の中にExpressを入れ子にする構成は技術的には可能だが、**標準ぶんの状態遷移課金が乗る**ので「コスト効率最優先」ではExpress単体に負ける。要件はシンプルに「安く大量に」なので、**わざわざ高い標準を混ぜる過剰構成**。
- **A：誤り** … 標準ワークフローは**状態遷移1回ごとに課金**。1秒10万回のような高頻度では遷移が膨大になり**コストが跳ね上がる**。「コストを抑えたい」と正反対。標準は長時間・監査重視向け。
- **D：誤り** … Lambdaの予約済み同時実行は**同時実行の枠を確保・制限する設定**で、**ワークフローの自動化（オーケストレーション）ではない**。要件は「ワークフローとして自動化」が前提でStep Functionsの型を選ばせており、Lambda単体に置き換える話ではない。予約はコスト削減手段でもない。

> [!tip] SAA着眼点
> - **「高頻度・大量イベント × 5分以内 × コスト重視」→ Step Functions の Express ワークフロー**。
> - **標準＝状態遷移ごと課金（長時間・厳密／exactly-once）／Express＝実行ベース課金（短時間・大量・安い）**。「1秒に何万回」「イベント処理」「コスト効率」の語が出たらExpress。
> - **Expressの上限は5分**。問題文の「5分以内」は"Expressで足りる"というヒント。逆に「数時間かかる」「厳密に1回だけ実行」なら標準。

## 第10問（Database ｜ 生成AI向けベクトル検索をコスト効率よく構築（RDS PostgreSQL＋pgvector））

> [!question] 問題（再掲）
> 生成AIを活用した検索機能をWebアプリに統合したい。ユーザーの問い合わせ内容と過去の**FAQデータをベクトル形式で比較**し、最も類似した回答を検索する。**ベクトル検索をサポート**し、**コスト効率の高い方法**で構築できるデータベースソリューションを探している。最もコスト効率よく満たすのは？
> - A. **ElastiCache** でFAQデータをインメモリ管理してベクトル検索を実装。
> - B. **OpenSearch Service** でFAQデータのベクトルインデックスを管理するクラスターを構築。
> - C. **RDS for PostgreSQL で pgvector 拡張を有効化**し、FAQベクトルデータをベクトルテーブルに保存して検索。
> - D. **S3** にベクトルファイルを保存し、**Athena** で検索クエリを実行。

> [!success]- 答え：**C**
> 決め手は「**ベクトル検索できること**」と「**最もコスト効率よく**」の両立。**RDS for PostgreSQL は `pgvector` という拡張機能を1つ有効化するだけでベクトル検索が使える**——つまり**既に慣れ親しんだマネージドRDBに機能を足すだけ**で、専用のベクトル検索基盤を新しく立てずに済みます。小〜中規模のFAQ検索ではこれが一番安い。他は「できるが高い（A・B）」か「そもそもベクトル検索できない（D）」で脱落します。

この問題は「ベクトル検索＝専用の高機能サービスが要る」と思い込むと、B（OpenSearch）に引っ張られる作りです。ポイントは**要件が"最もコスト効率よく"であること**。高機能な専用基盤ほど固定費（常時稼働クラスタ代）がかさむので、"安く"が最優先ならむしろ既存RDBの拡張で足りる、と発想を切り替えます。

### 用語ミニ辞書（ここが分かれば解ける）
- **ベクトル（埋め込み／エンベディング）** … 文章の"意味"を数百〜数千個の数値の並びに変換したもの。意味が近い文章どうしは、この数値の並びも近い場所に来る。生成AI検索の土台。
- **ベクトル検索（類似度検索・近傍検索）** … 質問文をベクトルに変換し、**保存済みベクトルの中から数値的に一番近いもの＝意味が最も似た回答**を探す仕組み。キーワード一致ではなく「意味が近い」で探すのが従来検索との違い。
- **pgvector** … PostgreSQL（オープンソースの定番RDB）に**ベクトル型の列と類似度検索を追加する拡張機能**。`CREATE EXTENSION vector;` で有効化するだけで、普通のテーブルにベクトルを保存して近傍検索できる。RDS for PostgreSQL / Aurora PostgreSQL で使える。
- **RDS** … AWSがOS・パッチ・バックアップまで面倒を見てくれるマネージドなRDB（リレーショナルDB）サービス。PostgreSQL や MySQL 等を選べる。
- **ElastiCache** … Redis / Valkey 等をマネージドで動かす**インメモリ**（全データをメモリに載せる）のキャッシュ／高速データストア。速い代わりにメモリ容量ぶんの費用がかかる。
- **OpenSearch Service** … 全文検索・ログ分析・ベクトル検索もできる検索エンジンのマネージドサービス。高機能だが**常時稼働の専用クラスター**を持つ必要がありコストは高め。

### 各選択肢をズバッと（不正解がなぜダメか）
- **C：正解** … 既存のRDS PostgreSQLに**pgvector拡張を足すだけ**でベクトル類似検索を実装できる。新しい基盤を増やさないので、FAQ規模なら**最もコスト効率が良い**。「ベクトル検索できる」「安い」を両方満たす唯一の選択肢。
- **B：誤り（最大の罠）** … OpenSearch Serviceも確かにベクトル検索に対応しており「正しそう」に見える。しかし**常時稼働の専用クラスターを立てる必要があり、料金が高い**。大規模データ・低レイテンシ・高度な検索が要るとき向けで、**FAQ程度の規模には過剰（オーバースペック）**。要件が「最もコスト効率よく」なので、機能は足りていてもコストで負ける。「ベクトル検索といえばOpenSearch」で飛びつくと外す。
- **A：誤り** … ElastiCacheは**全FAQをメモリに常駐**させる方式。データ量が増えるほど**メモリ課金がそのまま費用に跳ね返る**うえ、そもそも永続的な検索データストアというより高速キャッシュ用途。恒久的なベクトル検索基盤としてはコスト効率が悪い。
- **D：誤り** … S3（ファイル置き場）＋Athena（S3上のファイルにSQLを投げるクエリサービス）は、**そもそもベクトルの類似度検索＝近傍検索を行う仕組みを持っていない**。Athenaはスキャンした分だけ課金されるSQLクエリで、「意味が近いベクトルを探す」計算はできない。コスト以前に**機能要件を満たせない**用途違い。

> [!tip] SAA着眼点
> - **「ベクトル検索」＋「コスト効率よく」＋「既にRDB/PostgreSQLを使っている雰囲気」→ RDS/Aurora PostgreSQL の pgvector**。既存DBに拡張を足すのが最安の定番。
> - **「大規模・低レイテンシ・高度な検索が必要」ならOpenSearch Service**。逆に「小規模・安く」なら専用クラスタは過剰、とコスト観点で切り分ける。
> - **ElastiCache＝インメモリの高速キャッシュ**。「永続的に大量データを保持」用途で出たら、メモリ課金でコスト面の誤答になりやすい。
> - **S3＋Athenaはスキャン型のSQLクエリで、近傍検索（類似度検索）はできない**。ベクトル検索の文脈で出たら基本ひっかけ。
