## 第1問（Security / IAM ｜ 組織全体の未使用IAM権限検出（IAM Access Analyzer））

> [!question] 問題（再掲）
> OrganizationsとAWS Configで複数アカウントを管理している企業で、IAMユーザーやロールに付与された権限のうち、一度も使われていない権限が大量にあると判明した。組織全体で未使用権限を特定し、最小権限の原則を徹底したい。最も運用効率が高いソリューションは？
> - A. Security Hubを使用し、統合セキュリティダッシュボードでIAM関連のセキュリティ検出結果を確認する。
> - B. IAM Access Analyzerを組織レベルで有効化し、全アカウントの権限を確認する。
> - C. GuardDutyを組織全体で有効化し、セキュリティ脅威として未使用権限を特定する。
> - D. CloudTrailの証跡情報を使用して、通常使用されていない権限を自動的に特定する。

> [!success]- 答え：**B**
> 決め手は **未使用のIAM権限** と **組織全体**。IAM Access Analyzerの未使用アクセス分析を組織スコープで使うと、Organizations配下のIAMユーザー/ロールに付与された未使用アクセスをfindingとして確認できる。

この問題は「セキュリティっぽいサービス」が並んでいるので迷いやすいですが、役割を分けると一気に楽になります。

```text
未使用の権限を探したい
  -> IAM Access Analyzer

不審な攻撃や侵害の兆候を探したい
  -> GuardDuty

複数サービスの検出結果を集めたい
  -> Security Hub

誰が何をしたかの操作ログを残したい
  -> CloudTrail
```

今回の悩みは「攻撃されているか」ではありません。「過去に一度も使われていない権限が大量にある。最小権限にしたい」です。つまり、必要なのは脅威検知ではなく、IAM権限の棚卸しです。

IAM Access Analyzerには **unused access analyzer** があり、AWS Organizationsやアカウント内の未使用アクセスを継続的に監視してfindingを生成します。対象には、未使用ロール、IAMユーザーの未使用アクセスキー/パスワード、アクティブなユーザーやロールに付与されている未使用サービス・アクションが含まれます。

### たとえ話：社員証の入室権限を棚卸しする

会社の社員証に、会議室・倉庫・サーバールーム・金庫室など、たくさんの入室権限が付いているとします。

監査で「この人、金庫室の権限を持っているけど一度も入ったことがない」「このチーム、倉庫の権限を持っているけど業務で使っていない」と分かりました。

このとき必要なのは、警備員が不審者を探すことではありません。社員証の権限表と利用履歴を見て、**使っていない鍵を外すこと**です。

- IAM Access Analyzer：社員証の未使用権限を棚卸しする係
- GuardDuty：夜中に怪しい動きを見つける警備員
- Security Hub：各警備システムの警告を集める監視室
- CloudTrail：誰がどの部屋に入ったかを残す入退室ログ

だからBが最も運用効率の高い答えです。

### 各選択肢をズバッと

- **A：誤り** … Security Hubは検出結果の集約・セキュリティ態勢の可視化が主役。IAM Access Analyzerのfindingを集約して見ることはできても、「未使用IAM権限を特定する」本体ではない。
- **B：正解** … IAM Access Analyzerを組織レベルで使うと、Organizations配下のIAMユーザー/ロールの未使用アクセスを効率よく確認できる。最小権限化のための棚卸しに直結する。
- **C：誤り** … GuardDutyはCloudTrail、VPC Flow Logs、DNSログなどを分析して不審な振る舞いを検知するサービス。未使用権限は「攻撃の兆候」ではなく「過剰権限の棚卸し」なので役割が違う。
- **D：誤り** … CloudTrailはAPI操作の証跡を残す。ログを自前で集計すれば利用実績の分析材料にはなるが、組織全体で未使用権限を自動検出するには運用負荷が高い。SAAでは専用サービスがあるなら専用サービスを選ぶ。

> [!tip] SAA着眼点
> - **未使用権限 / 最小権限 / IAMユーザー・ロールの権限棚卸し** → IAM Access Analyzer。
> - **誰が何のAPIを呼んだか** → CloudTrail。
> - **不審なAPI呼び出し・侵害の兆候・脅威検知** → GuardDuty。
> - **複数のセキュリティ検出結果を集約** → Security Hub。
> - 問題文にAWS Configが出ても、Configは「リソース構成・準拠」の継続評価が中心。IAMの未使用アクション棚卸しの直球はIAM Access Analyzer。

## 第2問（Cost Optimization ｜ EMR短期バッチのSpot活用とAWS Config日次記録）

> [!question] 問題（再掲）
> Organizationsを利用してAWSアカウントを管理しているマーケティング分析企業がある。セキュリティアカウントがAWS ConfigやSecurity Hubなどを一元管理している。同社では毎日数百件以上のマーケティングデータ処理ジョブをEMR on EC2で実行し、ジョブごとに数十台から百台以上のEC2インスタンスを起動する。ほとんどのインスタンスは数時間以内に終了し、共通の起動テンプレートを使う。大量のインスタンス使用による計算コストと監視サービスの記録コストを最小化しつつ、コンプライアンスやセキュリティイベントを確実にモニタリングしたい。最適な組み合わせは？（2つ選択）
> - A. EMRクラスターのノードにリザーブドインスタンスを使用する。
> - B. Configを無効化し、CloudTrailの管理イベントをAthenaで定期クエリしてリソース構成の変更を追跡する。
> - C. Configの記録モードを日次記録に設定する。
> - D. EMRクラスターのノードにスポットインスタンスを使用する。
> - E. Configの記録モードを継続的（リアルタイム）記録に設定する。

> [!success]- 答え：**C, D**
> 決め手は **短時間で大量に起動するEMRのバッチ処理** と **Configの記録コスト**。計算コストはSpot Instancesで下げ、Configは無効化せず日次記録にしてConfiguration Itemの増加を抑える。

この問題は、2つのコストを別々に処理します。

```text
1. EMRジョブのEC2計算コスト
   -> Spot Instances

2. AWS Configの記録コスト
   -> 日次記録
```

まずEMR側です。問題文には「毎日数百件以上」「ジョブごとに数十台から百台以上」「数時間以内に終了」とあります。これは、常時ずっと同じ台数を動かすWebサーバーというより、必要なときだけ大量に計算するバッチ処理です。

EC2 Spot Instancesは、空きキャパシティを安く使う購入オプションです。AWS公式ドキュメントでも、データ分析・バッチジョブ・バックグラウンド処理など、中断を許容できるワークロードに向いていると説明されています。EMRでも、特にタスクノードはSpotと相性がよく、Amazon EMRはSpotの終了でジョブが失敗しにくいようにアプリケーションマスターをcore/on-demand側に寄せる設定を持っています。

次にAWS Config側です。Configはリソースの作成・変更・削除をConfiguration Item（CI）として記録します。短時間でEC2インスタンスが大量に作成・削除されると、CIが大量に発生して記録コストが増えます。AWS公式ドキュメントでも、EMRジョブやSpot Instancesのような一時的ワークロードではConfigの記録・評価が増える可能性があると説明されています。

そこで、Configを完全に止めるのではなく **日次記録** にします。日次記録は、直近24時間の最新状態を低頻度で記録する方式なので、継続記録よりコストを抑えやすいです。コンプライアンス監視を捨てずに記録量を減らす、という落としどころです。

### たとえ話：臨時作業員と入退室記録

EMRの大量EC2インスタンスを、イベント会場の臨時作業員だと考えます。

毎日イベントがあり、数時間だけ大量の作業員が来て、終わったら帰ります。この人たち全員に「長期契約の席」を買うより、必要なときだけ安く集める方が合います。これが **Spot Instances** です。

一方、AWS Configは入退室・設備変更の記録係です。作業員が数百人来てすぐ帰るたびに逐一細かく記録すると、記録用紙が山のように増えます。だから「記録係を廃止」ではなく、「1日単位でまとめて記録」にする。これが **Configの日次記録** です。

### 各選択肢をズバッと

- **A：誤り寄り** … Reserved Instancesは長期・安定利用に対する割引。短時間で大量に増減するEMRバッチの最小コスト策としてはSpotの方が問題文に合う。中断不可で長期に平均利用が安定しているならRI/Savings Plansも候補だが、この設問の軸は短期バッチ。
- **B：誤り** … Configを無効化すると、構成履歴・準拠評価・Security Hub連携の土台が弱くなる。CloudTrailは「誰が何のAPIを呼んだか」の証跡であり、「今リソース構成がどうなっているか」を評価するConfigの代替ではない。
- **C：正解** … Configの日次記録は、記録頻度を下げてCI数とコストを抑えられる。大量の短命リソースがある環境に合う。
- **D：正解** … Spot Instancesはデータ分析・バッチ処理のような短時間・大量計算・中断許容ワークロードのコスト削減に向く。EMRのタスクノードでは特に定番。
- **E：誤り** … 継続的記録はリアルタイム性が高いが、短命EC2が大量にある環境では記録コストが増える。コスト最小化の要件と逆。

> [!tip] SAA着眼点
> - **バッチ / データ分析 / 短時間 / 中断許容 / 大量EC2** → Spot Instances。
> - **常時稼働 / 長期安定 / 予測可能なベースライン** → Reserved Instances / Savings Plans。
> - **Configのコストが高い / 短命リソースが多い** → 日次記録や記録対象の絞り込みを検討。
> - **Configを無効化** は、コンプライアンス監視を求める問題では基本的に危険。
> - **CloudTrailは操作ログ、Configは構成状態と準拠評価**。この2つは代替ではなく補完関係。

## 第3問（Security / IAM ｜ LambdaからKMS暗号化SNSトピックへPublishする権限）

> [!question] 問題（再掲）
> 金融機関が、顧客に取引アラートを送信するアプリケーションをAWS上で開発している。Lambda関数を利用して、取引情報を含むメッセージをSNSトピックへ発行する。取引情報には機密性の高い財務情報が含まれるため、SNSトピックはAWS KMSのカスタマーマネージドキーを使用してサーバーサイド暗号化する。Lambda関数がSNSトピックに安全にメッセージを発行するための適切な権限設定は？（3つ選択）
> - A. SNSトピックのリソースポリシーで、Lambda実行ロールがトピックにメッセージを発行できるよう許可する。
> - B. KMSのAWSマネージドキーのキーポリシーを編集し、メッセージ発行に関与するプリンシパルを追加して権限を与える。
> - C. Lambda実行ロールに、SNSトピックへのメッセージ発行とKMSカスタマーマネージドキー利用を許可する権限を付与する。
> - D. Lambda関数のリソースベースポリシーで、SNSトピックへのメッセージ発行とKMSカスタマーマネージドキー利用を許可する権限を付与する。
> - E. SNSトピックの暗号化に使うKMSカスタマーマネージドキーのキーポリシーで、メッセージ発行に関与するプリンシパルがキーを利用できるよう許可する。

> [!success]- 答え：**A, C, E**
> 必要なのは **SNSトピック側の許可**、**Lambda実行ロール側の許可**、**KMSキー側の許可** の3点。Lambda関数のリソースベースポリシーは、Lambdaを呼び出す側の許可であり、LambdaからSNSへPublishする外向き権限ではない。

この問題は、権限の置き場所がごちゃっと見えるタイプです。まず登場人物を分けます。

```text
Lambda関数
  |
  | 実行ロールの権限で sns:Publish
  v
SNSトピック（KMSで暗号化）
  |
  | メッセージ暗号化のためKMSキーを使用
  v
KMSカスタマーマネージドキー
```

LambdaがSNSへメッセージを送るとき、実際にAWS APIを呼ぶ主体は **Lambda実行ロール** です。なので、まず実行ロールに `sns:Publish` を許可します。さらにSNSトピックがKMSで暗号化されているため、発行者にはKMSキーを使う権限も必要です。AWS公式ドキュメントでは、SSE有効なSNSトピックへ送信する発行者に `kms:GenerateDataKey*` と `kms:Decrypt` が必要と説明されています。

ただし、KMSはIAMポリシーだけ見て終わりではありません。カスタマーマネージドキーの場合、キーポリシー側でもそのプリンシパルがキーを使えるようにしておく必要があります。つまり、Lambda実行ロールに権限を付けるだけでなく、KMSキー側にも許可の入口を作ります。

最後にSNSトピック側です。SNSトピックはリソースポリシーを持てるため、問題の選択肢ではトピックのリソースポリシーでLambda実行ロールのPublishを許可するAも正解に含めます。実務上、同一アカウント内では実行ロールのアイデンティティポリシーだけで足りる場面もありますが、この設問は「3つ選択」で、SNS側・実行ロール側・KMS側の3点セットを選ばせる意図です。

### たとえ話：封筒を投函するには3つの許可がいる

Lambdaを「職員」、SNSトピックを「郵便ポスト」、KMSキーを「封筒をロックする鍵」だとします。

- SNSトピックポリシー：この職員はこのポストへ投函してよい
- Lambda実行ロール：この職員は投函作業と鍵の利用をしてよい
- KMSキーポリシー：この職員はこの鍵を使って封筒をロックしてよい

どれか1つが欠けると、ポストに近づけても投函できない、投函できても鍵が使えない、という状態になります。

### 各選択肢をズバッと

- **A：正解** … SNSトピックのリソースポリシーで、Lambda実行ロールに `sns:Publish` を許可する。トピック側の入口を開ける設定。
- **B：誤り** … AWSマネージドキーのキーポリシーは利用者が編集できない。さらに問題文はカスタマーマネージドキーを使うと言っている。
- **C：正解** … LambdaがSNSやKMSを呼び出す外向き権限は、Lambda実行ロールに付与する。`sns:Publish`、`kms:GenerateDataKey*`、`kms:Decrypt` などが必要。
- **D：誤り** … Lambda関数のリソースベースポリシーは「誰がこのLambdaを呼び出せるか」を許可するもの。LambdaがSNSへPublishするための権限ではない。
- **E：正解** … カスタマーマネージドKMSキーのキーポリシーで、メッセージ発行に関与するプリンシパルがキーを利用できるよう許可する。

> [!tip] SAA着眼点
> - **Lambdaが他サービスを呼ぶ** → Lambda実行ロール。
> - **他サービスがLambdaを呼ぶ** → Lambdaリソースベースポリシー。
> - **SNSトピックへPublish** → `sns:Publish`。
> - **KMS暗号化SNSへPublish** → 発行者に `kms:GenerateDataKey*` と `kms:Decrypt`。
> - **カスタマーマネージドキー** → キーポリシーを自分で管理できる。AWSマネージドキーのキーポリシー編集は不可。

## 第4問（Security / IAM ｜ ECSタスクロールによるS3読み取り最小権限）

> [!question] 問題（再掲）
> ECSクラスター上で実行されるアプリケーションに対して、複数の特定S3バケットへの読み取りアクセスのみを最小権限で許可したい。セキュリティ要件として、アプリケーションを実行するECSタスク以外のAWSリソースにS3バケットへのアクセス権限を付与してはならない。最適な組み合わせは？（2つ選択）
> - A. ECSクラスターを構成するコンテナインスタンスに関連付けられたIAMインスタンスロールに、作成したIAMポリシーをアタッチする。
> - B. ECSのタスク定義で`executionRoleArn`パラメータに指定するIAMロールに、作成したIAMポリシーをアタッチする。
> - C. バケット名にワイルドカード指定を含むIAMポリシーを作成する。
> - D. ECSのタスク定義で`taskRoleArn`パラメータに指定するIAMロールに、作成したIAMポリシーをアタッチする。
> - E. アクセスが必要なS3バケットのARNをIAMポリシーの`Resource`要素に明示し、必要な読み取りアクションのみを許可するIAMポリシーを作成する。

> [!success]- 答え：**D, E**
> アプリケーションコンテナがS3を読む権限は、ECSタスク定義の **`taskRoleArn`** に指定するタスクロールへ付与する。さらに、対象のS3バケット/オブジェクトARNを明示し、読み取り系アクションだけを許可して最小権限にする。

この問題は、ECSの2つのロールを見分ける問題です。

| ロール | 誰が使うか | 代表用途 |
|---|---|---|
| **task role** / `taskRoleArn` | コンテナ内のアプリケーション | S3を読む、DynamoDBに書く、SQSへ送る |
| **task execution role** / `executionRoleArn` | ECS/Fargateエージェント | ECRからイメージ取得、CloudWatch Logsへログ送信、Secrets取得 |
| **container instance role** | ECSクラスターを構成するEC2インスタンス | ECSエージェントがクラスターに参加する |

問題文は「アプリケーションがS3バケットを読む」と言っています。つまり、アプリケーションコードがAWS APIを呼ぶ話です。この場合は **task role** を使います。

AWS公式ドキュメントでも、ECSタスクにはIAMロールを関連付けられ、その権限はタスク内のコンテナへ提供され、アプリケーションコードがS3など他のAWSサービスを使うために必要だと説明されています。一方、task execution roleはECS/Fargateエージェントがイメージ取得やログ送信などを行うためのロールで、コンテナ内アプリが直接使うものではありません。

### たとえ話：運転手の免許と整備スタッフの鍵

ECSタスクを「配送車」、アプリケーションを「運転手」、ECSエージェントを「整備スタッフ」だとします。

- `taskRoleArn`：運転手の免許。運転手が倉庫（S3）に入って荷物を取り出すための許可。
- `executionRoleArn`：整備スタッフの鍵。車を用意し、燃料を入れ、走行ログを記録するための許可。
- コンテナインスタンスロール：駐車場全体の管理者権限。

今回必要なのは「運転手が特定の倉庫から読む」許可です。整備スタッフの鍵や駐車場全体の管理者権限を渡すと、権限の範囲がズレます。

### S3ポリシーはResourceを明示する

最小権限では、対象バケットをワイルドカードで広げず、必要なバケットだけを書きます。

```json
{
  "Effect": "Allow",
  "Action": [
    "s3:GetObject"
  ],
  "Resource": [
    "arn:aws:s3:::bucket-a/*",
    "arn:aws:s3:::bucket-b/*"
  ]
}
```

アプリがオブジェクト一覧を取得する必要があるなら、`s3:ListBucket` をバケットARNに対して追加します。

```json
{
  "Effect": "Allow",
  "Action": [
    "s3:ListBucket"
  ],
  "Resource": [
    "arn:aws:s3:::bucket-a",
    "arn:aws:s3:::bucket-b"
  ]
}
```

`s3:GetObject` は `bucket/*`、`s3:ListBucket` は `bucket` に対して許可する、というResourceの書き分けもSAAで地味に出ます。

### 各選択肢をズバッと

- **A：誤り** … ECS on EC2のコンテナインスタンスロールにS3権限を付けると、そのEC2上の他タスクにも権限が広がり得る。Fargateではインスタンスロールもない。タスク単位の最小権限ではない。
- **B：誤り** … `executionRoleArn` はECS/Fargateエージェント用。ECRイメージ取得、CloudWatch Logs送信、Secrets取得などのためで、アプリがS3を読む権限ではない。
- **C：誤り** … バケット名ワイルドカードは対象を広げやすい。問題文は「複数の特定S3バケット」なので、ARNを明示する方が最小権限。
- **D：正解** … `taskRoleArn` に指定するIAMロールへS3読み取りポリシーを付与する。アプリケーションコードが使うAWS API権限の置き場所。
- **E：正解** … 必要なS3バケット/オブジェクトARNを`Resource`に明示し、読み取りアクションだけを許可する。

> [!tip] SAA着眼点
> - **コンテナ内アプリがAWSサービスへアクセス** → ECS task role / `taskRoleArn`。
> - **ECSがタスクを起動するための権限** → task execution role / `executionRoleArn`。
> - **EC2起動タイプのホスト側権限** → container instance role。アプリ用権限をここへ入れると広すぎる。
> - **最小権限** → Actionを必要な操作だけ、Resourceを対象ARNだけにする。
> - S3は `GetObject` と `ListBucket` でResource ARNの形が違う。

## 第5問（Security / IAM ｜ インポートACM証明書の期限通知（EventBridge + SNS））

> [!question] 問題（再掲）
> 企業は社内のセキュリティ標準および監査要件により、SSL/TLS証明書を指定された外部認証局から取得している。その証明書をACMにインポートし、インターネット向けWebアプリケーションで利用している。インポート証明書の期限切れによるサービス影響を防ぐため、期限が近づいた際にメール通知を受けたい。通知先SNSトピックは作成済みで、担当者メールアドレスはそのトピックを購読済み。既存のSQS、SNS、Lambdaをターゲットとして利用できる。できるだけ運用負荷を抑えて実装するには？
> - A. EventBridgeで`ACM Certificate Approaching Expiration`イベントを対象にするルールを作成し、ターゲットにSQSキューを設定する。
> - B. ACMの証明書更新機能を利用してインポート済み証明書を自動更新し、更新イベントをトリガーにSNSトピックへ通知する。
> - C. EventBridgeで`ACM Certificate Expired`イベントを対象にするルールを作成し、ターゲットにLambdaを設定する。LambdaからSNSトピックへ通知する。
> - D. EventBridgeで`ACM Certificate Approaching Expiration`イベントを対象にするルールを作成し、通知先としてSNSトピックをターゲットに設定する。

> [!success]- 答え：**D**
> 決め手は **インポート証明書** と **期限が近づいたらメール通知**。インポート証明書はACMが自動更新しないため、EventBridgeの`ACM Certificate Approaching Expiration`イベントをSNSへ直接送る。

この問題は、やりたいことを1本の流れにするとシンプルです。

```text
ACMインポート証明書の期限が近づく
  |
  v
EventBridgeで Approaching Expiration イベントを検出
  |
  v
SNSトピックへ通知
  |
  v
購読済みメールアドレスへメール
```

まず、インポート証明書はACMのマネージド更新対象ではありません。AWS公式ドキュメントでも、インポート証明書は自分で期限を監視し、期限前に更新・再インポートする責任があると説明されています。つまりBの「ACMの証明書更新機能で自動更新」はこの時点で落ちます。

次に、通知タイミングです。サービス停止を防ぎたいなら、期限切れ後ではなく、期限が近づいた時点で通知する必要があります。ACMはEventBridgeへ `ACM Certificate Approaching Expiration` イベントを送信します。公式ドキュメントでは、インポート証明書やプライベート証明書は期限45日前から、ACM発行のパブリック証明書は期限30日前から日次の期限接近イベントが送られると説明されています。

最後に、通知先です。担当者のメールアドレスはすでにSNSトピックを購読しています。なら、EventBridgeルールのターゲットをSNSトピックにすればよく、SQSやLambdaを挟む必要はありません。EventBridgeはSNSトピックをターゲットにできます。

### たとえ話：免許更新のリマインダー

ACM証明書を「営業許可証」だとします。

- インポート証明書：外部機関から取得した営業許可証。AWSが勝手に更新してくれるわけではない。
- Approaching Expiration：期限が近づいたら届く更新リマインダー。
- SNS：担当者へメールを配る連絡網。
- Expired：すでに期限切れ。店を開けられなくなってから気づく状態。

この問題では、店が止まる前に気づきたいので、期限切れ後の`Expired`ではなく、期限接近の`Approaching Expiration`を使います。

### 各選択肢をズバッと

- **A：誤り** … イベントの種類は合っているが、ターゲットがSQS。SQSに入れてもメールは送られないため、別途処理が必要になる。問題文ではSNSトピックとメール購読が用意済みなので遠回り。
- **B：誤り** … インポート証明書はACMのマネージド更新対象外。外部CAで新しい証明書を取得し、ACMに再インポートする必要がある。
- **C：誤り** … `Expired`は期限切れ後なので、サービス影響を防ぐ目的に遅い。さらにACMの`Certificate Expired`イベントはインポート証明書では利用できない。Lambdaを挟むのも運用負荷が増える。
- **D：正解** … EventBridgeで`ACM Certificate Approaching Expiration`を検出し、SNSトピックへ直接通知する。メール購読済みなので最小構成で要件を満たす。

> [!tip] SAA着眼点
> - **インポート証明書** → ACMは自動更新しない。期限監視と再インポートが必要。
> - **期限切れを防ぐ通知** → `ACM Certificate Approaching Expiration`。
> - **期限切れ後の検知** → `Expired`は遅い。インポート証明書では対象外にも注意。
> - **メール通知** → SNSトピック + メール購読。
> - **低運用** → EventBridgeからSNSへ直接。SQS/Lambdaを挟む選択肢は必要性を見る。

## 第6問（Analytics ｜ ストリーミング収集とリアルタイム分析（Kinesis＋Managed Service for Apache Flink））

> [!question] 問題（再掲）
> 農業企業が、農場の**センサー/ドローン**から気象・土壌水分データを**リアルタイム収集・分析**したい。データは**平均5分おき**に届く。**異常な状態を即座に検出**したい。さらに**全データをデータレイクに保存**も計画。最も効果的なのは？
> - A. SNSで受け取り、DynamoDB Streamsでリアルタイム分析。
> - B. Kinesis Data StreamsでS3に保存し、Redshift Spectrumでリアルタイム分析。
> - C. API Gateway＋Lambdaで収集しS3に保存、Athenaでリアルタイム分析。
> - D. Kinesis Data Streamsで取り込み、Managed Service for Apache Flinkでリアルタイム分析、分析後データはすべてS3に保存。

> [!success]- 答え：**D**
> データ分析は「**①収集 → ②分析 → ③保存**」の3工程。それぞれに正しい道具を当てるとDだけが噛み合う：**①収集＝Kinesis**、**②即座に分析＝Apache Flink（流れながら分析）**、**③データレイク＝S3**。

肝は「**リアルタイム分析**」の一語です。あなたが選んだBは**収集（Kinesis）までは正しい**のですが、分析に選んだ**Redshift Spectrumが“リアルタイム”ではない**——ここが引っかけの核心です。

### たとえ話：川を流れる水を「流れながら」検査するか、「溜めてから」検査するか
センサーデータを**次々流れてくる川の水**だと思ってください。

- **D（Apache Flink）＝川に検査員を立たせて、流れる水をその場で即チェック**。異常（急な水質変化＝異常気象/土壌）をリアルタイムに見つけて即アラート。これが「ストリーム処理（流れながら分析）」。
- **B（Redshift Spectrum）＝水を一度ダム（S3）に溜めてから、後でバケツで汲んで検査**。SQLで問い合わせる分析で、「溜まったデータを後から調べる」バッチ寄り。**流れながら即座に、という要件に合わない**。
- **収集（Kinesis）＝川の水を引き込む取水口**。DはこれもちゃんとあるのでOK。

「即座に検出」なら**溜める前に、流れながら**見る＝Flink一択です。

### 各選択肢をズバッと
- **D：正解** … **Kinesis Data Streams（収集）→ Managed Service for Apache Flink（流れるデータをその場でリアルタイム分析・異常検出）→ S3（データレイクに全保存）**。3要件をきれいに満たす王道構成。
- **B：誤り（あなたの回答・最大の罠）** … 収集のKinesisは正しいが、**Redshift Spectrumは「S3上のデータをRedshiftに取り込まずSQLでクエリする」機能**。オンデマンドの問い合わせ分析で、ストリームを流れながら即検出する用途ではない。「Kinesisが入ってるから正解っぽい」に釣られる。
- **A：誤り** … **SNSは通知のPub/Sub**で、大量ストリーミングデータの取り込み基盤ではない。**DynamoDB Streamsはテーブルの変更履歴**を流す仕組みで、生センサーデータのリアルタイム分析用ではない。データレイク保存もなし。
- **C：誤り** … API Gateway＋Lambdaは大量ストリーミング収集に不向き。**AthenaもS3への都度SQLクエリ（インタラクティブ分析）**で、リアルタイムのストリーム処理ではない。

> [!tip] SAA着眼点
> - **データ分析は「収集→分析→保存」の3工程で部品を当てる**。①**ストリーミング収集＝Kinesis**（Data Streams/Firehose）or MSK。②**流れながらリアルタイム分析／異常検出＝Managed Service for Apache Flink**（旧 Kinesis Data Analytics）。③**データレイク＝S3**。
> - **「リアルタイム／即座に／ストリーム」→ Flink**。**「溜めたデータにSQLで問い合わせ（インタラクティブ/バッチ）→ Athena・Redshift(Spectrum)」**。この“流れながら”か“溜めてから”かの区別が最頻出。
> - **Redshift Spectrum＝S3のデータをRedshiftに取り込まず直接SQL**。既存DWHとS3の過去ログを結合する分析向けで、リアルタイムではない。
> - **SNS＝通知／DynamoDB Streams＝DB変更履歴**。どちらもストリーミングデータ分析の収集基盤ではない（ひっかけ）。

## 第7問（Management / Monitoring ｜ CloudWatch Agentの大規模一括導入（SSM Run Command））

> [!question] 問題（再掲）
> メディア制作会社が、ビデオエンコードのジョブ進行状況やリソース利用率を詳しく知るために、それらのメトリクスを継続的に監視したい。CloudWatchエージェントをすべてのEC2 Linuxインスタンスにインストールし、エンコードサーバーの詳細監視を行う計画。現在約5000台のEC2インスタンスがあり、インスタンスは異なる設定や用途で使用されている。効率的に実現するには？
> - A. Systems Manager Run Commandを使用して、すべてのEC2インスタンスにCloudWatchエージェントをインストールする。
> - B. Systems Manager Patch Managerを使用して、すべてのEC2インスタンスにCloudWatchエージェントのインストールを含むカスタムパッチを適用する。
> - C. EC2 Auto Scalingグループを利用し、CloudWatchエージェントがプリインストールされたAMIを起動する。
> - D. AWS Configを使用して、CloudWatchエージェントがインストールされていないEC2インスタンスを特定し、手動でエージェントをインストールする。

> [!success]- 答え：**A**
> 決め手は **既存の約5000台に一括インストール**。CloudWatch Agentの導入と起動は、Systems Manager Run Commandで管理対象インスタンスへリモート実行するのが最も効率的。

この問題は、CとDを落とせているならかなり良いところまで来ています。残るA/Bは、サービスの役割を分けると見えます。

```text
CloudWatch Agentを大量の既存EC2へ入れたい
  -> Systems Manager Run Command

OSやアプリのセキュリティ更新を管理したい
  -> Systems Manager Patch Manager

新しく起動するEC2へ最初から入れておきたい
  -> AMI / 起動テンプレート

設定状態を検出・評価したい
  -> AWS Config
```

今回の要件は「既存の約5000台にCloudWatchエージェントをインストールする」です。これはパッチ適用ではなく、リモートコマンド実行・パッケージ導入の問題です。

AWS公式ドキュメントでも、CloudWatch AgentをEC2フリートへ導入する方法としてSystems Managerを使う手順が示されています。Run Commandで`AWS-ConfigureAWSPackage`を使って`AmazonCloudWatchAgent`をインストールしたり、`AmazonCloudWatch-ManageAgent`で設定ファイルを使って起動・設定できます。

### たとえ話：5000台の編集室PCに監視ソフトを配る

会社に編集室PCが5000台あるとします。全部に監視ソフトを入れたい。

- **Run Command**：管理画面から「このソフトを入れて起動して」と全PCへ一斉指示する。
- **Patch Manager**：OSの更新プログラムやセキュリティパッチを当てる保守係。
- **AMI**：これから新しく買うPCに、最初から監視ソフトを入れておく設計。
- **Config**：「このPCには監視ソフトが入っていない」とチェックする台帳。

既存5000台へ今すぐ配るなら、配布指示を出せるRun Commandがいちばん自然です。

### 各選択肢をズバッと

- **A：正解** … Run CommandはSSM管理対象インスタンスへコマンドやSSMドキュメントを一括実行できる。既存の大量EC2にCloudWatch Agentを導入する用途に合う。
- **B：誤り** … Patch Managerはパッチ適用とパッチ準拠の管理が主役。CloudWatch Agentの新規導入・設定反映を行う最短ルートではない。「カスタムパッチ」という言葉でそれっぽく見せる罠。
- **C：誤り** … CloudWatch Agent入りAMIは今後の新規起動には便利。ただし問題文は既存の約5000台があり、異なる設定や用途で使われている。既存インスタンスへの一括導入にはならない。
- **D：誤り** … AWS Configは状態の検出・評価が得意。未導入インスタンスの特定には使えても、手動インストールは5000台規模で非効率。

> [!tip] SAA着眼点
> - **多数の既存EC2へコマンド実行・ソフト導入** → Systems Manager Run Command。
> - **パッチ適用・パッチ準拠** → Systems Manager Patch Manager。
> - **新規起動時に標準構成を入れる** → AMI / 起動テンプレート / Auto Scaling。
> - **構成の検出・準拠評価** → AWS Config。
> - 継続的に「インストール済み状態を維持」まで問われるなら、Run Commandに加えてState Manager associationも候補。ただし本問の選択肢ではRun Commandが最適。

## 第8問（Networking / CDN / LB ｜ VPC Latticeによるマイクロサービス間通信とサービスディスカバリ）

> [!question] 問題（再掲）
> 企業が新しい動画配信プラットフォームをAWS上に構築した。プラットフォームはECS上で動作する複数のマイクロサービスで構成され、各マイクロサービスは独立したチームによって複数AWSアカウントで運用されている。各マイクロサービスが相互通信できるよう、gRPCを介したサービス間通信が必要。さらに、マイクロサービスの位置を自動的に検出するサービスディスカバリーも必要。管理負担を最小限に抑えるには？
> - A. 各マイクロサービスを異なるリージョンにデプロイし、Direct Connectでリージョン間通信を確立する。サービスディスカバリーは各リージョンのローカルDNSで実装。
> - B. VPC Latticeでサービスネットワークを構築し、各マイクロサービスをそのネットワークに関連付ける。ECSタスクをターゲットグループに登録し、必要なアクセス制御ポリシーを設定する。通信が必要なVPCをサービスネットワークに接続する。
> - C. Transit Gatewayで各VPCを接続し、ルートテーブルでTransit Gatewayへルーティングする。サービスディスカバリーはRoute 53で実装する。
> - D. VPCピアリングで各VPCを直接接続し、セキュリティグループとルートテーブルを手動構成する。サービスディスカバリーはRoute 53で実装する。

> [!success]- 答え：**B**
> この問題は「VPC同士をつなぎたい」ではなく、**マイクロサービス同士を名前で見つけ、安全に呼び合いたい**問題。サービス間通信、サービスディスカバリー、アクセス制御、低運用をまとめて扱うのが **Amazon VPC Lattice**。

まず、問題文のキーワードを分解します。

```text
ECS上の複数マイクロサービス
複数AWSアカウント
gRPCでサービス間通信
サービスディスカバリー
管理負担を最小化
```

この時点で、ただのVPC接続サービスでは弱いです。欲しいのは、ネットワークの道だけではなく、次のようなアプリケーション層の仕組みです。

| 必要なもの | 何をしてほしいか |
|---|---|
| サービス間通信 | サービスAからサービスBを呼べる |
| サービスディスカバリー | サービスBがどこにいるか名前で見つかる |
| ルーティング | リクエストを正しいターゲットへ流す |
| アクセス制御 | 呼んでよいサービス/プリンシパルだけ許可する |
| 複数アカウント/VPC対応 | チームごとに別アカウントでもつなげる |

VPC Latticeは、AWS公式ドキュメントでも「サービスやリソースを接続・保護・監視するフルマネージドのアプリケーションネットワーキングサービス」と説明されています。複数VPC・複数アカウントにまたがるマイクロサービスを、サービスネットワークに関連付けて通信させられます。

### たとえ話：道路ではなく「社内電話帳と内線交換機」

Transit GatewayやVPCピアリングは、会社の建物同士をつなぐ道路のようなものです。

道路があれば移動はできます。ただし、誰がどの部署にいるのか、どの内線番号で呼べるのか、誰がどの部署へ電話してよいのかは、道路だけでは分かりません。

VPC Latticeは、道路というより **社内電話帳 + 内線交換機 + 入館許可** に近いです。

- サービス名：電話帳に載っている部署名
- サービスネットワーク：同じ内線網
- リスナー/ルール：かかってきた電話を正しい窓口へ回す仕組み
- ターゲットグループ：実際に応答する担当者たち
- 認証ポリシー：この部署に電話してよい人を決めるルール

今回ほしいのは道路そのものではなく、サービス同士が名前で見つけ合い、安全に呼び合う仕組みです。だからVPC Latticeです。

### gRPCはどう見るか

gRPCはHTTP/2を使うサービス間通信方式です。VPC LatticeはHTTP/HTTPS/TLSリスナーを持ち、ターゲットグループのプロトコルバージョンとしてHTTP/2やgRPCを扱えます。公式ドキュメントでも、gRPCリクエストを解析して、パッケージ・サービス・メソッドに基づいてターゲットグループへルーティングできると説明されています。

### 各選択肢をズバッと

- **A：誤り** … Direct Connectは主にオンプレミスとAWSを専用線で接続するためのサービス。複数マイクロサービスのサービスディスカバリーやアクセス制御を提供するものではない。そもそも「異なるリージョンにデプロイ」も要件から飛躍している。
- **B：正解** … VPC Latticeでサービスネットワークを作り、サービスとVPCを関連付け、ターゲットグループ・リスナー・ルール・認証ポリシーを設定する。複数アカウント/VPCのECSマイクロサービス間通信を低運用で扱える。
- **C：誤り** … Transit GatewayはVPC間をつなぐL3ネットワークのハブ。VPC同士は通信できるが、サービス単位のディスカバリー、gRPCルーティング、認可ポリシーは別途作り込む必要がある。
- **D：誤り** … VPCピアリングはVPCが増えるほどメッシュ接続になり、ルートテーブルやセキュリティグループ管理が重くなる。低運用・複数アカウントのマイクロサービス基盤としては弱い。

> [!tip] SAA着眼点
> - **マイクロサービス / 複数VPC / 複数アカウント / サービス間通信 / サービスディスカバリー / 認可 / 低運用** → VPC Lattice。
> - **VPC間のIPルーティングを集約** → Transit Gateway。
> - **少数VPCを直接つなぐ** → VPC Peering。ただし非推移・大規模では管理が重い。
> - **オンプレミスとAWSを専用線接続** → Direct Connect。
> - 「gRPC」「HTTP/2」「リスナー」「ターゲットグループ」「認証ポリシー」が出たら、L3ネットワークではなくアプリケーションネットワークを疑う。

## 第9問（Management / Monitoring ｜ DR環境のIaC再現（CloudFormation））

> [!question] 問題（再掲）
> 大手商社が、現在業務システムの運用環境をアジアリージョンで稼働させている。自然災害リスクを受け、DRのための冗長性を高めることにした。現在の運用環境と同じアーキテクチャを、北米リージョンに迅速かつ確実に構築したい。効率的な手段は？
> - A. Elastic Beanstalkを使用し、現在の運用環境の設定をエクスポートして、別リージョンに新しい環境を自動構築する。
> - B. Configで現在のインフラで使用されるリソースのインベントリを記録し、その情報を使って手動で別リージョンに構築する。
> - C. 現行インフラを元にCloudFormationテンプレートを作成する。CloudFormationを利用して新しいリージョンにテンプレートを適用し、DR用インフラを構築する。
> - D. Backupで現在リージョンのリソースのバックアップを取得し、そのバックアップを北米リージョンにリストアしてDR用環境を復元する。

> [!success]- 答え：**C**
> 決め手は **同じアーキテクチャを別リージョンに迅速かつ確実に構築**。これはバックアップ復元ではなく、インフラをコードとして再現する話なので **AWS CloudFormation** が最適。

この問題は、「DR」と聞いてAWS Backupへ飛びつくと間違えやすいです。問題文が求めているものを分けて見ます。

```text
求めているもの:
  現在と同じアーキテクチャを
  北米リージョンに
  迅速かつ確実に構築

つまり:
  インフラ構成そのものを再現したい
```

CloudFormationは、VPC、サブネット、セキュリティグループ、EC2、ALB、RDS、IAMなどのAWSリソースをテンプレートで定義し、スタックとして作成するサービスです。AWS公式ドキュメントでも、同じテンプレートを再利用して複数リージョンに同じリソースを一貫してプロビジョニングできる、と説明されています。

### たとえ話：設計図で別拠点に同じ店舗を作る

災害対策として、東京の店舗と同じ作りの店舗を北米に作りたいとします。

- **CloudFormation**：店舗の設計図。壁、電気、棚、レジ、倉庫の配置まで書いてあり、別拠点でも同じ店を作れる。
- **AWS Backup**：店舗内の商品や帳簿のコピー。中身の復元には強いが、店の建物全体の設計図ではない。
- **AWS Config**：今の店舗に何があるかを記録した点検簿。見ながら手作業で作ることはできるが、遅くてミスが起きやすい。
- **Elastic Beanstalk保存設定**：特定のアプリ実行環境の設定メモ。店舗全体の設計図ではない。

今回は「同じアーキテクチャを素早く確実に建てたい」ので、設計図であるCloudFormationを使います。

### 各選択肢をズバッと

- **A：誤り** … Elastic Beanstalkの保存設定は、Beanstalkアプリ環境の設定を別環境作成時に適用する仕組み。もし全システムがBeanstalkだけで完結しているなら一部候補になるが、設問は業務システム全体の同一アーキテクチャをDR用に再現したい話。汎用的なIaCであるCloudFormationが適切。
- **B：誤り** … AWS Configは構成履歴・設定状態・準拠評価の記録が主役。インベントリを見ながら手動構築するのは、迅速でも確実でもない。
- **C：正解** … CloudFormationテンプレートで現行インフラをコード化し、北米リージョンでスタックを作成する。繰り返し可能で、ミスを減らし、DR環境の再現性が高い。
- **D：誤り** … AWS Backupはデータや対応リソースのバックアップ/リストアに強い。リージョン間コピーもできるが、「同じアーキテクチャ全体を作る」主役ではない。DRではCloudFormationで器を作り、Backupでデータを戻す、のように組み合わせることはある。

> [!tip] SAA着眼点
> - **同じインフラを別リージョン/別環境に再現** → CloudFormation / CDK / IaC。
> - **データを守る・復元する** → AWS Backup。
> - **現在の構成を記録・準拠評価** → AWS Config。
> - **Beanstalkアプリ環境だけの設定再利用** → Elastic Beanstalk saved configuration。
> - DR問題では「インフラの器を作る」のか「データを戻す」のかを分ける。

## 第10問（Security / IAM ｜ Windows移行時のマネージドAD選定（連携不要のスタンドアロン））

> [!question] 問題（再掲）
> オンプレで複数の**Windowsサーバー**によるWebアプリを運用、各サーバーは**Microsoft AD**で管理。AWSへ移行し各サーバーを**EC2 Windowsインスタンス**に置換予定。**なお、オンプレのADはこのWebアプリのWindowsサーバー群のみを管理している**。移行で**運用負荷が最小**になる方法は？
> - A. AD用のEC2インスタンスを作成してWindowsインスタンス群を管理する。
> - B. オンプレミスのADと連携したAWS Directory Serviceを作成して管理する。
> - C. Windowsインスタンス群の中の1台にADを導入して他を管理する。
> - D. AWS Directory Serviceを作成してWindowsインスタンス群を管理する。

> [!success]- 答え：**D**
> 肝は問題文の一文「**オンプレのADはこのWebサーバー群だけを管理している**」。その群が丸ごとAWSへ移るので、**オンプレADに残る管理対象がゼロ＝連携する必要がない**。AWS内にマネージドAD（AWS Directory Service）を新規作成するのが最小運用。

あなたが選んだ **B（オンプレADと連携）は、一見「安全でしっかりした」ハイブリッド構成に見える最大の罠**です。でもよく読むと、オンプレADが面倒を見ているのは**移行するサーバーだけ**。全部AWSに引っ越したら、オンプレ側には守るものが残りません。だから“橋”をかけ続けるのは無駄な複雑さになります。

### たとえ話：引っ越しで「実家と回線をつなぎ続けるか」
オンプレADを**実家の管理人さん**、Windowsサーバー群を**実家に住む家族**だと思ってください。家族全員が新居（AWS）へ引っ越します。

- **前提の一文が決め手**：この管理人さんは**引っ越す家族しか管理していない**。全員引っ越したら、**実家にはもう管理する人がいない**。
- **D（新居にマネージド管理人を雇う）＝AWS Directory Service**。新居に、掃除も点検も全部やってくれる**プロの管理人（マネージドAD）**を新しく置く。実家とは縁を切ってスッキリ。運用は丸投げ＝最小。
- **B（実家と専用線でつなぎ続ける）＝オンプレAD連携**。誰もいない実家に**わざわざ回線を引いて連絡を取り続ける**。手間もコストも無駄。
- **A・C（自分でAD係を雇って自前運用）＝AD用EC2 / 1台にAD同居**。管理人を自分で雇って、給料・シフト・病欠対応（パッチ・冗長化・バックアップ）まで全部自分で面倒みる＝運用負荷マックス。

### 各選択肢をズバッと
- **D：正解** … **AWS Directory Service（AWS Managed Microsoft AD）** を新規作成し、移行後のWindowsインスタンス群を管理。DCのパッチ・冗長化・バックアップはAWS任せで運用最小。オンプレ連携が不要な本ケースの最適解。
- **B：誤り（あなたの回答・最大の罠）** … オンプレADとの連携（AD Connector／信頼関係）は、**オンプレ側にも管理を残す**場合に意味がある。本問は管理対象が全部AWSへ移るので、連携は**余計な複雑さ**にしかならない。
- **A：誤り** … AD用EC2で**自前AD運用**＝パッチ・冗長化・バックアップを自己管理。運用負荷大でマネージドに劣る。
- **C：誤り** … Windowsインスタンス1台にADを同居させる自前運用。運用負荷に加え、その1台が落ちると認証全体が止まる**単一障害点**。

> [!tip] SAA着眼点
> - **Windows/ADが絡む移行で「運用負荷最小」→ AWS Directory Service（AWS Managed Microsoft AD）**（マネージドAD）。**自前EC2でAD構築＝運用負荷大で不正解**寄り。
> - **オンプレADと“連携”が要るか否かは問題文で判断**：オンプレ側にも管理対象が残る／既存IDをそのまま使い続ける → **連携（AD Connector・信頼関係・Managed AD＋trust）**。本問のように**管理対象が丸ごとAWSへ移る**なら → **連携不要のスタンドアロンなManaged AD**。
> - **AD Connector**＝オンプレADに“橋渡し”するだけ（AWS側にディレクトリ実体を持たない）。**AWS Managed Microsoft AD**＝AWS内に本物のマネージドADを持つ。用途で使い分け。
> - 「一見しっかりしたハイブリッド構成」は、**その連携が本当に要るか**を問題文の条件で必ず確認（不要なら過剰設計＝不正解）。
