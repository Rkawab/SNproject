## 第1問（Security / IAM ｜ 脆弱性管理）

> [!question] 問題（再掲）
> EC2インスタンスとAmazon ECR内のコンテナイメージを継続的にスキャンし、既知のソフトウェア脆弱性を検出したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon Inspector
> - B. Amazon GuardDuty
> - C. Amazon Macie
> - D. AWS Shield

> [!success]- 答え：**A（Amazon Inspector）**
> EC2やECRイメージなどを継続評価し、CVEと実環境の露出を踏まえて修正優先度を付ける。

### 深掘り解説

Inspectorは「建物の弱い壁や古い鍵を探す点検員」。GuardDutyは「実際に怪しい動きをする人物を見つける警備員」で、弱点と攻撃兆候は別の軸である。

- **A**：パッケージ脆弱性とネットワーク露出を評価する正解。
- **B**：CloudTrail、DNS、VPCなどのシグナルから脅威を検知する。
- **C**：S3の機密データを発見・分類する。
- **D**：DDoS攻撃を緩和する。

SAAでは「CVE、脆弱性、EC2/ECRのスキャン」ならInspector、「不審なAPIや通信」ならGuardDutyと切り分ける。

## 第2問（Security / IAM ｜ セキュリティ検出結果の集約）

> [!question] 問題（再掲）
> 複数のAWSアカウントでAmazon GuardDuty、Amazon Inspector、Amazon Macieなどが生成する検出結果を、一元的に確認してセキュリティ態勢を評価したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Security Hub CSPM
> - B. Amazon Detective
> - C. AWS Artifact
> - D. AWS Audit Manager

> [!success]- 答え：**A（AWS Security Hub CSPM）**
> 多数の検出結果を標準形式で集約し、セキュリティ標準に対する態勢も一元表示する。

### 深掘り解説

Security Hub CSPMは「各警備装置の警報を集める中央防災センター」。警報を作るGuardDutyやInspectorを置き換えるのではなく、その結果をまとめて優先順位付けする。

- **A**：複数サービス・アカウントの検出結果集約に合う。
- **B**：個別インシデントの時系列や関係を調査する。
- **C**：AWSの監査報告書・契約文書の取得窓口。
- **D**：自社環境の監査証拠収集を自動化する。

「集約・セキュリティ態勢・標準チェック」はSecurity Hub、「根本原因調査」はDetectiveが決め手。

## 第3問（Security / IAM ｜ セキュリティ事象の調査）

> [!question] 問題（再掲）
> GuardDutyが不審なAPI操作を検出しました。担当者はCloudTrailやVPC Flow Logsなどの関連情報をたどり、影響を受けたリソースと根本原因を効率よく調査したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. Amazon Detective
> - B. AWS Security Hub CSPM
> - C. Amazon Inspector
> - D. AWS Config

> [!success]- 答え：**A（Amazon Detective）**
> ログと検出結果の関係をグラフ化し、何が起きたかを調査する。

### 深掘り解説

Detectiveは「防犯カメラ、入退室記録、人物相関を一つの捜査ボードへ並べる分析官」。すでに異常が見つかった後の追跡に強い。

- **A**：関係性、履歴、根本原因の調査に合う。
- **B**：多数の警報を集約する司令塔で、詳細捜査が中心ではない。
- **C**：攻撃前から存在する脆弱性を探す。
- **D**：構成状態と準拠状況を追跡する。

問題文に「検出後」「関連をたどる」「根本原因」がそろえばDetectiveを選ぶ。

## 第4問（Security / IAM ｜ 監査証拠の自動収集）

> [!question] 問題（再掲）
> 企業はPCI DSSに関する内部監査を定期的に実施しています。AWSリソースの構成やユーザー操作などの証拠を継続的に収集し、監査用の評価レポート作成を効率化したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Audit Manager
> - B. AWS Artifact
> - C. AWS CloudTrail
> - D. Amazon CloudWatch

> [!success]- 答え：**A（AWS Audit Manager）**
> 監査フレームワークを選び、自社AWS環境の証拠を継続収集して評価を整理する。

### 深掘り解説

Audit Managerは「監査用の証拠ファイルを自動で整理する事務局」。CloudTrailなどは証拠の材料だが、監査の枠組み、レビュー、評価全体は管理しない。

- **A**：証拠収集から評価レポートまでを支援する。
- **B**：AWS自身が受けた監査の報告書を取得する。
- **C**：API操作という一種類の証拠を記録する。
- **D**：システムの稼働監視を行う。

「自社環境の証拠を集める」はAudit Manager、「AWSのSOC報告書をもらう」はArtifact。

## 第5問（Management / Monitoring ｜ EC2への安全な接続）

> [!question] 問題（再掲）
> プライベートサブネット内のEC2インスタンスを運用管理したいと考えています。インバウンドのSSHポートを開けず、踏み台サーバーやSSH鍵の配布も避ける必要があります。
> 最も適した機能はどれですか。
>
> - A. AWS Systems Manager Session Manager
> - B. EC2 Instance Connect Endpoint
> - C. AWS Direct Connect
> - D. Amazon Inspector

> [!success]- 答え：**A（Session Manager）**
> SSM Agentからの管理チャネルを使い、IAMで認可された対話セッションを提供する。

### 深掘り解説

踏み台は「管理者専用の玄関を別に建てる」方式。Session Managerは「建物側から管理センターへ安全な内線を張る」方式なので、外向きのSSH玄関を作らずに済む。

- **A**：ポート、踏み台、SSH鍵を不要にできる正解。
- **B**：プライベートIPへのSSH/RDP経路にはなるが、SSH/RDPの仕組み自体は使う。
- **C**：ネットワーク専用線であり、EC2管理セッションではない。
- **D**：脆弱性を評価する。

SAAの典型は「ポート22を開かない」「踏み台不要」「IAMと操作ログ」からSession Managerを導く問題。

## 第6問（Cost Optimization ｜ リソースの適正サイズ推奨）

> [!question] 問題（再掲）
> EC2インスタンス、Auto Scalingグループ、EBSボリューム、Lambda関数などの利用状況を分析し、コストと性能の両面から適切なサイズへの変更候補を確認したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Compute Optimizer
> - B. AWS Cost Explorer
> - C. AWS Budgets
> - D. AWS Cost and Usage Report

> [!success]- 答え：**A（AWS Compute Optimizer）**
> 利用メトリクスを分析し、リソースごとのライトサイジング候補を生成する。

### 深掘り解説

Compute Optimizerは「身体測定を見て服の適正サイズを勧める店員」。Cost Explorerは「服にいくら使ったかを見る家計簿」であり、目的が違う。

- **A**：構成・利用率から具体的な変更候補を出す。
- **B**：支出の推移・内訳を分析する。
- **C**：設定した予算のしきい値を監視する。
- **D**：詳細な請求明細をデータとして出力する。

「m系から何へ変えるか」「過剰・不足プロビジョニング」はCompute Optimizerの語彙。

## 第7問（Management / Monitoring ｜ 承認済み製品の配布）

> [!question] 問題（再掲）
> 中央IT部門が承認したCloudFormationテンプレートだけを、各開発チームがセルフサービスで選択してプロビジョニングできるようにしたいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Service Catalog
> - B. AWS CloudFormation
> - C. AWS Organizations
> - D. AWS Systems Manager

> [!success]- 答え：**A（AWS Service Catalog）**
> 承認済みの製品をポートフォリオとして公開し、利用権限と起動制約を管理する。

### 深掘り解説

CloudFormationが「料理を再現するレシピ」なら、Service Catalogは「会社が安全確認したレシピだけを並べた社内メニュー」。利用者は許可された範囲で注文できる。

- **A**：標準構成のセルフサービス配布に合う。
- **B**：テンプレートを実行する基盤だが、製品カタログの統制は別機能。
- **C**：アカウントと組織ポリシーを管理する。
- **D**：ノード運用や自動化を管理する。

「承認済み」「カタログ」「セルフサービス」「起動制約」がService Catalogの決め手。

## 第8問（Management / Monitoring ｜ ランディングゾーン）

> [!question] 問題（再掲）
> 新たにAWSを利用する企業が、Organizationsを基盤としてログ集約、アカウント発行、統制ルールを含むベストプラクティス準拠のマルチアカウント環境を迅速に構築したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Control Tower
> - B. AWS Resource Access Manager
> - C. AWS Trusted Advisor
> - D. AWS License Manager

> [!success]- 答え：**A（AWS Control Tower）**
> Organizationsを土台に、ランディングゾーン、アカウント発行、コントロールをまとめて提供する。

### 深掘り解説

Organizationsが「町の区画と条例を管理する役所」なら、Control Towerは「道路、交番、標準区画まで整えた新しい町を開設する仕組み」。複数サービスを組み合わせた初期設定を自動化する。

- **A**：ランディングゾーン構築の正解。
- **B**：個別リソースをアカウント間で共有する。
- **C**：既存環境にベストプラクティスの助言を出す。
- **D**：ライセンスの使用状況を追跡する。

「アカウントファクトリー」「ランディングゾーン」「コントロール」はControl Towerの専用語。

## 第9問（Management / Monitoring ｜ アカウント間のリソース共有）

> [!question] 問題（再掲）
> AWS Organizations内の複数アカウントで、Transit GatewayやVPCサブネットなどの対応リソースを共有し、アカウントごとに同じリソースを重複作成することを避けたいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Resource Access Manager（AWS RAM）
> - B. AWS PrivateLink
> - C. VPCピアリング
> - D. AWS IAM Identity Center

> [!success]- 答え：**A（AWS RAM）**
> 対応リソースの所有を中央に保ったまま、他アカウントから利用可能にする。

### 深掘り解説

AWS RAMは「会社の共有会議室を各部署の予約表へ公開する」仕組み。PrivateLinkは「特定サービスの受付窓口だけを他部署へ公開する」仕組みである。

- **A**：Transit Gatewayやサブネットなどの共有に合う。
- **B**：サービスをプライベートエンドポイントとして公開する。
- **C**：VPC間を接続するが、リソースそのものを共有しない。
- **D**：人のSSOを管理する。

試験では共有対象がRAM対応リソースか、単なるネットワーク接続かをまず読む。

## 第10問（Security / IAM ｜ 組織全体のファイアウォール管理）

> [!question] 問題（再掲）
> Organizations配下の多数のアカウントに対して、AWS WAFルール、AWS Shield Advancedの保護、セキュリティグループポリシーなどを中央から一貫して適用・監査したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Firewall Manager
> - B. AWS Network Firewall
> - C. AWS WAF
> - D. AWS Security Hub CSPM

> [!success]- 答え：**A（AWS Firewall Manager）**
> 複数アカウントへ複数種類のセキュリティポリシーを中央配布し、準拠状況を管理する。

### 深掘り解説

WAFが「一店舗の入口に置く検査ルール」なら、Firewall Managerは「全店舗へ同じ検査ルールを配布し、未設置店舗を確認する本部」。保護機能そのものと中央管理機能を分ける。

- **A**：Organizations横断の適用と監査に合う。
- **B**：VPC内の通信を検査するファイアウォール本体。
- **C**：Webリクエストを個々のWeb ACLで検査する。
- **D**：検出結果を集約し、ポリシー配布は行わない。

「複数アカウントへWAF等を一括適用」はFirewall Managerの定型である。
