## 第1問（Compute ｜ ゴールデンイメージの自動作成）

> [!question] 問題（再掲）
> 企業は、最新パッチと社内セキュリティ設定を適用したEC2用AMIを定期的に作成し、テスト後に複数アカウントへ配布する処理を自動化したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. EC2 Image Builder
> - B. AWS Systems Manager Patch Manager
> - C. AWS CloudFormation
> - D. AWS CodeDeploy

> [!success]- 答え：**A（EC2 Image Builder）**
> イメージの構築、検証、配布をパイプラインとして自動化する。

### 深掘り解説

Image Builderは「承認済み部品から製品を組み立て、検品して各倉庫へ配る工場」。Patch Managerはすでに稼働している機械を整備する役割である。

- **A**：AMIの定期作成・テスト・配布に合う。
- **B**：稼働中ノードへのパッチ適用を管理する。
- **C**：インフラリソースをテンプレートから構築する。
- **D**：アプリケーションリビジョンを配布する。

「ゴールデンAMI」「イメージパイプライン」「複数アカウント配布」がImage Builderの決め手。

## 第2問（Compute ｜ オンプレミスでAWS基盤を実行）

> [!question] 問題（再掲）
> データ所在地と極低遅延の要件により、一部のワークロードを自社データセンター内で実行する必要があります。AWSと同じAPI・運用モデルを使い、AWSが管理するハードウェアをオンプレミスに設置したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Outposts
> - B. AWS Local Zones
> - C. AWS Wavelength
> - D. AWS Direct Connect

> [!success]- 答え：**A（AWS Outposts）**
> AWS管理のインフラを顧客施設へ設置し、リージョンと一貫したハイブリッド環境を作る。

### 深掘り解説

Outpostsは「AWS支店を自社ビルの中に設置する」イメージ。Direct Connectは自社ビルとAWSを結ぶ専用道路であり、支店そのものは置かない。

- **A**：顧客施設内という要件を満たす。
- **B**：AWSが選定した都市圏拠点。
- **C**：通信事業者の5G網内。
- **D**：オンプレミスとリージョン間の接続。

配置場所を読むことが重要。「顧客施設内」はOutpostsでほぼ確定する。

## 第3問（Compute ｜ 大都市圏での低レイテンシー）

> [!question] 問題（再掲）
> 動画制作会社は、AWSリージョンから遠い大都市圏の利用者に対し、1桁ミリ秒台の低レイテンシーが必要なアプリケーションをEC2で実行したいと考えています。設備を自社施設に置く必要はありません。
> 最も適した選択肢はどれですか。
>
> - A. AWS Local Zones
> - B. AWS Outposts
> - C. AWS Wavelength
> - D. Amazon CloudFront

> [!success]- 答え：**A（AWS Local Zones）**
> EC2やEBSなどをリージョンより利用者に近い都市圏で実行する。

### 深掘り解説

Local Zonesは「大都市に設けられたAWSの出張所」。Outpostsは顧客が指定する自社拠点、Wavelengthは5G通信事業者のネットワーク内という違いがある。

- **A**：都市圏のEC2低遅延要件に合う。
- **B**：自社施設への設置が必要な場合。
- **C**：5G端末が決め手の場合。
- **D**：キャッシュ配信には向くが、一般EC2ワークロードの配置先ではない。

「都市名・映像制作・ゲーム・低遅延・EC2」はLocal Zonesの典型。

## 第4問（Compute ｜ 5G端末向け超低遅延）

> [!question] 問題（再掲）
> 通信事業者の5Gネットワークに接続された車両やモバイル端末向けに、ネットワークの端にEC2ワークロードを配置して超低レイテンシーを実現したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Wavelength
> - B. AWS Local Zones
> - C. AWS Outposts
> - D. AWS Global Accelerator

> [!success]- 答え：**A（AWS Wavelength）**
> 対応通信事業者の5Gネットワークエッジにワークロードを配置する。

### 深掘り解説

Wavelengthは「携帯基地局に近い場所へアプリの処理場を置く」考え方。端末から遠いリージョンまで往復する区間を短くする。

- **A**：5Gネットワーク内という専用要件に合う。
- **B**：都市圏の一般利用者に近いAWS拠点。
- **C**：顧客施設内。
- **D**：Anycast IPとAWS網で通信を高速化するが、5G網へEC2を配置しない。

SAAでは「5G、通信事業者、モバイル超低遅延」の3語からWavelengthを選ぶ。

## 第5問（Analytics ｜ ストリームのリアルタイム処理）

> [!question] 問題（再掲）
> Kinesis Data Streamsに到着するクリックイベントを、ウィンドウ集計や状態を持つ処理によってリアルタイム分析したいと考えています。Apache Flinkをマネージドで実行する必要があります。
> 最も適したサービスはどれですか。
>
> - A. Amazon Managed Service for Apache Flink
> - B. Amazon Data Firehose
> - C. Amazon Athena
> - D. AWS Glue Data Catalog

> [!success]- 答え：**A（Amazon Managed Service for Apache Flink）**
> Flinkアプリケーションをマネージド実行し、連続データを状態付きで処理する。

### 深掘り解説

Flinkは「流れる荷物を見ながら直近10分の集計表を更新し続ける作業員」。Firehoseは「荷物をまとめて倉庫へ届ける配送便」で、複雑な状態管理が主役ではない。

- **A**：時間ウィンドウ、状態、リアルタイム計算に合う。
- **B**：ストリームをS3などへ配信する。
- **C**：主に保存済みS3データへSQLを実行する。
- **D**：テーブル定義などのメタデータを管理する。

「Flink」が明記されれば直接選べる。明記がなくても状態付きストリーム処理が決め手。

## 第6問（Analytics ｜ クラスター管理不要のビッグデータ処理）

> [!question] 問題（再掲）
> データ分析チームは、Apache SparkやApache Hiveを使うジョブを実行したいと考えています。EMRクラスターのプロビジョニングや容量管理は行わず、ジョブ実行時に使用したリソース分だけ支払う必要があります。
> 最も適したサービスはどれですか。
>
> - A. Amazon EMR Serverless
> - B. Amazon EMR on EC2
> - C. AWS Lambda
> - D. Amazon Redshift provisioned cluster

> [!success]- 答え：**A（Amazon EMR Serverless）**
> Spark/Hiveの実行基盤を自動プロビジョニングし、クラスター管理を不要にする。

### 深掘り解説

EMR on EC2は「分析工場を建てて機械台数を管理する」方式。EMR Serverlessは「ジョブを渡すと必要な作業場が自動で用意される」方式である。

- **A**：フレームワークは使いながら基盤管理を省ける。
- **B**：インスタンス構成を細かく管理したい場合に向く。
- **C**：短時間イベント処理用で、大規模Spark基盤ではない。
- **D**：SQLデータウェアハウス。

「Spark/Hive」と「クラスターを管理しない」が同時に出たらEMR Serverless。

## 第7問（Analytics ｜ データレイクの権限管理）

> [!question] 問題（再掲）
> 企業はS3上のデータレイクをAthena、Redshift、EMRから利用します。テーブル、列、行などのきめ細かなアクセス許可を中央管理し、データカタログと監査を統合したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Lake Formation
> - B. AWS Glue ETL
> - C. Amazon S3 ACL
> - D. Amazon QuickSight

> [!success]- 答え：**A（AWS Lake Formation）**
> データレイクのカタログときめ細かな権限を複数分析サービスへ一貫適用する。

### 深掘り解説

S3の権限が「倉庫の箱ごとの鍵」なら、Lake Formationは「箱の中の表・列・行まで閲覧範囲を決める資料室管理」。分析者ごとのデータ権限を中央化する。

- **A**：データレイク統制の正解。
- **B**：データを変換・加工する処理エンジン。
- **C**：オブジェクト単位のアクセス制御で粒度が合わない。
- **D**：BI可視化サービス。

「S3データレイク＋Athena/Redshift/EMR＋列レベル権限」ならLake Formation。

## 第8問（Migration / Transfer ｜ オンプレミス環境の移行調査）

> [!question] 問題（再掲）
> 大規模なデータセンター移行を計画しています。移行前に、オンプレミスサーバーの構成、使用率、稼働状況、サーバー間依存関係を収集して移行計画に利用したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Application Discovery Service
> - B. AWS Application Migration Service
> - C. AWS Database Migration Service
> - D. AWS DataSync

> [!success]- 答え：**A（AWS Application Discovery Service）**
> 移行前のサーバー棚卸し、利用状況、依存関係データを収集する。

### 深掘り解説

Discovery Serviceは「引っ越し前に家具の数、サイズ、使われ方、配線関係を調査する」。Application Migration Serviceは実際に荷物を新居へ運ぶ段階である。

- **A**：発見・評価フェーズに合う。
- **B**：サーバーを継続複製し、AWSへリホストする。
- **C**：DBデータを移行する。
- **D**：ファイル・オブジェクトを転送する。

「計画前の棚卸しと依存関係」はDiscovery、「移行実行」はMGNと整理する。

## 第9問（Migration / Transfer ｜ 移行状況の一元追跡）

> [!question] 問題（再掲）
> 複数のAWS移行ツールとパートナーツールを使って、多数のアプリケーションを段階的に移行します。各アプリケーションの移行状況を1つの画面で追跡したいと考えています。
> 最も適したサービスはどれですか。
>
> - A. AWS Migration Hub
> - B. AWS Control Tower
> - C. AWS Systems Manager
> - D. AWS Service Catalog

> [!success]- 答え：**A（AWS Migration Hub）**
> 複数ツールから進捗を集約し、アプリケーション単位で移行状況を表示する。

### 深掘り解説

Migration Hubは「複数の引っ越し業者の進捗を一枚の工程表に集める司令室」。移行作業そのものを一種類の方法で実施するサービスではない。

- **A**：移行ポートフォリオの進捗追跡に合う。
- **B**：新しいマルチアカウント環境を統制する。
- **C**：AWSノードとリソースの運用管理用。
- **D**：承認済み製品の配布用。

「複数の移行ツール」「一元ダッシュボード」「進捗追跡」がMigration Hubの決め手。

## 第10問（Cost Optimization ｜ 異常なコストの検知）

> [!question] 問題（再掲）
> 通常の利用傾向を機械学習で学習し、予想外のコスト増加をサービスやアカウントなどの単位で自動検知して担当者へ通知したいと考えています。
> 最も適した機能はどれですか。
>
> - A. AWS Cost Anomaly Detection
> - B. AWS Budgets
> - C. AWS Compute Optimizer
> - D. AWS Cost Explorer

> [!success]- 答え：**A（AWS Cost Anomaly Detection）**
> 支出パターンを学習し、通常から外れたコストと推定根本原因を検出する。

### 深掘り解説

Budgetsは「月額10万円を超えたら鳴る固定アラーム」。Cost Anomaly Detectionは「普段は少額の項目が急に5倍になったことへ気づく見張り役」である。

- **A**：動的な通常傾向から異常を発見する。
- **B**：利用者が定義した予算しきい値を監視する。
- **C**：リソースの適正サイズを推奨する。
- **D**：過去と現在の費用を分析する。

「機械学習・通常傾向・異常支出」はCost Anomaly Detection、「固定予算」はBudgets。
