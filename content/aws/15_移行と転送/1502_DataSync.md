# AWS DataSync

> [!info] このノートについて
> オンプレミス ⇔ AWS、あるいは AWS サービス間で **大量のファイル／オブジェクトを高速・自動・安全に転送** するマネージドサービス。SAAでは「NFS/SMB の大量データを S3/EFS/FSx へ定期的に同期・移行」という形で頻出。

## ひとことで言うと

引っ越しの「**オンライン高速配送便**」。荷物（ファイル）を自動で**梱包（暗号化）・検品（整合性チェック）**し、**前回から変わった分だけ（差分）**を**定期便（スケジュール）**で運んでくれる。オンプレ側には専属ドライバー（**DataSyncエージェント**）を1人配置するだけ。

> [!example] 例え：定期便の宅配サービス
> 自分でトラックを手配して荷造り・運転・検品…とやる代わりに、宅配業者（DataSync）に任せると「集荷→暗号化→高速輸送→中身チェック→差分だけ次回配送」を全部自動でやってくれる。人手（運用）がほぼ要らないのがポイント。

---

## 何と何をつなげるか（転送元・転送先）

| 種別 | 対応 |
|---|---|
| **オンプレ側** | NFS / SMB / HDFS / 自己管理オブジェクトストレージ（S3互換） |
| **AWS側** | **S3** / **EFS** / **FSx**（Windows・Lustre・NetApp ONTAP・OpenZFS） |
| **方向** | オンプレ ⇔ AWS、**AWSサービス間**（例：EFS→S3）、他クラウド ⇔ AWS |

> [!note] エージェントの要否
> - **オンプレ ⇔ AWS** … オンプレ側に **DataSyncエージェント（仮想アプライアンス）** をデプロイする
> - **AWSサービス間**（S3↔EFS↔FSx など） … **エージェント不要**（AWSマネージドで完結）

---

## 主な特徴（ここがDataSyncの強み）

| 特徴 | 内容 |
|---|---|
| **差分（増分）転送** | 前回から変わったファイルだけ転送。定期バックアップ・同期に最適 |
| **高速・並列化** | マルチスレッドで自前スクリプト(rsync等)より高速。回線を使い切る |
| **暗号化＋整合性チェック** | 転送中はTLS暗号化、転送後はチェックサムで中身を自動検証 |
| **スケジュール実行** | 毎時・毎日・毎週など定期同期をGUIで設定できる |
| **帯域制御（スロットリング）** | 業務時間帯の回線を圧迫しないよう転送速度を制限可能 |
| **メタデータ保持** | 権限・タイムスタンプ・所有者などのファイル属性を維持 |
| **フルマネージド＆監視** | エージェント死活監視・自動再試行・CloudWatch連携 |
| **料金** | 転送した**データ量(GB)課金**。初期費用なし |

---

## 使い方（オンプレ→AWSの基本フロー）

```
① オンプレに DataSyncエージェント（VM）をデプロイ
        ↓
② ソース（NFS/SMBの場所）を「ロケーション」として登録
        ↓
③ ターゲット（S3 / EFS / FSx）を「ロケーション」として登録
        ↓
④ タスクを作成（オプション：差分/全件・検証・帯域・除外設定）
        ↓
⑤ スケジュール設定（毎日・毎週…）or 手動実行
        ↓
⑥ 以降は自動で増分転送が続く（暗号化＋整合性チェック込み）
```

## SAA試験での判断ポイント

> [!important] DataSync を選ぶ合図
> 問題文に次のキーワードが揃ったら DataSync：
> - 「**オンプレの NFS / SMB**」の「**大量データ**」を
> - 「**S3 / EFS / FSx**」へ
> - 「**定期的に**」「**継続的に**」「**自動で**」「**運用の手間を最小化して**」転送・同期・移行したい

| 問題文のキーワード | 正解 |
|---|---|
| オンプレNFS/SMBの大量ファイルをS3/EFS/FSxへ移行・同期 | **DataSync** ✅ |
| 既存のSFTP/FTPワークフローを維持したまま受け渡し | Transfer Family |
| データベースを止めずに移行 | DMS（+SCT） |
| 回線が細い／超大容量で回線だと数週間かかる | Snow Family |
| オンプレを残してキャッシュ経由でクラウドを常用 | Storage Gateway |

## 紛らわしいサービスとの違い（最重要）

| 比較相手 | 相手の役割 | DataSyncとの違い |
|---|---|---|
| **Kinesis Data Firehose** | **ストリーミングデータ**（ログ・IoT・クリックストリーム）をリアルタイムに S3/Redshift/OpenSearch/Splunk へ配信 | DataSyncは「**ファイル**を一括/定期転送」。Firehoseは「**レコードを連続ストリーム**」。**FSx/EFSへは配信不可** |
| **Storage Gateway** | オンプレを残し、キャッシュ経由でクラウドを**常用**（ハイブリッドアクセス） | DataSyncは「**移して終わり・定期同期**」。SGは「**使い続ける**」 → [[1503_Storage Gateway]] |
| **Snow Family** | 物理デバイスで**オフライン**搬送 | DataSyncは**オンライン**。回線で現実的な時間に送れるならDataSync、無理ならSnow |
| **DMS** | **データベース**の移行（継続レプリケーション） | DataSyncは**ファイル/オブジェクト**。DBそのものはDMS |
| **Transfer Family** | **SFTP/FTPS/FTP** のプロトコルを維持して受け渡し | DataSyncは**速度・自動化**が目的。プロトコル維持や外部取引先の受け口ならTransfer Family |
| **S3 Transfer Acceleration** | S3アップロードを**高速化する機能** | あくまでS3への単発アップロード高速化。オンプレの**ファイル同期を自動化**するものではない |

> [!tip] 一行で区別
> **DataSync＝ファイルの「定期・自動・高速な引っ越し便」**。
> 「ストリーム」ならFirehose、「使い続ける」ならStorage Gateway、「DB」ならDMS、「回線が無理」ならSnow。
