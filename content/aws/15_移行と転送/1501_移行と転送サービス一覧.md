# AWS 移行と転送（Migration and Transfer）サービス一覧

> [!info] このノートについて
> AWSマネジメントコンソールの「移行と転送」カテゴリに含まれる主要サービスの整理。SAA試験を意識した選択判断つき。

## 全体像：引っ越し業者のチームに例える

オンプレミス（自宅）から AWS（新居）へ引っ越すイメージ。役割の違う専門業者がいる。

| 役割 | サービス | 引っ越しでの例え |
|---|---|---|
| 📋 進捗管理者 | **Migration Hub** | 引っ越し全体の工程表・進捗ボード |
| 🔍 下見調査員 | **Application Discovery Service** | 荷物量・配置・依存関係を調査 |
| 🚚 サーバー運送 | **Application Migration Service (MGN)** | 部屋ごとそのまま新居へ運ぶ |
| 🗄️ DB専門運送 | **Database Migration Service (DMS)** | 精密機器（DB）を専門に運ぶ |
| 📡 オンライン配送 | **DataSync** | ネット経由で大量の荷物を高速・自動転送 |
| 📁 宅配窓口 | **Transfer Family** | SFTP/FTP の宅配便でファイル受け渡し |
| 📦 物理トラック | **Snow Family** | 回線が細い時、物理デバイスで大量搬送 |

---

## 各サービス詳細（SAA重要度つき）

### 1. AWS Migration Hub　⭐重要
- 複数の移行ツールの **進捗を1画面で一元管理** するダッシュボード。
- それ自体は移行しない。「司令塔・進捗ボード」の役割。

### 2. AWS Application Discovery Service　⭐重要
- 移行前に **オンプレのサーバー構成・性能・サーバー間の依存関係** を自動収集。
- 「どのサーバーが何と通信しているか」を可視化し、移行計画を立てる。
- Migration Hub と連携して使う。

### 3. AWS Application Migration Service（MGN）　⭐⭐最重要
- サーバーを **そのまま AWS へ持ち上げる（リフト&シフト）** 代表サービス。旧称 CloudEndure Migration。
- OS・アプリ・設定を丸ごと継続的にレプリケーションし、最小ダウンタイムで切替。
- 「アプリを改修せず、とにかく早くクラウドへ」という時の第一候補。

> [!important] AWS SMS との関係（試験での混同注意）⚠️
> **AWS SMS（Server Migration Service）** は現在 **非推奨（サンセット）** されており、MGNがその後継サービスです。
> 
> | 項目 | AWS SMS | AWS MGN |
> |---|---|---|
> | **正式名** | Server Migration Service | Application Migration Service |
> | **状態** | 新規利用終了済み（レガシー） | 現行・推奨 |
> | **レプリケーション方式** | スナップショット単位の定期レプリケーション | 継続的・ブロックレベルのレプリケーション |
> | **ダウンタイム** | 比較的長い | 最小限（最終同期の短時間のみ） |
> | **試験での扱い** | 古い問題集では正解選択肢として残存 | 現行試験では MGN が正 |
> 
> **試験対策** 📚
> - SMS と MGN どちらか一方が選択肢に出た場合、**新しい方の MGN を選ぶ** が正解
> - 「SMS → MGN（後継）」という関係を覚えておくと、古い問題集での引っかかりを回避できる
> - 実務・試験最新版では MGN が標準

#### 移行の4ステップ（試験頻出の流れ）

> [!example] 例え：営業中の店舗を、閉店せずに新店舗へ引っ越す
> 旧店舗（オンプレVM）を開けたまま、新店舗（AWS）に“生き写しのコピー”を裏で作り続け、最後の一瞬だけ切り替える。

```
① エージェント導入
   移行対象VMに「AWS Replication Agent」をインストール
        ↓
② 継続レプリケーション
   ディスク内容をステージング領域へ自動・継続コピー（旧店舗は営業継続）
        ↓
③ テスト起動
   テストインスタンスを立ち上げ、AWS上で正しく動くか確認（本番はまだ旧側）
        ↓
④ カットオーバー（切替）
   旧VMへの書き込み停止 → 最終差分を同期 → カットオーバーインスタンス起動
   → これで本番がAWS側に移る（ダウンタイムは最終同期のわずかな時間だけ）
```

> [!warning] 紛らわしい：MGN と DRS は「同じエージェント名」
> **AWS Replication Agent** という同じ名前のエージェントを使うが目的が違う。
> - **MGN**（Application Migration Service）＝ **移行**（引っ越して旧環境は捨てる）
> - **DRS**（Elastic Disaster Recovery）＝ **災害復旧**（普段は待機、障害時だけ起動）
> 問題文が「移行（migration）」なら MGN、「DR・バックアップ・復旧」なら DRS。

> [!tip] なぜ DataSync + AMI 案は不正解になりやすいか
> DataSync は**ファイル転送**ツール。VMを丸ごと（OS・設定込み）移すには手順が増え、
> 「変更最小・短期間・低ダウンタイム」の要件に合わない。サーバー丸ごとは MGN が定石。

#### MGN の具体的な移行フロー（4ステップ）

MGNを使った移行では以下の流れを辿ります：

```
① Replication Agent をインストール（オンプレミスサーバーへ）
     ↓
② 継続的レプリケーション開始
   （AWS ステージングエリアへ自動同期）
     ↓
③ テストインスタンスを起動・動作確認
   （本番切替前にAWS上での動作を検証）
     ↓
④ カットオーバー実施
   オンプレミスへの書き込み停止
   → 最終同期実施
   → カットオーバーインスタンス起動
   → 完了
```

**重要ポイント**

| 段階 | 説明 | 目的 |
|---|---|---|
| **Replication Agent** | オンプレサーバーにインストールする専用エージェント | ディスク変更をAWSへ継続的に複製 |
| **ステージングエリア** | AWS内の準備領域 | テストやカットオーバー前の中間保管 |
| **テストインスタンス** | 本番切替前に起動するテスト用EC2 | ダウンタイムなしに動作検証可能 |
| **カットオーバー** | 最終的な本番環境への切替 | 書き込み停止→最終同期→本番開始 |

**試験での使い分け**

MGNが選ばれる条件（問題文の3つの要件が揃うと答え）：
- ✅ コード・構成の **変更を最小限に**（リフト&シフト）
- ✅ **短期間で完了**（継続レプリケーション型で準備期間を短縮）
- ✅ **ダウンタイムを最小限に**（カットオーバー直前まで同期し続ける）

### 4. AWS Database Migration Service（DMS）　⭐⭐最重要
- **稼働中の DB を止めずに** 移行（継続的レプリケーション対応）。
- 2パターン:
  - **同種移行**（Oracle→Oracle）：そのまま
  - **異種移行**（Oracle→Aurora など）：**AWS SCT（Schema Conversion Tool）** でスキーマ変換 → DMS でデータ移行
- 試験頻出：「最小ダウンタイムで DB 移行」「異種 DB へ移行」=**DMS（+SCT）**。
- 関連: [[0301_RDS]] / [[0303_DynamoDB]]
> [!note] 用語
> 選択肢では **AWS DMS** と略記されることがある。ファイル転送のDataSyncではなく、DB移行専用。

### 5. AWS DataSync　⭐⭐重要
- オンプレ ↔ AWS（S3 / EFS / FSx）間の **大量ファイル転送を自動化・高速化**。
- 専用エージェントが暗号化・整合性チェック・差分転送を担当。
- 試験頻出：「NFS/SMB の大量データを定期的に S3 へ」=**DataSync**。
- 詳細・他サービスとの使い分け → [[1502_DataSync]]
- 関連: [[0202_S3とEBS・EFSの違い]]

> [!note] DataSync（転送・移行） vs Storage Gateway（常用のハイブリッドアクセス）
> 「移して終わり・定期同期」なら **DataSync**、「オンプレを残してキャッシュ経由で**使い続ける**」なら → [[1503_Storage Gateway]]。

### 6. AWS Transfer Family　⭐重要
- **SFTP / FTPS / FTP** プロトコルのまま S3・EFS へファイル送受信できるマネージドサービス。
- 既存の取引先が「SFTP でしか送れない」場合に、サーバーを自前管理せず受け口を用意できる。
- 試験頻出：「既存の SFTP ワークフローを維持したまま S3 へ」=**Transfer Family**。

> [!info] そもそも SFTP / FTPS / FTP とは（ファイル転送プロトコル）
> ファイルをネットワーク越しに送受信する**昔ながらの取り決め（プロトコル）**。書類のやり取りに例えると：
>
> | プロトコル | 正式名 | たとえ | 暗号化の土台 |
> |---|---|---|---|
> | **FTP** | File Transfer Protocol | ハガキ（中身丸見え） | なし（平文） |
> | **FTPS** | FTP over SSL/TLS | 封筒に鍵 | SSL/TLS |
> | **SFTP** | **SSH** File Transfer Protocol | 金庫便（**現在の主流**） | SSH |
>
> - **SFTP と FTPS は名前は似て非なるもの**：暗号化の土台が SSH か SSL/TLS かで別物。実務で最もよく使われるのは **SFTP**。
> - 取引先が「SFTPでしか送れない」のは、**先方が長年その仕組みで運用しているから**。こちら都合で変えられない → だから受け口側（AWS）が合わせる、という発想で **Transfer Family** が選ばれる。
>
> **なぜ Transfer Family が「運用レス」なのか**：本来 SFTP で受け取るには自分で SFTP サーバーを立て、常時稼働・パッチ・監視が必要。Transfer Family は**そのサーバー部分をAWSがマネージドで肩代わり**し、受信したファイルをそのまま S3 / EFS に流し込む。

### 7. AWS Snow Family　⭐⭐重要
- **物理デバイスでオフライン搬送**。ネット回線が細い／データ量が膨大な時に使う。
  - **Snowcone**：最小（数TB）、携帯可
  - **AWS Snowball Edge**：数十TB〜PB級。Storage最適化型／Compute最適化型（エッジで処理も可）
  - ~~Snowmobile~~：トラック輸送（EB級、現在は新規提供終了）
- 試験の判断軸：「**回線で送ると数週間〜数ヶ月** → Snow デバイスで物理輸送」。

### （参考）その他
- **AWS Mainframe Modernization**：メインフレームのクラウド移行・モダナイズ。SAA では優先度低。
- **Migration Hub Refactor Spaces**：モノリス→マイクロサービス化（リファクタリング）支援。SAA では優先度低。
- **AWS Server Migration Service（SMS）**：⚠️ **非推奨（2024年頃にサンセット）**。現在は MGN に統合。古い教材では見かけることがあるが、現行試験では MGN が正解。

---

#### MGN と似たサービスの混同注意

| サービス | 用途 | 選ばれない理由 |
|---|---|---|
| **Elastic Disaster Recovery (DRS)** | 障害復旧（DR）用 | DR専用。移行には使わない |
| **AWS DataSync** | ファイル転送 | VMのOS/設定を含む丸ごと移行に不向き |
| **AWS Systems Manager** | パッチ管理・運用 | データ収集・移行用途ではない |

## ⭐SAA頻出の選択判断チャート

```
データ/システムをAWSへ移したい
│
├─ サーバーを丸ごと（リフト&シフト）   → MGN
├─ データベースを移したい
│    ├─ 同じDBエンジン同士            → DMS
│    └─ 違うエンジンへ変換も必要       → SCT + DMS
├─ ファイルをオンライン転送（継続的）  → DataSync
├─ SFTP/FTPのまま受け渡したい        → Transfer Family
├─ 回線が細い/超大容量をオフライン     → Snow Family
└─ 移行全体の進捗を管理したい          → Migration Hub（+ Discovery Service）
```

> [!warning] 間違えやすいペア
> - **DataSync**（オンライン・継続/定期転送） vs **Snowball**（オフライン・一回の大量搬送）
> - **DMS**（DB移行） vs **DataSync**（ファイル移行）
> - **Transfer Family**（プロトコル維持が目的） vs **DataSync**（速度・自動化が目的）

---

> [!tip] SMS と MGN の覚え方
> **S**MS（古い・**S**unset＝終了） → M**G**N（新しい・**G**o）
> 
> 名前が紛らわしいですが、「Gが付く方が新しい・Go forward」と区別すれば覚えやすい。
