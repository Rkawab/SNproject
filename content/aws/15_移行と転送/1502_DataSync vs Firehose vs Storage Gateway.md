# DataSync vs Firehose vs Storage Gateway の使い分け

> [!info] このノートについて
> オンプレミスのSMBファイルサーバーをAWS FSx for Windowsへバックアップ・同期する際、どのサービスを選ぶべきかの比較。SAA試験でも頻出の紛らわしいサービス選定問題。

## シナリオ：オンプレSMBファイルサーバー → FSx for Windows への定期バックアップ

**要件**
- オンプレミスのWindowsサーバー（SMBファイル共有）の内容を定期的にバックアップしたい
- バックアップ先：AWS FSx for Windows
- 運用の手間を最小化したい

---

## サービス別の判定

### ✅ 正解：AWS DataSync

**向いている理由**

| 観点 | 評価 |
|---|---|
| **SMB対応** | ✅ ネイティブにSMBをサポート |
| **配信先対応** | ✅ FSx for Windowsへの配信が可能 |
| **スケジュール実行** | ✅ 定期的な同期が簡単に設定できる |
| **差分転送** | ✅ 増分同期で効率的（バックアップに最適） |
| **マネージド** | ✅ 設定・運用の手間が少ない |
| **用途のマッチ** | ✅ まさに「ファイル同期」を目的としたサービス |

**DataSyncの具体的な使い方**

```
1. オンプレミスに DataSync エージェント をインストール
2. SMBサーバーのエンドポイント を設定
3. FSx for Windows をターゲット として指定
4. スケジュール設定：毎日・毎週など定期実行
5. 差分・フルスキャン・変更追跡などの戦略を選択
6. 実行開始 → 自動で増分転送が続く
```

**マネージドサービスのメリット**
- エージェントのヘルスチェック・自動再起動
- 暗号化・整合性チェックが自動化
- ログ記録・モニタリング統合（CloudWatch）
- セットアップ複雑度は低め

---

### ❌ 不正解1：Amazon Kinesis Data Firehose

**間違いの理由**

| 観点 | 問題点 |
|---|---|
| **配信先の制限** | ❌ Firehoseの配信先は **S3 / Redshift / OpenSearch / Splunk** のみ。**FSxには配信できない** |
| **用途が違う** | ❌ Firehoseは「**ストリーミングデータ**（ログ・IoT・クリックストリーム）をリアルタイムで処理」するもの |
| **要件のミスマッチ** | ❌ 「定期バックアップ」=スケジュール型。Firehoseの**継続的・リアルタイムストリーミング**とは目的が異なる |
| **ファイル構造の保持** | ❌ Firehoseはレコード（行）単位の処理。**ディレクトリ構造やメタデータ**の保持が難しい |

**わかりやすい例え**

> Firehoseは「**川の水を絶え間なく貯水池(S3など)へ流し込むパイプライン**」  
> 一方、SMBファイルサーバー → FSx の要件は「**箱詰めされた荷物を倉庫へ定期配送**」  
> 役割が根本的に異なる。そもそもFSxという蛇口口は用意されていない。

---

### ❌ 不正解2：AWS Storage Gateway（ボリュームゲートウェイ 保管型）

**間違いの理由**

| 観点 | 問題点 |
|---|---|
| **バックアップ先** | ❌ ボリュームゲートウェイのバックアップ先は **S3（EBSスナップショット形式）**。**FSxへは保存できない** |
| **ストレージ種別** | ❌ ボリュームゲートウェイは **iSCSI ブロックストレージ** を提供。問題文の **SMB（ファイル共有）** とは技術的に合わない |
| **保管型(Stored)の特性** | ❌ 保管型は「全データをオンプレに保持し、S3へ非同期バックアップ」。今回の「SMB→FSx同期」の要件とは無関係 |
| **用途** | ❌ Storage Gatewayは「ハイブリッドストレージ」（キャッシュ or バックアップ機能）。「ファイル移行」とは異なる |

**Storage Gatewayの実際の用途**

| タイプ | 用途 | 今回に適用できるか |
|---|---|---|
| **ボリュームゲートウェイ（保管型）** | オンプレの全データをS3にバックアップ | ❌ 配信先がS3固定、FSxには非対応 |
| **ボリュームゲートウェイ（キャッシュ型）** | 頻繁に使うデータをオンプレにキャッシュ | ❌ S3キャッシュが主目的、FSx非対応 |
| **ファイルゲートウェイ（S3向け）** | NFS/SMBでS3をマウント | ❌ 配信先がS3固定 |
| **ファイルゲートウェイ（FSx向け）** | NFS/SMBでFSx for Windowsをマウント | ⚠️ 可能だが「キャッシュアクセス」が目的。バックアップ同期目的なら DataSync が適切 |

> [!note] 補足：Storage Gateway ファイルゲートウェイ（FSx File Gateway）について
> **FSx File Gateway** は確かにFSxをサポートしていますが、用途は異なります：
> - **FSx File Gateway** = オンプレからFSxを「キャッシュ経由でアクセス」する（=ハイブリッドアクセス）
> - **DataSync** = オンプレ → FSxへ「定期的にバックアップ・同期」する（=ファイル転送）
> 
> 「バックアップ」「同期」「転送」が目的なら **DataSync** が最適解です。

## SAA試験での判断フロー

```
オンプレ → AWSストレージ間のファイル転送？
│
├─ 「定期的な同期・バックアップ」目的
│  ├─ 配信先がFSx or EFS or S3
│  └─ → DataSync ✅
│
├─ 「ストリーミングデータをリアルタイム処理」
│  ├─ 配信先がS3, Redshift, OpenSearch, Splunk
│  └─ → Kinesis Data Firehose ❌（今回は不適切）
│
├─ 「ハイブリッドストレージ・キャッシュアクセス」
│  ├─ 配信先がS3 or FSx（キャッシュ経由）
│  └─ → Storage Gateway ❌（今回は不適切）
│
└─ 「物理デバイスでオフライン転送」
   ├─ 大容量・回線が細い
   └─ → Snow Family ❌（今回は不適切）
```

## 重要ポイント：似ているが全く違う3つのサービス

| 項目 | DataSync | Firehose | Storage Gateway |
|---|---|---|---|
| **分類** | ファイル転送 | ストリーミング取り込み | ハイブリッドストレージ |
| **処理モデル** | 定期/継続同期 | リアルタイムストリーミング | キャッシュ or バックアップ |
| **対応プロトコル** | NFS / SMB | REST API（カスタムフォーマット） | iSCSI / NFS / SMB |
| **FSx対応** | ✅ DataSync native | ❌ 非対応 | ⚠️ FSx File GW のみ |
| **S3対応** | ✅ 可能 | ✅ 可能 | ✅ 可能 |
| **差分転送** | ✅ あり | ✅ あり（ストリーム処理） | ⚠️ バックアップのみ |
| **スケジュール** | ✅ あり | ❌ 継続的のみ | ⚠️ 定期スナップショット |
| **用途** | ファイル移行・同期 | ログ・IoT・クリックストリーム | ハイブリッド+バックアップ |

## 実践的な選定例

### 例1：EC2内のアプリケーションログをS3へ定期保存

```
選択肢:
A. DataSync
B. Firehose
C. CloudWatch Logs → S3
```

**正解：C（CloudWatch Logs） or B（Firehose も可）**
- DataSyncは「ファイルサーバー間」の転送専門
- ログストリーム → S3ならFirehose
- CloudWatch統合ならCloudWatch Logs

---

### 例2：オンプレのNFSストレージをEFS（AWS）へ毎週同期

```
選択肢:
A. DataSync
B. Storage Gateway
C. Kinesis
```

**正解：A（DataSync）**
- スケジュール定期同期 + ファイルサーバー間 → DataSync
- Storage Gatewayはキャッシュ/バックアップ機能で、「転送」がメイン目的ではない

---

### 例3：IoTセンサーからのリアルタイムデータをAmazon OpenSearchへ取り込む

```
選択肢:
A. DataSync
B. Firehose
C. SQS
```

**正解：B（Firehose）**
- リアルタイムストリーミング → OpenSearch
- Firehoseはこのシナリオの専門サービス
- DataSyncはファイル転送なので不適切

## まとめ：頻出パターンの記憶法

> [!important] SAA頻出パターン早見表
> 
> **1. ファイル移行・同期（オンプレ ⇔ FSx/EFS/S3）** → **DataSync**
> - 「定期的に」「バックアップ」「同期」のキーワード
> - スケジュール実行が可能
> - 差分転送で効率的
> 
> **2. リアルタイムストリーミング（ログ・IoT・クリックストリーム）** → **Kinesis/Firehose**
> - 「リアルタイム」「ストリーム」「連続的」のキーワード
> - Firehoseは配信先が限定（S3/Redshift/OpenSearch/Splunk）
> 
> **3. ハイブリッドアクセス（オンプレからクラウドをキャッシュ経由でアクセス）** → **Storage Gateway**
> - 「キャッシュ」「オンプレ側にデータ保持」のキーワード
> - バックアップ機能あり（ただしS3へのみ）
> 
> **4. 物理輸送（超大容量・回線貧弱）** → **Snow Family**
> - 「数週間かかる」「EB級」「オフライン」のキーワード
> - 初期導入費用あり
