# EC2やVPCとは（AWS主要サービスの全体像）

AWSの最重要サービスを「オンプレミス（自社運用）の何に相当するか」で整理する入門ノート。

## 主要サービスの定義

| サービス | 正式名称 | 何をするもの | オンプレでの相当物 |
|---|---|---|---|
| **VPC** | Virtual Private Cloud | AWS内に作る自分専用の仮想ネットワーク | 会社のネットワーク |
| **EC2** | Elastic Compute Cloud | 仮想サーバー（CPU・メモリ・OSを選んで起動） | 物理サーバー／レンタルサーバー |
| **RDS** | Relational Database Service | マネージドなリレーショナルDB（PostgreSQL/MySQL等） | DBサーバー |
| **S3** | Simple Storage Service | オブジェクトストレージ（ファイル置き場） | ファイルサーバー |
| **AMI** | Amazon Machine Image | EC2の起動テンプレート（OS＋設定＋ディスク内容） | サーバーのマスターイメージ |
| **IAM** | Identity and Access Management | ユーザー・権限の管理 | 社内の権限管理・入館証 |

> [!tip] 例え：AWS＝クラウド上の「会社まるごと」
> 会社ネットワーク（VPC）の中に、サーバー（EC2）・DBサーバー（RDS）を置き、ファイルサーバー（S3）に資料を保存し、入館証（IAM）で誰が何をできるか管理する──という対応で覚えると全体像がつかめる。
> ※ 厳密にはS3はVPCの**外**（リージョンサービス）にある点に注意。

## VPCとサブネット

**VPC** = AWS内に作る、論理的に分離された自分専用のネットワーク空間。IPアドレス範囲（CIDR、例: `10.0.0.0/16`）を自分で決める。

**サブネット** = VPCをさらに小分けにした区画。**1つのサブネットは1つのAZ（アベイラビリティゾーン）に属する**。

| 種類 | 定義 | 置くもの |
|---|---|---|
| **パブリックサブネット** | インターネットゲートウェイ（IGW）へのルートを持つ | Webサーバー、ALB、NAT Gateway |
| **プライベートサブネット** | IGWへの直接ルートを持たない | DB（RDS）、内部アプリサーバー |

```
VPC（10.0.0.0/16）
 ├ パブリックサブネット（10.0.1.0/24）
 │   └ EC2（Webサーバー）← インターネットから到達可能
 └ プライベートサブネット（10.0.2.0/24）
     └ RDS（DB）← インターネットから直接アクセス不可
```

> [!important] パブリック／プライベートの違いは「ルートテーブルにIGWへのルートがあるか」で決まる。サブネット自体に種類の設定があるわけではない。

詳細 → [[0401_IPアドレス・CIDR]] / [[0405_ネットワークとコンテンツ配信]]

## EC2

仮想サーバー。インスタンスタイプ（CPU・メモリの組み合わせ）とAMI（OSイメージ）を選んで数分で起動できる。

- 課金モデル（オンデマンド／リザーブド／スポット）→ [[0102_EC2購入プラン]]
- ストレージ（EBS／インスタンスストア／EFS）→ [[0103_EC2のストレージ]]
- 停止・休止・終了の違い → [[0107_EC2インスタンスの休止状態]]

## AMI

EC2を起動するためのテンプレート。**OS＋アプリ＋設定＋ディスク内容（EBSスナップショット）**のセット。

```
AMI（テンプレート）──起動──→ EC2インスタンス（実体）
```

同じ構成のサーバーを量産できるため、**Auto Scaling の前提**になる。詳細 → [[0104_スナップショット・AMIとは]]

## RDS / S3 / IAM

- **RDS**: DBエンジン（PostgreSQL/MySQL/Oracle等）をAWSが運用代行（パッチ・バックアップ・フェイルオーバー）→ [[0301_RDS]]
- **S3**: 容量無制限のオブジェクトストレージ。静的Webサイトホスティングも可能 → [[0201_S3]] / [[0108_EC2とS3のWebサイトの違い]]
- **IAM**: 「誰が（ユーザー/ロール）」「何に（リソース）」「何をできるか（ポリシー）」を管理 → [[0601_IAM基礎]]

## 典型的なWebアプリ最小構成

```
Internet
   ↓
ALB（ロードバランサー）          ← パブリックサブネット
   ↓
EC2（Django等のアプリ）          ← プライベートサブネット
   ↓
RDS（PostgreSQL）               ← プライベートサブネット
＋
S3（画像・静的ファイル）         ← VPC外（リージョンサービス）
```

## SAA試験ポイント

| 問われ方 | 答え |
|---|---|
| インターネットから直接アクセスさせたくないDBはどこに置く？ | **プライベートサブネット** |
| サブネットをパブリックにする条件は？ | ルートテーブルに**IGWへのルート**がある |
| 1つのサブネットは複数AZにまたがれる？ | **またがれない**（サブネットは単一AZ） |
| 同じ構成のEC2を自動で増やす（Auto Scaling）には何が必要？ | **AMI**（起動テンプレート） |
| S3はVPCの中にある？ | **ない**（リージョンサービス。VPC内からは VPCエンドポイント経由でアクセス可能） |

## 最低限覚える対応表

```
VPC   = 自分専用の仮想ネットワーク
EC2   = 仮想サーバー
RDS   = マネージドDB
AMI   = EC2の起動テンプレート
S3    = オブジェクトストレージ（VPC外）
IAM   = 権限管理
```
