# EBSボリュームタイプ

> [!info] 一言でいうと
> **EC2に付ける永続ブロックストレージの性能メニュー**。
> まずは汎用SSDの `gp3`、高性能DB向けの `io2`、大容量・順次アクセス向けの `st1` / `sc1` を見分ける。

> [!tip] たとえ
> 同じ「外付けディスク」でも、普段使いSSD、高級な業務用SSD、大容量HDD、安い保管用HDDを使い分けるイメージ。

---

## 最重要の選び方

| ボリューム | 種類 | 向く用途 | SAAキーワード |
|---|---|---|---|
| `gp3` | 汎用SSD | OS、一般的なアプリ、中規模DB | コスト効率、汎用、通常はこれ |
| `gp2` | 旧寄りの汎用SSD | 既存環境で見かける | サイズに性能が連動しがち |
| `io2` | プロビジョンドIOPS SSD | 高I/OのDB、低レイテンシ、高耐久 | 高IOPS、低レイテンシ、ミッションクリティカルDB |
| `st1` | スループット最適化HDD | ビッグデータ、DWH、ログ処理 | 大容量、順次読み書き、安い |
| `sc1` | コールドHDD | 低頻度アクセス、最低コスト | めったに読まない、最安 |

> [!warning] HDD系はブートボリュームに使えない
> `st1` / `sc1` は大容量の順次アクセス向け。OS起動ディスクや小さなランダムI/OのDBには向かない。

---

## gp3 と io2 の違い

| 観点 | `gp3` | `io2` |
|---|---|---|
| 位置づけ | 汎用SSD | 高性能・高耐久SSD |
| 主な用途 | 通常のEC2、一般的なDB | I/O集約型DB、厳しいレイテンシ要件 |
| 性能指定 | IOPS・スループットを独立して調整できる | 高いIOPSを安定してプロビジョニング |
| 試験での扱い | 迷ったら標準解になりやすい | 「高IOPS」「低レイテンシ」「ミッションクリティカルDB」で選ぶ |

> [!tip] SAAでは数値より判断軸
> 詳細な上限値は更新されるため、試験では「`gp3` = 一般用途」「`io2` = 高IOPS/低レイテンシDB」「`st1/sc1` = 大容量HDD」を優先して覚える。

---

## EBSと他ストレージの混同防止

| 要件 | 第一候補 |
|---|---|
| EC2に接続するOS/DB用ディスク | EBS |
| 複数EC2から同時にLinux共有 | EFS |
| Windows共有、SMB、AD統合 | FSx for Windows File Server |
| HPC/機械学習でS3連携の高速並列FS | FSx for Lustre |
| HTTPで大量オブジェクトを保存 | S3 |
| 消えてよい超低レイテンシ一時領域 | インスタンスストア |

---

## SAA判断フロー

```
EC2に付ける永続ディスク？
├── No → S3 / EFS / FSx / インスタンスストアを検討
└── Yes
    ├── 高IOPS・低レイテンシDB？ → io2
    ├── 大容量の順次アクセス？ → st1
    ├── 低頻度アクセスで最低コスト？ → sc1
    └── 一般的な用途？ → gp3
```
