# EC2 購入プラン（課金モデル）

EC2 (Amazon Elastic Compute Cloud) は、AWSが提供するクラウド上の仮想サーバーサービス。
数分でサーバーを作成し、CPU・メモリ・OSなどを自由に設定して利用できる。
基本 → [[0101_EC2やVPCとは]]

---

## 課金モデル一覧

| モデル | 概要 | 割引率 | 柔軟性 | 向いている用途 |
|---|---|---|---|---|
| オンデマンドインスタンス | 使った分だけ課金 | なし | 非常に高い | 開発・テスト・短期利用 |
| リザーブドインスタンス（RI） | 1年または3年の利用を予約 | 最大約72% | 低い | 長期運用 |
| Savings Plans | 1時間あたりの利用額をコミット | 最大約66% | 高い | 継続的な利用 |
| スポットインスタンス | AWSの余剰リソースを利用 | 最大約90% | 非常に低い（**中断あり**） | バッチ・データ処理 |

> [!warning] スポットの注意点
> AWS側の都合で**2分前の通知のみで中断**される。**中断されても問題ないワークロード**（バッチ・CI・ステートレス処理）にのみ使う。「止まってはいけない本番Webサーバー」にスポットは不正解。

## Reserved Instance の種類

| 種類 | 割引率 | 柔軟性 | 特徴 |
|---|---|---|---|
| Standard RI | 最大約72% | 低い | インスタンスタイプなどの変更がほぼできない |
| Convertible RI | 最大約66% | 高い | インスタンスファミリーなどを変更可能 |

| 項目 | Standard RI | Convertible RI |
|---|---|---|
| 割引率 | 高い | やや低い |
| インスタンスファミリー変更 | 不可 | 可能 |
| インスタンスタイプ変更 | 制限あり | 可能 |
| OS変更 | 不可 | 可能 |
| 向いている用途 | 構成が固定されたシステム | 将来変更の可能性があるシステム |

> [!warning] 「リザーブド」は物理サーバーの取り置き予約ではない
> RIは「1年/3年使い続けます」という**料金割引の契約**であって、特定の物理サーバーを自分専用に確保する意味ではない。
> - 物理サーバーを専有したい → **Dedicated Host**（下記）
> - 起動枠を確実に確保したい → **Capacity Reservation**（下記）
> インスタンス自体は通常どおり起動・停止でき、契約条件に合うインスタンスを動かしている間に割引が適用される。

> [!tip] オンデマンド vs リザーブドの判断フロー
> ```
> 1年以上使う？ → 24時間ずっと動かす？ → 利用量が予測できる？
>   → すべてYes：リザーブドインスタンス
> 逆に以下ならオンデマンド：
>   数日だけの検証／いつ停止するか分からない／
>   インスタンスタイプ未確定／長期契約を避けたい
> ```
> 「金融系・24時間365日・1年以上運用」のようなシナリオはEC2もDB（Aurora/RDS）もRIが定番解。DB側のRI（ストレージは別会計で従量課金のまま）は [[0307_Aurora詳細]] 参照。

## Savings Plans の種類

| 種類 | 柔軟性 | 割引率 | 特徴 |
|---|---|---|---|
| Compute Savings Plans | 非常に高い | 最大約66% | インスタンスファミリー・リージョン変更可能。**Fargate・Lambdaにも適用** |
| EC2 Instance Savings Plans | やや低い | 最大約72% | インスタンスファミリー固定 |

> [!info] RI と Savings Plans の違い
> - **RI** = 「このインスタンス構成を使い続けます」と**構成を**予約
> - **Savings Plans** = 「1時間あたり$Xぶん使い続けます」と**金額を**コミット
> 割引率はほぼ同等。新規ならSavings Plansの方が柔軟で推奨されることが多い。

## 専有系オプション（Dedicated Host / Dedicated Instance）

| 項目 | Dedicated Host（専有ホスト） | Dedicated Instance（専有インスタンス） |
|---|---|---|
| 専有単位 | **物理サーバー1台を専有** | 物理サーバーは専有だが**配置先のホストは指定できない** |
| 物理サーバーの可視性 | ソケット数・コア数が見える | 見えない |
| **ソケット/コア単位のライセンス（BYOL）** | ✅ **対応**（Oracle / SQL Server / Windows Server等） | ❌ 非対応 |
| 用途 | ライセンス要件・コンプライアンス | 他テナントとハードを共有したくないだけの場合 |

> [!important] SAA頻出の区別
> 「**ソケット単位・コア単位の既存ライセンスを持ち込みたい**」→ **Dedicated Host** 一択。
> Dedicated Instance は「物理的に他社と分離したい」だけの要件向けで、ライセンス紐付けはできない。

```
物理サーバー (Dedicated Host)
├ EC2インスタンス
├ EC2インスタンス
└ EC2インスタンス
```

## キャパシティ予約 (Capacity Reservation)

特定AZでEC2インスタンスを**確実に起動できるよう起動枠を確保する**仕組み。

| 項目 | 内容 |
|---|---|
| 目的 | EC2インスタンス起動枠の確保（割引ではない） |
| 割引 | **なし**（オンデマンド料金） |
| 課金 | **インスタンスを起動していなくても課金される**（席の予約料と同じ） |
| 組み合わせ | RI / Savings Plans と併用すれば割引も効かせられる |

| 仕組み | 目的 |
|---|---|
| Capacity Reservation | EC2**起動枠**の確保 |
| Reserved Instances | EC2**料金**割引 |
| Savings Plans | EC2**料金**割引 |
| Dedicated Host | 物理サーバー専有（ライセンス） |

## SAA試験ポイント

| 問題文のキーワード | 答え |
|---|---|
| 中断されても再実行できるバッチ／コストを最大限削減 | **スポットインスタンス** |
| 1〜3年継続利用・構成固定 | **Standard RI**（または EC2 Instance Savings Plans） |
| 1〜3年継続利用・将来構成が変わるかも | **Convertible RI** / **Compute Savings Plans** |
| Lambda・Fargateにも割引を効かせたい | **Compute Savings Plans** |
| ソケット/コア単位のライセンス持ち込み（BYOL） | **Dedicated Host** |
| イベント当日に確実にインスタンスを起動したい | **Capacity Reservation** |
| 短期・不定期・まずは試す | **オンデマンド** |

## 実務でよくある構成

| ワークロード | 利用モデル |
|---|---|
| ベース負荷 | Savings Plans / RI |
| 突発的なアクセス | オンデマンド |
| バッチ処理 | スポット |
| 起動保証が必要 | Capacity Reservation |

## まとめ（ひとこと暗記）

| モデル | 一言 |
|---|---|
| オンデマンド | とりあえず使う |
| Reserved Instance | **構成**を予約して割引 |
| Savings Plans | **金額**をコミットして割引 |
| Spot | 余剰リソース（中断あり・最安） |
| Dedicated Host | 物理専有（ライセンス持ち込み可） |
| Capacity Reservation | 起動枠の確保（割引なし） |
