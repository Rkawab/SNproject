# プレイスメントグループ（Placement Group）

**EC2インスタンスを物理ハードウェア上で「どう配置するか」を制御する仕組み**。
目的に応じて3種類あり、「近づけて高速化」か「離して耐障害化」かを選ぶ。

## 3種類の比較

| 種類 | 配置 | 特徴 | 用途 |
|---|---|---|---|
| **クラスター** | 同一AZ内で物理的に近接 | インスタンス間が超低遅延・高帯域 | HPC・ML・並列バッチ |
| **スプレッド** | 異なるラックに分離 | ハードウェア障害の影響を分離（**1AZあたり7台まで**） | 少数の重要サーバー |
| **パーティション** | ラックのグループ（パーティション）単位で分離 | 大規模でも分散配置できるバランス型 | HDFS・Kafka・Cassandra等の分散システム |

## スプレッドの「分離レベル」の位置づけ

| レベル | 分離対象 |
|---|---|
| リージョン | 東京 vs 大阪 |
| AZ | データセンター（建物）単位 |
| スプレッドPG | **ラック単位**（←ここ） |

```
スプレッドなし: 同じラックに2台 → ラックの電源障害で両方ダウン
スプレッドあり: ラックA/B/Cに1台ずつ → 1ラック壊れても他は無事
```

## クラスタープレイスメントグループの深掘り（SAA頻出）

大規模バッチ・HPC・ビッグデータのように、**インスタンス間で大量データを低遅延・高帯域で転送**したいときの定番。

> [!important] 単一AZ限定という制約
> クラスター配置は **必ず単一AZ内** でしか組めない。インスタンスを物理的に近接させて低遅延・高帯域を実現する仕組みなので、AZ(＝別の建物)をまたぐと原理的に成立しない。
> → **「クラスタープレイスメントグループ＋マルチAZ」という選択肢は成立せず、典型的な引っかけ**。

> [!tip] 単一AZでも回復性は確保できる
> 「単一AZ＝回復性ゼロ」ではない。**Auto Scalingグループ**と組み合わせれば、インスタンス単位の障害を自動で再起動・補充してくれる。
> → 「低遅延を維持したまま、できる範囲で回復性も確保したい」要件には **クラスターPG(単一AZ)＋Auto Scaling** が正解パターン。
> （AZ障害そのものへの耐性は捨てているが、低遅延が最優先のワークロードでは妥当なトレードオフ）

## SAA試験ポイント：キーワード → 配置タイプ

| 問題文のキーワード | 選ぶべき配置 |
|---|---|
| インスタンス間で低遅延・高帯域・大量データ転送 / HPC / ビッグデータ / 並列バッチ | **クラスター（単一AZ）** |
| 重要インスタンスの可用性を最大化 / ハードウェア障害の影響を分離 | **スプレッド** |
| 大規模分散システム（HDFS / Kafka / Cassandra）/ ラック単位で分けたい | **パーティション** |
