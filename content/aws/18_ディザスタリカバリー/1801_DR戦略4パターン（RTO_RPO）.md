# DR戦略4パターン（RTO / RPO）

## 概要

**ディザスタリカバリー（DR）** は、プライマリリージョン障害時にセカンダリリージョンへ業務を引き継ぐ仕組み。戦略は **RTO（復旧時間）** と **RPO（許容データ損失）** の要件で決まり、短いRTOほど高コスト。

| 戦略 | RTO | RPO | コスト | イメージ |
|-----|-----|-----|--------|---------|
| **バックアップ&リストア** | 数時間〜 | 数時間〜 | 最安 | 倉庫に設計図・在庫を置く |
| **パイロットライト** | 数分〜 | 数分〜 | 中 | 核だけ灯す |
| **ウォームスタンバイ** | 数分以内 | 数分以内 | 高 | 支店を最小人数で営業中 |
| **マルチサイト / ホットスタンバイ** | リアルタイム | リアルタイム | 最高 | 支店も本店と同規模で常時営業 |

---

## 戦略1：バックアップ&リストア（最遅・最安）

### 特徴

- **セカンダリに何も置かない**。バックアップ（ストレージ）のみ保管
- 障害時に、バックアップからゼロから構成を再構築
- **起動・登録・設定に数時間以上かかる** → 最も遅い復旧

### AWS実装

```text
バックアップ方式：
1. AWS Backup で定期的にEBS/RDS/EFS等をバックアップ
2. バックアップをセカンダリリージョンへクロスリージョンコピー
3. 障害時：
   - バックアップからAMI/スナップショットを復元
   - EC2/RDSを新規起動
   - ALB/セキュリティグループを再作成
   ⇒ 数時間かかる
```

### RTO目安

- **4時間以上**（復元開始→本格稼働）

### 使用場面

- 非ミッションクリティカルなシステム
- 復旧に数時間かかっても許容できる業務

### 最大の罠（試験頻出）

「定期バックアップは自動取得される」という安心感で選ぶ受験者が多いが、**バックアップから復元・再起動・再構成する時間は含まれない**。「**2分以内のRTO要件**」があったらこれはNG。

---

## 戦略2：パイロットライト（種火を灯す）

### 特徴

- **セカンダリリージョンに最小限のコンピュート（例：低スペックEC2 1台）だけ稼働**
- アプリケーションの核となるコンポーネントはセカンダリで動いているが、データベースはプライマリを指したまま
- 障害時に、セカンダリのEC2から**AMIを起動して追加のインスタンスを建てる**
- データはプライマリのバックアップから復元または、リアルタイムレプリケーション活用

### AWS実装

```text
セットアップ時：
1. セカンダリリージョンにEC2 1台（最小スペック）起動
   - アプリケーション環境をAMIとして保持
   - 常に稼働している（コストは低い）
2. プライマリ ⇄ セカンダリ間のデータレプリケーション
   - RDS Read Replica
   - S3 Cross-Region Replication (CRR)

障害時：
1. AMIからEC2を数台起動（数分以内）
2. Route 53 フェイルオーバーで名前解決をセカンダリに向ける
3. スケールアウトして負荷対応

⇒ 起動待ちで数分かかる
```

### RTO目安

- **5～10分以上**（起動待ち・登録待ち）

### 使用場面

- RTO数分程度が許容される中程度のミッションクリティカル業務
- 定期的にセカンダリでフェイルオーバーテスト可能

### 理由

- 起動を待つため、「2分以内」というRTO要件には間に合わない

---

## 戦略3：ウォームスタンバイ（支店を最小人数で営業中）→ **SAA試験では最頻出**

### 特徴

- **セカンダリリージョンで最小限のスケール（例：t3.medium 2台）で常時稼働**
- アプリケーション・データベース・ロードバランサーが既に起動している
- Route 53 ヘルスチェック＋フェイルオーバールーティングで、障害時に**トラフィックを瞬時に切替**
- 本格負荷への対応は、フェイルオーバー後に**オートスケーリング**で増員

### AWS実装

```text
セットアップ時：
1. セカンダリリージョンで小規模構成を常時稼働
   - EC2（t3.medium等）× 2台：常時起動
   - RDS（マルチAZ DBクラスター/読み取りレプリカ）：常時レプリケーション
   - ALB：常時起動
   - Auto Scaling グループ：最小2台～最大10台等
2. Route 53 フェイルオーバールーティング：ヘルスチェック＋自動切替

プライマリ障害時：
1. Route 53 が自動検知（ヘルスチェック失敗）
2. DNS 回答をセカンダリに即座に切替 ⇒ クライアントがセカンダリへ接続
3. セカンダリで既に稼働中なので "即座に受け入れ可能"
4. 負荷が増えたら Auto Scaling で増員

⇒ **起動待ちゼロ** = 数分以内のRTO達成
```

### RTO / RPO目安

- **RTO：2～5分以内**（フェイルオーバー検知＋DNS切替）
- **RPO：数分以内**（リアルタイムレプリケーション利用）

### 使用場面

- **「2分以内」「即座に」「常時切替」などの厳しいRTO要件**
- ミッションクリティカルなシステム（金融決済、医療等）

### SAA試験の見分け方

```
問題文キーワード：
✓ 「2分以内」「数分以内」「短いRTO」
✓ 「ウォームスタンバイ」と明示
✓ 「最小限の構成で常時稼働」
✓ 「フェイルオーバー自動化」
✓ 「Route 53」+「ヘルスチェック」

→ 答え：ウォームスタンバイ（選択肢C等）
```

## 戦略4：マルチサイト / ホットスタンバイ（支店も本店と同規模）

### 特徴

- **プライマリとセカンダリを同規模で常時運用**
- **Active-Active** 構成：両リージョンが並行して本番トラフィックを受ける
- 片方の障害でも、もう片方が全量引き受け可能
- **ゼロダウンタイム**

### AWS実装

```text
セットアップ時：
1. プライマリリージョン：フルスケール構成
   - EC2 × 20台、RDS Large等
2. セカンダリリージョン：同規模構成（常時稼働）
   - EC2 × 20台、RDS Large等
3. Global Accelerator / Route 53 (weighted, failover)
   - 両リージョンへ並行振分
   - 片方障害時は自動切替

プライマリ障害時：
⇒ セカンダリが既に全量を受けられる状態なので**ほぼゼロダウンタイム**
```

### RTO / RPO目安

- **RTO：ほぼゼロ（リアルタイム切替）**
- **RPO：ほぼゼロ（常時同期）**

### コスト

- **最高**（常に2倍のコンピュートコスト発生）

### 使用場面

- 金融取引所、オンライン決済、通信キャリア等
- ダウンタイムが絶対に許されない超ミッションクリティカル

## RTO / RPO の定義

### RTO（Recovery Time Objective：復旧時間目標）

```
障害発生時刻 ─────────────────→ サービス復旧時刻
              ← RTO の時間 →
```

- **障害検知～業務再開までの時間**
- 短いほど、戦略コストは高い
- 試験問題では「**2分以内**」「**即座に**」と書かれたら、すでに稼働している戦略を選ぶ

### RPO（Recovery Point Objective：目標復旧時点）

```
障害発生時刻 ─────────────────→
最後にセーフ ← RPO の時間 ←
データ時刻
```

- **障害時に許容できるデータ損失の最大範囲**
- 例：RPO=1時間 → 最大1時間前のデータが失われる可能性
- リアルタイムレプリケーション（RDS Multi-AZ DBクラスター、S3 CRR等）で RPO を小さく

## 4戦略の選択基準（早見表）

```
問題文に「RTO = ??分」という制約が書かれている？

┌─ RTO ≤ 数分（2分、5分など短い）
│  ⇒ 起動待ちのある戦略（バックアップ&リストア、パイロット）は脱落
│  ⇒ 「常時稼働」している戦略を選ぶ
│     → ウォームスタンバイ（最小構成で常時稼働）
│     → マルチサイト（同規模で常時稼働）
│
└─ RTO ≥ 数時間（許容度が大きい）
   ⇒ コスト最小化を重視
   ⇒ バックアップ&リストア（最安）を検討
```

### 各戦略の選ばれるシーン（問題文パターン）

| 問題文の条件 | 答え | 理由 |
|-------------|------|------|
| 「復旧が遅くても大丈夫」「定期バックアップは取得済み」 | バックアップ&リストア | 最安 |
| 「最小スケールで準備」「起動待ちは許容」「パイロット」と明示 | パイロットライト | 起動に数分かかるがOK |
| 「**2分以内**」「**即座に**」「**常時稼働**」「**フェイルオーバー**」 | **ウォームスタンバイ** | 起動待ちゼロで達成 |
| 「ゼロダウンタイム」「同規模」「Active-Active」 | マルチサイト | 最速・最高コスト |

## 試験頻出パターンと罠

### 罠1：「バックアップ」に釣られる

```
❌ よくある誤選択：
「定期的にバックアップを取得している」
→ 「だからバックアップ&リストア戦略で大丈夫」
   
✓ 正解の判断：
「RTO = 2分なら、バックアップから復元する時間は含まれない」
→ ウォームスタンバイを選ぶ
```

### 罠2：マルチAZ DB「インスタンス」のスタンバイは読取不可

```
❌ 誤り：
「マルチAZ DBインスタンスのスタンバイに監査クエリを向ける」
→ スタンバイは待機専用（通常は読取不可）

✓ 正解：
「マルチAZ DB"クラスター"を使う」
→ リーダーエンドポイント = 読取レプリカ複数台で読み分散
```

### 罠3：パイロットライトとウォームスタンバイの混同

```
違い：
- パイロットライト：   最小スケール + 障害時にAMI起動で増員（数分待ち）
- ウォームスタンバイ： 最小スケール + 常時稼働 + Route 53で瞬時切替（待ち時間ゼロ）

→ RTO要件が「2分以内」なら、起動待ちのあるパイロットはNG
```

## 関連AWS機能

### ディザスタリカバリー実装に使う主要機能

| 機能 | 用途 | 戦略 |
|-----|------|------|
| **AWS Backup** | バックアップ取得・保管 | バックアップ&リストア |
| **AMI** | サーバーイメージ保存 | パイロット / ウォーム / マルチサイト |
| **RDS Read Replica** | DB読み取りレプリカ | ウォーム / マルチサイト |
| **S3 Cross-Region Replication** | S3データのリージョン間同期 | 全戦略 |
| **Route 53 Failover Routing** | ヘルスチェック＋DNS自動切替 | ウォーム / マルチサイト |
| **Auto Scaling** | 負荷に応じた自動増減 | ウォーム / マルチサイト |
| **AWS CloudFormation** | セカンダリの一括デプロイ | 全戦略 |
| **Global Accelerator** | グローバルロードバランシング | マルチサイト |

## 本番例：金融決済システム

### 要件

- RTO = 2分以内（決済が停止しない）
- RPO = 数分以内（取引記録の損失最小化）

### 選択戦略

**ウォームスタンバイ**

```
プライマリ（東京リージョン）:
- EC2（t3.medium） × 4台（常時稼働）
- RDS Multi-AZ DBクラスター（ライター＋リーダー ×2）
- ALB
- Auto Scaling（最小4台～最大20台）

セカンダリ（大阪リージョン）:
- EC2（t3.medium） × 2台（常時稼働）
- RDS Read Replica（リーダー ×2、同期レプリケーション）
- ALB
- Auto Scaling（最小2台～最大20台）

フェイルオーバー：
- Route 53 ヘルスチェック（プライマリ監視）
- 障害検知 → DNS切替（数秒）
- クライアント再接続 → セカンダリから応答

復旧後：
- セカンダリのデータをプライマリに同期
- Route 53 を元に戻す
```

## SAA試験での出題レベル

- **頻度**：高（毎回のように出題）
- **難度**：中（キーワード「2分」「常時稼働」を読めれば正答率UP）
- **ポイント**：RTO要件 ⇒ 戦略の逆引き
